# AGENTS.md

This file provides guidance to AI agents working with this repository.

## What this repo is

EDEN is a **protocol** for directed-code-evolution orchestration, not a single product. The repo contains three distinct kinds of artifact, each with different change discipline:

| Layer | Path | Authoritative? | Change discipline |
|---|---|---|---|
| Protocol specification | [`spec/`](spec/) | Yes — the source of truth | RFC-style: versioned, normative (MUST/SHOULD/MAY), carefully reviewed |
| Reference implementation | [`reference/`](reference/) | No — *one* valid impl | Normal code-review discipline |
| Conformance suite | [`conformance/`](conformance/) | Normative for tests | Black-box; must be implementation-agnostic |

Non-normative human docs live in [`docs/`](docs/).

**Phase 10 chunk 10d follow-up A complete (container-isolated `*_command`).** Each subprocess-mode worker host gains an opt-in `--exec-mode docker` that wraps every spawn of the user's `*_command` in a sibling container via DooD (Docker outside of Docker — host `/var/run/docker.sock` mounted into the worker). The wrap shape lives in [`reference/services/_common/src/eden_service_common/container_exec.py`](reference/services/_common/src/eden_service_common/container_exec.py) and is `docker run --rm -i --init --cidfile <path> --label eden.{role,host,task_id} --mount type=volume,source=<name>,target=<path>[:readonly] --mount type=bind,source=<host-path>,target=<path>[:readonly] -w <cwd> -e <KEYS>... <image> bash -lc '<original-command>'`. Mount targets inside the spawned child match the worker host's exact paths so worker-internal env vars (`EDEN_TASK_JSON`, `EDEN_OUTPUT`, `EDEN_WORKTREE`, `EDEN_EXPERIMENT_DIR`) resolve consistently in both views. The set of mounts is supplied by repeatable `--exec-volume <name:target[:ro|rw]>` and `--exec-bind <host-path:target[:ro|rw]>` flags driven by the compose overlay; eden-bare-repo, eden-worktrees, and eden-artifacts-data are pinned to literal `name:` in compose.yaml so docker daemon resolves the wrap's `--mount source=<literal>` against the same volume the worker sees. **Container lifecycle:** every spawn writes its container id to a unique `<cidfile_dir>/<role>-<spawn-uuid>.cid`; a cleanup_callback unlinks the cidfile on every terminal exit branch (graceful, fast-path, and SIGKILL escalation); a separate post_kill_callback runs `docker kill && docker rm -f` on the SIGKILL escalation branch only — necessary because killing the local `docker run` client does NOT kill the spawned container, the daemon parent keeps it alive. Both callbacks are stored on the `Subprocess` instance at spawn time (no per-call threading). At host startup, `reap_orphaned_containers(role, host=gethostname())` removes any containers labeled `eden.host=<this>` from a prior crash, scoped by host so cross-host races are impossible by construction. **Identity:** `eden-runtime:dev` mirrors the `eden:1000` user from `eden-reference:dev` — without this, worktree files written by the worker host (uid 1000) trip git's `dubious ownership` check inside the spawned child (running as root by default), and any commits the child produces would be uid-0-owned, breaking subsequent integrator reads. The wrap deliberately does not pass `--user` so experiment images that override USER (e.g. for system-level installs) are honored. **DooD socket permissions:** the worker host gets `group_add: ["${EDEN_DOCKER_GID}"]` from the compose overlay; setup-experiment probes the gid with `docker run --rm -v /var/run/docker.sock:/var/run/docker.sock alpine stat -c '%g' /var/run/docker.sock` (probe-from-inside, not host-side stat — Docker Desktop's VM has its own gid namespace). **Image strategy:** if the experiment dir contains a Dockerfile, setup-experiment builds it as `eden-experiment-<id>:dev` and writes that tag into `.env`; otherwise the default `eden-runtime:dev` (built from [`reference/compose/Dockerfile.runtime`](reference/compose/Dockerfile.runtime), python3 + git + bash + ca-certificates) is used. **Security boundary:** DooD is a soft boundary — secrets passed via `-e KEY` are visible to anyone with daemon access (which under DooD includes any other concurrent `*_command`); a malicious `*_command` has full daemon access. Documented as informative §7 in [`spec/v0/reference-bindings/worker-host-subprocess.md`](spec/v0/reference-bindings/worker-host-subprocess.md). The reference deployment trades that boundary for operational simplicity; the intent is bug isolation + dependency isolation, not hostile-code containment. **Compose overlay split:** [`compose.subprocess.yaml`](reference/compose/compose.subprocess.yaml) is the **host-mode** subprocess overlay (chunk-10d soft isolation, no docker socket access); a new [`compose.docker-exec.yaml`](reference/compose/compose.docker-exec.yaml) layered ON TOP is the only place the docker socket gets mounted into worker hosts and `group_add: ["${EDEN_DOCKER_GID}"]` applied — so a layering bug can't accidentally grant DooD privilege to the host-mode path. Both smoke scripts assert this directly via `docker inspect eden-implementer-host`: host mode requires the socket bind to be ABSENT; docker mode requires it to be PRESENT. New `compose-smoke-subprocess-docker` CI job mirrors `compose-smoke-subprocess` end-state assertions (≥3 trial.integrated, ≥9 task.completed, ≥3 plan-task.completed) plus a per-task orphan-container assertion (no `eden.role=implementer|evaluator` containers left after quiescence) and a clean-teardown check (after `compose stop --timeout 15`, no `eden.role=planner` sibling remains — clean-SIGTERM or SIGKILL-escalation is acceptable for the smoke). The dedicated SIGKILL-escalation invariant (post_kill_callback actually `docker kill`s a sibling whose user command ignored SIGTERM) is exercised by the `pytest.mark.docker` integration test `test_terminate_sigkill_path_invokes_post_kill_callback` in [`reference/services/_common/tests/test_container_exec_integration.py`](reference/services/_common/tests/test_container_exec_integration.py), which also drives the actual production worktree-with-gitlink mount shape (bare repo + worktree's `.git` gitlink, both mounted into the spawned child). Branch protection is **not** updated to require the new job in this chunk (same posture chunks 10a / 10b / 10c / 10d / 10e took for newly-added jobs).

## Contribution conventions

### Spec edits

+ Every cross-component contract (wire format, state machine, invariant) is defined in `spec/` first, then implemented in `reference/`. When they disagree, the spec wins and the impl gets a bug.
+ Spec chapters use RFC 2119 language: MUST, SHOULD, MAY.
+ Spec versions are frozen once stable (`spec/v0/` is a single lineage; breaking changes go to `spec/v1/`).
+ Schema changes must update the spec prose, the JSON Schema file, and (from Phase 3 onward) the Pydantic bindings in lockstep. CI will enforce schema ↔ model parity once both exist.

### Reference implementation

+ Normal code-review discipline.
+ Normative behavior is tested against the conformance suite (once it exists in Phase 11).
+ The reference impl is explicitly not a monopoly on correctness. A third-party implementation that passes conformance is equally valid.

### Adding a new service or package with its own `tests/` directory

Pytest's default `prepend` import mode names test modules by file basename, so two `tests/test_app.py` files in different packages collide on collection (`import file mismatch`). The repo's tests-dir layout is mixed — some packages (e.g. [`eden-contracts/tests/`](reference/packages/eden-contracts/tests/)) have `__init__.py` and use `from .conftest import …`; most service test dirs (e.g. [`web-ui/tests/`](reference/services/web-ui/tests/)) have no `__init__.py` and use `from conftest import …`.

When adding a new tests directory:

+ Use a basename that is unique across the whole `testpaths` set in [`pyproject.toml`](pyproject.toml). Grep `find reference -name 'test_*.py' -exec basename {} \;` to verify before merging.
+ Do **not** add `__init__.py` to a service tests dir while another tests dir already has one — both directories become the `tests` package and the second conftest collides on `tests.conftest` import/registration. Under default `prepend` mode this surfaces as `ImportPathMismatchError`; under `importlib` mode it surfaces as `Plugin already registered under a different name`.
+ Switching the whole repo to `--import-mode=importlib` looks tempting but is **not** a clean fix today: the existing eden-contracts tests use relative `.conftest` imports that depend on the `__init__.py` package layout, and importlib mode keeps that collision while removing the sys.path hack the rest of the suite relies on.

If the constraint becomes load-bearing later (Phase 10 will add several more service test dirs), the cleanest fix is a three-step lockstep: rename each tests dir to a unique package name (e.g. `web_ui_tests/` instead of `tests/`), switch every bare `from conftest import …` to a package-relative `from .conftest import …`, and add `--import-mode=importlib` to pytest's `addopts`. A mechanical basename-uniqueness check in CI is also a sensible follow-up — AGENTS-only guidance is weaker than a guardrail.

### Adding or extending a JSON Schema + Pydantic binding

The `schema-parity` CI job is only as strong as what both sides of the test actually enforce. Several Pydantic and `jsonschema` defaults let drift through silently. When adding a new schema — or a new field type to an existing one — evaluate each of the following; reusable implementations live in [`reference/packages/eden-contracts/src/eden_contracts/_common.py`](reference/packages/eden-contracts/src/eden_contracts/_common.py).

+ **Strict numeric parsing.** Top-level models set `ConfigDict(strict=True, extra="allow")`. Non-strict mode coerces `True`/`"2"` into int, but the schemas treat `type: integer` / `type: number` as strict JSON types. New models must keep `strict=True`.
+ **Format assertions on both sides.** `format` keywords (`uri`, `date-time`, …) are advisory by default in both the `jsonschema` library and Pydantic. The schema-side validator wires a custom `FormatChecker` in [`tests/conftest.py`](reference/packages/eden-contracts/tests/conftest.py), and the model side uses the reusable types in `_common.py`. A new `format` keyword in any schema requires handlers on *both* sides; the `test_format_coverage` test fails loudly if the schema-side handler is missing.
+ **Real date-time / URI validation, not just regex or `urlparse`.** The regex on `DateTimeStr` accepts impossible values like `2026-99-99T…Z`; an `AfterValidator` runs `datetime.fromisoformat` to reject those. `UriStr` uses `rfc3986-validator`, not `urllib.parse.urlparse`, which accepts malformed schemeful URIs (e.g., spaces in the host).
+ **Null vs absent.** JSON Schema's `type: X` rejects explicit `null`, but Pydantic's `X | None = None` accepts it. Wrap every optional typed field with `NotNone` from `_common.py` so absent is accepted and explicit null is rejected.
+ **Round-trip emission.** `model.model_dump(mode="json", exclude_none=True)` must re-validate against the schema. Add any new model to `tests/test_roundtrip.py` so this is checked.
+ **Corpus coverage.** Parity is asserted over the fixture corpus in `tests/cases.py`. A new field type deserves at least one accept fixture and one reject fixture per constraint the schema imposes (required, pattern, enum, min/max, format, cross-field).

### Adding a new CI job

GitHub Actions' `ubuntu-latest` is now Ubuntu 24.04 with a system Python protected by [PEP 668](https://peps.python.org/pep-0668/) — bare `pip install` fails with `error: externally-managed-environment` instead of installing. Bash scripts that run on both CI (bash 5+) and contributor laptops (macOS ships bash 3.2 by default) hit the parallel constraint that bash-4-only builtins like `mapfile` / `readarray` silently work in CI but fail locally with `command not found`.

When adding a new CI job that needs Python or runs bash scripts:

+ **Python-using jobs must declare an interpreter via `actions/setup-python`** before any `pip install`. setup-python provisions an isolated interpreter that pip can write to freely, and adds it to PATH so a downstream `python3 …` resolves to the same interpreter the workflow installed into. Use `python3 -m pip install …`, not bare `pip install …`, to make the install/import binding explicit. The chunk-10e [`compose-e2e`](.github/workflows/ci.yml) job is the canonical small example: setup-python step, then `python3 -m pip install 'httpx>=0.27,<1'`, then a bash wrapper that calls `python3` on the driver. Jobs that don't need a custom Python (the existing `compose-smoke` etc. that only call `docker compose` / `jq` / `curl`) skip setup-python entirely.

+ **Bash scripts under [`reference/compose/healthcheck/`](reference/compose/healthcheck/) and [`reference/scripts/`](reference/scripts/) must work under bash 3.2.** macOS' default `/bin/bash` is 3.2 and contributors run these scripts locally to repro CI failures. The two non-obvious incompatibilities to watch for: (1) `mapfile -t arr < <(cmd)` does not exist — use `arr=(); while IFS= read -r line; do [[ -n "$line" ]] && arr+=("$line"); done < <(cmd)` instead. (2) Associative arrays (`declare -A`) do not exist either; use a parallel-arrays pattern or a different data structure. `set -euo pipefail` and process substitution `< <(…)` both work fine in bash 3.2; only the named bash-4 builtins are off-limits.

## Plan-writing pitfalls

These are the recurring traps that have shown up across chunks 9c / 9d / 10d / 10e / 10d-followup-A / 10d-followup-B during `/codex-review`. Check each before submitting a plan that touches the orchestrator loop, the wire surface, or any role's submission flow.

+ **Verify wire-touching shapes against the actual dataclasses, not the spec prose alone.** The submission types in [`reference/packages/eden-storage/src/eden_storage/submissions.py`](reference/packages/eden-storage/src/eden_storage/submissions.py) carry only `status` + role-specific id/payload fields — there is no free-form `description`. If your plan threads a `description` through the wire, you'll have to either drop it (route to host log only) or amend the spec + bindings + contracts together. Same discipline applies to event payloads: check [`reference/packages/eden-contracts/src/eden_contracts/event.py`](reference/packages/eden-contracts/src/eden_contracts/event.py) before assuming a field exists.
+ **Any code path that produces a trial must run `Store.create_trial(status="starting")` before any observable repo write.** This is [`spec/v0/03-roles.md`](spec/v0/03-roles.md) §3.2 step 1, and the chunk-9c web-ui implementer at [`reference/services/web-ui/src/eden_web_ui/routes/implementer.py`](reference/services/web-ui/src/eden_web_ui/routes/implementer.py) is the canonical reference flow. Both the chunk-9c and 10d plans skipped the starting-trial step in their first revision until `/codex-review` caught it. The recovery path (chunk-1 expired-claim sweeper composite-commits the orphan trial to `error`) only works if the trial actually exists in `starting` state when the host crashes.

+ **For compose-bound integration tests, audit which always-on services would race with the test.** The reference-host services (`planner-host`, `implementer-host`, `evaluator-host`) tight-loop-poll their respective task kinds on startup and claim *every* eligible task without filtering. So a test that wants to drive a task through the web UI cannot simply `compose up && stop <competing-host>` — by the time the stop returns, the competing host has already drained the seed. The chunk-10e [`e2e.sh`](reference/compose/healthcheck/e2e.sh) shows the right pattern: stage 1 brings up everything *except* the competing host, the test claims its targets via the UI, then stage 2 brings the competing host online so the rest of the experiment proceeds. Rule of thumb when planning: enumerate the polling services and ask "could the user's intent be drained before my test runs?" — if yes, defer the competing service to stage 2.

+ **DooD wraps need explicit `name:` on every forwarded compose volume, and the spawned-child user identity must match the worker host's.** Two traps the 10d-followup-A implementation hit in succession during local-smoke debugging: (1) compose auto-prefixes volume names with the project (`eden-reference_eden-bare-repo`); a wrap that does `--mount source=eden-bare-repo` against the host docker daemon resolves to a *fresh empty volume* unless the compose volume has explicit `name:` matching the wrap's literal — see the `name:` discipline on [`compose.yaml`](reference/compose/compose.yaml) and [`compose.subprocess.yaml`](reference/compose/compose.subprocess.yaml). (2) The default container user differs between `eden-reference:dev` (eden:1000) and a freshly-pulled base image (root); spawned children that run as root against worktree files written by uid 1000 trip git's `dubious ownership` check, AND any commits they produce are uid-0-owned breaking subsequent integrator reads. The fix is to mirror the `eden:1000` user in `eden-runtime:dev` (the wrap deliberately doesn't pass `--user` so experiment-image USER overrides are honored). Rule of thumb when planning DooD: enumerate every forwarded volume and confirm `name:` is set; pin the spawned-child user to match the worker host's, not the spawned image's default.

+ **DooD sibling containers survive SIGKILL of the local `docker run` client; the cidfile-driven `post_kill_callback` is load-bearing.** Killing the local `docker run` process via `os.killpg(SIGKILL)` does NOT propagate to the spawned container — the daemon parent keeps it running until the workload exits or it's killed externally. Plans that wrap `subprocess.Popen` with `docker run` must register a `post_kill_callback` (stored on the `Subprocess` instance, fires on the SIGKILL escalation branch) that reads the cidfile and runs `docker kill && docker rm -f`. The cleanup callback that unlinks the cidfile must run on **every** terminal exit branch (graceful, fast-path, and SIGKILL) — not just SIGKILL — because `--cidfile` errors if the path exists, so cidfile reuse without cleanup is a hard failure. See [`reference/services/_common/src/eden_service_common/container_exec.py`](reference/services/_common/src/eden_service_common/container_exec.py) and the §D.3 design notes in [`docs/plans/eden-phase-10d-followup-a-container-isolation.md`](docs/plans/eden-phase-10d-followup-a-container-isolation.md).

+ **Three load-bearing wiring traps in any DooD sibling-container wrap.** The container-isolation chunk's CI burned through six rounds catching these in the wild; check each before submitting a plan that mounts the host docker socket into a worker. (1) **Privilege opt-in via separate overlay.** Putting `/var/run/docker.sock` mounts unconditionally in `compose.subprocess.yaml` would have granted DooD privilege to host-mode users by accident. The fix was a third overlay file ([`compose.docker-exec.yaml`](reference/compose/compose.docker-exec.yaml)) layered ON TOP of the subprocess overlay — that's the only file that mounts the socket and applies `group_add`. Both smoke scripts assert this directly via `docker inspect`. Rule of thumb: any deployment mode that needs new privilege (capability, host-network, socket mount, etc.) goes in its own compose overlay so the default keeps the privilege off. (2) **Paths in `docker run` flags resolve on the daemon side, NOT in the calling container's mount namespace.** `--cidfile <path>`, `--mount source=<path>`, `-v <host>:<container>` — all of these are interpreted on the host docker daemon, against the host filesystem. So a worker container that bind-mounts `${EDEN_CIDFILES_DIR_HOST}` at `/var/lib/eden/cidfiles` and passes `--cidfile /var/lib/eden/cidfiles/X.cid` will fail "permission denied" — that path doesn't exist on the host. The fix is to bind the cidfile dir at the SAME path inside the worker as on the host (`${EDEN_CIDFILES_DIR_HOST}:${EDEN_CIDFILES_DIR_HOST}`) so worker and daemon agree. Same posture for any path passed through `--mount source=`. (3) **`docker run -i` exits early on closed stdin.** Compose containers run with stdin closed by default; `docker run -i` propagates that closed-stdin to its parent's wait loop and exits *before* the workload has done useful work. Make `-i` conditional on whether the caller will pipe stdin to the spawn: planner subprocesses (long-running JSON-line protocol) need it; per-task implementer/evaluator spawns must NOT include it. See [`reference/services/_common/src/eden_service_common/container_exec.py`](reference/services/_common/src/eden_service_common/container_exec.py)'s `attach_stdin` parameter. (Bonus trap: `subprocess.Popen(shell=True)` runs `/bin/sh -c '<wrap>'`; on Linux that's typically dash, which exits on SIGTERM without forwarding to the docker run child. Prefix the wrap with `exec` so the shell `exec`s docker run in place — `popen.pid` then IS docker run.)

+ **Any wire/network write that can leave server state unknown to the client needs a 3-outcome read-back ladder: confirmed-success / confirmed-divergence / indeterminate.** A `git push` that returns a transport error after the daemon may have already accepted the ref is the canonical example — there's no "did it land?" answer in the exit code alone. The 10d-followup-B integrator handles this by following the failed push with `ls_remote(trial.branch)` and routing on the three outcomes: remote==our-SHA → success (proceed to step 3), remote==different-SHA → divergence (compensating-delete local, raise `RefRefused`), remote-absent-or-unreadable → indeterminate (compensating-delete local, raise `IndeterminatePush`). Same shape as `StoreClient.integrate_trial`'s read-back from Phase 8a (chapter 07 §5). Rule of thumb when planning: any write to a remote system whose response can be lost in transit (HTTP `integrate_trial`, `git push`, future Postgres-over-network commits) needs a read-back step; do NOT collapse "transport error" into "definitely failed" — that would be a silent atomicity violation when the remote in fact accepted.

+ **Recovery code must use spec-authoritative content paths, NOT ref-name parsing.** The 10d-followup-B remote-orphan reconciler needs to recover a `trial_id` from a remote-only ref (one that exists on Gitea but has no corresponding `Trial` record in the store). The first-cut plan parsed `refs/heads/trial/<id>-<slug>` with a regex; round-2 codex review caught that [`spec/v0/02-data-model.md`](spec/v0/02-data-model.md) §1.3 keeps `trial_id` opaque (any string ≤ 64 chars, the integrator's branch-name shape is non-normative). The fix walks the ref's commit tree and reads `.eden/trials/<id>/eval.json` — the spec-authoritative manifest path defined in chapter 6 §3.2. Rule of thumb when planning: anywhere recovery / reconciliation / GC code needs to extract a spec-defined identifier from a git artifact, the source of truth is the on-tree manifest the spec mandates, not the implementation's chosen branch-naming convention. Branch names are a reference-impl convenience.

+ **Long-lived bare/non-bare clones need read-before-write/display fetches against the remote.** Two related round-0/round-1 findings on 10d-followup-B: (1) The admin work-refs page rendered from the local clone; if the integrator pushed a ref since the page-load, the admin's view was stale and the CAS-guarded delete-button could fail confusingly. The fix is `fetch_all_heads` immediately before reading refs to display, and `delete_remote_ref` BEFORE `delete_ref` so the remote-of-record is the source of truth. (2) The integrator's local clone may not have the work-branch the implementer pushed after orchestrator startup; an integrator that only fetches at startup misses any post-startup push. The fix is a per-promote-call `fetch_ref(trial.branch)` immediately before chapter-6 §2 reachability check. Rule of thumb when planning: any service that holds a clone of a remote-of-record must answer "when does this clone re-sync?" — at startup is rarely sufficient; the answer is usually "immediately before each operation that reads or writes a ref the remote could have changed."

+ **Narrow exception handling on store reads when classifying authoritative-orphan vs indeterminate.** Impl round-0 on 10d-followup-B: `reconcile_remote_orphans` initially treated any `Exception` from `Store.read_trial(trial_id)` as "trial doesn't exist → orphan → delete the remote ref". A transient transport blip (StoreClient HTTP timeout, postgres connection drop) would have deleted legitimate in-flight integrator work. The fix narrows to `except NotFound: trial = None  # authoritative orphan` and routes any other exception to `continue  # indeterminate, leave the remote alone`. Rule of thumb when planning: anywhere a "missing → take destructive recovery action" branch is being written, the read that classifies "missing" must distinguish definitive `NotFound` from transport-indeterminate failure; collapsing the two is a silent data-loss vector.

+ **Role-specific git/wire transport-failure mapping must match chapter 3 contracts.** Plan round-1 on 10d-followup-B: the first cut mapped any transport failure during git operations to a single failure mode. [`spec/v0/03-roles.md`](spec/v0/03-roles.md) actually differentiates by role — implementer transport failure on `push_ref` submits `status="error"` (chapter 3 §3.3); evaluator transport failure on `fetch_ref(trial.branch)` (preventing trial inspection) submits `status="eval_error"` (chapter 3 §4.4); integrator transport failure runs the chapter 6 §3.4 compensating-delete ladder (the integrator never "submits" — it composite-commits or rolls back). Rule of thumb when planning: when a wire/transport failure interrupts a role's flow, look up the failure-status vocabulary for THAT role in chapter 3 / chapter 6 — there is no shared "transport-failed" status across roles.

+ **Tests for callbacks/lifecycle/atomicity must drive the actual code path through the public entry point, not just call the helper directly.** The chunk's first SIGKILL test called `kill_via_cidfile()` directly — it passed even though the production path through `Subprocess.terminate()`'s SIGKILL escalation branch was broken (the branch was being skipped because `popen.wait` returned success in 0.5s). Codex round-0 caught this; the fix was a rewrite that uses `eden_service_common.spawn(...)` + `sub.terminate(shutdown_deadline=0.5)` and asserts the spawned sibling container is gone after the public terminate call. See [`test_terminate_sigkill_path_invokes_post_kill_callback`](reference/services/_common/tests/test_container_exec_integration.py) for the canonical shape. Rule of thumb: when a callback is supposed to fire on a specific branch of a public method, the test must (a) construct the same setup the production caller would, (b) invoke the public method, and (c) assert the side effect. Helper-only tests can pass while the integrated path silently skips the branch.

+ **Spec inter-chapter restatement is a conflict surface.** When the same MUST is restated across multiple chapters with the canonical statement in one and a role-side restatement elsewhere, the restatements drift over time and codex-review on a conformance plan is the place that drift surfaces. Chunk-11c hit this on evaluator-resubmit equivalence: [`spec/v0/04-task-protocol.md`](spec/v0/04-task-protocol.md) §4.2 (canonical task-store rule) listed `trial_id`+`status`+`metrics` as the equivalence formula; [`spec/v0/03-roles.md`](spec/v0/03-roles.md) §4.4 (role-side restatement) listed `trial_id`+`status`+`metrics`+`artifacts_uri`. The reference impl followed §4.2 (per its own docstring at [`reference/packages/eden-storage/src/eden_storage/submissions.py`](reference/packages/eden-storage/src/eden_storage/submissions.py) `submissions_equivalent`); §4.4 was the drift. The fix shape: the role-side restatement defers to the canonical statement (chunk-11c amended §4.4 to drop `artifacts_uri` and reference §4.2). Rule of thumb when planning: any chunk that touches a role-contract MUST should grep both chapter 03 AND chapters 04/05/07/08 for the same field/operation/invariant; if both chapters restate it with different lists, surface the conflict in the plan and resolve by deferring the role-side to the canonical chapter — don't just author tests against whichever side the impl happens to follow, because that codifies one side of the drift as conformance contract. The plan-review codex pass is the right place to catch this; it caught chunk-11c's drift on round 0.

+ **Conformance assertions need three-legged traceability: group identity + MUST citation + group-relevance.** When adding a scenario to [`conformance/scenarios/`](conformance/scenarios/), three checks are gated by [`conformance/src/conformance/tools/check_citations.py`](conformance/src/conformance/tools/check_citations.py) (run in CI after pytest): (1) the file declares a module-level `CONFORMANCE_GROUP = "<group name>"` matching one of the chapter-9 §5 index groups; (2) the test docstring's first line cites `spec/v0/<chapter>.md §<sec>` whose section text (or any ancestor section) contains a `\bMUST\b` token; (3) the citation lies inside the declared group's chapter-9 §5 entry (or a descendant section). The three legs together prevent group bleed (groups that share a cited section, e.g. several §7 producers) AND identity-only misclassification (a file declaring any group regardless of what its tests cite). Phase-11 round-2/3/4 codex-review iterations all turned on these legs missing — both single-leg and dual-leg variants permitted regressions. If the assertion describes behavior the reference impl exhibits but no spec MUST mandates, the choice is to amend the spec (so the assertion is grounded) or drop the assertion; codifying impl observations as conformance assertions silently creates a second source of truth and locks third-party implementations into reference-impl quirks.

+ **Subprocess adapter lifecycle has three load-bearing traps.** The chunk-11 reference adapter ([`conformance/src/conformance/adapters/reference/adapter.py`](conformance/src/conformance/adapters/reference/adapter.py)) hit each in succession during impl review; check each before adding another subprocess-spawning adapter or test fixture. (1) **`secrets.token_urlsafe()` is NOT argparse-safe** — its base64url alphabet includes `-`, so a token starting with `-` is interpreted as a flag and the child subprocess exits with code 2 before announcing its port. Use `secrets.token_hex()` (only `[0-9a-f]`) for any secret passed through argparse-driven CLIs. (2) **`proc.stdout.readline()` blocks indefinitely** until a newline OR EOF — a hung child that never prints anything wedges the test process for the lifetime of the harness. The right pattern is a daemon thread + `queue.Queue` so `q.get(timeout=...)` makes the deadline real; see [`adapter.py`'s `_read_port_announcement`](conformance/src/conformance/adapters/reference/adapter.py) for the canonical shape. (3) **Partial startup leaks**: an adapter's `start()` typically does `Popen` + read-port-announcement + start-stderr-drain (+ optional outer-wrapper proxy startup); any post-Popen step that raises must run cleanup before re-raising, otherwise the subprocess (and any wrapper layers already started) leak because the fixture never reached `yield` and the harness never calls `stop()`. The fix is `try: …; except BaseException: self.stop(); raise` — and for layered adapters (e.g. `MisbehavingAdapter` wrapping `ReferenceAdapter`), the wrapper must clean up its OWN partial state (proxy thread + sockets) before tearing down the upstream.

+ **Multi-process e2e tests must not pile log output into undrained `subprocess.PIPE` handles — the 64 KiB pipe buffer fills up and wedges the writer mid-handler.** The reference logger writes to `sys.stdout` ([`reference/services/_common/src/eden_service_common/logging.py`](reference/services/_common/src/eden_service_common/logging.py)). With 5 subprocesses each logging a JSON line per HTTP request, the orchestrator e2e tests cross 64 KiB of stdout in ~10 s; if the test harness is using `subprocess.PIPE` and only reading the first line for the port announcement, every later write blocks. In an `async def` server that does logging inside the request handler, that blocking write happens on the event loop thread — the entire server stops responding to other requests, and from the orchestrator's view it looks like a `httpx.ReadTimeout` on `list_trials`. The fix is to redirect each subprocess's `stdout`+`stderr` to a real per-process file (Popen `stdout=open(path, "wb"), stderr=subprocess.STDOUT`) and read it from disk for the port announcement and for failure dumps; that removes the back-pressure entirely and preserves the log for diagnostics. The orchestrator e2e tests at [`reference/services/orchestrator/tests/test_e2e.py`](reference/services/orchestrator/tests/test_e2e.py) and [`reference/services/orchestrator/tests/test_subprocess_e2e.py`](reference/services/orchestrator/tests/test_subprocess_e2e.py) carry the canonical `_spawn` / `_read_port_announcement` / `_dump_logs` shape (eden#39). Rule of thumb when planning: any test that spawns >1 subprocess for >5 s and uses `subprocess.PIPE` must have an explicit drainer (file redirect or background thread); the symptom of failing this rule is intermittent `ReadTimeout` rather than an obvious crash, so it tends to look like a server-side bug.

+ **Adding a new uv-workspace member requires updating the reference Dockerfile.** uv's `--frozen --all-packages` insists every declared workspace member be present in the build context. The reference Dockerfile at [`reference/compose/Dockerfile`](reference/compose/Dockerfile) only copies `reference/packages` and `reference/services`, so a new top-level member (Phase 11 added `conformance/`) breaks every compose-* CI job with `Failed to determine installation plan` until a corresponding `COPY <member-dir> <member-dir>` is added. If the member's deps shouldn't bleed into the runtime image, options are: (a) accept the bloat (Phase 11's choice — pytest/aiohttp/etc. are ~30 MB and never imported by services); (b) restructure the member's deps under `[dependency-groups] dev` and gate the conformance CI job's sync with `--all-groups` or per-member-group flags (Phase 11 tried this and reverted: workspace-member dev groups don't activate from `uv sync` at the root by default, so the conformance job lost its test deps); (c) drop `--all-packages` and enumerate the runtime services explicitly with `--package …` flags. Rule of thumb: when adding a workspace member that ships only test-time tools, audit the Dockerfile FIRST and decide which of (a)/(b)/(c) you want before opening the PR.
