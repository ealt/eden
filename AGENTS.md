# AGENTS.md

This file provides guidance to AI agents working with this repository.

## What this repo is

EDEN is a **protocol** for directed evolution orchestration, not a single product. The repo contains three distinct kinds of artifact, each with different change discipline:

| Layer | Path | Authoritative? | Change discipline |
|---|---|---|---|
| Protocol specification | [`spec/`](spec/) | Yes — the source of truth | RFC-style: versioned, normative (MUST/SHOULD/MAY), carefully reviewed |
| Reference implementation | [`reference/`](reference/) | No — *one* valid impl | Normal code-review discipline |
| Conformance suite | [`conformance/`](conformance/) | Normative for tests | Black-box; must be implementation-agnostic |

Non-normative human docs live in [`docs/`](docs/).

## Recording chunk completions

Every shipped chunk MUST land an entry in `CHANGELOG.md` and a one-line status flip in `docs/roadmap.md`. Do NOT edit AGENTS.md to record chunk completions.

For chunks that have a chunk-plan in `docs/plans/`, the final-wave "docs PR" MUST:

1. Commit the impl-stage codex-review iteration record under `docs/plans/review/<chunk>/impl/<timestamp>/` (the plan-stage record is committed at plan-merge; this step covers the impl-stage record produced during chunk execution). The top-level `.gitignore` drops regenerable transcripts (`*.jsonl` / `*.stderr` / `*.stdout` / `prompt.txt`); commit the remaining durable artifacts the run produced (typically `*.md` and `*.patch`, plus any session-state files the skill emits).
2. Append the chunk-completion entry to `CHANGELOG.md [Unreleased]` describing the shipped work. On merge, that entry flips to a dated chunk heading (`## Phase <id> — <title> — <YYYY-MM-DD>`) at the top of the file.
3. Flip the chunk's `docs/roadmap.md` status pointer to "shipped": `- [<id>](plans/eden-phase-<id>-<slug>.md) — <title> — **shipped <YYYY-MM-DD>** (see [CHANGELOG](../CHANGELOG.md))`.

For planless chunks (hotfixes, mechanical renames, single-PR doc fix-ups) the same shape applies, except the roadmap one-liner points at the merged PR instead of a plan: `- [<id>](https://github.com/<org>/<repo>/pull/<N>) — <title> — **shipped <YYYY-MM-DD>** (see [CHANGELOG](../CHANGELOG.md))`. Use the planless shape when no plan was authored — never invent a plan path after the fact. Planless chunks that go through codex-review still commit the review record under `docs/plans/review/<slug>/impl/<timestamp>/`.

## Contribution conventions

### Spec edits

+ Every cross-component contract (wire format, state machine, invariant) is defined in `spec/` first, then implemented in `reference/`. When they disagree, the spec wins and the impl gets a bug.
+ Spec chapters use RFC 2119 language: MUST, SHOULD, MAY.
+ Spec versions are frozen once stable (`spec/v0/` is a single lineage; breaking changes go to `spec/v1/`).
+ Schema changes must update the spec prose, the JSON Schema file, and (from Phase 3 onward) the Pydantic bindings in lockstep. CI will enforce schema ↔ model parity once both exist.

### Reference implementation

+ Normal code-review discipline.
+ Normative behavior is tested against the conformance suite (once it exists in Phase 11).
+ The reference impl is explicitly not a monopoly on correctness. A third-party implementation that passes conformance is equally valid.

### Adding a new service or package with its own `tests/` directory

Pytest's default `prepend` import mode names test modules by file basename, so two `tests/test_app.py` files in different packages collide on collection (`import file mismatch`). The repo's tests-dir layout is mixed — some packages (e.g. [`eden-contracts/tests/`](reference/packages/eden-contracts/tests/)) have `__init__.py` and use `from .conftest import …`; most service test dirs (e.g. [`web-ui/tests/`](reference/services/web-ui/tests/)) have no `__init__.py` and use `from conftest import …`.

When adding a new tests directory:

+ Use a basename that is unique across the whole `testpaths` set in [`pyproject.toml`](pyproject.toml). Grep `find reference -name 'test_*.py' -exec basename {} \;` to verify before merging.
+ Do **not** add `__init__.py` to a service tests dir while another tests dir already has one — both directories become the `tests` package and the second conftest collides on `tests.conftest` import/registration. Under default `prepend` mode this surfaces as `ImportPathMismatchError`; under `importlib` mode it surfaces as `Plugin already registered under a different name`.
+ Switching the whole repo to `--import-mode=importlib` looks tempting but is **not** a clean fix today: the existing eden-contracts tests use relative `.conftest` imports that depend on the `__init__.py` package layout, and importlib mode keeps that collision while removing the sys.path hack the rest of the suite relies on.

If the constraint becomes load-bearing later (Phase 10 will add several more service test dirs), the cleanest fix is a three-step lockstep: rename each tests dir to a unique package name (e.g. `web_ui_tests/` instead of `tests/`), switch every bare `from conftest import …` to a package-relative `from .conftest import …`, and add `--import-mode=importlib` to pytest's `addopts`. A mechanical basename-uniqueness check in CI is also a sensible follow-up — AGENTS-only guidance is weaker than a guardrail.

### Adding or extending a JSON Schema + Pydantic binding

The `schema-parity` CI job is only as strong as what both sides of the test actually enforce. Several Pydantic and `jsonschema` defaults let drift through silently. When adding a new schema — or a new field type to an existing one — evaluate each of the following; reusable implementations live in [`reference/packages/eden-contracts/src/eden_contracts/_common.py`](reference/packages/eden-contracts/src/eden_contracts/_common.py).

+ **Strict numeric parsing.** Top-level models set `ConfigDict(strict=True, extra="allow")`. Non-strict mode coerces `True`/`"2"` into int, but the schemas treat `type: integer` / `type: number` as strict JSON types. New models must keep `strict=True`.
+ **Format assertions on both sides.** `format` keywords (`uri`, `date-time`, …) are advisory by default in both the `jsonschema` library and Pydantic. The schema-side validator wires a custom `FormatChecker` in [`tests/conftest.py`](reference/packages/eden-contracts/tests/conftest.py), and the model side uses the reusable types in `_common.py`. A new `format` keyword in any schema requires handlers on *both* sides; the `test_format_coverage` test fails loudly if the schema-side handler is missing.
+ **Real date-time / URI validation, not just regex or `urlparse`.** The regex on `DateTimeStr` accepts impossible values like `2026-99-99T…Z`; an `AfterValidator` runs `datetime.fromisoformat` to reject those. `UriStr` uses `rfc3986-validator`, not `urllib.parse.urlparse`, which accepts malformed schemeful URIs (e.g., spaces in the host).
+ **Null vs absent.** JSON Schema's `type: X` rejects explicit `null`, but Pydantic's `X | None = None` accepts it. Wrap every optional typed field with `NotNone` from `_common.py` so absent is accepted and explicit null is rejected.
+ **Round-trip emission.** `model.model_dump(mode="json", exclude_none=True)` must re-validate against the schema. Add any new model to `tests/test_roundtrip.py` so this is checked.
+ **Corpus coverage.** Parity is asserted over the fixture corpus in `tests/cases.py`. A new field type deserves at least one accept fixture and one reject fixture per constraint the schema imposes (required, pattern, enum, min/max, format, cross-field).

### Adding a new CI job

GitHub Actions' `ubuntu-latest` is now Ubuntu 24.04 with a system Python protected by [PEP 668](https://peps.python.org/pep-0668/) — bare `pip install` fails with `error: externally-managed-environment` instead of installing. Bash scripts that run on both CI (bash 5+) and contributor laptops (macOS ships bash 3.2 by default) hit the parallel constraint that bash-4-only builtins like `mapfile` / `readarray` silently work in CI but fail locally with `command not found`.

When adding a new CI job that needs Python or runs bash scripts:

+ **Python-using jobs must declare an interpreter via `actions/setup-python`** before any `pip install`. setup-python provisions an isolated interpreter that pip can write to freely, and adds it to PATH so a downstream `python3 …` resolves to the same interpreter the workflow installed into. Use `python3 -m pip install …`, not bare `pip install …`, to make the install/import binding explicit. The chunk-10e [`compose-e2e`](.github/workflows/ci.yml) job is the canonical small example: setup-python step, then `python3 -m pip install 'httpx>=0.27,<1'`, then a bash wrapper that calls `python3` on the driver. Jobs that don't need a custom Python (the existing `compose-smoke` etc. that only call `docker compose` / `jq` / `curl`) skip setup-python entirely.

+ **Bash scripts under [`reference/compose/healthcheck/`](reference/compose/healthcheck/) and [`reference/scripts/`](reference/scripts/) must work under bash 3.2.** macOS' default `/bin/bash` is 3.2 and contributors run these scripts locally to repro CI failures. The two non-obvious incompatibilities to watch for: (1) `mapfile -t arr < <(cmd)` does not exist — use `arr=(); while IFS= read -r line; do [[ -n "$line" ]] && arr+=("$line"); done < <(cmd)` instead. (2) Associative arrays (`declare -A`) do not exist either; use a parallel-arrays pattern or a different data structure. `set -euo pipefail` and process substitution `< <(…)` both work fine in bash 3.2; only the named bash-4 builtins are off-limits.

## Plan-writing pitfalls

These are the recurring traps that have shown up across chunks 9c / 9d / 10d / 10e / 10d-followup-A / 10d-followup-B during `/codex-review`. Check each before submitting a plan that touches the orchestrator loop, the wire surface, or any role's submission flow.

+ **Verify wire-touching shapes against the actual dataclasses, not the spec prose alone.** The submission types in [`reference/packages/eden-storage/src/eden_storage/submissions.py`](reference/packages/eden-storage/src/eden_storage/submissions.py) carry only `status` + role-specific id/payload fields — there is no free-form `description`. If your plan threads a `description` through the wire, you'll have to either drop it (route to host log only) or amend the spec + bindings + contracts together. Same discipline applies to event payloads: check [`reference/packages/eden-contracts/src/eden_contracts/event.py`](reference/packages/eden-contracts/src/eden_contracts/event.py) before assuming a field exists.
+ **Any code path that produces a variant must run `Store.create_variant(status="starting")` before any observable repo write.** This is [`spec/v0/03-roles.md`](spec/v0/03-roles.md) §3.2 step 1, and the chunk-9c web-ui executor at [`reference/services/web-ui/src/eden_web_ui/routes/executor.py`](reference/services/web-ui/src/eden_web_ui/routes/executor.py) is the canonical reference flow. Both the chunk-9c and 10d plans skipped the starting-variant step in their first revision until `/codex-review` caught it. The recovery path (chunk-1 expired-claim sweeper composite-commits the orphan variant to `error`) only works if the variant actually exists in `starting` state when the host crashes.

+ **For compose-bound integration tests, audit which always-on services would race with the test.** The reference-host services (`ideator-host`, `executor-host`, `evaluator-host`) tight-loop-poll their respective task kinds on startup and claim *every* eligible task without filtering. So a test that wants to drive a task through the web UI cannot simply `compose up && stop <competing-host>` — by the time the stop returns, the competing host has already drained the seed. The chunk-10e [`e2e.sh`](reference/compose/healthcheck/e2e.sh) shows the right pattern: stage 1 brings up everything *except* the competing host, the test claims its targets via the UI, then stage 2 brings the competing host online so the rest of the experiment proceeds. Rule of thumb when planning: enumerate the polling services and ask "could the user's intent be drained before my test runs?" — if yes, defer the competing service to stage 2.

+ **DooD wraps need explicit `name:` on every forwarded compose volume, and the spawned-child user identity must match the worker host's.** Two traps the 10d-followup-A implementation hit in succession during local-smoke debugging: (1) compose auto-prefixes volume names with the project (`eden-reference_eden-bare-repo`); a wrap that does `--mount source=eden-bare-repo` against the host docker daemon resolves to a *fresh empty volume* unless the compose volume has explicit `name:` matching the wrap's literal — see the `name:` discipline on [`compose.yaml`](reference/compose/compose.yaml) and [`compose.subprocess.yaml`](reference/compose/compose.subprocess.yaml). (2) The default container user differs between `eden-reference:dev` (eden:1000) and a freshly-pulled base image (root); spawned children that run as root against worktree files written by uid 1000 trip git's `dubious ownership` check, AND any commits they produce are uid-0-owned breaking subsequent integrator reads. The fix is to mirror the `eden:1000` user in `eden-runtime:dev` (the wrap deliberately doesn't pass `--user` so experiment-image USER overrides are honored). Rule of thumb when planning DooD: enumerate every forwarded volume and confirm `name:` is set; pin the spawned-child user to match the worker host's, not the spawned image's default.

+ **DooD sibling containers survive SIGKILL of the local `docker run` client; the cidfile-driven `post_kill_callback` is load-bearing.** Killing the local `docker run` process via `os.killpg(SIGKILL)` does NOT propagate to the spawned container — the daemon parent keeps it running until the workload exits or it's killed externally. Plans that wrap `subprocess.Popen` with `docker run` must register a `post_kill_callback` (stored on the `Subprocess` instance, fires on the SIGKILL escalation branch) that reads the cidfile and runs `docker kill && docker rm -f`. The cleanup callback that unlinks the cidfile must run on **every** terminal exit branch (graceful, fast-path, and SIGKILL) — not just SIGKILL — because `--cidfile` errors if the path exists, so cidfile reuse without cleanup is a hard failure. See [`reference/services/_common/src/eden_service_common/container_exec.py`](reference/services/_common/src/eden_service_common/container_exec.py) and the §D.3 design notes in [`docs/archive/eden-phase-10d-followup-a-container-isolation.md`](docs/archive/eden-phase-10d-followup-a-container-isolation.md).

+ **Three load-bearing wiring traps in any DooD sibling-container wrap.** The container-isolation chunk's CI burned through six rounds catching these in the wild; check each before submitting a plan that mounts the host docker socket into a worker. (1) **Privilege opt-in via separate overlay.** Putting `/var/run/docker.sock` mounts unconditionally in `compose.subprocess.yaml` would have granted DooD privilege to host-mode users by accident. The fix was a third overlay file ([`compose.docker-exec.yaml`](reference/compose/compose.docker-exec.yaml)) layered ON TOP of the subprocess overlay — that's the only file that mounts the socket and applies `group_add`. Both smoke scripts assert this directly via `docker inspect`. Rule of thumb: any deployment mode that needs new privilege (capability, host-network, socket mount, etc.) goes in its own compose overlay so the default keeps the privilege off. (2) **Paths in `docker run` flags resolve on the daemon side, NOT in the calling container's mount namespace.** `--cidfile <path>`, `--mount source=<path>`, `-v <host>:<container>` — all of these are interpreted on the host docker daemon, against the host filesystem. So a worker container that bind-mounts `${EDEN_CIDFILES_DIR_HOST}` at `/var/lib/eden/cidfiles` and passes `--cidfile /var/lib/eden/cidfiles/X.cid` will fail "permission denied" — that path doesn't exist on the host. The fix is to bind the cidfile dir at the SAME path inside the worker as on the host (`${EDEN_CIDFILES_DIR_HOST}:${EDEN_CIDFILES_DIR_HOST}`) so worker and daemon agree. Same posture for any path passed through `--mount source=`. (3) **`docker run -i` exits early on closed stdin.** Compose containers run with stdin closed by default; `docker run -i` propagates that closed-stdin to its parent's wait loop and exits *before* the workload has done useful work. Make `-i` conditional on whether the caller will pipe stdin to the spawn: ideator subprocesses (long-running JSON-line protocol) need it; per-task executor/evaluator spawns must NOT include it. See [`reference/services/_common/src/eden_service_common/container_exec.py`](reference/services/_common/src/eden_service_common/container_exec.py)'s `attach_stdin` parameter. (Bonus trap: `subprocess.Popen(shell=True)` runs `/bin/sh -c '<wrap>'`; on Linux that's typically dash, which exits on SIGTERM without forwarding to the docker run child. Prefix the wrap with `exec` so the shell `exec`s docker run in place — `popen.pid` then IS docker run.)

+ **Any wire/network write that can leave server state unknown to the client needs a 3-outcome read-back ladder: confirmed-success / confirmed-divergence / indeterminate.** A `git push` that returns a transport error after the daemon may have already accepted the ref is the canonical example — there's no "did it land?" answer in the exit code alone. The 10d-followup-B integrator handles this by following the failed push with `ls_remote(variant.branch)` and routing on the three outcomes: remote==our-SHA → success (proceed to step 3), remote==different-SHA → divergence (compensating-delete local, raise `RefRefused`), remote-absent-or-unreadable → indeterminate (compensating-delete local, raise `IndeterminatePush`). Same shape as `StoreClient.integrate_variant`'s read-back from Phase 8a (chapter 07 §5). Rule of thumb when planning: any write to a remote system whose response can be lost in transit (HTTP `integrate_variant`, `git push`, future Postgres-over-network commits) needs a read-back step; do NOT collapse "transport error" into "definitely failed" — that would be a silent atomicity violation when the remote in fact accepted.

+ **Recovery code must use spec-authoritative content paths, NOT ref-name parsing.** The 10d-followup-B remote-orphan reconciler needs to recover a `variant_id` from a remote-only ref (one that exists on Forgejo but has no corresponding `Variant` record in the store). The first-cut plan parsed `refs/heads/variant/<id>-<slug>` with a regex; round-2 codex review caught that [`spec/v0/02-data-model.md`](spec/v0/02-data-model.md) §1.3 keeps `variant_id` opaque (any string ≤ 64 chars, the integrator's branch-name shape is non-normative). The fix walks the ref's commit tree and reads `.eden/variants/<id>/evaluation.json` — the spec-authoritative manifest path defined in chapter 6 §3.2. Rule of thumb when planning: anywhere recovery / reconciliation / GC code needs to extract a spec-defined identifier from a git artifact, the source of truth is the on-tree manifest the spec mandates, not the implementation's chosen branch-naming convention. Branch names are a reference-impl convenience.

+ **Long-lived bare/non-bare clones need read-before-write/display fetches against the remote.** Two related round-0/round-1 findings on 10d-followup-B: (1) The admin work-refs page rendered from the local clone; if the integrator pushed a ref since the page-load, the admin's view was stale and the CAS-guarded delete-button could fail confusingly. The fix is `fetch_all_heads` immediately before reading refs to display, and `delete_remote_ref` BEFORE `delete_ref` so the remote-of-record is the source of truth. (2) The integrator's local clone may not have the work-branch the executor pushed after orchestrator startup; an integrator that only fetches at startup misses any post-startup push. The fix is a per-integrate-call `fetch_ref(variant.branch)` immediately before chapter-6 §2 reachability check. Rule of thumb when planning: any service that holds a clone of a remote-of-record must answer "when does this clone re-sync?" — at startup is rarely sufficient; the answer is usually "immediately before each operation that reads or writes a ref the remote could have changed."

+ **Narrow exception handling on store reads when classifying authoritative-orphan vs indeterminate.** Impl round-0 on 10d-followup-B: `reconcile_remote_orphans` initially treated any `Exception` from `Store.read_variant(variant_id)` as "variant doesn't exist → orphan → delete the remote ref". A transient transport blip (StoreClient HTTP timeout, postgres connection drop) would have deleted legitimate in-flight integrator work. The fix narrows to `except NotFound: variant = None  # authoritative orphan` and routes any other exception to `continue  # indeterminate, leave the remote alone`. Rule of thumb when planning: anywhere a "missing → take destructive recovery action" branch is being written, the read that classifies "missing" must distinguish definitive `NotFound` from transport-indeterminate failure; collapsing the two is a silent data-loss vector.

+ **Role-specific git/wire transport-failure mapping must match chapter 3 contracts.** Plan round-1 on 10d-followup-B: the first cut mapped any transport failure during git operations to a single failure mode. [`spec/v0/03-roles.md`](spec/v0/03-roles.md) actually differentiates by role — executor transport failure on `push_ref` submits `status="error"` (chapter 3 §3.3); evaluator transport failure on `fetch_ref(variant.branch)` (preventing variant inspection) submits `status="evaluation_error"` (chapter 3 §4.4); integrator transport failure runs the chapter 6 §3.4 compensating-delete ladder (the integrator never "submits" — it composite-commits or rolls back). Rule of thumb when planning: when a wire/transport failure interrupts a role's flow, look up the failure-status vocabulary for THAT role in chapter 3 / chapter 6 — there is no shared "transport-failed" status across roles.

+ **Tests for callbacks/lifecycle/atomicity must drive the actual code path through the public entry point, not just call the helper directly.** The chunk's first SIGKILL test called `kill_via_cidfile()` directly — it passed even though the production path through `Subprocess.terminate()`'s SIGKILL escalation branch was broken (the branch was being skipped because `popen.wait` returned success in 0.5s). Codex round-0 caught this; the fix was a rewrite that uses `eden_service_common.spawn(...)` + `sub.terminate(shutdown_deadline=0.5)` and asserts the spawned sibling container is gone after the public terminate call. See [`test_terminate_sigkill_path_invokes_post_kill_callback`](reference/services/_common/tests/test_container_exec_integration.py) for the canonical shape. Rule of thumb: when a callback is supposed to fire on a specific branch of a public method, the test must (a) construct the same setup the production caller would, (b) invoke the public method, and (c) assert the side effect. Helper-only tests can pass while the integrated path silently skips the branch.

+ **Spec inter-chapter restatement is a conflict surface.** When the same MUST is restated across multiple chapters with the canonical statement in one and a role-side restatement elsewhere, the restatements drift over time and codex-review on a conformance plan is the place that drift surfaces. Chunk-11c hit this on evaluator-resubmit equivalence: [`spec/v0/04-task-protocol.md`](spec/v0/04-task-protocol.md) §4.2 (canonical task-store rule) listed `variant_id`+`status`+`metrics` as the equivalence formula; [`spec/v0/03-roles.md`](spec/v0/03-roles.md) §4.4 (role-side restatement) listed `variant_id`+`status`+`metrics`+`artifacts_uri`. The reference impl followed §4.2 (per its own docstring at [`reference/packages/eden-storage/src/eden_storage/submissions.py`](reference/packages/eden-storage/src/eden_storage/submissions.py) `submissions_equivalent`); §4.4 was the drift. The fix shape: the role-side restatement defers to the canonical statement (chunk-11c amended §4.4 to drop `artifacts_uri` and reference §4.2). Rule of thumb when planning: any chunk that touches a role-contract MUST should grep both chapter 03 AND chapters 04/05/07/08 for the same field/operation/invariant; if both chapters restate it with different lists, surface the conflict in the plan and resolve by deferring the role-side to the canonical chapter — don't just author tests against whichever side the impl happens to follow, because that codifies one side of the drift as conformance contract. The plan-review codex pass is the right place to catch this; it caught chunk-11c's drift on round 0.

+ **Conformance-plan MUSTs must be filtered through the IUT contract before drafting scenarios.** Chapter 9 §6 says "the chapter-7 HTTP binding is the only IUT contract a conformance harness can rely on" — the suite cannot inspect git refs, filesystem state, internal store transactions, or any other artifact the IUT doesn't expose through chapter 7. So when a conformance plan names a normative MUST as in-scope, ask first: is this MUST observable through the IUT contract? If the answer is no, the MUST cannot be asserted at this level — it has to be deferred to a future binding chapter (e.g. a hypothetical "conformance + git access" contract). Chunk-11d hit this when chapter 9 §4 named four chapter-06 MUSTs as v1+roles+integrator scope ("squash shape, eval-manifest shape, `work/*` access discipline, atomicity ladder under transport-indeterminate failures") — three of those (squash, manifest, work/*) are pure git-side artifacts and the chapter-7 binding doesn't expose them. The fix amended chapter 9 §4 to scope the level to the wire-observable projection of chapter 06 (§2 integration preconditions, §3.4 atomicity-of-(field, event), §5.3 no-overwrite) and explicitly defer the rest. Rule of thumb when planning: walk each candidate spec MUST through `chapter 9 §6 IUT contract → wire endpoint(s) involved → what artifacts can the harness read?` and only keep MUSTs whose required observables are reachable through that pipeline. Surfacing the off-wire MUSTs in the plan as "explicitly out-of-scope, deferred to future binding" is the right shape — silently dropping them is the failure mode that lets the spec scope-claim drift away from what the suite actually delivers. This is a cousin of the spec-inter-chapter-restatement pitfall above (chapter 9 §4 was over-promising what §6's IUT-contract restriction allows), and the same plan-review codex pass tends to catch both shapes.

+ **Conformance assertions need three-legged traceability: group identity + MUST citation + group-relevance.** When adding a scenario to [`conformance/scenarios/`](conformance/scenarios/), three checks are gated by [`conformance/src/conformance/tools/check_citations.py`](conformance/src/conformance/tools/check_citations.py) (run in CI after pytest): (1) the file declares a module-level `CONFORMANCE_GROUP = "<group name>"` matching one of the chapter-9 §5 index groups; (2) the test docstring's first line cites `spec/v0/<chapter>.md §<sec>` whose section text (or any ancestor section) contains a `\bMUST\b` token; (3) the citation lies inside the declared group's chapter-9 §5 entry (or a descendant section). The three legs together prevent group bleed (groups that share a cited section, e.g. several §7 producers) AND identity-only misclassification (a file declaring any group regardless of what its tests cite). Phase-11 round-2/3/4 codex-review iterations all turned on these legs missing — both single-leg and dual-leg variants permitted regressions. If the assertion describes behavior the reference impl exhibits but no spec MUST mandates, the choice is to amend the spec (so the assertion is grounded) or drop the assertion; codifying impl observations as conformance assertions silently creates a second source of truth and locks third-party implementations into reference-impl quirks.

+ **Subprocess adapter lifecycle has three load-bearing traps.** The chunk-11 reference adapter ([`conformance/src/conformance/adapters/reference/adapter.py`](conformance/src/conformance/adapters/reference/adapter.py)) hit each in succession during impl review; check each before adding another subprocess-spawning adapter or test fixture. (1) **`secrets.token_urlsafe()` is NOT argparse-safe** — its base64url alphabet includes `-`, so a token starting with `-` is interpreted as a flag and the child subprocess exits with code 2 before announcing its port. Use `secrets.token_hex()` (only `[0-9a-f]`) for any secret passed through argparse-driven CLIs. (2) **`proc.stdout.readline()` blocks indefinitely** until a newline OR EOF — a hung child that never prints anything wedges the test process for the lifetime of the harness. The right pattern is a daemon thread + `queue.Queue` so `q.get(timeout=...)` makes the deadline real; see [`adapter.py`'s `_read_port_announcement`](conformance/src/conformance/adapters/reference/adapter.py) for the canonical shape. (3) **Partial startup leaks**: an adapter's `start()` typically does `Popen` + read-port-announcement + start-stderr-drain (+ optional outer-wrapper proxy startup); any post-Popen step that raises must run cleanup before re-raising, otherwise the subprocess (and any wrapper layers already started) leak because the fixture never reached `yield` and the harness never calls `stop()`. The fix is `try: …; except BaseException: self.stop(); raise` — and for layered adapters (e.g. `MisbehavingAdapter` wrapping `ReferenceAdapter`), the wrapper must clean up its OWN partial state (proxy thread + sockets) before tearing down the upstream.

+ **Multi-process e2e tests must not pile log output into undrained `subprocess.PIPE` handles — the 64 KiB pipe buffer fills up and wedges the writer mid-handler.** The reference logger writes to `sys.stdout` ([`reference/services/_common/src/eden_service_common/logging.py`](reference/services/_common/src/eden_service_common/logging.py)). With 5 subprocesses each logging a JSON line per HTTP request, the orchestrator e2e tests cross 64 KiB of stdout in ~10 s; if the test harness is using `subprocess.PIPE` and only reading the first line for the port announcement, every later write blocks. In an `async def` server that does logging inside the request handler, that blocking write happens on the event loop thread — the entire server stops responding to other requests, and from the orchestrator's view it looks like a `httpx.ReadTimeout` on `list_variants`. The fix is to redirect each subprocess's `stdout`+`stderr` to a real per-process file (Popen `stdout=open(path, "wb"), stderr=subprocess.STDOUT`) and read it from disk for the port announcement and for failure dumps; that removes the back-pressure entirely and preserves the log for diagnostics. The orchestrator e2e tests at [`reference/services/orchestrator/tests/test_e2e.py`](reference/services/orchestrator/tests/test_e2e.py) and [`reference/services/orchestrator/tests/test_subprocess_e2e.py`](reference/services/orchestrator/tests/test_subprocess_e2e.py) carry the canonical `_spawn` / `_read_port_announcement` / `_dump_logs` shape (eden#39). Rule of thumb when planning: any test that spawns >1 subprocess for >5 s and uses `subprocess.PIPE` must have an explicit drainer (file redirect or background thread); the symptom of failing this rule is intermittent `ReadTimeout` rather than an obvious crash, so it tends to look like a server-side bug.

+ **Adding a new uv-workspace member requires updating the reference Dockerfile.** uv's `--frozen --all-packages` insists every declared workspace member be present in the build context. The reference Dockerfile at [`reference/compose/Dockerfile`](reference/compose/Dockerfile) only copies `reference/packages` and `reference/services`, so a new top-level member (Phase 11 added `conformance/`) breaks every compose-* CI job with `Failed to determine installation plan` until a corresponding `COPY <member-dir> <member-dir>` is added. If the member's deps shouldn't bleed into the runtime image, options are: (a) accept the bloat (Phase 11's choice — pytest/aiohttp/etc. are ~30 MB and never imported by services); (b) restructure the member's deps under `[dependency-groups] dev` and gate the conformance CI job's sync with `--all-groups` or per-member-group flags (Phase 11 tried this and reverted: workspace-member dev groups don't activate from `uv sync` at the root by default, so the conformance job lost its test deps); (c) drop `--all-packages` and enumerate the runtime services explicitly with `--package …` flags. Rule of thumb: when adding a workspace member that ships only test-time tools, audit the Dockerfile FIRST and decide which of (a)/(b)/(c) you want before opening the PR.

+ **The "Commands" section above is the literal pre-push validation gate; narrowed subsets are not.** `uv run pytest -q reference/services/<one>/tests/` is NOT a substitute for the full `uv run pytest -q`, and `pytest -q` itself is NOT a substitute for the bash smoke scripts (`smoke.sh` / `smoke-subprocess.sh` / `e2e.sh`) which exercise the full Compose pipeline that pytest can't reach. Multiple regressions have shipped to PR-CI when the local validation only ran a scoped pytest invocation: an outdated env-var name in `.env.example` that survived a rename pass (caught only by the smoke scripts that load `.env`), an audit PR's smoke regression from a leaked `--max-quiescent-iterations 36000` workaround, and SIM105/markdownlint violations the canonical commands would have caught. Rule of thumb: before any `git push`, run the literal commands listed in the "Commands" table above (or at least the lint/typecheck/pytest/smoke quartet) — not a narrowed equivalent. The smokes are the most commonly skipped because they take minutes; they're also the ones that catch pipeline-shape regressions.

+ **Cherry-pick contamination from dev-only workarounds.** WIP commits often carry dev-only convenience changes ("bumped X for my manual testing", "patched Y to skip a check while iterating") that are NEVER intended to land. When such a commit is later cherry-picked or rebased into a real PR branch, those workarounds slip into the merged state silently. The audit PR's smoke regression came from cherry-picking a manual-UI WIP commit that had bumped `--max-quiescent-iterations` from 30 to 36000 (a 10-hour quiescence wait that broke the 180s smoke budget). Rule of thumb: WIP commits intended as session-local snapshots should be scrubbed before any cherry-pick / rebase / squash that would land them on a shared branch. Either (a) keep dev-only changes in the worktree's working tree only, never committed; (b) tag dev-only commits with a `[dev-only]` prefix so they're easy to spot when squashing; or (c) when cherry-picking from a WIP branch, diff against the destination first and surgically include only the substantive changes.

+ **CI failure diagnosis: local repro beats log-tail reading.** When a CI smoke / integration test fails, the failure-mode output you see in the GH Actions log is typically truncated (`docker compose logs --tail 30 orchestrator` is the default in `smoke.sh`) and shows only the LAST few seconds of the failing service's log. Reasoning from that snippet alone — without local reproduction — is unreliable: an audit PR's smoke regression got misdiagnosed as "orchestrator seed step is broken" because the visible polling pattern at the end of the log looked like nothing had been seeded; a local repro showed the seed worked fine and the actual issue was a 10-hour quiescence wait swallowing the smoke's 180s budget. Rule of thumb: for any failing compose smoke or e2e CI job, run `bash reference/compose/healthcheck/smoke.sh` (or the relevant script) locally and inspect the live state with `docker compose logs --tail 200 <service>` BEFORE proposing a fix. The CI log is a hint; the local stack is the ground truth.

+ **Compose smoke tests need explicit volume cleanup between runs.** `setup-experiment.sh` rotates the postgres password (and the `EDEN_SHARED_TOKEN` and other secrets) on every invocation; the generated `.env` carries the new password but the postgres data volume from a prior run still has the old password baked into `pg_authid`. The next `compose up` then fails task-store-server's healthcheck with a confusing `psycopg.OperationalError: ... password authentication failed for user "eden"`, reported as "dependency failed to start: container eden-task-store-server is unhealthy". The fix is volume cleanup before each smoke run: `docker rm -f $(docker ps -aq --filter 'name=eden-' 2>/dev/null) ; docker volume ls -q | grep eden | xargs -I{} docker volume rm {} 2>&1 | head`. CI gets this for free because each job starts on a fresh runner; the local-repro path doesn't, and the cryptic auth error sends diagnosis down a wrong path. Run the cleanup once at the start of any smoke-debugging session and after every smoke run.

+ **Substrate plans (Helm, k8s manifests, Compose overlays) must call out operator-required values explicitly and may need to bundle small impl changes when the substrate forces a CLI/API surface that doesn't yet exist.** Two related traps from chunk-13a's review rounds: (1) a chart that ships a fictional default for a value the operator must supply (e.g., `image.repository: eden-reference` when there is no published `eden-reference` image — Compose builds it locally from a Dockerfile target) makes `helm install` look runnable on a vanilla cluster but fails at pod-pull time with a confusing `ErrImagePull`. The fix is to leave the value's default empty and gate `values.schema.json` with `minLength: 1` so `helm install` fails loud at lint/template time with a clear "image.repository is required" message rather than at runtime. (2) the existing impl's CLI may not match the substrate's operational shape — chunk-13a hit this with `--max-quiescent-iterations`, where the Compose deployment's quiescence-exit pattern (orchestrator exits 0 when no work remains; `restart: "on-failure"` keeps it pinned to its last exit) doesn't translate to a long-running k8s workload. The right move is to bundle the small impl change (CLI accepts `0` as a "never exit" sentinel; `loop.py` skips the quiescence-exit branch under that value) INTO the substrate plan rather than fragmenting it into its own chunk, because the chart can't ship without it. Round-1 codex caught this on 13a; the scope expansion was the right shape. Rule of thumb when planning a substrate chunk: walk every container's CLI surface against the substrate's lifecycle assumptions (long-running vs short-lived; restart-policy semantics; readiness/liveness probes; graceful-shutdown signals) and surface any mismatch as either (a) operator-required values with no defaults + clear error messages, or (b) bundled impl changes scoped to the same chunk.

+ **Multi-tenant lifecycle in deployment substrates: per-experiment bootstrap belongs OUTSIDE the chart/template owner, not inside.** Chunk 13a's round-1 review caught this: the first cut had `templates/repo-init-job.yaml` as a chart-managed Job. That works for a single-experiment deployment but breaks the moment you want to run multiple experiments against the same release — the helm release "owns" the repo-init Job from experiment 1 forever, and re-running setup against the same release for experiment 2 either no-ops (Job already exists) or replaces the experiment-1 record. The right shape mirrors Compose's `setup-experiment.sh`: the bootstrap Job lives at `bootstrap/repo-init-job.yaml.tmpl` (outside `templates/`), is rendered by `setup-experiment-helm.sh` with the per-experiment values, and is `kubectl apply`-ed directly — so each experiment gets its own Job lifecycle, independent of the helm release lifecycle. Same posture is likely correct for any future per-experiment artifact (per-experiment ConfigMaps holding the experiment YAML, per-experiment ad-hoc seed Jobs, etc.) on substrates that have a separate notion of release-managed resources. Rule of thumb when planning a multi-tenant substrate chunk: enumerate every artifact whose lifecycle is per-experiment (NOT per-release), and route each through an out-of-tree template + setup script rather than `templates/` — the chart should describe the steady-state deployment, not the per-experiment bootstraps.

## Naming discipline

EDEN's vocabulary is verb-noun-coherent and role-symmetric. Before introducing or renaming any identifier — class, function, JSON enum value, CLI flag, env var, spec heading, doc cross-reference — read [`docs/glossary.md`](docs/glossary.md) and validate the proposed name against the canonical patterns there.

The patterns:

| Role (-or) | Verb | Task kind (gerund) | Submission class | Artifact |
|---|---|---|---|---|
| ideator    | ideate    | `ideation`   | `IdeaSubmission`       | `idea`       |
| executor   | execute   | `execution`  | `VariantSubmission`    | `variant`    |
| evaluator  | evaluate  | `evaluation` | `EvaluationSubmission` | `evaluation` |
| integrator | integrate | (n/a — runs in-orchestrator) | (n/a)            | (canonical lineage entry) |

Rules of thumb:

+ **Helper-function names use the artifact noun.** `submit_idea`, `submit_variant`, `submit_evaluation` — never `submit_ideate` / `submit_execute` / `submit_evaluate` (verb-on-verb). <!-- rename-discipline:cite -->
+ **Task kinds are gerund nouns.** Wire enum, JSON schema enum, `task.kind == "X"` literals, CLI flags, env vars all use `ideation` / `execution` / `evaluation`.
+ **Submission classes use the artifact noun**, not the verb. `IdeaSubmission` / `VariantSubmission` / `EvaluationSubmission`.
+ **Status enums are gerund-as-noun.** `evaluation_error`, not `eval_error`. <!-- rename-discipline:cite -->
+ **Don't introduce synonyms.** The integrator integrates; it does not "promote". `eval_error` was retired — don't reintroduce it. <!-- rename-discipline:cite -->
+ **The glossary wins over other docs.** If a spec chapter, README, or plan disagrees with the glossary, fix the chapter / README / plan, not the glossary.

The `rename-discipline` CI job (`scripts/check-rename-discipline.py`) catches the specific legacy patterns retired by past renames; it is a backstop, not a substitute for consulting the glossary before introducing a new identifier.
