I don’t see further feasibility blockers at the same level as the prior rounds. The next substantive feedback is at **Alternatives**.

**Findings**
1. The plan should more explicitly justify keeping termination as a 5th §6.2 decision type instead of modeling it as a separate experiment-lifecycle concern. In 12a-2, the four §6.2 decisions are homogeneous orchestrator actions: create ideation tasks, dispatch execution tasks, dispatch evaluation tasks, and integrate variants ([12a-2 lines 223-239](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:223>)). In 12a-3, “termination” is qualitatively different: it mutates experiment state, gates task creation, gates claim, is invokable both by policy and by operator, and yet still leaves integration running ([12a-3 lines 200-236](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:200>), [254-255](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:254>), [361-380](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:361>), [638](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:638>), [656](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:656>)). That can still work, but it is no longer “just another decision type”; it is a lifecycle gate over the rest of the system. I would want the plan to explicitly compare the two models and say why the §6.2 placement is still the better abstraction.

2. If you keep termination as an experiment-lifecycle concern, the generic `update_experiment_state` / `PATCH .../state` surface looks broader than the v0 contract needs. The plan already says the only allowed transition is `running -> terminated` ([lines 256-265](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:256>)), while the operator-facing action is specifically `POST .../terminate` ([lines 463-470](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:463>), [655-656](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:655>)). A generic state-mutation op expands the normative surface without adding real capability in v0, and it makes future unsupported transitions look half-specified. I would at least consider collapsing the public model to `terminate_experiment` / `POST .../terminate` only, with no general-purpose `PATCH .../state` until there is a second legal lifecycle transition.

**Open Questions**
- What is the concrete benefit of keeping termination inside the §6.2 decision enumeration, now that it has different behavior from the other four and survives by special-casing integration?
- Is there any v0 caller that truly needs “set experiment state” rather than “terminate experiment”? If not, why standardize the broader endpoint now?

**Overall Assessment**
This is now much closer to implementation-ready. The remaining substantive question is architectural: whether termination really belongs as a 5th orchestrator decision type, or whether the plan is already describing a separate experiment-lifecycle control plane and should expose it that way more directly.
