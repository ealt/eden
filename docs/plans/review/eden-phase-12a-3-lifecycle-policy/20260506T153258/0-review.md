**Findings**
1. The termination model currently strands already-successful variants and conflicts with the 12a-2 integration contract. In 12a-3, termination runs first and, on `Terminate`, skips all four operational decisions forever ([plan lines 188-219](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:188>), [326-353](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:326>)). But 12a-2’s orchestrator contract says every `status == "success"` variant without `variant_commit_sha` must eventually be integrated ([12a-2 lines 236-239](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:236>)). `terminated` in 12a-3 only discusses task creation / in-flight tasks ([plan lines 221-245](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:221>)), and integration is not a task lifecycle at all ([spec/v0/03-roles.md lines 19, 162-179](</Users/ericalt/Documents/eden/spec/v0/03-roles.md:19>)). As written, a variant that is already `success` when termination fires can be left permanently unintegrated.

2. The new `experiment.state` + `experiment.terminated` atomic commit has no normative home in the current event/storage chapters, and the plan does not schedule the needed spec edits. The plan relies on a composite commit for experiment-state transition plus event ([plan lines 206-209](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:206>), [342-344](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:342>), [456-459](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:456>), [483-485](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:483>)). But the current transactional invariant is explicitly scoped to `tasks`, `ideas`, and `variants`, not experiments ([spec/v0/05-event-protocol.md lines 31-37](</Users/ericalt/Documents/eden/spec/v0/05-event-protocol.md:31>), [45-54](</Users/ericalt/Documents/eden/spec/v0/05-event-protocol.md:45>); [spec/v0/08-storage.md lines 161-179](</Users/ericalt/Documents/eden/spec/v0/08-storage.md:161>)). 12a-3’s file list edits chapter 05 only to add event types and does not touch chapter 08 at all ([plan lines 570-579](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:570>)). Until those chapters are widened to make `Experiment` protocol-owned state with the same atomicity guarantees, the central lifecycle transition is underspecified.

3. The four-field removal is not mechanically aligned with the live implementation surface. The plan says to change `reference/packages/eden-contracts/src/eden_contracts/experiment.py` ([plan lines 258-260](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:258>), [595-596](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:595>)), but the live `ExperimentConfig` model is currently in [config.py](</Users/ericalt/Documents/eden/reference/packages/eden-contracts/src/eden_contracts/config.py:32>) and still directly defines `max_variants`, `max_wall_time`, `convergence_window`, and `target_condition` ([config.py lines 32-48](</Users/ericalt/Documents/eden/reference/packages/eden-contracts/src/eden_contracts/config.py:32>)). The main conformance fixture also still carries `max_variants` / `max_wall_time` ([minimal-experiment.yaml lines 1-12](</Users/ericalt/Documents/eden/conformance/src/conformance/fixtures/minimal-experiment.yaml:1>)). So the plan’s removal is directionally right, but the file-touch list is not executable as written.

4. The terminated-state task-creation rule is internally inconsistent. The lifecycle prose only names “new ideation tasks” as rejected after termination ([plan line 234](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:234>)), but the implementation/test sections expect all task-creation paths to fail after termination, including execution and evaluation ([plan lines 486-487](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:486>), [670-672](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:670>), [731-736](</Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:731>)). Since 12a-2 explicitly added admin-created evaluation tasks and is lifting execution-task authority here, chapter-02/chapter-04 text needs to say “no new tasks of any kind” if that is the intended contract.

**Open Questions**
- Should termination stop only new task creation, while still draining already-succeeded variants through integration?
- Are `experiment.terminated` / `experiment.policy_error` meant to be first-class registered protocol events, or implementation telemetry? The answer changes whether chapter 05/08 must broaden their entity model.
- Is the intent to expose only `POST /terminate`, or is a general `PATCH /state` endpoint actually part of v0? With only `running -> terminated` allowed, the broader endpoint looks like unused surface.

**Overall Assessment**
Not ready as written. The plan needs one explicit decision about post-termination integration behavior, and one explicit spec broadening so `Experiment` state participates in the same atomic event/storage contract as tasks, ideas, and variants. After that, the field-removal and `intended_executor` parts look tractable.
