## 1. Context

After 12a-2 the orchestrator is well-defined as a role with four
decision types (`ideation_creation`, `execution_dispatch`,
`evaluation_dispatch`, `integration`). Each decision is gated by
`dispatch_mode` and authenticated as the `orchestrators` group.

What's still missing ŌĆö and what makes the orchestrator's loop today
either run-forever or terminate-on-quiescence ŌĆö is a principled
decision about *when an experiment is done*. The four termination
fields in [`spec/v0/02-data-model.md`](../../spec/v0/02-data-model.md)
┬¦2.1 / ┬¦2.3 (`max_variants`, `max_wall_time`, `convergence_window`,
`target_condition`) are normative in the spec but **enforced
nowhere** in the reference impl
([`MANUAL_UI_ISSUES.md`](../../MANUAL_UI_ISSUES.md) ┬¦14). The
operational escape hatch from PR #47
(`EDEN_MAX_QUIESCENT_ITERATIONS`) is a quiescence heuristic, not a
termination condition; it conflates "no work right now" with
"experiment is over" (issue #1).

12a-3 settles this by making **termination a deployment policy, not
a spec mechanism**:

- The spec defines the *termination decision* contract (a fifth
  orchestrator decision type that runs before the existing four
  each iteration) and the *experiment lifecycle* (`running` ŌåÆ
  `terminated`).
- The spec does NOT enumerate specific termination conditions. The
  four legacy fields in chapter 02 are removed; deployments that
  want them re-implement them as a callable.

Separately, 12a-3 adds `intended_executor` to ideas. This was the
field 12a-2 needed to support operator-driven `create_task(kind=
execution)`: an operator wants to push an idea to a specific
executor; without `intended_executor`, the operator-created
execution task would fall back to `target=null`, which is
indistinguishable from auto-orchestrator output ŌĆö so the operator
path was deferred. With `intended_executor` landed, the operator
authority on `create_task(kind=execution)` becomes meaningful.

## 2. Decisions

These are the load-bearing design calls; ┬¦3 unpacks each.

1. **Termination becomes the orchestrator's 5th decision type.** It
   runs *before* the four existing decision types in each
   iteration. If the policy returns `terminate(reason)`, the
   orchestrator emits `experiment.terminated`, transitions the
   experiment state to `terminated`, and the orchestrator's loop
   exits cleanly.

2. **`experiment.state` becomes a first-class field.** Two values
   in v0: `"running"` (default at experiment creation) and
   `"terminated"`. Transition is one-way. The state lives on the
   experiment row in storage, not on the experiment-config (config
   is declarative input; state is observed runtime).

3. **Termination-policy callable.** Reference impl exposes
   `--termination-policy <module:callable>` parallel to 12a-2's
   `--ideation-policy`. Default reference policy is
   `eden_orchestrator.policies:never_terminate` (matches today's
   "run until operator stops" behavior). A reference library of
   common policies ships separately:
   - `max_variants_policy(N)` ŌĆö terminate when integrated variants
     Ōēź N.
   - `max_wall_time_policy(duration)` ŌĆö terminate when wall-time
     exceeds duration.
   - `convergence_window_policy(N)` ŌĆö terminate when objective
     hasn't improved in N integrated variants.
   - `target_condition_policy(expr)` ŌĆö terminate when an objective
     expression evaluates true.
   These match the four legacy fields' semantics ŌĆö they're just
   not normative anymore.

4. **The four legacy termination fields are removed from the
   normative experiment-config schema.** Greenfield: no shim. Any
   deployment that still sets them gets them silently ignored
   under the existing `additionalProperties` posture.

5. **`Idea.intended_executor: TaskTarget | null`.** A new optional
   field on the idea schema. Reuses the 12a-1 tagged-target shape
   (`{kind: "worker", id: "..."}` | `{kind: "group", id: "..."}` |
   `null`). When the orchestrator's `execution_dispatch` decision
   creates an execution task from a `ready` idea, it copies
   `idea.intended_executor` to `task.target`. Default `null` (the
   12a-2 status quo).

6. **Operator-driven `kind=execution` task creation is now in
   scope.** The authority restriction in 12a-2 ┬¦3.7
   (`create_task(kind=execution)` was orchestrators-only) lifts:
   admins MAY now create execution tasks directly, with explicit
   `idea_id` + an OPTIONAL `target` override (defaulting to the
   referenced idea's `intended_executor` if the field exists).

7. **The termination decision dispatch_mode key is `"manual"` by
   default for backward compatibility.** Adding `termination` as
   `auto` by default would change the runtime behavior of every
   existing deployment (terminate via the default never-terminate
   policy, which is harmless, but the dispatch_mode default on
   the new key still has to be chosen). Default `"manual"` means:
   the auto-orchestrator does NOT consult any termination policy
   unless the operator explicitly opts in via the wire op or
   compose env var. This matches today's effective behavior
   (no termination ever fires) and makes the new mechanism
   strictly opt-in.

## 3. Design

### 3.1 Spec chapter 03 ┬¦6 ŌĆö adding the termination decision type

The 12a-2 ┬¦6.2 list of four decision types becomes a list of
five, with `termination` added as the first item (it runs before
any of the other four each iteration). Sketch:

> ### 6.2 Decision types
>
> The orchestrator runs five decision types per iteration. The
> "termination" decision is consulted **first**; if it returns
> `Terminate`, the orchestrator stops creating new tasks but MUST
> continue integrating any `status == "success"` variants whose
> `variant_commit_sha` is unset (per ┬¦3 integrator contract).
> Integration continues until no such variants remain, at which
> point the orchestrator's loop has no remaining work and the
> deployment may shut it down.
>
> **Termination decision** (`dispatch_mode.termination`).
> Before each iteration, the orchestrator MAY consult a
> deployment-supplied termination policy. The policy is invoked
> with a read-only view of experiment state and returns one of:
>
> - `Continue` ŌĆö proceed to all five operational decisions below
>   for this iteration.
> - `Terminate(reason: str)` ŌĆö emit `experiment.terminated` with
>   the policy's reason, atomically transition the experiment's
>   `state` from `"running"` to `"terminated"`. From the next
>   iteration onward (and in the rest of this iteration after the
>   transition fires), only the **integration** decision continues
>   to run; the other three operational decisions
>   (`ideation_creation`, `execution_dispatch`,
>   `evaluation_dispatch`) MUST NOT run.
>
> The policy is invoked only when `dispatch_mode.termination ==
> "auto"`. When `"manual"`, the orchestrator runs operational
> decisions unconditionally (the historical behavior). A
> `terminated` experiment is absorbing in v0 ŌĆö there is no
> `terminated ŌåÆ running` transition.
>
> Why integration continues post-termination: the ┬¦3 integrator
> contract requires every `status == "success"` variant without
> `variant_commit_sha` to be eventually integrated. Stranding a
> success variant unintegrated would violate the canonical
> lineage invariant. Termination stops *new work*; it does not
> abandon *committed work in flight*.
>
> The four operational decisions (ideation_creation,
> execution_dispatch, evaluation_dispatch, integration) are
> otherwise unchanged from 12a-2 ┬¦6.2.

### 3.2 Spec chapter 02 ŌĆö `experiment.state` field

The experiment object (currently a thin wrapper around
experiment-config) gains a `state` field. Sketch:

> ### 2.5 Experiment lifecycle state
>
> The experiment runtime carries a `state` field independent of
> the declarative `experiment-config`:
>
> | Value | Meaning |
> |---|---|
> | `"running"` | Default at creation. The orchestrator MAY make any decision per chapter 03 ┬¦6.2. |
> | `"terminated"` | The Store MUST reject all task-creation ops (`create_ideation_task`, `create_execution_task`, `create_evaluation_task`) with `eden://error/illegal-transition`. In-flight tasks complete or are reclaimed per the normal task-lifecycle. The orchestrator's integration decision (┬¦6.2 decision-type 4) continues to run until no `status == "success"` variants without `variant_commit_sha` remain. Once integration drains, the orchestrator's loop has no remaining work; the deployment may stop the orchestrator container. |
>
> Transition: `running ŌåÆ terminated` is one-way; emitted with an
> `experiment.terminated` event carrying a `reason: str`.
> `terminated ŌåÆ running` is NOT a v0 transition; reactivation of a
> terminated experiment is reserved for a future spec change.

The `state` field is observed-only ŌĆö it's NOT part of
`experiment-config` (which is declarative input) and NOT
mutable through any wire op except `update_experiment_state` (a
new admin-gated op for explicit operator-driven termination, in
addition to the policy-driven path).

### 3.4 Reference termination-policy library

New module
[`reference/packages/eden-orchestrator/src/eden_orchestrator/policies.py`](../../reference/packages/eden-orchestrator/src/eden_orchestrator/policies.py)
(extending 12a-2's policy module):

```python
def never_terminate(state: ExperimentStateView) -> TerminationDecision:
    return Continue()

def max_variants_policy(target: int) -> Policy:
    def _policy(state: ExperimentStateView) -> TerminationDecision:
        if state.integrated_variant_count >= target:
            return Terminate(reason=f"max_variants={target} reached")
        return Continue()
    return _policy

def max_wall_time_policy(duration: timedelta) -> Policy:
    def _policy(state: ExperimentStateView) -> TerminationDecision:
        elapsed = datetime.now(UTC) - state.experiment_created_at
        if elapsed >= duration:
            return Terminate(reason=f"max_wall_time={duration} reached")
        return Continue()
    return _policy

def convergence_window_policy(window: int) -> Policy:
    def _policy(state: ExperimentStateView) -> TerminationDecision:
        ...  # inspect ordered objective values; terminate if no improvement
    return _policy

def target_condition_policy(expr: str) -> Policy:
    ...  # parse expr against the latest variant's metrics; terminate if true
```

`Policy = Callable[[ExperimentStateView], TerminationDecision]`.
`TerminationDecision = Continue | Terminate`. Both shapes match
the design doc's "Continue / Terminate(reason?)" sketch.

The `ExperimentStateView` was introduced in 12a-2 for the
ideation-policy callable; 12a-3 extends it with new counters
needed by termination policies:

- `integrated_variant_count: int` ŌĆö count of variants with
  `variant_commit_sha` set (i.e., integration completed).
- `experiment_created_at: datetime` ŌĆö for wall-time computation.
- `objective_history: list[float]` ŌĆö ordered list of objective
  values (one per integrated variant); used by
  `convergence_window_policy`.
- `latest_metrics: dict[str, Any]` ŌĆö the latest variant's
  evaluation; used by `target_condition_policy`.

These are derived from existing Store reads; no new normative
ops required.

### 3.5 The termination decision flow

Each orchestrator iteration runs:

```text
1. Read experiment.state.
   - If "running": proceed to step 2.
   - If "terminated": SKIP step 2 (no policy consultation;
     terminated is absorbing) and proceed to step 4 with only
     the integration decision enabled.
2. Read dispatch_mode (12a-2). If dispatch_mode.termination ==
   "auto":
   a. Build ExperimentStateView.
   b. Invoke the termination policy.
   c. If Terminate(reason): atomically transition state to
      "terminated", emit experiment.terminated. Continue to step 4
      with only the integration decision enabled.
3. (Reached only when state was "running" entering this iteration
   and the policy returned Continue or termination was manual.)
4. Run the operational decisions per dispatch_mode (12a-2 ┬¦6.2):
   - If experiment.state was "running" at iteration start AND
     stayed "running" through step 2: run all four decisions.
   - If experiment.state is "terminated" (either entering this
     iteration or after step 2c's transition): run only the
     integration decision; the other three are skipped.
5. If integration drained on this iteration (no success variants
   without variant_commit_sha remained), record that the
   orchestrator has no further work and the loop may exit on the
   next quiescence check.
```

The atomic transition in step 2c is a Store-level composite
commit: state field update + event emit, single transaction. It
mirrors the 12a-2 ┬¦3.6 reassign_task pattern.

Multi-instance safety: with N orchestrator replicas, each
independently reading `experiment.state == "running"` and
calling its own termination policy, only one of them needs to
"win" the transition. The Store's atomic update + event-emit
gives idempotency: the second replica's transition attempt sees
the state is already `"terminated"` and no-ops (no second event,
no error). The 12a-2 ┬¦6.4 "exact-idempotent decisions" class
covers this. Both replicas then proceed to step 4 with only
integration enabled; integration is itself
exact-idempotent per 12a-2 ┬¦6.4.

### 3.8 Edge-case: termination policy raising

A misbehaving termination policy that raises rather than returning
`Continue` or `Terminate` would be a runtime fault. Contract:

- Raised exception ŌåÆ orchestrator logs the exception with
  `policy_error` and treats the iteration as if `Continue` were
  returned (i.e., the operational decisions still run). The
  orchestrator does NOT terminate due to a policy error.
- The orchestrator emits an `experiment.policy_error` event so
  operators see the failure in the admin event log.
- This contract is documented in the spec chapter 03 ┬¦6 (the
  termination decision type's "fault tolerance" subsection) so
  alternative implementations honor it.

Why not exit on policy error: a failing policy is the operator's
config bug, not a deployment failure. The historical
"never-terminate" behavior is a safer fallback than crashing the
orchestrator container.

### 3.9 Manual termination via wire op

In addition to the policy-driven path, an operator can terminate
an experiment directly: new wire op `POST
/v0/experiments/{E}/terminate` with body `{reason: str}`. Authority:
admin. Effect: same atomic transition as the policy-driven path
(state ŌåÆ "terminated", `experiment.terminated` event), but with
the operator-supplied reason.

This is the "operator stops the container; want to mark the
experiment officially done" path. Without this op, an operator
would have to write a one-shot termination policy callable,
which is friction.

- Multi-experiment orchestrator (one orchestrator process running
  N experiments concurrently): Phase 12c (control plane).
- Spec-level role-binding chapter: chapter 03 ┬¦7 stays a stub
  through 12a; lands when the binding pattern is stable.

### 4.3 Out of scope

- Custom policy DSL (the design doc ┬¦9 mentions "DSL expression"
  as one valid implementation; the reference impl uses Python
  callables only).
- Termination-policy hot-reload (a deployment that wants to
  change policy mid-experiment must restart the orchestrator
  container).
- Per-experiment termination policy override (12a-3's policy is a
  deployment-level CLI flag; per-experiment policies are a future
  control-plane feature).

### 4.4 Non-goals

- Backwards compatibility with deployments that set the four
  removed fields. Greenfield posture; the fields are silently
  ignored.
- Spec-mandated default policies. The reference impl ships
  `never_terminate` as default; conforming impls MAY ship any
  default (or no default).
- Operator-supplied policy via wire (the reference impl loads
  callables at startup; a deployment that wants live policy
  uploads is out of scope).

## 7. Tricky areas

### 7.1 The policy-fault contract is intentionally lenient

┬¦3.8 specifies that a policy that raises does NOT cause
termination ŌĆö the orchestrator continues running operational
decisions. A reader's first instinct may be "fail-safe means
default-to-terminate", but that's the wrong default for this
system: a misbehaving termination policy is a config bug; the
deployment's recovery is "fix the callable and restart the
container", not "the experiment is over because my code threw".

The `experiment.policy_error` event makes the failure visible in
the admin event log, so it's not silent. Operators can build
alerting on top of that event.

### 7.2 Multi-instance termination

When N orchestrator replicas all observe `state == "running"` and
each invokes its own termination policy, several can simultaneously
return `Terminate(reason)`. Per ┬¦3.5, the Store's atomic update
serializes them: the first replica's transition wins, emitting
`experiment.terminated`. The second replica's call sees state is
already `terminated` and no-ops. The 12a-2 ┬¦6.4 idempotency class
covers this.

The test for this case is in `test_termination_decision.py` ŌĆö
two `StoreClient` instances each call `terminate_experiment`
concurrently; assert exactly one `experiment.terminated` event in
the log.

### 7.3 Schema-removal greenfield

Removing the four termination fields is a schema-breaking change
in the strict sense ŌĆö a deployment whose `experiment-config.yaml`
has them will continue to validate (because of
`additionalProperties`) but those fields will be silently
ignored. The field consumer is the reference policy library; if
the library's policy callable still reads from those fields, it
needs explicit re-pointing to the policy's own argument:

```python
# Before (12a-2 reference impl behavior, if any):
def max_variants_policy_from_config(state):
    return Terminate() if state.integrated_count >= state.config.max_variants else Continue()

# After (12a-3):
def max_variants_policy(target: int) -> Policy:
    def _policy(state):
        return Terminate() if state.integrated_count >= target else Continue()
    return _policy
```

The greenfield assumption (no external users) means we don't ship
a compat layer that reads the old fields if present. The `--termination-policy
eden_orchestrator.policies.max_variants_policy:N=20`-style CLI
flag is the new shape.

### 7.4 `Idea.intended_executor` resolution at dispatch time vs claim time

Per 12a-1 ┬¦D.3, target eligibility is resolved at claim time,
not at assignment. Same posture for `intended_executor`: when
the orchestrator copies `idea.intended_executor` to
`task.target`, the resolution of "is this worker still in the
group?" happens when a worker tries to claim. The orchestrator
does NOT validate that the target's group still has members at
dispatch time.

If `idea.intended_executor` names a worker who's been
deregistered between idea creation and dispatch, the resulting
task's `target` is stale ŌĆö no worker can authenticate as that
ID. The task stays in `pending` until an operator runs
`reassign_task` (12a-2).

This is consistent with the design ŌĆö claim-time resolution is the
single point of authority ŌĆö but it means an operator who creates
an idea with a specific `intended_executor` and then deletes the
worker has effectively orphaned the resulting task. The Web UI
admin idea-detail page MUST surface `intended_executor` so an
operator can see the wiring before deletion.

### 7.5 Manual termination + policy-driven termination interaction

If `dispatch_mode.termination == "auto"` and an operator hits
`POST .../terminate` while the policy is also returning
`Continue`, both transitions could race. The Store's atomic update
ensures only one wins; the second sees state is already
`terminated` and no-ops. Either path produces an
`experiment.terminated` event with whichever caller's reason
got there first.

If the operator wants their reason to win (vs. being raced by
the policy), they can flip `dispatch_mode.termination` to
`"manual"` first, then call `/terminate`. The dispatch_mode flip
is per-experiment per 12a-2 ┬¦3.2.

### 7.6 The `experiment` object schema is new

12a-3 introduces `Experiment` as a first-class object distinct
from `experiment-config`. The schema split mirrors task vs
task-payload: `experiment-config` is declarative input;
`experiment` is observed runtime state. The new
`experiment.schema.json` is small (3 fields: `experiment_id`,
`state`, `created_at`), but adding it as a new top-level schema
expands the chapter-02 surface.

The naming risk: a reader might confuse `experiment` (runtime)
with `experiment-config` (declarative). Mitigation: chapter 02
┬¦2.5's prose explicitly names the split and cites the two
schema files side-by-side.

## 8. Risks

1. **The four removed fields are referenced by user docs or
   external tooling.** Greenfield project, no external users ŌĆö
   risk is theoretical. Mitigation: a grep for the field names
   across the repo before merge confirms no surviving normative
   references.

2. **Reference policy library is too coupled to specific
   metric shapes.** `target_condition_policy(expr)` parses an
   expression against `latest_metrics`; if a deployment's
   evaluation_schema doesn't match the expr's free variables,
   the policy raises (caught per ┬¦3.8). Mitigation: the policy's
   docstring documents the expression language and links to the
   experiment-config's `objective.expr` parser, which is the
   canonical reference.

3. **`Idea.intended_executor` UI dropdown can grow unwieldy.**
   With many workers + groups, the dropdown is long. Mitigation:
   the reference UI uses an HTML datalist (autocomplete) for the
   field; not a hard cap on entries. Out of scope for a stronger
   filter.

4. **Termination event ordering during high concurrency.** If
   the policy returns `Terminate` simultaneously with several
   in-flight task submissions, the events interleave in the log
   in some order. The spec doesn't pin the order beyond "the
   atomic transition's `experiment.terminated` event is observed
   before any subsequent task-creation attempt's
   illegal-transition error". Operators reading the log
   chronologically may see, e.g., a `task.completed` after
   `experiment.terminated` (the task was already in flight; the
   completion is legitimate). Mitigation: documented in chapter
   02 ┬¦2.5.

5. **Re-introducing legacy vocab in the new spec text.** The
   strengthened guardrail from PR #60 catches the patterns; this
   chunk adds new spec surface that opens new opportunity for
   slip-ins. Mitigation: pre-submit
   `python3 scripts/check-rename-discipline.py` clean.
