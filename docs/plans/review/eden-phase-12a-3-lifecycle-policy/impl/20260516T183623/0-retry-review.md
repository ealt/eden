**Overall assessment**
The core Store-layer lifecycle work is generally well structured and the spec/storage/wire surfaces mostly line up, but I found two merge-blocking contract violations and two follow-up gaps. I did not rerun the full suite; the findings below come from code review plus focused repros against the current branch.

**Blocking issues**
- `experiment.policy_error` cannot be emitted on the deployed reference path. The spec makes this a MUST on termination-policy faults ([spec/v0/03-roles.md:210](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/spec/v0/03-roles.md:210)), but the orchestrator service always runs through `StoreClient` ([reference/services/orchestrator/src/eden_orchestrator/cli.py:331](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/reference/services/orchestrator/src/eden_orchestrator/cli.py:331), [reference/services/orchestrator/src/eden_orchestrator/cli.py:377](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/reference/services/orchestrator/src/eden_orchestrator/cli.py:377)). `StoreClient.emit_policy_error()` is a hard `NotImplementedError` ([reference/packages/eden-wire/src/eden_wire/client.py:904](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/reference/packages/eden-wire/src/eden_wire/client.py:904)), and the dispatch driver explicitly degrades that to log-only ([reference/packages/eden-dispatch/src/eden_dispatch/driver.py:227](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/reference/packages/eden-dispatch/src/eden_dispatch/driver.py:227), [reference/packages/eden-dispatch/src/eden_dispatch/driver.py:273](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/reference/packages/eden-dispatch/src/eden_dispatch/driver.py:273)). In other words, a real policy exception in the shipped orchestrator violates the normative observability contract.
- The v5 migrations clobber persisted unknown `dispatch_mode` keys. The spec says unknown keys must be accepted and omitted keys preserved across merge/readback ([spec/v0/02-data-model.md:88](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/spec/v0/02-data-model.md:88), [spec/v0/07-wire-protocol.md:114](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/spec/v0/07-wire-protocol.md:114), [spec/v0/05-event-protocol.md:148](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/spec/v0/05-event-protocol.md:148)). Both backends instead rewrite the whole stored object to a fixed 5-key literal during v5 ([reference/packages/eden-storage/src/eden_storage/_schema.py:185](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/reference/packages/eden-storage/src/eden_storage/_schema.py:185), [reference/packages/eden-storage/src/eden_storage/_postgres_schema.py:157](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/reference/packages/eden-storage/src/eden_storage/_postgres_schema.py:157)). Any existing extension key is silently lost on upgrade.

**Should-fix**
- The new termination conformance file does not actually exercise the decision-type-0 contract it cites. Despite the module docstring and citations, every scenario in [conformance/scenarios/test_termination_decision.py:1](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/conformance/scenarios/test_termination_decision.py:1) through [conformance/scenarios/test_termination_decision.py:121](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/conformance/scenarios/test_termination_decision.py:121) drives the admin `/terminate` endpoint directly; none covers `dispatch_mode.termination == "auto"`, a `Continue` path, or the required `experiment.policy_error` path. That blind spot is what lets the blocking `StoreClient.emit_policy_error` regression land undetected.
- Legacy termination fields are still presented as active inputs after the “fully removed” change. The user guide still marks `max_variants` and `max_wall_time` as required and shows them in the canonical config example ([docs/user-guide.md:50](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/docs/user-guide.md:50)), and the subprocess-env tests still seed configs with those fields ([reference/services/ideator/tests/test_ideator_subprocess_env.py:64](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/reference/services/ideator/tests/test_ideator_subprocess_env.py:64), [reference/services/evaluator/tests/test_evaluator_subprocess_env.py:32](/Users/ericalt/Documents/eden-worktrees/phase-12a-3-lifecycle-policy/reference/services/evaluator/tests/test_evaluator_subprocess_env.py:32)). That keeps the pre-12a-3 shape alive in active docs/tests even though the phase claims greenfield removal.
