**2. Feasibility**

Assessment: there are significant feasibility issues here, so I would stop before alternatives/completeness/risks.

- The migration design has no renderable switch. Decision 6 and §3.6 require `blob.backend=s3|gcs` deployments to keep the file PVC mounted read-only and instantiate a composite backend during migration ([docs/plans/eden-phase-13d-blob-backend.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13d-blob-backend.md:329), [docs/plans/eden-phase-13d-blob-backend.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13d-blob-backend.md:937)). But the values surface only exposes `blob.backend` ([docs/plans/eden-phase-13d-blob-backend.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13d-blob-backend.md:484)), and elsewhere the artifacts PVC is conditional on `blob.backend=file` only ([docs/plans/eden-phase-13d-blob-backend.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13d-blob-backend.md:223), [docs/plans/eden-phase-13d-blob-backend.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13d-blob-backend.md:1472)). Fresh S3 installs and migrated S3 installs need different manifests, but the chart has no value that distinguishes them.
- The worker-host artifact story contradicts the current subprocess binding. The plan says the subprocess protocol does not change and worker-host changes are basically CLI plumbing ([docs/plans/eden-phase-13d-blob-backend.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13d-blob-backend.md:302)), but §3.4 depends on the role process emitting a local file path that the host uploads ([docs/plans/eden-phase-13d-blob-backend.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13d-blob-backend.md:804)). That field does not exist today. The ideator subprocess emits `rationale` or a final `artifacts_uri` ([spec/v0/reference-bindings/worker-host-subprocess.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/spec/v0/reference-bindings/worker-host-subprocess.md:103), [reference/services/ideator/src/eden_ideator_host/subprocess_mode.py](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/reference/services/ideator/src/eden_ideator_host/subprocess_mode.py:250)); the evaluator subprocess also emits a final `artifacts_uri` ([spec/v0/reference-bindings/worker-host-subprocess.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/spec/v0/reference-bindings/worker-host-subprocess.md:205), [reference/services/evaluator/src/eden_evaluator_host/subprocess_mode.py](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/reference/services/evaluator/src/eden_evaluator_host/subprocess_mode.py:246)). As written, the host cannot perform the proposed upload without a binding change.
- Remote idea artifacts also break the executor subprocess path unless the host downloads them locally. The executor currently derives `rationale_path` only from `file://` URIs ([reference/services/executor/src/eden_executor_host/subprocess_mode.py](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/reference/services/executor/src/eden_executor_host/subprocess_mode.py:384), [reference/services/executor/src/eden_executor_host/subprocess_mode.py](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/reference/services/executor/src/eden_executor_host/subprocess_mode.py:493)), and the subprocess contract hands the user command a filesystem path, not bytes ([spec/v0/reference-bindings/worker-host-subprocess.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/spec/v0/reference-bindings/worker-host-subprocess.md:145)). If web-ui/ideator starts issuing `s3://` or `gs://` idea artifacts, execution tasks lose rationale access unless 13d adds a fetch-to-temp step or changes the binding.

**Overall Assessment**

The goal is sound, but this plan is not implementation-ready yet. The main blockers are structural: it needs an explicit migration mode in the chart, and it needs a coherent story for how non-`file://` artifacts interact with the existing subprocess contracts, especially the executor rationale path.
