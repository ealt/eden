**1. Missing Context**

Assessment: the problem statement in [issue-122-baseline-variant.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/docs/plans/issue-122-baseline-variant.md:1) is clear, but two inputs are still missing before the design is fully pinned.

- The plan treats seed SHA as a single-process CLI/env input to the orchestrator ([issue-122-baseline-variant.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/docs/plans/issue-122-baseline-variant.md:85)), but the repo already has a multi-experiment orchestrator mode where one process drives many experiments ([multi_loop.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/reference/services/orchestrator/src/eden_orchestrator/multi_loop.py:67), [cli.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/reference/services/orchestrator/src/eden_orchestrator/cli.py:201)). The plan needs to say where a per-experiment seed SHA lives in that mode.
- The plan does not say whether a baseline should count in termination-policy state. Today `attempted_variant_count` counts every variant, while `recent_evaluations` / `latest_evaluation` only consider integrated variants ([state_view.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/reference/packages/eden-dispatch/src/eden_dispatch/state_view.py:105), [termination.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/reference/packages/eden-dispatch/src/eden_dispatch/termination.py:84)). A default-on, never-integrated baseline therefore changes `max_variants_policy` behavior but is invisible to target/convergence policies. That needs an explicit rule.

**2. Feasibility**

Assessment: significant concerns here. I would stop at this level and revise the plan before judging alternatives/completeness/edge cases.

- Reusing generic `create_variant` for the override path is not viable as written. The plan relaxes `create_variant` so a baseline can be created directly in `success` ([issue-122-baseline-variant.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/docs/plans/issue-122-baseline-variant.md:38), [issue-122-baseline-variant.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/docs/plans/issue-122-baseline-variant.md:104)), but `POST /variants` is only worker-authenticated today, not orchestrator/admin gated ([07-wire-protocol.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/spec/v0/07-wire-protocol.md:185), [server.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/reference/packages/eden-wire/src/eden_wire/server.py:963)). That would let any registered worker fabricate a successful baseline with arbitrary metrics. If this path stays, the plan needs a new authority rule for baseline creation; otherwise use a dedicated orchestrator/admin-only op.
- The direct-success override also needs a real event-model amendment, not just an impl note. Current events define `variant.started` as `— → starting` with required `idea_id`, and `variant.succeeded` as `starting → success`; the success event is currently described as the evaluation-task terminal composite ([05-event-protocol.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/spec/v0/05-event-protocol.md:45), [05-event-protocol.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/spec/v0/05-event-protocol.md:121)). The plan says to emit the success event “in addition to (or composed with) `variant.started`” ([issue-122-baseline-variant.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/docs/plans/issue-122-baseline-variant.md:108)) but does not include [05-event-protocol.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/spec/v0/05-event-protocol.md:115) in the touched files list ([issue-122-baseline-variant.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/docs/plans/issue-122-baseline-variant.md:233)). As written, that contract is not coherent.
- The `idea_id` relaxation is much wider than schema/Pydantic parity. The plan focuses on `variant.py` / schema changes ([issue-122-baseline-variant.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/docs/plans/issue-122-baseline-variant.md:74), [issue-122-baseline-variant.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/docs/plans/issue-122-baseline-variant.md:139)), but current event payloads, integrator code, manifest generation, UI lineage, and admin observability all assume `variant.idea_id` exists ([05-event-protocol.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/spec/v0/05-event-protocol.md:121), [integrator.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/reference/packages/eden-git/src/eden_git/integrator.py:137), [_manifest.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/reference/packages/eden-git/src/eden_git/_manifest.py:45), [_lineage.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/reference/services/web-ui/src/eden_web_ui/routes/_lineage.py:228), [observability.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/reference/services/web-ui/src/eden_web_ui/routes/admin/observability.py:315)). Even if baselines are never integrated, these surfaces need explicit carve-outs or fallback behavior.
- The termination-drain carve is incomplete against the actual implementation. The plan correctly calls out the main integration predicate and §2.5 drain text ([issue-122-baseline-variant.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/docs/plans/issue-122-baseline-variant.md:118)), but the control-plane loop also decides “drained terminated” by checking that every `success` variant has `variant_commit_sha` ([multi_loop.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/reference/services/orchestrator/src/eden_orchestrator/multi_loop.py:227)), and the reference integration walk still iterates every `status=="success"` variant ([driver.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/reference/packages/eden-dispatch/src/eden_dispatch/driver.py:473)). Without touching those paths, multi-experiment mode still wedges on a successful baseline.
- The blast-radius analysis is understated. The plan only audits the compose smoke lower-bound assertions ([issue-122-baseline-variant.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/docs/plans/issue-122-baseline-variant.md:211)), but the full reference suite has exact-count orchestrator e2e expectations of exactly 3 variants / 9 tasks ([test_e2e.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/reference/services/orchestrator/tests/test_e2e.py:286), [test_e2e.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/reference/services/orchestrator/tests/test_e2e.py:345), [test_subprocess_e2e.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-122-baseline/reference/services/orchestrator/tests/test_subprocess_e2e.py:315)). Default-on baseline will break those unless the plan explicitly includes them.

**Overall Assessment**

The core motivation is sound, and the termination-hang insight is real. But the current plan is not ready as an implementation contract. The biggest blockers are the per-experiment seed-SHA source, the authority/event contract for direct-success baseline creation, and the much larger-than-stated `idea_id` / multi-experiment blast radius. I would revise those first, then re-review levels 3-5.
