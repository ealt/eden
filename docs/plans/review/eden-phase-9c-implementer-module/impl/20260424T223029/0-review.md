- **Bug** — [reference/services/web-ui/src/eden_web_ui/templates/implementer_claim.html](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/templates/implementer_claim.html:24): `proposal.artifacts_uri` is injected directly into `<a href=...>`. A malicious proposal can set `artifacts_uri` to `javascript:...` or `data:...`, and the draft page will render an executable link. The §A.1 trust boundary only guards inline file reads; it does not make arbitrary URI schemes safe to hyperlink. Fix by validating allowed schemes before rendering a link (`http`, `https`, maybe `file`), and render all other schemes as escaped text/code instead of a clickable `href`.

- **Risk** — [reference/services/web-ui/src/eden_web_ui/routes/implementer.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/implementer.py:388), [reference/services/web-ui/src/eden_web_ui/templates/implementer_orphaned.html](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/templates/implementer_orphaned.html:19): when submit retries are exhausted and `read_task()` itself fails, `_retry_submit_with_readback()` returns `("auto", ...)`. That tells the template to show “auto-recovers via reclaim”, but at that point the outcome is actually indeterminate: the submit may already have committed. This contradicts the §C / §C-recovery design, which distinguishes confirmed-committed from confirmed-uncommitted via read-back. Fix by returning a distinct indeterminate/transport outcome here and rendering operator-investigation prose, not auto-recovery prose.

- **Risk** — [reference/services/web-ui/src/eden_web_ui/routes/implementer.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/implementer.py:286): Phase 1 only catches `DispatchError` around `store.create_trial()`. In the real configured path this call goes through `StoreClient`, so transport exceptions (`httpx` timeout/connect errors) will bubble out as a 500. Worse, if the server committed the trial before the transport failed, the UI loses the ability to explain or reconcile that partial state. Fix by catching transport-shaped exceptions here too and either reconciling with a read-back or at least rendering an indeterminate/orphan page instead of crashing.

- **Risk** — [reference/services/web-ui/tests/test_implementer_partial_write.py](/Users/ericalt/Documents/eden/reference/services/web-ui/tests/test_implementer_partial_write.py:135), [reference/services/web-ui/tests/test_implementer_partial_write.py](/Users/ericalt/Documents/eden/reference/services/web-ui/tests/test_implementer_partial_write.py:170): the tests that claim to cover §C-recovery sub-case A / A' do not actually exercise the read-back path. In both monkeypatches, the first `store.submit()` call returns normally, so the route exits immediately with success and never retries or reconciles. That leaves the most important recovery logic effectively unpinned. Fix by having the first patched call commit and then raise a transport exception, so `_retry_submit_with_readback()` must enter the read-back branch.

Overall assessment: plan adherence is strong on the main flow, route wiring, and test breadth, but the implementation is weaker on transport-indeterminate behavior than the plan intends, and the new draft template introduces an unsafe link sink. I would not treat this as review-complete until the hyperlink sanitization and the misclassified read-back failure path are fixed, and the partial-write tests are tightened to cover the real reconciliation branches.
