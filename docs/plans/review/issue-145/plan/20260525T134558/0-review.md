**Level 1: Missing Context**
The problem statement is directionally right, but the plan is missing two pieces of implementation-defining context: where per-experiment config actually comes from in the current system, and how “registered in control plane” differs from “seeded in the task store” for operator-visible behavior. The route/test/template inventory also needs to be re-cut from the current tree, not inherited from the original 12c deferral text.

**Level 2: Feasibility**
As written, I do not think the approach is implementable without redesign. The config path in §3.4 is based on a non-existent API; the stale-selection logic misclassifies legitimate registry/task-store split states; and the credential-bootstrap design in §3.2 omits the locking and reissue behavior that the existing auth bootstrap needs to stay correct under concurrency. I stopped here rather than doing a full alternatives/completeness/risk pass, because those downstream judgments depend on a corrected design.

**Overall Assessment**
The plan is not ready for implementation. I would revise §3.2, §3.4, and §3.8 first around the actual current interfaces: pick a real source for per-experiment config, define the operator experience for control-plane-registered but task-store-unseeded experiments, and specify a locked credential-bootstrap path with a concrete control-plane auth story. After that, a second review pass should revisit alternatives, full surface completeness, and edge-case coverage.
