**Missing Context**

Assessment: the problem statement is mostly clear, but the plan is missing the one piece of context that matters most here: an explicit authority matrix for the existing admin surfaces.

- [docs/plans/issue-143-non-admin-default-signup.md](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/docs/plans/issue-143-non-admin-default-signup.md:103) treats "`/admin/*`" as one permission domain, but the current app does not. The router surface includes experiment-local admin pages, per-experiment worker/group registry pages, a read-only artifacts page, and deployment-wide control-plane pages; see [app.py](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/reference/services/web-ui/src/eden_web_ui/app.py:109), [admin_artifacts.py](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/reference/services/web-ui/src/eden_web_ui/routes/admin_artifacts.py:104), and [admin_experiments.py](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/reference/services/web-ui/src/eden_web_ui/routes/admin_experiments.py:105). Before the plan can prescribe a blanket guard, it needs a route-by-route table saying which surfaces are experiment-`admins`, which remain deployment-admin-only, and which are control-plane-admin.
- [docs/plans/issue-143-non-admin-default-signup.md](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/docs/plans/issue-143-non-admin-default-signup.md:121) discusses self-protection as if `admins` membership were always direct, but the actual model is transitive; see [spec/v0/02-data-model.md](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/spec/v0/02-data-model.md:316). The plan needs to say explicitly whether nested groups inside `admins` are supported on this surface, because that changes what “self-revoke” means.

**Feasibility**

Assessment: the wire-level direction is better than full web-ui proxying for promotion, but the proposed amendment is not internally consistent yet.

- [docs/plans/issue-143-non-admin-default-signup.md](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/docs/plans/issue-143-non-admin-default-signup.md:65) under-specifies the §13.3 change. Today §13.3 allows only `admin-gated`, `worker-gated`, or `either`, and only worker-gated endpoints may add a group gate; see [spec/v0/07-wire-protocol.md](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/spec/v0/07-wire-protocol.md:401). Adding `admin-OR-admins-group-gated` means rewriting the classification framework itself, not just adding one new summary bullet. The current dispatcher also hard-codes `require_admin()` on these routes; see [server.py](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/reference/packages/eden-wire/src/eden_wire/server.py:1437). Also, the claim at plan line 72 that old IUTs would “continue to be conforming” is wrong once [spec/v0/09-conformance.md](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/spec/v0/09-conformance.md:71) grows a MUST-backed scenario for worker-in-`admins` mutation.
- [docs/plans/issue-143-non-admin-default-signup.md](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/docs/plans/issue-143-non-admin-default-signup.md:103) says every `/admin/*` route should get `require_admin`, but several of those routes still proxy deployment-admin-only operations through `app.state.admin_store`: [admin_workers.py](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/reference/services/web-ui/src/eden_web_ui/routes/admin_workers.py:9), [admin_groups.py](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/reference/services/web-ui/src/eden_web_ui/routes/admin_groups.py:1), and especially [admin_experiments.py](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/reference/services/web-ui/src/eden_web_ui/routes/admin_experiments.py:170). As written, any experiment admin would inherit deployment-admin powers via the UI proxy. That directly contradicts the plan’s own decision to keep `register_group`, `register_worker`, and `reissue_credential` admin-principal-only.
- Extending the dual gate to `delete_group` is unsafe in the form proposed at [docs/plans/issue-143-non-admin-default-signup.md](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/docs/plans/issue-143-non-admin-default-signup.md:67) and [docs/plans/issue-143-non-admin-default-signup.md](/Users/ericalt/Documents/eden-worktrees/issue-143-plan/docs/plans/issue-143-non-admin-default-signup.md:128). The direct-id self-protection rail does not protect against indirect self-demotion through nested groups, and it does nothing about deleting the `admins` group outright. If `delete_group("admins")` stays in scope for the new dual gate, the plan needs a real “would the caller still resolve into `admins` after this mutation?” check, or a rule that reserved `admins` is not deletable through this path.

I’d stop here for this round. The core product decision is sound: non-admin by default, explicit promotion via `admins`, and worker-bearer authority for promotion is the right direction. But the plan is not ready to implement until it splits experiment-admin, deployment-admin, and control-plane-admin surfaces cleanly and turns the §13.3 change into a coherent dispatcher model.
