The listing builds `serve_uri` as `f"file://{resolved}"`, which is not a valid file URI for paths containing reserved characters like `?` or `#`. When the user clicks that link, `/artifacts` reparses it with `urlparse()`, so `a?b.txt` is interpreted as path `a` plus query `b.txt`, and the fetch 404s. I reproduced this against the app: `/admin/artifacts/` rendered a link for `a?b.txt`, and following that link returned 404. The current tests miss it because they assert the same raw-URI construction and the “link back” test calls `/artifacts` directly instead of following the href emitted by the listing.  
The fix should be to generate a real file URI with percent-encoded path bytes, e.g. `resolved.as_uri()`, then keep query-param escaping in the template. Add an end-to-end test that creates a filename containing `?` and `#`, extracts/follows the rendered `/artifacts?uri=...` href, and asserts a 200 with the file contents.
