# Implementation Review: issue-107-artifacts-listing

## Issue (planless PR — issue is the contract)

- GitHub: <https://github.com/ealt/eden/issues/107>
- PR: <https://github.com/ealt/eden/pull/108>
- Branch: `impl/issue-107-artifacts-listing` (one commit ahead of `main`)

### Issue summary

Add a read-only directory-listing route to the reference web-ui that
surfaces the contents of `--artifacts-dir` to operators. The existing
`GET /artifacts?uri=file://...` route serves an exact file but offers
no way to *discover* what artifacts exist; operators currently have to
`docker exec` into the container or read the host bind-mount. The new
route lives under `/admin/*` (the operator-inspection namespace), is
read-only, reuses the same session-cookie auth + path-containment
posture as the existing artifacts route, and renders an empty-state
banner for scripted-mode deployments (which produce zero artifact
files by design).

### Issue-specified contract (paraphrased from #107)

| Requirement | Source |
|---|---|
| List every file under `--artifacts-dir`, recursively | "Idea" §1 |
| Per file: size + mtime + link back to existing `/artifacts?uri=file://...` | "Idea" §1–2 |
| Session-cookie auth, admin-eligible (same as `/artifacts`) | "Idea" §3 |
| Path-containment: rooted at `--artifacts-dir`; listing never escapes the jail | "Idea" §4 |
| Empty-listing banner explaining scripted-mode produces nothing | "Idea" §5 |
| **Out of scope**: upload / delete / writes | "Out of scope" §1 |
| **Out of scope**: non-`file://` URIs (defer to #102) | "Out of scope" §2 |

## Change Summary

- New `GET /admin/artifacts/` endpoint. Auth: `get_session(request)` —
  unauthenticated GET returns 303 to `/signin`. (Same helper as
  `routes/artifacts.py` and `routes/admin_workers.py`.)
- `os.walk(base, followlinks=False)` enumerates the artifacts dir
  recursively. Each candidate is `Path.resolve()`-ed and
  `is_relative_to(base)`-checked; non-file inodes are dropped. This
  mirrors the trust-boundary helper at
  `reference/services/web-ui/src/eden_web_ui/routes/_helpers.py:_read_inline_artifact`.
- Symlink containment: `followlinks=False` on `os.walk` plus the
  `resolve()`+`is_relative_to` re-check on every leaf rejects
  symlinks whose targets land outside the jail. The link itself is
  followed by `resolve()` so the check is on the target, not the
  symlink path.
- Each rendered entry carries: relative path (POSIX), `st_size`,
  ISO-8601 UTC mtime, and an `<a href="/artifacts?uri=file://...">`
  link pointing to the existing serving route. The link uses Jinja's
  `urlencode` filter (which leaves `/` unescaped per
  `urllib.parse.quote`'s default).
- Empty-state branch in the template renders a `banner banner-warn`
  explaining that scripted-mode deployments produce no artifact files
  — heading off "why is this blank?" support traffic.
- Read-only. No POST / DELETE / PUT routes are registered on the
  module. No mutation controls are rendered in the template.
- Tests cover: auth gate (unauth GET → 303); empty-state banner;
  populated render with size + mtime + link target; recursive walk
  at depth > 1; symlink-to-outside-jail is silently dropped (target
  path, link path, and payload bytes all absent from rendered HTML);
  traversal URI returns 404 from the serving route (sanity check that
  the existing posture is intact — the listing route never accepts a
  path from the caller).

## Specific surfaces worth focused review

These are the areas the user flagged as benefiting from a second pass:

1. **Auth surface.** The route inherits the same `get_session(request)`
   shape as `routes/artifacts.py` and the other admin routes. There is
   no admin-bearer check (the existing artifacts route doesn't gate on
   `admin_store`; any signed-in session can fetch files). The listing
   surfaces only the *contents* the serving route would already serve;
   anyone who could read the cookie could already enumerate the
   directory via the serving route by variant-and-error. Is the
   posture-parity argument sound, or should the listing route gate on
   `admin_store is not None` (treating discovery as a stricter
   capability than serving)?

2. **Path containment (OWASP-class).** The listing walks
   `app.state.artifacts_dir.resolve()` and applies
   `Path.resolve()` + `is_relative_to(base)` to each leaf:
   - `followlinks=False` on `os.walk` prevents descent into a
     symlinked directory. A symlinked *file* under the jail still has
     its target re-checked via `resolve()`+`is_relative_to`, so a leaf
     symlink pointing outside the jail is silently dropped.
   - `Path.resolve()` is strict by default in Python 3.13+? Actually
     `Path.resolve()` returns the canonicalized path without
     requiring existence; `is_relative_to` then compares. A path that
     can't be resolved (broken symlink, EPERM on parent) is caught by
     `except OSError: continue`.
   - **TOCTOU**: the resolved path is later embedded in the rendered
     anchor's `uri` querystring; the serving route re-resolves and
     re-checks on the click. So the listing's `is_relative_to` check
     is not load-bearing for the eventual fetch — it's only
     load-bearing for what gets *enumerated*. Is that defensible?
   - The `serve_uri` is constructed as `f"file://{resolved}"` without
     URL-encoding the path; Jinja's `urlencode` filter then encodes
     the whole querystring. Are there path characters that survive
     that pipeline to confuse the serving route's `urlparse` +
     `unquote`?

3. **Operator-supplied path configurability.** `--artifacts-dir` is
   operator-supplied; the listing trusts the operator's resolution of
   that path. A misconfigured deployment that points `--artifacts-dir`
   at `/` would surface the entire host filesystem. That's a
   deployment-time misconfiguration, not a runtime escape — but worth
   flagging if the review thinks the route should validate the
   configured root (e.g., refuse `/`, refuse a path that isn't a
   directory at boot).

4. **Plain `.resolve()` follows symlinks at every component.** If
   `<artifacts-dir>/a` is itself a symlink to a directory outside the
   base, `os.walk(base, followlinks=False)` will NOT descend into it,
   so the contents are safely skipped. But are there other shapes
   (hard-link, mount-bind, race on the directory) that the listing
   should care about?

5. **No max-files / pagination cap.** The issue lists this as an
   optional extension. The route walks the entire tree synchronously
   on every render; a pathological artifacts dir (10⁵+ files) would
   render very slowly and produce a huge HTML response. Defer to
   follow-up?

6. **`_format_mtime` always returns UTC.** Operators viewing the
   listing in another timezone will see UTC timestamps. Consistent
   with the rest of the admin UI (which also serializes ISO-8601 UTC
   in event tables / task tables), but worth a sanity check.

7. **No CSRF token in the template.** Read-only route, no POST forms,
   so no CSRF is needed. Confirm.

## Review Status

Round 0 — codex flagged one Bug: `serve_uri = f"file://{resolved}"`
is not a valid file URI when the path contains URI-reserved bytes
(`?`, `#`, `%`, …). Codex reproduced end-to-end: a file named
`a?b.txt` rendered a clickable listing entry whose link 404'd when
followed (the serving route's `urlparse` split the path at the
unescaped `?`).

### Round 1 — addressed

- **Fix** `reference/services/web-ui/src/eden_web_ui/routes/admin_artifacts.py:91-99`:
  build `serve_uri` with `resolved.as_uri()` instead of
  `f"file://{resolved}"`. `Path.as_uri()` percent-encodes path
  bytes that are reserved in a URI.
- **New test**
  `reference/services/web-ui/tests/test_admin_artifacts_routes.py::TestAdminArtifactsPopulated::test_rendered_link_round_trips_for_reserved_chars`:
  creates files named `a?b.txt`, `c#d.txt`, and `e f.txt`; extracts
  the rendered `<a href>` from the listing HTML; follows the link;
  asserts a 200 + correct payload.
- **Update** the existing `test_lists_file_with_size_mtime_and_serve_link`
  assertion to match the new `as_uri()`-based shape (was previously
  asserting the bugged `f"file://..."` shape, which masked the bug).
