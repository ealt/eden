## 1. Context

### 1.0 Substrate baseline (post-13a / 13d)

After 13a + 13b + 13c + 13d the reference k8s deployment runs:

- A **Gitea `StatefulSet`** (forecast by 13a §3.4 + §5.1) at
  the chart-managed name `<release>-eden-gitea-0`, image
  `gitea/gitea:1.22.6-rootless` (matching Compose's pin at
  [`reference/compose/compose.yaml`](../../reference/compose/compose.yaml)
  line 33), backed by a PVC sized `storage.giteaSize` (default
  `20Gi`) on the cluster's default `StorageClass`. Replicas: 1.
- **Auth posture (per 13a §3.3a, mirroring Compose's
  10d-followup-B baseline):** a single `eden` admin user with
  a per-experiment HTTP Basic password generated by
  `setup-experiment.sh` and exposed to every worker host
  service via a credential-helper script. The script reads
  `GITEA_REMOTE_PASSWORD` from the env (chart Secret →
  envFrom) and hands it to `git -c credential.helper=...`
  during clone / push / fetch.
- **No branch protection.** The integrator and any worker-host
  identity can write `variant/*`; any worker-host identity can
  write `work/*`. Chapter 6 §1.2 / §1.3 ownership invariants
  are enforced **only by the integrator's behavior** — the
  protocol's "integrator is the sole writer of `variant/*`"
  rule lives in
  [`spec/v0/06-integrator.md`](../../spec/v0/06-integrator.md)
  §1.2 and is honored by the reference integrator at
  [`reference/packages/eden-git/src/eden_git/integrator.py`](../../reference/packages/eden-git/src/eden_git/integrator.py),
  but Gitea would happily accept a `git push` to
  `variant/<id>-<slug>` from any worker that has the shared
  `eden` user's password.
- **No PR usage.** The integrator merges by direct push of the
  squashed `variant/*` commit (per chapter 6 §3.4's atomic
  three-step write).
- **Per-experiment Gitea repo, owned by a USER.** The
  bootstrap creates `eden/<experiment_id>` via
  `gitea admin user create` +
  `POST /api/v1/user/repos`
  ([`reference/scripts/setup-experiment/setup-experiment.sh`](../../reference/scripts/setup-experiment/setup-experiment.sh)
  lines 354-394). The repo's owner is the `eden` USER (not
  a Gitea organization). This matters because Gitea's
  branch-protection feature only supports team-based push
  allow-lists on **organization-owned** repositories
  ([Gitea docs — Protected branches](https://docs.gitea.com/usage/access-control/protected-branches));
  user-owned repos can only use the
  `push_whitelist_usernames` form. 13e MUST switch the
  repo ownership to an organization to support the §3.2
  team-based ACL design — see Decision 0 and §3.0 for the
  new org topology + migration.
- **Worker identity (post-12a-1).** Every worker is registered
  via `Store.register_worker(worker_id, ...)` and authenticates
  to the task-store-server via a per-worker credential
  (Argon2id-hashed). The
  [`reference/packages/eden-storage/src/eden_storage/protocol.py`](../../reference/packages/eden-storage/src/eden_storage/protocol.py)
  `Worker` record carries the per-experiment registry. A
  separate `reissue_credential(worker_id)` op lets admins
  rotate per-worker credentials. **None of this propagates to
  Gitea today** — the Gitea side is still single shared
  password.
- **Web-UI admin's work-refs GC view (9e §5).** Today the
  operator-facing PR-shape work surfaces only as a chart-9
  admin-side reclamation page; there are no Gitea-native PRs
  involved.

13e's job: replace these four soft postures (shared password,
no ACLs, no PRs, no rotation alignment) with a hardened
deployment posture aligned with the rest of Phase 13's
substrate-as-deployment-policy story.

### 1.1 What 13e changes

13e adds:

1. **Per-worker Gitea identities.** Each registered worker
   (per 12a-1's `register_worker`) gets a corresponding Gitea
   user provisioned by an admin-bootstrap helper (`eden_gitea`
   module). The Gitea username derives from the
   12a-1 `worker_id` via a documented mapping
   (§3.1.2 covers the underscore-vs-hyphen edge). The Gitea
   user gets a per-worker scoped access token; that token is
   the credential the worker host injects into git operations
   instead of the shared `eden` password.
2. **Per-branch ACLs.** Three Gitea branch-protection rules
   per repo (`main`, `variant/*`, `work/*`), provisioned by
   the same admin-bootstrap helper, mapping the chapter-6 §1
   namespace ownership rules to Gitea's branch-protection
   feature (`/api/v1/repos/{owner}/{repo}/branch_protections`).
   §3.2 details each rule and its push-allow-list.
3. **Native PR review** — opt-in for two specific surfaces:
   - **Human-executor `work/*` submissions** (when the
     operator's deployment uses a human-driven executor
     instead of the scripted / k8s-job executor): the
     human-executor opens a PR from `work/<branch>` against
     a per-experiment review branch; the human-evaluator (or
     a designated reviewer) approves. Decision in §3.3.
   - **Operator-mediated integrator approval** (optional,
     for high-stakes deployments): the integrator opens a
     PR for each `variant/*` and auto-merges via API as part
     of its existing chapter-6 §3.4 atomic write — same
     atomicity, with audit trail. Default OFF; opt-in via
     `gitea.pr.integratorApproval.enabled`.

   Programmatic submissions (scripted executor, scripted
   evaluator, scripted ideator, integrator-without-the-PR
   knob) stay direct-push, gated only by the per-branch ACLs.
4. **OAuth/OIDC SSO** as a values-driven option for
   operator-facing logins (the human-executor / evaluator /
   admin web flows). Auth for the Gitea API used by EDEN
   worker hosts is **always per-worker scoped tokens** — SSO
   doesn't apply to programmatic worker auth. §3.4.
5. **Operator-required values** posture per the AGENTS.md
   substrate-plan pitfall: empty defaults +
   `values.schema.json minLength: 1` for OAuth client secrets
   / OIDC issuer URLs / SSO config; per-experiment
   admin-token references via `existingSecret`; no fictional
   defaults. §3.5.
6. **Credential rotation alignment.** When EDEN's
   `reissue_credential(worker_id)` rotates a worker's
   task-store credential, the same admin-bootstrap helper
   ALSO rotates that worker's Gitea token. The Gitea token
   is treated as a **derived** secret of the EDEN worker
   credential — operators never rotate the two
   independently. §3.6.
7. **Compose-side compatibility** — Compose stays on the
   shared-password posture by default to keep dev workflow
   simple; 13e adds **opt-in** Compose overlays
   (`compose.gitea-acl.yaml` + `compose.gitea-oauth.yaml`)
   for operators who want the hardened posture in Compose.
   §3.7.
8. **Migration runbook** at
   `docs/deployment/migrating-to-gitea-hardening.md`
   covering the rollover from 13a's shared `eden` user to
   per-worker users + branch protection. §3.8.
9. **A new CI smoke job** `helm-smoke-gitea-oauth` that
   stands up Gitea with OAuth enabled, branch protection,
   and per-worker identities, runs the fixture experiment to
   quiescence, and asserts that (a) the same end-state
   invariants the existing helm smokes assert hold, AND
   (b) Gitea-side ACLs reject a forged push from a
   wrong-role worker. §3.9.

13e does NOT change:

- Any normative spec text. The wire protocol gets ONE
  small additive operation —
  `update_worker_labels(worker_id, labels)` — described
  in §3.0a.4 below. This is informative-binding additive
  (the 12a-1 plan reserved the worker-record's `labels`
  field as free-form operator metadata, and 12a-1's wire
  surface only exposed `register_worker` for writing it;
  13e needs label mutation for the migration backfill +
  runtime relabeling). The new op is non-breaking
  (existing 12a-1 clients keep working without it) and
  conformance-impact-free (the existing scenarios test
  observable behavior of registered workers, not the
  mutation surface). Chapter 6 §1's branch-namespace
  ownership rules are already normative; 13e adds
  Gitea-side enforcement in addition to the integrator-
  behavior enforcement that satisfies the spec today.
- The Compose deployment's default behavior. Operators who
  don't opt into the new overlays continue using the shared
  `eden` password.
- The integrator's chapter-6 §3.4 atomic-three-step write
  contract. The opt-in PR-mediated integrator path performs
  the merge via Gitea's API in a way that preserves
  atomicity (§3.3.2).

## 2. Decisions

These are the load-bearing design calls; §3 unpacks each.

1. **Repo ownership moves from a USER to a Gitea
   ORGANIZATION.** 13e introduces a per-deployment Gitea
   organization named `eden` (one org per chart release;
   the org name is `eden` literal, not parameterized — the
   chart's release-name namespacing already isolates
   deployments at the k8s namespace + Gitea-instance
   level). Per-experiment repos move from `eden/<experiment_id>`
   (user-owned in the 10d-fb baseline) to
   `eden/<experiment_id>` (same path, but `eden` is now an
   organization). This is required because Gitea's
   branch-protection allow-lists support
   `push_whitelist_teams` ONLY on organization-owned
   repositories ([Gitea docs — Protected branches](https://docs.gitea.com/usage/access-control/protected-branches));
   the §3.2 team-based ACL design depends on this. The
   org is created at chart-install time by the
   bootstrap Job (idempotent — uses
   `POST /api/v1/orgs` and ignores
   `422 Unprocessable Entity` on already-exists). Existing
   workers + the `eden` user identity continue to exist;
   the user becomes a member of the new org with admin
   role for the migration window (§3.8.2). See §3.0 for
   the topology details + migration story.

2. **Worker role is sourced from a reserved label
   `eden.role` on the 12a-1 `Worker` record.** 12a-1's
    `register_worker(worker_id, labels?)` accepts free-form
    `labels` (per
    [`docs/plans/eden-phase-12a-1-worker-identity.md`](eden-phase-12a-1-worker-identity.md)
    §"Worker shape"); 13e reserves the
    `eden.role` label key for the role-value that
    `eden-gitea-admin provision-user` uses to choose the
    Gitea token's scopes + team membership. The legal
    values match the §3.1.1 set
    (`executor | evaluator | ideator | integrator |
    operator | human-executor | human-evaluator |
    human-ideator`). Workers without an `eden.role` label
    are NOT auto-provisioned a Gitea identity — operators
    can run them as scripted-mode workers that don't need
    Gitea writes (e.g., ideator subprocess hosts that only
    read), OR explicitly opt them out via
    `labels["eden.role"] = "none"`. Rejected: deriving
    role from `worker_id` substring matches (fragile);
    making it a separate `Worker.role` field on the
    record (12a-1 §10 spec is frozen for v0; adding a
    field requires a future spec amendment); reading it
    from a separate per-experiment YAML the operator
    maintains alongside the chart values (one more file
    to keep in sync). See §3.0a for the label-convention
    details. The `eden.` label-prefix namespace is
    consistent with the reserved-identifier rules in
    12a-1 §"Worker-id grammar" — `eden.*` labels are
    reserved for protocol-binding-defined metadata; other
    labels remain operator-free-form.

3. **Per-worker Gitea identities, not shared `eden` user.**
   Each worker registered via 12a-1's `register_worker` gets a
   corresponding Gitea user provisioned by the new
   `eden_gitea` admin-bootstrap helper. The Gitea username is
   derived from the EDEN `worker_id` per §3.1.2's mapping
   rule. Each user gets a per-worker scoped access token
   (Gitea's "scoped tokens" feature lets us limit a token to
   `write:repository` coarse-scope + team-bound to a
   single repo via §3.0.4 — fine-grained isolation comes
   from team membership + branch protection, not the
   token scope itself; see §3.0a's note + §8.2). Rejected: keeping
   the shared `eden` user with finer-grained scope (Gitea's
   per-token scoping helps, but a shared identity defeats
   the audit-log + per-worker-rotation properties);
   reusing the EDEN bearer-token directly as the Gitea token
   (Gitea has its own auth model + the wire protocols don't
   intersect; cross-binding the credentials adds operational
   coupling for no benefit). See §3.1.

4. **Three branch-protection rules mirror chapter 6 §1.**
   - `main`: write-protected, no exceptions in the running
     experiment. Pushed only at experiment-create time by
     the bootstrap helper using the admin token; after that,
     `BLOCK_ON_REJECTED_REVIEWS=true` + push allow-list of
     `[]` (empty list = nobody).
   - `variant/*`: writeable only by the integrator's Gitea
     user (push allow-list = `[<integrator-worker-id>]`).
     The push-allow-list is dynamic — the chart's
     post-install Job can populate it after the integrator's
     Gitea user is registered. When the chart deploys
     multiple orchestrator replicas (per 13a §6 default
     `replicas.orchestrator=2`), each integrator replica gets
     its own Gitea user; the allow-list contains all of them.
   - `work/*`: writeable by any registered executor or
     human-executor user (push allow-list = members of a
     Gitea team `eden-executors`). The team is populated by
     the same bootstrap helper.
   See §3.2. Rejected: enforcing branch protection only on
   `main` and `variant/*` and leaving `work/*` open (would
   miss the pitfall where a misconfigured ideator pushes a
   commit directly to `work/<id>-<slug>`); enforcing via
   Gitea pre-receive hooks instead of branch protection
   (more flexible but more code; the branch-protection API
   covers our four rules cleanly).

5. **Native PRs are opt-in, NOT a default.** Default
   posture: `gitea.pr.integratorApproval.enabled=false`,
   `gitea.pr.humanExecutor.enabled=false`. Programmatic
   integrator + scripted executor stay direct-push, gated by
   the §3.2 ACLs. Why: the chapter-6 §3.4 atomic-three-step
   contract is satisfied trivially by direct push; PRs add a
   review/merge surface that's only worth the operational
   cost when (a) the operator wants human approval before
   integration, OR (b) the operator's executor is a human at
   a keyboard. Neither applies to the default
   helm-smoke-style automated experiment. See §3.3.

6. **OAuth/OIDC is for operator-facing logins only;
   programmatic worker auth is always per-worker scoped
   tokens.** The Gitea web UI (operator browses repos /
   reviews PRs / inspects history) authenticates via the
   operator-configured OAuth provider when
   `gitea.auth.oauth.enabled=true`; the EDEN worker hosts
   continue authenticating via per-worker scoped tokens
   regardless. Why: SSO works well for human logins (browser
   redirect flow) but is awkward for programmatic clients
   (would require the client to maintain a refresh token).
   Per-worker tokens align cleanly with 12a-1's
   `reissue_credential` flow + the §3.6 rotation
   alignment. See §3.4.

7. **Operator-required values; no fictional defaults.**
   When `gitea.auth.oauth.enabled=true`,
   `gitea.auth.oauth.issuerUrl` MUST be non-empty AND either
   `gitea.auth.oauth.clientSecretKey` references a non-empty
   key in `gitea.auth.oauth.existingSecret`, OR an inline
   secret is supplied via the chart-managed Secret. Same
   posture for `gitea.users.adminExistingSecret` (the admin
   token used by the bootstrap helper).
   `values.schema.json`'s `if/then` clauses enforce. Same
   13a/13c/13d posture per the AGENTS.md substrate-plan
   pitfall. See §3.5.

8. **Gitea worker tokens are derived secrets of the EDEN
   worker credential.** When EDEN's
   `reissue_credential(worker_id)` mints a new task-store
   credential, the same operation triggers the
   `eden_gitea` helper to revoke the old Gitea token and
   mint a new one for that user. The new Gitea token is
   propagated to the worker host via the same chart-managed
   Secret update (operators pulling secrets via External
   Secrets Operator or Sealed Secrets see the rotation as
   a single Secret update). Operators MAY rotate the Gitea
   token independently (out-of-band via
   `eden-gitea-admin rotate-token`), but that's documented
   as a recovery path, not a routine flow. See §3.6.
   Rejected: making the Gitea token a fully-independent
   secret (operationally cleaner per provider but loses the
   "rotate once, both rotate" property the
   operator-experience benefits from); making the Gitea
   token literally equal to the EDEN bearer (cross-binding
   couples the two auth surfaces' security models, which
   are independently scoped today).

9. **Compose stays on the default shared-password path;
   13e adds opt-in overlays.** Per the same posture 13d
   took for blob backends: forcing OAuth or per-worker
   identities on Compose dev breaks the local-iteration
   flow. The new `compose.gitea-acl.yaml` overlay adds
   branch protection (which works fine without per-worker
   identities — it just allow-lists the shared `eden` user
   on `variant/*` etc.); the new `compose.gitea-oauth.yaml`
   overlay adds OAuth using a static-config OAuth provider
   (e.g., a co-located dex container for dev, or the
   GitHub OAuth app the operator already has). Both are
   opt-in via `docker compose -f compose.yaml -f
   compose.gitea-acl.yaml ...`. See §3.7.

10. **Migration is documented, not scripted, with an
   exception for the `eden-gitea-admin migrate` command.**
   Per the 13c/13d substrate-plan pattern: a migration
   runbook covers the rollover from shared `eden` user to
   per-worker users. The shape is: register every existing
   worker via `eden-gitea-admin provision-user` (driven by
   `Store.list_workers()` output); rotate each worker's
   credential to pick up the new Gitea token; provision
   branch protection. The `eden-gitea-admin migrate`
   command bundles those steps for a typical happy path.
   Operators with non-typical setups (custom Gitea orgs,
   external user databases) follow the manual runbook.
   See §3.8.

11. **CI: `helm-smoke-gitea-oauth` uses a static-config
   OAuth provider, not real GitHub/Google.** Real-provider
   tokens in CI are a flaky-network + secret-management
   problem. A static-config provider (the
   [`oauth2-mock-server`](https://www.npmjs.com/package/oauth2-mock-server)
   project, MIT-licensed, runs as a single Node container)
   speaks the standard OIDC flow at a deterministic
   endpoint and lets us test Gitea's OAuth integration
   end-to-end without external dependencies. See §3.9.

12. **Conformance impact is none.** Gitea-side ACLs are
    a deployment-substrate concern; chapter 9 §6 says
    conformance is via the chapter-7 HTTP binding, which
    Gitea isn't part of. The conformance suite continues
    passing across all four substrate variants
    (Compose-default + Compose-with-overlays +
    Helm-default + Helm-with-OAuth). See §3.10.

## 3. Design

### 3.0 Gitea organization topology

#### 3.0.1 Why an organization (not a user-owned repo)

Per Decision 0: branch protection in Gitea supports
`push_whitelist_teams` ONLY for repositories owned by a
Gitea organization. User-owned repositories
(the 10d-followup-B baseline shape, with the `eden` user
owning every experiment's repo) can only allow-list
specific usernames via `push_whitelist_usernames` — they
can't reference a team. The §3.2 ACL design uses teams
(`eden-<experiment_id>-integrator`,
`eden-<experiment_id>-executor`,
`eden-<experiment_id>-human-executor`) so the allow-list
stays correct as workers are added or rotated; with the
username-only form, every membership change would require
re-PATCH'ing every branch-protection rule.

#### 3.0.2 The `eden` organization

13e introduces a per-deployment Gitea organization with
the literal name `eden`. One org per chart release; one
chart release per k8s namespace per Gitea instance (no
cross-deployment collisions because each chart release
runs its own Gitea instance per the 13a
gitea-StatefulSet posture).

The org is created at chart-install time by the new
bootstrap Job `gitea-org-provision-job.yaml.tmpl` (under
`bootstrap/`, out-of-tree per the 13a §3.5 + 13c §3.8
substrate-plan posture). The Job runs:

```bash
curl -X POST \
  -H "Authorization: token ${GITEA_ADMIN_TOKEN}" \
  -H "Content-Type: application/json" \
  http://eden-gitea:3000/api/v1/orgs \
  -d '{"username": "eden", "visibility": "private"}'
# 201 Created on first run; 422 Unprocessable Entity
# on already-exists — both are success exits per the
# helper's idempotency rule.
```

The Job is gated on `gitea.org.bootstrap=true` (default
true). When false, the chart assumes the operator has
pre-created the org out-of-band (e.g., via a federated
external Gitea instance).

#### 3.0.3 Per-experiment repos under the org

Per-experiment repos move from `eden/<experiment_id>`
(user-owned in 10d-fb) to `eden/<experiment_id>`
(org-owned in 13e). The path string is identical — the
ownership type changes underneath. Workers' clone URLs
do not change (the `eden/<experiment_id>` path resolves
to org-owned in 13e).

Repo creation:

```bash
curl -X POST \
  -H "Authorization: token ${GITEA_ADMIN_TOKEN}" \
  -H "Content-Type: application/json" \
  http://eden-gitea:3000/api/v1/orgs/eden/repos \
  -d '{"name": "<experiment_id>", "private": true, "auto_init": false}'
```

Replaces the existing
`POST /api/v1/user/repos` in
[`setup-experiment.sh`](../../reference/scripts/setup-experiment/setup-experiment.sh)
line 382 with the org-scoped variant.

#### 3.0.4 Teams and their lifecycle

For each registered worker role per experiment, the
bootstrap helper creates a corresponding Gitea team
under the `eden` org:

- `eden-<experiment_id>-executor`
- `eden-<experiment_id>-human-executor`
- `eden-<experiment_id>-evaluator`
- `eden-<experiment_id>-human-evaluator`
- `eden-<experiment_id>-ideator`
- `eden-<experiment_id>-human-ideator`
- `eden-<experiment_id>-integrator`
- `eden-<experiment_id>-operator`
- `eden-<experiment_id>-integrator-approver` (only
  rendered when `gitea.pr.integratorApproval.enabled=true`)

Teams are created lazily (only the roles for which at
least one worker has been registered) via
`POST /api/v1/orgs/eden/teams`. Each team is granted
`Read` repository access by default; the `executor`,
`human-executor`, and `integrator` teams are additionally
granted `Write` via team-repository assignment
(`PUT /api/v1/teams/{id}/repos/{owner}/{repo}`). The
team-level access is the coarse gate; the §3.2
branch-protection rules are the fine gate that picks
which branches each team can push.

Teams are NOT shared across experiments — each
experiment gets its own team set so that
`eden-exp-1-executor`'s allow-list rights on
`eden/exp-2` are nonexistent by construction.

#### 3.0.5 Migration: existing `eden` USER → new `eden` ORG

10d-fb-era deployments have an `eden` USER owning
existing repos. 13e's migration runbook (§3.8.2)
covers the rename:

1. Bootstrap creates the new `eden` ORG.
2. For each pre-existing repo `eden/<experiment_id>`
   (user-owned), the runbook transfers ownership to the
   org via `POST /api/v1/repos/{owner}/{repo}/transfer`
   with `new_owner: "eden"`. Gitea handles this without
   downtime; clone URLs are preserved (the
   `eden/<experiment_id>` path resolves identically
   post-transfer).
3. The old `eden` USER (now distinct from the `eden`
   ORG — Gitea allows the same login name for a user and
   an org? Actually it does NOT — Gitea's username
   uniqueness covers both users and organizations). The
   pre-existing `eden` USER MUST be renamed BEFORE the
   org is created. The runbook walks through:
   `eden-gitea-admin rename-user eden eden-legacy`
   (calls `PATCH /api/v1/admin/users/eden` with the new
   name), then bootstrap creates the `eden` org. After
   the migration soak, the operator deletes the
   `eden-legacy` user via
   `DELETE /api/v1/admin/users/eden-legacy`.

The runbook + the chart's NOTES.txt flag this rename
step. Operators on fresh installs (no prior `eden`
user) skip step 3.

#### 3.0.6 Why one `eden` org per deployment, not one per experiment

Per-experiment orgs would isolate experiments at the org
level + simplify cleanup (delete the org = delete every
team + repo + member-binding in one shot). But:

- The chart already runs its own Gitea instance per
  deployment, so experiments are already isolated at the
  Gitea-instance level.
- One org per experiment means N orgs per deployment.
  Gitea handles this fine, but the operator-facing
  surface (UI navigation, audit-log filtering) gets
  noisier.
- The per-experiment team naming
  (`eden-<experiment_id>-*`) keeps team-level isolation
  within the shared org.

Rejected. Documented as a future amendment in §11 in
case a future deployment needs harder per-experiment
isolation (e.g., shared Gitea across multiple charts —
which is itself a §4.4 non-goal).

### 3.0a Worker-role label convention

#### 3.0a.1 The `eden.role` label

Per Decision 0a: 12a-1's `Worker.labels` field is the
canonical source for the worker's role. 13e reserves the
`eden.role` label key. Operators set it at
`register_worker` time:

```python
store.register_worker(
    worker_id="exec-1",
    labels={"eden.role": "executor"},
)
```

The `eden-gitea-admin provision-user` helper reads
`worker.labels["eden.role"]` to determine:

- The Gitea team membership (per §3.0.4).
- The token scope (per §3.1.1).
- Whether to provision at all (workers with
  `eden.role=none` or no `eden.role` label are skipped).

Legal values for `eden.role`:

| Value | Team membership | Token scope |
|---|---|---|
| `executor` | `eden-<experiment_id>-executor` | `write:repository` |
| `human-executor` | `eden-<experiment_id>-human-executor` | `write:repository` |
| `evaluator` | `eden-<experiment_id>-evaluator` | `read:repository` |
| `human-evaluator` | `eden-<experiment_id>-human-evaluator` | `read:repository` |
| `ideator` | `eden-<experiment_id>-ideator` | `read:repository` |
| `human-ideator` | `eden-<experiment_id>-human-ideator` | `read:repository` |
| `integrator` | `eden-<experiment_id>-integrator` | `write:repository` |
| `operator` | `eden-<experiment_id>-operator` | `write:repository,read:organization,write:organization` |
| `none` | n/a (skip Gitea provisioning) | n/a |

**Token scopes are NOT per-repository.** Per Gitea's
documented scope vocabulary
([API Usage — Scoped tokens](https://docs.gitea.com/development/api-usage)),
scopes are coarse role-level grants like
`write:repository`, `read:repository`, `read:organization` —
they DO NOT support a per-repo gate. The fine-grained
isolation comes from (a) the user's TEAM membership
(per §3.0.4: a user is only a member of the
`eden-<experiment_id>-<role>` team, which is bound to
exactly that experiment's repo) AND (b) the §3.2
branch-protection rules. The token scope is the coarse
"this token can write to any repo this user has team
access to"; the team membership is what restricts WHICH
repo. Defense in depth comes from the §3.2 branch
allow-lists rejecting wrong-branch pushes even if the
user is a team member.

Values outside this set are rejected at provisioning
time with a clear error message; the runbook documents
the closed set.

#### 3.0a.2 Where the `eden.role` label gets set

For Helm deployments: `setup-experiment-helm.sh` writes
`eden.role` per-worker as part of its
`register_worker` invocation. The chart's values
expose the role mapping per-service-type:

- `replicas.executorHost` worker instances get
  `eden.role=executor`.
- `replicas.evaluatorHost` instances get
  `eden.role=evaluator`.
- `replicas.ideatorHost` instances get
  `eden.role=ideator`.
- `replicas.orchestrator` instances get
  `eden.role=integrator` (because the orchestrator IS
  the integrator per 12a-2's "orchestrator as role"
  forecast; if a future amendment splits them, the
  chart values gain a separate
  `replicas.integratorHost`).
- `replicas.webUi` instances get
  `eden.role=operator` (admin-level access for the
  web-ui's executor / evaluator route plumbing).

For Compose deployments + `compose.gitea-perworker.yaml`
overlay: `setup-experiment.sh` similarly maps each
service to its role and writes the label at
`register_worker` time.

For runtime worker-add (via 12c's control plane): the
operator passes `--label eden.role=<value>` to the
worker-host CLI; the worker's first
`register_worker` call includes the label. The 12c
post-commit hook reads it.

#### 3.0a.3 12a-1 label-namespace reservation

12a-1's existing posture is that `Worker.labels` is
free-form operator metadata. 13e doesn't change that
generic posture — it only reserves the `eden.*`
prefix for protocol-binding-defined keys (the same
mechanism 13a Decision 8 used for `eden.*` k8s labels
on Job resources, and 13b uses for `eden.task_id` /
`eden.role` k8s labels on Job pods).

The reservation is informative on the 12a-1 spec
side (a future v1 amendment might pin this formally);
on the implementation side, the
`eden_storage.WorkerLabels` type-check accepts any
keys but the documentation flags `eden.*` as reserved.
`register_worker` does NOT reject `eden.*` labels with
unknown sub-keys (forward-compat with future
binding-defined labels); `eden-gitea-admin` silently
ignores unknown `eden.*` keys.

#### 3.0a.4 New wire op: `update_worker_labels`

12a-1 only exposes `register_worker(worker_id, labels?)`
for writing the labels field — `labels` are set
idempotently at registration time but there's no mutation
path. 13e's migration runbook (§3.8.2) needs to backfill
`eden.role` on existing worker records that were
registered without it. To avoid the workaround of
"de-register + re-register" (which would invalidate the
worker's existing credential per the 12a-1
idempotency-on-existing-record semantics), 13e adds a
minimal new op:

```python
# eden_storage.Store
def update_worker_labels(
    self,
    worker_id: str,
    labels: dict[str, str],
    *,
    mode: Literal["merge", "replace"] = "merge",
) -> Worker:
    """Update an existing worker's labels.

    ``merge`` (default): existing labels survive; the
    supplied keys overwrite or add. ``replace``: the
    supplied dict becomes the new labels in full.

    Raises NotFound if no worker matches. Raises
    InvalidPrecondition if an `eden.*` key is set to a
    type-invalid value (the only enforced rule today;
    `eden.role` must be one of §3.0a.1's legal values).
    """
```

Wire shape: `POST /v0/experiments/{E}/workers/{W}/labels`
with body `{"labels": {...}, "mode": "merge"}`. Returns
the updated `Worker` record. Authorization: requires the
admin credential (per 12a-1 §"RBAC on the RBAC" — labels
are operator-controlled, not worker-controlled).

This is the SINGLE wire-surface addition 13e makes;
everything else is deployment-substrate work. The op is
non-breaking (clients that don't call it keep working);
conformance scenarios are unaffected (existing scenarios
exercise observable behavior of registered workers, not
the mutation surface).

The `eden-gitea-admin backfill-labels` command (§3.8.2)
calls `update_worker_labels` with
`mode="merge"`. Operators who want a runtime label-
rewrite (e.g., re-roleing an existing worker) also call
the op directly via the wire.

### 3.1 Per-worker Gitea identities

#### 3.1.1 Provisioning model

Each `register_worker(worker_id, labels={"eden.role": ...})`
call produces an EDEN worker record. The new `eden_gitea`
helper module ships an `eden-gitea-admin provision-user`
CLI that takes a worker record (looked up via the
task-store wire) and provisions the corresponding Gitea
identity:

```bash
eden-gitea-admin provision-user \
  --task-store-url http://task-store-server:8080 \
  --eden-token-env EDEN_SHARED_TOKEN \
  --gitea-url http://gitea:3000 \
  --admin-token-env GITEA_ADMIN_TOKEN \
  --worker-id <worker_id> \
  --experiment-id <experiment_id>
```

The CLI does NOT take a `--role` flag — the role is read
from the worker record's `eden.role` label per §3.0a. A
worker without the label OR with `eden.role=none` is
skipped (the CLI exits 0 with a "skipped: no eden.role
label" message, NOT an error — same shape as a no-op
re-run). Workers with an unknown `eden.role` value exit
non-zero with an `unknown role <bad-value>; legal values:
{...}` message.

The command:

1. POSTs `/api/v1/admin/users` with username derived from
   `worker_id` per §3.1.2's mapping. The Gitea password is
   set to a freshly-generated 32-byte random string (the
   user-facing password is irrelevant — the worker
   authenticates via token, not password; the password just
   has to be syntactically valid + unguessable so it can't
   be used as an alternate auth path). Per Gitea's API
   ([API Usage docs](https://docs.gitea.com/development/api-usage)),
   admin user-creation is the supported path under
   `BasicAuth+sudo` via the admin token.
2. Mints a scoped access token on behalf of the user.
   Gitea's API endpoint `POST /api/v1/users/{username}/tokens`
   requires `BasicAuth` with the **target user's**
   password — admin tokens are NOT accepted there
   ([API Usage](https://docs.gitea.com/development/api-usage)).
   Two paths are viable; 13e picks the CLI path:

   - **Chosen — CLI inside the Gitea pod via remote
     exec.** Run
     `gitea admin user generate-access-token
     --username <user> --scopes "<scopes>"
     --token-name "<name>"` inside the Gitea
     container. The CLI subcommand runs as the
     in-process Gitea admin against the local DB —
     no API authentication needed
     ([Gitea command-line — admin user generate-access-token](https://docs.gitea.com/administration/command-line)).
     The CLI prints the token to stdout; the
     bootstrap Job captures stdout and writes the value
     to the chart-managed Secret. The exec mechanism
     is selected via `--exec-backend
     {kubernetes-api,docker-cli}` on `eden-gitea-admin`:
     - **kubernetes-api** (chart-managed bootstrap Jobs
       running inside the cluster): the Python
       `kubernetes>=29.0.0` client's
       `kubernetes.stream.stream(...)` exec API hits
       the API server directly; no `kubectl` binary
       in the bootstrap image required. The Job's
       ServiceAccount needs `pods/exec:create` on the
       Gitea pod (label-selected; scoped to the
       release namespace via a `Role`, per §5.3).
     - **docker-cli** (Compose's `setup-experiment.sh`
       AND operator-side workstation invocations
       against a Compose deployment): runs
       `docker exec -i eden-gitea ...`.
   - *Rejected — BasicAuth-as-user.* Would require the
     bootstrap Job to know each user's password, which
     it does (we just generated it in step 1 with
     `--must-change-password false`). Functional but
     means leaking the password through one more
     channel; the CLI path keeps the password local to
     the Gitea container.
   - *Rejected — Sudo header.* Gitea's `Sudo` header
     lets admins act as other users for some endpoints,
     but the token-mint endpoint specifically does not
     accept it (the docs flag this endpoint as
     "BasicAuth only").

   The token's scopes depend on the worker's
   `eden.role` label per §3.0a's table:
   - `executor` / `human-executor`:
     `write:repository` (coarse — applied to the user;
     the user is bound to ONE repo via team membership
     per §3.0.4, so this effectively grants write to
     `eden/<experiment_id>` only).
   - `evaluator` / `human-evaluator`:
     `read:repository` (the evaluator reads the worker
     branch but never writes; the chapter-3 §4.1
     "evaluator MUST NOT modify the worker branch" rule
     is enforced by team-level read-only + branch
     protection + read-only token; defense in depth).
   - `ideator` / `human-ideator`:
     `read:repository` (ideator produces ideas, not git
     commits — read-only access is for inspecting
     parent_commits etc.).
   - `integrator`: `write:repository`. The branch-
     protection rules (§3.2) restrict WHICH branches the
     integrator can push; the token scope is coarse.
   - `operator`:
     `write:repository,read:organization,write:organization`
     (operator manages teams + their repo bindings;
     needs org-admin scope for membership ops). Used
     only by the bootstrap + reclamation paths.
3. Adds the user to the Gitea team
   `eden-<experiment_id>-<role>` (one team per role per
   experiment); the team membership is what the
   branch-protection allow-list references (§3.2).
4. Stores the token in a chart-managed Secret named
   `eden-<release>-gitea-tokens` under the key
   `<worker_id>.token` (or in
   `gitea.users.workerTokenStore=externalSecret` mode,
   updates an external-Secrets-managed Secret).

The command is **idempotent**: if the user already exists,
the existing user is reused; if a token with the same
description already exists, it's revoked and a fresh one
minted (the chart-managed Secret is updated).

#### 3.1.2 worker_id → Gitea username mapping

12a-1 grammar: `worker_id` matches
`^[a-z0-9][a-z0-9_-]{0,63}$` (lowercase alphanumeric +
hyphen + underscore; first char non-hyphen; ≤ 64 chars;
plus a reserved set `{admin, system, internal, _*}`).

Gitea usernames: per Gitea's `validation.AllowDotsInUsernames`
config and the default user-validation rules
([gitea/services/auth/source/db/user.go#L41](https://github.com/go-gitea/gitea/blob/main/services/auth/source/db/user.go)),
usernames must match `^[a-zA-Z0-9_.-]{1,40}$` AND not start
with `-` or `.`. So Gitea allows `_` and `.` (which EDEN's
worker_id grammar excludes); EDEN's worker_id allows `_` and
`-` (which Gitea allows). The intersection is **trivial**:
any `worker_id` is a valid Gitea username (modulo the 64 vs
40 length cap).

Mapping rule: when `len(worker_id) <= 40`, Gitea username =
`worker_id` literal. When `len(worker_id) > 40`, the helper
truncates to 32 chars + `-` + 7-char content-hash:
`<worker_id[:32]>-<hash[:7]>`. Truncation keeps the
human-readable prefix; the hash suffix prevents collisions
between two long worker_ids that share a 40-char prefix.

The reserved EDEN worker_ids `{admin, system, internal, _*}`
are never used as worker identities (per 12a-1 §D.1), so the
mapping never has to account for them. Gitea's own reserved
usernames (`admin`, `assets`, `api`, etc.) intersect with
EDEN's reserved set on `admin`; for the bootstrap-helper's
own admin token, the chart uses a fixed username
`eden-bootstrap` (NOT `admin` and NOT `eden-admin` — both
are Gitea-reserved or close to reserved).

#### 3.1.3 Coordination with 12a-1

12a-1's `register_worker` is the trigger; the EDEN admin
calls `eden-gitea-admin provision-user` immediately after
each successful `register_worker` call. For the bootstrap
flow, `setup-experiment-helm.sh` (and `setup-experiment.sh`
when the Compose `compose.gitea-acl.yaml` overlay is
active) runs both calls in sequence. For runtime
worker-add operations (via the 12c control plane), the
control plane's `register_worker` handler fires a
post-commit hook that calls `eden-gitea-admin
provision-user`; failure to provision the Gitea user is
**non-blocking** for the EDEN registration (the worker's
EDEN record exists; the Gitea side will be provisioned by
the next bootstrap-script reconciliation). See §8.4.

### 3.2 Per-branch ACLs

Three branch-protection rules per repo, provisioned by
`eden-gitea-admin provision-branch-protection`:

#### 3.2.1 `main`

```json
{
  "branch_name": "main",
  "rule_name": "eden-main-protection",
  "enable_push": true,
  "enable_push_whitelist": true,
  "push_whitelist_usernames": [],
  "push_whitelist_teams": [],
  "push_whitelist_deploy_keys": false,
  "enable_force_push": false,
  "enable_status_check": false
}
```

Empty `push_whitelist_usernames` = nobody can push to
`main` once the rule is in place. The bootstrap helper's
admin token bypasses the rule at experiment-create time
(Gitea's branch protection allows site-admins to bypass
when configured — `gitea.acl.allowAdminBypass` defaults
to true for the bootstrap path).

#### 3.2.2 `variant/*`

```json
{
  "branch_name": "variant/*",
  "rule_name": "eden-variant-protection",
  "enable_push": true,
  "enable_push_whitelist": true,
  "push_whitelist_usernames": [],
  "push_whitelist_teams": ["eden-<experiment_id>-integrator"],
  "enable_force_push": false,
  "enable_status_check": false
}
```

`push_whitelist_teams` references the Gitea team that holds
the integrator workers. When the chart deploys multiple
orchestrator replicas (per 13a), each replica's integrator
identity is added to the same team.

#### 3.2.3 `work/*`

```json
{
  "branch_name": "work/*",
  "rule_name": "eden-work-protection",
  "enable_push": true,
  "enable_push_whitelist": true,
  "push_whitelist_usernames": [],
  "push_whitelist_teams": [
    "eden-<experiment_id>-executor",
    "eden-<experiment_id>-human-executor"
  ],
  "enable_force_push": false,
  "enable_status_check": false
}
```

Both scripted-executor and human-executor teams are
allow-listed. The integrator team is NOT — the integrator
reads `work/*` but its `variant/*` write goes through the
`variant-protection` rule.

#### 3.2.4 ACL mode: `enforce` vs `advisory-only`

`gitea.acl.mode = enforce` (default): branch protection is
enforced; a misconfigured push is rejected by Gitea with
HTTP 403. `gitea.acl.mode = advisory-only`: the rules are
provisioned with `enable_push_whitelist = true` but
`enable_push = false` — a misconfigured push triggers a
Gitea webhook notification (which the operator monitors)
but is not rejected. Used during the §3.8 migration window
when an operator is rolling out per-worker identities and
wants to catch misconfigurations before they harden the
ACLs. After the migration soak, the operator switches to
`enforce`.

### 3.3 Native PR review

#### 3.3.1 Default: no PRs

`gitea.pr.integratorApproval.enabled=false` and
`gitea.pr.humanExecutor.enabled=false` is the default. With
both off, programmatic submissions and integrator merges
go direct-push as in 13a. The §3.2 branch-protection rules
gate which workers can write where; PRs add no value.

#### 3.3.2 Opt-in: integrator-approval PR

When `gitea.pr.integratorApproval.enabled=true`, the
integrator's `Integrator.integrate(variant_id)` flow
changes:

1. (Existing) Build the squash commit per chapter-6 §3.2.
2. (Existing) Push to a temp branch `_eden_internal/pending-integrate/<variant_id>`
   (a NEW namespace 13e introduces, write-protected to the
   integrator team — same allow-list as `variant/*`).
3. (New) `POST /api/v1/repos/{owner}/{repo}/pulls` to open
   a PR from the temp branch to `variant/<variant_id>-<slug>`.
4. (New) Wait up to `gitea.pr.integratorApproval.timeoutSeconds`
   (default 300) for the PR to be approved by a configured
   reviewer team `eden-<experiment_id>-integrator-approver`.
   The operator pre-populates this team with human approvers
   (could be the same as integrator if the operator wants
   the integrator to self-approve via a separate human
   identity).
5. (New) On approval: `POST /api/v1/repos/{owner}/{repo}/pulls/{n}/merge`
   with `Do=squash` (the PR's merge IS the chapter-6 §3.4
   atomic write into `variant/*`). The integrator polls
   `GET /api/v1/repos/{owner}/{repo}/pulls/{n}` to
   confirm `merged=true` AND captures the resulting
   `merge_commit_sha`. The Store's `integrate_variant`
   call records the same SHA in `variant.variant_commit_sha`
   AND emits the `variant.integrated` event — this is the
   chapter-6 §3.4 atomic three-step (PR merge = ref write
   on Gitea; field write + event emit happen atomically in
   the Store). Failure of the PR merge (timeout, reject)
   triggers the existing chapter-6 §3.4 rollback ladder
   — delete the temp branch, do NOT emit the
   `variant.integrated` event.
6. (New) Delete the temp branch via
   `DELETE /api/v1/repos/{owner}/{repo}/branches/{branch}`.

Atomicity-on-the-wire is preserved: a subscriber observes
the `variant.integrated` event iff the `variant/*` ref +
`variant.variant_commit_sha` field both landed (per
chapter-6 §3.4). The PR merge is just the mechanism by
which the integrator obtains operator approval before
performing the canonical write; from the protocol's
perspective, the visible behavior is identical to
direct-push integration.

The integrator's claim TTL must accommodate the PR-approval
timeout: the chart's
`config.integrator.claimTtlSeconds` (forecast by 12c)
defaults to 600 when
`gitea.pr.integratorApproval.enabled=true` AND
`gitea.pr.integratorApproval.timeoutSeconds=300`,
giving a 2× buffer for the approval window.

#### 3.3.3 Opt-in: human-executor PR

When `gitea.pr.humanExecutor.enabled=true`, the
`work/*`-write flow for human-executor identities changes:
the human's local push to `work/<variant_id>-<slug>`
opens (via Gitea web UI or the operator's git client) a PR
from `work/...` to a per-experiment review branch
`_eden_internal/work-staging`. A reviewer (typically an
human-evaluator) approves the PR; on merge, the commit
lands on `_eden_internal/work-staging` AND the executor
host watches `work-staging` for new commits and re-creates
the `work/<branch>` ref pointing at the merge commit. The
executor then proceeds to its normal chapter-3 §3.3 submit
(`Store.submit` with the now-canonical commit_sha).

This path is opt-in because most deployments don't have
human executors. It does not affect any other role.

The implementation lives in the human-executor's Web-UI
flow (chunk 9c of Phase 9, forecast-extended here). When
`gitea.pr.humanExecutor.enabled=false`, the human-executor
path is the existing direct-push-to-`work/*` flow.

#### 3.3.4 PR review never applies to scripted submissions

Scripted ideator / executor / evaluator (per 12a-2's
default `--mode scripted` worker hosts) never use PRs.
Their submissions are programmatic and the review surface
adds no value. The chart's `values.schema.json` does NOT
allow `gitea.pr.humanExecutor.enabled=true` in
combination with a scripted-only deployment (i.e., when
no human-executor users have been registered) — it's
documented in NOTES.txt that turning the flag on without
a registered human-executor team produces dead PRs. (The
schema can't directly enforce "at least one
human-executor user exists" because that's runtime state;
NOTES.txt is the right place for the warning.)

### 3.4 OAuth/OIDC SSO

#### 3.4.1 Configuration shape

```yaml
gitea:
  auth:
    mode: "basic"            # ∈ {basic, oauth}

    # Basic mode: 13a's HTTP-Basic-with-shared-eden-user
    # baseline. Mostly retained for the migration window.
    basic:
      adminUsernameKey: "GITEA_ADMIN_USERNAME"
      adminPasswordKey: "GITEA_ADMIN_PASSWORD"

    # OAuth mode: SSO for operator-facing logins.
    oauth:
      provider: ""           # required when mode=oauth (e.g., "github", "gitlab", "google", "openidConnect")
      issuerUrl: ""          # required when provider=openidConnect; ignored otherwise
      existingSecret: ""     # references a Secret with clientId + clientSecret keys
      clientIdKey: "OAUTH_CLIENT_ID"
      clientSecretKey: "OAUTH_CLIENT_SECRET"
      # Per Gitea config: the OAuth login button label.
      providerName: "EDEN SSO"
      # Per-deployment redirect URL — the chart computes it
      # from values.ingress.hosts.gitea + the standard
      # /user/oauth2/<providerName>/callback path.
      autoRegister: true     # auto-create Gitea users for new OAuth logins
```

#### 3.4.2 Gitea env-var wiring

When `auth.mode=oauth`, the Gitea StatefulSet's pod template
gets:

```yaml
env:
  - name: GITEA__oauth2_client__ENABLE_AUTO_REGISTRATION
    value: "{{ .Values.gitea.auth.oauth.autoRegister | toString }}"
  - name: OAUTH_CLIENT_ID
    valueFrom:
      secretKeyRef:
        name: {{ .Values.gitea.auth.oauth.existingSecret }}
        key: {{ .Values.gitea.auth.oauth.clientIdKey }}
  - name: OAUTH_CLIENT_SECRET
    valueFrom:
      secretKeyRef:
        name: {{ .Values.gitea.auth.oauth.existingSecret }}
        key: {{ .Values.gitea.auth.oauth.clientSecretKey }}
```

The OAuth provider configuration itself is provisioned via
Gitea's CLI (`gitea admin auth add-oauth ...`) at
chart-install time, run by a one-shot Job under
`bootstrap/gitea-oauth-config-job.yaml.tmpl` (out-of-tree,
following the 13a §3.5 + 13c §3.8 substrate-plan posture
for per-experiment / per-deployment artifacts that don't
belong under `templates/`).

#### 3.4.3 Programmatic worker auth is unchanged

Per Decision 4: worker hosts authenticate to Gitea via
per-worker scoped tokens regardless of
`gitea.auth.mode`. The OAuth toggle affects only the Gitea
web UI's user-facing login flow.

### 3.5 Operator-required values + schema enforcement

#### 3.5.1 Schema clauses

```json
{
  "if": { "properties": { "gitea": { "properties": { "auth": { "properties": { "mode": { "const": "oauth" } } } } } } },
  "then": {
    "properties": {
      "gitea": {
        "properties": {
          "auth": {
            "properties": {
              "oauth": {
                "required": ["provider", "existingSecret"],
                "properties": {
                  "provider":      { "type": "string", "minLength": 1, "enum": ["github", "gitlab", "google", "openidConnect"] },
                  "existingSecret":{ "type": "string", "minLength": 1 },
                  "clientIdKey":   { "type": "string", "minLength": 1 },
                  "clientSecretKey": { "type": "string", "minLength": 1 }
                }
              }
            }
          }
        }
      }
    }
  }

```

Plus a paired clause that requires `oauth.issuerUrl` to be
non-empty when `oauth.provider == "openidConnect"`. The
generic OIDC provider needs the issuer URL; the named
providers (github / gitlab / google) have hard-coded
endpoints in Gitea so the URL is unnecessary.

#### 3.5.2 Admin-token contract

`gitea.users.adminExistingSecret` MUST be non-empty when
the chart is in any mode that provisions users
(`gitea.acl.enabled=true` OR
`gitea.pr.integratorApproval.enabled=true` OR any
`auth.mode`). The Secret carries the bootstrap user's
admin token under `gitea.users.adminTokenKey` (default
`GITEA_ADMIN_TOKEN`). The `eden-gitea-admin` CLI reads
it via `--admin-token-env GITEA_ADMIN_TOKEN`. Same
posture 13c took for `EDEN_STORE_URL` (single-Secret-key
contract).

### 3.6 Credential-rotation alignment

#### 3.6.1 `reissue_credential` extended path

12a-1's `Store.reissue_credential(worker_id)` mints a new
EDEN bearer for the worker. After 13e, the
admin-bootstrap helper extends this with a Gitea-side
rotation:

1. Existing 12a-1 step: rotate the EDEN credential.
2. New 13e step: `eden-gitea-admin rotate-token
   --worker-id <worker_id>` revokes the worker's existing
   Gitea token (`DELETE /api/v1/users/{username}/tokens/{token-name}`,
   which IS admin-token-authenticated) and mints a new one
   via the remote-exec path from §3.1.1 (the `--exec-backend`-
   selected `kubernetes.stream.stream(...)` or
   `docker exec` invocation of
   `gitea admin user generate-access-token`; admin-token
   API path is not available for token mint per Gitea's
   BasicAuth-only requirement).
3. The new Gitea token is written to the chart-managed
   Secret under the same `<worker_id>.token` key.

   **Propagation to running workers.** Kubernetes Secret
   value updates DO NOT propagate to a pod's environment
   variables without a pod restart
   ([Distribute Credentials Securely](https://kubernetes.io/docs/tasks/inject-data-application/distribute-credentials-secure/)).
   13e therefore mounts the per-worker Gitea token as a
   FILE from a Secret VOLUME, not as an env var. Mounted
   Secret volume updates ARE propagated by kubelet to
   running containers within the kubelet's
   sync-frequency (defaults to ~60s; controllable via the
   pod-spec's
   `containers[].env[].valueFrom.secretKeyRef` vs
   `volumes[].secret` distinction). The credential-helper
   script reads the token from
   `/etc/eden/gitea-token` (file mount) on every git
   operation; a Secret update lands as a new file content
   within the kubelet sync window. The worker host's
   `--gitea-token-env <name>` CLI flag is renamed in 13e
   to `--gitea-token-file <path>` to match this file-
   mount posture; the credential-helper Python helper
   (§5.2's `credential_helper.py`) reads the file on
   every invocation.

   For workers that need IMMEDIATE rotation (no
   kubelet-sync delay), the operator can additionally
   trigger a pod restart via
   `kubectl rollout restart statefulset/<worker>` after
   the Secret update; the runbook documents this for the
   compliance-driven case. The default flow tolerates
   the ~60s sync window.

The Store-side hook is a post-commit listener wired into
the wire layer's `reissue_credential` handler (per 12a-1
§5's `register_worker` shape). The hook is **best-effort**:
if the Gitea-side rotation fails, the EDEN credential
rotation is still committed (the worker's task-store
auth is rotated). The operator is paged via a
`gitea_token_rotation_failed` event on the wire (per
chapter 5 §3 conventions); the runbook covers manual
recovery via `eden-gitea-admin rotate-token` invoked
out-of-band.

#### 3.6.2 Why best-effort, not transactional

A transactional `EDEN-rotate + Gitea-rotate or both
roll back` would require either two-phase commit across
the Store and Gitea (operationally heavy + Gitea has no
2PC support) OR a saga with compensating actions (revert
the EDEN rotation if Gitea fails). The cost outweighs the
benefit: a stuck-in-old-Gitea-token state means the worker
can still authenticate to the task-store with its new EDEN
credential AND still authenticate to Gitea with its old
token (which hasn't been revoked yet because the rotation
failed); from the worker's perspective, nothing breaks
visibly until the operator runs the manual rotation. The
window is bounded by operator response time; the runbook
flags the alert + the recovery shape.

Documented as a §11 future amendment: a job-queue-driven
saga that retries Gitea rotations until success. Not
load-bearing for v0.

### 3.7 Compose-side compatibility

Three opt-in overlays:

| Overlay | Effect |
|---|---|
| `compose.gitea-acl.yaml` | Provisions the three branch-protection rules from §3.2 against the existing shared `eden` user. The rules' allow-lists allow-list `eden` for both `variant/*` and `work/*` (so the existing shared-password path still works); the value is in being able to test the ACL mechanism end-to-end in dev. Operators who want per-worker identities in dev layer this on top of `compose.gitea-perworker.yaml`. |
| `compose.gitea-perworker.yaml` | Replaces the shared-password posture with per-worker users + scoped tokens. Requires `setup-experiment.sh` to provision N+1 users (N workers + the bootstrap admin); slower setup but matches the Helm posture exactly. Includes the `eden-gitea-admin` invocation. |
| `compose.gitea-oauth.yaml` | Adds OAuth wiring against an operator-supplied provider (typically GitHub or a co-located dex container). Compose's lack of a built-in Secret manager means inline values + the `.env` file; documented as dev-only. |

Default `docker compose up` continues to use the
shared-`eden`-password baseline. The compose-smoke jobs
do NOT switch to the new overlays in 13e — keeping the
helm-smoke-gitea-oauth job (§3.9) as the canonical
hardened-deployment smoke.

The compose `.env.example` file is documented + shipped
with empty placeholders for `GITEA_OAUTH_*` etc.

### 3.8 Migration runbook

`docs/deployment/migrating-to-gitea-hardening.md` covers
two paths.

#### 3.8.1 Path A: fresh deployment with Gitea hardening

Operator-side prerequisites: an OAuth provider (when
`auth.mode=oauth`), an admin-token Secret pre-created in
the cluster's namespace. Walk-through:

1. Set chart values:

   ```yaml
   gitea:
     auth:
       mode: oauth
       oauth:
         provider: github
         existingSecret: eden-gitea-oauth
     acl:
       enabled: true
       mode: enforce
     pr:
       integratorApproval:
         enabled: false
       humanExecutor:
         enabled: false
     users:
       adminExistingSecret: eden-gitea-admin
   ```

2. `helm install eden ./reference/helm/eden -f values.yaml`.
3. The chart's bootstrap `Job` runs `eden-gitea-admin
   provision-branch-protection` against the
   newly-deployed Gitea.
4. `setup-experiment-helm.sh` runs `register_worker` for
   each scripted-mode worker AND then runs
   `eden-gitea-admin provision-user` for each. The
   operator confirms via the Gitea web UI that the
   per-worker users + tokens exist + the branch-protection
   rules are visible.

#### 3.8.2 Path B: migrating from 13a's shared-`eden` baseline

Operator-side prerequisites: an existing 13a-style
deployment running with `gitea.auth.mode=basic` +
shared `eden` user. Walk-through:

1. **Drain artifact writers.** Same shape as 13c §3.8.2's
   step 1 (workers scale to 0 first, orchestrator-driven
   reclaim of stuck claims via web-ui admin path, then
   orchestrator + web-ui to 0). The Gitea pod stays up.
2. **Backfill `eden.role` labels on existing workers.** If
   the existing deployment registered workers without
   the `eden.role` label (the 12a-1 / pre-13e shape),
   the operator runs
   `eden-gitea-admin backfill-labels --from-pod-role` once:
   it iterates `Store.list_workers()`, infers the role
   from each worker's `worker_id` prefix (`executor-*` →
   `executor`, etc. per the chart's standard naming
   convention), and writes the label via the existing
   12a-1 `update_worker_labels` op. The runbook documents
   that operators with non-standard worker_id prefixes
   set the labels manually via
   `eden-gitea-admin set-role --worker-id ... --role ...`
   before continuing.
3. **Rename `eden` USER → `eden-legacy` and create the
   `eden` ORG.** Per §3.0.5: Gitea-side username
   uniqueness means the `eden` USER must vacate the name
   before the org can claim it. Run
   `eden-gitea-admin rename-user eden eden-legacy` then
   `eden-gitea-admin provision-org eden`. Pre-existing
   repos owned by the user are unaffected by the rename
   (Gitea preserves the repo's path under the new
   username temporarily); step 4 transfers ownership.
4. **Transfer repos to the new org.** For each
   per-experiment repo `eden-legacy/<experiment_id>`,
   run `eden-gitea-admin transfer-repo
   --from eden-legacy --to eden --repo <experiment_id>`.
   Clone URLs flip from `eden-legacy/<id>.git` to
   `eden/<id>.git`. Workers still use the path
   `eden/<experiment_id>` — the runbook's notes section
   walks the operator through updating
   `GITEA_REMOTE_URL` in `.env` (for Compose
   deployments using the per-worker overlay) or
   `values.yaml`'s
   `gitea.repo.path` (for Helm).
5. **Provision per-worker users + teams + branch
   protection.** With Gitea still up and the chart in
   `basic` mode, run `eden-gitea-admin migrate
   --from-shared-user eden-legacy --to-per-worker`
   against the experiment's repo. The command iterates
   `Store.list_workers()`, provisions a Gitea user per
   worker (per §3.1.1), creates per-role teams under the
   `eden` org (per §3.0.4), writes the per-worker token
   to the chart-managed Secret, and runs
   `provision-branch-protection` with
   `gitea.acl.mode=advisory-only` (so any pre-existing
   pushes don't get rejected during the migration).
6. **Verify advisory-only ACLs.** Watch the Gitea webhook
   notifications for any `would-have-rejected` events —
   operator confirms no legitimate-but-misconfigured
   push patterns exist. Soak for the operator's chosen
   window.
7. **Switch to enforce mode.** Update values:
   `gitea.acl.mode=enforce`. `helm upgrade`. Push
   attempts that were silently logged in advisory-only
   are now rejected.
8. **Switch to OAuth (optional).** Update values:
   `gitea.auth.mode=oauth` plus the oauth sub-block.
   `helm upgrade`. Operators sign in via the new SSO.
9. **Decommission `eden-legacy` user (optional).** After
   the migration soak, delete the renamed user via
   `eden-gitea-admin delete-user --username eden-legacy`.
   Skip this if the operator wants to retain the legacy
   identity for break-glass access (it has no team
   memberships in the new model so it can't push to any
   protected branch anyway).
10. **Bring workers back up.** Reverse-order from §3.8.2
    step 1.

Total drain window: bounded by step 1's scale-down
(typically minutes per the 13c §3.8.2 sweeper budget) +
step 2's user-provisioning loop (seconds per worker) +
the operator's chosen advisory-only soak.

#### 3.8.3 Why a runbook + an `eden-gitea-admin migrate` command

Per 13c/13d posture: a runbook covers the operator-specific
parts (which OAuth provider, which networks, how long to
soak); the `migrate` command bundles the mechanical parts
(provision-user iteration, ACL flip from advisory-only to
enforce). Operators can opt out of the bundled command
and run the steps manually if their setup is unusual.

### 3.10 Conformance impact

13e adds ONE additive wire op
(`update_worker_labels`, §3.0a.4); it does not change
any normative spec text or modify any existing wire op.
Conformance scenarios are binding-agnostic per chapter
9 §6 — they exercise observable behavior of registered
workers, not the mutation surface. A deployment running
with Gitea hardening passes the same scenarios as a
deployment running with shared `eden` user, and a
deployment whose Store backend doesn't implement
`update_worker_labels` passes them too (the existing
scenarios don't invoke the op). No new conformance
scenarios are added in 13e.

The §3.9 CI job is the substrate-level proof; it asserts
the same end-state invariants the existing helm smokes do
plus the new "ACLs reject forged pushes" check.

There is one place where the spec and 13e's ACL design
intersect worth flagging: chapter 3 §4.1 says the
evaluator MUST NOT modify the worker branch. 13e's
`evaluator` Gitea token has `read:repository` only —
defense in
depth that satisfies the spec rule with belt-and-
suspenders. If a future spec change relaxed the
no-evaluator-write rule, 13e's ACLs would also need to
relax. Documented in §8.6.

## 8. Tricky areas

### 8.1 Gitea username constraints

§3.1.2 covers the simple cases; the complications:

- Gitea's username validation is partly configurable via
  `validation.AllowDotsInUsernames` (allows `.` — we
  don't use it). The default disallows leading `-`
  and `.`; EDEN's worker_id grammar already disallows
  leading `-`.
- Gitea's case-insensitive username matching means
  `Worker1` and `worker1` would conflict; the EDEN
  grammar's lowercase-only constraint avoids this trap
  by construction.
- Reserved Gitea usernames (`admin`, `assets`, `api`,
  `attachments`, `avatars`, `commits`, `debug`,
  `error`, `explore`, `favicon.ico`, `ghost`, `help`,
  `install`, `issues`, `less`, `login`, `logout`,
  `manifest.json`, `metrics`, `milestones`, `new`,
  `notifications`, `org`, `plugins`, `pulls`,
  `raw`, `release`, `repo`, `robots.txt`, `search`,
  `serviceworker.js`, `signin`, `signout`, `signup`,
  `ssh_info`, `stars`, `template`, `user`,
  `v2`) — EDEN's reserved-worker_ids set
  (`{admin, system, internal, _*}` per 12a-1 §D.1)
  intersects on `admin` only. The §3.1.2 mapping uses
  `eden-bootstrap` for the chart's own admin identity to
  sidestep both reservations.

### 8.2 Gitea token scope granularity

Gitea's documented scope vocabulary
([API Usage — Scoped tokens](https://docs.gitea.com/development/api-usage))
is coarse: `read:repository`, `write:repository`,
`read:organization`, `write:organization`,
`read:user`, etc. There is **no** per-repository
sub-scope (Gitea's tokens grant the scope to ANY repo
the user has access to). Per-repo isolation in 13e
comes from THREE layered mechanisms:

1. **Team membership** (per §3.0.4): each user is a
   member of exactly the `eden-<experiment_id>-<role>`
   team. The team is bound to ONE repo
   (`eden/<experiment_id>`) via team-repo binding.
2. **Repo-level team access mode**: read-only teams get
   `Read` access; write-capable teams get `Write` access.
3. **Branch-protection allow-lists** (per §3.2): the
   fine ACL that picks which TEAM can push which BRANCH.

The token's `write:repository` scope is the coarse
"this token can write to any repo this user has team-
write to" gate; the team-membership gate restricts that
to ONE repo; the branch-protection gate restricts that
to specific branches. Effective allow-list is the
intersection of all three.

This is fine for our threat model (bad config, not
malicious workers) but worth flagging — the token scope
is not by itself a tight ACL. Documented in §8.6.

### 8.3 Gitea branch-protection idempotency on re-apply

The `provision_branch_protection` API endpoint is
`POST /api/v1/repos/{o}/{r}/branch_protections` for
create + `PATCH /.../branch_protections/{id}` for update.
The bootstrap Job's idempotent shape:

1. `GET /api/v1/repos/{o}/{r}/branch_protections` — list
   existing rules.
2. For each rule we want to provision: if a matching
   rule exists (by `branch_name`), `PATCH` it; otherwise
   `POST` it.
3. For any pre-existing rule whose `branch_name` is one
   of `main` / `variant/*` / `work/*` but whose
   `rule_name` doesn't start with `eden-` (operator-
   created), leave it alone — the operator may have
   layered their own rules.

### 8.4 `register_worker` post-commit hook can fail

Per §3.1.3: a `register_worker` call followed by a
failed `provision-user` leaves the EDEN side registered
but Gitea side empty. The next `bootstrap-script
reconciliation` rerun re-attempts the Gitea side.
Documented; the `eden-gitea-admin reconcile` subcommand
iterates `Store.list_workers()` and provisions any
missing Gitea users. Operators run it via the runbook's
"recovery" section.

A future amendment MAY make `register_worker` itself
fail when the Gitea side fails (transactional shape).
Out of v0 scope per the same reasoning as §3.6.2.

### 8.5 PR-mediated integrator timeout vs claim TTL

Per §3.3.2: when
`gitea.pr.integratorApproval.enabled=true`, the integrator
holds a claim for the PR-approval window plus the merge
plus event-write time. The claim TTL must accommodate this.
The chart's `config.integrator.claimTtlSeconds` defaults
to 600 when PR approval is enabled (per §3.3.2). The
12c control plane's claim-sweeper respects per-task
`expires_at`; the integrator's claim DOES carry one when
PR-approval is enabled.

If the PR sits in approval longer than the TTL, the
sweeper reclaims; on next dispatch the new integrator
sees the existing PR (idempotent — the PR's existence is
queryable via the Gitea API) and either continues
waiting or, if the prior integrator's temp branch is
stale, deletes it and re-opens. Documented in §8.7's
recovery flow.

### 8.6 Token-scope vs branch-protection layering

§8.2 documents that token scope is coarser than what we'd
ideally have. The §3.2 ACLs are the tight layer. There's
one subtle case worth pinning: an operator who disables
branch protection (`gitea.acl.enabled=false`) AND uses
per-worker scoped tokens (`gitea.users.workerTokenStore=chart-managed`)
ends up with weaker security than 13a's shared-password
baseline — every worker can push to every branch
(token-scope-only enforcement). The schema rejects this
combo: `gitea.users.workerTokenStore=chart-managed`
requires `gitea.acl.enabled=true`. Operators who want
per-worker tokens without ACLs (a deliberate threat-model
choice, e.g., for a dev environment with audit-only
goals) opt into `gitea.acl.mode=advisory-only` instead.

### 8.7 PR-stuck-mid-approval recovery

When PR approval is enabled and a PR sits unmerged past
the integrator's claim TTL:

1. Sweeper reclaims the integrator's task.
2. Next dispatch: a new integrator sees `variant_id` in
   `success` state without `variant_commit_sha`.
3. New integrator queries
   `GET /api/v1/repos/{o}/{r}/pulls?state=open`,
   matches by source-branch
   `_eden_internal/pending-integrate/<variant_id>`.
4. Two outcomes:
   - PR found, still open: the new integrator waits on
     it (same path as the original). The PR's auto-merge
     hook fires when approved.
   - PR found, closed but not merged: the new integrator
     deletes the temp branch + re-opens a fresh PR with
     the same source-branch shape.
   - PR not found: the temp branch is the only state;
     re-open as a fresh PR.

The §3.8.2 runbook covers the `eden-gitea-admin pr-status`
command for operators inspecting stuck-PR cases.

### 8.8 OAuth provider consistency across pod restarts

Gitea's OAuth provider config lives in the SQLite DB
(or external Postgres if the operator configured Gitea's
`database.DB_TYPE` accordingly — out of 13e's scope; 13e
keeps Gitea on its bundled sqlite per 13a's image
choice). Pod restart preserves the config. Helm upgrade
that re-runs the bootstrap Job: the Job is idempotent
(checks via `gitea admin auth list-oauth` first); a
duplicate provider name is rejected by Gitea, so the
Job's idempotency is enforced.

If the operator changes `gitea.auth.oauth.provider` in
values across upgrades (e.g., switching from github to
google), the Job's idempotent path adds the NEW provider
without removing the old one. Operators do that via
`eden-gitea-admin remove-oauth-provider` separately.
Documented in §8.

### 8.9 Compose `.env` secret-rotation alignment

Compose deployments using
`compose.gitea-perworker.yaml` need per-worker tokens
written to `.env`. The §3.6 EDEN-rotation hook can't
trigger a Compose `.env` re-write without operator
involvement. Documented as a runbook item: when
`reissue_credential` runs on a Compose deployment, the
operator manually re-runs `setup-experiment.sh` to
regenerate the `.env` with the new tokens (the script's
idempotent re-run path is preserved).

### 8.10 Chart upgrade: 13a → 13e default values keep behavior

Operators on chart 0.4.0 (13d) upgrade to 0.5.0 (13e)
without changing values. Defaults:
`gitea.auth.mode=basic`, `gitea.acl.enabled=false`,
`gitea.pr.*.enabled=false`,
`gitea.users.workerTokenStore=externalSecret` with the
existing `secrets.giteaRemotePassword` reference.
Result: no behavior change. Operators opt into the new
features explicitly via values updates.

The legacy `secrets.giteaRemotePassword` value (per 13a
§3.3a) stays valid in `gitea.users.workerTokenStore=externalSecret`
mode (i.e., when EDEN doesn't manage the worker tokens).
When operators flip to
`gitea.users.workerTokenStore=chart-managed`, the
legacy value becomes ignored and the chart-managed
Secret takes over. Documented in §3.8.2.

## 9. Risks

1. **Gitea API instability across versions.** Gitea
   1.22.x is the pin; minor versions have occasionally
   tweaked the branch-protection JSON shape. Mitigation:
   pin the image tag in 13a's values; CI runs against
   the pinned version; bumping the pin is a deliberate
   operator action.
2. **OAuth provider drift.** If the operator's OAuth
   provider changes its OIDC discovery endpoint (e.g.,
   GitHub deprecating `https://github.com/login/oauth`
   for `https://api.github.com/login/oauth`), Gitea's
   built-in providers handle the transition; the
   custom `openidConnect` provider does NOT and needs
   the issuerUrl updated. Documented in the runbook.
3. **Token-rotation timing windows.** The §3.6
   best-effort rotation has a brief window where the
   EDEN credential is rotated but the Gitea token
   isn't. Workers that hit Gitea during that window
   either see success (old token still valid) or get
   a 401 (admin already rotated). Documented; the
   credential-helper script's exit code propagates the
   401 → next worker iteration retries with the
   refreshed env.
4. **Branch-protection rejection-during-bootstrap.**
   The `setup-experiment` flow does an initial seed-push
   to `main`. If the chart's bootstrap-Job order
   provisions ACL BEFORE the seed-push runs, the
   seed-push gets rejected. Mitigation: §3.5's bootstrap
   Job ordering puts seed-push BEFORE ACL provisioning;
   the chart's `helm.sh/hook-weight` annotations
   enforce this.
5. **Per-worker user proliferation.** Long-running
   experiments accumulate Gitea users (one per
   registered worker). Each user costs a row in Gitea's
   user table + a token row. Gitea handles thousands
   without issue; documented as an operator-monitoring
   item.
6. **PR-mediated integrator latency.** §3.3.2's PR
   approval adds 0-300s latency per integration. For
   high-throughput experiments (many variants/min),
   this is a real bottleneck. Default OFF mitigates
   for the typical case; operators who turn it on are
   making a deliberate latency-vs-audit tradeoff.
7. **OAuth + on-prem provider TLS.** Operators using a
   self-hosted OIDC IdP whose CA isn't in Gitea's
   trust bundle hit "x509: certificate signed by
   unknown authority" on the OAuth callback. Mitigation:
   the chart's `gitea.tls.caBundleSecret` (mirroring
   13c's Postgres CA pattern) lets the operator mount
   their CA into the Gitea pod's trust store.
   Documented in the runbook.
8. **Compose-side `setup-experiment.sh` divergence.**
   The new per-worker overlay path adds significant
   complexity to `setup-experiment.sh`. Risk: the
   default (no-overlay) path regresses. Mitigation:
   compose-smoke continues running against the default
   path unchanged; the overlay path has its own
   compose-smoke variant if/when 13e ships an
   `compose-smoke-gitea-perworker` job (out of v0
   scope; future amendment).

## 10. Sequencing

Recommended PR shape (in order):

1. **`eden-gitea` package PR.** New workspace member;
   `GiteaAdmin` class; CLI; unit tests against mocked
   responses + integration tests against a real Gitea
   container.
2. **Service integration PR.** `--gitea-token-file` flag
   plumbing across executor / evaluator / ideator /
   web-ui; credential-helper Python helper; `register_worker`
   post-commit hook.
3. **Chart values + schema PR.** `gitea.*` block;
   `values.schema.json` clauses; `_helpers.tpl`
   additions.
4. **Chart templates PR.** OAuth env-var wiring on
   gitea-statefulset.yaml; bootstrap Job templates
   (out-of-tree).
5. **Compose overlays PR.** Three opt-in overlays plus
   `setup-experiment.sh` updates.
6. **CI PR.** `helm-smoke-gitea-oauth` + `eden-gitea-tests`.
7. **Integrator PR-path PR.** `--integrator-pr-approval-enabled`
   plumbing in the orchestrator's integrator
   call site.
8. **Web-UI human-executor PR-path PR.**
   `--human-executor-pr-enabled` plumbing in chunk 9c's
   executor route.
9. **Runbook + docs PR.** `migrating-to-gitea-hardening.md`;
   `helm.md` extension; roadmap delta.

A reviewer going from PR 1 to PR 9 should expect the
`helm-smoke-gitea-oauth` job to first appear in PR 6 and
go green on first run; if it doesn't, the failure mode
is most likely the oauth2-mock startup race
(documented in §3.9 + the §6.3 fixture wait pattern).

## 13. What lands at the end of 13e

After 13e merges, an operator can run
`helm install eden ./reference/helm/eden -f values.yaml`
against any conforming Kubernetes cluster, configure
their preferred OAuth provider, and get a
production-grade EDEN deployment where:

- Operators sign in to the Gitea web UI via SSO.
- Each worker has a per-worker Gitea identity with
  scoped tokens.
- Branch protection enforces the chapter-6 §1
  namespace ownership rules at the Gitea layer.
- Optional: human-executor flows go through native PRs
  with reviewer approval.
- Optional: integrator merges go through PRs for
  audit + human-approval gates.
- EDEN credential rotation aligns with Gitea token
  rotation as a single operator action.

The Compose deployment retains its simple shared-`eden`
posture by default; operators who want the hardened
posture in dev opt into the new overlays. The
13c→13d→13e substrate sequence is complete: managed
Postgres + cloud blob storage + hardened Gitea give
operators a deployment they can stand behind.
