**Missing Context**

Assessment: the round-0 blockers are addressed. The repo-ownership model and the role source are now explicit enough to move on.

**Feasibility**

Assessment: this is now the first level with material blockers, so I would stop here before reviewing alternatives, completeness, or edge cases.

- `CRITICAL` The token-mint / token-rotate path is not supported as written. The plan has an admin-token-driven helper minting PATs via `POST /api/v1/users/{username}/tokens` ([eden-phase-13e-gitea-hardening.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13e-gitea-hardening.md:771), [eden-phase-13e-gitea-hardening.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13e-gitea-hardening.md:798), [eden-phase-13e-gitea-hardening.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13e-gitea-hardening.md:1196)). Gitea’s API docs say that endpoint is special and requires `BasicAuth` with the target user’s password, not token auth ([API Usage](https://docs.gitea.com/next/development/api-usage)). The plan needs a version-verified minting mechanism, and rotation has to use that same mechanism too.

- `CRITICAL` The scoped-token model is still based on capabilities Gitea does not document. The plan repeatedly uses repo-specific scopes like `repo:write on eden/<experiment_id>` / `repo:read` ([eden-phase-13e-gitea-hardening.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13e-gitea-hardening.md:360), [eden-phase-13e-gitea-hardening.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13e-gitea-hardening.md:691), [eden-phase-13e-gitea-hardening.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13e-gitea-hardening.md:801), [eden-phase-13e-gitea-hardening.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13e-gitea-hardening.md:1835)). Gitea’s token docs describe coarse permission blocks such as `read:repository`, `write:repository`, `read:organization`, etc.; they are not repo-specific, and the documented syntax is `write:repository`, not `repo:write` ([API Usage](https://docs.gitea.com/next/development/api-usage)). The plan should treat team/repo membership plus branch protection as the real isolation boundary, and rewrite the token-scope sections accordingly.

- `CRITICAL` The Secret-rotation propagation story is wrong for Kubernetes env vars. The plan says workers will pick up a rotated token because the credential helper re-reads the env and kubelet will refresh the pod env within seconds ([eden-phase-13e-gitea-hardening.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13e-gitea-hardening.md:1201)). Kubernetes docs say Secret updates are not reflected in container environment variables unless the container is restarted ([Distribute Credentials Securely Using Secrets](https://kubernetes.io/docs/tasks/inject-data-application/distribute-credentials-secure/)). If hot rotation matters, the token needs to be read from a mounted Secret file or another reloadable source; otherwise the plan should explicitly require pod restart on rotation.

- `CRITICAL` The migration backfill path depends on an interface that the predecessor contract does not define. Path B says `backfill-labels` writes `eden.role` through an existing 12a-1 `update_worker_labels` op ([eden-phase-13e-gitea-hardening.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13e-gitea-hardening.md:1317)), but 12a-1 only defines `register_worker(worker_id, labels?)` and this plan still says 13e does not change the wire protocol/spec ([eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-12a-1-worker-identity.md:137), [eden-phase-13e-gitea-hardening.md](/Users/ericalt/Documents/eden-worktrees/phase-13cde-substrate-plans/docs/plans/eden-phase-13e-gitea-hardening.md:214)). Either 13e needs to add a deliberate worker-label-mutation API/spec amendment, or the migration/backfill story needs to be redesigned around already-defined operations.

I’d skip levels 3-5 until those feasibility issues are resolved.

**Overall Assessment**

The plan is materially better after the round-0 fixes, but it is still not implementation-ready. The remaining blockers are now concrete mechanism problems rather than missing context: token issuance, token-scope semantics, Secret rotation delivery, and the nonexistent worker-label update path.
