**Blockers**
- The `Store` Protocol does not faithfully encode chapter 8 as claimed. [protocol.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/protocol.py:53) exposes workflow-specific methods like `create_plan_task()`/`create_implement_task()`/`create_evaluate_task()` and a single `events()` snapshot, but chapter 8 normatively names `create_task`, `read_range`, `subscribe`, and `replay`, and also defines an artifact-store contract. See [08-storage.md §1.1](/Users/ericalt/Documents/eden/spec/v0/08-storage.md:35), [§2.1](/Users/ericalt/Documents/eden/spec/v0/08-storage.md:136), and [§5.1](/Users/ericalt/Documents/eden/spec/v0/08-storage.md:274). The repo explicitly declares artifact storage a non-goal in [README.md](/Users/ericalt/Documents/eden/reference/packages/eden-storage/README.md:33), so “pins the chapter-8 union” in [roadmap.md](/Users/ericalt/Documents/eden/docs/roadmap.md:195) and [0.md](/Users/ericalt/Documents/eden/docs/plans/review/phase-6-storage/impl/20260423T163433/0.md:171) overstates what shipped. A cleaner reading would be either separate `TaskStore`/`EventLog`/`ArtifactStore` protocols plus a façade, or a spec-shaped single protocol that still includes the event-log and artifact-store operations.
- `SqliteStore`’s durability setting does not satisfy a literal reading of `08-storage.md §3.1`’s “crash of the store’s host”. The code sets WAL + `synchronous=NORMAL` in [sqlite.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/sqlite.py:161), and the brief/roadmap claim that this satisfies §3.1 in [0.md](/Users/ericalt/Documents/eden/docs/plans/review/phase-6-storage/impl/20260423T163433/0.md:95) and [roadmap.md](/Users/ericalt/Documents/eden/docs/roadmap.md:201). SQLite’s own docs say WAL+NORMAL remains consistent but “does lose durability”; committed transactions may roll back after power loss or system crash, while WAL+FULL adds the per-commit sync for durability: https://www.sqlite.org/pragma.html#pragma_synchronous
- `_max_event_seq()` is not just future-fragile; it is already wrong once the default counter reaches `evt-1000000`. The default generator emits `evt-{n:06d}` in [_base.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/_base.py:175), but [sqlite.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/sqlite.py:232) only scans exactly six digits. I reproduced this locally: after persisting `evt-1000000`, reopen seeds from `999999`, and the next append fails with `UNIQUE constraint failed: event.event_id`. That makes the “resumes the counter from the persisted max” claim false in [0.md](/Users/ericalt/Documents/eden/docs/plans/review/phase-6-storage/impl/20260423T163433/0.md:104) and [roadmap.md](/Users/ericalt/Documents/eden/docs/roadmap.md:205).
- Metrics-schema immutability is enforced as literal JSON-string equality, which is stricter than the spec’s semantic “schema MUST NOT be mutated” rule. [sqlite.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/sqlite.py:193) serializes the schema and compares `stored_schema != schema_json` at [sqlite.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/sqlite.py:215). Because JSON object key order is not semantically meaningful here, reopening with the same metric map in a different insertion order is rejected, even though the schema did not mutate under [08-storage.md §4.2](/Users/ericalt/Documents/eden/spec/v0/08-storage.md:235). I reproduced that false rejection locally.

**Assessment**
The shared `_StoreBase` design is solid, and I did not find a current partial-visibility bug in the composite-commit paths themselves. I ran `uv run pytest -q reference/packages/eden-storage/tests`; all 119 tests passed. The main issue is that the implementation and docs over-claim chapter-8 conformance: atomic rollback looks good, but the public contract, durability setting, and reopen semantics are not fully aligned with the spec.

**Non-Blocker Suggestions**
- Atomicity looks correct for current composite commits. `_apply_commit()` runs inside one explicit SQLite transaction in [sqlite.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/sqlite.py:379), and the injected mid-commit failure test in [test_restart_safety.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/tests/test_restart_safety.py:225) cleanly rolls back. I could not construct a current sequence where a partial composite commit becomes visible after an exception; the real issue is durability under host crash, not atomicity.
- The `_in_txn` re-entrancy model is acceptable today but discipline-based. [sqlite.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/sqlite.py:260) protects nested helper calls in SQLite, but [memory.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/memory.py:49) has no rollback layer. If a future refactor re-enters public methods or calls `_apply_commit()` before all validation is finished, SQLite could mask the mistake while the in-memory backend would not. A stricter “stage first, single commit point second” structure would remove that class of risk.
- The brief has a few accuracy issues beyond the blockers. “Monkey-patched commit failure verifies rollback” in [0.md](/Users/ericalt/Documents/eden/docs/plans/review/phase-6-storage/impl/20260423T163433/0.md:124) and [CLAUDE.md](/Users/ericalt/Documents/eden/CLAUDE.md:29) is overstated; the test injects a failure during `_apply_commit`, not during SQLite `COMMIT`. “`ensure_schema` applies every pending one under a single outer transaction” in [0.md](/Users/ericalt/Documents/eden/docs/plans/review/phase-6-storage/impl/20260423T163433/0.md:100) is also overstated; [_schema.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/_schema.py:93) never issues `BEGIN`, and `executescript()` under autocommit is not one outer migration transaction.
- I did not find remaining mutable-reference leaks in the public read/list/events/submission boundary. The deep-copy and JSON rehydration story in [_base.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/_base.py:269) and [sqlite.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/sqlite.py:295) looks complete.
- Event retention and no-fabrication are mostly fine in practice: I found no deletion/compaction path, and persisted entities/events come from explicit caller-facing operations. The missing chapter-8 behavior is API surface, not hidden background behavior: `replay`/`subscribe` are absent, and artifact-store §5 is absent entirely.
