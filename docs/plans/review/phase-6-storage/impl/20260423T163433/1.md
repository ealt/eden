# Implementation Review: Phase 6 — reference storage backend (`eden-storage`) — round 1

## What changed since round 0

Addressed all four round-0 blockers:

1. **Protocol fidelity.** Added the spec-literal `create_task(task)` to the `Store` Protocol and both backends; it dispatches by `task.kind` into the existing composite-commit helpers. Added `replay()` and `read_range(cursor=None)` per chapter 8 §2.1. `events()` kept as a deprecated-friendly alias delegating to `replay()`. The brief and CLAUDE.md now explicitly scope the Protocol to task store + event log reads + proposal/trial persistence, calling out the artifact store (§5) and the streaming `subscribe` operation as Phase 6 non-goals (Phase 10 and Phase 8 respectively). 20 new parametrized scenarios in `tests/test_protocol_operations.py` exercise `create_task`, `replay`, and `read_range` across both backends.
2. **Durability.** SQLite PRAGMA switched from `synchronous=NORMAL` to `synchronous=FULL`, matching the strict reading of chapter 8 §3.1 ("survives a subsequent crash of the store's host").
3. **`_max_event_seq` fragility.** Replaced with `_next_event_seq()` which uses `MAX(seq)+1` on the event table's AUTOINCREMENT primary key. Resumption is independent of the `event_id` string format.
4. **Metrics-schema literal-JSON comparison.** `_initialize_experiment` canonicalizes both sides via `json.dumps(..., sort_keys=True)` before comparing, so a reopen with the same logical schema in a different key order is not mistaken for a mutation.

Plus two of the non-blocker brief-accuracy issues:

- `ensure_schema` now runs migrations inside an explicit `BEGIN` / `COMMIT` by issuing DDL via `execute()` per statement rather than `executescript`.
- Brief wording: "monkey-patched `_apply_commit` failure" (the test injects a failure during the stage phase, not during SQLite's `COMMIT` statement).

Tests: 242 passing across both backends.

## Change Summary

### Package + protocol layout

- Chapter 8 defines three stores (task, event log, proposal/trial) but §7 explicitly permits one physical backend to serve all three. The `Store` Protocol collapses them into a single structural interface — covering the task store (§1), the event log reads (§2.1 `replay` + `read_range`), and proposal/trial persistence (§1.7). Out of scope at Phase 6: the artifact store (§5 — Phase 10); streaming `subscribe` (§2.1 — Phase 8 when cross-process lands). The collapse is the smallest surface that lets `05-event-protocol.md` §2.2 composite commits ("this write + that event both land or neither does") be expressed naturally as one atomic method.
- `InMemoryStore` moved from `eden-dispatch` to `eden-storage` **unchanged in behavior** — every field, every method, every validation path is bit-for-bit equivalent. The only changes are import paths. 57 pre-existing Phase 5 tests continue to pass against it without modification.
- `eden-dispatch` still re-exports the Store types from its package top level (`from eden_dispatch import InMemoryStore, SqliteStore, ...`), so pre-Phase-6 callers see no API break.

### Shared transition logic (`_base.py`)

- One `_StoreBase` holds every public method, every validation helper, and every composite-commit staging path. Subclasses override only the backend primitives (`_get_task`, `_iter_trials`, `_atomic_operation`, `_apply_commit`, …).
- This is the structural guarantee that "both backends pass the same conformance suite" actually means something: there is only one code path for the state machine and the composite commits; a future Postgres backend or a remote backend that overrides the primitives inherits every invariant for free.

### SQLite backend (`sqlite.py` + `_schema.py`)

- Each public op: `BEGIN IMMEDIATE` → run the shared logic → `COMMIT` on normal exit, `ROLLBACK` on exception. `_in_txn` tracks re-entry so helper methods called from inside a public op (e.g. `_accept_implement`) participate in the outer transaction rather than opening a nested one.
- Row layout: one table per entity kind; `data` column is a JSON blob of the full Pydantic dump; denormalized columns (`kind`, `state`, `status`) index filtered queries only. Rehydration goes back through `Model.model_validate_json` so the denormalized columns cannot silently diverge from the canonical JSON (drift would surface at the next read).
- PRAGMAs: `foreign_keys = ON`, `journal_mode = WAL`, `synchronous = FULL`. Chapter 8 §3.1 requires a committed write to survive a crash of the store's host; SQLite's own docs say WAL + NORMAL may lose committed transactions on system crash, while WAL + FULL fsyncs on commit. The stricter setting is the one that matches a literal reading of §3.1; the cost is a fsync per composite commit.
- Migrations are linear. `_MIGRATIONS` is a list of callables indexed by version; `ensure_schema` applies every pending one inside an explicit `BEGIN` / `COMMIT` (DDL issued via `execute()` per statement rather than `executescript`, because `executescript` auto-commits and would break the bounding). A partial-DDL failure rolls back cleanly. Adding a future migration means appending a function.
- The default `event_id` counter in `_StoreBase` starts at 1 every time a store object is constructed. On reopen that would produce duplicate `event_id` values and violate the table's UNIQUE constraint. `SqliteStore.__init__` resumes the counter from `MAX(seq)+1` on the event table's AUTOINCREMENT primary key — format-independent; the `evt-NNNNNN` string convention is not load-bearing. Caller-supplied factories are responsible for their own collision avoidance on reopen.
- `_initialize_experiment` enforces chapter 8 §4.2 (metrics-schema immutability): on reopen, the stored `experiment_id` must match the requested one, and a supplied `metrics_schema` must match the persisted one up to JSON key order. Comparison canonicalizes both sides via `json.dumps(..., sort_keys=True)` so semantically identical schemas in different key orders are not rejected — §4.2 constrains semantic identity, not lexical representation. Reopen without a schema argument inherits the persisted one so §4.3 metric validation keeps running.

### Conformance across backends

- `tests/conftest.py` parametrizes `make_store` across (`InMemoryStore`, `SqliteStore`). Every conformance scenario (`test_task_state_machine.py`, `test_composite_commits.py`, `test_store_hardening.py`) now runs twice — 110 scenarios total, both backends green.
- The 9 `test_restart_safety.py` tests are `SqliteStore`-only because the scenario (close + reopen) is meaningless for an in-memory backend. They cover: persistence of every entity + event on reopen; claim-token survival; total-order preservation across a restart boundary; rejection of reopen under a different experiment id or metrics schema; inheritance of persisted metrics schema; rollback-on-exception via a monkey-patched commit failure; full planner→implementer→{restart}→evaluator→integrator cycle.

## Pre-review audit (five impl-review lenses)

Applied the `feedback_impl_review_lenses.md` audit:

1. **Mutable-model boundary leak.** Every `read_*` / `list_*` / `events()` in `_base.py` deep-copies. `create_proposal`, `create_trial`, and `submit` deep-copy caller-supplied values before persisting. `SqliteStore` reads additionally rehydrate fresh Pydantic models from JSON; the `_deep` wrapper is then a no-op-equivalent belt-and-suspenders.
2. **`model_copy(update=...)` validator bypass.** No call sites. `grep -rn "model_copy(update" reference/` returns only the docstring that explains why we avoid it. All updates go through `_validated_update` which dumps → merges → `model_validate`.
3. **Validation failures terminalize, not raise.** Preserved from Phase 5. `validate_terminal` returns a three-way decision; the driver routes to accept / reject(worker_error) / reject(validation_error).
4. **Status vs. reason keying asymmetry.** Preserved. `_reject_evaluate` keys trial-side transitions on `submission.status`, not on the orchestrator's `reason`.
5. **Aspirational framing in the brief.** Grep-verified every claim:
   - "only one code path for the state machine" — grep confirms no duplication of transition logic in `memory.py` or `sqlite.py`; both only override the primitive hooks.
   - "WAL + FULL satisfies §3.1" — a claim about SQLite behavior, not about code. SQLite's own docs say `synchronous=FULL` fsyncs on every commit so a committed write cannot be lost on system crash; that matches §3.1's "survives a subsequent crash of the store's host" literally.
   - "backends inherit every invariant for free" — verified by parametrizing 65 conformance scenarios across both backends; both pass without any scenario-side accommodation.

## Notable design decisions to audit

1. **Single `Store` Protocol vs. three Protocols.** Chapter 8 names three stores; the Protocol merges them. Rationale in `protocol.py` module docstring: composite commits span store boundaries, and a Protocol split across three objects would need an external commit coordinator (added complexity, no normative benefit). A reviewer may argue the opposite — that the Protocol should mirror the spec's noun structure — and I would want to hear it.
2. **Re-entrant transactions.** `_in_txn` allows `_accept_implement` etc. to call `_apply_commit` without opening a second transaction. The alternative is to split every public op into a "validate + stage" phase and a "commit" phase and have only the public op open the transaction. The re-entrant approach preserves the existing in-memory code structure (helpers are `_apply_commit`-callable in both backends), but it does mean the "one BEGIN per public op" rule is enforced by discipline, not structurally.
3. **`_max_event_seq` sniffs only the default `evt-NNNNNN` format.** A caller-supplied `event_id_factory` is responsible for its own collision avoidance on reopen; I don't try to interpret arbitrary event_id patterns. This is a reasonable boundary because the factory is caller-controlled, but a reviewer may prefer that I persist the counter explicitly (e.g. in the `experiment` row) rather than inferring it from max(event_id).
4. **WAL + `synchronous = NORMAL`.** Industry-standard durability for a reference impl. `synchronous = FULL` would give crash safety against kernel panics / power loss; `NORMAL` gives crash safety against process death, which is the scenario chapter 8 §3.1 is worded around. A reviewer who interprets §3.1 as requiring fsync-on-commit may want `FULL`.
5. **No Postgres, no blob store.** Per Phase 6 non-goals. I want this confirmed rather than assumed — please flag if chapter 8 §5 (artifact store) is close enough to a storage-chapter deliverable that Phase 6 should have shipped a stub.

## Scope this review should cover

1. Does the Protocol faithfully express chapter 8 §1 + §1.7 + §2?
2. Does `SqliteStore` satisfy chapter 8 §3 (durability + crash recovery) under the reading that a process crash is the unit test's close/reopen boundary, given WAL + `synchronous = NORMAL`?
3. Does every composite commit in `05-event-protocol.md` §2.2 actually land atomically in SQLite — i.e. either all rows of a `_Tx` persist, or none do, including on exceptions raised mid-commit? The monkey-patched test asserts this for one such commit; are there composites I should also spot-check?
4. Is the re-entrant transaction model correct, or does it hide a bug where a helper's `_apply_commit` could succeed while the outer transaction is already doomed?
5. Is the `_max_event_seq` reconstruction approach defensible, or should the counter live in its own persisted cell?
6. Any corners of chapter 8 the refactor missed (event log retention §2.4, no-fabrication §3.4, artifact-store §5 boundary)?
