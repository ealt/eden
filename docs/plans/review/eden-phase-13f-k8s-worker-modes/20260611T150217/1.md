# Phase 13f — Kubernetes-native subprocess + pod-exec worker hosts

**Status.** Draft. Plan for
[#291](https://github.com/ealt/eden/issues/291).

**Predecessors.** 13a (base Helm chart, shipped 2026-06-04), 13c
(managed Postgres, shipped 2026-06-10), 13d (S3/GCS blob backend,
shipped 2026-06-09). The chart this plan extends is the
**as-implemented** chart at [`reference/helm/eden/`](../../reference/helm/eden/)
— which diverged from the 13a plan doc in load-bearing ways
(the #128 identity-gating three-phase bring-up, lease mode off by
default with no control-plane Deployment, the 13d `blob.*`
backend) — not the 13a plan text.

**Why now.** The operator wants real (user-supplied `*_command`)
experiments on AWS/EKS. The merged chart runs `--mode scripted`
reference workers only (13a Decision 10); subprocess and
docker-exec modes exist only on Compose. #291 is the gap between
"infra validates on EKS" and "real experiments run on EKS".

**Chunk id: 13f.** The roadmap's Phase 13 allocates 13a–13e.
This chunk's scope — all three worker roles, two new modes,
chart + host-code + CI — is broader than 13b's "Executor as a
k8s Job (GPU node selection)" slot, and it **subsumes** 13b
rather than fitting inside it (§1.2). A fresh id keeps the
roadmap history legible: 13b's line is annotated as subsumed,
13f gets its own line, and this plan PR performs both flips.

[`docs/glossary.md`](../glossary.md) and AGENTS.md "Naming

## 1. Context

### 1.0 Substrate baseline (the as-implemented chart)

Facts about the merged chart that constrain this design:

- **Worker hosts are mode-hardcoded.** All three hosts pass
  `--mode {{ .Values.config.workerMode }}` where
  `values.schema.json` pins `workerMode` to `enum: ["scripted"]`.
  The evaluator's per-replica clone PVC exists but is unused in
  scripted mode ("reserved for subprocess" —
  [`values.yaml`](../../reference/helm/eden/values.yaml) line 160).
- **Identity gating.** Each identity-consuming service renders
  only once `identity.<svc>.{workerId,token}` are both set;
  `setup-experiment-helm.sh` runs the three-phase bring-up
  (infra → seed + store tier → mint identities + app tier).
  13f's new modes change worker-pod *internals* (args, mounts,
  images), not the render gates or the bring-up phases — the
  setup script is untouched except for docs.
- **Artifacts are web-ui-only** (issue
  [#285](https://github.com/ealt/eden/issues/285)): the
  artifacts PVC is a web-ui `volumeClaimTemplate`; no worker
  host and not the task-store-server mounts it, and the
  task-store-server's `/_reference/.../artifacts/` route is not
  wired (no `--artifacts-dir`). Subprocess mode breaks this
  posture (§1.4).
- **Whole-Secret env injection.** Every service pod uses
  `envFrom: secretRef` against the chart-managed Secret, so each
  worker-host process environment contains `EDEN_ADMIN_TOKEN`,
  `POSTGRES_PASSWORD`, `FORGEJO_REMOTE_PASSWORD`,
  `EDEN_STORE_URL`, and the rest. Harmless while workers are
  scripted; a **secret-exfiltration hazard** the moment a
  user-supplied `*_command` runs in that pod, because
  [`subprocess_runner.spawn`](../../reference/services/_common/src/eden_service_common/subprocess_runner.py)
  builds the child env as
  `full_env = dict(os.environ); full_env.update(env)` — the
  child inherits everything. §2 Decision 6 bundles the fix.
- **In-cluster DNS.** k8s pods resolve Service names natively
  (`<release>-task-store-server`, `<release>-forgejo`). The
  Compose DooD `--exec-network` trap (issue #155 — siblings land
  on the docker bridge and can't resolve compose hostnames) has
  **no k8s analog**: a per-task pod is a first-class pod on the
  cluster network. No `--exec-network`-equivalent flag is needed.
- **No docker socket.** EKS (and every modern managed cluster)
  runs containerd; `/var/run/docker.sock` does not exist on the
  node. `--exec-mode docker` cannot port
  ([`worker-host-subprocess.md`](../../spec/v0/reference-bindings/worker-host-subprocess.md)
  §8). The k8s-native analog is spawning per-task pods via the
  k8s API.

### 1.1 What 13f adds

Two opt-in worker modes on the Helm substrate, per-role:

1. **Subprocess mode in-pod** (`workers.<role>.mode: subprocess`,
   all three roles): the existing `--mode subprocess` host runs
   inside its pod, spawning the user's `*_command` in the same
   container. The pod's image is an operator-built experiment
   image layered on the reference image (§2 Decision 3). This is
   the k8s port of
   [`compose.subprocess.yaml`](../../reference/compose/compose.subprocess.yaml).
   On k8s every host already runs in its own pod, so subprocess
   mode is already container-isolated *from the other services*;
   what it does not isolate is user code from the host process
   sharing the pod — that residual is what pod-exec adds.
2. **Pod-exec mode** (`workers.<role>.mode: podExec`, executor +
   evaluator only): each claimed task becomes one `batch/v1 Job`
   whose pod clones from Forgejo into an emptyDir, runs the
   user's command in the operator's experiment image, and reports
   its outcome via a stdout sentinel line the host reads through
   the `pods/log` API. This is the k8s-native analog of
   [`compose.docker-exec.yaml`](../../reference/compose/compose.docker-exec.yaml)
   *and* the realization of the roadmap's "executor as a k8s Job"
   (§1.2). The ideator never gets pod-exec (§2 Decision 1).

Plus the substrate work both modes force: the shared artifacts
PVC (§1.4), subprocess env isolation (Decision 6), per-role chart
values with schema gates, the RBAC opt-in, two new kind smokes,
and an informative `worker-host-pod-exec.md` reference binding.

### 1.2 Relationship to 13b / #172: subsume, don't duplicate

[`docs/plans/eden-phase-13b-executor-k8s-job.md`](eden-phase-13b-executor-k8s-job.md)
(tracked by [#172](https://github.com/ealt/eden/issues/172))
already designed executor-as-Job in detail, and several of its
calls survive review and are **adopted wholesale** here (§3.5):
Job-per-task with `backoffLimit: 0`; stdout-sentinel outcome
plumbing via `pods/log`; init-container clone to emptyDir;
namespace-scoped RBAC; the duplicate-Pod selection rule; the
image-pull early-give-up deadline; the log-read-vs-GC race
mitigation.

But the 13b plan cannot be executed as written:

- It predates the **as-implemented** 13a chart. It assumes a
  `config.workerMode`-era chart with no identity gating, an
  always-on control plane, control-plane experiment
  registration in the smoke, and a `helm-smoke` shape that no
  longer exists.
- It covers the **executor only**. #291 needs evaluator
  pod-exec and the subprocess mode for all three roles; landing
  13b first and 13f after would build the Job substrate twice
  or force 13f to refactor freshly-merged 13b code.
- Its wrapper/init details predate #166 (wire artifacts) and
  the #128 opaque-identity model.

**Decision: 13f subsumes 13b.** Specifically:

- The per-task Job substrate lands here, shared by executor and
  evaluator (`eden_service_common/pod_exec.py`).
- The GPU scheduling surface — `nodeSelector`, `tolerations`,
  `resources` (including `limits."nvidia.com/gpu"`-shaped keys)
  — ships here as pure pass-through values under
  `workers.<role>.podExec.*` (§3.3). It is three pass-through
  fields on a pod template this chunk must define anyway;
  splitting them into a separate chunk would be artificial.
- This plan PR: marks the 13b plan superseded in place (banner
  at the top of
  [`eden-phase-13b-executor-k8s-job.md`](eden-phase-13b-executor-k8s-job.md));
  rewrites the roadmap's 13b line to point at 13f; adds the 13f
  roadmap line. The physical move to
  `docs/archive/eden-phase-13b-executor-k8s-job.md` rides 13f's
  final docs wave (wave 5).
- At plan-merge, #172 gets a comment explaining the
  subsumption; it is **closed when 13f's pod-exec wave ships**.
  13b deferrals that remain real are re-filed as their own
  issues at that point (per-experiment `execution_resources`
  overrides, `podFailurePolicy` on k8s ≥ 1.31, static wrapper
  binary for shell-less images, per-Job sentinel nonce — §11).
- GPU validation on a **real** GPU cluster (vs the kind smoke's
  synthetic label) stays operator-side, as 13b already scoped
  it.

### 1.4 Artifact substrate reconciliation (#285 / #166 / #290)

Subprocess mode depends on the shared-filesystem artifact
substrate that the chart deliberately does not provide (#285):
the ideator host writes `ideas/<idea_id>/content.md` under
`--artifacts-dir` and the executor's `task.json.content_path`
points into that same tree; the evaluator reads it read-only.
The wire-deposit endpoints from #166 exist, but the hosts still
*write and resolve* `file://` artifacts until the
[#290](https://github.com/ealt/eden/issues/290) hard cutover —
which is unscheduled and explicitly the highest-blast-radius
wave of #166.

Options considered:

- **(a) Make #290 a hard predecessor.** Cleanest end-state (no
  shared FS at all; per-task pods fetch over the wire), but it
  blocks the operator's EKS goal on an unscheduled
  cross-cutting cutover. Rejected for sequencing, not for
  direction.
- **(b) Shared artifacts PVC, mirroring Compose (chosen).**
  Promote the artifacts volume from a web-ui
  `volumeClaimTemplate` to a **standalone chart-managed PVC**
  (the `blob-pvc.yaml` pattern, `helm.sh/resource-policy: keep`)
  mounted by: web-ui (rw, as today), ideator-host (rw, when
  subprocess), executor-host + per-task executor pods (ro),
  evaluator-host + per-task evaluator pods (ro), and the
  task-store-server (ro + `--artifacts-dir`, which finally wires
  the `/_reference/.../artifacts/` route so `EDEN_ARTIFACT_URL`
  substrate access works — the interim posture #285 itself
  names). New `storage.artifactsAccessModes` value, default
  `["ReadWriteOnce"]`.
- **(c) Inline idea content into the task payload / per-task
  ConfigMaps.** Rejected: invents a third artifact transport
  that #290 would immediately retire, and fights the 1 MiB-ish
  object caps.

The (b) constraint is honest and documented rather than hidden:
**RWO works only while all artifact-mounting pods land on one
node** (true in the kind smokes; true on a single-node dev
cluster). Multi-node deployments MUST set
`storage.artifactsAccessModes: ["ReadWriteMany"]` with an RWX
StorageClass — on EKS that is the standard EFS CSI pattern, which
is precisely the operator's target environment. The chart cannot
see cluster topology, so this is a values-comment + README +
`docs/deployment/helm.md` discipline, not a schema gate. #290
retires the shared PVC entirely; this chunk must not deepen the
`file://` coupling beyond what Compose already has (no new
`file://` writers or readers — only mounts for the existing
ones).

## 2. Decisions

1. **Per-role mode values; pod-exec for executor + evaluator
   only; the ideator never gets pod-exec.** `workers.<role>.mode`
   defaults to `scripted` per role; mixed deployments (ideator
   `subprocess` + executor/evaluator `podExec`) are the expected
   production posture, mirroring how Compose layers
   `compose.docker-exec.yaml` over `compose.subprocess.yaml`.
   The ideator's binding is a **long-running** stdin/stdout
   JSON-line protocol
   ([`worker-host-subprocess.md`](../../spec/v0/reference-bindings/worker-host-subprocess.md)
   §2) that exists so user code can hold accumulating LLM
   context across tasks; a per-task pod fights that semantics
   (13b §11 reached the same conclusion), and a long-running
   *sibling* pod would need a durable attached-stdin transport —
   the k8s attach API is an interactive-debugging surface, not a
   production IPC channel. On k8s the ideator's isolation story
   is its own pod: subprocess mode with an operator experiment
   image gives the ideator dependency isolation already; the
   in-pod host-vs-user-code residual is the same trust boundary
   the Compose host-mode overlay ships today.

2. **13f subsumes 13b** (§1.2). Adopted 13b decisions, restated
   so this plan is self-contained: Job-per-task (not bare Pod —
   controller-maintained terminal status, `ttlSecondsAfterFinished`
   GC, standard ecosystem shape) with `parallelism: 1`,
   `completions: 1`, `backoffLimit: 0` (the host owns retry via
   claim-TTL + sweeper; a node disruption fails the Job and maps
   to the role's chapter-3 failure status — no k8s-level retry);
   stdout sentinel `EDEN_OUTCOME <json>` as the last line, read
   via `pods/log` (rejected alternatives unchanged from 13b §3.1:
   per-task RWX PVC for outcome files, HTTP callback to the
   host, self-annotation patching); the §3.5 duplicate-Pod
   selection rule; the image-pull early-give-up deadline; the
   `ttlSecondsAfterFinished: 60` + read-logs-immediately
   mitigation for the log-GC race. **One deliberate deviation:**
   13b put the `git push` inside the user-facing wrapper script,
   which would hand the Forgejo deployment credential to user
   code and skip the AGENTS.md push read-back ladder; 13f moves
   publishing into a dedicated **publisher container** instead
   (Decision 10 / §3.4).

3. **Experiment image story: one operator-built image layered on
   the reference image; per-role override; experiment dir baked
   in.** In subprocess mode the user's code and deps must be in
   the worker pod's image *alongside* the eden host code (the
   host `Popen`s the command in its own container). The
   supported pattern is `FROM <registry>/eden-reference:<tag>`
   plus user deps plus `COPY <experiment-dir>` to the path named
   by `workers.experimentDir` (default
   `/etc/eden/experiment-dir`, matching Compose's mount target).
   `workers.<role>.image.{repository,tag}` overrides the chart
   image per role and defaults to the chart image; the schema
   enforces repository+tag are supplied together
   (both-or-neither `oneOf`, the `identity.*` pattern). **The
   default is a mechanical fallback, not a working deployment:**
   the chart image carries no experiment dir, so any `*_command`
   that references `${EDEN_EXPERIMENT_DIR}` (the normal case —
   the fixture's commands all do) REQUIRES the effective image
   for that role to be an experiment image carrying
   `workers.experimentDir`. The chart cannot inspect image
   contents, so the gate is a bundled **host startup check**:
   when `--mode subprocess` or `--mode pod-exec` is set and
   `--experiment-dir` does not exist (or is empty), the host
   exits non-zero with a message naming the expected path and
   the image-build pattern — a loud CrashLoopBackOff at the
   right pod instead of a silent per-task failure loop. The
   README + `docs/deployment/helm.md` state the requirement; the
   smokes build the fixture experiment image rather than leaning
   on the fallback. For
   pod-exec, `workers.<role>.podExec.image` selects the per-task
   pod's image, defaulting to `workers.<role>.image` then the
   chart image. The 13b minimum-surface contract for pod-exec
   images carries over: `/bin/sh`, `git` ≥ 2.20, `python3`,
   `ca-certificates`, SHOULD run as `eden:1000`; distroless /
   `FROM scratch` are unsupported (static-wrapper follow-up,
   §11). Experiment-dir-as-ConfigMap was considered and rejected:
   it caps at ~1 MiB, can't carry deps, and would be a second
   delivery mechanism for the same bytes the image already
   carries.

4. **Worktrees on emptyDir; per-replica clone PVCs stay; uid
   story per mode.** Subprocess-mode executor/evaluator hosts
   pass `--worktrees-dir /var/lib/eden/worktrees` backed by an
   **emptyDir** (per-task scratch; pod restart = clean slate; the
   binding's §6 per-hostname isolation still applies since pod
   names are unique). The bare clone stays on the existing
   per-replica `repo` PVC; `git worktree add` across the two
   filesystems is a plain gitlink and is the same shape the
   Compose stack runs (named volume + bind mount). **uid:** in
   subprocess mode the user command runs in the same container
   as the host (uid 1000, `fsGroup: 1000` already set), so the
   DooD `dubious ownership` trap is structurally absent. In
   pod-exec mode the init container (chart image, eden:1000)
   populates the emptyDir and the main container (experiment
   image) reads it: the pod template sets
   `securityContext.fsGroup: 1000`, and the image contract
   SHOULD pin `eden:1000` (Decision 3); a root-running
   experiment image is the operator's deviation and the README
   documents the `safe.directory` consequence.

5. **Shared artifacts PVC** (§1.4, option b): standalone
   chart-managed PVC; web-ui rw; ideator rw / executor ro /
   evaluator ro mounts rendered only when that role's mode is
   `subprocess` or `podExec`; per-task pods mount it ro at the
   same `/var/lib/eden/artifacts` target so
   `task.json.content_path` resolves identically in every view
   (the DooD same-path discipline); task-store-server gains ro
   mount + `--artifacts-dir` when any worker mode is
   non-scripted, wiring `EDEN_ARTIFACT_URL`. Default RWO,
   documented RWX requirement for multi-node. Retired by #290.

6. **Subprocess env isolation — bundled host-code change.**
   Today `spawn()` merges the constructed child env *over* the
   host's full `os.environ`
   ([`subprocess_runner.py`](../../reference/services/_common/src/eden_service_common/subprocess_runner.py)
   line ~230). On the chart, `envFrom: secretRef` puts every
   deployment secret in the host's environ, so subprocess mode
   as-is would hand `EDEN_ADMIN_TOKEN` + the Postgres + Forgejo
   credentials to user code. Two-layer fix, both in this chunk:
   - **Host code (all substrates):** `spawn()` gains
     `inherit_env: bool`; subprocess-mode spawns pass
     `inherit_env=False`, and the child env becomes a minimal
     process baseline allowlist (`PATH`, `HOME`, `LANG`,
     `LC_*`, `TZ`, `TMPDIR`) + the constructed
     `_build_subprocess_env` dict (EDEN\_\* vars + env-file
     keys). The intended contract was already this — user env
     arrives via `--*-env-file`, and the DooD wrap already
     forwards only constructed keys (`-e KEY`) — so this aligns
     host mode with the documented binding rather than changing
     the contract. The binding doc §1.1 is amended to state it.
     Compose subprocess smokes re-validate (the fixture commands
     use only EDEN\_\* vars).
   - **Chart (defense in depth):** worker-host pod templates
     replace whole-Secret `envFrom` with explicit `env:` entries
     for exactly the keys each host consumes
     (`EDEN_ADMIN_TOKEN` for argv, `FORGEJO_REMOTE_PASSWORD`
     for the credential helper, `EDEN_READONLY_STORE_URL` for
     the substrate forward). `POSTGRES_PASSWORD`,
     `EDEN_STORE_URL`, Forgejo's internal secrets, and the
     web-ui session secret never enter a worker-host pod.
   `EDEN_WORKER_CREDENTIAL` **is** forwarded to user code by
   design (the §9 substrate bearers need it); `EDEN_ADMIN_TOKEN`
   never is.

7. **Pod-exec child lifecycle: three nested safety layers.**
   The DooD `post_kill_callback` analog must answer "who deletes
   the Job when the host dies at each badness level":
   - **Graceful (SIGTERM, rolling update):** the host's
     per-task `finally` deletes its in-flight Job
     (`propagation_policy="Background"`), the direct analog of
     the DooD cleanup callback.
   - **Pod deletion (eviction, spec-change rollout, node
     drain):** every Job carries an `ownerReference` to the
     host's own Pod (name/uid via the downward API,
     `blockOwnerDeletion: false`), so k8s garbage-collects
     orphaned Jobs when the owning Pod object goes away — no
     reaper invocation needed.
   - **In-place container restart (OOM-kill, panic):** the Pod
     object survives (same uid), so ownerReference does NOT
     fire; the restarted host's **startup reaper** (the
     `reap_orphaned_containers` analog) deletes Jobs labeled
     `eden.host=<own pod name>` whose task is no longer
     `claimed`. StatefulSet pod names are stable across
     restarts, so the label filter is exact.
   **Claim expiry is part of this chunk, not an assumption.**
   The recovery story above leans on the claim-TTL sweeper, but
   the worker hosts today call `store.claim(...)` with no
   `expires_at` (the wire client already supports it —
   [`client.py`](../../reference/packages/eden-wire/src/eden_wire/client.py)
   `claim(..., expires_at=…)`), so a wedged or in-place-restarted
   host would leave its task claimed forever. 13f bundles a
   `--claim-ttl-seconds` flag on the executor + evaluator hosts:
   in pod-exec mode the chart ALWAYS sets it, derived as
   `podExec.activeDeadlineSeconds` + a fixed watch/submit slack
   (one derived value, so the two knobs cannot silently invert);
   in subprocess mode it is optional and defaults to unset
   (Compose behavior unchanged). Unit tests cover claim-with-
   expiry and sweeper recovery after a simulated host wedge.
   `activeDeadlineSeconds` bounds a runaway pod that outlives
   its claim. Unique-per-attempt Job names
   (`eden-execute-<task-prefix>-<nonce>` /
   `eden-evaluate-<task-prefix>-<nonce>`, 63-char-safe) avoid
   the reclaimed-task-id collision the DooD cidfile uuid solves;
   labels carry the full `task_id`.

8. **stdin semantics.** Per-task pods never attach stdin — the
   DooD `-i` trap (closed stdin makes `docker run -i` exit
   early) has no pod-exec analog because nothing attaches to the
   pod at all; the Job pod's only channels are its filesystem
   (init-provisioned), env, and stdout/stderr. The ideator's
   attached-stdin requirement is satisfied by subprocess mode
   in-pod (same-container pipe), which is part of why pod-exec
   excludes the ideator (Decision 1).

9. **RBAC is opt-in, namespace-scoped, and rendered only when
   pod-exec is on.** A `ServiceAccount` + `Role` + `RoleBinding`
   render only when `workers.executor.mode == "podExec"` or
   `workers.evaluator.mode == "podExec"` (and
   `workers.rbac.create`, default true, for operators with
   GitOps-managed RBAC). Role verbs: `batch/v1 Job`
   create/get/list/watch/delete; `Pod` get/list/watch;
   `pods/log` get — nothing else, release namespace only. This
   is the AGENTS.md privilege-overlay rule translated to Helm:
   Compose isolates new privilege in a separate overlay *file*;
   the chart's equivalent is a conditional template gated on the
   opt-in mode value, and the default (`scripted` everywhere)
   renders **zero** RBAC objects — asserted mechanically in
   `helm-lint` (§6.1) and at runtime in the pod-exec smoke
   (§6.3), mirroring how `smoke-subprocess.sh` asserts the
   docker socket is absent in host mode.

10. **Per-task pods confine the Forgejo credential to non-user
    containers; user-container env is explicit-only.** The task
    pod has up to three containers with distinct trust levels:
    - **init (provision)** — chart image; clones from Forgejo +
      creates the worktree + writes `task.json`. Carries
      `FORGEJO_REMOTE_PASSWORD` + the credential-helper mount.
      Runs and exits before user code starts; container env is
      not readable across container boundaries.
    - **main (user command)** — experiment image; wrapper runs
      the user's `*_command`. Env is exactly the host's
      constructed `_build_subprocess_env` dict (never the admin
      token — Decision 6) plus `EDEN_EXECUTION_COMMAND` /
      `EDEN_EVALUATION_COMMAND`. **No Forgejo credential, no
      credential-helper mount** — the clone it works in has no
      stored credentials (the helper supplies them on demand and
      only the init/publisher containers mount it).
    - **publisher (executor pods only)** — chart image; performs
      the `work/*` push after user code exits (§3.4) and emits
      the outcome sentinel. Carries the Forgejo credential.
    On Compose, the push always ran host-side, so user code
    never held the Forgejo credential; this split preserves that
    property on k8s (13b's wrapper-push would have broken it).
    There is no environment inheritance channel at all in
    pod-exec. Remaining caveat documented in the binding: the
    user container's env values sit in the Job spec, readable by
    anyone with `jobs get` in the namespace — the same soft
    boundary as DooD's `docker inspect` (binding §7.5); routing
    the worker credential through a per-task Secret is a named
    follow-up (§11), not v0.

11. **Repeatable `--*-env-file` — bundled CLI change.** Compose
    feeds the ideator a deployment-generated env file
    (`EDEN_BASE_COMMIT_SHA`, `EDEN_IDEAS_PER_IDEATION`) and lets
    operators extend the executor/evaluator files with API keys.
    On the chart those two sources are different objects (a
    chart-rendered ConfigMap vs an operator Secret), so the
    single-value `--ideation-env-file` / `--execution-env-file`
    / `--evaluation-env-file` flags become repeatable
    (`action="append"`; later files win on key conflict; the
    reserved-substrate-key strip still runs after the merge).
    The chart mounts the chart-managed env-file ConfigMap always
    (when subprocess/podExec) and the operator's
    `workers.<role>.envFileSecret` additionally when set. This
    is the "substrate forces a CLI surface that doesn't exist →
    bundle the small impl change" pitfall pattern, same as 13a's
    quiescence sentinel.

12. **Chart surface: a `workers:` block replaces
    `config.workerMode`.** Greenfield rename, no alias: the
    scripted default renders byte-identical worker pods modulo
    the Decision 6 env-entry narrowing, so existing
    deployments' values files change one key. Schema gates:
    `workers.ideator.mode` enum excludes `podExec`;
    `workers.<role>.image` both-or-neither; `podExec.*` values
    are validated for shape (objects/arrays, GPU keys NOT
    enumerated — vendor-agnostic pass-through per 13b Decision
    6). 13f introduces no value the schema must `minLength`-gate
    — every new value has a mechanical default (chart-image
    fallbacks, conventional paths). The one operator-required
    input the schema *cannot* see — the experiment image
    actually containing `workers.experimentDir` — is enforced by
    the Decision 3 host startup check instead, keeping the 13a
    "fail loud, never fictional" posture with the loud failure
    moved to the earliest layer that can detect it.

### 3.2 Subprocess mode per role (chart wiring)

Each worker-host template grows a mode conditional on
`workers.<role>.mode`. Args/mounts delta vs scripted, mirroring
the Compose overlay flag-for-flag:

- **ideator-host** (`Deployment`, stays `Recreate`): adds
  `--mode subprocess --experiment-config
  /etc/eden/experiment-config.yaml --experiment-dir
  {{ workers.experimentDir }} --artifacts-dir
  /var/lib/eden/artifacts --repo-path /var/lib/eden/clone/repo
  --forgejo-url … --credential-helper … --ideation-env-file
  /etc/eden/ideation.env [--ideation-env-file
  /etc/eden/ideation-extra.env]`. New mounts: shared artifacts
  PVC (rw), a **new per-replica repo volume** — the ideator has
  no clone PVC today; subprocess mode adds an emptyDir-backed
  clone (the ideator's clone is a cache, re-cloned on pod
  restart; a PVC would be the only Deployment-attached RWO PVC
  in the chart and `Recreate` makes emptyDir re-clone safe and
  simpler), experiment-config ConfigMap, chart-rendered
  ideation-env ConfigMap (`EDEN_BASE_COMMIT_SHA={{
  experiment.baseCommitSha }}`, `EDEN_IDEAS_PER_IDEATION`),
  optional operator env-file Secret. Substrate env
  (`EDEN_REPO_DIR`, `EDEN_ARTIFACT_URL`,
  `EDEN_ARTIFACT_PATH_ROOT`, `EDEN_READONLY_STORE_URL`) set as
  explicit pod env exactly as the Compose overlay does.
- **executor-host** (`StatefulSet`): adds `--mode subprocess
  --experiment-config … --experiment-dir … --worktrees-dir
  /var/lib/eden/worktrees --execution-env-file …`. New mounts:
  worktrees emptyDir, artifacts PVC (ro), experiment-config
  ConfigMap, env-file sources. Repo clone PVC unchanged.
- **evaluator-host** (`StatefulSet`): symmetric; its
  already-reserved clone PVC becomes load-bearing.

The worker pod image resolves through
`workers.<role>.image → image.*`. Decision 6's explicit `env:`
narrowing applies to all three templates in the same wave.

### 3.3 Pod-exec mode: shared module + chart wiring

New `eden_service_common/pod_exec.py` (the `container_exec.py`
sibling; `kubernetes` client imported lazily so non-k8s services
don't pay the import) owning: Job-manifest build
(role-parameterized), Job watch with the duplicate-Pod selection
read, sentinel-line log parse, image-pull-stall detection,
`delete_job`, and `reap_orphaned_jobs(role, host, namespace)`.
Per-role `pod_exec_mode.py` in the executor and evaluator hosts
drives the chapter-3 flow through it. The `kubernetes` dep
(`>=29,<31`) is declared on the two host packages; in-cluster
config via `load_incluster_config` with `load_kube_config`
fallback for tests.

The executor StatefulSet, when `mode: podExec`, passes `--mode
pod-exec --pod-namespace {{ .Release.Namespace }} --pod-image …
--pod-service-account … --pod-template-config /etc/eden/
pod-template.json --pod-active-deadline-seconds …` etc., runs
under the chart-managed ServiceAccount, and keeps its repo clone
PVC (reachability validation stays host-side). The
`pod-template.json` ConfigMap carries the rendered
nodeSelector/tolerations/resources/env. The host pod itself
needs no artifacts mount in pod-exec mode (content resolution
happens in the task pod).

### 3.4 Per-task sequence (executor; evaluator deltas in §3.5)

1. **Host:** claim **with `expires_at` from
   `--claim-ttl-seconds`** (Decision 7) → pre-Phase-1
   ref-collision guard → `Store.create_variant(status="starting")`
   (chapter 3 §3.2 step 1 — before the Job exists, because the
   Job pod's push is an observable repo write).
2. **Host:** build + create the Job: labels
   (`eden.role/host/task_id/experiment_id`), ownerReference to
   own Pod, `backoffLimit: 0`, `restartPolicy: Never`,
   `activeDeadlineSeconds`, `ttlSecondsAfterFinished: 60`;
   volumes: work emptyDir, wrapper ConfigMap,
   git-credential-helper ConfigMap (mounted into init +
   publisher only — Decision 10), artifacts PVC (ro);
   `fsGroup: 1000`. Three containers:
   - **init (provision; chart image, Forgejo credential):**
     bare-clone from Forgejo into the emptyDir,
     `git worktree add --detach` at `parent_commits[0]`, write
     `.eden/task.json` (including `content_path`, which resolves
     against the ro artifacts mount at the same path the host
     computed).
   - **main (user command; pod-exec image, no credential):**
     `command: ["/bin/sh", "/etc/eden/wrapper.sh"]` — runs the
     user command with cwd=worktree (the command string arrives
     via `EDEN_EXECUTION_COMMAND` env, not argv — shell-quoting
     fidelity), then writes the done-marker
     `.eden/command-exited` (carrying the exit code) and exits.
     It never touches refs and never emits the sentinel.
   - **publisher (chart image, Forgejo credential):** a plain
     python entrypoint (`eden_executor_host.pod_publish`, using
     `eden_git` — not POSIX sh) that waits for the done-marker
     (bounded by the pod's remaining deadline), reads
     `outcome.json`, and on `status=success` runs local
     `update-ref` + `push_ref` with the **three-outcome
     read-back ladder** the repo mandates for ambiguous pushes
     (AGENTS.md / `GitRepo.push_ref`): `RefRefused` → rollback
     local, outcome `error`; `GitTransportError` → immediate
     `ls_remote(branch)`: remote == our SHA → confirmed success;
     remote divergent or absent → rollback local, `error`;
     `ls_remote` also fails → rollback local, `error`. The
     indeterminate branch collapses to the executor's chapter-3
     §3.3 failure status — never silently to success. A remote
     `work/*` ref leaked by the indeterminate branch (push
     landed, read-back failed) is benign-but-orphaned: the
     variant terminalizes as `error`, no integrator ever reads
     the ref, and the existing admin work-refs GC page is the
     cleanup surface (§7 "indeterminate-push residue"). Last
     stdout line: `EDEN_OUTCOME <json>` reflecting the
     *post-push* outcome.
3. **Host:** watch the Job at `--pod-poll-interval`; the
   image-pull deadline gives up early on
   `ImagePullBackOff`/`ErrImagePull`/`InvalidImageName`. On
   terminal: select the outcome pod per the duplicate-Pod rule
   (13b §8.10, adopted verbatim — partition by phase, prefer
   the Succeeded pod, require byte-equal outcomes if two
   Succeeded, fail closed otherwise), read the **publisher
   container's** log (`pods/log` takes a container name), parse
   the **last** sentinel line.
4. **Host:** validate — outcome shape, then `fetch_ref(branch)`
   plus `is_ancestor(parent, commit_sha)` for every parent against
   the host's clone (chapter 3 §3.3). Any failure → the role's
   chapter-3 failure status (executor: `error`).
5. **Host:** existing submit with retry-before-orphan +
   read-back; **finally** delete the Job.

The chart derives `--claim-ttl-seconds` as
`activeDeadlineSeconds` + watch/submit slack so the sweeper
never reclaims a task whose Job is alive (Decision 7).

Container ordering note: `initContainers` guarantee init runs
first; `main` and `publisher` are both regular containers and
start concurrently — the publisher's done-marker wait is the
ordering mechanism (same shared-emptyDir signaling the DooD wrap
gets for free from process order). The Job is terminal only when
both containers exit; the publisher always exits 0 (the outcome
travels in the sentinel, not the exit code), so Job-level
`failed` is reserved for infrastructure failures (deadline,
eviction, image pull), keeping the §8.10-adopted classification
clean.

### 3.5 Evaluator pod-exec deltas

Symmetric, minus the push — so **no publisher container**: init
(chart image, credential) fetches `variant.branch` and adds the
worktree at `variant.commit_sha` and writes
`.eden/eval-task.json`; main (pod-exec image, no credential)
runs `EDEN_EVALUATION_COMMAND` via the wrapper, never touches
refs, and emits the evaluation outcome sentinel itself as its
last stdout line (nothing credentialed has to happen after user
code, so the sentinel can live in main; the host reads the
**main** container's log for evaluator Jobs). Host-side
validation runs `Store.validate_evaluation` against
`evaluation_schema`; transport-shaped failures map to
`evaluation_error` (chapter 3 §4.4), per-role per the AGENTS.md
transport-failure-mapping pitfall — there is no shared
"transport failed" status. Job names use `eden-evaluate-…`.

### 3.6 Wrapper script + publisher entrypoint

The **wrapper** is one POSIX-sh `wrapper.sh` shipped in a chart
ConfigMap (the authoritative copy, mounted at
`/etc/eden/wrapper.sh` and used as `command:`, overriding any
image entrypoint) and baked into the reference image for
non-Helm invocation — the dual-shipping pattern 13b Decision 8
chose, same as the credential-helper ConfigMap.
Role-parameterized via env
(`EDEN_POD_EXEC_ROLE=executor|evaluator`) rather than two
near-identical scripts. With publishing moved out (Decision 10),
the wrapper's job shrinks to: cd worktree, run the user command,
write the done-marker (executor) or emit the outcome sentinel
(evaluator) — it needs only `/bin/sh`, trimming the 13b
minimum-surface contract (`python3` is no longer required in the
*user* image for the executor path; the evaluator wrapper still
needs no JSON parsing because it forwards `outcome.json`'s bytes
into the sentinel verbatim and the host does the parsing).

The **publisher** (`eden_executor_host.pod_publish`, executor
pods only) is real python running in the chart image — it gets
`eden_git`'s typed `RefRefused` / `GitTransportError` +
`ls_remote` for the §3.4 read-back ladder for free, instead of
reimplementing atomicity logic in shell.

Sentinel-spoofing defense: the host parses only the **last**
`EDEN_OUTCOME`-prefixed line of the *sentinel container's* log —
for executor Jobs that container is the publisher, which user
code cannot write to at all; for evaluator Jobs it is main, where
the wrapper emits after the user command has exited, so the last
line wins. Per-Job random nonce hardening is a named follow-up.

New informative

### 3.9 Alternatives considered (beyond those adopted from 13b)

- **`--exec-mode k8s` riding the subprocess flow** (the literal
  DooD shape: host creates the worktree, spawn is wrapped):
  rejected — there is no shared filesystem between the host pod
  and an arbitrarily-scheduled task pod, so the host-side
  worktree + host-side push that define the subprocess binding
  can't be preserved; pretending pod-exec is a spawn wrapper
  would misdescribe the binding. It is a distinct `--mode` with
  its own binding doc.
- **Sidecar container per worker pod for user code:** rejected
  — k8s has no exec-into-sibling-container primitive a host
  could use as a spawn channel short of `pods/exec` RBAC (an
  interactive API, bigger privilege than `jobs create`), and it
  still couldn't do per-task images or GPU placement.
- **DooD-on-k8s via docker-socket hostPath / sysbox / Kata:**
  rejected for the reference chart — node-runtime-specific,
  privileged, and exactly what #291 calls non-portable. A
  hardened-isolation chunk may revisit (§11).
- **Reaper-only lifecycle (13b's shape) without
  ownerReferences:** rejected — a deleted host Pod whose
  replacement never schedules (scale-down, quota) would leak
  Jobs forever; ownerReference GC covers that branch for free
  and costs one downward-API env pair.

## 7. Tricky areas

Carried forward from 13b (still load-bearing, adopted with the
substrate): Job-name collision under task reclaim (per-attempt
nonce); image-pull stalls vs activeDeadline (early give-up);
pod-log read vs TTL GC race (read immediately on terminal;
404 → role failure status); user-emitted sentinel spoofing
(last-line parse); RBAC blast radius (namespace Role; an
executor-host compromise can run arbitrary Jobs in-namespace —
same soft boundary as the DooD socket, documented); rolling
update vs in-flight Job (finally-delete + reaper + ownerRef);
`kubernetes` client version pin; wrapper POSIX-sh portability.

New in 13f:

- **Artifacts PVC migration is a substrate migration** — the
  AGENTS.md same-PR-audit rule applies: grep every consumer of
  the web-ui artifacts `volumeClaimTemplate` name and
  `/var/lib/eden/artifacts` in chart templates, smokes, and
  `docs/deployment/helm.md` in the wave that moves it; a stale
  reference fails silent (web-ui serving an empty dir), not
  loud. Existing deployments lose nothing by policy: the chunk
  documents that the rename orphans the old
  `artifacts-<release>-web-ui-0` PVC (operator deletes it
  manually; greenfield posture, no migration script).
- **Decision 6 changes child-env behavior on Compose too.** A
  Compose operator who relied on host-environ inheritance
  (instead of `--*-env-file`) for API keys would break. The
  binding doc never promised inheritance and the overlays ship
  env-file plumbing, so this is contract-alignment, but the
  CHANGELOG entry must call it out loudly.
- **emptyDir worktrees vs huge repos:** emptyDir draws from the
  node's ephemeral storage; a large experiment repo can evict
  the pod. `workers.<role>.podExec` and the subprocess worktree
  volume accept an optional `sizeLimit`; README notes the
  failure shape (Evicted pod → Job failed → variant `error`).
- **Init-container clone cost per task:** a full bare clone per
  task pod is O(repo) network from Forgejo per execution.
  Acceptable at fixture scale; named follow-up for a
  shallow/reference-clone optimization (§11) rather than
  premature cleverness now.
- **Claim-TTL vs activeDeadlineSeconds coupling:** the chart
  derives the host's `--claim-ttl-seconds` from
  `podExec.activeDeadlineSeconds + slack` rather than exposing
  two independently-settable values that can silently invert.
  Note the hosts today claim with **no** expiry at all — the
  Decision 7 plumbing is what makes the sweeper-recovery story
  real, and the pod-exec smoke implicitly depends on it.
- **Indeterminate-push residue:** the publisher's
  "push-landed-but-read-back-failed" branch terminalizes the
  variant as `error` while the `work/*` ref may exist on
  Forgejo. That residue is inert (no integrator reads a ref for
  an `error` variant) and visible/cleanable in the existing
  admin work-refs GC page; the binding documents it rather than
  inventing a new reconciler for an unreachable-by-construction
  ref class.
- **`fsGroup` + RWX (EFS):** EFS access points ignore fsGroup
  semantics differently than block storage; the README's EKS
  notes must include the EFS CSI `gid`/access-point guidance —
  doc-only, but the kind smoke can't catch it, so it needs the
  fresh-operator walkthrough at impl time.

## 8. Risks

1. **kind resource pressure** — two more multi-service smokes;
   mitigate with single-replica everything, pre-loaded images,
   and path-filtered job triggers (existing `helm` bucket).
2. **`kubernetes` client / cluster API drift** — pin major;
   README documents the supported range; mocked-client unit
   tests don't catch wire-shape drift, the smoke does.
3. **Schema complexity creep** in `values.schema.json` if/then
   chains — keep per-role gates flat; `helm-lint` rejection
   tests pin the behavior.
4. **Env-isolation false confidence**: the chart-side narrowing
   is testable in CI; the host-side allowlist is unit-tested;
   neither proves a future template doesn't re-add `envFrom` —
   add a `helm-lint` grep asserting worker-host manifests
   contain no `envFrom` (cheap mechanical backstop).
5. **Subsumption hygiene**: archiving 13b + closing #172 risks
   dropping a deferral on the floor — §11 enumerates the
   carried-forward set explicitly; the chunk-completion review
   checks each has an issue per the deferral-tracking rule.

## 12. What lands at the end of 13f

An operator with an EKS cluster builds one experiment image
`FROM <registry>/eden-reference:<tag>`, sets

```yaml
workers:
  ideator:   { mode: subprocess, image: { repository: ..., tag: ... } }
  executor:  { mode: podExec,    podExec: { image: { repository: ..., tag: ... },
               nodeSelector: { eks.amazonaws.com/nodegroup: gpu },
               resources: { limits: { nvidia.com/gpu: "1" } } } }
  evaluator: { mode: podExec,    podExec: { image: { repository: ..., tag: ... } } }
storage:
  artifactsAccessModes: ["ReadWriteMany"]   # EFS StorageClass
```

and runs `setup-experiment-helm.sh` — and their real
`ideation_command` / `execution_command` / `evaluation_command`
drive the experiment end-to-end on Kubernetes: the ideator
holding LLM context in its own pod, every execution and
evaluation running as a GPU-schedulable per-task Job, no docker
socket anywhere, and no deployment secret reachable from user
code. Compose remains the local-development path, byte-for-byte
unchanged except the env-isolation contract alignment.
