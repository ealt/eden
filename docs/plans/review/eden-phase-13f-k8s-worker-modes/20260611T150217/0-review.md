**Missing Context**

Assessment: mostly well-defined. The plan gives enough substrate and predecessor context to understand #291.

One gap: the experiment image requirement is softer than the design needs. The plan says non-scripted worker images default to the chart image and that 13f adds “no operator-required-with-no-default values” ([plan:292](/Users/ericalt/Documents/eden-worktrees/plan-issue-291-k8s-worker-modes/docs/plans/eden-phase-13f-k8s-worker-modes.md:292), [plan:466](/Users/ericalt/Documents/eden-worktrees/plan-issue-291-k8s-worker-modes/docs/plans/eden-phase-13f-k8s-worker-modes.md:466)), but the fixture commands run files under `${EDEN_EXPERIMENT_DIR}` and the base chart image does not contain an experiment dir. The plan should state explicitly that enabling `subprocess`/`podExec` requires either the global chart image or the role/podExec image to be an experiment image containing `workers.experimentDir`.

**Feasibility**

Assessment: not ready to implement as written. I found three load-bearing gaps.

1. The pod-exec Forgejo credential story contradicts the security claim. The Job env is specified as exactly `_build_subprocess_env` plus the command env ([plan:427](/Users/ericalt/Documents/eden-worktrees/plan-issue-291-k8s-worker-modes/docs/plans/eden-phase-13f-k8s-worker-modes.md:427)), but the task pod must clone and push to Forgejo ([plan:598](/Users/ericalt/Documents/eden-worktrees/plan-issue-291-k8s-worker-modes/docs/plans/eden-phase-13f-k8s-worker-modes.md:598), [plan:607](/Users/ericalt/Documents/eden-worktrees/plan-issue-291-k8s-worker-modes/docs/plans/eden-phase-13f-k8s-worker-modes.md:607)). The existing credential helper reads `FORGEJO_REMOTE_PASSWORD` from env ([git-credential-helper-configmap.yaml:1](/Users/ericalt/Documents/eden-worktrees/plan-issue-291-k8s-worker-modes/reference/helm/eden/templates/git-credential-helper-configmap.yaml:1)). Either the task pod gets a deployment secret, making “no deployment secret reachable from user code” false ([plan:1045](/Users/ericalt/Documents/eden-worktrees/plan-issue-291-k8s-worker-modes/docs/plans/eden-phase-13f-k8s-worker-modes.md:1045)), or clone/push cannot authenticate. The plan needs a concrete design here: accept and document the secret exposure, or split user execution from Forgejo publishing so arbitrary user code never gets the Forgejo credential.

2. The task-pod push path misses the repo’s required read-back ladder. The plan says the wrapper does `git push` and compensating local delete on failure, but does not include `ls_remote` classification ([plan:607](/Users/ericalt/Documents/eden-worktrees/plan-issue-291-k8s-worker-modes/docs/plans/eden-phase-13f-k8s-worker-modes.md:607)). AGENTS.md explicitly requires confirmed-success / confirmed-divergence / indeterminate handling for ambiguous `git push` failures ([AGENTS.md:154](/Users/ericalt/Documents/eden-worktrees/plan-issue-291-k8s-worker-modes/AGENTS.md:154)), and `GitRepo.push_ref` documents transport failure as ambiguous and requiring `ls_remote` disambiguation ([repo.py:654](/Users/ericalt/Documents/eden-worktrees/plan-issue-291-k8s-worker-modes/reference/packages/eden-git/src/eden_git/repo.py:654)). Put that ladder in the wrapper before emitting the sentinel, or move publishing back to a host-side component that can do it.

3. The recovery story depends on claim expiry, but worker-host claims currently have no expiry. The plan says claim TTL is configured greater than `activeDeadlineSeconds` and derived by the chart ([plan:627](/Users/ericalt/Documents/eden-worktrees/plan-issue-291-k8s-worker-modes/docs/plans/eden-phase-13f-k8s-worker-modes.md:627), [plan:927](/Users/ericalt/Documents/eden-worktrees/plan-issue-291-k8s-worker-modes/docs/plans/eden-phase-13f-k8s-worker-modes.md:927)). Existing worker hosts call `store.claim(...)` without `expires_at` ([executor subprocess:197](/Users/ericalt/Documents/eden-worktrees/plan-issue-291-k8s-worker-modes/reference/services/executor/src/eden_executor_host/subprocess_mode.py:197), [evaluator subprocess:159](/Users/ericalt/Documents/eden-worktrees/plan-issue-291-k8s-worker-modes/reference/services/evaluator/src/eden_evaluator_host/subprocess_mode.py:159)), though the wire client supports it ([client.py:417](/Users/ericalt/Documents/eden-worktrees/plan-issue-291-k8s-worker-modes/reference/packages/eden-wire/src/eden_wire/client.py:417)). For pod-exec, that is load-bearing: if the host container restarts in-place, ownerReference GC does not fire, and without claim expiry the task can remain claimed forever. The plan should add explicit pod-exec claim-expiry plumbing and tests.

I’m skipping Alternatives, Completeness, and Edge Cases for now because the feasibility issues above change the shape of the design.

Overall: strong context and a plausible direction, but the plan needs another revision before implementation. The main unresolved question is where Forgejo publishing authority lives relative to untrusted user code; once that is settled, the push read-back and claim-expiry fixes are straightforward plan amendments.
