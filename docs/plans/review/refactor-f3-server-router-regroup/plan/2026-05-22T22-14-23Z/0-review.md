If route order is an invariant, the plan currently violates its own rule.
- `Must-fix:` The proposed regression test in [§6.2](</Users/ericalt/Documents/eden-worktrees/issue-115-server-router-regroup-plan/docs/plans/refactor-f3-server-router-regroup.md:812>) uses `/tasks`, which is before `experiment` in both the current file and the planned order. It would not catch the only disputed ordering edges (`/events`, `/workers`, `/groups`).

If you keep a test, make it hit `/events` or `/workers`, not just `/tasks`.

**Criterion 8 — Wave Robustness**
- `Must-fix:` The one-function-per-module `build_router(deps) -> APIRouter` shape in [Decision 1](</Users/ericalt/Documents/eden-worktrees/issue-115-server-router-regroup-plan/docs/plans/refactor-f3-server-router-regroup.md:146>) and [§3.3](</Users/ericalt/Documents/eden-worktrees/issue-115-server-router-regroup-plan/docs/plans/refactor-f3-server-router-regroup.md:461>) likely recreates the Tier-1 function-length violation. `tasks.py` is estimated at ~250 LOC in [§3.1](</Users/ericalt/Documents/eden-worktrees/issue-115-server-router-regroup-plan/docs/plans/refactor-f3-server-router-regroup.md:370>), while the gate fails at function length `> 100` in [check-complexity.py](</Users/ericalt/Documents/eden-worktrees/issue-115-server-router-regroup-plan/scripts/check-complexity.py:55>). - Concrete edit: change the design so `build_router` stays tiny. For example, use per-route handler factories or per-route `_register_*` helpers, and add an explicit budget line to the plan: no `build_router` or registration helper may exceed 100 LOC. Wave 1 should explicitly run `python3 scripts/check-complexity.py` against the new router module, not just against `server.py`.
