**Feasibility**

The plan is materially better. The auth-boundary split, tagged `Task.target`, and explicit credential reissue close several round-0 problems. I don’t think feasibility is fully clear yet, though; I’d stay at this level for one more round.

1. The `submit` ownership check still has no single, coherent enforcement point. [D.0](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:68) says auth is binding-only and Store signatures stay as-is, but [D.6](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:411) requires `authenticated_worker_id == task.claim.worker_id`, and the storage worklist still treats `WrongClaimant` as a store-layer concern in [§4.1](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:461) and [§5.3](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:531). The current `Store.submit` shape still has no worker identity input in [protocol.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/protocol.py:209). Suggestion: either add `worker_id` to `Store.submit`, or move claimant-matching and `WrongClaimant` entirely into the binding layer and stop specifying it as a store semantic.

2. The startup credential-verification story is still internally inconsistent and uses the wrong probe. [D.1](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:154) says startup should verify an existing credential via `read_worker(self_id)`, but [D.1 read semantics](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:130) describe `read_worker` / `list_workers` as experiment-level reads, so they are not a reliable auth check. Separately, [§8.2](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:699) still says failed verification falls through to fresh register, which contradicts the new `reissue_credential` escalation path in [D.1](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:159). Suggestion: standardize on one canonical op name (`reissue_credential` vs `update_worker_credential`) and use `verify_worker_credential` or a dedicated authenticated ping as the startup check.

3. `admin` is still a valid `worker_id`, but D.5 also uses it as the auth sentinel. The grammar in [D.1](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:86) allows `admin`, while [D.5](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:307) reserves `Authorization: Bearer admin:<token>` for admin auth. That makes a legitimate worker named `admin` ambiguous or unrepresentable. Suggestion: reserve `admin` in the identifier grammar, or change the wire encoding so principal type is tagged out of band.

4. The web-ui sign-in strategy reintroduces app/session coupling by rotating credentials on every login. The shipped choice in [D.5b](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:366) calls `reissue_credential(worker_id)` on every sign-in and explicitly invalidates the prior session’s credential in [D.5b](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:370). That means a second sign-in as the same worker can strand an active claim in the first session, and it undercuts the cross-application claim story the plan still asserts in [§6.4](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:636). Suggestion: either accept that limitation explicitly and scope the UI path as non-cross-app in 12a-1, or avoid per-sign-in credential rotation for the reference UI.

**Overall Assessment**

This draft is much closer; the major round-0 structural issues are mostly addressed. I still wouldn’t move to alternatives/completeness yet, because the claimant-auth enforcement point and the session credential lifecycle are both still under-specified enough to cause implementation drift.
