**Missing Context**

Assessment: mostly sufficient. The problem statement, scope split, and upstream rationale are clear enough to review. The one context gap that needs to be made explicit before implementation is whether 12a-1 is intentionally moving authentication from a reference-only binding concern into the normative protocol contract. The plan assumes that shift in [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:201), but the current wire chapter still says auth is outside the binding in [07-wire-protocol.md](/Users/ericalt/Documents/eden/spec/v0/07-wire-protocol.md:204). That boundary decision drives the rest of the chunk.

**Feasibility**

Assessment: there are blocking feasibility issues here. I would stop at this level before evaluating alternatives or completeness.

1. The authenticated-worker model does not have a coherent path through the storage contract once claim tokens are removed. The plan says `Store.claim` performs authentication and `submit` authorizes by authenticated worker identity in [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:157), [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:255), and [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:309), but the actual store interface has no auth context on either operation beyond `worker_id` on claim and `token` on submit in [protocol.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/protocol.py:199) and [protocol.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/protocol.py:209). As written, the plan mixes wire-layer auth, store-layer auth, and transport-independent `Store` semantics without choosing which layer is authoritative. Suggestion: either keep auth as a wire-only precondition for this chunk, or change the normative store contract so claim/submit carry the authenticated actor explicitly across all bindings.

2. The web UI flow cannot satisfy the proposed identity rules as described. The plan says claim eligibility and submit attribution come from the authenticated worker id in [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:216) and [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:259), but it also says 12a-1 will only authenticate the web-ui process itself in [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:413). Today the UI signs users in as a session worker and claims with that session identity in [auth.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/auth.py:30) and [ideator.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/ideator.py:112). If the process authenticates only as `web-ui`, then either all human actions collapse to `web-ui` or the UI cannot submit on behalf of the signed-in worker at all. Suggestion: either make UI requests authenticate as the session worker in 12a-1, or defer browser/manual flows out of scope for this chunk.

3. `Task.target` is ambiguous because `worker_id` and `group_id` are both plain strings in the same field. The plan defines independent per-experiment uniqueness for workers and groups in [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:68) and [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:111), then uses an untagged `worker_id | group_id | null` target in [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:144). Claim-time enforcement in [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:163) assumes the implementation can tell which kind of identifier it is. That is not deterministic if both registries can contain the same string. Suggestion: either enforce a single shared namespace across workers and groups, or make `target` a tagged structure like `{kind, id}`.

4. Idempotent re-registration conflicts with the recovery story for lost credentials. The plan says re-registering an existing worker returns no new token in [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:98), but the startup logic later says that if the persisted credential is missing or invalid, the host should fall through to a fresh register in [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:551). Those two rules do not compose: a host that has lost its local token but still has a registry row cannot recover without an explicit reissue path. Suggestion: define a real credential-rotation/reissue operation and make startup recovery use that path intentionally.

5. The bearer format is underspecified because it relies on `:` splitting while `worker_id` is operator-supplied and opaque. The plan allows arbitrary operator-supplied worker ids in [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:68), then defines `Authorization: Bearer <worker_id>:<registration_token>` in [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:213). Unless worker ids are constrained to exclude `:`, auth parsing is ambiguous. Suggestion: either tighten the identifier grammar in the data model or use a structured transport format that does not depend on a delimiter inside an opaque id.

Because these are blocking feasibility issues, I would skip alternatives, completeness, and edge-case review for now.

**Overall Assessment**

The plan has a solid motivation and a sensible chunk boundary, but it is not implementation-ready yet. The core auth and identity model is internally inconsistent across the normative store contract, the wire binding, and the existing human-facing UI flow; that needs to be resolved before the rest of the plan can be evaluated usefully.
