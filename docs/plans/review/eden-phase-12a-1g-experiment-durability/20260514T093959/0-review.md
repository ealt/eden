**Findings**
1. The proposed spec text changes experiment lifetime semantics, not just durability, and it conflicts with the current spec. The plan’s draft §13 says lifetime continues until explicit operator termination and that the protocol defines no automatic termination mechanism [plan](/Users/ericalt/Documents/eden-worktrees/phase-12a-1g-experiment-durability/docs/plans/eden-phase-12a-1g-experiment-durability.md:59). But the current spec says experiments have termination conditions [01-concepts.md](/Users/ericalt/Documents/eden-worktrees/phase-12a-1g-experiment-durability/spec/v0/01-concepts.md:13) and that an experiment terminates per those configured conditions [01-concepts.md](/Users/ericalt/Documents/eden-worktrees/phase-12a-1g-experiment-durability/spec/v0/01-concepts.md:124). This is a load-bearing contradiction.
2. The Compose migration is not feasible with the stated scope. The plan says the subprocess and docker-exec overlays need no changes because they “inherit the new bind-mount shape transparently” [plan](/Users/ericalt/Documents/eden-worktrees/phase-12a-1g-experiment-durability/docs/plans/eden-phase-12a-1g-experiment-durability.md:187), but the current overlays still depend on named volumes: [compose.subprocess.yaml](/Users/ericalt/Documents/eden-worktrees/phase-12a-1g-experiment-durability/reference/compose/compose.subprocess.yaml:103), [compose.subprocess.yaml](/Users/ericalt/Documents/eden-worktrees/phase-12a-1g-experiment-durability/reference/compose/compose.subprocess.yaml:107), [compose.subprocess.yaml](/Users/ericalt/Documents/eden-worktrees/phase-12a-1g-experiment-durability/reference/compose/compose.subprocess.yaml:157), [compose.subprocess.yaml](/Users/ericalt/Documents/eden-worktrees/phase-12a-1g-experiment-durability/reference/compose/compose.subprocess.yaml:185), [compose.docker-exec.yaml](/Users/ericalt/Documents/eden-worktrees/phase-12a-1g-experiment-durability/reference/compose/compose.docker-exec.yaml:110), [compose.docker-exec.yaml](/Users/ericalt/Documents/eden-worktrees/phase-12a-1g-experiment-durability/reference/compose/compose.docker-exec.yaml:113), [compose.docker-exec.yaml](/Users/ericalt/Documents/eden-worktrees/phase-12a-1g-experiment-durability/reference/compose/compose.docker-exec.yaml:115), [compose.docker-exec.yaml](/Users/ericalt/Documents/eden-worktrees/phase-12a-1g-experiment-durability/reference/compose/compose.docker-exec.yaml:168). With only the base file changed, those paths will either stay on Docker-managed storage or silently mount fresh empty volumes.
3. The plan omits `eden-worktrees` entirely from the durability model. That is current Compose state in subprocess mode [compose.subprocess.yaml](/Users/ericalt/Documents/eden-worktrees/phase-12a-1g-experiment-durability/reference/compose/compose.subprocess.yaml:103) [compose.subprocess.yaml](/Users/ericalt/Documents/eden-worktrees/phase-12a-1g-experiment-durability/reference/compose/compose.subprocess.yaml:157). Either it is intentional scratch and should be explicitly excluded, or it also needs migration under `EDEN_EXPERIMENT_DATA_ROOT`.

**1. Missing Context**
Assessment: The operator problem is clear, but the plan is missing two pieces of context a reader needs before implementation.

- It does not reconcile “durable for its entire operational lifetime” with the existing spec’s configured termination conditions. That has to be spelled out before drafting normative text.
- It does not enumerate the full current Compose storage surface. The base named volumes are listed, but the overlay-only volumes and the docker-exec volume-forwarding contract are part of the current deployment shape too.

**2. Feasibility**
Assessment: Not feasible as written.

- For the spec, I would not ship the current §13 draft without first deciding whether durability lasts until configured terminal state, operator-declared terminal state, or both. If the chunk is meant to stay narrow, the safest shape is to anchor the invariant to the existing termination model rather than replacing it.
- For Compose, the plan needs either:
  - expanded scope that updates `compose.subprocess.yaml`, `compose.docker-exec.yaml`, and any exec-volume assumptions to use bind paths, or
  - an explicit narrowing to scripted-mode only, with subprocess/docker-exec called out as deferred. Right now it claims both full conversion and no overlay work, which cannot both be true.

I’m stopping here because these are blocking feasibility issues.

**Overall Assessment**
The plan is pointed at a real problem and the binding-agnostic split is directionally right, but it is not ready to implement yet. Resolve the lifetime contradiction and re-scope the Compose work to include the existing overlay storage contracts, then it will be worth reviewing alternatives, completeness, and edge cases.
