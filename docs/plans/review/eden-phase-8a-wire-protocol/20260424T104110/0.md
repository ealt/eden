# EDEN Phase 8a — Wire Protocol & Standalone Orchestrator

## Goal

Define an HTTP binding for the task/event-log operations that chapters 4, 5, and 8 specify behaviorally, and split the reference deployment into two processes that communicate over it: a **task-store server** that holds the `Store` (and its SQLite database), and an **orchestrator service** that runs `run_experiment` as a client of that server.

At the end of Phase 8a, the 3-trial end-to-end experiment runs with the store in one process and the orchestrator in another, with every interaction crossing an HTTP boundary.

## Non-goals (deferred to 8b / 8c)

- **Worker process extraction.** Scripted workers keep running inside the orchestrator process at 8a. Phase 8b moves each worker role into its own host.
- **Authentication.** The server listens on loopback at 8a; the shared-token scaffolding the roadmap calls for lands with 8b when workers become independent principals.
- **Push-based event subscription (SSE / WebSocket).** Polling against `read_range` over HTTP satisfies at-least-once + replay for 8a; push delivery is a transport optimization that can land in 8b or 8c without changing subscriber-observable behavior.
- **Removing in-process paths.** Phase 8c is the explicit cut-over that deletes the direct-`Store` instantiation path from `eden-dispatch`. At 8a, running dispatch against a local `Store` instance must remain supported; the tests still exercise it.
- **Artifact store (§5) transport.** Blob upload/fetch lands with Phase 10's object-store backend. The wire protocol carries only URIs, which remain opaque at 8a.
- **Composite-commit transport semantics.** The server holds the `Store` directly; composite commits stay atomic inside a single in-process transaction on the server side. 8a does not attempt to distribute a composite commit across processes.

## Protocol-first framing

The wire protocol is a **transport binding** for chapter 4 (task state machine), chapter 5 (event log), and the parts of chapter 8 the reference deployment has already implemented. Every wire endpoint maps 1:1 to a `Store` Protocol operation; no endpoint introduces new normative semantics. The binding chapter is therefore a translation layer — request URLs, methods, status codes, error envelopes — not a re-specification of behavior.

The new spec chapter is [`spec/v0/07-wire-protocol.md`](../../spec/v0/07-wire-protocol.md) (slot `07` was unassigned between `06-integrator.md` and `08-storage.md`). It is an HTTP binding **only**; it does not rule out other bindings (gRPC, in-process, message-broker-mediated). A non-HTTP conforming deployment simply targets a different binding.

## What gets created

### Spec: `spec/v0/07-wire-protocol.md`

Normative chapter covering:

1. **Scope.** An HTTP binding for the operations in chapters 4, 5, and §§1.1 and 2.1 of chapter 8. Non-normative commentary cross-links to `04-task-protocol.md` for every endpoint that touches the task state machine.
2. **Transport.**
   - HTTP/1.1 minimum; HTTP/2 MAY. Request/response bodies are `application/json; charset=utf-8`.
   - Every request carries `X-Eden-Experiment-Id: <experiment_id>`; the server MUST reject a request whose header disagrees with the path's experiment segment. The experiment-ID is also in the URL path (see §3), so the header is defense-in-depth against misrouted clients.
3. **URL shape.** Resource-oriented REST with explicit operations for state transitions. The path root is `/experiments/{experiment_id}`; below that:
   - `POST   tasks`                              — `create_task` (wraps the typed `create_plan_task` / `create_implement_task` / `create_evaluate_task` helpers via the task `kind` field).
   - `GET    tasks`                              — `list_tasks` (supports `?kind=` and `?state=` query params).
   - `GET    tasks/{task_id}`                    — `read_task`.
   - `POST   tasks/{task_id}/claim`              — `claim`.
   - `POST   tasks/{task_id}/submit`             — `submit`.
   - `POST   tasks/{task_id}/accept`             — `accept`.
   - `POST   tasks/{task_id}/reject`             — `reject`.
   - `POST   tasks/{task_id}/reclaim`            — `reclaim`.
   - `GET    tasks/{task_id}/validate-terminal`  — `validate_terminal` (read-only; returns the decision tuple).
   - `GET    tasks/{task_id}/submission`         — `read_submission`.
   - `GET    proposals`, `GET proposals/{id}`    — list / read.
   - `POST   proposals`                          — `create_proposal`.
   - `POST   proposals/{id}/mark-ready`          — `mark_proposal_ready`.
   - `GET    trials`, `GET trials/{id}`          — list / read.
   - `POST   trials`                             — `create_trial`.
   - `POST   trials/{id}/declare-eval-error`     — `declare_trial_eval_error`.
   - `POST   trials/{id}/integrate`              — `integrate_trial` (server-side write of `trial_commit_sha`; the `Integrator.integrate` caller already holds the ref/manifest invariants — the server only persists the CAS'd field and event).
   - `GET    events`                             — `read_range` / `replay` (supports `?cursor=<int>`; omitting = full replay).
   - `POST   validate/metrics`                   — `validate_metrics` (request body: metrics dict; response body: OK or structured error per §4).
4. **Request and response bodies.**
   - Every body is a JSON object with a stable field layout defined by a JSON Schema under `spec/v0/schemas/wire/`. At minimum: - `wire/task-create-request.schema.json` - `wire/claim-request.schema.json` / `claim-response.schema.json` - `wire/submit-request.schema.json` - `wire/reject-request.schema.json` (carries `reason`) - `wire/reclaim-request.schema.json` (carries optional `cause`) - `wire/events-response.schema.json` - `wire/error.schema.json`
   - Response bodies for read endpoints serialize the existing entity schemas (task, proposal, trial, event) verbatim — they're already the canonical JSON shapes.
5. **Error envelope.** All non-2xx responses return `application/problem+json` (RFC 7807 profile) with the `type` slot set to a closed vocabulary mapping to the `Store` error hierarchy:

   | `type` | HTTP status | `Store` error |
   |---|---|---|
   | `eden://error/not-found` | 404 | `NotFound` |
   | `eden://error/already-exists` | 409 | `AlreadyExists` |
   | `eden://error/illegal-transition` | 409 | `IllegalTransition` |
   | `eden://error/wrong-token` | 403 | `WrongToken` |
   | `eden://error/conflicting-resubmission` | 409 | `ConflictingResubmission` |
   | `eden://error/invalid-precondition` | 422 | `InvalidPrecondition` |
   | `eden://error/bad-request` | 400 | body schema violation |
   | `eden://error/experiment-id-mismatch` | 400 | header vs. path disagreement |

The `title` is a short human phrase; `detail` is a longer human-readable message; `instance` is the request URL. Clients MUST key retries / error handling on `type`, not on HTTP status alone (several conflict-class errors share 409).
6. **Idempotency.** The task-protocol §4.2 idempotency rule binds the wire binding: a repeated `submit` with the same token and content-equivalent payload MUST return the same 200 outcome, and a repeated `submit` with a divergent payload MUST return `conflicting-resubmission`. The spec MAY note that HTTP-level retries by a client library (e.g. automatic retry on a transport timeout) rely on this binding for correctness; no HTTP-level `Idempotency-Key` header is required because the claim token already uniquely identifies the attempt.
7. **Event delivery.**
   - `GET /experiments/{E}/events?cursor=N` returns events after index `N` in log order; `cursor` is the cumulative count the caller has seen (same semantics as `Store.read_range`). The response contains the delivered batch and the new cursor.
   - Polling clients re-issue after a caller-chosen interval.
   - The binding MAY also offer Server-Sent Events at `GET /experiments/{E}/events/stream` in a later phase; 8a does not mandate this.
   - At-least-once and full-replay guarantees (chapter 5 §4.3, §4.4) are preserved by the binding because they are preserved by the underlying `read_range` semantics.
8. **Versioning.** All URLs are under `/v0/` (e.g. `/v0/experiments/{id}/tasks`). Breaking changes go to `/v1/` per the same discipline as spec chapter versioning.

### Schemas: `spec/v0/schemas/wire/`

Request/response JSON Schemas listed in §4 above. Each gets a passing test under `reference/packages/eden-contracts/tests/` (extending the existing `schema-validity` + `schema-parity` jobs).

The error envelope schema is shared across all endpoints and is cited from each endpoint's prose in the chapter.

### Package: `reference/packages/eden-wire/`

A new workspace member depending on `eden-contracts` and `eden-storage`. Provides three cooperating pieces:

1. **`eden_wire.models`** — Pydantic request/response types for every endpoint. Each body type has a schema-parity test matching `spec/v0/schemas/wire/*.schema.json`. Follows the same `strict=True`, `NotNone`, `UriStr`, `DateTimeStr` discipline as `eden-contracts`.
2. **`eden_wire.server`** — A FastAPI (or Starlette; see "Open questions") application factory: `make_app(store: Store) -> ASGIApp`. Each endpoint is a thin adapter that:
   - Validates the request body against its Pydantic model.
   - Validates the experiment-ID header against the URL segment.
   - Invokes the matching `Store` method.
   - Catches each `StoreError` subclass and maps it to the RFC 7807 envelope.
   - Returns a Pydantic-serialized response body.
   The server does not embed business logic — every operation delegates to `Store`. This keeps the server's conformance story trivial: if the underlying `Store` is conforming, the HTTP facade is conforming.
3. **`eden_wire.client`** — A `StoreClient` class whose public API satisfies the `Store` Protocol. Every method makes the corresponding HTTP call via `httpx.Client` and either:
   - Returns the decoded Pydantic entity, or
   - Raises the matching `StoreError` subclass reconstructed from the problem+json body.
   `StoreClient` is used wherever a `Store` is used today: dispatch driver, integrator, test fixtures. Structural Protocol conformance means callers don't need to know whether they hold an in-process `Store` or a wire proxy.

4. **`eden_wire.errors`** — Serde for the RFC 7807 envelope, including the `eden://error/...` type → exception class mapping. Used by both server (raise → serialize) and client (deserialize → raise).

### Services

- **`reference/services/task-store-server/`** — Entry point `eden_task_store_server.__main__` that:
  1. Parses CLI args (`--db-path`, `--experiment-id`, `--experiment-config`, `--host`, `--port`).
  2. Constructs either `SqliteStore.open(...)` or `InMemoryStore(...)` based on `--db-path`.
  3. Calls `eden_wire.server.make_app(store)`.
  4. Runs it via `uvicorn`.
  Logs structured events (stdout) for observability.

- **`reference/services/orchestrator-service/`** — Entry point `eden_orchestrator.__main__` that:
  1. Parses CLI args (`--task-store-url`, `--experiment-id`, `--repo-path`, `--plan-tasks`, `--implement-task-prefix`, `--evaluate-task-prefix`).
  2. Constructs `StoreClient(base_url, experiment_id)`.
  3. Constructs scripted workers (Phase 5 / 8a flavor) against the `StoreClient`.
  4. Constructs an `Integrator(store=client, repo=..., author=...)`; passes `integrator.integrate` as `integrate_trial`.
  5. Calls `run_experiment(...)` to completion.
  6. Exits zero on clean termination.

- Existing directories `reference/services/orchestrator/`, `planner/`, `implementer/`, `evaluator/`, `web-ui/`, `control-plane/` remain stubbed with `.gitkeep`. `orchestrator-service/` is a new directory; the bare `orchestrator/` stays reserved as an alias in case a later phase prefers that shorter name.

### Tests

1. **Parametrized conformance** — `eden-wire/tests/` adds a new backend to the `make_store` parametrization used across `eden-storage` and `eden-dispatch`: a `ClientStore` that wraps a `StoreClient` pointed at an in-process FastAPI `TestClient` (no network, but the full serialization + dispatch round-trip happens). Every existing conformance scenario runs against this backend unchanged. Drift between the wire binding and the direct-store semantics surfaces as a test failure.
2. **End-to-end two-process test** — `eden-wire/tests/test_e2e.py` actually boots the task-store server as a subprocess on an ephemeral port, runs the orchestrator service against it, and asserts the post-run state matches the Phase 5 / 7b end-to-end fixtures (3 trials, 3 evaluations, 3 promotions visible in `list_trials(status="success")` with `trial_commit_sha` set).
3. **Error-envelope round-trip** — every `StoreError` subclass gets a test that triggers it server-side and verifies the client reconstructs the same exception class.
4. **Experiment-id header check** — a request whose header disagrees with the URL segment is rejected with `experiment-id-mismatch`.
5. **Schema parity** — each new wire schema has a `tests/test_wire_schema_parity.py` that round-trips Pydantic dumps against the JSON Schema.

## Algorithm — what the orchestrator's turn looks like over HTTP

For reference, a single iteration of `run_experiment` — unchanged from Phase 7b structurally — now performs these HTTP calls via `StoreClient`:

1. `planner.run_pending(client)`:
   - `GET /tasks?kind=plan&state=pending`
   - For each: `POST /tasks/{id}/claim`, run scripted planner, per drafted proposal `POST /proposals`, per ready proposal `POST /proposals/{id}/mark-ready`, `POST /tasks/{id}/submit`.
2. `_finalize_submitted(client, kind="plan")`:
   - `GET /tasks?kind=plan&state=submitted`
   - For each: `GET /tasks/{id}/validate-terminal` → `POST /tasks/{id}/accept` or `POST /tasks/{id}/reject`.
3. `_dispatch_implement_tasks`:
   - `GET /proposals?state=ready`
   - For each: `POST /tasks` (kind=implement).
4. Implementer and evaluator loops follow the same pattern.
5. `_promote_successful_trials`:
   - `GET /trials?status=success`
   - For each without a `trial_commit_sha`: `integrator.integrate(id)`. Integrator calls into `client.read_trial`, `client.validate_metrics`, and — on success — `POST /trials/{id}/integrate` which atomically writes the field and the event per chapter 6 §3.4 (server-side).

Per-iteration HTTP count is proportional to the number of tasks in flight; the polling loop quiesces in O(1) round-trips after the queue drains.

## Atomicity and composite-commit handling

Composite commits (chapter 5 §2.2) stay atomic on the server side: each endpoint that the server handles runs under the existing `Store._commit` transaction. The binding does not distribute transactions. A client invoking `POST /tasks/{id}/accept` on an `implement` task receives a single success response after both the task-completed event and the proposal-completed event have been durably appended.

This is consistent with chapter 8 §6.1 and the single-process atomicity guarantees `SqliteStore` already provides; crossing the process boundary doesn't weaken them because each HTTP call corresponds to exactly one server-side transaction.

## Migration from Phase 7b

- `eden-dispatch.run_experiment` needs no signature change; it already accepts a `Store` and an `integrate_trial` callable. A `StoreClient` *is* a `Store`, so the driver works unchanged.
- `eden-git.Integrator` needs no change for the same reason.
- `Store.integrate_trial` server endpoint must preserve the §3.4 invariant: the HTTP handler must invoke the same atomic transaction the direct call does. Integrator first writes the ref locally (it holds the repo), then calls the HTTP endpoint, then compensates on failure — same order as 7b.
- Existing tests that construct `Store` directly stay. The parametrized fixtures gain an additional `ClientStore` backend; no direct-`Store` test is removed.

## Open questions for review

1. **FastAPI vs. Starlette.** FastAPI gives Pydantic-first request validation, dependency injection, and OpenAPI spec generation for free. Starlette is lighter, closer to ASGI. FastAPI is the better default here because:
   - Every endpoint's request body is already a Pydantic model.
   - Auto-generated OpenAPI is a nice sanity check against the hand-written spec chapter.
   - The overhead is negligible for a reference impl.
   Is anyone opposed?

2. **Separate wire schema directory.** Should `spec/v0/schemas/wire/*.schema.json` live under the same `schemas/` directory as the entity schemas, or in a sibling `wire/` directory (e.g. `spec/v0/wire/`)? The draft assumes a subdirectory `schemas/wire/` — parallel to how the integrator design note lives under `design-notes/`. An alternative is to put wire-specific schemas one level up (e.g. `spec/v0/schemas/wire/*.json` vs `spec/v0/wire-schemas/*.json`). I'd prefer `schemas/wire/` because it keeps the schema root singular.

3. **Problem+json vs. vanilla JSON errors.** RFC 7807 is the IETF standard for HTTP API errors and lets clients key on `type` URIs without parsing human text. A vanilla `{"error": ..., "code": ...}` is lighter. I'd prefer problem+json because the `type` field survives future additions without client parsing changes; the cost is a more formal error envelope.

4. **Should `validate_terminal` remain server-side?** The orchestrator currently relies on `store.validate_terminal(task_id)` to decide accept vs. reject before issuing the transition. We could move that decision into the client — the client has enough information (the submission payload, the task's expected contract) — which would cut the number of round-trips per submitted task in half. The cost is duplicating the decision logic in the client. I'd prefer keeping it server-side for 8a; the extra RTT is cheap and the logic stays in one place. Revisit in 8c.

5. **Subscription shape for 8a.** Polling-only is the proposal. The cost is higher latency (poll interval) for workers that want to react to events; the benefit is a shorter critical path for 8a. Is polling acceptable, or should 8a include SSE?

6. **`/v0/` URL prefix.** The spec is versioned at `v0/`; the URL mirrors it. Cleaner alternative: version in the `Accept: application/vnd.eden.v0+json` header, keep URLs unprefixed. Path-versioning is simpler for logs/grepping; I'd take it.

## Out of scope for 8a

- Worker process extraction.
- Authentication or authorization.
- SSE / WebSocket push subscription.
- Removing in-process dispatch (that is Phase 8c).
- Artifact-store HTTP binding.
- OpenAPI publication / docs site generation (the app exposes it for introspection; shipping it as a consumable artifact is later).

## Risks

- **HTTP error-envelope ↔ exception hierarchy drift.** Mitigation: a table in the spec chapter pairs every `eden://error/<name>` with a `Store` exception; both sides of the mapping assert against the table in a parity test.
- **Subprocess e2e test flakiness.** Mitigation: boot with a readiness probe (poll `GET /experiments/{id}/events?cursor=0` until 2xx) before the orchestrator starts; tear down with SIGTERM and a graceful shutdown window.
- **FastAPI heavy-dependency surface.** Mitigation: pin narrowly in the per-package `pyproject.toml`; the workspace-wide `uv.lock` captures the resolved set.
- **Problem+json serialization drift.** Mitigation: the error schema is under `spec/v0/schemas/wire/` and both server + client validate against it.

## Summary

Phase 8a adds one new spec chapter (`07-wire-protocol.md`), one new package (`eden-wire`), two new services (task-store server, orchestrator service), and a new conformance backend (`ClientStore`). No changes to existing spec chapters, no changes to `eden-dispatch` / `eden-git` / `eden-storage` / `eden-contracts` APIs. At completion, the 3-trial end-to-end experiment runs across a process boundary with every interaction carrying full HTTP serialization, and the conformance scenarios pass identically against both the direct `Store` and the wire-mediated `ClientStore`.
