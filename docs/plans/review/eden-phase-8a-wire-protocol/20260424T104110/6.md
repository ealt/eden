# EDEN Phase 8a — Wire Protocol & Standalone Orchestrator

## Goal

Define an HTTP binding for the task/event-log operations that chapters
4, 5, and 8 specify behaviorally, and split the reference deployment
into two processes that communicate over it: a **task-store server**
that holds the `Store` (and its SQLite database), and an
**orchestrator service** that runs `run_experiment` as a client of
that server.

At the end of Phase 8a, the 3-trial end-to-end experiment runs with
the store in one process and the orchestrator in another, with every
interaction crossing an HTTP boundary.

## Non-goals (deferred to 8b / 8c)

- **Worker process extraction.** Scripted workers keep running inside
  the orchestrator process at 8a. Phase 8b moves each worker role
  into its own host.
- **Authentication.** The server listens on loopback at 8a; the
  shared-token scaffolding the roadmap calls for lands with 8b when
  workers become independent principals.
- **Push-based event subscription (SSE / WebSocket).** Polling against
  `read_range` over HTTP satisfies at-least-once + replay for 8a;
  push delivery is a transport optimization that can land in 8b or
  8c without changing subscriber-observable behavior.
- **Removing in-process paths.** Phase 8c is the explicit cut-over
  that deletes the direct-`Store` instantiation path from
  `eden-dispatch`. At 8a, running dispatch against a local `Store`
  instance must remain supported; the tests still exercise it.
- **Artifact store (§5) transport.** Blob upload/fetch lands with
  Phase 10's object-store backend. The wire protocol carries only
  URIs, which remain opaque at 8a.
- **Composite-commit transport semantics.** The server holds the
  `Store` directly; composite commits stay atomic inside a single
  in-process transaction on the server side. 8a does not attempt to
  distribute a composite commit across processes.

## Protocol-first framing

The wire protocol is a **transport binding** for chapter 4 (task
state machine), chapter 5 (event log), and the parts of chapter 8 the
reference deployment has already implemented. Every wire endpoint
maps 1:1 to a `Store` Protocol operation; no endpoint introduces new
normative semantics. The binding chapter is therefore a translation
layer — request URLs, methods, status codes, error envelopes — not a
re-specification of behavior.

The new spec chapter is [`spec/v0/07-wire-protocol.md`](../../spec/v0/07-wire-protocol.md)
(slot `07` was unassigned between `06-integrator.md` and
`08-storage.md`). It is an HTTP binding **only**; it does not rule
out other bindings (gRPC, in-process, message-broker-mediated). A
non-HTTP conforming deployment simply targets a different binding.

## What gets created

### Spec: `spec/v0/07-wire-protocol.md`

Normative chapter covering:

1. **Scope.** An HTTP binding for the operations in chapters 4, 5, and
   §§1.1 and 2.1 of chapter 8. Non-normative commentary cross-links
   to `04-task-protocol.md` for every endpoint that touches the task
   state machine.
2. **Transport.**
   - HTTP/1.1 minimum; HTTP/2 MAY. Request/response bodies are
     `application/json; charset=utf-8`.
   - Every request carries `X-Eden-Experiment-Id: <experiment_id>`;
     the server MUST reject a request whose header disagrees with
     the path's experiment segment. The experiment-ID is also in the
     URL path (see §3), so the header is defense-in-depth against
     misrouted clients.
3. **URL shape.** Resource-oriented REST with explicit operations
   for state transitions. The path root is
   `/experiments/{experiment_id}`; below that:
   - `POST   tasks`                              — `create_task`
     (wraps the typed `create_plan_task` /
     `create_implement_task` / `create_evaluate_task` helpers via
     the task `kind` field).
   - `GET    tasks`                              — `list_tasks`
     (supports `?kind=` and `?state=` query params).
   - `GET    tasks/{task_id}`                    — `read_task`.
   - `POST   tasks/{task_id}/claim`              — `claim`.
   - `POST   tasks/{task_id}/submit`             — `submit`.
   - `POST   tasks/{task_id}/accept`             — `accept`.
   - `POST   tasks/{task_id}/reject`             — `reject`.
   - `POST   tasks/{task_id}/reclaim`            — `reclaim`.
   - `GET    tasks/{task_id}/submission`         — `read_submission`.
   - `GET    proposals`, `GET proposals/{id}`    — list / read.
   - `POST   proposals`                          — `create_proposal`.
   - `POST   proposals/{id}/mark-ready`          — `mark_proposal_ready`.
   - `GET    trials`, `GET trials/{id}`          — list / read.
   - `POST   trials`                             — `create_trial`.
   - `POST   trials/{id}/declare-eval-error`     — `declare_trial_eval_error`.
   - `POST   trials/{id}/integrate`              — `integrate_trial`
     (server-side atomic write of `trial_commit_sha` + event;
     **same-value idempotent** per §3.4-binding below). The
     `Integrator.integrate` caller has already performed the ref
     write under its own atomicity rule; the HTTP endpoint persists
     the store-side half.
   - `GET    events`                             — `read_range` /
     `replay` (supports `?cursor=<int>`; omitting = full replay).
   - `GET    events/subscribe`                   — the chapter 8 §2.1
     `subscribe` binding; see §7 below.

   Reference-implementation helpers (`validate_terminal`,
   `validate_metrics`) are **not** normative wire endpoints and are
   not listed in `07-wire-protocol.md`. The orchestrator service and
   the `eden_wire` server expose them at `/_reference/...` as
   non-normative affordances so the dispatch driver's Phase 5
   decision helpers keep working against the `StoreClient`; a
   third-party conforming orchestrator is free to implement the
   same decisions inline. See the new "Reference-only helper
   endpoints" section below.
4. **Request and response bodies.**
   - Every body is a JSON object with a stable field layout defined
     by a JSON Schema under `spec/v0/schemas/wire/`. At minimum:
     - `wire/task-create-request.schema.json`
     - `wire/claim-request.schema.json` / `claim-response.schema.json`
     - `wire/submit-request.schema.json`
     - `wire/reject-request.schema.json` (carries `reason`)
     - `wire/reclaim-request.schema.json` (carries optional `cause`)
     - `wire/events-response.schema.json`
     - `wire/error.schema.json`
   - Response bodies for read endpoints serialize the existing entity
     schemas (task, proposal, trial, event) verbatim — they're
     already the canonical JSON shapes.
5. **Error envelope.** All non-2xx responses return
   `application/problem+json` (RFC 7807 profile) with the `type`
   slot set to a closed vocabulary mapping to the `Store` error
   hierarchy:

   | `type` | HTTP status | `Store` error |
   |---|---|---|
   | `eden://error/not-found` | 404 | `NotFound` |
   | `eden://error/already-exists` | 409 | `AlreadyExists` |
   | `eden://error/illegal-transition` | 409 | `IllegalTransition` |
   | `eden://error/wrong-token` | 403 | `WrongToken` |
   | `eden://error/conflicting-resubmission` | 409 | `ConflictingResubmission` |
   | `eden://error/invalid-precondition` | 409 | `InvalidPrecondition` |
   | `eden://error/bad-request` | 400 | body schema violation |
   | `eden://error/experiment-id-mismatch` | 400 | header vs. path disagreement |

   The `title` is a short human phrase; `detail` is a longer
   human-readable message; `instance` is the request URL. Clients
   MUST key retries / error handling on `type`, not on HTTP status
   alone (several conflict-class errors share 409).
6. **Idempotency.**
   - **Submit (chapter 4 §4.2).** A repeated `submit` with the same
     token and content-equivalent payload MUST return the same 200
     outcome; a divergent payload MUST return
     `conflicting-resubmission`. The claim token uniquely identifies
     the attempt, so no HTTP-level `Idempotency-Key` header is
     required.
   - **Integrate-trial (chapter 6 §3.4 binding).** The wire endpoint
     `POST /trials/{id}/integrate` MUST be idempotent for **the same
     `trial_commit_sha`** value: a repeated call against a trial whose
     `trial_commit_sha` already equals the request's value MUST
     succeed as a no-op and MUST NOT append a second
     `trial.integrated` event. A repeated call with a **different**
     `trial_commit_sha` MUST be rejected with
     `eden://error/invalid-precondition` (HTTP 409). The chapter
     binds this error type explicitly: it mirrors the `Store`-side
     `InvalidPrecondition` already raised by
     `Integrator._check_idempotent` when a stored SHA mismatches a
     re-derived one, and avoids overloading
     `conflicting-resubmission` (which is reserved for the chapter 4
     §4.2 submit-idempotency rule).
   - **Transport-indeterminate handling (integrator-client rule).**
     The client-side `Integrator.integrate` compensating-delete path
     MUST NOT run blindly on any HTTP error. On an *indeterminate*
     failure — a transport error raised after the request was sent
     but before a 2xx or terminal 4xx/5xx response was received
     (connection reset, timeout, proxy-side disconnect) — the
     client MUST issue a reconciliation read of `GET /trials/{id}`
     and resolve the outcome per the three-way rule in
     "Atomicity and composite-commit handling" below: observed
     SHA equals expected → success, no compensation; observed SHA
     differs → `AtomicityViolation`, no compensation; observed
     SHA absent or read-back itself fails →
     `IndeterminateIntegration`, no compensation (a `null`
     `trial_commit_sha` is not positive proof the original request
     failed — it may still be in flight). The Integrator
     compensates **only** when the server demonstrably rejected
     the write via a non-transport store error that is **not** the
     different-SHA divergence branch — i.e. a synchronous
     `invalid-precondition` that the Integrator can act on without
     implying another integrator already owns the ref target
     (different-SHA divergence surfaces as `AtomicityViolation`
     instead; see "Atomicity and composite-commit handling"
     below). Combined with server-side same-value idempotency,
     this preserves the chapter 6 §3.4 atomicity invariant across
     the process boundary.
   - **Other mutations.** Claim, reject, reclaim, and accept are
     **not blindly retry-safe** on indeterminate transport
     failures. The natural temptation is to treat a retry's
     `illegal-transition` response as "the prior call committed,"
     but that conflates several distinct situations — a racing
     actor, a reclaim between attempts, or a genuine state-machine
     violation — and `claim` additionally loses its token on a
     lost response with no way to recover it. 8a does **not**
     standardize an HTTP-level retry convention for these
     endpoints. The reference `StoreClient` surfaces transport
     errors as `IndeterminateRequest` exceptions that the caller
     must handle with operation-specific knowledge: a dispatch
     driver may choose to read the task state back and decide
     whether to reclaim, bail, or proceed, but the decision is
     the driver's, not the binding's. A future phase MAY introduce
     per-operation idempotency tokens or an `Idempotency-Key`
     mechanism; 8a does not.
7. **Event delivery — the `subscribe` binding.**
   - The chapter 8 §2.1 `subscribe` operation has an HTTP binding at
     `GET /experiments/{E}/events/subscribe?cursor=N`. In the 8a
     binding, this is **long-poll**: the server returns as soon as
     one or more events are available after index `N`, and returns
     an empty batch after a server-chosen timeout (default 30s)
     otherwise. The response body includes the delivered batch and
     the new cursor. A caller iterating this endpoint in a loop
     observes the log in order with unbounded retention and
     at-least-once delivery (chapter 5 §4.3, §4.4).
   - `GET /experiments/{E}/events?cursor=N` is a non-blocking
     companion corresponding to `read_range`: it returns whatever
     is available immediately (possibly zero events). Clients that
     prefer explicit polling use this endpoint; clients that want
     near-real-time delivery use `/subscribe`.
   - Both endpoints accept the same `cursor` semantics: cumulative
     count of events the caller has already observed. Cursor `0`
     (or omitted) triggers `replay` — the full stream from the
     first event.
   - **Out of scope for 8a: push transports.** Server-Sent Events
     and WebSocket bindings are additional *transport* layers that
     a later phase MAY add; they do not change the normative
     `subscribe` contract. A conforming 8a deployment offers the
     long-poll binding; richer transports are optimizations.
   - At-least-once and full-replay guarantees are preserved by the
     binding because they are preserved by the underlying `events`
     + `read_range` semantics.
8. **Versioning.** All URLs are under `/v0/` (e.g.
   `/v0/experiments/{id}/tasks`). Breaking changes go to `/v1/`
   per the same discipline as spec chapter versioning.

### Reference-only helper endpoints (non-normative)

The following endpoints are exposed by the reference `eden_wire`
server for the convenience of the reference orchestrator, but are
**not** part of the normative wire protocol and are **not** listed
in `07-wire-protocol.md`. They live under `/_reference/` (not
`/v0/`) to make the non-normative status syntactically obvious:

- `GET  /_reference/tasks/{task_id}/validate-terminal` — wraps
  `Store.validate_terminal`. Returns the `(decision, reason)`
  tuple the Phase 5 dispatch driver uses to pick
  accept/reject_worker/reject_validation. A third-party
  orchestrator is free to implement this decision itself by
  reading the submission payload and task state via the normative
  `GET /tasks/{id}` + `GET /tasks/{id}/submission` endpoints.
- `POST /_reference/validate/metrics` — wraps
  `Store.validate_metrics`. Returns OK or a problem+json error.
  A third-party caller can validate locally against the
  experiment's metrics schema, or skip the pre-check and let the
  server reject malformed payloads at submit time.

The reference Python `StoreClient` calls these `/_reference/`
endpoints through `validate_terminal` / `validate_metrics` methods
that match the `Store` Protocol; this keeps the existing dispatch
driver working verbatim. The `Store` Protocol stays what it is; the
wire chapter stays narrower than the Protocol.

If a future phase decides one of these helpers deserves normative
status (e.g. a conformance test needs it), it gets promoted into the
chapter — and into `/v0/` — at that time, with a spec version bump
if it's a breaking change. At 8a, neither is normative.

### Schemas: `spec/v0/schemas/wire/`

Request/response JSON Schemas listed in §4 above. Each gets a
passing test under `reference/packages/eden-contracts/tests/`
(extending the existing `schema-validity` + `schema-parity` jobs).

The error envelope schema is shared across all endpoints and is
cited from each endpoint's prose in the chapter.

### Package: `reference/packages/eden-wire/`

A new workspace member depending on `eden-contracts` and
`eden-storage`. Provides three cooperating pieces:

1. **`eden_wire.models`** — Pydantic request/response types for
   every endpoint. Each body type has a schema-parity test matching
   `spec/v0/schemas/wire/*.schema.json`. Follows the same
   `strict=True`, `NotNone`, `UriStr`, `DateTimeStr` discipline as
   `eden-contracts`.
2. **`eden_wire.server`** — A FastAPI application factory:
   `make_app(store: Store) -> ASGIApp`. Each endpoint is a thin
   adapter that:
   - Validates the request body against its Pydantic model.
   - Validates the experiment-ID header against the URL segment.
   - Invokes the matching `Store` method.
   - Catches each `StoreError` subclass and maps it to the RFC 7807
     envelope.
   - Returns a Pydantic-serialized response body.

   The server does not embed business logic — every operation
   delegates to `Store`. This keeps the server's conformance
   story trivial: if the underlying `Store` is conforming, the
   HTTP facade is conforming.
3. **`eden_wire.client`** — A `StoreClient` class whose public API
   satisfies the `Store` Protocol. Every method makes the
   corresponding HTTP call via `httpx.Client` and either:
   - Returns the decoded Pydantic entity, or
   - Raises the matching `StoreError` subclass reconstructed from
     the problem+json body.

   `StoreClient` is used wherever a `Store` is used today: dispatch
   driver, integrator, test fixtures. Structural Protocol conformance
   means callers don't need to know whether they hold an in-process
   `Store` or a wire proxy.

4. **`eden_wire.errors`** — Serde for the RFC 7807 envelope,
   including the `eden://error/...` type → exception class mapping.
   Used by both server (raise → serialize) and client (deserialize
   → raise).

   `list_trials(status="success")` with `trial_commit_sha` set).
3. **Error-envelope round-trip** — every `StoreError` subclass gets
   a test that triggers it server-side and verifies the client
   reconstructs the same exception class.
4. **Experiment-id header check** — a request whose header
   disagrees with the URL segment is rejected with
   `experiment-id-mismatch`.
5. **Schema parity** — each new wire schema has a
   `tests/test_wire_schema_parity.py` that round-trips Pydantic
   dumps against the JSON Schema.
6. **Integrate-over-HTTP reconciliation — three-outcome coverage.**
   One test per branch of the three-outcome rule:
   - *Confirmed success.* Server's
     `POST /trials/{id}/integrate` commits durably; the response
     is then lost (injected transport disconnect after the
     transaction but before serialization). The
     `StoreClient.integrate_trial` read-back observes
     `trial_commit_sha` equal to the expected value; the
     Integrator returns success and MUST NOT compensate-delete
     the ref.
   - *Confirmed divergence.* Server has already committed a
     *different* `trial_commit_sha` (simulated by seeding the
     store with one SHA, then calling integrate with another).
     The request is rejected inline with
     `invalid-precondition` (same-value idempotency fails);
     alternatively, the transport fails and the read-back
     observes the different SHA. Both paths raise
     `AtomicityViolation`; neither compensates.
   - *Indeterminate.* Server has no `trial_commit_sha` at
     read-back time (request never arrived, or transaction
     rolled back, or is still in flight) or the read-back
     itself fails. The client MUST raise
     `IndeterminateIntegration` and MUST NOT compensate; the
     `null`-SHA reading is not treated as positive proof of
     non-commit.

   A separate test verifies the compensation path is still
   exercised by a **non-transport, non-divergence** rejection.
   The test injects a mock `Store` (or uses a direct-`Store`
   subclass) whose `integrate_trial` raises a synthetic
   `InvalidPrecondition` without the different-SHA branch (for
   example, a simulated "trial status flipped out of success
   concurrently" condition). The Integrator MUST compensate-
   delete the ref it wrote and re-raise, matching the Phase 7b
   rule. This preserves the one branch where compensating
   deletion is correct.
7. **Same-value idempotency on `integrate_trial`.** Calling the
   endpoint twice with the same `trial_commit_sha` emits exactly
   one `trial.integrated` event; calling it with a different SHA
   returns `invalid-precondition`. Verified by counting events and
   by asserting the error type.
8. **Long-poll `subscribe` semantics.** A subscriber connected at
   `GET /events/subscribe?cursor=N` receives events appended after
   cursor `N` within the poll window, receives an empty batch on
   timeout, and resumes from the returned cursor on the next call.
   Replay from cursor `0` returns the full experiment stream.

## Algorithm — what the orchestrator's turn looks like over HTTP

For reference, a single iteration of `run_experiment` — unchanged
from Phase 7b structurally — now performs these HTTP calls via
`StoreClient`:

1. `planner.run_pending(client)`:
   - `GET /tasks?kind=plan&state=pending`
   - For each: `POST /tasks/{id}/claim`, run scripted planner, per
     drafted proposal `POST /proposals`, per ready proposal
     `POST /proposals/{id}/mark-ready`,
     `POST /tasks/{id}/submit`.
2. `_finalize_submitted(client, kind="plan")`:
   - `GET /tasks?kind=plan&state=submitted`
   - For each: `GET /_reference/tasks/{id}/validate-terminal`
     (reference-only helper; not in `07-wire-protocol.md`) →
     `POST /tasks/{id}/accept` or `POST /tasks/{id}/reject`.
3. `_dispatch_implement_tasks`:
   - `GET /proposals?state=ready`
   - For each: `POST /tasks` (kind=implement).
4. Implementer and evaluator loops follow the same pattern.
5. `_promote_successful_trials`:
   - `GET /trials?status=success`
   - For each without a `trial_commit_sha`: `integrator.integrate(id)`.
     Integrator calls into `client.read_trial`,
     `client.validate_metrics`, and — on success —
     `POST /trials/{id}/integrate` which atomically writes the
     field and the event per chapter 6 §3.4 (server-side).

Per-iteration HTTP count is proportional to the number of tasks in
flight; the polling loop quiesces in O(1) round-trips after the
queue drains.

## Atomicity and composite-commit handling

### In-process composite commits stay atomic

Chapter 5 §2.2 composite commits (task-completed + proposal-completed,
evaluate-terminal + trial.succeeded, reclaim + trial.errored,
trial-promotion ref + field + event) still commit atomically on the
server side: each endpoint runs under the existing `Store._commit`
transaction. A client invoking `POST /tasks/{id}/accept` on an
`implement` task receives one response after the task-completed
event and the proposal-completed event have both been durably
appended.

The binding does not distribute transactions across HTTP calls. No
endpoint splits a composite commit into two HTTP requests; each
composite commit maps to exactly one server transaction.

### Chapter 6 §3.4 across the process boundary

The trial-promotion invariant (ref + trial_commit_sha + event all
consistent post-promotion) is the one composite commit that spans
the HTTP boundary: the ref write happens client-side in the
Integrator, and the store-side half (`trial_commit_sha` + event)
happens server-side through `POST /trials/{id}/integrate`. In-
process at Phase 7b, a `Store` exception from `integrate_trial`
caused the Integrator to compensate-delete the ref — the
"post-promotion reading" from
[`spec/v0/design-notes/integrator-atomicity.md`](../../spec/v0/design-notes/integrator-atomicity.md).

Over HTTP, a transport failure — connection reset, timeout, proxy
disconnect — between the request being sent and the response being
received is **indeterminate**: the server may have durably committed
the transition before the response was lost. A blind
compensating-delete in that case would violate §3.4: the ref
disappears while the `trial_commit_sha` and `trial.integrated`
event stay, producing exactly the split-brain state §3.4 forbids
after a promotion returns.

Two mechanisms together close this gap:

1. **Server-side same-value idempotency** on the endpoint (see
   "Request and response bodies" §6 above): a repeated
   `POST /trials/{id}/integrate` with the **same** `trial_commit_sha`
   is a no-op; a repeated call with a **different** value is an
   error. This is the same rule `Integrator.integrate` already
   enforces in-process for `_check_idempotent` — now surfaced at
   the HTTP boundary so retries are safe.
2. **Client-side reconciliation on indeterminate failure.** The
   `StoreClient.integrate_trial` path MUST NOT raise a bare
   transport exception back to `Integrator.integrate`. Instead,
   on `httpx.TransportError` / `httpx.RemoteProtocolError` /
   `httpx.ReadTimeout` (and equivalents), it MUST attempt a
   reconciliation read of `GET /trials/{id}` and resolve to
   exactly one of three outcomes:
   a. **Confirmed success.** If the trial's `trial_commit_sha`
      equals the SHA the client sent, the server has durably
      committed. Return success; Integrator skips the compensating
      delete.
   b. **Confirmed divergence.** If the trial's `trial_commit_sha`
      is set but does **not** equal the SHA the client sent, some
      other integrator call committed a different SHA. §1.2
      forbids this — trial-commit promotion is sole-writer. Raise
      `AtomicityViolation` for operator intervention. The client
      MUST NOT compensate-delete the ref, because the ref may
      correspond to the SHA the server did commit.
   c. **Indeterminate.** In every other case — read-back shows no
      `trial_commit_sha`, or the reconciliation read itself fails,
      or only some attempts succeed — raise
      `IndeterminateIntegration`. Read-back observing a null
      `trial_commit_sha` is **not** positive proof that the
      original request failed, because the original can still be
      in flight and commit moments later; the client must treat
      this as unresolvable and MUST NOT compensate.

The Integrator's compensating-delete flow runs **only** on a
non-transport store error where the server demonstrably rejected
the write **and** the rejection is not the different-SHA branch of
same-value idempotency. Concretely:

- A synchronous `InvalidPrecondition` (or any other non-transport
  rejection) that is **not** "same trial, different already-stored
  `trial_commit_sha`" → compensate. The ref was written in
  anticipation of a store-side commit that will not happen; the
  ref must be removed to keep §3.4's three-artifact invariant
  intact. Examples: the trial was flipped out of `"success"` by
  operator action after the Integrator's pre-read, or the store
  returned a generic validation failure the Integrator cannot
  otherwise act on.
- A synchronous `InvalidPrecondition` that **is** "same trial,
  different already-stored `trial_commit_sha`" → `AtomicityViolation`
  and do **not** compensate. The already-stored SHA names a
  different ref that presumably already exists; deleting **this**
  Integrator's ref could remove a valid artifact the other
  integrator wrote, or leave the store's SHA pointing at a ref
  that was never intended to correspond to it. Operator
  intervention only.
- Any transport-level failure → three-way reconciliation above;
  never compensate without positive confirmation.

This asymmetry is what preserves the chapter 6 §3.4 post-
promotion invariant across the process boundary: after the call
returns — by success, by a compensated non-transport rejection,
by `AtomicityViolation`, or by `IndeterminateIntegration` — the
three artifacts are either consistent or the inconsistency is
explicitly named and surfaced to operators.

This is the one place where the HTTP binding needs more than a
transport-layer translation of the `Store` API. The plan calls it
out explicitly so the Integrator changes land with the
`eden-wire` package, not as a later bug fix.

## Migration from Phase 7b

- `eden-dispatch.run_experiment` needs no signature change; it
  already accepts a `Store` and an `integrate_trial` callable. A
  `StoreClient` *is* a `Store`, so the driver works unchanged.
- `eden-git.Integrator` gains one new failure mode —
  `IndeterminateIntegration` — raised when a wire-mediated
  `integrate_trial` call's outcome cannot be determined. Direct-
  `Store` callers never see it (no transport involved); wire-
  mediated callers treat it as operator-intervention territory,
  same as `AtomicityViolation`. Existing exception handlers in
  `eden-dispatch` propagate it without special handling because
  `run_experiment` currently doesn't catch integrator errors.
- The `StoreClient.integrate_trial` wrapper owns the
  transport-indeterminate reconciliation (read-back + retry loop)
  described in "Atomicity and composite-commit handling" above.
  The Integrator sees either a successful return, a definite
  failure it can compensate, or an `IndeterminateIntegration` it
  must re-raise — the same three-way decision it would have in a
  direct-`Store` world, with the client absorbing the HTTP
  complexity.
- `Store.integrate_trial` server endpoint must preserve the §3.4
  invariant: the HTTP handler invokes the same atomic transaction
  the direct call does, and adds same-value idempotency on top
  (no-op if `trial_commit_sha` already equals the request value;
  error if it equals a different value). The same-value
  idempotency also applies to the direct-`Store` path, so
  `Store.integrate_trial` gains this behavior uniformly (the
  `_check_idempotent` helper in `Integrator` already expressed
  the same rule externally; moving the same-value check down into
  `Store.integrate_trial` keeps the HTTP binding's contract and
  the direct-call contract identical).
- Existing tests that construct `Store` directly stay. The
  parametrized fixtures gain an additional `ClientStore` backend;
  no direct-`Store` test is removed.

## Design decisions settled in round 0

The following were posed as open questions in the draft and resolved
by codex-review round 0:

- **FastAPI over Starlette** for the reference server (Pydantic-
  native request validation; auto OpenAPI; the dependency surface
  is not a concern for a reference impl).
- **Wire schemas under `spec/v0/schemas/wire/`** — kept inside the
  single `schemas/` root rather than a sibling `wire-schemas/`.
- **RFC 7807 problem+json** for error envelopes. `type` URIs close
  over the error vocabulary and decouple the client's retry logic
  from HTTP status codes alone.
- **Helper endpoints (`validate_terminal`, `validate_metrics`) are
  reference-only**, not part of `07-wire-protocol.md`, and live
  under `/_reference/` to make the non-normative status
  syntactically obvious.
- **InvalidPrecondition → 409**, not 422. These are state conflicts
  against existing protocol objects, not malformed bodies.
- **Long-poll is the 8a `subscribe` binding**, not a skip of the
  subscribe contract. SSE / WebSocket bindings are optional
  transport upgrades that later phases may add without changing
  the normative contract.
- **Path-versioned URLs (`/v0/`)** over header-versioned content
  types — simpler for logs and grepping; mirrors the spec's
  `spec/v0/` directory structure.
- **Integrate-over-HTTP uses server-side same-value idempotency +
  client-side read-back reconciliation** to preserve chapter 6 §3.4
  atomicity across the process boundary. See "Atomicity and
  composite-commit handling" above.

## Risks

- **HTTP error-envelope ↔ exception hierarchy drift.** Mitigation:
  a table in the spec chapter pairs every `eden://error/<name>`
  with a `Store` exception; both sides of the mapping assert
  against the table in a parity test.
- **Subprocess e2e test flakiness.** Mitigation: boot with a
  readiness probe (poll `GET /experiments/{id}/events?cursor=0`
  until 2xx) before the orchestrator starts; tear down with
  SIGTERM and a graceful shutdown window.
- **FastAPI heavy-dependency surface.** Mitigation: pin narrowly
  in the per-package `pyproject.toml`; the workspace-wide `uv.lock`
  captures the resolved set.
- **Problem+json serialization drift.** Mitigation: the error
  schema is under `spec/v0/schemas/wire/` and both server + client
  validate against it.

## Summary

Phase 8a adds one new spec chapter (`07-wire-protocol.md`), one new
package (`eden-wire`), two new services (task-store server,
orchestrator service), and a new conformance backend
(`ClientStore`). No changes to existing spec chapters, no changes to
`eden-dispatch` / `eden-git` / `eden-storage` / `eden-contracts`
APIs. At completion, the 3-trial end-to-end experiment runs across
a process boundary with every interaction carrying full HTTP
serialization, and the conformance scenarios pass identically
against both the direct `Store` and the wire-mediated
`ClientStore`.
