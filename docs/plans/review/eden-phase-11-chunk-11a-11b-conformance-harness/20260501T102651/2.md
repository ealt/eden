# EDEN Phase 11 chunk 11a + 11b — Conformance harness + state-machine scenarios

## Context

[`docs/roadmap.md`](../roadmap.md) Phase 11 chunks the conformance suite as
`11a + 11b` (one chunk), `11c` (one), `11d` (one). The harness and the
first scenarios travel together because the harness has nothing to
validate itself against until at least one scenario is in place — the
"first scenarios validate the harness itself" bootstrap problem.

[`spec/v0/`](../../spec/v0/) currently ships chapters 00 – 08 plus the wire
binding (07) and is missing chapter 09. [`conformance/`](../../conformance/)
is a `README.md` stub reserving the location.

This plan delivers exactly what chunk 11a + 11b says it should:

1. The harness — pytest-driven, IUT-adapter-shaped.
2. The first scenarios — task lifecycle, claim-token semantics,
   transactional event invariant including all chapter-05 §2.2
   composite commits, plus the chapter-07 wire-binding shapes that
   the v1 suite exercises through HTTP.
3. The new normative chapter — [`spec/v0/09-conformance.md`](../../spec/v0/09-conformance.md)
   — declaring what conformance means and the relationship between the
   spec, the suite, and any IUT.
4. The reference IUT adapter that drives the existing
   `task-store-server` against the harness.
5. A self-validation scenario that asserts the harness rejects a
   deliberately-broken adapter on a specific test for a specific
   reason.
6. A new `conformance` CI job that runs the suite against the reference
   impl on every PR.

## Reframe

The conformance suite is the **second** thing the spec promises (after
the spec itself) and the **first** thing that makes spec compliance an
externally-verifiable claim. Until this chunk lands, "the reference
impl conforms to the spec" is a sentence the maintainers say. After,
"the reference impl passes the v1 conformance suite, scenario by
scenario, every PR" is a sentence CI says.

The chunk's success criterion is therefore **not** "the reference impl
passes the suite" — that is a tautology if I author the suite by
inspecting the reference impl. The criterion is the chapter-09 +
scenarios pair: **every assertion the harness makes MUST be backed by
a normative MUST in chapters 02 / 04 / 05 / 06 / 07 / 08.** No "the
reference impl happens to do this" assertions; every pass/fail must
be cite-able to a `§N.M` in a specific chapter. That discipline is the
load-bearing part of the chunk; the pytest plumbing is incidental.

## Conformance posture for v1 (resolved from review rounds 0–1)

The v0 spec defines exactly one wire binding ([`07-wire-protocol.md`](../../spec/v0/07-wire-protocol.md)).
A conforming v0 IUT MAY in principle implement the chapter-04/05/08
behavior over a non-HTTP transport, but no such transport is currently
specified, and the v1 suite drives the IUT exclusively through chapter
07's HTTP binding.

**Chapter 09 therefore declares the v1 conformance level as a partial
claim, not full v0 conformance.** Specifically, the v1 suite covers
the **task-store + wire-binding subset**: chapters 02 / 04 / 05 / 07
plus the storage MUSTs in chapter 08 that the wire endpoints expose
(§1.1, §1.7). Chapter-06 integrator MUSTs and the chapter-03 role
submission contracts that the worker hosts uphold are explicitly out
of scope for v1 and ship in chunks 11d (integrator) and 11c (role
contracts). Chapter 09 §4 names the levels as `v1` (this chunk),
`v1+roles` (after 11c), and `v1+roles+integrator` (after 11d). The
final level is the closest the suite gets to "full v0 conformance"
without a non-HTTP binding.

Accordingly:

- The suite uses **one** pytest marker, `conformance`, applied at the
  scenario-directory level via `conftest.py`. There is no `_core` /
  `_wire_http` split in v1. Round-0 review correctly observed that a
  marker split would be a fiction: every scenario in v1 drives the IUT
  through HTTP and most scenarios assert both the wire shape and the
  underlying state-machine effect. Pretending those layers can be
  exercised separately would make the markers a lie.
- A future spec lineage that adds a non-HTTP binding will refactor the
  suite into transport-neutral semantic tests + per-binding wire tests
  at that point. Chapter 09 §4 describes that future shape as
  informative; it does **not** ship in v1.
- Chapter 07's `/_reference/` paths and the §12 informative bearer-
  token authentication are out of normative scope for the suite.

## Scope

### In scope (this chunk)

1. **Harness infrastructure** under [`conformance/harness/`](../../conformance/harness/):
   - `IutAdapter` Protocol — start a subject, return a handle that
     exposes the wire-protocol base URL and any auth material the
     subject needs.
   - pytest plugin that wires fixtures (`iut`, `wire_client`,
     `experiment_id`, `event_cursor_helper`) and the `conformance`
     marker.
   - `WireClient` — thin httpx wrapper that talks raw HTTP/JSON, NOT
     through the existing `eden_wire.StoreClient`. The conformance
     suite needs to see status codes, problem+json bodies, and
     content-type headers; `StoreClient` papers over exactly those.

2. **Reference IUT adapter** under
   [`conformance/adapters/reference/`](../../conformance/adapters/reference/):
   - Spawns `python3 -m eden_task_store_server` in a subprocess on an
     ephemeral port, parses the `EDEN_TASK_STORE_LISTENING` line for the
     port, and tears the subprocess down on `stop()`.
   - The reference adapter is a **task-store-only** subject. The chunk
     does NOT spawn the reference orchestrator or worker hosts — chunk
     11a + 11b's scope is task-store + wire-binding behavior. Role
     contracts (which would require driving the orchestrator alongside
     the store) are chunk 11c.
   - Adapter lifecycle is **per-scenario**: each test gets a fresh
     subprocess. Sharing across scenarios is incompatible with the
     reference task-store-server, which binds one `--experiment-id` at
     start (codex review round 0 finding #6); per-scenario lifecycle
     also gives free isolation guarantees.

3. **State-machine scenarios** under
   [`conformance/scenarios/`](../../conformance/scenarios/) (chapters
   02 / 04 / 05 / 08):
   - Task lifecycle: every legal transition; every illegal transition
     returns the spec-pinned rejection.
   - Claim tokens: §3.2 freshness/uniqueness, §3.3 authorization, §3.4
     no-reclaim-while-claimed.
   - Submit idempotency: §4.2 content-equivalent acceptance vs
     divergent rejection vs post-terminal rejection.
   - Reclamation: §5.1 case matrix; §5.2 token invalidation; §5.4
     starting-trial reconciliation on implement reclaim.
   - Event log: §1 envelope shape; §3 per-type payload fields
     (`task.claimed.worker_id`, `task.failed.reason`,
     `task.reclaimed.cause`, `proposal.dispatched.task_id`,
     `trial.integrated.trial_commit_sha`, etc.); §4.1 total order; §4.3
     at-least-once via replay; §4.4 replayability from cursor 0.
   - Composite commits: §2.2 — implement dispatch, implement terminal,
     evaluate terminal (`success` / `error` / `eval_error`), implement
     reclaim with in-flight trial.

4. **Wire-binding scenarios** (chapter 07):
   - Status-code mapping per §§2-6: every operation returns the
     spec-pinned status.
   - RFC 7807 problem+json envelope shape on every non-2xx response per
     §7 (content-type, all five envelope fields, `type` matching
     `eden://error/<name>`).
   - Closed `eden://error/<name>` vocabulary per §7: every error type
     observed across the suite belongs to the closed table; every
     entry in the table is observed at least once across the suite.
   - `X-Eden-Experiment-Id` header **disagreement** is rejected with
     `eden://error/experiment-id-mismatch` per §1.3. The suite does NOT
     test header absence; §1.3 only pins the disagreement case (codex
     review round 0 finding #2).
   - Same-value idempotency on `integrate_trial` per §5: first call
     200 + emit `trial.integrated`; same-SHA call 200 + no second
     event; different-SHA call 409 `invalid-precondition`; non-success
     trial 409 `invalid-precondition`.
   - The suite does NOT test `/v0/`-rooting (chapter 07 §1.2 makes
     `/v0/` a positive requirement, not a "non-`/v0/` returns 404"
     requirement) or auth (chapter 07 §11-12 informative).

5. **Self-validation scenario** (`conformance/scenarios/_meta/`):
   A scenario that runs the suite as a subprocess against a wrapper
   adapter that injects one specific deviation, then asserts:
   - the targeted test failed by exact test ID,
   - the failure message contains the expected substring,
   - the wrapper proxy's hit counter recorded ≥ 1 mutated response, and
   - the **same** suite passes against the unmodified reference adapter
     in the same meta test (control). This is the bootstrap-validation
     proof: the suite has teeth AND the teeth aren't biting innocents.

6. **Spec chapter [`spec/v0/09-conformance.md`](../../spec/v0/09-conformance.md)**:
   See §G. The chapter is short and primarily declarative. The IUT
   adapter and the v0-reference auth scheme are explicitly **informative**,
   not normative — the only normative content is "to be tested by the
   v1 suite, an IUT MUST implement chapter 07."

7. **CI wiring** (`conformance` job in
   [`.github/workflows/ci.yml`](../../.github/workflows/ci.yml)):
   `uv sync --all-packages` + `uv run pytest -q conformance/`. Branch
   protection NOT updated to require it in this chunk (same posture as
   compose-smoke / compose-smoke-subprocess / compose-smoke-subprocess-
   docker / compose-e2e took for newly-added jobs).

8. **AGENTS.md commands table** entry for the suite invocation, plus
   citation tooling (§I).

### Out of scope (deferred to 11c / 11d / 12 / 13)

- **Role-contract scenarios** (planner / implementer / evaluator
  submission semantics): chunk 11c.
- **Integrator scenarios** (squash shape, eval-manifest shape, `work/*`
  access discipline): chunk 11d.
- **Multi-experiment / lease scenarios**: Phase 12.
- **Non-HTTP wire bindings**: not currently spec'd. The marker split
  is forward-looking only.
- **Conformance against deployed Compose stack**: the reference adapter
  spawns one Python subprocess against an in-process `SqliteStore` at
  `:memory:`. Container-level conformance is `compose-smoke` /
  `compose-e2e`'s job.
- **Concurrency stress / chaos scenarios**: future Phase 11 follow-up
  if they prove necessary; the v1 suite asserts spec-level behavior,
  not behavior under fault injection. The atomicity scenario in §E.4
  is explicitly framed as a regression test, not a certification.
- **SSE/WebSocket push transports**: chapter 07 §6.2 long-poll
  subscribe IS in scope; SSE/WebSocket is OPTIONAL per chapter 07 §6.2
  and out of scope here.
- **Non-`/v0/` path behavior**: chapter 07 §1.2 does not pin a
  response shape for non-`/v0/` paths; the suite does not assert one.
- **Authentication conformance**: chapter 07 §§11-12 are informative.

### B. The IUT-adapter Protocol

In [`conformance/harness/adapter.py`](../../conformance/harness/adapter.py):

```python
from __future__ import annotations

from dataclasses import dataclass, field
from pathlib import Path
from typing import Mapping, Protocol


@dataclass(frozen=True)
class IutHandle:
    """Handle returned by IutAdapter.start; everything the harness
    needs to drive the IUT through chapter-07 HTTP.
    """

    base_url: str
    experiment_id: str
    # Headers the harness should attach to every request. Empty for
    # IUTs without auth; the reference adapter sets
    # {"Authorization": "Bearer <token>"}. Auth is informative per
    # chapter-07 §12, so the conformance contract does not require
    # any specific scheme; it just gives the IUT a way to inject one.
    extra_headers: Mapping[str, str] = field(default_factory=dict)


class IutAdapter(Protocol):
    """Conformance-suite adapter for an EDEN task-store implementation.

    A conforming third-party implementation provides ONE concrete
    subclass. The reference adapter under conformance/adapters/reference
    is one example.

    The adapter contract is informative per chapter 09 §6. Non-Python
    implementors are free to write their own runner that drives an
    IUT through chapter-07 HTTP without using this Python Protocol at
    all; the spec only requires the IUT to implement chapter 07.
    """

    def start(
        self,
        *,
        experiment_config_path: Path,
        experiment_id: str,
    ) -> IutHandle: ...

    def stop(self) -> None: ...
```

The handle is **flat** (no callable for fetching a fresh token mid-
scenario); every secret a scenario needs is observable through the
wire binding. The handle's `extra_headers` field replaces a bespoke
`shared_token` so the contract covers header-based auth (`Authorization:
Bearer …`, custom header, or none at all). Transport-level auth
(mTLS, IP allowlists) is NOT supported by this adapter shape — those
schemes would need an extended contract that exposes the underlying
httpx client config (TLS context, transport hooks). v1 declines to
support them; an IUT that requires transport-level auth must front
itself with a TLS-terminating proxy that adds an `Authorization`
header for the harness, or implement a richer adapter for its own
suite.

The adapter is **per-scenario**: each test gets `start()` → use →
`stop()`. The reference adapter takes ~2 seconds per cycle (subprocess
spawn + uvicorn startup + port readiness); ≤ 50 scenarios in v1 means
the full suite runs in ~2 minutes locally and ~5 minutes in CI. That
is an acceptable cost for the isolation guarantees per-scenario start
provides.


The `WireClient` is a thin httpx wrapper that:

- Talks raw HTTP/JSON, returning httpx `Response` objects (not decoded
  models) so scenarios can assert on status codes, headers, and body.
- Always sets `X-Eden-Experiment-Id` per chapter 07 §1.3 (the suite
  tests the disagreement case explicitly by overriding it).
- Adds `iut.extra_headers` to every request.
- Does NOT retry, NOT decode problem+json into typed exceptions, NOT
  paper over errors. All of those would mask the wire shape the
  suite is verifying.

### D. The reference IUT adapter

In [`conformance/adapters/reference/adapter.py`](../../conformance/adapters/reference/adapter.py):

```python
import secrets
import subprocess
import sys
import threading
import time
from pathlib import Path

from conformance.harness.adapter import IutAdapter, IutHandle


class ReferenceAdapter:
    """Spawns a fresh task-store-server subprocess against in-memory storage."""

    _PORT_ANNOUNCE_TIMEOUT = 15.0
    _STOP_TIMEOUT = 5.0

    def __init__(self) -> None:
        self._proc: subprocess.Popen | None = None
        self._stderr_buf: list[str] = []
        self._stderr_thread: threading.Thread | None = None

    def start(self, *, experiment_config_path: Path, experiment_id: str) -> IutHandle:
        token = secrets.token_urlsafe(24)
        self._proc = subprocess.Popen(
            [
                sys.executable, "-m", "eden_task_store_server",
                "--store-url", ":memory:",
                "--experiment-id", experiment_id,
                "--experiment-config", str(experiment_config_path),
                "--shared-token", token,
                "--subscribe-timeout", "2.0",
                "--port", "0",
                "--host", "127.0.0.1",
            ],
            stdout=subprocess.PIPE,
            stderr=subprocess.PIPE,
            text=True,
        )
        port = self._read_port_announcement()
        self._start_stderr_drain()
        return IutHandle(
            base_url=f"http://127.0.0.1:{port}",
            experiment_id=experiment_id,
            extra_headers={"Authorization": f"Bearer {token}"},
        )

    def stop(self) -> None:
        if self._proc and self._proc.poll() is None:
            self._proc.terminate()
            try:
                self._proc.wait(timeout=self._STOP_TIMEOUT)
            except subprocess.TimeoutExpired:
                self._proc.kill()
                self._proc.wait(timeout=2.0)
        # Drain stderr so the next scenario's pytest output isn't polluted.
        if self._stderr_thread:
            self._stderr_thread.join(timeout=1.0)
```

The `_read_port_announcement` helper duplicates the corresponding
helper from
[`reference/services/orchestrator/tests/test_e2e.py`](../../reference/services/orchestrator/tests/test_e2e.py)
intentionally — copy is cheap, dependency from `conformance/` into
`reference/services/orchestrator/tests/` is wrong direction.
`conformance/` MUST NOT import from `reference/` outside
`conformance/adapters/reference/`. CI gates this; see §H.

The reference adapter takes a deliberate shortcut: `SqliteStore(":memory:")`
rather than `PostgresStore`. Conformance does not test durability;
[`reference/packages/eden-storage/tests/`](../../reference/packages/eden-storage/tests/)
does. SQLite-in-memory gives the fastest possible test cycle.

`--subscribe-timeout 2.0` is passed so the timeout-path subscribe test
in §E.9 (`test_subscribe_empty_batch_after_timeout`) completes in 2-3
seconds rather than the 30-second default. Chapter 07 §6.2 makes 30s
a `RECOMMENDED:` value (i.e. SHOULD), so a 2-second window is
spec-conforming. The reference-adapter-specific override is acceptable;
a third-party adapter SHOULD pass an equivalently-short timeout for
its own subject. The chapter-07 §6.2 conformance assertion in the
test is "the subscribe response after timeout has the empty-batch
shape," not "the timeout is 30 seconds."

### E. Scenario inventory

Every scenario's first docstring line cites the exact normative MUST
it tests, in the format `spec/v0/<chapter>.md §<sec>`. The
`tools/check_citations.py` helper (§I) walks every test and verifies
the citation resolves. Any test whose citation cannot be verified is
a CI failure.

#### E.1 Task lifecycle (`test_task_lifecycle.py`)

Each test creates a fresh `plan` task and exercises one transition.
Tests assert both the post-condition (state machine) and the wire
response (status + error type for negative cases).

| Test | Asserts | Citation |
|---|---|---|
| `test_create_task_pending` | After `POST /tasks`, the task exists with `state == "pending"`, no `claim`. | 04 §2; 02 §3.1 |
| `test_pending_to_claimed` | `POST /tasks/{T}/claim` returns 200 with claim object; subsequent read shows `state == "claimed"`. | 04 §1.2; §3.2 |
| `test_claimed_to_submitted` | Submit with current token + plan payload returns 200; `state == "submitted"`. | 04 §4.1 |
| `test_submitted_to_completed` | `POST /tasks/{T}/accept` returns 200; `state == "completed"`. | 04 §4.3 |
| `test_submitted_to_failed_via_reject` | Reject with valid `reason` returns 200; `state == "failed"`. | 04 §4.3 |
| `test_terminal_completed_rejects_writes` | After accept, every write op (claim, submit, accept, reject, reclaim) returns 409 `eden://error/illegal-transition`. | 04 §1.1, §4.4 |
| `test_terminal_failed_rejects_writes` | Same for `failed`. | 04 §1.1 |
| `test_pending_rejects_submit` | Submit on pending task returns 409 `illegal-transition`. | 04 §1.2 |
| `test_pending_rejects_accept` | Accept on pending task returns 409. | 04 §1.2 |
| `test_pending_rejects_reject` | Reject on pending task returns 409. | 04 §1.2 |
| `test_claimed_rejects_accept` | Accept-without-submit returns 409. | 04 §1.2 |
| `test_claim_rejected_when_not_pending` | Re-claim a `claimed` task returns 409. | 04 §3.4 |
| `test_terminal_rejects_reclaim` | Reclaim of a `completed` or `failed` task returns 409. | 04 §5.1 |

#### E.2 Claim tokens (`test_claim_tokens.py`)

| Test | Asserts | Citation |
|---|---|---|
| `test_token_present_on_claim` | Successful claim response carries `claim.token`. | 02 §3.4 |
| `test_token_unique_across_tasks` | Two parallel claims on different tasks produce different tokens. | 04 §3.2 |
| `test_token_unique_across_reclaim_cycles` | Reclaim → re-claim same task → token differs. | 04 §3.2; §5.2 |
| `test_wrong_token_rejected` | Submit with a fabricated token returns 403 `eden://error/wrong-token`. | 04 §3.3 |
| `test_token_invalidated_by_reclaim` | After operator reclaim, the prior token returns 403 wrong-token on submit. | 04 §5.2 |
| `test_no_reclaim_while_claimed` | Second worker's claim against a claimed task returns 409 illegal-transition. | 04 §3.4 |

The harness does NOT attempt to assert "tokens are unforgeable" — that
is a SHOULD/MAY mechanism in chapter 04 §3.2. The MUST is uniqueness,
which is testable.

### F. Self-validation scenario

In [`conformance/scenarios/_meta/test_self_validation.py`](../../conformance/scenarios/_meta/test_self_validation.py).

The pre-review draft's `pytest.main()`-in-pytest design is replaced
with a subprocess pytest invocation per codex review round 0 finding 5.
The meta scenario:

1. **Constructs a `MisbehavingAdapter`** that wraps `ReferenceAdapter`
   with an in-process httpx-Mocked-Transport-shaped HTTP proxy
   (see below). The proxy:
   - Listens on its own ephemeral port.
   - Forwards all requests transparently to the underlying real IUT.
   - Special-cases ONE URL pattern (`POST /v0/.../tasks/.../submit`
     with a token mismatch detected by reading and re-emitting the
     real backend's response): if the real IUT would have returned
     403 `eden://error/wrong-token`, the proxy mutates the response
     to 200 OK with a fabricated success body.
   - Maintains a `mutation_count` counter incremented whenever it
     mutated a response.
   - Lifecycle is context-managed: enter → start proxy → real IUT;
     exit → real IUT.stop() → proxy.stop().

2. **Runs three subprocess pytest invocations** via
   `subprocess.run([sys.executable, "-m", "pytest", ...])`:

   a. **Control run (must pass).** Target:
      `conformance/scenarios/test_claim_tokens.py::test_wrong_token_rejected`
      against the unmodified `ReferenceAdapter`.
      Assertion: exit code 0.

   b. **Broken run (must fail).** Same target test against
      `MisbehavingAdapter`. Assertion: exit code != 0 AND
      `--json-report-file` shows `test_wrong_token_rejected` as
      `outcome=failed` AND its `longrepr` contains the substring
      `"Expected 403 wrong-token, got 200"` (the exact assertion
      text from the broken scenario; this proves the failure was the
      expected assertion, not an import error or proxy-startup
      error).

   c. **Hit counter check (must be ≥ 1).** After the broken run,
      read the `MisbehavingAdapter`'s `mutation_count` (exposed via
      a sidecar file the proxy writes to; the meta test reads it
      after the subprocess exits). Assertion: count ≥ 1.

3. **All three assertions pass** ⇒ the meta scenario is the proof
   that the suite has teeth. Any single-failure fails the meta.

The proxy implementation lives at
[`conformance/scenarios/_meta/_misbehave_proxy.py`](../../conformance/scenarios/_meta/_misbehave_proxy.py)
and uses `aiohttp` (pinned in `conformance/pyproject.toml`). It is
~80 lines: a single-purpose forwarder that mutates one specific
response shape. It is **not** a general MITM library.

The misbehaving adapter exposes its own `--shared-token` because the
sidecar misbehaving proxy needs to read+forward authenticated requests
unchanged.

#### F.1 Why three sub-runs

- The control run (a) defends against false-positives: if a real bug
  has already broken the wrong-token assertion in the suite, the
  control fails and we know the meta scenario isn't the right
  diagnosis.
- The broken run (b) defends against silent harness regressions: if
  someone refactors the wrong-token check away, the broken run starts
  passing, and the meta fires.
- The hit counter (c) defends against drift in the proxy itself: if
  the harness changes how it tests wrong-token (e.g., starts using a
  different URL or status code), the proxy stops matching the request,
  the broken run still fails (because `wrong-token` no longer fires),
  but the hit counter goes to 0 and the meta fires.

#### F.2 Why this is not "just an extra test"

The self-validation scenario is the closest thing to a proof that the
suite asserts what it claims. Chapter 09 §3 leans on this scenario
when it says "every assertion in the suite is keyed off a normative
MUST." The meta is the integrity check that connects the assertion
text to a concrete observable failure.

### G. Spec chapter 09

Chapter 09 is short by design. The chapter's job is to declare the
relationship between the spec, the suite, and any IUT — not to
duplicate content from chapters 02 / 04 / 05 / 06 / 07 / 08.

```markdown
# Conformance

This chapter declares what conformance to EDEN protocol v0 means and
how it is verified.

## 1. What conformance is

A v0-conforming IUT (implementation under test) implements chapter
07's HTTP wire binding with the normative semantics of chapters 02 /
03 / 04 / 05 / 06 / 08.

Conformance is verified by passing the conformance suite under
[`conformance/`](../../conformance/) in this repo: a set of black-box
scenarios that drive an IUT through chapter-07 HTTP and assert MUSTs
from those chapters. The suite is delivered in three increments —
**v1**, **v1+roles**, and **v1+roles+integrator** — each adding a
group of scenarios to the prior level. An IUT MAY claim the level
that matches the scenarios it passes; "v0 conformance" without a
qualifier means the highest level shipped at the time of the claim.

The suite is one expression, not the only one. A future suite — in a
different language, against a different harness — that drives the same
chapter-07 endpoints and asserts the same MUSTs would be equally
authoritative. The spec is the source of truth; the suite is one
faithful implementation of "what would test that?"

## 2. The IUT

An IUT MUST:

- Expose chapter 07's HTTP binding at a deployment-chosen base URL.
- Accept an experiment configuration whose shape matches
  `experiment-config.schema.json`.

Anything else — language, deployment topology, persistence, auth
scheme, scaling profile — is a deployment choice. Chapter 07 §11
enumerates further latitude.

## 3. Assertion vocabulary

Every assertion in the v1 suite is keyed off a normative MUST in
chapters 02 / 04 / 05 / 06 / 07 / 08. The suite does NOT assert
SHOULDs; SHOULDs are interop guidance, not interop contracts.

When the spec evolves, the suite evolves with it. A change that
removes a MUST or downgrades it to SHOULD MUST be accompanied by a
matching scenario removal or rewrite.

The suite intentionally does NOT certify §1.3-class atomicity or §3.2
unforgeability properties; black-box testing cannot prove the absence
of a sufficiently-narrow window or the absence of a sufficiently-weak
random source. The v1 suite asserts the testable consequences of these
invariants (event presence, uniqueness across observed claims) and
labels the regression-style tests as such.

## 4. Conformance levels

The v0 spec defines exactly one wire binding (chapter 07 HTTP). The
suite drives every IUT through it; transport-neutral semantic tests
vs. per-binding wire tests cannot be meaningfully separated until a
second binding exists. Levels are therefore distinguished by
**scenario coverage**, not transport:

- **v1** — task-store and event-log MUSTs from chapters 02 / 04 /
  05 / 07 plus the chapter-08 storage MUSTs the wire binding exposes
  (§1.1, §1.7). v1 does NOT cover chapter-03 role contracts or the
  chapter-06 integrator atomicity ladder.

- **v1+roles** — adds chapter-03 role-contract scenarios (per-role
  submission semantics, backpressure, idempotency). Lands in chunk
  11c.

- **v1+roles+integrator** — adds chapter-06 integrator scenarios
  (squash shape, eval-manifest shape, `work/*` access discipline,
  the §3.4 atomicity ladder under transport-indeterminate failures).
  Lands in chunk 11d.

A future spec lineage that introduces a second wire binding will at
that point split the suite into transport-neutral semantic tests +
per-binding wire tests, and chapter 09 will gain a `core` level
claimable by IUTs implementing only the semantic layer. The marker
structure in [`conformance/`](../../conformance/) anticipates that
refactor but does not enable it in v0.

## 6. Adapter (informative)

A v1 IUT does NOT need to ship a Python adapter. The reference suite's
adapter shape — under
[`conformance/harness/adapter.py`](../../conformance/harness/adapter.py)
— is one convenience for IUTs that prefer to integrate with the
reference Python harness; it is not a normative requirement. A non-
Python implementor MAY:

- Write a Python `IutAdapter` subclass that spawns their service.
- Run the suite against an already-running IUT pointed at by an
  environment variable, via a thin Python adapter that simply
  returns an `IutHandle` to that URL.
- Re-implement the suite in another language entirely, asserting the
  same MUSTs.

The contract between an IUT and a conformance harness is the chapter
07 HTTP binding. Everything else is convenience.

## 7. Reference posture

The reference implementation in [`reference/`](../../reference/) is
one demonstration of v0 conformance at the highest currently-shipped
level (v1 at the time chapter 09 first lands; v1+roles after chunk
11c; v1+roles+integrator after chunk 11d). CI gates the reference
impl on every level the suite has shipped, via the `conformance` job.
A reference-impl test failure that cannot be cited to a normative
MUST is either a suite bug or a spec gap; the resolution is one of:
amend the suite, amend the spec, fix the impl. The reference impl
does NOT have authority to reinterpret the spec.
```

The chapter is intentionally informative on the adapter shape and on
the v0-reference auth mechanism (chapter 07 §12). Codex review round 0

party implementations to expose a Python-harness-friendly lifecycle
shim or a bearer-token auth scheme.

1. Walks every test under `conformance/scenarios/`.
2. Extracts the first docstring line.
3. Parses out `spec/v0/<chapter>.md §<sec>`.
4. Asserts the cited section exists (using the existing
   [`scripts/spec-xref-check.py`](../../scripts/spec-xref-check.py)
   helper as a model).
5. Validates that every entry in the chapter 09 §5 scenario index has
   at least one citing test (drift between the chapter and the suite
   is a CI failure — codex review round 0 implicit risk).
6. Emits a coverage report: which spec sections have at least one
   citing test, and the union of cited sections.

CI fails if (3) doesn't parse, (4) doesn't resolve, or (5) leaves an
index entry uncovered.

### K. Self-host / packaging

The harness is invoked **without** the conformance/ package being a
development dependency of the reference services — i.e. running
`uv run pytest -q reference/` doesn't pull in conformance machinery.
This is enforced by:

- `conformance/pyproject.toml` declares `conformance` as a separate uv
  workspace member with its own dependency closure (pytest, httpx,
  aiohttp, pytest-ordering, pytest-json-report).
- The reference services' tests do NOT import from `conformance/`.
- Root `pyproject.toml` lists `conformance` under
  `[tool.uv.workspace] members` but reference packages don't depend
  on it.

A non-Python implementor can clone the repo, run `uv sync --package
conformance` (or any equivalent venv), and point pytest at their
adapter. They never run `uv sync --all-packages`.

## Known risks / things to watch

- **Chapter 09 minimalism vs. discoverability.** The chapter is short
  on purpose; an implementor who wants to know "how do I structure my
  service?" reads chapters 02-08, not 09. Round-1 review may push
  back ("chapter 09 is too short, expand it") — the response is that
  chapter 09 is a meta-chapter and the suite + the other chapters
  carry the real content. If round-1 disagrees with strong reasons,
  the chapter expands; if it disagrees on style, hold firm.

- **Atomicity scenario is best-effort by design.** The §E.5 docstring
  is explicit. A reviewer who mistakes it for a §1.3 certification
  will push back; the response is that black-box testing cannot prove
  §1.3 in the strict linearizability sense, and a Jepsen-style harness
  is out of scope for v1.

- **Self-validation broken-adapter is a moving target.** If a future
  refactor changes the wrong-token error format, the proxy's mutation
  no longer triggers. Mitigation: §F.1 hit-counter check fires at
  count=0; meta fails.

- **`pytest-ordering` for the vocabulary-closure test in §E.12.** The
  closure test must run last. `pytest-ordering` is a small pinned
  plugin; if it ever breaks, scenario ordering becomes filesystem-
  walk-dependent and the closure test runs with a partial vocabulary.
  Mitigation: document the dependency in `conformance/README.md` and
  in the test's docstring.

- **Adapter per-scenario start cost.** ~2 seconds × ~70 tests ~= 140s
  locally, 250-300s in CI. If the suite grows much larger, the
  per-scenario lifecycle becomes slow. Mitigation for v1 is "accept
  it"; a future optimization is `--iut-mode=session` once the adapter
  contract supports per-experiment provisioning. Out of scope for
  this chunk.

- **`integrate_test_trial_factory` is the longest single fixture.**
  6 wire calls to set up an integrate test's prerequisites. A bug in
  the fixture becomes a bug in 4 unrelated tests. Mitigation: the
  fixture is itself unit-tested by an `_meta` scenario
  (`test_integrate_test_trial_factory_produces_success_trial`) that
  asserts the post-condition `read_trial(T).status == "success"`.

- **Reference-impl quirks fossilizing into the suite.** Chapter 09 §3
  is the discipline. Concrete example: if the reference impl emits
  `worker_id` on `task.submitted` (chapter 05 §3.1 says SHOULD), the
  suite MUST NOT assert it. `tools/check_citations.py` walks the cited
  section and rejects scenarios whose docstring cites a SHOULD/MAY
  rather than a MUST.

- **Per-event-type assertions explode in count.** §E.7 has 15 tests,
  one per registered type. Maintenance burden if §3 changes; gain is
  exhaustive coverage. Mitigation: each test is mechanical
  (load JSON Schema for that type → assert payload validates), so
  refactoring is straightforward.
