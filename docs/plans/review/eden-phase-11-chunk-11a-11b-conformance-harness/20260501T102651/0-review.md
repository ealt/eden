**Findings**

1. The `core` vs `wire-http` split is not actually drawn on semantic boundaries. [plan L265](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:265>) defines `conformance_core` as chapter 02/04/05/06/08 only, but the proposed core tests assert HTTP statuses and `eden://error/...` types, for example [plan L413](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:413>), [plan L432](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:432>), [plan L449](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:449>), and [plan L467](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:467>). That makes `core` unclaimable by any future non-HTTP binding even though chapter 09 says it should exist independently. Either collapse v1 to one HTTP-only level, or add a transport-neutral semantic layer and move raw HTTP assertions entirely into `conformance_wire_http`.

2. Several proposed assertions are not grounded in normative MUSTs, they encode current reference behavior. [plan L491](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:491>) expects `/experiments/...` without `/v0/` to return `404`, but `07 §1.2` only says normative paths are rooted at `/v0/`; it does not require `404` for aliases or redirects. [plan L492](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:492>) expects a missing `X-Eden-Experiment-Id` header to be specifically `experiment-id-mismatch`, but `07 §1.3` only pins that error for disagreement, not absence. [plan L526](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:526>) asserts unauthenticated requests return `401`, but auth is outside the normative binding and may be absent entirely.

3. Chapter 09, as sketched, makes the suite adapter and auth shape normative when they should stay informative. [plan L638](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:638>) says an IUT “MUST” be startable/stoppable via an adapter shim, and [plan L215](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:215>) / [plan L701](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:701>) make `shared_token` part of the handle contract. That conflicts with `07 §11-12`, where auth is out of scope and the reference shared-token scheme is informative only. The suite can document the adapter it consumes, but the spec should not require third-party implementations to expose a Python-harness-friendly lifecycle shim or bearer-token auth.

4. The atomicity scenario is too weak to certify the `04 §1.3` / `08 §6` invariant it claims to test. [plan L474](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:474>) says alternating `GET /tasks/{T}` and `GET /events` proves “state without event” cannot occur. It does not: a server can expose the new task state, append the event a few milliseconds later, and still pass because the next event read sees it. Long-poll subscribe narrows the window but does not remove the race. This should be framed as a best-effort black-box regression test, not as the conformance proof of atomicity.

5. The self-validation design is fragile, and its pass condition is too weak. [plan L588](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:588>) uses `pytest.main(...)` inside pytest and only asserts “non-zero”. That also passes on import failures, proxy startup failures, or re-entrant pytest/plugin-state breakage. The selected proxy approach at [plan L606](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:606>) also needs explicit lifecycle handling: ephemeral port allocation, startup synchronization, teardown of both proxy and real IUT, and a hit counter proving the proxy actually mutated the targeted wrong-token response. Use a subprocess `python -m pytest ...` and assert the specific target test failed for the expected reason.

6. The fixture model is internally inconsistent with the proposed adapter contract. [plan L307](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:307>) says session mode starts the IUT once, while [plan L308](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:308>) says each test gets a fresh `experiment_id`. But the reference adapter start command hard-binds one `--experiment-id` at process start in [plan L359](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:359>). Under the current `IutAdapter` shape, session reuse across fresh experiments is impossible. Either drop `--iut-mode=session` for v1, or expand the adapter contract to provision/reset experiments after startup.

7. Chapter 05 coverage is under-specified, and one event assertion is too loose for the spec. [plan L473](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:473>) mostly checks that some matching event exists, but `05 §1` and `05 §3` also normatively pin event envelope fields and per-type payload fields such as `task.claimed.worker_id`, `task.failed.reason`, `task.reclaimed.cause`, `proposal.dispatched.task_id`, and `trial.integrated.trial_commit_sha`. Also, [plan L478](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:478>) says `proposal.state-changed (or equivalent)`, but `05 §2.2` / `05 §3.2` require `proposal.dispatched` specifically, not an equivalent event name.

8. The scope cut around 11c is too loose in one direction and too wide in another. [plan L544](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:544>) seeds integrate tests by running plan, implement, and evaluate flows through submit/accept, which quietly pulls role-specific chapter-03 semantics back into 11a/11b. At the same time, the v1 index omits evaluate-terminal composite commits from `04 §4.3` and `05 §2.2`, even though those are task-store/event semantics and do not require worker-host bindings. The cleaner split is: seed proposals/trials directly through normative endpoints unless the scenario is explicitly about role contracts, and keep evaluate-terminal composite effects in v1.

9. `test_reclaim_cause_vocabulary_closed` is misclassified. [plan L467](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:467>) belongs under `conformance_wire_http`, not `conformance_core`. The interesting assertion there is the HTTP/schema mapping to `400 bad-request`, which is chapter 07 behavior, not chapter 04/05/08 core semantics.

**Open Questions / Assumptions**

1. I treated `spec/v0/08-storage.md` as the intended “chapter 08” source of normative storage guarantees; the repo does not have `08-control-plane.md`.

2. I assumed “`GET /events` filtered to a task ID” in [plan L475](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:475>) means client-side filtering after replay. If it means a server-side query parameter, that is another unspec'd wire assertion.

3. If the intent is “v1 is explicitly HTTP-only even though chapter 07 is optional,” chapter 09 should say that directly and defer the `core` taxonomy until a second binding harness exists.

**Overall Assessment**

The framing is strong: the plan correctly centers “spec citations, not reference observations,” and the directionality / citation-discipline ideas are good. But it is not ready to implement as written. The main problem is conceptual, not editorial: it still mixes transport semantics into `core`, and it still contains a few reference-implementation assumptions that would fossilize into chapter 09.

I would revise the plan before implementation. The minimal bar is: fix the marker boundary, remove the non-citeable HTTP assertions, make chapter 09’s adapter/auth material informative rather than normative, and either weaken or redesign the atomicity and meta-test claims so the suite does not overstate what it can prove black-box.
