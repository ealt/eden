[plan L637](</Users/Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:637>) uses repeated `GET /events?cursor=0`, which exercises replayability, not `05 §4.3` subscriber delivery semantics. [plan L279](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:279>) through [plan L285](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:285>) and [plan L313](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:313>) say `extra_headers` is enough even for mTLS, but the fixture at [plan L381](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11-chunk-11a-11b-conformance-harness.md:381>) only passes headers into a plain httpx client. Since the adapter is informative this is not a spec blocker, but the plan should stop claiming those schemes fit this interface unless you add a client-factory / httpx-config hook.

**Overall Assessment**

This is materially better than round 0. The marker-boundary problem, the non-citeable HTTP assertions, the auth/adapter overreach, the `pytest.main()` meta-test design, and the direct-role-flow seeding issue are mostly fixed.

Fix that, add the missing `trial.eval_errored` and `already-exists` coverage, and tighten the `§4.3` at-least-once test, and the plan looks ready.
