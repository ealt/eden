**2. Feasibility**

Assessment: the port is mostly mechanical, but the persisted-credential design is not solid enough as written.

- The current plan will predictably fail on a normal experiment switch. With credentials stored as `{worker_id: token}` [plan](/Users/ericalt/Documents/eden-worktrees/phase-12a-1h-cli-port/docs/plans/eden-phase-12a-1h-cli-port.md:266) and persisted tokens returned blindly if present [plan](/Users/ericalt/Documents/eden-worktrees/phase-12a-1h-cli-port/docs/plans/eden-phase-12a-1h-cli-port.md:287), rerunning `setup-experiment.sh` for a new experiment but keeping the same default worker id will reuse the old token and hit 401 on the first worker-gated call. The repo already has the canonical fix in [auth.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1h-cli-port/reference/services/_common/src/eden_service_common/auth.py:141): verify persisted creds via `/whoami`, then reissue if stale through [auth.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1h-cli-port/reference/services/_common/src/eden_service_common/auth.py:193). The plan should either adopt that behavior or namespace credentials by `EDEN_EXPERIMENT_ID`.
- The auth taxonomy in the plan is partly wrong. [docs/plans/eden-phase-12a-1h-cli-port.md](/Users/ericalt/Documents/eden-worktrees/phase-12a-1h-cli-port/docs/plans/eden-phase-12a-1h-cli-port.md:195) classifies `create_idea`, `mark_idea_ready`, and `create_variant` as “either” endpoints, but the wire spec makes them worker-gated in [spec/v0/07-wire-protocol.md](/Users/ericalt/Documents/eden-worktrees/phase-12a-1h-cli-port/spec/v0/07-wire-protocol.md:75), [same](/Users/ericalt/Documents/eden-worktrees/phase-12a-1h-cli-port/spec/v0/07-wire-protocol.md:78), and [same](/Users/ericalt/Documents/eden-worktrees/phase-12a-1h-cli-port/spec/v0/07-wire-protocol.md:86), and the auth tests enforce 403 for admin on worker-gated mutations at [test_auth.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1h-cli-port/reference/packages/eden-wire/tests/test_auth.py:416). The proposed call sites still use worker bearers, so this is plan drift rather than a fatal implementation bug, but it should be fixed before coding.

I’d stop here before evaluating alternatives/completeness/risks. The main blocker is that the plan has not settled the lifecycle of persisted worker credentials. Once it either scopes `eden-manual` state per experiment or reuses the existing verify-and-reissue bootstrap pattern, the rest of the port looks straightforward.
