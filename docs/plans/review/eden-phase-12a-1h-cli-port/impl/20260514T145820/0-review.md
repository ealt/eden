**Findings**

- `Bug` [reference/scripts/manual-ui/eden-manual](/Users/ericalt/Documents/eden-worktrees/phase-12a-1h-cli-port/reference/scripts/manual-ui/eden-manual:139): Plan §4.D says a hand-edited or wiped `.credentials.json` should recover automatically, but `_read_credentials()` blindly `json.loads()` the file and assumes it is a `dict[str, str]`. A truncated file, invalid JSON, or a non-object value crashes the CLI before the register-or-reissue ladder runs. Fix: treat parse/shape failures as recoverable input corruption, warn, and continue with an empty credential map (or move the bad file aside and rewrite it on successful recovery).

- `Risk` [reference/scripts/manual-ui/eden-manual](/Users/ericalt/Documents/eden-worktrees/phase-12a-1h-cli-port/reference/scripts/manual-ui/eden-manual:145), [reference/scripts/manual-ui/eden-manual](/Users/ericalt/Documents/eden-worktrees/phase-12a-1h-cli-port/reference/scripts/manual-ui/eden-manual:185): The new credential ladder has no serialization or atomic write around the shared `.credentials.json`. Two concurrent `eden-manual` sessions can both read stale state, then one reissue invalidates the other’s freshly minted token, or one write drops another worker’s entry. The direct `write_text(...); chmod(0o600)` path also leaves a partial-write / brief wrong-permissions window for a sensitive file. Fix: add a lock file around read/verify/register/reissue/write, and persist via a `0o600` temp file plus `os.replace()`.

- `Risk` [.claude/skills/eden-manual-executor/SKILL.md](/Users/ericalt/Documents/eden-worktrees/phase-12a-1h-cli-port/.claude/skills/eden-manual-executor/SKILL.md:112), [reference/scripts/manual-ui/eden-manual](/Users/ericalt/Documents/eden-worktrees/phase-12a-1h-cli-port/reference/scripts/manual-ui/eden-manual:607): The executor skill still says phase 3 “creates a `Variant(starting, ..., commit_sha=<sha>)` in the store”, but the CLI actually creates the starting variant without `commit_sha` and only writes `commit_sha` during task submit. That mismatch matters when troubleshooting partial failures, which this flow is already sensitive to. Fix: update the skill text to match the real three-phase order: `create_variant(starting, no commit_sha)` → push canonical ref → submit with `commit_sha`.

**Overall assessment**

The implementation mostly matches the plan: the auth port is wired correctly, the claim/submit payloads match the post-12a-1 wire shape, and the no-op guard is in the planned spot. The main gap is robustness in the new credential ladder: the recovery story is weaker than the plan claims, especially around corrupted local state and concurrent sessions. I did not run the full Compose manual verification flow; this review is based on code inspection plus targeted static/tool checks.
