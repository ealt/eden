**Findings**
- `Bug` [reference/packages/eden-dispatch/src/eden_dispatch/driver.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1-worker-identity/reference/packages/eden-dispatch/src/eden_dispatch/driver.py:191): `run_orchestrator_iteration` lets `create_execution_task` / `create_evaluation_task` conflicts escape. Plan §3.4’s exact-idempotent story explicitly requires replica B to observe the first commit and treat the duplicate as benign; here the exception bubbles out and kills the replica instead. I confirmed this locally by forcing `create_execution_task` to raise `AlreadyExists`; `run_orchestrator_iteration` propagated it. Fix: catch duplicate-dispatch conflicts (`AlreadyExists` / the stale-read `InvalidPrecondition` path) per task and continue the iteration.

- `Bug` [reference/services/orchestrator/src/eden_orchestrator/loop.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1-worker-identity/reference/services/orchestrator/src/eden_orchestrator/loop.py:40): `_read_dispatch_mode()` fails open to all-`auto` on any read error. That matches the plan text, but it does not match the shipped spec: `03-roles.md §6.1` says manual mode means the orchestrator `MUST NOT` run that decision. A transient store read failure can therefore cause a forbidden dispatch/integration during an operator manual window. Fix: fail closed for the four gated decisions on read failure, while optionally still finalizing already-submitted tasks.

- `Bug` [reference/packages/eden-wire/src/eden_wire/models.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1-worker-identity/reference/packages/eden-wire/src/eden_wire/models.py:195), [reference/packages/eden-wire/src/eden_wire/server.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1-worker-identity/reference/packages/eden-wire/src/eden_wire/server.py:705), [reference/packages/eden-storage/src/eden_storage/_base.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1-worker-identity/reference/packages/eden-storage/src/eden_storage/_base.py:1163): `PATCH /dispatch_mode` misclassifies invalid values on unknown keys. Because the request model uses `extra="allow"`, a payload like `{"future_key":"paused"}` reaches the Store, which raises `InvalidPrecondition`; the wire then returns `409 eden://error/invalid-precondition`. I confirmed that behavior locally. `04-task-protocol.md §7.1` / `07-wire-protocol.md §2.8` require bad values to be a 400-class request error. Fix: validate all patch values in the wire layer and raise `BadRequest`, or change the Store-side error class/mapping for this path.

- `Risk` [reference/packages/eden-storage/src/eden_storage/_schema.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1-worker-identity/reference/packages/eden-storage/src/eden_storage/_schema.py:129), [reference/packages/eden-storage/src/eden_storage/_postgres_schema.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1-worker-identity/reference/packages/eden-storage/src/eden_storage/_postgres_schema.py:111): wave 2 never added the live-task unique indexes promised in plan §5.2. The current reference server still works because one store instance serializes operations in-process, but the DB no longer enforces the “mechanical” at-most-one-live execution/evaluation guarantee the plan called for. Fix: add the partial unique indexes from the plan, or explicitly narrow the guarantee/docs to the current single-process server posture.

- `Risk` [reference/compose/healthcheck/smoke-multi-orchestrator.sh](/Users/ericalt/Documents/eden-worktrees/phase-12a-1-worker-identity/reference/compose/healthcheck/smoke-multi-orchestrator.sh:112): the failover smoke does not actually prove secondary-only takeover. It `docker kill`s the primary, but `orchestrator` is configured with `restart: "on-failure"`, and the script merely sleeps 3 seconds before waiting on `orchestrator-2`. The restarted primary can continue doing work, so the smoke can pass without demonstrating the property claimed in plan §4.1 / the phase summary. Fix: disable restart or explicitly stop/remove the primary after the kill, then assert the secondary alone drives quiescence.

- `Risk` [reference/compose/healthcheck/e2e_drive.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1-worker-identity/reference/compose/healthcheck/e2e_drive.py:275), [reference/compose/healthcheck/e2e.sh](/Users/ericalt/Documents/eden-worktrees/phase-12a-1-worker-identity/reference/compose/healthcheck/e2e.sh:210): the reassign drill no longer verifies what it claims. `_reassign_drill()` checks only for `303 ...reassigned=ok`; it never re-reads the task to assert the target became `ideator-1`, and the shell smoke only asserts that some `task.reassigned` event exists. A regression in form wiring or target selection could still pass. Fix: re-read the task after POST and assert the exact target, then tie the final event assertion to that specific `task_id` / `new_target` / `reassigned_by`.

**Overall Assessment**
The implementation mostly covers the planned surface area, but I would not call it cleanly conformant yet. The largest problems are in the exact-idempotent dispatch path and failure-mode semantics: duplicate dispatch races currently crash a replica instead of collapsing benignly, manual mode fails open on `dispatch_mode` read errors, and the wire layer returns the wrong error class for at least one invalid `PATCH /dispatch_mode` input. The smoke/e2e coverage also overstates what it proves, and the wave-2 migration stopped short of the plan’s DB-level uniqueness enforcement.

I did not run the full suite; this was a static review plus two targeted local reproductions (`run_orchestrator_iteration` propagating `AlreadyExists`, and `PATCH {"future_key":"paused"}` returning `409 invalid-precondition`).
