## Change Summary

- **Dispatch mode gating** — Orchestrator reads `dispatch_mode` at iteration start; each of the 4 decisions (`ideation_creation` / `execution_dispatch` / `evaluation_dispatch` / `integration`) gated independently. Finalize (`accept`/`reject` of submitted tasks) intentionally NOT gated.
- **Multi-instance safety** — Bounded-overshoot for ideation creation (N*T cap), exact-idempotent for execution/evaluation/integrate via at-most-one-live invariants in the Store.
- **§3.7 authority matrix** — `create_task(ideation|evaluation)` requires `admins` OR `orchestrators`; `create_task(execution)` requires `orchestrators` only; `accept`/`reject`/`integrate_variant` require `orchestrators`; `reassign_task`/`update_dispatch_mode` require `admins`. The deployment-admin bearer is bootstrap-only.
- **Web UI** — operator-facing routes for dispatch-mode toggle + reassign; web-ui's `web-ui-1` worker pre-registered in `admins` so it can drive the admin-gated routes (until the per-user session-bearer retrofit lands).
- **Smoke scripts** — `smoke-manual-mode.sh` PATCHes dispatch_mode BEFORE workers come up (the post-`compose up --wait` race is unwinnable because the ideator-host claims+submits in ms); `smoke-multi-orchestrator.sh` uses the §6.4 N*T bounds for cardinality (not strict equality).
