**Stopped At Level 2: Feasibility**

- The manual-evaluation path is still not propagated coherently through the rest of the plan. The updated decision and authority table now put operator-created evaluation tasks in scope at [Decision 8](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:177), [3.7](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:468), and [6.1](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:775), but [3.9](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:529) still says “No operator-driven execution / evaluation task creation,” and [4.2](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:627) still defers `kind=evaluation` task creation to 12a-3. That leaves the implementation scope contradictory: an executor reading the non-goals/followups section would reasonably conclude the new admin evaluation-create path should not land in 12a-2.

- The worker bootstrap fix is still internally inconsistent with 12a-1. The revised design now uses `register_worker(self_id)` plus `add_to_group(self_id, "orchestrators")` at [Decision 6](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:160), which is the right 12a-1 shape. But [3.7](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:463) and [3.8](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:493) still describe `register_worker(self_id)` as open self-registration, while 12a-1 clearly makes `register_worker` admin-authenticated at [12a-1 D.5](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:322). The files-to-touch list also still encodes the old shape in the shared helper signature at [5.6](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:711), which still takes a `group` parameter. As written, the plan mixes three models: open self-registration, admin-gated registration, and helper-level implicit group assignment.

- The new barrier-based HA test design deadlocks as written. [6.3](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:813) says each `create_execution_task` / `create_evaluation_task` call enters a shared `threading.Barrier(n)` immediately after acquiring the per-store lock. But the current shared store base explicitly uses a single in-memory `RLock` transaction scope at [reference/packages/eden-storage/src/eden_storage/_base.py:22](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/_base.py:22). If thread A acquires that lock and then waits at the barrier, thread B cannot acquire the same lock to reach the barrier, so the test wedges. The synchronization point has to move before the store lock/transaction begins, or the instrumentation has to be attached deeper at a backend-specific point where both contenders can enter without serializing on the same lock first.

**Overall Assessment**

The round-2 fixes improved the main design sections, but feasibility is still not clear enough to proceed to alternatives or completeness. The remaining problems are now narrower: scope propagation for manual evaluation, one consistent registration model, and a race test that can actually run. Once those are fixed, the next pass should be able to move past Level 2.
