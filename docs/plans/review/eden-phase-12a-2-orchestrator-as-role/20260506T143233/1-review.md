**Stopped At Level 2: Feasibility**

- Manual `evaluation_dispatch` is still internally contradictory. The plan says manual mode means a human/operator uses “the same wire ops” the auto-orchestrator would use at [3.1 §6.1](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:197) and the Web UI test plan explicitly requires an operator to create an evaluation task while `evaluation_dispatch=manual` at [6.2](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:760). But [Decision 8](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:172) defers operator-driven `kind=evaluation` creation to 12a-3, and the authority table only allows `create_task(kind=execution | evaluation)` for `orchestrators` at [3.7](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:442). Those cannot all be true at once. If `evaluation_dispatch=manual` is in scope now, operator-created evaluation tasks must be in scope now too; `intended_executor` is only relevant to execution tasks, not evaluation tasks.

- The worker/group bootstrap model in 12a-2 does not match the landed 12a-1 contract. 12a-2 repeatedly uses `register_worker(..., group="orchestrators")` at [Decision 6](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:159) and [3.8](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:467), and the authority table even lists `register_worker(other_id, group=...)` and open self-registration at [3.7](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:438). But 12a-1 defines `register_worker(worker_id, labels?)` with no group argument at [12a-1 D.1](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:136), separate group mutation ops at [12a-1 D.2](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:239), and admin-gated registration at [12a-1 D.5](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:322). This is a real interface mismatch, not just wording drift. The plan needs one coherent bootstrap sequence such as `register_worker` → `register_group`/`add_to_group`, and 3.7/3.8/5.2/5.4/5.7 should all use that same shape.

- The ideation overshoot bound is mathematically inconsistent inside the plan. Decision 4 says concurrent ideation creation yields at most `policy.max_pending + 1` tasks at [2.4](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:143), but the worked example in [3.4](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:357) produces 5 pending from a target of 3 and initial pending of 1, and the test plan codifies that same 5-task outcome at [6.3](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:779). For the stated `max(0, target - pending)` policy with two concurrent replicas, the bound is not `target + 1` in that case. The spec text and tests need one exact formula.

- The multi-instance tests still do not reliably exercise the races that §3.4 relies on. [6.3](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:767) says two orchestrators “race,” but [Risk 3](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:1010) explicitly accepts that CI may not race at all and still pass because the assertion is only end-state `≤ N`. That means the suite can pass without ever driving the duplicate-create / AlreadyExists path, which is exactly the branch the HA argument depends on for execution and evaluation dispatch. The plan should either add a testing-only synchronizer/barrier for the unit/integration layer, or narrow the conformance claim so it stops pretending those tests prove the collision path.

- The `dispatch_mode` storage entry still does not match the design it is supposed to implement. [3.2](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:264) defines a four-key object, but the migration note in [5.2](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:646) backfills `{"auto":"auto","auto":"auto",...}` and calls the SQLite/Postgres column `JSONB` for both backends. That reads like a placeholder that was not reconciled with the actual shape.

**Overall Assessment**

The baseline/context problems are fixed; the plan is now reviewable. I stopped at feasibility because the manual-evaluation path, the worker/group bootstrap API, and the HA test story are still inconsistent in ways that would send implementation in conflicting directions. Once those are resolved, the next pass should move to alternatives and completeness rather than finding more internal contradictions.
