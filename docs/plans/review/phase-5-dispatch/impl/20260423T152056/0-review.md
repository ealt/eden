**Findings**

- **Bug** — [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:160), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:226), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:247), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:529), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:567), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:899) leak live mutable task/proposal/trial/event objects. `read_*`, `list_*`, `events()`, and `subscribe()` return shared instances, and `create_*` stores caller-owned instances directly. Because the `eden_contracts` models are mutable, any caller can mutate store state or past events in place with no store method, no lock, and no matching event append. That is a direct violation of 05 §2 / 04 §1.3 / 08 §6. I verified this locally: mutating the object returned by `read_task()` changed stored state, and mutating the object returned by `events()` rewrote the log entry.

- **Bug** — [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:645), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:667), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:702), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:741), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:780), [driver.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/driver.py:72) do not validate that a submission actually belongs to the dispatched task. `accept`/`reject` trust `submission.trial_id` instead of binding it back to `task.payload.trial_id` / `task.payload.proposal_id`; plan success is accepted without checking that the referenced proposals exist or are all non-`drafting`. An evaluate task for trial A can complete trial B; an implement task can stamp `commit_sha` onto an unrelated trial; an implement `status="error"` can fail the task and complete the proposal even if no errored trial exists, despite 03 §3.4 requiring one. Malformed success submissions should turn into `task.failed(reason="validation_error")` per 04 §4.3, but the driver only branches on `submission.status` and blindly calls `accept`/`reject`.

- **Bug** — [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:121), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:690), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:763), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:807), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:913) never enforce the required payload validity on trial writes. There is nowhere to register an experiment metrics schema, so 08 §4.1–§4.3 cannot be satisfied at all. On top of that, the store uses `model_copy(update=...)`, which bypasses Pydantic validation, so invalid `commit_sha`, invalid `artifacts_uri`, and arbitrary non-JSON / wrong-typed metrics values are persisted. I verified the store accepts `commit_sha='not-a-sha'`, `artifacts_uri='not a uri'`, and `metrics={'score': object()}` and then continues processing. That breaks 03 §3.4, 03 §4.4, and 08 §4.

- **Risk** — [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:231) is not a conforming event-log subscription surface. The method is explicitly a one-shot snapshot, with no cursor, no live delivery, and no `read_range`/`replay` equivalent matching 08 §2.1 / 05 §4.1–§4.4. That is enough for the synchronous in-process tests, but it is not the storage contract the spec pins and it will break as soon as Phase 6/8 needs a real subscriber.

- **Risk** — [workers.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/workers.py:110), [workers.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/workers.py:176), [workers.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/workers.py:242), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:529), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:567), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:611) show that 03 §1.2 is enforced by convention, not structure. Workers get the raw store and raw model objects; the store exposes public proposal/trial/integration mutators to any caller. So planners/evaluators are not structurally unable to create trials, implementers are not structurally unable to integrate, and any in-process adapter can bypass the worker negative rules. That is legal only because the scripted workers are well-behaved.

- **Risk** — [test_task_state_machine.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/tests/test_task_state_machine.py:39), [test_composite_commits.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/tests/test_composite_commits.py:63), [test_end_to_end.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/tests/test_end_to_end.py:38) materially overstate conformance. The suite passed (`36 passed`), but it only covers internally consistent happy-path submissions and state transitions. It does not test malformed-success rejection, `validation_error` emission, mismatched `trial_id`/`proposal_id`, invalid metrics/artifact/commit payloads, mutation of returned objects, or subscription/replay semantics.

**Open Questions / Assumptions**

- I did not count the current `commit_sha` write timing as a defect. The spec text is ambiguous enough that “persist on implement accept” is defensible.
- I also did not count evaluate resubmit comparison on `artifacts_uri` as a defect; 03 §4.4 includes it, while 04 §4.2 omits it.

**Assessment**

Structural problem that needs a larger change. The happy-path state machine and composite commits are there, but the store does not actually preserve the transactional invariant under normal API use, and the orchestrator/store boundary is not validating role outputs the way 04 §4.3 and 08 §4 require. This is not ready for Phase 5 sign-off without a hardening pass on the store API, validation path, and tests.
