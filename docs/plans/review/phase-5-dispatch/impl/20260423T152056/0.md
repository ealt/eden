## Change Summary

- `InMemoryStore` collapses the task store, event log, and
  proposal/trial persistence into a single in-process object
  (collapsing is allowed by `08-storage.md` §7). Every mutating
  public method stages writes in a `_Tx` and applies them via a
  single `_commit`; precondition failures raise before commit, so
  readers never observe a partial state change.
- All six composite commits enumerated in `05-event-protocol.md`
  §2.2 land atomically: implement dispatch, implement terminal,
  evaluate terminal (success/error), implement reclaim with
  in-flight trial, retry-exhausted `trial.eval_errored`, and trial
  promotion.
- The implementer's `commit_sha` lands on the trial atomically with
  the implement-accept composite commit (task.completed +
  proposal.completed), not at submit time. Rationale: the trial's
  `commit_sha` is not itself a state-machine-owned field (trial
  status stays `starting` until the evaluator writes it), and
  writing it inside the composite keeps every protocol-owned
  mutation behind the transactional invariant.
- Workers go through the store's public API only; they never touch
  the store's internal dicts. The store enforces role negative
  rules from `03-roles.md` §1.2 structurally because there is no
  back door.
- The dispatch driver (`run_experiment`) is deliberately thin: it
  shuttles work between roles by polling `list_tasks` /
  `list_proposals` / `list_trials` and finalizes submitted tasks by
  reading the recorded submission. It does not implement retry
  policy, priority ordering, or any scheduling beyond "do every
  piece of work currently possible, to quiescence."

## Notable design decisions to audit

1. **`commit_sha` write timing.** See above. Alternative: have
   `submit` for implement also write `commit_sha` on the trial
   atomically. That would match `03-roles.md` §3.2 ("set the trial's
   commit_sha to the tip of the work/* branch" as an implementer
   step) more literally. Current design interprets §3.2 as "the
   implementer provides commit_sha as an output; the orchestrator
   persists it atomically with terminal acceptance."
2. **Content equivalence for resubmit.** `_submissions_equivalent`
   compares status + role-specific canonical fields per
   `04-task-protocol.md` §4.2: plan compares `proposal_ids` as sets;
   implement compares `trial_id` + `commit_sha`; evaluate compares
   `trial_id` + `metrics` (dict equality) + `artifacts_uri`.
3. **Submission storage separate from the task object.** Submissions
   live in `InMemoryStore._submissions` keyed by task_id, not on the
   task's extra fields. This keeps the `Task` Pydantic model aligned
   with the wire schema (no result field) and makes the submission
   cleared on reclaim via a single set operation.
4. **Automatic reclaim of submitted tasks.** Per §5.1, only operator
   reclaim may move a submitted task. `reclaim(task_id, cause)`
   enforces this by rejecting `expired` / `health_policy` against
   `submitted`.
5. **Trial `commit_sha` precondition on evaluate dispatch.**
   `create_evaluate_task` requires `trial.commit_sha` is set. This
   follows from `03-roles.md` §4.1 (the orchestrator ensures
   `T.commit_sha` is set before dispatch).

## What's explicitly NOT in this chunk (by roadmap non-goals)

- No git integration — commit SHAs are fabricated by scripted
  workers; no `work/*` branches are actually written. Lands in
  Phase 7.
- No SQLite or any persistence. The store is in-memory only. Lands
  in Phase 6.
- No cross-process protocol. Lands in Phase 8.
- No retry policy, priority scheduling, or operator-facing
  observability beyond the event snapshot. Lands later.
