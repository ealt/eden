# Phase 13a — Helm base chart

**Status.** Draft.

**Predecessors.** Phase 12 plans all merged (PRs #57 / #61 / #62
/ #63 / #64). 13a assumes the v0 protocol is feature-complete: the
deployment runs multiple experiments via the control plane, has
portable checkpoints, supports termination policies, and walks the
12a-1/2/3 worker-identity / lease-driven orchestrator model. 13a
does NOT change protocol; it changes the substrate that hosts the
reference services.

**Roadmap.** [`docs/roadmap.md`](../roadmap.md) §"Phase 13 —
Kubernetes reference deployment" lists five sub-projects: base
Helm chart, executor as k8s Job, managed Postgres migration, S3/GCS
blob backend, Gitea with auth + ACLs. 13a is the **first** chunk
— the base chart that the other four extend.

**Naming.** Pre-draft check against
[`docs/glossary.md`](../glossary.md) and AGENTS.md "Naming discipline":

- Kubernetes / Helm vocabulary (chart, release, values,
  Deployment, StatefulSet, Service, Ingress, ConfigMap, Secret,
  Job, PVC, namespace, ClusterRole, ServiceAccount) is the
  upstream vocabulary; no collisions with EDEN identifiers.
- The reference services keep their existing names (no
  service-renaming); the chart just packages them.
- The chart values file's structure mirrors the existing
  `.env.example` shape so operators familiar with the Compose
  deployment can find familiar knobs.

## 1. Context

Today the reference deployment runs as a Compose stack: nine
services (task-store-server, control-plane, orchestrator,
ideator-host, executor-host, evaluator-host, web-ui, gitea,
postgres) plus a setup-time `eden-repo-init` profile. Operators
run `setup-experiment.sh` to bootstrap, then `docker compose up`.
The Compose model is the canonical reference deployment for v0
through Phase 12.

13a converts that into a Helm chart suitable for any Kubernetes
distribution (kind, k3s, EKS, GKE, AKS). Goals:

- Operators run `helm install eden ./chart -f values.yaml` and
  get the same nine services running on Kubernetes.
- The `values.yaml` shape mirrors `.env.example` so existing
  deployment configs port over predictably.
- The chart is **Kubernetes-vanilla** in v0 — no operator-only
  features (CRDs, mutating webhooks, etc.), no cloud-provider
  assumptions (Ingress class, StorageClass, etc.) baked in.
  Cloud-specific extensions are 13b / 13c / 13d / 13e's jobs.
- Existing CI smoke jobs (`compose-smoke`, etc.) keep running
  unchanged. A new `helm-smoke` CI job spins up a kind cluster,
  installs the chart, and runs the same end-to-end fixture.

13a explicitly does NOT cover:

- GPU node selection for the executor (13b).
- Managed cloud Postgres (13c) — the 13a chart bundles its own
  Postgres StatefulSet for v0.
- S3/GCS blob backend (13d) — 13a uses PVC-backed `file://`
  artifacts.
- Gitea auth + per-branch ACLs (13e) — 13a runs Gitea
  unauthenticated within the cluster, same as Compose.

## 2. Decisions

These are the load-bearing design calls; §3 unpacks each.

1. **One Helm chart, not an umbrella.** The chart at
   `reference/helm/eden/` packages all nine services as a
   single chart with one `values.yaml`. Subcharts add
   complexity (each subchart needs its own values, depends-on
   wiring, version pinning) for no v0 benefit — every service
   ships in lockstep with the chart's version. A future
   amendment MAY split out specific services as subcharts if
   they need independent versioning.

2. **Vanilla Kubernetes only — no cloud-specific assumptions.**
   The chart works on any conforming Kubernetes cluster (1.27+):
   - **Storage**: uses `PersistentVolumeClaim` with the
     cluster's default `StorageClass`. Chart values let
     operators pin a specific class.
   - **Networking**: services are `ClusterIP`. The Web UI and
     Gitea expose `Ingress` resources gated on the `ingress`
     toggle in values; off by default. Operators pick their
     own ingress controller.
   - **Image registry**: defaults to the same
     `eden-reference:dev` tag the Compose stack uses. Operators
     point at their registry via `image.repository` /
     `image.tag` values.

3. **Stateful services use StatefulSet; stateless use
   Deployment.** Postgres, Gitea, and the artifacts/repo
   volumes need stable identities + persistent volumes. The
   stateless services (task-store-server, control-plane,
   orchestrator replicas, worker hosts, web-ui) use
   Deployments.

4. **The Compose `setup-experiment.sh` workflow stays for
   Compose; a parallel `setup-experiment-helm.sh` script
   bootstraps the Helm deployment.** The Helm workflow:
   1. Operator writes / generates `values.yaml`.
   2. `helm install eden ./chart -f values.yaml` brings up
      Postgres, Gitea, task-store-server, control-plane.
   3. The operator runs `setup-experiment-helm.sh
      <experiment-config>.yaml` which: creates the experiment
      via the control plane (12c register_experiment); seeds
      the bare repo via a one-shot Job (mirrors the existing
      `eden-repo-init` Compose service); bumps the orchestrator
      Deployment to start the loop.

5. **Secret management is Kubernetes-native.** All deployment
   secrets (`EDEN_ADMIN_TOKEN`, postgres password, gitea admin
   credentials) ship as Kubernetes `Secret` resources. The
   chart accepts secret values from `values.yaml` (via Helm's
   `--values` / `--set`) for development; for production, the
   values file references existing Secrets via `existingSecret`
   knobs (so operators integrate with Sealed Secrets, External
   Secrets Operator, Vault, etc.).

6. **Replica count is operator-configurable per service.** The
   12c control plane supports multi-replica orchestrators per
   the lease model. The chart exposes `replicaCount` for each
   stateless service. Default for orchestrator is 2 (HA); for
   worker hosts it's 1 (operator scales as load demands).

7. **Image build is out of scope.** The chart pulls
   `eden-reference:dev` from a registry. Building + pushing the
   image is a separate operator step (or a separate CI
   pipeline). 13a doesn't add a `Dockerfile.helm` or anything;
   the existing `reference/compose/Dockerfile` stays the
   single image build, and operators tag/push it for their
   cluster's registry. A future amendment MAY add a
   `chart/.helmignore` + `helm package` workflow alongside the
   existing image build.

8. **Bare-repo persistence stays inside the cluster (PVC) in
   13a.** Compose uses a Docker named volume; Helm uses a PVC.
   Cross-cluster portability of the bare repo (e.g., backing
   it with a cloud storage backend) lands in 13d / 13e.

`eden-reference:dev`-equivalent image tag.

### 3.2 `values.yaml` structure

The values file mirrors the Compose `.env.example` shape so
operators porting from Compose can find their knobs in
predictable places. Sketch:

```yaml
# Image
image:
  repository: ghcr.io/your-org/eden-reference
  tag: 0.x.0
  pullPolicy: IfNotPresent
  pullSecrets: []

# Per-service replica counts
replicas:
  taskStoreServer: 1     # singleton in v0; clustered in 13c
  controlPlane: 1        # singleton SPOF per 12c §3.9
  orchestrator: 2        # HA per 12c lease model
  ideatorHost: 1
  executorHost: 1
  evaluatorHost: 1
  webUi: 1

# Secrets (DEV: inline; PROD: existingSecret)
secrets:
  adminToken: ""                       # required
  postgresPassword: ""                 # required
  giteaAdminPassword: ""               # required
  webUiSessionSecret: ""               # required
  existingSecret: ""                   # if set, all above ignored

# Storage
storage:
  className: ""                        # cluster default if empty
  postgresSize: 10Gi
  giteaSize: 20Gi
  bareRepoSize: 5Gi
  artifactsSize: 50Gi

# Networking
ingress:
  enabled: false
  className: ""                        # cluster default
  hosts:
    webUi: eden.example.com
    gitea: gitea.example.com
  tls: []                              # operator supplies cert refs

# Compose-mirror env knobs (per 12a-1/2/3 + 12b + 12c)
config:
  leaseDurationSeconds: 30             # 12c
  stateSyncIntervalSeconds: 30         # 12c §3.4a
  stateSyncFailureThreshold: 10        # 12c §3.4a
  ideationPolicyTargetPending: 3       # 12a-2
  ideationPolicyMaxTotal: ""           # unbounded
  terminationPolicy: "eden_orchestrator.policies:never_terminate"  # 12a-3
  maxQuiescentIterations: 30           # PR #47
  pollIntervalSeconds: 1.0
  experimentDirHost: ""                # for subprocess workers
  execMode: "host"                     # or "docker"
```

The `values.schema.json` enforces shape: required fields
(`secrets.adminToken` etc. unless `existingSecret` is set);
type checks (replica counts are positive ints; storage sizes
match `[0-9]+(Gi|Mi)`); enum values (`execMode ∈ {host,
docker}`).

### 3.3 Secrets handling

The chart's secret strategy:

**Development.** Operator sets `secrets.adminToken=...` etc.
inline in `values.yaml`. The chart materializes a single
`Secret` resource named `<release>-eden-secrets` containing all
of them. Pros: works out-of-the-box. Cons: secrets in plaintext
on disk (wherever the values file lives).

**Production.** Operator pre-creates the Secret externally
(via Sealed Secrets, External Secrets Operator, Vault, etc.)
and passes `secrets.existingSecret=my-secret` in values. The
chart's templates reference the existing secret name in
`envFrom: secretRef:` blocks; no secret values flow through
the chart itself.

The `_helpers.tpl` exposes `eden.secretName` which returns
either the chart-managed name or `existingSecret` based on the
values setting. All templates use this helper.

### 3.4 Per-service Deployments / StatefulSets

Each service gets a template file. Common shape:

- `apiVersion: apps/v1`
- `kind: Deployment` (stateless) or `StatefulSet` (stateful)
- `replicas: {{ .Values.replicas.<service> }}`
- `selector` + `template` with consistent labels per
  `_helpers.tpl`'s `eden.labels`.
- `containers`: one container per pod; image from
  `.Values.image`; envFrom: secretRef: `eden.secretName`.
- `volumeMounts` + `volumes` for stateful services'
  PVCs.

**Service-specific knobs** carried via `args:` lists, mirroring
the Compose `command:` shape. Example for orchestrator:

```yaml
args:
  - "--worker-id"
  - "$(POD_NAME)"                    # k8s downward API
  - "--control-plane-url"
  - "http://{{ include "eden.fullname" . }}-control-plane:8080"
  - "--lease-duration-seconds"
  - "{{ .Values.config.leaseDurationSeconds }}"
  - "--task-store-url"
  - "http://{{ include "eden.fullname" . }}-task-store-server:8080"
  - "--ideation-policy"
  - "$(EDEN_IDEATION_POLICY)"
  - "--termination-policy"
  - "{{ .Values.config.terminationPolicy }}"
  - "--max-quiescent-iterations"
  - "{{ .Values.config.maxQuiescentIterations }}"
env:
  - name: POD_NAME
    valueFrom:
      fieldRef:
        fieldPath: metadata.name
```

Using the pod name as `--worker-id` automatically gives each
orchestrator replica a unique worker_id (per 12a-1 §D.1's
grammar). The pod name is `<release>-eden-orchestrator-0`,
`-1`, etc., which matches the `^[a-z0-9][a-z0-9_-]{0,63}$`
grammar.

**Worker host services** (ideator, executor, evaluator) follow
the same shape but the args differ per service (each has its
own `--mode subprocess` flag and binds the experiment-specific
config dir as a hostPath/PVC mount). The worker hosts'
`--worker-id` similarly defaults to `$(POD_NAME)`.

**Stateful services** (Postgres, Gitea):

- Postgres: `StatefulSet` with PVC template; image is the
  upstream `postgres:16.6-alpine` (matches Compose's choice).
  `replicas: 1` in v0 — clustering / managed Postgres is 13c.
- Gitea: `StatefulSet` with PVC for its data dir; image is
  the upstream Gitea image. `replicas: 1`. Auth + ACLs are
  13e.

### 3.5 Setup workflow

`reference/scripts/setup-experiment-helm.sh` mirrors the
existing `setup-experiment.sh` but for Helm:

```bash
# Operator workflow:
helm install eden ./reference/helm/eden \
  -f values.yaml \
  --create-namespace \
  --namespace eden-prod

# Wait for the chart to come up:
kubectl -n eden-prod rollout status deployment/eden-task-store-server
kubectl -n eden-prod rollout status deployment/eden-control-plane

# Bootstrap an experiment:
bash reference/scripts/setup-experiment-helm.sh \
  --namespace eden-prod \
  --experiment-config tests/fixtures/experiment/.eden/config.yaml \
  --experiment-id exp-1

# The script:
# 1. kubectl exec into the control-plane pod, calls
#    register_experiment.
# 2. Submits a one-shot Job (the chart's repo-init template,
#    enabled by --set repoInit.runOnce=true) that seeds the
#    bare repo. Waits for completion.
# 3. The orchestrator Deployment is already running; it
#    auto-acquires a lease for the new experiment within
#    one poll-interval.
# 4. Prints the seed SHA + experiment URLs (web-ui ingress
#    if enabled, else port-forward instructions).
```

The `--set repoInit.runOnce=true` mechanism: the
`repo-init-job.yaml` template is conditional on
`.Values.repoInit.enabled`. Setting it to `true` materializes
a `kind: Job` that runs once and exits; subsequent
`helm upgrade` runs are idempotent (same Job name; if the
prior one succeeded, k8s skips).

### 3.6 Helm smoke CI job

A new `helm-smoke` CI job mirrors `compose-smoke`:

- Spins up a kind cluster.
- Installs the chart with `helm install eden
  reference/helm/eden -f reference/helm/eden/ci-values.yaml
  --wait`.
- Runs `setup-experiment-helm.sh` with the fixture experiment.
- Waits for the orchestrator to drive 3 ideation tasks → 3
  variants → 3 integrations.
- Asserts (via `kubectl exec` into the task-store-server pod
  and querying the wire endpoint): ≥3 `variant.integrated`
  events, ≥9 `task.completed` events, ≥3 ideate-task
  `task.completed` events. Same shape as `compose-smoke`.
- Tears down with `helm uninstall eden && kind delete cluster`.

`ci-values.yaml` pins specific values for CI (small storage
sizes, dev secrets inline, no ingress).

The kind cluster's setup uses
`actions/setup-kind@v1` (or equivalent); install Helm via
`azure/setup-helm@v3`. Both are standard GitHub Actions
patterns.

### 3.7 Conformance: helm-smoke is the same end-state assertion

The Helm deployment passes the same chapter-9 conformance
contracts as the Compose deployment — the protocol is
unchanged. No new conformance scenarios are added in 13a; the
existing v1, v1+roles, v1+roles+integrator, v1+checkpoints,
v1+multi-experiment levels (post-12c) all apply.

The new `helm-smoke` smoke is the substrate-level assertion
that the protocol works on Kubernetes; conformance scenarios
(which run against the wire, not against a substrate) cover
the protocol assertions.

### 3.8 What 13a does NOT do

- **Cloud-provider helm extensions.** The chart doesn't ship
  AWS / GCP / Azure value files. Operators target their
  cloud's specifics (IRSA, GKE Workload Identity, ingress
  classes) via standard Helm value overrides.
- **Operator pattern.** No CRDs, no controllers. v0 stays
  Helm-vanilla.
- **Multi-tenancy at the cluster level.** A single chart
  release is one deployment. Multiple releases per cluster
  (different namespaces) are supported but each is independent
  — there's no shared control plane across releases.
- **Helm test resources beyond a connection check.** The
  `tests/test-connection.yaml` is a minimal `helm test`
  resource that verifies each service's HTTP endpoint
  responds. Full smoke testing happens in CI via the
  `helm-smoke` job, not via `helm test`.
- **Image build pipeline.** The chart pulls the existing
  `eden-reference:dev` image; building + pushing is the
  operator's separate concern (or a separate CI pipeline).

- **GPU node selection for the executor** — 13b. Adds
  `nodeSelector`, `tolerations`, `resources.limits.nvidia.com/gpu`

- **Managed Postgres migration** — 13c. The chart's
  `postgres-statefulset.yaml` becomes optional (gated on

  value lets operators point at RDS / Cloud SQL.
- **S3/GCS blob backend** — 13d. The artifacts PVC becomes

  routes the artifact-store URI.

  The chart adds Gitea's auth knobs + creates per-experiment
  ACL rules.

### 4.3 Out of scope

- **Operator chart packaging** (Helm OCI registry push). The
  chart lives in the repo as source; operators clone or
  vendor it.
- **Helm hook ordering complexity.** v0 doesn't use
  pre-install / post-install hooks for ordering; relies on
  k8s's standard Service / Pod readiness.
- **Backup + restore at the chart level.** Operators handle
  PVC snapshots / Postgres dumps via their own infrastructure
  (or 12b portable checkpoints, which work the same in Helm
  as in Compose).

### 4.4 Non-goals

- Backwards compat with the Compose stack at the data level.
  The two are independent deployments; an operator can't
  "upgrade" a Compose stack to Helm (or vice versa) without
  exporting via 12b portable checkpoints and re-importing.
- Distroless / minimal base images. The existing
  `eden-reference:dev` image stays; minimization is a separate
  optimization.
- Strict resource requests/limits. The chart sets reasonable
  defaults via `resources` blocks but doesn't pin to specific
  values; operators tune for their cluster.

## 6. Test design

### 6.1 Unit-equivalent tests (`helm template` + `helm lint`)

Helm doesn't have a unit-test framework in the traditional
sense; the closest is `helm template` (renders templates to
YAML) + `helm lint` (validates chart structure).

- `helm lint reference/helm/eden`: pass.
- `helm template eden reference/helm/eden -f
  reference/helm/eden/ci-values.yaml`: produces valid YAML
  parsed by `kubectl apply --dry-run=client`.
- `helm template ... | yamllint`: pass (consistent
  formatting).

These run in a new lightweight CI job `helm-lint` that's fast
(seconds) and runs on every push. The full `helm-smoke`

### 6.2 `helm-smoke` integration test

Per §3.6:

- Spin up kind via `actions/setup-kind`.
- `helm install eden reference/helm/eden -f
  reference/helm/eden/ci-values.yaml --create-namespace
  --namespace eden-test --wait --timeout 5m`.

  --namespace eden-test

  --experiment-id exp-1`.

- Install the chart at `0.1.0`.

- `helm upgrade eden ./reference/helm/eden ...` (no version
  bump but exercises the upgrade machinery).
- Assert the experiment is unaffected (still has its tasks /
  variants; lease holders unchanged within
  `lease_duration`).

This catches PVC reclaim policy mistakes (data loss on

## 7. Tricky areas

### 7.1 PVC retention vs reclaim policy

The default `StorageClass` in many clusters has
`reclaimPolicy: Delete`, so when a PVC is deleted the
underlying storage is reclaimed. For Postgres + Gitea + the
bare-repo + artifacts volumes, that means a careless `helm
uninstall` deletes ALL experiment data.

The chart's PVCs use `volumeClaimTemplates` (StatefulSet) which
don't auto-delete on `helm uninstall` — k8s leaves the PVCs
behind for manual cleanup. Operators who want to fully
tear down run `kubectl delete pvc -n eden-test --all` after
`helm uninstall`. Documented in the README.

For the CI smokes, this is fine — `kind delete cluster`
removes everything regardless.

### 7.2 Image pull secrets in private registries

The `image.pullSecrets` value lets operators reference a
pre-created `imagePullSecret` for private registries. Default
empty (assumes public registry). Documented in the README
with a worked example for ECR / GCR / GHCR.

### 7.3 `replicaCount: 0` to disable a service

Operators may want to run only some services (e.g., disable
the orchestrator for a manual-UI-only deployment). Setting
`.Values.replicas.orchestrator = 0` produces a Deployment with
0 replicas — the service is "deployed but idle". This is
preferred over conditional template inclusion (which makes
upgrade behavior fragile).

The same pattern applies to all stateless services. Stateful
services (Postgres, Gitea) cannot reasonably scale to 0 in v0
— if you don't want them, edit the chart.

### 7.4 Pod restarts vs lease handoff

A Helm-driven rolling restart of the orchestrator Deployment
(e.g., from `helm upgrade --recreate-pods`) terminates pods
one at a time. Each terminating pod's leases expire (per 12c
§3.2); during the brief window before re-acquisition by the
new pod, the experiment is un-leased and no dispatch happens.

For typical deployments this is fine — the window is
sub-`lease_duration` (30s default). For deployments where
even 30s of un-dispatched is unacceptable, the chart's default
`replicas.orchestrator = 2` ensures another replica is
available to acquire the lease during the rolling restart.

### 7.5 Helm + Kubernetes version constraints

The chart targets Kubernetes 1.27+ (matches what most clouds
support as of 2026). It uses:

- `apps/v1` Deployments and StatefulSets (stable since 1.9).
- `networking.k8s.io/v1` Ingress (stable since 1.19).
- `helm test` framework (Helm 3.0+).
- `values.schema.json` (Helm 3.x).

Older clusters fail at install time with a clear error.
Documented in the README.

### 7.6 `setup-experiment-helm.sh` vs control-plane availability

The script calls the control plane to register the experiment.
If the control plane isn't ready yet (the chart just installed,
control-plane pod is still starting), the call fails. The
script polls the control-plane's `/healthz` for up to 60s
before attempting registration. Documented in the script's
help text.

## 8. Risks

1. **Default StorageClass missing.** A cluster without a
   default `StorageClass` rejects PVC creation; install
   fails. Mitigation: README requires the operator confirms
   storage provisioning before installing. The
   `values.schema.json` could enforce that
   `storage.className` is non-empty if no default is
   configured, but the chart doesn't have visibility into
   the cluster's default — operator responsibility.

2. **kind cluster resource limits in CI.** kind in CI runs in
   a single Docker container with limited CPU/memory. The
   chart's nine services may push the limits — Postgres alone
   is hungry. Mitigation: `ci-values.yaml` sets minimal
   storage sizes + relaxed `resources.requests` on
   non-critical services. If `helm-smoke` is flaky in CI, the
   operator-side tuning lives in `ci-values.yaml` rather than
   the production-default `values.yaml`.

3. **Image build vs chart version drift.** The chart's
   `appVersion` and the actual `eden-reference` image tag can
   drift if the operator builds a new image but doesn't bump
   the chart. Mitigation: a future amendment (13b/c/d/e or
   later) wires the chart's `appVersion` to the project's
   semver via a build-time substitution; v0 documents the
   drift risk.

4. **Re-introducing legacy vocab in chart docs.** PR #60's
   guardrail catches the patterns. The chart's README is a
   new doc; the values file mirrors `.env.example` shapes
   that are already known clean. Pre-submit
   `python3 scripts/check-rename-discipline.py` clean.

5. **Helm's templating quirks.** The chart's `_helpers.tpl`
   uses Sprig functions for string manipulation. A typo in a
   helper produces a chart that lints clean but renders
   broken YAML. Mitigation: `helm template` + `kubectl apply
   --dry-run=client` pre-merge catches this.

## 9. Sequencing

Recommended PR shape (in order):

1. **Chart skeleton PR.** `Chart.yaml`, `values.yaml`,
   `values.schema.json`, `_helpers.tpl`, `secret.yaml`,
   `README.md`. Lints clean, but contains no service
   templates yet. Reviewable as a structural foundation.

2. **Stateful services PR.** Postgres + Gitea StatefulSets +
   their Services. Operators can `helm install` and have the
   stores up.

3. **Stateless control-plane services PR.**
   task-store-server + control-plane Deployments + Services.
   The protocol-level coordination layer is up.

4. **Orchestrator + worker hosts PR.** orchestrator,
   ideator-host, executor-host, evaluator-host Deployments.
   Multi-replica orchestrator + worker fleet.

5. **Web UI + ingress PR.** web-ui Deployment + Service +
   optional Ingress. Full operator surface.

6. **Repo-init Job + setup script PR.** `repo-init-job.yaml`
   plus `setup-experiment-helm.sh`. End-to-end operator
   workflow.

7. **CI PR.** `helm-lint`, `helm-smoke`, `helm-upgrade-smoke`
   jobs. Validation in CI.

8. **Docs PR.** `docs/deployment/helm.md` + AGENTS.md
   commands + roadmap delta.

A reviewer going from PR 1 to PR 8 should expect the
`helm-smoke` job to go red around PR 4 (when the orchestrator
is added but worker hosts aren't all green yet) and come back
green at PR 6.
