**Feasibility Findings**
- Critical: Decision 9’s `EDEN_MAX_QUIESCENT_ITERATIONS=0` strategy is not compatible with the current orchestrator implementation. The plan treats `0` as an existing “run forever” escape hatch ([docs/plans/eden-phase-13a-helm-base-chart.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-13a-helm-base-chart.md:83), [docs/plans/eden-phase-13a-helm-base-chart.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-13a-helm-base-chart.md:239), [docs/plans/eden-phase-13a-helm-base-chart.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-13a-helm-base-chart.md:375), [docs/plans/eden-phase-13a-helm-base-chart.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-13a-helm-base-chart.md:704)), but the current CLI rejects any value `< 2` ([reference/services/orchestrator/src/eden_orchestrator/cli.py](/Users/ericalt/Documents/eden/reference/services/orchestrator/src/eden_orchestrator/cli.py:25), [reference/services/orchestrator/src/eden_orchestrator/cli.py](/Users/ericalt/Documents/eden/reference/services/orchestrator/src/eden_orchestrator/cli.py:33), [reference/services/orchestrator/src/eden_orchestrator/cli.py](/Users/ericalt/Documents/eden/reference/services/orchestrator/src/eden_orchestrator/cli.py:124)). - High: the repo-init bootstrap model still does not work for a long-lived multi-experiment Helm release. The current `repo_init` tool is idempotent only for “this repo path is already seeded” ([reference/services/_common/src/eden_service_common/repo_init.py](/Users/ericalt/Documents/eden/reference/services/_common/src/eden_service_common/repo_init.py:97)), not for “a Helm release can keep reusing one Job manifest to initialize arbitrary future experiments.” The `kubectl exec into the control-plane pod` step ([docs/plans/eden-phase-13a-helm-base-chart.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-13a-helm-base-chart.md:530)) is another sign the bootstrap path is still not modeled as a clean operator-side API call. This wants an ad hoc Job/Pod or direct API client path, not a chart-managed singleton Job.

- High: the plan still claims “Kubernetes-vanilla” while carrying subprocess-worker assumptions that are node-local and Docker-specific. The values surface still includes `experimentDirHost` and `execMode` ([docs/plans/eden-phase-13a-helm-base-chart.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-13a-helm-base-chart.md:377)), and §3.4 still says subprocess workers bind the experiment directory as a `hostPath/PVC` mount ([docs/plans/eden-phase-13a-helm-base-chart.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-13a-helm-base-chart.md:492)). And the reference `--exec-mode docker` path explicitly depends on mounting `/var/run/docker.sock` into the worker host ([reference/services/_common/src/eden_service_common/cli.py](/Users/ericalt/Documents/eden/reference/services/_common/src/eden_service_common/cli.py:90), [spec/v0/reference-bindings/worker-host-subprocess.md](/Users/ericalt/Documents/eden/spec/v0/reference-bindings/worker-host-subprocess.md:355), [spec/v0/reference-bindings/worker-host-subprocess.md](/Users/ericalt/Documents/eden/spec/v0/reference-bindings/worker-host-subprocess.md:444)). That is not a vanilla-cluster substrate story, especially on containerd-based clusters. **Overall Assessment**

The missing-context issues are fixed; the review now stops at `2. Feasibility`. The biggest blocker is Decision 9: as written, the chart depends on orchestrator behavior that the current code does not implement. After that, the bootstrap workflow and the subprocess-runtime story both still need redesign before this reads like a chart that will actually work on a vanilla Kubernetes cluster.
