**Findings**
- High: the plan does not establish the current git-storage topology, and several later sections imply the wrong one. The current reference deployment is not “one bare repo volume plus Gitea”; it is Gitea as the central remote plus per-service local clones for the orchestrator, executor, evaluator, and web UI ([reference/compose/compose.yaml](/Users/ericalt/Documents/eden/reference/compose/compose.yaml:194), [reference/compose/compose.yaml](/Users/ericalt/Documents/eden/reference/compose/compose.yaml:268), [reference/compose/compose.yaml](/Users/ericalt/Documents/eden/reference/compose/compose.yaml:336), [reference/compose/compose.yaml](/Users/ericalt/Documents/eden/reference/compose/compose.yaml:380)). Because that context is missing up front, the plan’s singular `bareRepoSize` / “bare-repo persistence” framing reads as if 13a only needs one repo PVC ([docs/plans/eden-phase-13a-helm-base-chart.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-13a-helm-base-chart.md:243), [docs/plans/eden-phase-13a-helm-base-chart.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-13a-helm-base-chart.md:163), [docs/plans/eden-phase-13a-helm-base-chart.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-13a-helm-base-chart.md:470)). An outside reviewer new to the chart needs that substrate baseline before they can evaluate PVC shape, StatefulSet vs Deployment, or repo-init semantics.

- High: the Gitea auth baseline is misstated. The plan says 13a runs Gitea “unauthenticated within the cluster, same as Compose” ([docs/plans/eden-phase-13a-helm-base-chart.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-13a-helm-base-chart.md:64)), but current Compose is already using per-experiment HTTP Basic auth plus credential-helper wiring: the environment carries `GITEA_REMOTE_PASSWORD` / `GITEA_REMOTE_URL` ([reference/compose/.env.example](/Users/ericalt/Documents/eden/reference/compose/.env.example:25)), `setup-experiment.sh` preserves and provisions that password ([reference/scripts/setup-experiment/setup-experiment.sh](/Users/ericalt/Documents/eden/reference/scripts/setup-experiment/setup-experiment.sh:176)), and the orchestrator / executor / web UI all mount and pass `--credential-helper` ([reference/compose/compose.yaml](/Users/ericalt/Documents/eden/reference/compose/compose.yaml:196), [reference/compose/compose.yaml](/Users/ericalt/Documents/eden/reference/compose/compose.yaml:270), [reference/compose/compose.yaml](/Users/ericalt/Documents/eden/reference/compose/compose.yaml:338)). That is missing context for the Secrets section: as written, the plan presents parity with Compose where it is actually changing the auth posture.

- Medium: the plan omits one piece of current deployment context that materially shapes the Kubernetes design review: the orchestrator is currently a quiescence-exiting process, and Compose deliberately uses `restart: "on-failure"` so a clean `0` exit sticks ([reference/compose/compose.yaml](/Users/ericalt/Documents/eden/reference/compose/compose.yaml:176), [reference/services/orchestrator/src/eden_orchestrator/loop.py](/Users/ericalt/Documents/eden/reference/services/orchestrator/src/eden_orchestrator/loop.py:67)). Section 1 describes “the same nine services running on Kubernetes” ([docs/plans/eden-phase-13a-helm-base-chart.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-13a-helm-base-chart.md:42)) without surfacing that lifecycle fact. An outside reviewer needs that baseline before they can reason about workload kind, rollout behavior, and the 12c lease model.

**Overall Assessment**

I stopped at `1. Missing context`, because there is substantive feedback there. The plan is not ready for a feasibility review yet: it needs to re-anchor itself to the actual current substrate baseline first, especially the per-service git-clone topology, the existing Gitea credential model, and the orchestrator’s quiescent lifecycle. Once that context is corrected, the feasibility pass will be much sharper.
