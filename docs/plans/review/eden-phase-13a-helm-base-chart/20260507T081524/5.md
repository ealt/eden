# Phase 13a — Helm base chart

**Status.** Draft.

**Predecessors.** Phase 12 plans all merged (PRs #57 / #61 / #62
/ #63 / #64). 13a assumes the v0 protocol is feature-complete: the
deployment runs multiple experiments via the control plane, has
portable checkpoints, supports termination policies, and walks the
12a-1/2/3 worker-identity / lease-driven orchestrator model. 13a
does NOT change protocol; it changes the substrate that hosts the
reference services.

**Roadmap.** [`docs/roadmap.md`](../roadmap.md) §"Phase 13 —
Kubernetes reference deployment" lists five sub-projects: base
Helm chart, executor as k8s Job, managed Postgres migration, S3/GCS
blob backend, Gitea with auth + ACLs. 13a is the **first** chunk
— the base chart that the other four extend.

**Naming.** Pre-draft check against
[`docs/glossary.md`](../glossary.md) and AGENTS.md "Naming discipline":

- Kubernetes / Helm vocabulary (chart, release, values,
  Deployment, StatefulSet, Service, Ingress, ConfigMap, Secret,
  Job, PVC, namespace, ClusterRole, ServiceAccount) is the
  upstream vocabulary; no collisions with EDEN identifiers.
- The reference services keep their existing names (no
  service-renaming); the chart just packages them.
- The chart values file's structure mirrors the existing
  `.env.example` shape so operators familiar with the Compose
  deployment can find familiar knobs.

## 1. Context

Today the reference deployment runs as a Compose stack: nine
services (task-store-server, control-plane, orchestrator,
ideator-host, executor-host, evaluator-host, web-ui, gitea,
postgres) plus a setup-time `eden-repo-init` profile. Operators
run `setup-experiment.sh` to bootstrap, then `docker compose up`.
The Compose model is the canonical reference deployment for v0
through Phase 12.

### 1.0 Substrate baseline (current Compose deployment)

Several Compose-side details shape what 13a must replicate:

- **Git-storage topology — Gitea-as-remote + per-service local
  clones.** Phase 10d follow-up B moved the deployment off a
  single shared `eden-bare-repo` named volume. Today, Gitea is
  the central git remote (running as a Compose service); the
  orchestrator, executor-host, evaluator-host, and web-ui each
  hold their OWN local clone of the experiment's bare repo,
  populated via `git clone` from Gitea at first start and
  refreshed via `git fetch` on each ref-touching op
  ([`reference/compose/compose.yaml`](../../reference/compose/compose.yaml)
  lines 194, 268, 336, 380; per-service named volumes
  `eden-orchestrator-repo`, `eden-executor-repo`,
  `eden-evaluator-repo`, `eden-web-ui-repo`). The Helm chart
  needs **four** independent PVCs, not one.
- **Gitea auth — HTTP Basic with credential helper.** The
  Compose stack uses per-experiment HTTP Basic credentials
  generated by `setup-experiment.sh` and stored in
  `GITEA_REMOTE_PASSWORD` plus a per-experiment
  `credential-helper` script directory mounted into each
  consuming service
  ([`reference/compose/.env.example`](../../reference/compose/.env.example)
  line 25;
  [`reference/scripts/setup-experiment/setup-experiment.sh`](../../reference/scripts/setup-experiment/setup-experiment.sh)
  line 176). 13a mirrors that exact auth posture (HTTP Basic
  plus credential helper); it does NOT change auth. 13e adds
  per-branch ACLs + native PR review on top.
- **Orchestrator lifecycle — quiescence-exit with
  `restart: on-failure`.** The orchestrator process exits 0
  when its quiescence threshold trips (12a-2 default 30
  iterations); Compose's `restart: "on-failure"` policy
  honors exit-0 by NOT restarting the container
  ([`reference/compose/compose.yaml`](../../reference/compose/compose.yaml)
  line 176;
  [`reference/services/orchestrator/src/eden_orchestrator/loop.py`](../../reference/services/orchestrator/src/eden_orchestrator/loop.py)
  line 67). Kubernetes Deployments do NOT have a
  `restart: on-failure` equivalent — `restartPolicy: Always`
  is the only valid value for a Deployment's pod template,
  and it restarts on ANY exit including exit 0. 13a addresses
  this with the `EDEN_MAX_QUIESCENT_ITERATIONS=0` env-var
  pattern from PR #47 (orchestrator never exits on
  quiescence; runs forever until SIGTERM). See §3.4.

### 1.1 What 13a converts to Kubernetes

13a converts the Compose stack into a Helm chart suitable for
any Kubernetes distribution (kind, k3s, EKS, GKE, AKS). Goals:

- Operators run `helm install eden ./chart -f values.yaml` and
  get the same nine services running on Kubernetes.
- The `values.yaml` shape mirrors `.env.example` so existing
  deployment configs port over predictably.
- The chart is **Kubernetes-vanilla** in v0 — no operator-only
  features (CRDs, mutating webhooks, etc.), no cloud-provider
  assumptions (Ingress class, StorageClass, etc.) baked in.
  Cloud-specific extensions are 13b / 13c / 13d / 13e's jobs.
- Existing CI smoke jobs (`compose-smoke`, etc.) keep running
  unchanged. A new `helm-smoke` CI job spins up a kind cluster,
  installs the chart, and runs the same end-to-end fixture.

13a explicitly does NOT cover:

- GPU node selection for the executor (13b).
- Managed cloud Postgres (13c) — the 13a chart bundles its own
  Postgres StatefulSet for v0.
- S3/GCS blob backend (13d) — 13a uses PVC-backed `file://`
  artifacts.
- **Gitea auth changes** — 13a mirrors Compose's existing
  HTTP Basic auth + credential-helper posture verbatim. 13e
  adds per-branch ACLs and native PR review on top of that
  baseline.

## 2. Decisions

These are the load-bearing design calls; §3 unpacks each.

1. **One Helm chart, not an umbrella.** The chart at
   `reference/helm/eden/` packages all nine services as a
   single chart with one `values.yaml`. Subcharts add
   complexity (each subchart needs its own values, depends-on
   wiring, version pinning) for no v0 benefit — every service
   ships in lockstep with the chart's version. A future
   amendment MAY split out specific services as subcharts if
   they need independent versioning.

2. **Vanilla Kubernetes only — no cloud-specific assumptions.**
   The chart works on any conforming Kubernetes cluster (1.27+):
   - **Storage**: uses `PersistentVolumeClaim` with the
     cluster's default `StorageClass`. Chart values let
     operators pin a specific class.
   - **Networking**: services are `ClusterIP`. The Web UI and
     Gitea expose `Ingress` resources gated on the `ingress`
     toggle in values; off by default. Operators pick their
     own ingress controller.
   - **Image registry**: NO default works on a fresh cluster.
     The Compose stack builds `eden-reference:dev` locally
     ([`reference/compose/Dockerfile`](../../reference/compose/Dockerfile));
     a Kubernetes cluster cannot pull that local image.
     Operators MUST build + push the image to a registry
     they control and supply `image.repository` +
     `image.tag` in `values.yaml`. The chart's
     `values.schema.json` enforces both as non-empty.

3. **Stateful services use StatefulSet; truly-stateless
   services use Deployment.** The current substrate baseline
   (§1.0) shows that the orchestrator, executor-host,
   evaluator-host, and web-ui each hold a per-service local
   clone of the experiment's bare repo. With multiple replicas
   of any of these, each replica needs its OWN clone (sharing
   would require ReadWriteMany PVCs, which aren't universally
   available; concurrent writes from multiple git clients to
   the same `.git` dir would race anyway). So:
   - **StatefulSet**: orchestrator, executor-host,
     evaluator-host, web-ui (each holds local-clone PVC),
     plus Postgres and Gitea.
   - **Deployment**: task-store-server, control-plane,
     ideator-host (no per-replica state — ideator does NOT
     hold a git clone since it doesn't touch refs).
   StatefulSet's `volumeClaimTemplates` mints one PVC per
   replica automatically, sized per the `storage.<service>RepoSize`
   values. StatefulSet also gives stable pod names
   (`eden-orchestrator-0`, `-1`, etc.) which the chart uses
   directly as `--worker-id` (the §3.4 POD_NAME pattern).
   Decision 1's "single chart" framing still holds; the
   workload-kind split is per-service.

4. **The Compose `setup-experiment.sh` workflow stays for
   Compose; a parallel `setup-experiment-helm.sh` script
   bootstraps the Helm deployment.** The Helm workflow:
   1. Operator writes / generates `values.yaml`.
   2. `helm install eden ./chart -f values.yaml` brings up
      Postgres, Gitea, task-store-server, control-plane.
   3. The operator runs `setup-experiment-helm.sh
      <experiment-config>.yaml` which: creates the experiment
      via the control plane (12c register_experiment); seeds
      the bare repo via a one-shot Job (mirrors the existing
      `eden-repo-init` Compose service); bumps the orchestrator
      Deployment to start the loop.

5. **Secret management is Kubernetes-native.** All deployment
   secrets (`EDEN_ADMIN_TOKEN`, postgres password, gitea admin
   credentials) ship as Kubernetes `Secret` resources. The
   chart accepts secret values from `values.yaml` (via Helm's
   `--values` / `--set`) for development; for production, the
   values file references existing Secrets via `existingSecret`
   knobs (so operators integrate with Sealed Secrets, External
   Secrets Operator, Vault, etc.).

6. **Replica count is operator-configurable per service.** The
   12c control plane supports multi-replica orchestrators per
   the lease model. The chart exposes `replicaCount` for each
   stateless service. Default for orchestrator is 2 (HA); for
   worker hosts it's 1 (operator scales as load demands).

7. **Image build is out of scope; image distribution is the
   operator's responsibility.** The chart pulls a published
   image from a registry the operator controls. The Compose
   stack's local-build target (`eden-reference:dev` from
   [`reference/compose/Dockerfile`](../../reference/compose/Dockerfile))
   is NOT pullable by a Kubernetes cluster. Operators must
   build + push the image (`docker build` + `docker push`,
   or via their CI) and supply
   `image.repository=<registry>/<repo>` + `image.tag=<version>`
   in `values.yaml`. The chart fails install if either is
   empty (per `values.schema.json`'s `minLength: 1`
   constraint). The existing `reference/compose/Dockerfile`
   stays the single canonical image build; 13a doesn't add a
   `Dockerfile.helm`. A future amendment MAY add a
   `chart/.helmignore` + `helm package` workflow alongside
   a published-by-CI image, but v0 keeps the operator-builds-
   and-pushes posture.

8. **Per-service local-clone PVCs (4 of them) plus one
   gitea-side bare repo PVC.** Per §1.0 substrate baseline,
   the Compose stack already moved off a single shared bare
   repo to a Gitea-as-remote + per-service local-clone model.
   The Helm chart preserves that topology: four PVCs for the
   per-service clones (orchestrator, executor, evaluator,
   web-ui) plus one PVC for Gitea's data dir (which holds the
   authoritative bare repo). Cross-cluster portability of the
   git data (e.g., backing it with a cloud-managed git
   forge) lands in 13e; 13a uses cluster-local PVCs.

9. **Orchestrator runs forever on Kubernetes — orchestrator
   CLI/loop change required, not a chart-only switch.** Per
   §1.0 baseline, the orchestrator currently exits 0 on
   quiescence. Kubernetes Deployments support only
   `restartPolicy: Always`, which would restart the
   orchestrator on every quiescence exit — burning CPU on
   restarts for no progress.

   The "obvious" fix would be `EDEN_MAX_QUIESCENT_ITERATIONS=0`,
   but the current CLI rejects values `< 2`
   ([`reference/services/orchestrator/src/eden_orchestrator/cli.py`](../../reference/services/orchestrator/src/eden_orchestrator/cli.py)
   lines 25, 33, 124), and the loop's `quiescent >= max_quiescent_iterations`
   check would terminate on the FIRST zero-progress
   iteration if `max == 0`. So the chart cannot just set the
   env var to 0 — there's no implementation behind it.

   13a therefore BUNDLES a small orchestrator code change:
   - Accept `--max-quiescent-iterations 0` as a sentinel
     meaning "never exit on quiescence; run until SIGTERM".
   - Update the loop's check: `if max == 0: continue;
     elif quiescent >= max: return`.
   - Validation in `cli.py` accepts 0 specifically (other
     values < 2 still rejected).

   The chart sets `EDEN_MAX_QUIESCENT_ITERATIONS=0` (the §5.5
   default) and the orchestrator runs forever, ending on
   SIGTERM via Kubernetes pod termination. The Compose
   stack continues to default to non-zero (preserving
   quiescence-exit on Compose); the operator who deploys via
   Helm gets the run-forever lifecycle automatically.

10. **Subprocess + DooD worker mode is OUT OF SCOPE for 13a.**
    The `--mode subprocess` worker hosts (Phase 10d) require
    `--experiment-dir`, `--artifacts-dir`, `--worktrees-dir` as
    filesystem paths on the worker host's filesystem. The
    `--exec-mode docker` path additionally requires
    `/var/run/docker.sock` mounted into the worker host
    container — viable on Docker-runtime nodes but not
    universally on containerd / cri-o clusters
    (`spec/v0/reference-bindings/worker-host-subprocess.md`
    §8). 13a ships only the `--mode scripted` worker hosts
    (the test-fixture-style deterministic workers). User-
    supplied LLM workers via subprocess + DooD are deferred
    to a later 13 sub-chunk that defines a Kubernetes-native
    per-task packaging mechanism (likely "executor as k8s Job"
    per the roadmap entry; see 13b). v0 13a operators who
    need user-supplied LLM workers continue using the Compose
    stack until 13b lands.

### 3.2 `values.yaml` structure

The values file mirrors the Compose `.env.example` shape so
operators porting from Compose can find their knobs in
predictable places. Sketch:

```yaml
# Image
# Image — REQUIRED. The Compose stack builds eden-reference:dev as
# a local target ([reference/compose/Dockerfile](../../reference/compose/Dockerfile));
# the chart cannot pull that local image from a Kubernetes
# cluster. Operators MUST build + push the image to their own
# registry and supply repository + tag. The chart's
# values.schema.json enforces `image.repository != ""` AND
# `image.tag != ""`.
image:
  repository: ""                       # required (e.g., ghcr.io/your-org/eden-reference)
  tag: ""                              # required (e.g., 0.1.0)
  pullPolicy: IfNotPresent
  pullSecrets: []

# Per-service replica counts
replicas:
  taskStoreServer: 1     # singleton in v0; clustered in 13c
  controlPlane: 1        # singleton SPOF per 12c §3.9
  orchestrator: 2        # HA per 12c lease model
  ideatorHost: 1
  executorHost: 1
  evaluatorHost: 1
  webUi: 1

# Secrets (DEV: inline; PROD: existingSecret)
secrets:
  adminToken: ""                       # required
  postgresPassword: ""                 # required
  giteaAdminPassword: ""               # required
  giteaRemotePassword: ""              # required (per §3.3a; HTTP Basic
                                       #  credential mirroring the Compose
                                       #  GITEA_REMOTE_PASSWORD)
  webUiSessionSecret: ""               # required
  existingSecret: ""                   # if set, all above ignored

# Storage
storage:
  className: ""                        # cluster default if empty
  postgresSize: 10Gi
  giteaSize: 20Gi
  # Per-service local-clone PVCs (per §1.0 substrate baseline:
  # Gitea is the central remote; orchestrator/executor/evaluator/
  # web-ui each hold their OWN local clone). Sizes can be smaller
  # than the gitea-side bare repo since the per-service clones
  # are working trees.
  orchestratorRepoSize: 5Gi
  executorRepoSize: 5Gi
  evaluatorRepoSize: 5Gi
  webUiRepoSize: 5Gi
  artifactsSize: 50Gi

# Networking
ingress:
  enabled: false
  className: ""                        # cluster default
  hosts:
    webUi: eden.example.com
    gitea: gitea.example.com
  tls: []                              # operator supplies cert refs

# Compose-mirror env knobs (per 12a-1/2/3 + 12b + 12c)
config:
  leaseDurationSeconds: 30             # 12c
  stateSyncIntervalSeconds: 30         # 12c §3.4a
  stateSyncFailureThreshold: 10        # 12c §3.4a
  ideationPolicyTargetPending: 3       # 12a-2
  ideationPolicyMaxTotal: ""           # unbounded
  terminationPolicy: "eden_orchestrator.policies:never_terminate"  # 12a-3
  # On Kubernetes (Decision 9) the orchestrator runs forever
  # because Deployments only support restartPolicy: Always; a
  # quiescence-exit would just produce CrashLoopBackOff. Default
  # 0 disables the quiescence exit. Operators who specifically
  # want the Compose-style quiescence behavior can set this to a
  # non-zero value AND repackage the orchestrator as a Job (out
  # of scope for v0).
  maxQuiescentIterations: 0
  pollIntervalSeconds: 1.0
  # NOTE: --mode subprocess + --exec-mode docker are OUT OF
  # SCOPE for 13a (Decision 10). The chart only supports
  # --mode scripted workers in v0; user-supplied LLM workers
  # need a Kubernetes-native packaging mechanism (planned for
  # 13b "executor as k8s Job"). Operators with subprocess +
  # DooD workloads stay on the Compose stack until 13b lands.
  workerMode: "scripted"               # only "scripted" in 13a
```

The `values.schema.json` enforces shape: required fields
(`secrets.adminToken` etc. unless `existingSecret` is set);
type checks (replica counts are positive ints; storage sizes
match `[0-9]+(Gi|Mi)`); enum values (`workerMode ∈
{scripted}` for 13a).

### 3.3 Secrets handling

The chart's secret strategy mirrors the Compose stack's
existing posture (per §1.0 baseline: Gitea is HTTP-Basic-auth
behind a per-experiment password + credential helper).

**Development.** Operator sets `secrets.adminToken=...`,
`secrets.giteaAdminPassword=...`, `secrets.giteaRemotePassword=...`
etc. inline in `values.yaml`. The chart materializes a single
`Secret` resource named `<release>-eden-secrets` containing all
of them. Pros: works out-of-the-box. Cons: secrets in plaintext
on disk (wherever the values file lives).

**Production.** Operator pre-creates the Secret externally
(via Sealed Secrets, External Secrets Operator, Vault, etc.)
and passes `secrets.existingSecret=my-secret` in values. The
chart's templates reference the existing secret name in
`envFrom: secretRef:` blocks; no secret values flow through
the chart itself.

The `_helpers.tpl` exposes `eden.secretName` which returns
either the chart-managed name or `existingSecret` based on the
values setting. All templates use this helper.

### 3.3a Gitea credential-helper wiring

The Compose stack provisions a per-experiment `git
credential-helper` script directory and bind-mounts it into
each consuming service (orchestrator, executor-host,
evaluator-host, web-ui). Each service passes
`--credential-helper <path>` to `eden_git`'s clone/fetch ops;
the helper script reads `GITEA_REMOTE_PASSWORD` from the env
and emits the credential in git's expected format.

The Helm chart preserves this exact mechanism:

- A new `ConfigMap` resource holds the helper script (it's
  pure shell, doesn't carry the password).
- Each consuming service's pod template mounts the ConfigMap
  as a file at `/etc/eden/git-credential-helper`.
- The pod template injects `GITEA_REMOTE_PASSWORD` as an env
  var from the chart-managed Secret (or `existingSecret`).
- The service's args include `--credential-helper
  /etc/eden/git-credential-helper`.

A new value `secrets.giteaRemotePassword` carries the
password. The chart's `secret.yaml` template includes it
alongside `adminToken` and the other deployment credentials.
For production, the `existingSecret` knob lets operators
supply all credentials via Sealed Secrets / External Secrets
Operator / Vault.

The auth posture is byte-for-byte compatible with the Compose
stack: same HTTP Basic credentials, same git-credential-helper
shape, same per-service mount points. 13e will REPLACE this
HTTP-Basic baseline with per-branch ACLs and Gitea-native PR
review; 13a is mirroring the existing Compose posture, not
upgrading it.

### 3.4 Per-service Deployments / StatefulSets

Each service gets a template file. Common shape:

- `apiVersion: apps/v1`
- `kind: Deployment` (stateless) or `StatefulSet` (stateful)
- `replicas: {{ .Values.replicas.<service> }}`
- `selector` + `template` with consistent labels per
  `_helpers.tpl`'s `eden.labels`.
- `containers`: one container per pod; image from
  `.Values.image`; envFrom: secretRef: `eden.secretName`.
- `volumeMounts` + `volumes` for stateful services'
  PVCs.

**Service-specific knobs** carried via `args:` lists, mirroring
the Compose `command:` shape. Example for orchestrator:

```yaml
args:
  - "--worker-id"
  - "$(POD_NAME)"                    # k8s downward API
  - "--control-plane-url"
  - "http://{{ include "eden.fullname" . }}-control-plane:8080"
  - "--lease-duration-seconds"
  - "{{ .Values.config.leaseDurationSeconds }}"
  - "--task-store-url"
  - "http://{{ include "eden.fullname" . }}-task-store-server:8080"
  - "--ideation-policy"
  - "$(EDEN_IDEATION_POLICY)"
  - "--termination-policy"
  - "{{ .Values.config.terminationPolicy }}"
  - "--max-quiescent-iterations"
  - "{{ .Values.config.maxQuiescentIterations }}"
env:
  - name: POD_NAME
    valueFrom:
      fieldRef:
        fieldPath: metadata.name
```

Using the pod name as `--worker-id` automatically gives each
orchestrator replica a unique worker_id (per 12a-1 §D.1's
grammar). The pod name is `<release>-eden-orchestrator-0`,
`-1`, etc., which matches the `^[a-z0-9][a-z0-9_-]{0,63}$`
grammar.

**Worker host services** (ideator, executor, evaluator) follow
the same shape but the args differ per service (each has its
own `--mode subprocess` flag and binds the experiment-specific
config dir as a hostPath/PVC mount). The worker hosts'
`--worker-id` similarly defaults to `$(POD_NAME)`.

**Stateful services** (Postgres, Gitea):

- Postgres: `StatefulSet` with PVC template; image is the
  upstream `postgres:16.6-alpine` (matches Compose's choice).
  `replicas: 1` in v0 — clustering / managed Postgres is 13c.
- Gitea: `StatefulSet` with PVC for its data dir; image is
  the upstream Gitea image. `replicas: 1`. Auth + ACLs are
  13e.

### 3.5 Setup workflow

The Helm release is **long-lived** and hosts MANY experiments
over its lifetime (per 12c's multi-experiment control plane).
That means a chart-managed `repo-init` Job (with one fixed
release-scoped resource) doesn't fit — a chart-managed
singleton can't bootstrap experiment 2, 3, 4, ... over time
with distinct per-experiment inputs.

The bootstrap workflow instead uses **ad-hoc Jobs created by
`setup-experiment-helm.sh`**, not chart-managed Jobs. The
chart ships a Job *manifest template* under
`reference/helm/eden/bootstrap/repo-init-job.yaml.tmpl` (NOT
under `templates/` — Helm doesn't render this file at
install time); the script renders it per-experiment with
`envsubst` or `jq`/`yq`, then `kubectl apply`s the resulting
manifest. Each experiment gets its own Job named
`eden-repo-init-<experiment_id>`.

```bash
# Operator workflow:
helm install eden ./reference/helm/eden \
  -f values.yaml \
  --create-namespace \
  --namespace eden-prod

# Wait for the chart to come up:
kubectl -n eden-prod rollout status statefulset/eden-postgres
kubectl -n eden-prod rollout status statefulset/eden-gitea
kubectl -n eden-prod rollout status deployment/eden-task-store-server
kubectl -n eden-prod rollout status deployment/eden-control-plane

# Bootstrap an experiment:
bash reference/scripts/setup-experiment-helm.sh \
  --namespace eden-prod \
  --experiment-config tests/fixtures/experiment/.eden/config.yaml \
  --experiment-id exp-1

# The script:
# 1. Calls the control-plane register_experiment endpoint via
#    the API directly (not kubectl exec — uses port-forward
#    or an Ingress endpoint if available).
# 2. Renders the bootstrap/repo-init-job.yaml.tmpl with the
#    experiment_id substituted; kubectl apply -f - to create
#    the Job. Waits for completion via kubectl wait
#    --for=condition=complete.
# 3. The orchestrator StatefulSet is already running; it
#    auto-acquires a lease for the new experiment within
#    one poll-interval.
# 4. Prints the seed SHA + experiment URLs (web-ui ingress
#    if enabled, else port-forward instructions).

# To bootstrap a SECOND experiment later:
bash reference/scripts/setup-experiment-helm.sh \
  --namespace eden-prod \
  --experiment-config /path/to/exp-2.yaml \
  --experiment-id exp-2
# Creates eden-repo-init-exp-2 Job; same flow.
```

Each per-experiment Job is independent: distinct name,
distinct input experiment-config, distinct seed output. The
`setup-experiment-helm.sh` script's idempotency rule is
"run twice with the same `--experiment-id` and the second
run sees the registry entry exists, the bare repo is already
seeded, and exits 0 without re-running the Job".

A future amendment may move bootstrap into the control
plane's wire surface as a first-class `bootstrap_experiment`
op (so operators don't need direct cluster access — just an
admin-token POST). For 13a, the script-driven flow is the
canonical mechanism; `kubectl` access is required.

### 3.6 Helm smoke CI job

A new `helm-smoke` CI job mirrors `compose-smoke`:

- Spins up a kind cluster.
- Installs the chart with `helm install eden
  reference/helm/eden -f reference/helm/eden/ci-values.yaml
  --wait`.
- Runs `setup-experiment-helm.sh` with the fixture experiment.
- Waits for the orchestrator to drive 3 ideation tasks → 3
  variants → 3 integrations.
- Asserts (via `kubectl exec` into the task-store-server pod
  and querying the wire endpoint): ≥3 `variant.integrated`
  events, ≥9 `task.completed` events, ≥3 ideate-task
  `task.completed` events. Same shape as `compose-smoke`.
- Tears down with `helm uninstall eden && kind delete cluster`.

`ci-values.yaml` pins specific values for CI (small storage
sizes, dev secrets inline, no ingress).

The kind cluster's setup uses
`actions/setup-kind@v1` (or equivalent); install Helm via
`azure/setup-helm@v3`. Both are standard GitHub Actions
patterns.

### 3.7 Conformance: helm-smoke is the same end-state assertion

The Helm deployment passes the same chapter-9 conformance
contracts as the Compose deployment — the protocol is
unchanged. No new conformance scenarios are added in 13a; the
existing v1, v1+roles, v1+roles+integrator, v1+checkpoints,
v1+multi-experiment levels (post-12c) all apply.

The new `helm-smoke` smoke is the substrate-level assertion
that the protocol works on Kubernetes; conformance scenarios
(which run against the wire, not against a substrate) cover
the protocol assertions.

### 3.8 Alternatives considered

Two architectural choices in this chunk benefit from explicit
compare-and-reject paragraphs.

**Embedded Postgres + Gitea (chosen) vs. upstream chart
dependencies.** Decision 1 picked "one chart, no umbrella";
that decided the EDEN-internal split. A separate question:
should the chart EMBED its own Postgres + Gitea
StatefulSets, or DEPEND on the maintained upstream Helm
charts (e.g., `bitnami/postgresql`, `gitea/gitea`)? The
upstream charts handle a lot of operational concerns —
backup hooks, init-container migrations, monitoring
endpoints, replication options — that the embedded version
doesn't. For 13a, embedding wins:

1. **13c and 13e replace these anyway.** The Postgres and
   Gitea bundles in 13a are explicit stopgaps. 13c migrates
   to a managed external Postgres (chart's `postgres`
   becomes optional + a connection-string knob); 13e
   replaces Gitea with a auth-aware version (TBD whether
   that's still embedded). Investing in upstream-chart
   plumbing now (which means picking + pinning a chart
   version, vendoring its values surface, handling its
   upgrade semantics) creates churn in 13c / 13e for no
   v0 benefit. A 50-line StatefulSet template is cheaper to
   replace than a vendored subchart's complete value
   structure.
2. **Conformance smoke determinism.** The `helm-smoke` CI
   job runs against the chart-embedded versions; the
   versions are pinned to specific Postgres + Gitea image
   tags that the existing `compose-smoke` already exercises.
   Depending on an upstream chart introduces another moving
   piece in CI (the upstream chart's release cadence vs
   ours).
3. **Vendoring overhead.** Helm's chart-dependency mechanism
   (`Chart.yaml`'s `dependencies:` block + `helm dep
   update`) requires the operator (or the CI pipeline) to
   resolve + pull dependencies before install. That's a
   minor friction but real; the embedded approach has zero
   external dependencies once the chart is cloned.

If a future deployment needs production-grade Postgres or
Gitea operationally (HA, encryption-at-rest, backup
hooks), 13c and 13e are the right place — not retrofitting
upstream charts into 13a's stopgap.

**Helm-vanilla (chosen) vs. small operator/controller.** The
chart pushes a fair amount of lifecycle work outside Helm's
static install: per-experiment bootstrap Jobs (§3.5),
script-driven control-plane registration calls, kubectl
exec-equivalent flows. A small operator (a Custom Resource
Definition like `Experiment` plus a controller that watches
it and handles bootstrap + registration + lease assignment)
would consolidate that lifecycle into a single declarative
pattern.

Why Helm-vanilla anyway:

1. **One substrate dependency vs. two.** Helm + scripts is
   "kubectl + bash"; an operator pattern adds a controller
   binary, RBAC for CRD watch + Pod create, plus operator
   versioning. For 13a that's significant new substrate.
   The operator pays off when there are MANY experiments
   per deployment (each Custom Resource is a tidy
   declarative entry); in v0 a deployment may host 1-10
   experiments and the script-driven path is simpler.
2. **Iteration speed.** Helm + scripts can be edited
   without an operator-build cycle. The bootstrap flow's
   shape is still settling (the §3.5 design is plausible
   but unproven); freezing it into a controller binary too
   early locks in a design that may need revision in 13b/c.
   13a's script-driven flow makes those revisions cheap.
3. **Ecosystem reach.** Operators / controllers require
   cluster-admin rights to install (CRDs are
   cluster-scoped). Some shared k8s clusters don't grant
   that easily. Helm-vanilla works in namespace-only
   environments.

A future amendment MAY add an operator pattern as a
deployment option (alongside, not replacing, the Helm
chart). v0 keeps the substrate footprint minimal.

### 3.9 What 13a does NOT do

- **Cloud-provider helm extensions.** The chart doesn't ship
  AWS / GCP / Azure value files. Operators target their
  cloud's specifics (IRSA, GKE Workload Identity, ingress
  classes) via standard Helm value overrides.
- **Operator pattern.** No CRDs, no controllers. v0 stays
  Helm-vanilla.
- **Multi-tenancy at the cluster level.** A single chart
  release is one deployment. Multiple releases per cluster
  (different namespaces) are supported but each is independent
  — there's no shared control plane across releases.
- **Helm test resources beyond a connection check.** The
  `tests/test-connection.yaml` is a minimal `helm test`
  resource that verifies each service's HTTP endpoint
  responds. Full smoke testing happens in CI via the
  `helm-smoke` job, not via `helm test`.
- **Image build pipeline.** The chart pulls the existing
  `eden-reference:dev` image; building + pushing is the
  operator's separate concern (or a separate CI pipeline).

## 6. Test design

### 6.1 Unit-equivalent tests (`helm template` + `helm lint`)

Helm doesn't have a unit-test framework in the traditional
sense; the closest is `helm template` (renders templates to
YAML) + `helm lint` (validates chart structure).

- `helm lint reference/helm/eden`: pass.
- `helm template eden reference/helm/eden -f
  reference/helm/eden/ci-values.yaml`: produces valid YAML
  parsed by `kubectl apply --dry-run=client`.
- `helm template ... | yamllint`: pass (consistent
  formatting).

These run in a new lightweight CI job `helm-lint` that's fast
(seconds) and runs on every push. The full `helm-smoke`
(below) is heavier and runs on a slower schedule.

### 6.2 `helm-smoke` integration test

Per §3.6:

- Spin up kind via `actions/setup-kind`.
- `helm install eden reference/helm/eden -f
  reference/helm/eden/ci-values.yaml --create-namespace
  --namespace eden-test --wait --timeout 5m`.
- `bash reference/scripts/setup-experiment-helm.sh
  --namespace eden-test
  --experiment-config tests/fixtures/experiment/.eden/config.yaml
  --experiment-id exp-1`.
- Poll the task-store-server's wire endpoint via
  `kubectl exec` until quiescence (orchestrator exits 0 OR 3
  `variant.integrated` events seen — same shape as
  `compose-smoke`'s 180s budget).
- Assert via the wire endpoint (run a one-shot `kubectl run
  --rm -it ... -- python -c "..."` curl-equivalent):
  - ≥3 `variant.integrated` events
  - ≥9 `task.completed` events
  - ≥3 ideate-task `task.completed` events
- `helm uninstall eden -n eden-test && kind delete cluster`.

### 6.3 Chart upgrade test

A separate `helm-upgrade-smoke` test:

- Install the chart at `0.1.0`.
- Run a small fixture (1 variant).
- `helm upgrade eden ./reference/helm/eden ...` (no version
  bump but exercises the upgrade machinery).
- Assert the experiment is unaffected (still has its tasks /
  variants; lease holders unchanged within
  `lease_duration`).

This catches PVC reclaim policy mistakes (data loss on
upgrade) early.

### 6.4 `helm test` connection probe

`reference/helm/eden/templates/tests/test-connection.yaml`
materializes a Job that:

- Curls each service's healthcheck endpoint
  (task-store-server `/healthz`, control-plane `/healthz`,
  web-ui `/healthz`).
- Asserts each returns 200.

Run via `helm test eden -n eden-test` after install. The
`helm-smoke` CI job calls this as part of its post-install
verification.

## 7. Tricky areas

### 7.1 PVC retention vs reclaim policy

The default `StorageClass` in many clusters has
`reclaimPolicy: Delete`, so when a PVC is deleted the
underlying storage is reclaimed. For Postgres + Gitea + the
bare-repo + artifacts volumes, that means a careless `helm
uninstall` deletes ALL experiment data.

The chart's PVCs use `volumeClaimTemplates` (StatefulSet) which
don't auto-delete on `helm uninstall` — k8s leaves the PVCs
behind for manual cleanup. Operators who want to fully
tear down run `kubectl delete pvc -n eden-test --all` after
`helm uninstall`. Documented in the README.

For the CI smokes, this is fine — `kind delete cluster`
removes everything regardless.

### 7.2 Image pull secrets in private registries

The `image.pullSecrets` value lets operators reference a
pre-created `imagePullSecret` for private registries. Default
empty (assumes public registry). Documented in the README
with a worked example for ECR / GCR / GHCR.

### 7.3 `replicaCount: 0` to disable a service

Operators may want to run only some services (e.g., disable
the orchestrator for a manual-UI-only deployment). Setting
`.Values.replicas.orchestrator = 0` produces a Deployment with
0 replicas — the service is "deployed but idle". This is
preferred over conditional template inclusion (which makes
upgrade behavior fragile).

The same pattern applies to all stateless services. Stateful
services (Postgres, Gitea) cannot reasonably scale to 0 in v0
— if you don't want them, edit the chart.

### 7.4 Pod restarts vs lease handoff

A Helm-driven rolling restart of the orchestrator StatefulSet
(per Decision 3 — `kubectl rollout restart statefulset/...`)
terminates pods one at a time, ordered (StatefulSet's default
rolling-update is `OrderedReady`, decreasing-ordinal). Each terminating pod's leases expire (per 12c
§3.2); during the brief window before re-acquisition by the
new pod, the experiment is un-leased and no dispatch happens.

For typical deployments this is fine — the window is
sub-`lease_duration` (30s default). For deployments where
even 30s of un-dispatched is unacceptable, the chart's default
`replicas.orchestrator = 2` ensures another replica is
available to acquire the lease during the rolling restart.

### 7.5 Helm + Kubernetes version constraints

The chart targets Kubernetes 1.27+ (matches what most clouds
support as of 2026). It uses:

- `apps/v1` Deployments and StatefulSets (stable since 1.9).
- `networking.k8s.io/v1` Ingress (stable since 1.19).
- `helm test` framework (Helm 3.0+).
- `values.schema.json` (Helm 3.x).

Older clusters fail at install time with a clear error.
Documented in the README.

### 7.6 `setup-experiment-helm.sh` vs control-plane availability

The script calls the control plane to register the experiment.
If the control plane isn't ready yet (the chart just installed,
control-plane pod is still starting), the call fails. The
script polls the control-plane's `/healthz` for up to 60s
before attempting registration. Documented in the script's
help text.

## 8. Risks

1. **Default StorageClass missing.** A cluster without a
   default `StorageClass` rejects PVC creation; install
   fails. Mitigation: README requires the operator confirms
   storage provisioning before installing. The
   `values.schema.json` could enforce that
   `storage.className` is non-empty if no default is
   configured, but the chart doesn't have visibility into
   the cluster's default — operator responsibility.

2. **kind cluster resource limits in CI.** kind in CI runs in
   a single Docker container with limited CPU/memory. The
   chart's nine services may push the limits — Postgres alone
   is hungry. Mitigation: `ci-values.yaml` sets minimal
   storage sizes + relaxed `resources.requests` on
   non-critical services. If `helm-smoke` is flaky in CI, the
   operator-side tuning lives in `ci-values.yaml` rather than
   the production-default `values.yaml`.

3. **Image build vs chart version drift.** The chart's
   `appVersion` and the actual `eden-reference` image tag can
   drift if the operator builds a new image but doesn't bump
   the chart. Mitigation: a future amendment (13b/c/d/e or
   later) wires the chart's `appVersion` to the project's
   semver via a build-time substitution; v0 documents the
   drift risk.

4. **Re-introducing legacy vocab in chart docs.** PR #60's
   guardrail catches the patterns. The chart's README is a
   new doc; the values file mirrors `.env.example` shapes
   that are already known clean. Pre-submit
   `python3 scripts/check-rename-discipline.py` clean.

5. **Helm's templating quirks.** The chart's `_helpers.tpl`
   uses Sprig functions for string manipulation. A typo in a
   helper produces a chart that lints clean but renders
   broken YAML. Mitigation: `helm template` + `kubectl apply
   --dry-run=client` pre-merge catches this.

## 9. Sequencing

Recommended PR shape (in order):

1. **Chart skeleton PR.** `Chart.yaml`, `values.yaml`,
   `values.schema.json`, `_helpers.tpl`, `secret.yaml`,
   `README.md`. Lints clean, but contains no service
   templates yet. Reviewable as a structural foundation.

2. **Orchestrator code change PR (Decision 9).** The
   `--max-quiescent-iterations 0` "never exit" sentinel
   lands in the orchestrator service per §5.1a. This must
   precede the chart PRs since the chart's k8s-default
   value depends on it.

3. **Stateful infra PR.** Postgres + Gitea StatefulSets +
   their Services. Operators can `helm install` and have the
   stores up.

4. **Stateless control-plane services PR.**
   task-store-server + control-plane Deployments + Services.
   The protocol-level coordination layer is up.

5. **Orchestrator + worker hosts PR.** orchestrator
   StatefulSet, executor-host StatefulSet, evaluator-host
   StatefulSet (each with per-replica clone PVCs via
   volumeClaimTemplates per Decision 3); ideator-host
   Deployment. Multi-replica orchestrator + worker fleet.

6. **Web UI + ingress PR.** web-ui StatefulSet (per
   Decision 3 — repo-clone + artifacts PVCs) + Service +
   optional Ingress. Full operator surface.

7. **Bootstrap-template + setup script PR.**
   `bootstrap/repo-init-job.yaml.tmpl` (NOT under templates/)
   plus `setup-experiment-helm.sh`. End-to-end operator
   workflow.

8. **CI PR.** `helm-lint`, `helm-smoke`,
   `helm-upgrade-smoke` jobs. Validation in CI.

9. **Docs PR.** `docs/deployment/helm.md` + AGENTS.md
   commands + roadmap delta.

A reviewer going from PR 1 to PR 9 should expect the
`helm-smoke` job to go red around PR 5 (when the orchestrator
is added but worker hosts aren't all green yet) and come back
green at PR 7.

## 12. What lands at the end of 13a

After 13a merges, an operator can run `helm install eden
./reference/helm/eden -f values.yaml` on any conforming
Kubernetes cluster and get a complete EDEN deployment running
the full Phase-12 protocol (workers + groups + leases +
control plane + portable checkpoints). The Compose deployment
stays as the "local development" path; Helm is the
"production-ish" path. 13b through 13e add cloud-specific
extensions to the foundation 13a establishes.
