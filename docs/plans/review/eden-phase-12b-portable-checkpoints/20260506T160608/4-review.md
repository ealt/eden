**Findings**

1. The plan still needs an explicit rejected-alternatives note for the archive encoding. The design source keeps the logical format as a directory tree and treats tar/zip as transport conveniences [portable-checkpoints.md:36](/Users/ericalt/Documents/eden/docs/design/portable-checkpoints.md:36), [portable-checkpoints.md:78](/Users/ericalt/Documents/eden/docs/design/portable-checkpoints.md:78), but the plan hard-codes a tar media type on both export and import [eden-phase-12b-portable-checkpoints.md:423](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:423), [eden-phase-12b-portable-checkpoints.md:435](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:435). That choice is plausible, but the plan should say why tar beats zip for v0, and why a newline-JSON-only / directory-only wire shape was rejected. Right now the rationale is implicit rather than design-recorded.

2. The `sha256` artifact-addressing choice also deserves a brief alternatives note. The current justification is basically “it matches existing SHA-shaped vocabulary” [eden-phase-12b-portable-checkpoints.md:25](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:25), while the choice itself becomes normative for artifact identity [eden-phase-12b-portable-checkpoints.md:122](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:122). Since you specifically make artifacts content-addressed across implementations, the plan should record why not BLAKE3 and why not git-style object IDs. A short note like “BLAKE3 is faster but less universal in standard tooling; git object IDs would couple artifact identity to git hash-family/runtime choices even though artifacts are not git objects” would close this out cleanly.

3. The separate `/v0/checkpoints/import` root is now coherent, but the plan should still document why that exception is preferable to a fully experiment-scoped binding. You now have to amend chapter 07’s global scoping rule to make it legal [eden-phase-12b-portable-checkpoints.md:763](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:763), which means this is not a free choice. Because the design source simply proposes the path [portable-checkpoints.md:218](/Users/ericalt/Documents/eden/docs/design/portable-checkpoints.md:218), the plan should add one paragraph comparing the chosen global root with an alternative like a two-step experiment-scoped import flow. The current choice may still be right; it just needs the tradeoff written down.

**Overall Assessment**

Round 4 gets the plan past feasibility for me. I would now stop at alternatives: the core design is coherent, but the plan still needs a small amount of “why this, not that” documentation around the archive wrapper, the hash scheme, and the global import root so those normative choices are deliberate rather than merely inherited.
