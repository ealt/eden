**Findings**

1. The plan still has no explicit contract for concurrent imports of the same checkpoint. It has a dedicated section for multiple concurrent exports [12b:1213](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:1213), but the import side only covers the sequential “experiment already exists” case [12b:938](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:938). For the chosen single-call import design, the edge case needs to be pinned: if two operators concurrently submit the same checkpoint to the same target `experiment_id` (or same `as_experiment_id`), does exactly one win and the other(s) get `409 eden://error/experiment-id-conflict`? That should be stated in §7 and tested in §6.

2. The “JSONL references refs the bundle doesn’t have” case is still treated as “impossible,” which is too narrow for an import surface. The plan correctly handles the opposite direction by ignoring bundle refs unreferenced by protocol objects [12b:1060](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:1060). But for the case where protocol objects reference refs missing from the bundle, it says that is impossible because of the reference exporter’s ordering [12b:1062](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:1062), [12b:1064](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:1064). That is not enough once checkpoints are portable files that can come from other implementations or be corrupted in transit. The importer needs an explicit validation-and-reject rule here, plus a malformed-checkpoint test in §6.

**Overall Assessment**

Round 6 gets the plan past completeness. I would stop at edge cases: the terminated-experiment and mid-stream export-interruption cases are now covered well enough, but concurrent-import semantics and malformed “refs-missing-from-bundle” checkpoints still need explicit contract text and tests.
