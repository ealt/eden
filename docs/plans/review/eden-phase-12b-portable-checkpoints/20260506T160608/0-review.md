**Findings**

1. The plan does not pin the authoritative 12a object shapes it intends to serialize, so an outside reviewer cannot tell whether the checkpoint format is preserving protocol objects or inventing a new lossy projection. It defines `workers.jsonl` as `worker_id`, `created_at`, `groups` [12b:375](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:375) and `groups.jsonl` as `group_id`, `created_at`, `members` [12b:380](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:380), but 12a-1’s actual `Worker` shape is `worker_id`, `experiment_id`, `registered_at`, `registered_by`, `labels`, `auth_credential_hash` [12a-1:109](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:109) and its `Group` shape is `group_id`, `experiment_id`, `members`, `created_at`, `created_by` [12a-1:207](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:207). The uncertainty note that `Worker` “may already exist post-12a-1; verify when implementing” [12b:632](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:632) confirms the dependency is not nailed down. Before feasibility review, this needs an explicit mapping table: for each checkpoint file, which protocol object(s) it carries, which fields are preserved verbatim, which are omitted, and why.

2. The plan also lacks one authoritative statement of the import posture: is a checkpoint supposed to restore exact live state, or restore only structurally equivalent resumable state? Right now that contract is scattered across `requires_credential_reissue` and “revert claimed tasks to pending” [12b:318](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:318), [12b:395](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:395), the round-trip bullet that says claims are identical except they “MAY be reset” [12b:446](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:446), and the separate preservation rule for terminated experiments [12b:458](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:458). That missing summary matters because 12a-1’s credential model distinguishes stable worker registration from explicit credential reissue [12a-1:155](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:155), [12a-1:173](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:173), and the web UI already has intentionally non-stable per-session credentials [12a-1:393](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:393). The plan should say plainly, near §1 or §2, whether import preserves “exact state” or “resume-safe state with volatile auth/claim material normalized.”

**Overall Assessment**

I would stop at missing context. The plan has enough high-level structure, but it has not yet pinned the exact predecessor shapes and the exact import-state contract, so deeper feasibility review would be partly speculative. Once those two contracts are made explicit, the feasibility questions you called out become much easier to review cleanly.
