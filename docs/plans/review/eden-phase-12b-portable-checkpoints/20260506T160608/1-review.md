**Findings**

1. The terminated-experiment path is still internally broken. The plan says import preserves `experiment.state`, including `terminated` [12b:503](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:503), but it also says every claimed task is reverted to `pending` on import [12b:183](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:183), [12b:202](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:202), [12b:440](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:440). That conflicts with 12a-3’s termination contract, where already-claimed work is what drains after termination and new claims are rejected [12a-3:122](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:122), [12a-3:645](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:645), [12a-3:706](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:706). A checkpoint taken from a terminated experiment mid-drain would import into a stuck state: the experiment is terminated, the in-flight tasks are now pending, and those pending tasks cannot be claimed. There is also a direct self-contradiction in §3.6 step 5, which says the experiment starts in `state="running"` [12b:440](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:440), versus §3.8 saying terminated state is preserved [12b:503](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:503). This needs one consistent rule.

2. The round-trip equivalence contract has not actually been rewritten to match the new resume-safe import model. Decision 10 now says auth/claim state is not preserved [12b:172](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:172), but §3.8 still says tasks are identical, including claims [12b:491](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:491), and still calls export→import a no-op modulo URI/event-id/credential reassignment [12b:508](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:508). The tests are written against that stronger claim too: storage tests promise “full equivalence” [12b:724](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:724) and conformance promises “round-trip equivalence” [12b:774](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:774). As written, the plan simultaneously requires and forbids preserving claim state. §3.8 and the test plan need to define equivalence over structural state only, with task-state normalization called out explicitly.

3. The atomicity/streaming story is not internally consistent. Decision 8 makes chunked transfer the endpoint contract [12b:155](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:155), but §4.2 says server-side chunked encoding is only a recommendation, not a normative requirement [12b:613](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:613), while wire tests assert chunked transfer specifically [12b:748](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:748). On the atomicity side, §3.7 says the transactional snapshot commits “at the end of the export” [12b:460](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:460), but §7.2’s mitigation closes the transaction after writing a temp file and only then serves the HTTP stream [12b:851](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:851). Those are two different contracts. The plan needs to choose whether HTTP chunking is required or optional, and whether “export” means “finish materializing the snapshot artifact” or “finish sending bytes to the client.”

4. The experiment-id collision rule is still underspecified against retry/idempotency semantics. The plan says a same-id import returns 409 unless the operator supplies `as_experiment_id` [12b:128](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:128), [12b:395](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:395), and the backend tests encode that as `AlreadyExists` [12b:730](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:730). But this is a large streaming upload endpoint [12b:388](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12b-portable-checkpoints.md:388): if the server commits the import and the client loses the `201 Created`, a safe retry becomes impossible under the current rule. The plan needs to say explicitly whether import is intentionally non-idempotent create-only, or whether replaying the same checkpoint bytes to the same target ID is treated as idempotent success.

**Overall Assessment**

The new context sections are coherent enough to move past level 1. I would stop at feasibility: the plan now has a clearer model, but the resume-safe import contract has not been propagated consistently through §3.6, §3.8, the terminated-experiment behavior, the streaming/atomicity story, or the retry semantics for import.
