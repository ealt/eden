**Findings**
- Bug — [04-task-protocol.md](/Users/ericalt/Documents/eden/spec/v0/04-task-protocol.md:162), §4.1 says: “A successful submit transitions `claimed → submitted` (or, under the conditions in §4.3, `claimed → failed`)”. But [04-task-protocol.md](/Users/ericalt/Documents/eden/spec/v0/04-task-protocol.md:37), §1.2 says the permitted transitions are exactly `claimed → submitted` and then `submitted → failed`. That is a direct contradiction in the state machine.

- Bug — [03-roles.md](/Users/ericalt/Documents/eden/spec/v0/03-roles.md:41), §1 step 3 says: “The worker MAY report progress; it MUST NOT mutate any protocol-owned object other than the task it holds a claim on.” That conflicts with required role outputs in [03-roles.md](/Users/ericalt/Documents/eden/spec/v0/03-roles.md:173), §3.2 (“Persist a trial...”) and [03-roles.md](/Users/ericalt/Documents/eden/spec/v0/03-roles.md:246), §4.2 (“Set the trial’s `completed_at` timestamp”). The evaluator section then repeats the problem in [03-roles.md](/Users/ericalt/Documents/eden/spec/v0/03-roles.md:256): “The evaluator MUST NOT modify... any other protocol-owned mutable state.”

- Bug — [02-data-model.md](/Users/ericalt/Documents/eden/spec/v0/02-data-model.md:187), §3.4 says: “The `claim` field MUST be present when `state == "claimed"`, and MUST NOT be present for any other state.” The schema enforces that in [task.schema.json](/Users/ericalt/Documents/eden/spec/v0/schemas/task.schema.json:102). But [04-task-protocol.md](/Users/ericalt/Documents/eden/spec/v0/04-task-protocol.md:168), §4.2 defines idempotent resubmit as “a second submit carrying the same token as an earlier successful submit”, and [04-task-protocol.md](/Users/ericalt/Documents/eden/spec/v0/04-task-protocol.md:240), §5.1 allows `submitted` reclamation based on “`claim.expires_at`”. Once the task is `submitted`, the normative object no longer carries the claim or its expiry, so the spec does not define what remains authoritative.

- Bug — [03-roles.md](/Users/ericalt/Documents/eden/spec/v0/03-roles.md:143), §2.4 says `proposal_ids` are reported “in the order the planner considers most preferred.” [04-task-protocol.md](/Users/ericalt/Documents/eden/spec/v0/04-task-protocol.md:176), §4.2 then defines plan resubmission equivalence as “the set of `proposal_ids` (as a set, not a sequence)”. That discards declared preference semantics during idempotency checks.

- Bug — [02-data-model.md](/Users/ericalt/Documents/eden/spec/v0/02-data-model.md:288), §7.1 and [trial.schema.json](/Users/ericalt/Documents/eden/spec/v0/schemas/trial.schema.json:27) include trial status `eval_error`, and [01-concepts.md](/Users/ericalt/Documents/eden/spec/v0/01-concepts.md:86) presents it as a real trial state. But [03-roles.md](/Users/ericalt/Documents/eden/spec/v0/03-roles.md:278), §4.4 says on evaluator `eval_error`: “The trial’s final status remains `starting`”, and [04-task-protocol.md](/Users/ericalt/Documents/eden/spec/v0/04-task-protocol.md:208), §4.3 says the same. That leaves a frozen enum value with no reachable transition or defined meaning.

- Gap — [01-concepts.md](/Users/ericalt/Documents/eden/spec/v0/01-concepts.md:71) says proposal lifecycle “`drafting → ready → dispatched → completed`” “is defined in `04-task-protocol.md`.” But [04-task-protocol.md](/Users/ericalt/Documents/eden/spec/v0/04-task-protocol.md:90), §2 only normatively defines `ready → dispatched`, and [04-task-protocol.md](/Users/ericalt/Documents/eden/spec/v0/04-task-protocol.md:324), §7 says `dispatched → completed` only “MAY” happen. Proposal completion is therefore optional even though the frozen chapters present it as part of the defined lifecycle.

- Gap — [03-roles.md](/Users/ericalt/Documents/eden/spec/v0/03-roles.md:176), §3.2 requires the implementer to persist a `starting` trial before repo writes. [04-task-protocol.md](/Users/ericalt/Documents/eden/spec/v0/04-task-protocol.md:260), §5.2 then says reclaimed tasks do not delete “starting trials”. The spec never says whether a reclaimed/retried implement task must reuse that trial, may create a new one, or how many `starting` trials may exist for one proposal/task. Implementations will diverge.

- Gap — [01-concepts.md](/Users/ericalt/Documents/eden/spec/v0/01-concepts.md:13) says an experiment has “A starting commit on a designated branch of a git repository.” [03-roles.md](/Users/ericalt/Documents/eden/spec/v0/03-roles.md:167), §3.1 says the implementer gets “Access to the git repository identified by the experiment config.” But [02-data-model.md](/Users/ericalt/Documents/eden/spec/v0/02-data-model.md:86) and [experiment-config.schema.json](/Users/ericalt/Documents/eden/spec/v0/schemas/experiment-config.schema.json:6) still define the host-independent config without any repository or starting-commit field. The new chapters depend on protocol-core data that the frozen config does not define.

- Risk — [04-task-protocol.md](/Users/ericalt/Documents/eden/spec/v0/04-task-protocol.md:243), §5.1 permits reclaim when “the claiming worker is unreachable” even from `submitted`, while [04-task-protocol.md](/Users/ericalt/Documents/eden/spec/v0/04-task-protocol.md:276), §5.4 says a `submitted` task “MUST be processed by the orchestrator” and automatic reclaim “SHOULD NOT” apply there. After submit, worker reachability is no longer the relevant liveness signal, so the reclamation boundary for `submitted` is underspecified.

- Nit — [03-roles.md](/Users/ericalt/Documents/eden/spec/v0/03-roles.md:264), §4.3 says evaluation side effects “MUST be local to the evaluator’s own workspace.” That is the only place I saw meaningful host-model leakage. The portable invariant is that side effects must not escape through protocol-owned refs/stores except via declared outputs; “workspace” reads subprocess/local-host flavored.

**Assumptions**
- No project `CLAUDE.md` exists in this repo; I used [AGENTS.md](/Users/ericalt/Documents/eden/AGENTS.md:1) as the governing project instructions.
- I treated `00`/`01`/`02` and the frozen schemas as authoritative; where `03`/`04` disagree, I treated that as a defect in the new chapters.

**Assessment** These chapters are not ready to merge. The blocking issues are the contradictory submit path, the unresolved claim/token model once a task reaches `submitted`, and the lifecycle mismatches against frozen proposal/trial semantics and experiment config. Once those are fixed, a shorter follow-up pass should be enough to clean up the remaining wording-level issues.
