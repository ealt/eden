**Findings**

1. Bug — [spec/v0/06-integrator.md](/Users/ericalt/Documents/eden/spec/v0/06-integrator.md:136) §3.2 conflicts with [spec/v0/06-integrator.md](/Users/ericalt/Documents/eden/spec/v0/06-integrator.md:200) §4. The squash rule says the canonical commit `T` MUST satisfy `tree(T) == tree(commit_sha)`, but §4 also requires `T` to contain `.eden/trials/<trial_id>/eval.json`. Those cannot both be true unless the worker branch already contains the manifest, which the implementer/evaluator contracts do not require and effectively forbid. Fix by changing the tree invariant to “worker tip tree plus the eval-manifest path” or by moving the manifest out of the committed tree; then update the repeat-promotion/idempotency check in §5.3 to match.

2. Bug — [spec/v0/06-integrator.md](/Users/ericalt/Documents/eden/spec/v0/06-integrator.md:221) §4 requires eval-manifest fields the rest of the protocol never produces. `evaluator.{name,version}` and the per-artifact `{uri, sha256, bytes}` entries have no protocol-defined source: [spec/v0/03-roles.md](/Users/ericalt/Documents/eden/spec/v0/03-roles.md:285) only gives evaluator submissions `trial_id`, `status`, `metrics`, and optional `artifacts_uri`, and [spec/v0/08-storage.md](/Users/ericalt/Documents/eden/spec/v0/08-storage.md:237) only defines artifact `upload`/`fetch`, not listing or metadata discovery. A conforming integrator cannot build the required manifest from protocol-owned state. Fix by carrying evaluator identity/version and artifact inventory/hash data in the evaluator submission/trial model, or relax the manifest requirements until those fields exist elsewhere in the spec.

3. Bug — [spec/v0/05-event-protocol.md](/Users/ericalt/Documents/eden/spec/v0/05-event-protocol.md:251) §3.4 promises that registered events alone can reconstruct every protocol-owned state change, but the registry payloads do not carry the fields those transitions write. `proposal.drafted`/`proposal.ready` omit proposal contents, `trial.started` omits branch/parents/`started_at`, `trial.succeeded`/`trial.errored` omit metrics/artifacts/`completed_at`, and `task.created` omits task payload/timestamps. [spec/v0/schemas/event.schema.json](/Users/ericalt/Documents/eden/spec/v0/schemas/event.schema.json:43) and [reference/packages/eden-contracts/src/eden_contracts/event.py](/Users/ericalt/Documents/eden/reference/packages/eden-contracts/src/eden_contracts/event.py:49) are in lockstep with that minimal registry, so schema/model parity is fine, but the protocol claim is not. Fix by either expanding required event payloads to include the full state delta/snapshot for each protocol-owned write, or narrowing the spec’s reconstruction guarantee to lifecycle transitions only and aligning the cross-chapter observability language.

4. Risk — [spec/v0/08-storage.md](/Users/ericalt/Documents/eden/spec/v0/08-storage.md:24) specifies concrete operations only for tasks, events, and artifacts, even though [spec/v0/05-event-protocol.md](/Users/ericalt/Documents/eden/spec/v0/05-event-protocol.md:53) makes the transactional invariant apply equally to proposals and trials and [spec/v0/03-roles.md](/Users/ericalt/Documents/eden/spec/v0/03-roles.md:109) requires durable proposal/trial writes. As written, a reimplementation can satisfy the storage chapter literally while leaving proposal/trial persistence semantics under-specified. Fix by defining the proposal/trial repository contract explicitly, or by stating which store owns those writes and that the same durability, read-after-write, and atomicity rules apply.

5. Risk — [spec/v0/05-event-protocol.md](/Users/ericalt/Documents/eden/spec/v0/05-event-protocol.md:310) says replay from an experiment’s first event is required, but [spec/v0/05-event-protocol.md](/Users/ericalt/Documents/eden/spec/v0/05-event-protocol.md:335) and [spec/v0/08-storage.md](/Users/ericalt/Documents/eden/spec/v0/08-storage.md:134) allow compaction to reject older replays. That ambiguity weakens the replayability floor and leaves room for incompatible implementations. Fix by choosing one contract and stating it consistently: either lifetime full replay while an experiment is retained, or replay only from a declared retention horizon.

**Overall Assessment**

Phase 4 does land the planned 4a+4b+4c artifacts, and I did not find a schema↔model literal drift on the implemented 15-event registry: the enum values and commit-SHA pattern are aligned between `event.schema.json` and `event.py`. I would still request changes before calling the phase coherent. The main blockers are protocol-level: the integrator chapter contains two hard contradictions, and the event/storage chapters currently promise reconstruction and replay guarantees that the defined registry and storage contract do not actually make implementable.
