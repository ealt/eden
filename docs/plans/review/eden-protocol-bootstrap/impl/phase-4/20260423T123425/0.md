## Change Summary

- `05-event-protocol.md` pins the envelope, the transactional
  invariant (atomic event+state), the closed registry of 15 v0 event
  types (6 task / 4 proposal / 5 trial), payload shapes per type,
  composite-commit cases enumerated (e.g. `task.created` +
  `proposal.dispatched` for implement dispatch), and delivery
  guarantees (per-experiment total order, causal consistency,
  at-least-once, replayability, durability).
- `event.schema.json` now carries `$defs` for shared types
  (`taskId`, `commitSha`, `failReason`, `reclaimCause`, …) and an
  `allOf` array of 15 `if/then` clauses that dispatch on `type` to
  enforce per-type `data` payload shapes. Unregistered types pass
  the envelope only (§3.5 open extension).
- `06-integrator.md` pins: three-namespace topology with `main`
  immutability; sole-integrator rule for `trial/*`; promotion
  trigger (`status==success`, `commit_sha` set,
  `trial_commit_sha` absent); squash rule (single commit with
  `tree(T)==tree(commit_sha)` and parents from proposal's
  `parent_commits`); commit-message format (`trial: <trial_id>
  <slug>`); atomic-three invariant (ref write + `trial_commit_sha`
  field + `trial.integrated` event); eval manifest at
  `.eden/trials/<trial_id>/eval.json` with required fields
  (trial_id, proposal_id, commit_sha, parent_commits, metrics,
  artifacts, evaluator, evaluated_at); artifacts entries include
  `sha256` and `bytes` for reproducibility.
- `08-storage.md` specifies the three stores. Task store: operation
  table, atomic claim (linearizable), token-authorization rule,
  idempotent resubmit, terminal immutability, per-task
  serialization. Event log: append / read_range / subscribe /
  replay operations, ordering + causality, at-least-once delivery,
  retention floor. Durability: write durability, read-after-write
  consistency, crash recovery, no-fabrication. Per-experiment
  metrics-schema enforcement. Artifact store: upload/fetch,
  durability tied to retention, content-integrity (no overwrite
  under a referenced URI). Transactional guarantees consolidated
  for composite commits (§6).
- `event.py` adds `RegisteredEvent = Annotated[... | ..., Field(
  discriminator="type")]` with one model per registered type, a
  `RegisteredEventAdapter`, and a `REGISTERED_EVENT_TYPES` frozenset
  for the test harness to dispatch on. Each per-type payload is its
  own private `BaseModel` (`_TaskCreatedData`, `_ProposalAndTaskData`,
  …). Optional fields still follow the `NotNone` discipline from
  Phase 3; but registered payloads here are all-required at v0.
- Round-trip and schema-parity tests now dispatch through the typed
  registered-event path when applicable, so a regression that
  dropped per-type payload enforcement would fail CI.

## Review focus

Please prioritize substantive review in this order:

1. **Protocol coherence.** Do the three chapters agree with
   `04-task-protocol.md` and `03-roles.md` on every state
   transition? Is the event registry complete — can a subscriber
   reconstruct every protocol-owned state change from the log
   alone?
2. **Composite-commit correctness.** §2.2 of 05-event-protocol
   enumerates multi-entity transitions that MUST commit atomically.
   Does the list match the lifecycles in the prior chapters? Any
   composite I missed?
3. **Schema ↔ model parity.** The `event.schema.json` `if/then`
   dispatch and the Pydantic discriminated union encode the same
   registry. Have I kept them in lockstep (required fields,
   enum values for `reason`/`cause`, `commit_sha` pattern)?
4. **Integrator invariants.** Squash rule, commit-message format,
   and eval-manifest shape are new normative content — any
   ambiguity or underspecification?
5. **Storage contract tightness.** Does §6 (transactional
   guarantees) correctly reference §1/§2 operations? Is the
   at-least-once + replayability floor stated clearly enough that
   a reimplementation can't accidentally weaken it?
