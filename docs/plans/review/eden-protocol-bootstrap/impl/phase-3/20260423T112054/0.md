## Change Summary

- Added a uv workspace rooted at the repo top, with
  `reference/packages/eden-contracts` as the first (and currently only)
  member. Workspace members are listed explicitly to avoid picking up
  the placeholder directories under `reference/packages/*`.
- Hand-wrote Pydantic v2 models that mirror each JSON Schema under
  `spec/v0/schemas/`. Constraints that JSON Schema expresses via
  `if/then/else` (Task claim-presence, per-kind payload) are encoded as
  discriminated unions and `@model_validator(mode="after")` hooks.
  Reserved-metric-name and minProperties constraints on `MetricsSchema`
  are encoded via a `RootModel` validator.
- The accept/reject test corpus is the single source of truth for
  parity: the same cases drive per-model behavioral tests and the
  schema-parity test. Any asymmetry between model and schema over the
  corpus fails the parity job.
- Added a round-trip test: every accept fixture is dumped with
  `mode="json", exclude_none=True` and re-validated against the JSON
  Schema. This catches the one known asymmetry between models and
  schemas: Pydantic accepts explicit `null` for optional fields, JSON
  Schema does not. Documented in the package README.
- Four new CI jobs land alongside the existing `docs-lint` and
  `schema-validity`: `python-lint` (ruff), `python-typecheck` (pyright),
  `python-test` (full pytest), `schema-parity` (runs the parity and
  round-trip tests specifically, for branch-protection visibility).
- Local verification is green: `uv run ruff check .`, `uv run pyright`,
  `uv run pytest -q` (62 passed), markdownlint, and
  `check-jsonschema --check-metaschema`.

## Notable design choices to scrutinize

- **Hand-authored models, not codegen.** `datamodel-code-generator`
  would produce models from schemas but handles
  `allOf/if/then/else` awkwardly; hand-authored keeps discriminated
  unions and cross-field validators clean. Parity is asserted by test
  fixtures rather than by byte-identical schema regeneration.
- **One-way parity.** The parity test asserts model and schema agree
  on every corpus case. It does not claim semantic equivalence across
  all possible inputs — only that the corpus (which aims to cover every
  schema constraint) agrees. The round-trip test catches the
  absent-vs-null asymmetry.
- **`extra="allow"` everywhere.** JSON Schema does not set
  `additionalProperties: false`, so any extra field is schema-legal.
  Models preserve them on round-trip. Consequence: unknown fields are
  not flagged — a drawback the reference impl could tighten later per
  its own policy.
- **Workspace `members` is explicit, not a glob.** A glob over
  `reference/packages/*` picks up the Phase 0 `.gitkeep`-only
  placeholder directories and fails `uv sync`. Switching to explicit
  members means each future package must be added by name when it
  lands. Alternative: scaffold a minimal `pyproject.toml` in each
  placeholder. I opted for the simpler list; trade-off is a one-line
  edit per package rather than silent inclusion.
- **Pyright is `standard`, not `strict`.** Matches direvo. Strict with
  Pydantic is noisy; we can tighten later once we see concrete types
  flowing through the reference impl.
