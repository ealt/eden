**Findings**

1. High: the proposed CI gate will skip exactly the kinds of non-doc changes that can stale the docs. The plan says the new job should gate on `docs || python` and treats that as covering the script surface ([issue-184-doc-instructions-smoke-tested.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-184-doc-instructions-smoke-tested/docs/plans/issue-184-doc-instructions-smoke-tested.md:327), [issue-184-doc-instructions-smoke-tested.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-184-doc-instructions-smoke-tested/docs/plans/issue-184-doc-instructions-smoke-tested.md:392)), but the workflow classifies `reference/scripts/**` and `reference/compose/**` under `compose`, not `python` ([ci.yml](/Users/ericalt/Documents/eden-worktrees/plan-issue-184-doc-instructions-smoke-tested/.github/workflows/ci.yml:94), [ci.yml](/Users/ericalt/Documents/eden-worktrees/plan-issue-184-doc-instructions-smoke-tested/.github/workflows/ci.yml:103)). A change to `setup-experiment.sh` or a compose command can drift the docs while this job stays skipped. The gate should include `compose`, not just `docs` and `python`.

2. High: the lite validation mechanism does not cover the flag surfaces the plan says it will cover. The plan explicitly says lite should catch documented flags like `--seed-from` and `--exec-mode` ([issue-184-doc-instructions-smoke-tested.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-184-doc-instructions-smoke-tested/docs/plans/issue-184-doc-instructions-smoke-tested.md:101)), but the only concrete L1 validator described is parsing `eden-manual` subcommands/flags ([issue-184-doc-instructions-smoke-tested.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-184-doc-instructions-smoke-tested/docs/plans/issue-184-doc-instructions-smoke-tested.md:238)). Those flags are documented on `setup-experiment.sh`, not `eden-manual` ([user-guide.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-184-doc-instructions-smoke-tested/docs/user-guide.md:116), [user-guide.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-184-doc-instructions-smoke-tested/docs/user-guide.md:145)). The plan needs either separate validators per command family or a narrower claim about what lite actually checks.

3. Medium: the dependency model for the standalone gate is still unresolved. The plan wants `python3 scripts/check-docs-commands.py --strict` locally and in CI, and explicitly prefers a lightweight job without `uv sync` ([issue-184-doc-instructions-smoke-tested.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-184-doc-instructions-smoke-tested/docs/plans/issue-184-doc-instructions-smoke-tested.md:398), [issue-184-doc-instructions-smoke-tested.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-184-doc-instructions-smoke-tested/docs/plans/issue-184-doc-instructions-smoke-tested.md:460)). But it also promises YAML/schema validation via the same `check-jsonschema` flow as `schema-validity` ([issue-184-doc-instructions-smoke-tested.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-184-doc-instructions-smoke-tested/docs/plans/issue-184-doc-instructions-smoke-tested.md:245)). Today that tooling exists either in the uv dev environment ([pyproject.toml](/Users/ericalt/Documents/eden-worktrees/plan-issue-184-doc-instructions-smoke-tested/pyproject.toml:53)) or via an explicit CI install step ([ci.yml](/Users/ericalt/Documents/eden-worktrees/plan-issue-184-doc-instructions-smoke-tested/.github/workflows/ci.yml:237)). The plan should say which one it is adopting; otherwise the main command is not actually runnable as written.

**Level assessment**

Missing context: strong. The plan is clear about scope, clearly separates protocol/reference/conformance concerns, and does a good job explaining why naive “run every bash block” does not work here.

Feasibility: not quite there yet. The lite/heavy split is sensible, but the implementation contract still has the three gaps above: wrong CI trigger surface, incomplete command-surface validation, and an unresolved dependency story for schema/YAML checks.

I stopped at feasibility because those are material enough that alternatives/completeness/risk analysis would mostly be downstream of fixing them.

**Overall assessment**

Directionally, this is a good plan: the problem is real, the lite/heavy split is sensible, and the coverage-visibility discipline is the right instinct. But it is not implementation-ready yet. Tighten the trigger surface, make the L1 validators match the actual documented command families, and pin the dependency/runtime model for the standalone checker before treating this as the execution contract.
