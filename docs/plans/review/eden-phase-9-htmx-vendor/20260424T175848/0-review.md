**Findings**

1. Medium: the HTMX `add_row` path breaks the no-JS contract on redirect/error branches, because it returns ordinary `303` redirects even for `HX-Request: true`. The button is wired for fragment swaps in [planner_claim.html](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/templates/planner_claim.html:47), and [add_row](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/planner.py:154) returns `303` for missing session/claim at lines [155-164](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/planner.py:155). Per the official [htmx docs](https://htmx.org/docs/) and [HX-Redirect docs](https://htmx.org/headers/hx-redirect/), 3xx response headers are not processed by htmx; the browser follows the redirect and htmx receives the redirected HTML. I reproduced that: an HTMX POST without a claim ended at `/planner/?banner=claim+missing+from+session` and returned a full document (`<html>`, `<header>`), which htmx would then swap into `#proposal-rows`. The no-JS path navigates away; the HTMX path corrupts the draft DOM instead. The new tests in [test_planner_flow.py](/Users/ericalt/Documents/eden/reference/services/web-ui/tests/test_planner_flow.py:148) only cover the happy-path fragment and miss this branch. This needs an htmx-aware redirect/error response plus tests for missing-claim and signed-out HTMX requests.

The main tech-stack text says the decisive chunk-1 use case is “add another proposal row” at [docs/plans/eden-phase-9-web-ui.md:66](/Users/ericalt/Documents/eden/docs/plans/eden-phase-9-web-ui.md:66), but the alternatives section still says HTMX was picked for the “planner-list → claim-form transition” at [docs/plans/eden-phase-9-web-ui.md:159](/Users/ericalt/Documents/eden/docs/plans/eden-phase-9-web-ui.md:159).
