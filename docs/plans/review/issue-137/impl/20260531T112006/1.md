## Change Summary

- **Sort/filter/group state in URL query params** (decision §2.2): `parse_list_view` allow-lists `sort` ∈ {priority,slug}, `dir` ∈ {asc,desc}, `eligible`, `target` ∈ {all,targeted,untargeted}, `group`; `build_list_links` builds every href from validated values only (no `?sort=` reflection — §D.2).
- **Sort** (§D.2): priority-DESC default, `created_at` ASC always-on secondary tiebreak (two-pass stable sort). Degraded rows (idea/variant unreadable) partition to the bottom under both directions — no sentinel.
- **EligibilityResolver** (§D.4): registration resolved once per render via `read_worker`; group membership memoized by `group_id`. 3-outcome discipline: registered / `NotFound` (definitively ineligible) / transport-indeterminate (unknown). Unknown-eligibility rows stay visible (disabled) even under the eligible-only filter so an outage never hides claimable work. Two distinct warning counters (content-read vs eligibility-resolution).
- **Claim button** (§D.5): live form for eligible rows; disabled + tooltip for definitively-ineligible (filter OFF) and unknown-eligibility rows. Server-side prong already satisfied by #134; added a forged-POST regression (303 banner-redirect, not 500).
- **Expansion** (§D.3): click-to-expand `<details>` context-links row replaces the inline preview — task/creator/idea/variant admin links + artifact-shape-aware "view content" links (bundle → per-entry, single file → one link, http(s) → external, opaque → text).
- **Symmetry** (§9): both list templates are thin wrappers over a single `_pending_list.html` macro partial so executor/evaluator can't drift.

## Round 1 — addressed codex round-0 feedback

**1. Bug (parse_list_view honored `dir` without a valid `sort`) — FIXED.**
`parse_list_view` now only honors `dir` when `sort` is an explicit valid
key; otherwise `direction = _default_direction(sort)`. `?dir=asc` and
`?sort=garbage&dir=asc` now yield the §D.2 default (priority DESC).
Tests: `TestListViewParsing` (3 direct unit cases) +
`TestSort::test_dir_without_valid_sort_keeps_priority_desc` (route-level).

**2. Risk (group memoization skipped the transport-unknown case) — FIXED.**
`EligibilityResolver._group_cache` now stores the full tri-state
`(eligible, unknown)` outcome, including transport failures, so N rows
against the same group cost one DAG walk + one warning increment.
Test: `TestEligibilityResolverMemo::test_failing_group_probe_walks_once`
asserts `store.calls == 1` and `unresolved_count == 1` across two probes.

**3. Risk (tooltip inferred reason from `row.target` presence) — FIXED.**
New `disabled_claim_reason(resolver, *, eligible, unknown)` classifies the
registration-first ladder outcome (`"unregistered"` vs `"target"`); both
row-builders attach `eligibility_reason`, and `_claim_cell` selects the
tooltip from it. An unregistered worker viewing a worker-targeted task now
sees "You are not registered for this experiment.", not "you are not in its
target." Test:
`TestRegistrationLadder::test_unregistered_targeted_row_tooltip_says_not_registered`.

**4. Risk (§D.1/§D.3 deviation: parent-commit text + main-row affordance) —
parent-commit text FIXED; expand-affordance kept as the no-JS two-row
`<details>`.**
- The expansion now renders the parent-commit SHAs as cross-reference text
  for both roles (`row.parent_commits`: `idea.parent_commits` for the
  executor, `variant.parent_commits` for the evaluator), abbreviated to 12
  chars, with an inline note that linking them to a repo browser is deferred
  to #250 (per §D.3's deliberate deferral).
- On the "dedicated main-row expand affordance": §D.3 specifies "toggles a
  `<details>` block spanning the row (same primitive the current preview
  uses — repurposed)" AND "No JS required for the baseline." A toggle living
  in a main-row column that controls a *row-spanning* block is not
  expressible in native HTML without JS (a `<details>`'s `<summary>` and its
  content must be siblings inside the same element). The implemented
  two-row pattern — a full-width `<tr class="expand-row"><td colspan="5">`
  containing the `<details>` whose `<summary>` carries the native disclosure
  triangle (▸/▾) — is the faithful no-JS realization: the affordance is the
  native triangle, the block spans the row, and multiple rows can be open at
  once. This is an implementation-detail choice fully inside §D.3's stated
  constraints, not a scope change; flagging here rather than re-litigating.
