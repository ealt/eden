**Edge Cases And Risks**
- `POST /logout` currently just deletes the browser cookie [auth.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-140-operator-identity-worker/reference/services/web-ui/src/eden_web_ui/routes/auth.py:45). Once the cookie carries a bearer-equivalent `registration_token`, logout no longer revokes the server-side credential. A copied/stolen cookie remains usable until a later reissue. The plan should either accept that explicitly or add a revocation step.
- `selected_experiment_id` already exists in session state [sessions.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-140-operator-identity-worker/reference/services/web-ui/src/eden_web_ui/sessions.py:28), but the current experiment-selection route explicitly says it does not swap per-route store binding [admin_experiments.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-140-operator-identity-worker/reference/services/web-ui/src/eden_web_ui/routes/admin_experiments.py:257). A request-scoped session client helper should state whether it remains default-experiment-only or starts honoring selected experiment; otherwise this plan risks dragging in an unplanned multi-experiment semantic change.
- The migration note around checkpoints is too loose. The real implementation risk is not simply “`web-ui-1` no longer exists”; it is that #128 changes worker registration semantics and #140 changes who bears credentials. The plan should describe the actual failure mode operators will hit, not a generic “non-restorable” conclusion.
- Because the test suite is heavily in-process and admin-seeded [tests/conftest.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-140-operator-identity-worker/reference/services/web-ui/tests/conftest.py:43), auth bugs specific to real `StoreClient` bearer handling are easy to miss. Wave 2 needs explicit real-client coverage, not just the current `InMemoryStore` matrix.

**Overall Assessment**
The direction is sound, and the CLI self-heal is feasible in the reference stack because the CLI already carries the admin token. The plan is not ready as an implementation contract yet. The biggest gaps are: it understates the breadth of the `app.state.store` and `app.state.worker_id` sweep, it does not make post-#128 bootstrap mechanically workable, and it assumes a request-scoped `StoreClient` model without acknowledging the resulting app/test-harness redesign. I would revise the plan before implementation: make the bootstrap capture minted ids explicitly, reconcile the `web-ui-1`/`admin_store` story with #143 and the current admin routes, inventory every session rewrite and `app.state.worker_id` site, and state the admin-token dependency and limits of CLI recovery plainly.
