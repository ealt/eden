## Change summary

- New `eden-git` workspace package scaffolded under `reference/packages/eden-git/` and wired into the uv workspace, pyright include, pytest testpaths, and dev dependency group.
- `GitRepo` class wraps a repository path with roughly 25 methods split across factories, introspection, plumbing, and branch / worktree management. All operations shell out to the local `git` binary via `subprocess.run`; failures raise `GitError` carrying the full argv plus captured streams.
- Per-invocation hardening: `-c safe.directory=<repo>` (no wildcard), `-c commit.gpgsign=false` (prevents ambient signing config from breaking integrator commits), and `commit_tree` sets `GIT_AUTHOR_*` / `GIT_COMMITTER_*` env vars explicitly so `user.name` / `user.email` from the user's global config is never consulted.
- `write_tree_with_file` — the main non-trivial primitive — builds a new tree equal to `base_tree_sha` with exactly one file added, using an isolated `GIT_INDEX_FILE` in a tempdir so no index state escapes. Rejects if the target path already exists in the base tree (the §3.2 "worker branch MUST NOT contain the eval-manifest path" guard that 7b will invoke).
- `create_ref` calls `update-ref name sha <zero>` to atomically create a ref that must not currently exist, supporting the §1.2 `trial/*` immutability invariant.
- 36 tests run against a real `git` subprocess on a `tmp_path` repo, covering every public method and the two explicit reject paths (pre-existing path in `write_tree_with_file`, pre-existing ref in `create_ref`).

## Focus areas for this review

1. **Spec-literal audit.** The contract-bearing methods (`write_tree_with_file`, `create_ref`, `update_ref`, `commit_tree`) will be invoked by 7b to satisfy §3.2 / §3.4. Are they shaped such that a 7b implementation could build the squash and publish it atomically without reaching for private state? Specifically: is there anything §3.2 demands that 7a can't express today?
2. **Environment isolation.** `commit.gpgsign=false`, explicit identity env vars, `safe.directory=<path>`. Is there any other ambient git config or env var that could leak into integrator behavior (e.g., `core.autocrlf`, `core.hooksPath`, `init.templateDir`, `GIT_CONFIG_GLOBAL`, `GIT_SSH_COMMAND`)?
3. **`GIT_INDEX_FILE` scoping.** `write_tree_with_file` isolates the index via a tempdir. Is there any case where a `git read-tree` / `update-index` / `write-tree` sequence would still mutate repo state or leak state to other concurrent invocations on the same repo?
4. **Zero-OID hard-coding.** `_ZERO_SHA = "0" * 40` assumes SHA-1. The package README notes this as a known limitation. Is there a cheap way to make this hash-algo-aware now (e.g., query `extensions.objectFormat`) or is deferring to Phase 13 fine?
5. **Test coverage gaps.** 36 tests run green; are there obvious edge cases that aren't covered yet and would bite 7b hard? Examples that come to mind: a worker branch with a directory entry (tree, not blob) at the eval-manifest path; a symlink (`120000`) in the base tree; `update-ref` CAS against a ref that simultaneously moved; orphan / unborn-HEAD repos.

## Review status

### Round 0 — findings addressed

- **Bug** — `_env()` leaking ambient git vars (`GIT_DIR`, `GIT_INDEX_FILE`, …). Fixed by introducing a module-level `_sanitized_git_env()` that strips every entry in a `_GIT_ENV_VARS_TO_STRIP` frozenset (the redirect-capable vars plus the committer-identity env vars, which `commit_tree` sets back explicitly per invocation) and also scrubs any `GIT_CONFIG_KEY_*` / `GIT_CONFIG_VALUE_*` pairs. Every child env — instance `_env()`, factory `_run_git()` — now routes through it. Regression tests in `test_environment_isolation.py` verify that ambient `GIT_DIR`, `GIT_INDEX_FILE`, and `GIT_COMMITTER_NAME` / `GIT_COMMITTER_EMAIL` cannot steer the wrapper.
- **Risk** — `init()` / `init_bare()` inheriting system/global git config and user templates. Fixed by (a) routing init through the same `_sanitized_git_env` (which pins `GIT_CONFIG_NOSYSTEM=1` and `GIT_CONFIG_GLOBAL=/dev/null`) and (b) adding `--template=` on the init command line to disable template-dir copying. Regression tests verify that a poisoned `GIT_CONFIG_GLOBAL` and a poisoned `GIT_TEMPLATE_DIR` cannot leak into a fresh repo.
- **Risk** — `_ZERO_SHA = "0" * 40` hard-coding SHA-1. Fixed by replacing the module constant with a cached instance method `zero_oid()` that queries `git rev-parse --show-object-format` once and returns 40 zeros for SHA-1 / 64 zeros for SHA-256. `create_ref` now calls `self.zero_oid()`. SHA-256 regression test skips cleanly when the local git build does not include SHA-256 support.
- **Nit** — README/module docstring drift (`write_tree`, `create_commit`). Fixed: README now enumerates the actual public surface (`write_tree_from_entries`, `write_tree_with_file`, `commit_tree`, `empty_tree_sha`, `zero_oid`, …) and calls out the per-invocation environment sanitization. Module docstring in `repo.py` now references `commit_tree` (not `create_commit`) and lists the full plumbing set.

### Additional coverage added alongside round-0 fixes

- Write-tree collision edge cases: target path nested under a pre-existing blob (type conflict), and target path already present as a symlink (mode `120000`). Both reject as expected — the symlink case validates that `tree_entry_exists` catches non-blob existing entries too.

All gates green: 294 tests (48 in `eden-git`), `ruff` clean, `pyright` 0 errors / 0 warnings, markdownlint clean.
