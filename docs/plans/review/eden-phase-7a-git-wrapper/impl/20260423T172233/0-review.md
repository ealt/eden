**Findings**

- Bug — [reference/packages/eden-git/src/eden_git/repo.py](/Users/ericalt/Documents/eden/reference/packages/eden-git/src/eden_git/repo.py:489): `_env()` copies the ambient git environment straight through, so `GitRepo` is not actually path-scoped. In a quick repro, `GIT_DIR=/other/repo git -C <intended-repo> rev-parse --git-dir` still targeted the other repo, which means `GIT_DIR`, `GIT_WORK_TREE`, `GIT_INDEX_FILE`, `GIT_OBJECT_DIRECTORY`, replace refs, etc. can redirect or skew wrapper behavior. The same problem flows through `_run()` and `_run_with_env()` at [repo.py](/Users/ericalt/Documents/eden/reference/packages/eden-git/src/eden_git/repo.py:497). Fix by building a sanitized env for every git subprocess: remove the repo-local vars listed by `git rev-parse --local-env-vars`, force `GIT_TERMINAL_PROMPT=0` unconditionally, and only add back the specific vars this wrapper intentionally controls (`GIT_INDEX_FILE`, `GIT_AUTHOR_*`, `GIT_COMMITTER_*`).

- Risk — [reference/packages/eden-git/src/eden_git/repo.py](/Users/ericalt/Documents/eden/reference/packages/eden-git/src/eden_git/repo.py:82) and [reference/packages/eden-git/src/eden_git/repo.py](/Users/ericalt/Documents/eden/reference/packages/eden-git/src/eden_git/repo.py:530): `init()` / `init_bare()` also inherit ambient config and templates. I reproduced `init.templateDir` injecting a hook into a newly created repo, so fixture repos and future callers are not deterministic even before `GitRepo` starts operating on them. That undercuts the “isolated reference wrapper” goal and can make tests depend on user machine state. Fix by running init under the same sanitized env and disabling external config/template loading (`GIT_CONFIG_NOSYSTEM=1`, `GIT_CONFIG_GLOBAL=/dev/null`, and `--template=` or equivalent).

- Risk — [reference/packages/eden-git/src/eden_git/repo.py](/Users/ericalt/Documents/eden/reference/packages/eden-git/src/eden_git/repo.py:36) and [reference/packages/eden-git/src/eden_git/repo.py](/Users/ericalt/Documents/eden/reference/packages/eden-git/src/eden_git/repo.py:364): `create_ref()` hard-codes a 40-zero old OID. That fails immediately in SHA-256 repos; `git update-ref ... 0000...` errors with “not a valid old SHA1”, so 7b cannot enforce the `trial/*` immutability invariant there. This is cheap to fix now: derive the zero OID length from the repo’s object format or from `empty_tree_sha()`, then use that value instead of `_ZERO_SHA`. Add a SHA-256 regression test.

- Nit — [reference/packages/eden-git/README.md](/Users/ericalt/Documents/eden/reference/packages/eden-git/README.md:23), [reference/packages/eden-git/src/eden_git/repo.py](/Users/ericalt/Documents/eden/reference/packages/eden-git/src/eden_git/repo.py:15), and [reference/packages/eden-git/src/eden_git/__init__.py](/Users/ericalt/Documents/eden/reference/packages/eden-git/src/eden_git/__init__.py:5): the docs drift from the actual API. They mention `write_tree` / `create_commit`, but the shipped surface is `write_tree_from_entries`, `write_tree_with_file`, and `commit_tree`. Fix the docs so Phase 7b consumers do not have to reconcile README/docstrings against the source.
