# EDEN Phase 10e â€” End-to-end Compose integration test

## Context

Phase 10b/10c shipped `compose-smoke` (scripted-mode 3-trial loop,
headless-host only). Phase 10d shipped `compose-smoke-subprocess`
(subprocess-mode same shape). Both verify that the orchestrator + the
three headless worker hosts can drive an experiment to quiescence; both
ignore the web UI surface, the admin-reclaim path, and the stack's
termination behavior.

Phase 10e closes that gap. The exit criterion in [`docs/roadmap.md`](../roadmap.md)
is "End-to-end Compose integration test exercising the full loop
including Web UI, admin-reclaim, and termination." This chunk delivers
exactly that, in one new `compose-e2e` CI job.

## Reframe

The value of `compose-e2e` is the **stack shape**, not duplicate role
coverage. The chunk-9b / 9c / 9d real-subprocess pytest tests in
[`reference/services/web-ui/tests/test_*_e2e.py`](../../reference/services/web-ui/tests/)
already exercise each role's UI flow against a forked task-store-server
+ web-ui pair on an ephemeral port. What those tests can NOT prove is
that the same flows work when the web-ui is talking to a
`task-store-server` running in *another container*, sharing volumes
with the *real* orchestrator and *real* worker hosts, behind compose's
healthcheck and restart-policy machinery, on the published
`${WEB_UI_HOST_PORT}` host port.

So this chunk picks **one** representative role flow (planner â€” the
simplest, no repo writes, no metrics-schema validation) and drives it
end-to-end against the live stack. Implementer and evaluator UI flows
are out of scope: their pytest e2e tests give equivalent role-contract
signal, and adding two more compose-bound walkthroughs multiplies
fragility (extra container starts/stops, extra orchestration races)
without adding stack-shape signal beyond the planner walkthrough.

## Roadmap delta

- Container isolation of the `*_command` subprocess (docker-in-docker,
  experiment-specific image build, auth-secret mounting): still
  deferred to a 10d follow-up sub-chunk per the chunk-10d plan.
- Workers integrate with Gitea as their actual git remote: still
  deferred to a 10d follow-up sub-chunk.
- Branch protection is **not** updated to require `compose-e2e` in
  this chunk â€” same posture 10c / 10d took for newly-added compose
  jobs (let it run cleanly for a few iterations on `main` first).

## Scope

### In scope

1. **Web UI planner walkthrough.** Sign in to the web UI on the
   published host port, GET `/planner/`, claim one plan task, submit a
   one-proposal draft, verify the task moved to `submitted` and the
   resulting trial integrates downstream.
2. **Admin-reclaim drill.** Claim a *separate* plan task via the web
   UI without submitting (so it sits in `claimed` state with a live
   token), POST `/admin/tasks/<id>/reclaim`, verify the redirect to
   `/admin/tasks/<id>/?reclaimed=ok`, and verify the event log shows a
   `task.reclaimed` event with `cause=operator`.
3. **Termination drill.** `docker compose stop --timeout 10` against
   the still-up stack, then verify every container in `compose ps -a`
   exited cleanly: status=`exited`, exit code != 137 (the SIGKILL
   marker â€” anything else, including 0 from a graceful handler and
   143 from a default-handler SIGTERM, is acceptable).
4. **End-state assertions.** Same final-state shape as compose-smoke,
   adjusted for `EDEN_PLAN_TASKS=4`: `>= 4 trial.integrated`,
   `>= 12 task.completed`, `>= 4` plan-task `task.completed`, plus
   `>= 1 task.reclaimed` with `cause=operator`.
5. **CI wiring.** New `compose-e2e` job mirroring the existing
   `compose-smoke` and `compose-smoke-subprocess` jobs.

### Out of scope

- Subprocess-mode coverage. `compose-smoke-subprocess` already covers
  that path; layering UI walkthroughs on top is duplicative.
- Implementer / evaluator UI walkthroughs. Their chunk-9c / 9d
  pytest e2e tests give equivalent role-contract signal.
- Container isolation of the subprocess (10d follow-up).
- Gitea as the workers' actual git remote (10d follow-up).
- Branch-protection requirement for the new job (deferred follow-up).
- Multi-experiment / control-plane (Phase 12).
- Kubernetes deployment (Phase 13).

## Design

### A. Test layout

Two new files under `reference/compose/healthcheck/`:

- **`e2e.sh`** â€” bash wrapper. Owns: tooling preflight, setup-experiment
  invocation, staged `compose up -d`, dispatch to the python driver,
  post-run event-log assertions via `jq`, termination drill, teardown.
- **`e2e_drive.py`** â€” python driver. Owns: web UI sign-in, CSRF
  extraction, planner-walkthrough claim/submit, admin-reclaim POST,
  exit non-zero with diagnostic on assertion failure.

The split is deliberate: bash is the natural shape for `compose` /
`docker inspect` / `jq` plumbing; python is the natural shape for
cookie-aware HTTP and form-CSRF round-tripping. Mirroring the existing
smoke shells keeps the CI job shape uniform.

### B. Bring-up sequence â€” the race-against-planner-host fix

**Codex round-0 finding (blocker, addressed):** the scripted planner-host
drains pending plan tasks in a tight loop on startup
([`workers.py`](../../reference/packages/eden-dispatch/src/eden_dispatch/workers.py)).
If the stack starts with planner-host running and the orchestrator
seeds 4 plan tasks via `--plan-tasks 4`, the planner-host can claim
all 4 before the python driver finishes signing in. So a "stop
planner-host after up" pattern is fundamentally racy â€” the bad race
is during initial startup, not steady state.

**Fix:** start the stack *without* planner-host, run the UI
walkthroughs against the seeded-but-unclaimed plan tasks while no
planner-host is competing, then bring planner-host online:

```bash
# Stage 1: bring up everything we actually USE in the drill, minus
# planner-host. We deliberately omit gitea: it has a 30s healthcheck
# start_period and is not consumed by 10e (Gitea-as-remote is the
# 10d follow-up sub-chunk, deferred). The orchestrator has no
# healthcheck and exits after 30 Ă— 1s of no progress, so we must NOT
# wait on a 30s+ healthcheck during stage-1 --wait, or the
# orchestrator can quiesce and exit 0 mid-wait â€” `compose up --wait`
# treats an exited service during the wait window as failure
# (codex round-1 finding).
#
# Also omit planner-host (the whole reason for staging).
#
# What we DO wait on: postgres (5s start_period), task-store-server
# (10s), web-ui (15s). Plus blob-init completes successfully in ~2s.
# Worst-case wait is ~40s â€” well under 30s of orchestrator quiescence
# from when it actually starts polling, because the orchestrator only
# starts after task-store-server is healthy.
docker compose -f compose.yaml --env-file "$ENV_FILE" up -d --wait \
    --wait-timeout 120 \
    postgres blob-init \
    task-store-server orchestrator \
    implementer-host evaluator-host web-ui

# Sanity: the orchestrator must still be running by the time --wait
# returns. If it exited early (quiesced because we took too long,
# or crashed), the drill cannot proceed.
status="$(docker inspect --format '{{.State.Status}}' eden-orchestrator)"
test "$status" = "running" || {
    echo "orchestrator already exited at end of stage 1 (status=$status)" >&2
    docker compose -f compose.yaml --env-file "$ENV_FILE" \
        logs --tail 60 orchestrator >&2
    exit 1
}

# orchestrator seeded plan-0001..plan-0004 at startup. None claimed.
# Run python driver â€” claims plan-0001 (submit) and plan-0002
# (admin-reclaim drill). The claim events count as "progress" and
# reset the orchestrator's quiescence counter, so it stays running.

python3 healthcheck/e2e_drive.py

# Stage 2: bring planner-host online. It now picks up plan-0002
# (returned to pending by admin-reclaim), plan-0003, and plan-0004.
# Gitea is still down â€” not started, not used.
docker compose -f compose.yaml --env-file "$ENV_FILE" up -d \
    --wait --wait-timeout 60 planner-host
```

**Note on `EDEN_PLAN_TASKS=4`:** the orchestrator's `--plan-tasks` flag
defaults to 3 in the fixture. We override to 4 in `.env` so the four
deterministic IDs `plan-0001..plan-0004` are seeded. The python driver
uses two specific IDs (`plan-0001` for the submit walkthrough,
`plan-0002` for the admin-reclaim drill); the other two go through the
headless planner-host on the success path.

```bash
# After setup-experiment.sh writes .env, override:
sed -i.bak 's/^EDEN_PLAN_TASKS=.*/EDEN_PLAN_TASKS=4/' "$ENV_FILE"
rm -f "${ENV_FILE}.bak"
```

The `sed -i.bak` form is GNU/BSD-portable (BSD `sed -i ''` rejects
the bare `-i`; GNU `sed -i.bak` accepts it identically).

**Confirmed plan-task ID shape** ([`orchestrator/cli.py:116`](../../reference/services/orchestrator/src/eden_orchestrator/cli.py)):
numeric `--plan-tasks N` expands to `plan-0001..plan-{N:04d}`. So
with `N=4` we have exactly `plan-0001`, `plan-0002`, `plan-0003`,
`plan-0004`.

### E. Event-log read-back

`task-store-server` is internal-only â€” no published host port. The
python driver cannot read events directly; the post-walkthrough
`task.reclaimed` assertion runs from `e2e.sh` via `compose exec`,
exactly the way `smoke.sh` already does:

```bash
EVENTS_JSON="$(
    docker compose -f compose.yaml --env-file "$ENV_FILE" \
        exec -T task-store-server \
        curl -fsS \
            -H "Authorization: Bearer ${EDEN_SHARED_TOKEN}" \
            -H "X-Eden-Experiment-Id: ${EXPERIMENT_ID}" \
            "http://localhost:8080/v0/experiments/${EXPERIMENT_ID}/events"
)"

RECLAIMED_OPERATOR="$(
    echo "$EVENTS_JSON" \
        | jq '(.events // .) | [.[] | select(
              .type == "task.reclaimed"
              and .data.cause == "operator"
            )] | length'
)"
test "$RECLAIMED_OPERATOR" -ge 1
```

### H. Termination drill â€” full container set

**Codex round-0 finding (medium, addressed):** the original drill
narrowed to `eden-*` containers, missing `eden-postgres`, `eden-gitea`,
and `eden-blob-init`. Also: `eden-repo-init` ran via
`compose run --rm --no-deps` and is gone by termination time
([`setup-experiment.sh:251`](../../reference/scripts/setup-experiment/setup-experiment.sh)).

The drill operates over the actual `compose ps -a` output:

```bash
echo "--- termination drill: stop with 10s budget ---"
docker compose -f compose.yaml --env-file "$ENV_FILE" stop --timeout 10

# Iterate every container compose still knows about, regardless of
# name pattern. This naturally covers postgres, gitea, blob-init,
# task-store-server, orchestrator, planner-host, implementer-host,
# evaluator-host, and web-ui. eden-repo-init is gone (--rm) so it
# does not appear.
for name in $(
    docker compose -f compose.yaml --env-file "$ENV_FILE" ps -a \
        --format '{{.Name}}'
); do
    status="$(docker inspect --format '{{.State.Status}}' "$name")"
    code="$(docker inspect --format '{{.State.ExitCode}}' "$name")"
    test "$status" = "exited" || {
        echo "$name still in status $status after stop --timeout 10" >&2
        exit 1
    }
    test "$code" != "137" || {
        echo "$name was SIGKILLed (exit code 137)" >&2
        docker compose -f compose.yaml --env-file "$ENV_FILE" \
            logs --tail 30 "$name" >&2
        exit 1
    }
    # blob-init is a one-shot that already exited 0 during bring-up;
    # this is just a paranoia check, not a status-change.
done
```

**Why `!= 137` instead of `== 0`:**
[`reference/services/_common/`](../../reference/services/_common/)'s
SIGTERM handler exits the EDEN python services cleanly with code 0
(verified by codex against
[`web-ui cli.py:155`](../../reference/services/web-ui/src/eden_web_ui/cli.py),
[`task-store cli.py:171`](../../reference/services/task-store-server/src/eden_task_store_server/cli.py),
[`planner cli.py:109`](../../reference/services/planner/src/eden_planner_host/cli.py),
[`implementer cli.py:85`](../../reference/services/implementer/src/eden_implementer_host/cli.py),
[`evaluator cli.py:90`](../../reference/services/evaluator/src/eden_evaluator_host/cli.py)).
But postgres / gitea use external images we don't control; if a future
upstream tweak changes their default SIGTERM handler to default-handler
(producing 143), `== 0` would falsely fail. `!= 137` captures the only
thing we actually care about: **the container did NOT need SIGKILL**.

### I. Teardown

`e2e.sh`'s `cleanup()` trap runs `docker compose down -v` on EXIT,
exactly as smoke.sh does. The termination drill's `compose stop` does
NOT call `down -v` itself â€” that would lose the volumes before the
drill verifies anything. `down -v` is the trap's responsibility;
`stop` is the drill's.

### K. Failure-mode handling (revised)

| Failure | Surface | Recovery |
|---|---|---|
| `compose up --wait` (stage 1) times out on a service | bash exits non-zero | trap dumps `compose ps -a` + `compose logs --tail 30` for unhealthy services; teardown |
| Python driver: claim returns no eligible task | driver exits non-zero with diagnostic naming the task IDs it tried | trap dumps `compose logs task-store-server`; teardown |
| Python driver: planner submit returns non-200 | driver exits non-zero with the response status + body excerpt | trap dumps `compose logs web-ui`; teardown |
| Python driver: admin-reclaim CSRF rejected | response is bare 403 (per [`admin.py:317`](../../reference/services/web-ui/src/eden_web_ui/routes/admin.py)); driver exits with diagnostic â€” indicates a CSRF-extraction bug in the driver | shell trap dumps body; teardown |
| Python driver: admin-reclaim 303 to `?error=â€¦` instead of `?reclaimed=ok` | driver exits with the error-banner code | trap dumps `compose logs web-ui`; teardown |
| `compose up planner-host` (stage 2) fails healthcheck | planner-host has no healthcheck so this is just `compose up -d` succeeding when the container is started; no race | not applicable |
| Orchestrator doesn't quiesce in 240s | bash exits non-zero | trap dumps `compose logs --tail 60 orchestrator planner-host implementer-host evaluator-host`; teardown |
| End-state assertion fails (counts low) | bash exits non-zero with the failed comparison | trap dumps `compose logs --tail 60 orchestrator`; teardown |
| Container SIGKILLed during stop (exit code 137) | bash exits non-zero in the termination loop | trap dumps `compose logs --tail 30 <name>` for the offending container; teardown |

The **TTL-expiry concern** flagged in round-0 is moot: the web-ui's
`--claim-ttl-seconds` default is **3600** (1 hour) per
[`reference/services/web-ui/src/eden_web_ui/cli.py:53`](../../reference/services/web-ui/src/eden_web_ui/cli.py),
and the admin-reclaim drill runs in <5s after the claim. No new flag
is needed.

### L. Deterministic bring-up choreography

The staged `compose up -d` (stage 1 omits planner-host and gitea;
stage 2 adds planner-host) makes the UI's claims uncontested. The only
component that could touch plan tasks during the drill window is the
expired-claim sweeper that runs inside the orchestrator's poll loop;
it only reclaims claims older than their TTL. The web-ui's
`--claim-ttl-seconds` defaults to 3600 and the drill holds claims for
<5s, so the sweeper is a no-op here.

**Startup race against task seeding:** the orchestrator has no
compose-level healthcheck, so `compose up --wait` for orchestrator is
satisfied as soon as the container is in `running` state â€” which can
happen *before* the orchestrator has finished seeding the 4 plan tasks
into the task store. The python driver MUST therefore poll the task
list at startup rather than assume tasks are present.

The driver's first action is:

```python
deadline = time.monotonic() + 30.0
while time.monotonic() < deadline:
    resp = ui.get("/admin/tasks/?kind=plan")
    if all(f"plan-{i:04d}" in resp.text for i in (1, 2, 3, 4)):
        break
    time.sleep(0.5)
else:
    sys.exit("orchestrator did not seed all 4 plan tasks within 30s")
```

This polls (rather than sleeping a fixed interval) so the drill is
both robust against slow orchestrator startup AND fast on the happy
path. Once the four IDs are visible, the walkthrough proceeds.

**Why not poll task-store-server directly:** that service is
internal-only (no published host port). The web-ui's `/admin/tasks/`
list reads through `StoreClient` and is the public-facing equivalent.

## Lessons-learned hooks (preview for /codify after impl review)

- The "stop after up" pattern doesn't work against tight-loop drainers;
  the right shape is staged-up where the competing service is added
  *after* the test claims its targets. Worth codifying if it survives
  impl review.
- `!= 137` is more robust than `== 0` for SIGKILL guards because
  external images don't always install graceful handlers; codify if it
  survives codex-review on the impl.
- Plan-task IDs are zero-padded 4-digit (`plan-0001`), not bare-int â€”
  doc-rot risk if not codified somewhere search-able.
