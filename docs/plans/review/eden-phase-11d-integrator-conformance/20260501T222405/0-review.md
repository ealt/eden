**Findings**

1. High: the scope deferral is too broad. The plan treats the git-aware parts of chapter 06 as untestable through a wire-only suite ([plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11d-integrator-conformance.md:40>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11d-integrator-conformance.md:156>)), but §2’s “`commit_sha` is set and resolves to a commit on the trial’s `branch`” has a wire-observable consequence: `/integrate` must reject and must not write `trial_commit_sha` or emit `trial.integrated` ([06-integrator.md](</Users/ericalt/Documents/eden/spec/v0/06-integrator.md:41>)). The harness can seed that state through normal endpoints by creating a `starting` trial with arbitrary `branch`/`commit_sha`, driving only the evaluator leg to `success`, then calling `/integrate`. Deferring that case leaves a real chapter-06 MUST untested.

2. High: chapter 06 §5.3 is not actually covered by the proposed group/citation shape. The plan says existing v1 `Integrate idempotency` coverage is enough ([plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11d-integrator-conformance.md:11>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11d-integrator-conformance.md:51>)), but those tests cite chapter 07 §5, not chapter 06 §5.3 ([test_integrate_idempotency.py](</Users/ericalt/Documents/eden/conformance/scenarios/test_integrate_idempotency.py:15>)). If chunk 11d is the chapter-06 addendum, at least one new scenario should cite §5.3, or the chapter-9 index row for `Integrator atomicity` should include §5.3. As written, `divergent-resubmit-leaves-no-second-event` is really a repeat-promotion / no-overwrite assertion, not purely §3.4 atomic-write ([06-integrator.md](</Users/ericalt/Documents/eden/spec/v0/06-integrator.md:185>)).

3. Medium: one proposed atomicity test is weaker than the missing coverage it displaces. `event-implies-field` is close to a restatement of `field-implies-event`, and `failed-integrate-leaves-no-artifacts` is really a §2 precondition rejection case, not the strongest §3.4-specific delta ([plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11d-integrator-conformance.md:102>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-11d-integrator-conformance.md:114>)). I would spend one of those slots on the missing “success trial with bogus branch/commit rejects integrate” scenario instead.

4. Medium: the plan underestimates the reference-IUT impact. The current conformance adapter only spawns `eden_task_store_server` ([adapter.py](</Users/ericalt/Documents/eden/conformance/src/conformance/adapters/reference/adapter.py:1>)), and the reference `integrate_trial` implementation behind that server checks status/idempotency only; it does not perform repo-backed chapter-06 promotion validation ([eden_storage/_base.py](</Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/_base.py:885>)). If 11d adds the missing §2 branch/commit test, the plan needs to acknowledge likely adapter/IUT work instead of assuming “no adapter changes.”

**Overall assessment**

Needs revisions.

**By question**

1. Spec scope correctness: partly right. Deferring direct git-artifact assertions for squash shape, manifest shape, `work/*`, and ref durability is the right shape. Blanket deferral of repo-sensitive promotion preconditions is too aggressive; at least the “`commit_sha` resolves on `branch`” refusal path is black-box testable.

2. Coverage completeness: the plan misses one wire-observable §2 MUST: reject integrate for a `success` trial whose recorded `branch`/`commit_sha` is not a real branch-tip commit. It also treats chapter 06 §5.3 as already covered, but existing coverage is only via chapter 07 citations and does not satisfy chapter-06 traceability.

3. Test design: the cross-artifact consistency framing is good, and reading state after the call returns matches §3.4’s “once a promotion has returned” wording ([06-integrator.md](</Users/ericalt/Documents/eden/spec/v0/06-integrator.md:100>)). I would keep one success-path consistency test, one event-side/cardinality retry test, and shift one of the current duplicates to the missing §2 invalid-branch/commit case.

4. Citation-check discipline: `check_citations.py` will accept a separate v1+roles+integrator table inside chapter 9 §5; it parses all rows until the next `##` ([check_citations.py](</Users/ericalt/Documents/eden/conformance/src/conformance/tools/check_citations.py:49>)). But if you want a test to cite chapter 06 §5.3, the matching chapter-9 row must also call out §5.3, or the group-relevance check will fail ([check_citations.py](</Users/ericalt/Documents/eden/conformance/src/conformance/tools/check_citations.py:150>), [check_citations.py](</Users/ericalt/Documents/eden/conformance/src/conformance/tools/check_citations.py:295>)).

5. Helper extensions: `drive_to_error_trial` and `drive_to_eval_error_trial` are good additions and match the hardened `_seed` pattern. They are not enough for the missing invalid-branch/commit case; that needs either a new helper for “manual starting trial -> evaluator success” or an extension to `drive_to_success_trial`.

6. Inter-chapter restatement drift: the proposed chapter-9 amendment is directionally correct, but I would narrow it. Better resolution: chapter 9 should say the wire-only suite covers wire-observable consequences of chapter 06 §2 / §3.4 / §5.3, while direct inspection of git artifacts remains deferred to a future binding.

7. Other risks/gaps: chapter 9 §5 prose currently says the integrator level adds “its own group” singular ([09-conformance.md](</Users/ericalt/Documents/eden/spec/v0/09-conformance.md:61>)); if 11d adds two groups, that text should be updated too. The metrics-schema-at-promotion case is reasonable to defer for now; I do not see a clean normative-endpoint path to seed a `success` trial with invalid metrics.
