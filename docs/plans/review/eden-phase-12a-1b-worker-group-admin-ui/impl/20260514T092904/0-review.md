**Findings**
- `Bug` [admin_groups.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/services/web-ui/src/eden_web_ui/routes/admin_groups.py:217) treats any member that is not a group as a worker and adds it to the transitive closure, so [admin_group_detail.html](/Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/services/web-ui/src/eden_web_ui/templates/admin_group_detail.html:73) can advertise dangling IDs as “workers that could claim” the group. That is incorrect against store semantics in [eden_storage/_base.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/packages/eden-storage/src/eden_storage/_base.py:1785): an unregistered ID resolves to membership `false`. I verified this with a live TestClient check: `ghost-worker` renders in the closure while `resolve_worker_in_group("ghost-worker", ...)` returns `False`.

- `Bug` Cross-registry namespace collisions are misclassified. `register_worker` can fail definitively with `AlreadyExists` when a group already owns that ID [eden_storage/_base.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/packages/eden-storage/src/eden_storage/_base.py:1563), but [admin_workers.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/services/web-ui/src/eden_web_ui/routes/admin_workers.py:331) does not catch that and redirects to `?error=transport`. The symmetric group path collapses the same worker/group collision into a generic `already-exists` banner at [admin_groups.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/services/web-ui/src/eden_web_ui/routes/admin_groups.py:336). That loses the real cause and makes operator recovery guesswork.

- `Risk` Membership-derived views can silently under-report on transient store/wire failures. [admin_workers.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/services/web-ui/src/eden_web_ui/routes/admin_workers.py:205) swallows exceptions from `resolve_worker_in_group` and just omits the group preview; [admin_groups.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/services/web-ui/src/eden_web_ui/routes/admin_groups.py:227) does the same mid-walk for nested members. Because no banner is set, the page can look authoritative while actually incomplete.

- `Risk` A wrong or stale admin token degrades into opaque “transport” banners instead of a clear auth failure. Startup only validates the worker bearer via [cli.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/services/web-ui/src/eden_web_ui/cli.py:229) and never probes `admin_store` after constructing it at [cli.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/services/web-ui/src/eden_web_ui/cli.py:242). The write handlers then catch `Forbidden`/`Unauthorized` under broad `Exception` arms like [admin_workers.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/services/web-ui/src/eden_web_ui/routes/admin_workers.py:331) and [admin_groups.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/services/web-ui/src/eden_web_ui/routes/admin_groups.py:344), so a misconfigured admin token looks like a retryable transport blip.

- `Nit` Several planned UI details were not implemented. The group list only shows direct-member counts/preview in [admin_groups.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/services/web-ui/src/eden_web_ui/routes/admin_groups.py:269) and [admin_groups.html](/Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/services/web-ui/src/eden_web_ui/templates/admin_groups.html:64), omitting the planned transitive-worker count. Invalid label parsing is reduced to a generic redirect at [admin_workers.py](/Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/services/web-ui/src/eden_web_ui/routes/admin_workers.py:314) instead of the planned inline error naming the offending line.
