**3. Alternatives**
- The two-file split in `§D.1` looks right. I do not see a strong reason to collapse workers and groups back into one file.
- The admin-token-bearer posture is the correct short-term choice because the write endpoints are admin-gated today ([server.py](</Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/packages/eden-wire/src/eden_wire/server.py:641>), [server.py](</Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/packages/eden-wire/src/eden_wire/server.py:747>), [server.py](</Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/packages/eden-wire/src/eden_wire/server.py:796>), [server.py](</Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/packages/eden-wire/src/eden_wire/server.py:832>)). But the plan should say more plainly that, until 12a-2 lands, “sign in to web-ui” effectively means “act with deployment-admin power”.
- The in-response HTML token display is also the right alternative to a 303 with query params. But `§8.2` only says to `"consider issuing a Cache-Control: no-store, no-cache header"` ([plan](</Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/docs/plans/eden-phase-12a-1b-worker-group-admin-ui.md:986>)). That should be a design requirement, not an implementation maybe.

**4. Completeness**
- The GET routes are missing the chunk-9e “transport-shaped read failure” contract. `admin.py` consistently catches non-`StorageNotFound` read failures and returns `_read_failure_response(...)` ([admin.py](</Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/services/web-ui/src/eden_web_ui/routes/admin.py:274>), [admin.py](</Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/reference/services/web-ui/src/eden_web_ui/routes/admin.py:753>)). `§D.4`/`§D.5` mention 404 handling and POST banners, but not what the new GET list/detail pages do on `StoreClient` transport errors.
- The worker-detail scope is not fully reconciled with `§D.8`. `§D.4` promises attributed tasks, ideas, and variants plus “show more” links ([plan](</Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/docs/plans/eden-phase-12a-1b-worker-group-admin-ui.md:312>)), but `§D.8` only adds worker filtering to tasks and variants ([plan](</Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/docs/plans/eden-phase-12a-1b-worker-group-admin-ui.md:518>)). That leaves the ideas branch incomplete.
- `§6.7` still has unresolved test expectations: `"redirects ... (or returns 403 — pick one in impl)"` ([plan](</Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/docs/plans/eden-phase-12a-1b-worker-group-admin-ui.md:876>)). A plan should not defer the route contract to implementation.

**5. Edge Cases and Risks**
- `§8.2` correctly calls out refresh-repost on the token page, but the mitigation is too soft for the non-idempotent reissue flow. If this stays as a 200-render design, require `Cache-Control: no-store` and document the exact browser-refresh behavior.
- The main operational risk is not the token page; it is privilege concentration. `§2` decision 2 acknowledges “every signed-in session is implicitly an operator” ([plan](</Users/ericalt/Documents/eden-worktrees/phase-12a-1b-worker-group-admin-ui/docs/plans/eden-phase-12a-1b-worker-group-admin-ui.md:61>)), but the plan should treat that as a first-class risk, not background context.

Overall, the plan’s scope and top-level architecture are reasonable, and the naming is consistent with the repo’s discipline. But it is not ready to implement as written: there are several factual mismatches against the shipped 12a-1 wire/session behavior, plus an unresolved disabled-mode contract and an incomplete worker-detail story. Fix those first; after that, the chunk looks straightforward.
