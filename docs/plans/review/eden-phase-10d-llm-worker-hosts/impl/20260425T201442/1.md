# Implementation Review: eden-phase-10d-llm-worker-hosts (Round 1)

## Round 1 — Issues Addressed

| # | Severity | Finding | Fix |
|---|---|---|---|
| 1 | Bug | Compose overlay missing `--implement-env-file` / `--evaluate-env-file` plumbing | Added `eden-implement-env` and `eden-evaluate-env` configs (default empty); wired both `--*-env-file` flags into the implementer/evaluator services in `compose.subprocess.yaml`. |
| 2 | Bug | Implementer container in subprocess overlay didn't mount the artifacts volume; rationale_path lookups would fail in real deployments | Added a read-only mount of `eden-artifacts-data` at `/var/lib/eden/artifacts` to the `implementer-host` overlay. |
| 3 | Risk | Planner CLI overlaid `EDEN_EXPERIMENT_DIR` then merged user env file on top, letting users redirect a host-owned reserved variable | Reordered: load env file first, then write reserved `EDEN_*` keys. |
| 4 | Risk | Planner host respawns on subprocess crash forever — broken `plan_command` would thrash | Added `_MAX_CONSECUTIVE_RESPAWNS = 5` ceiling tracked across both startup + handler failures, reset on success. |
| 5 | Risk | Claim race (WrongToken / IllegalTransition) tore down the long-running planner subprocess and discarded accumulated context | `WrongToken` / `IllegalTransition` are caught and treated as normal "skip this task" without subprocess teardown. |
| 6 | Risk | Worker-side failure context (planner `plan-error.reason`, implementer/evaluator `outcome.description`, stderr) wasn't tied to `task_id` in host logs | `dispatch_plan` now returns the full terminator dict so `reason` is logged; impl/eval log `outcome.description` before normalizing to wire submission; `spawn(...)` accepts `task_id` and `_pump_lines_to_log` includes it on stderr forwarding entries. |
| 7 | Nit | `worker-host-subprocess.md` had broken relative links to chapter files and schemas | Fixed to `../03-roles.md`, `../04-task-protocol.md`, `../schemas/...`. |
