**Findings**
- Critical: the plan still routes worker failure detail through submission shapes that do not exist in the current protocol. It calls for `PlanSubmission(status="error", description=reason)` and multiple `ImplementSubmission(status="error", description=…)` paths, but the actual submission dataclasses have no `description` field, and the only built-in reject channel is the closed `reason` enum on `reject`. As written, those failure paths cannot be implemented without a broader spec/storage/wire change that the plan does not currently call out. See [eden-phase-10d-llm-worker-hosts.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-llm-worker-hosts.md:243), [eden-phase-10d-llm-worker-hosts.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-llm-worker-hosts.md:272), [eden-phase-10d-llm-worker-hosts.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-llm-worker-hosts.md:336), [eden-phase-10d-llm-worker-hosts.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-llm-worker-hosts.md:355), [submissions.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/submissions.py:35), [models.py](/Users/ericalt/Documents/eden/reference/packages/eden-wire/src/eden_wire/models.py:51), and [\_base.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/_base.py:724).
- High: the host-scoped worktree-pruning mitigation is not implementable as described. The plan relies on “only prune that host’s subtree” to avoid cross-host races, but the proposed helper is still `prune_worktrees(repo_path)` and the cleanup mechanism is still `git worktree prune`, which is repo-global rather than path-scoped. That means the stated race mitigation does not actually follow from the design. This needs a different cleanup approach, not just a note in the helper docstring.
