# EDEN Phase 10 chunk 10d: Subprocess-driven worker hosts

## Context

Phase 10 chunks 10b + 10c shipped the dockerized scripted services and
the `setup-experiment` bootstrap. The fixture experiment runs to
quiescence in Compose using **scripted** plan/implement/evaluate
behavior baked into the host process — no user code participates in
the loop.

The roadmap's Phase 10d wording is:

> 10d — LLM worker hosts: planner host (context-accumulating Claude
> session), implementer host (spawns per-task sandbox containers
> running `implement_command`), evaluator host.

Read literally, "LLM" is the headline. But the load-bearing
contribution is **letting user code drive the loop** — the planner,
implementer, and evaluator each run a user-supplied command from the
experiment config (`plan_command`, `implement_command`,
`evaluate_command`, all already accepted by the schema and present in
[`tests/fixtures/experiment/.eden/config.yaml`](../../tests/fixtures/experiment/.eden/config.yaml)). Whether
that command shells out to Claude, Codex, a local LLM, a
deterministic script, or `cat` is the user's business; the host's
contract is the **invocation protocol** between host and command.

This chunk defines and ships that invocation protocol. The
`context-accumulating Claude session` and `per-task sandbox
container` pieces are concrete consequences of the protocol — a
long-running planner subprocess that holds Claude session state
across plan tasks, and a per-task implementer subprocess whose cwd
is a freshly-created worktree the user's command commits into. **No
LLM client code lands in this repo**; users wire their own LLM into
their `*_command` scripts.

## Scope (one chunk)

Three new modes alongside the existing scripted profiles, selected
per-host via a CLI flag — `--mode {scripted,subprocess}`:

1. **Planner subprocess mode.** One long-running planner subprocess
   per host. Tasks dispatched as JSON lines on stdin; proposals
   returned as JSON lines on stdout, terminated by a sentinel.
2. **Implementer subprocess mode.** One short-lived subprocess per
   implement task. cwd is a freshly-created worktree pinned to the
   first parent commit. Subprocess produces a commit (or fails) and
   writes a JSON outcome file; host reads HEAD, validates parent
   reachability, pushes the work branch into the bare repo, submits
   the trial.
3. **Evaluator subprocess mode.** One short-lived subprocess per
   evaluate task. cwd is a freshly-created worktree pinned to the
   trial commit. Subprocess writes a metrics JSON file; host
   validates against `metrics_schema` and submits.

The fixture experiment grows three deterministic command scripts
(`plan.py`, `implement.py`, `eval.py`) that exercise the protocol
without any LLM dependency, so the existing `compose-smoke` job can
run a subprocess-mode end-to-end smoke without external services or
secrets. The fixture's `*_command` config keys are amended to use
`${EDEN_EXPERIMENT_DIR}/<script>.py` so the commands resolve from
any cwd (see §D.0 for the command/cwd/env contract).

## Design

### D.0 Command source, cwd, and environment contract

This section pins the single source of truth for "where does the
host get the command, how is it invoked, and what does it see?" so
that subsequent sections don't drift.

- **Single source of truth: experiment-config.** The host reads
  `plan_command` / `implement_command` / `evaluate_command` from the
  same `experiment-config.yaml` the task-store-server already
  consumes (the schema accepts these as additional properties; no
  schema change). There is no CLI flag to override the command
  string — that would split the source of truth between two places.
- **Experiment directory.** Each host gains `--experiment-dir
  <path>` (required in subprocess mode). The directory is the
  user's experiment checkout — for the fixture, it's
  `tests/fixtures/experiment/`. Compose mounts this into each
  subprocess-mode host via a new bind-mount (setup-experiment grows
  an `--experiment-dir` flag and writes the host-side path into
  `.env` as `EDEN_EXPERIMENT_DIR`).
- **Command invocation: shell mode.** The host invokes the command
  with `subprocess.Popen(cmd_str, shell=True, …)` so user
  expressions like `python3 ${EDEN_EXPERIMENT_DIR}/plan.py` expand
  correctly without the host having to parse and rewrite paths.
  This is acceptable because the command string comes from the
  trusted experiment-config (operator-supplied at setup time), not
  from any per-task / per-trial source. Documented in the protocol
  README.
- **cwd.**
  - Planner subprocess: `cwd = experiment-dir`. One process, lives
    across all plan tasks; doesn't touch git.
  - Implementer subprocess: `cwd = <task worktree>`. The user's
    script runs inside the per-task git worktree and can `git
    commit` directly from cwd.
  - Evaluator subprocess: `cwd = <task worktree>`. The trial's
    commit is checked out at cwd; the user's script reads the tree
    and writes metrics.
- **No free-form failure surface on the wire.** The submission
  dataclasses ([`reference/packages/eden-storage/src/eden_storage/submissions.py`](../../reference/packages/eden-storage/src/eden_storage/submissions.py))
  carry only `status` + role-specific id/payload fields; there is no
  `description` field. Worker-side failure context (subprocess exit
  reason, malformed outcome JSON, deadline-exceeded, etc.) is
  surfaced **only via the host's structured logger** (info/warning
  with `task_id`, `subprocess_role`, `reason`). The worker's own
  `outcome.json` may include a `description` string for operator
  diagnostics, but the host does **not** propagate it onto the wire.
  Expanding the wire failure vocabulary (a normative
  `Submission.diagnostic` field, or a dedicated `task.diagnostic`
  event) is out of scope for this chunk and would require a spec
  amendment + schema change + binding update.
- **Environment.** Every subprocess receives, in addition to the
  host's environment:
  - `EDEN_EXPERIMENT_DIR` — absolute path to the experiment
    directory.
  - `EDEN_TASK_JSON` — relative path (under cwd) to the task brief
    JSON file the host wrote.
  - `EDEN_OUTPUT` — relative path (under cwd) to the file the
    subprocess must write its outcome JSON to.
  - For implementer/evaluator: `EDEN_WORKTREE` — absolute path to
    the worktree (also equals cwd; redundant for convenience).
  - User-supplied env from `--*-env-file` (intended for LLM API
    keys etc.).
- **Fixture-config change.** The fixture's
  [`tests/fixtures/experiment/.eden/config.yaml`](../../tests/fixtures/experiment/.eden/config.yaml)
  is amended in this chunk:

  ```yaml
  plan_command: "python3 ${EDEN_EXPERIMENT_DIR}/plan.py"
  implement_command: "python3 ${EDEN_EXPERIMENT_DIR}/implement.py"
  evaluate_command: "python3 ${EDEN_EXPERIMENT_DIR}/eval.py"
  ```

  Old bare values (`python3 plan.py`) wouldn't have resolved from
  the implementer/evaluator's worktree cwd; the explicit
  `${EDEN_EXPERIMENT_DIR}/` makes the intent explicit and uniform
  across the three roles.

### D.1 Mode selection and CLI shape

Each of the three host services gains a `--mode` flag:

```text
eden-planner-host    --mode {scripted,subprocess}  ...
eden-implementer-host --mode {scripted,subprocess} ...
eden-evaluator-host  --mode {scripted,subprocess}  ...
```

Default stays `scripted` so existing CI jobs and the merged
`compose-smoke` keep working unchanged. New flags appear only in
subprocess mode.

All three hosts gain in subprocess mode:

- `--experiment-config PATH` — already present on evaluator + web-ui
  for chunk-9d/9e parity; added to planner + implementer hosts
  (required in subprocess mode). The host loads the YAML and reads
  the relevant `*_command`.
- `--experiment-dir PATH` — host-side absolute path to the
  experiment directory (required).
- `--*-env-file PATH` — additional env vars to inject (KEY=VALUE
  per line); intended for `ANTHROPIC_API_KEY` and similar.

**Planner-host subprocess flags (added):**

- `--plan-startup-deadline FLOAT` — seconds to wait for the
  subprocess to print a `ready` line (default 30).
- `--plan-task-deadline FLOAT` — seconds to wait for proposals from a
  single dispatched plan task (default 120).
- `--plan-shutdown-deadline FLOAT` — seconds to allow for graceful
  shutdown when the host stops (default 10).
- `--plan-env-file PATH`

**Implementer-host subprocess flags (added):**

- `--worktrees-dir PATH` — base directory under which per-task
  worktrees are created (default: `/var/lib/eden/worktrees` in
  Compose; not on the same volume as the bare repo to avoid
  ownership clashes — see §D.5).
- `--implement-task-deadline FLOAT` (default 600).
- `--implement-env-file PATH`

**Evaluator-host subprocess flags (added):**

- `--worktrees-dir PATH`
- `--evaluate-task-deadline FLOAT` (default 300).
- `--evaluate-env-file PATH`

The implementer's existing `--repo-path` stays required (the host
needs to manipulate the bare repo); the evaluator gains a new
`--repo-path` requirement in subprocess mode.

### D.2 Planner subprocess protocol

Wire format: one JSON object per line, on both stdin and stdout. The
host writes "dispatch" lines, the worker writes "proposal" /
"plan-done" / "plan-error" / "ready" lines. The subprocess runs
once per host with cwd=experiment-dir per §D.0; it reads
`EDEN_EXPERIMENT_DIR` from the env (also accessible as cwd) for
auxiliary file lookup, and uses stdin/stdout exclusively for
protocol traffic.

**Startup handshake.** When the host launches the subprocess, the
subprocess must print exactly one line `{"event": "ready"}` to
stdout within `--plan-startup-deadline`. Until ready, the host
ignores other lines (logged at debug). After ready, the host begins
dispatching tasks.

**Dispatching a plan task.** The host writes:

```json
{"event": "plan", "task_id": "plan-…", "experiment_id": "exp-1",
 "objective": {"expr": "score", "direction": "maximize"},
 "metrics_schema": {"score": "real"},
 "history": [{"trial_id": "trial-…", "status": "success",
              "commit_sha": "abc…", "metrics": {"score": 0.7}}]}
```

The `history` field is a flat list of completed trials available so
far (an LLM planner uses it to inform the next proposal; a
deterministic planner ignores it).

**Worker response.** The worker writes any number of proposal lines
followed by exactly one terminator:

```json
{"event": "proposal", "task_id": "plan-…", "slug": "p0",
 "priority": 1.0, "parent_commits": ["abc…"],
 "artifacts_uri": "file:///var/lib/eden/artifacts/plan-…/p0",
 "rationale": "free-form text"}
{"event": "plan-done", "task_id": "plan-…"}
```

`task_id` on every line scopes the response so a misbehaving worker
can't bleed state across tasks. `rationale` is optional; if present,
the host writes it to `<artifacts_dir>/proposals/<proposal_id>/rationale.md`
and stamps the proposal's `artifacts_uri` accordingly (overrides any
worker-supplied `artifacts_uri`). This anchors the chunk-9c "rationale
markdown rendered inline" path.

**Errors.** A `{"event": "plan-error", "task_id": "plan-…",
"reason": "..."}` line causes the host to submit
`PlanSubmission(status="error")` and continue.
A protocol violation (missing required field, wrong `task_id`,
deadline exceeded) causes the host to: (a) treat the current task
as `plan-error`; (b) tear down and respawn the subprocess if the
violation suggests state corruption (deadline + truncated stdout);
(c) continue.

**Stdout discipline.** The worker's stdout is consumed line-by-line
on a background thread per task; non-JSON lines (or JSON missing
`event`) are logged at debug and dropped. Stderr is line-buffered
and forwarded to the host's logger at info, which gives users
visibility into LLM agent output.

### D.3 Implementer subprocess protocol

The flow honors [`spec/v0/03-roles.md`](../../spec/v0/03-roles.md) §3.2
step 1: the trial MUST be persisted with `status == "starting"`
**before** any repository write becomes observable. This matches the
chunk-9c web-ui implementer's ordering and the scripted
`ScriptedImplementer._drive_starting_trial` pattern. Per-task
short-lived subprocess. The host:

1. Generates a host-owned `trial_id` (server-only, never round-trips
   through the request surface or the subprocess; matches chunk-9c).
2. **Pre-Phase-1 ref-collision guard.** Computes the canonical work
   branch name `work/<slug>-<trial_id>`. If
   `repo.ref_exists(f"refs/heads/{branch}")`, short-circuits with
   an `ImplementSubmission(status="error", trial_id=…)` (the
   trial_id is host-generated; no `create_trial` is called, so the
   id is unbound — the `_reject_implement` path tolerates this per
   the chunk-9c pattern). The collision context is logged but does
   not appear on the wire (Submission has no free-form fields).
3. **Phase 1: Persist the starting trial.** Calls
   `store.create_trial(trial_id=…, proposal_id=…, status="starting")`.
   No `commit_sha` yet (it's set later by `_accept_implement` once
   the submission lands). Crash after this point leaves a
   recoverable orphan: the chunk-1 expired-claim sweeper composite-
   commits the trial to `error` atomically with the implement task
   reverting to `pending`.
4. **Phase 2a: Create the worktree.** Resolves
   `proposal.parent_commits[0]` against the bare repo and creates a
   fresh worktree at `<worktrees_dir>/<host>-<task_id>/` checked out
   to that commit via `git worktree add --detach <path> <commit>`.
   (If `parent_commits` has multiple entries, the worktree is
   single-parent; the user's `implement_command` is responsible for
   producing a merge commit by writing one with `git commit-tree -p
   <p1> -p <p2>` or equivalent. The host's job is to start the
   worktree at the *first* parent.)
5. **Phase 2b: Write the task brief.** Writes
   `<worktree>/.eden/task.json`:

   ```json
   {
     "task_id": "implement-…",
     "trial_id": "trial-…",
     "proposal_id": "proposal-…",
     "proposal_slug": "p0",
     "parent_commits": ["abc…", …],
     "rationale_path": "/var/lib/eden/artifacts/proposals/proposal-…/rationale.md",
     "branch": "work/p0-trial-…",
     "output_path": ".eden/outcome.json"
   }
   ```

   `branch` is the canonical name from step 2; the user's script
   may use it but cannot override (Phase 2c uses the canonical name
   regardless of what the subprocess writes back). `rationale_path`
   is absolute on the host filesystem and only present if the
   proposal was created with a rationale.
6. **Phase 2c: Run the subprocess.** Spawns `implement_command` per
   §D.0 (cwd=worktree, env includes `EDEN_TASK_JSON`,
   `EDEN_OUTPUT`, `EDEN_WORKTREE`, `EDEN_EXPERIMENT_DIR`, plus
   `--implement-env-file` content). Awaits exit within
   `--implement-task-deadline`. On timeout: SIGTERM, wait 10 s,
   SIGKILL.
7. **Phase 2d: Read outcome.** On exit-zero, reads
   `<worktree>/.eden/outcome.json`:

   ```json
   {"status": "success", "commit_sha": "def…",
    "description": "free-form summary"}
   ```

   Or `{"status": "error", "description": "…"}`. The user's script
   does **not** set `branch` (host owns it).
8. **Phase 2e: Validation (success path):**
   - `commit_sha` exists in the bare repo's object database
     (`git -C <bare> cat-file -e <sha>`). The user's `git commit`
     inside the worktree updates the bare repo's object DB
     automatically (worktrees share the parent's `.git/objects`).
   - `is_ancestor(parent, commit_sha)` for **every** parent in
     `proposal.parent_commits` per spec §3.3 reachability.
   - Validation failures → submit
     `ImplementSubmission(status="error", trial_id=…)` (Phase 3
     below). The trial ends up `error` via the store's
     `_reject_implement` path, which atomically composite-commits
     the `starting` trial to `error`.
9. **Phase 2f: Create the work-branch ref.** `repo.create_ref(
   f"refs/heads/{branch}", commit_sha)`. The pre-Phase-1 collision
   guard makes this safe; an unexpected EEXIST here (race against
   another host) → submit `ImplementSubmission(status="error",
   trial_id=…)`; the collision context goes to the host log only.
10. **Phase 3: Submit.** Calls `store.submit(...)` with retry-before-
    orphan (3 attempts, exponential backoff) per the existing
    Phase-8a path. On retry exhaustion, runs the same committed-state
    read-back the chunk-9c implementer route uses (read_task +
    read_submission + submissions_equivalent) to classify
    success / orphan / conflict.
11. **Cleanup.** Removes the worktree (`git worktree remove --force
    <path>`). Cleanup failure logs a warning; the next host startup
    runs `sweep_host_worktrees` against this host's own subdir as a
    maintenance step (see §D.5; path-scoped, not the repo-global
    `git worktree prune`).
12. **Failure paths:** exit-nonzero → submit
    `ImplementSubmission(status="error", trial_id=…)`; timeout →
    same; missing `outcome.json` → same; outcome `status="error"`
    → same. All four routes go through Phase 3 with the same
    retry-before-orphan + read-back classification.

**Failure modes covered:**

- Subprocess writes a `commit_sha` that's not a valid object →
  submitted as `ImplementSubmission(status="error",
  trial_id=…)`; the rejection reason goes to the host log;
  trial-creation ordering ensures the `starting` trial is
  composite-committed to `error` atomically with the task
  transition (`_reject_implement`).
- Subprocess produces a commit whose parents don't satisfy spec
  §3.3 → same.
- Subprocess succeeds but exits non-zero → host trusts exit code
  (treats as error). Document this in the protocol README so users
  don't get bitten.
- Worktree cleanup failure → logged warning; the next task uses a
  new task-scoped path so leakage doesn't cascade.
- Crash mid-flow (host killed): the worktree leaks. The next host
  startup runs `sweep_host_worktrees` against the host's own
  subdir, removing leftover dirs via path-scoped
  `git worktree remove --force` (see §D.5).

### D.4 Evaluator subprocess protocol

Symmetric to implementer (the chapter-9d evaluator has no `starting`
trial-creation step because the trial already exists from the
implementer's flow; the evaluator only ever reads it):

1. Worktree at `trial.commit_sha` (detached).
2. Write `.eden/eval-task.json`:

   ```json
   {
     "task_id": "evaluate-…",
     "trial_id": "trial-…",
     "trial_branch": "trial/…-p0",
     "trial_commit_sha": "ghi…",
     "metrics_schema": {"score": "real"},
     "objective": {"expr": "score", "direction": "maximize"},
     "output_path": ".eden/eval-outcome.json"
   }
   ```

3. Run `evaluate_command` per §D.0 (cwd=worktree, env includes
   `EDEN_TASK_JSON`, `EDEN_OUTPUT`, `EDEN_WORKTREE`,
   `EDEN_EXPERIMENT_DIR`, plus `--evaluate-env-file` content).
4. Read outcome JSON:

   ```json
   {"status": "success", "metrics": {"score": 0.83},
    "artifacts_uri": "file:///var/lib/eden/artifacts/evaluate-…"}
   ```

   Or `status: error|eval_error`.
5. Validate metrics against `metrics_schema` using the same
   `Store.validate_metrics` path the chunk-9d evaluator route uses.
6. Submit; clean up worktree.

### D.5 Worktree management

Both the implementer and evaluator subprocess modes need ephemeral
worktrees over the bare repo. New helper module:

[`reference/services/_common/src/eden_service_common/worktrees.py`](../../reference/services/_common/src/eden_service_common/worktrees.py)

```python
class TaskWorktree:
    def __init__(self, repo_path: Path, base_dir: Path, task_id: str): ...
    def create(self, *, commit: str) -> Path: ...
    def head_sha(self) -> str: ...
    def remove(self) -> None: ...

def sweep_host_worktrees(repo_path: Path, host_subdir: Path) -> None:
    """Remove every leftover worktree directory under host_subdir.

    Walks host_subdir's immediate children; for each directory that
    looks like a worktree (presence of a `.git` text file pointing
    at the bare repo's worktrees/ admin dir), runs
    `git worktree remove --force <path>` against the bare repo to
    deregister and delete it.

    Path-scoped: never touches admin entries outside host_subdir,
    so a sweep in one host can't race a `git worktree add` in
    another host whose host_subdir is different.
    """
```

Internally it shells out to `git worktree add --detach`, `git -C
<wt> rev-parse HEAD`, `git worktree remove --force`. The bare-repo
git invocation needs a `--git-dir <bare>` form because there's no
working directory; the worktree paths live outside the bare repo
proper.

The host process owns the worktree. The user's command sees only
its cwd; it should not be aware that this is a worktree (it's just
a checkout it can `git commit` in). The host's
`commit_exists`/`is_ancestor` queries go to the **bare** repo
(where the commit object lives after the worker's `git commit`
inside the worktree).

**Volume layout.** The worktree base directory is **not** on the
same Compose volume as the bare repo. `git worktree add` writes
admin metadata under `<bare>/worktrees/<name>/`, so the bare-repo
volume needs to be writable by the host user — but the worktree
checkout itself is bigger and gets created/destroyed per task; mixing
the two on one volume confuses cleanup. Layout:

- Volume `eden-bare-repo` mounted at `/var/lib/eden/repo` (bare
  repo, read-write by hosts that need it).
- Volume `eden-worktrees` mounted at `/var/lib/eden/worktrees`
  (worktree checkouts, ephemeral, host-private subtrees).

**Cross-host isolation.** Each host writes worktrees under a
host-private subdir
`<worktrees_dir>/<container_hostname>/<task_id>/`. The host's
startup-time cleanup walks **only its own subdir** and runs
`git worktree remove --force <path>` for each leftover (path-scoped,
not the repo-global `git worktree prune`). Because every targeted
remove is restricted to the host's own subdir, two hosts running
sweeps concurrently never touch each other's worktree paths or
their admin entries. The bare-repo's `worktrees/<name>/` admin
entry is named after the **path basename** (`<task_id>`), so as
long as `task_id` values are globally unique (UUID-suffixed, per
the existing `_trial_id` factory) two hosts cannot collide on an
admin entry either.

The host's own `task_id` paths are tracked in an in-process
registry; per-task cleanup at submit-time uses
`TaskWorktree.remove()`, which calls `git worktree remove --force`
on that exact path. Crash-recovery cleanup runs only at startup
via `sweep_host_worktrees`. There is no periodic cross-host prune.

### D.6 Subprocess execution and signal handling

Implemented in a new `eden_service_common.subprocess` module so the
three host services share one execution path:

- `subprocess.Popen` with `start_new_session=True` (each subprocess
  is its own process group, so SIGTERM during host shutdown can
  cleanly kill the worker tree without nuking the host).
- Stdout consumed on a background thread per process; lines pushed
  into a thread-safe queue the main thread reads with timeouts.
- Stderr consumed on a separate background thread, forwarded to
  `logger.info(line, extra={"subprocess": "<role>", "task_id": …})`.
- Termination ladder: `SIGTERM`, wait `--*-shutdown-deadline`,
  `SIGKILL`. Same ladder used for per-task timeouts and for host
  graceful shutdown.
- On host receiving SIGTERM/SIGINT (existing `StopFlag` pattern),
  the subprocess receives the same signal; in-flight tasks are
  abandoned (no submission); claim sweeper at the next host startup
  reclaims them.

### D.7 Fixture experiment scripts

Three new files under
[`tests/fixtures/experiment/`](../../tests/fixtures/experiment/):

- `plan.py` — long-running JSON-line worker that prints `ready`,
  reads dispatched plan tasks from stdin, and writes one proposal
  per task. Deterministic: priority equals N − i for the i-th
  proposal of an N-proposal plan; `parent_commits` from
  `EDEN_BASE_COMMIT_SHA`. Written so the existing fixture config's
  `plan_command: "python3 plan.py"` invokes it.
- `implement.py` — short-lived; reads `EDEN_TASK_JSON`, makes a
  trivial blob change, runs `git add`/`git commit`, writes
  `EDEN_OUTPUT` with the resulting commit SHA. Mirrors the scripted
  implementer's blob shape so the existing `compose-smoke`
  assertions pass without parameterization.
- `eval.py` — short-lived; reads `EDEN_TASK_JSON`, emits a
  deterministic `score` matching the scripted evaluator.

These exist to make the protocol testable in CI without LLM
secrets. Real users supply their own `*.py` (or other interpreter)
that does LLM work.

The fixture's existing `tests/fixtures/experiment/.eden/config.yaml`
already declares `plan_command` / `implement_command` /
`evaluate_command` keys; this chunk amends their values to the
`${EDEN_EXPERIMENT_DIR}/<script>.py` form (see §D.0). The fixture
config change is mechanical and the schema-parity tests are
unaffected (these keys are accepted as additional properties).

### D.8 Compose updates

A new variant of the stack runs in subprocess mode by setting two
new `.env` keys:

```text
EDEN_WORKER_MODE=subprocess        # default: scripted
EDEN_EXPERIMENT_DIR_HOST=/abs/host/path/to/experiment
```

`EDEN_EXPERIMENT_DIR_HOST` is the **host-side** absolute path to
the user's experiment directory, written by setup-experiment when
invoked with `--experiment-dir <path>`. Compose bind-mounts it
read-only into each subprocess-mode host at
`/etc/eden/experiment-dir`, and the host's `--experiment-dir` flag
points at that container path. The `experiment-config.yaml` config
mount stays as-is (chunk-9c/9d wiring); the host loads the YAML and
reads the `*_command` keys from it.

When `EDEN_WORKER_MODE=subprocess`, each host service's `command:`
gains `--mode subprocess --experiment-dir /etc/eden/experiment-dir`
plus the per-host subprocess flags from §D.1. The command strings
are **not** plumbed through Compose env — they live in the
experiment-config YAML.

A new named volume `eden-worktrees` is added for the implementer/
evaluator's per-task worktrees (mounted at
`/var/lib/eden/worktrees`); see §D.5 for why this is separate from
the bare-repo volume.

Compose-side flag plumbing uses `${EDEN_WORKER_MODE:-scripted}` so
the scripted-mode default is unchanged. The subprocess-mode flags
appear as either-or blocks in the YAML (Compose has no native
conditional, so the simplest approach is two compose files merged
via `compose -f compose.yaml -f compose.subprocess.yaml`; the
subprocess overlay adds the new flags and bind-mounts).

The compose-smoke job stays in scripted mode (the merged shape from
10c) **plus** a new `compose-smoke-subprocess` matrix entry that
runs the same end-to-end assertions in subprocess mode against the
fixture scripts. Two CI signals: scripted protects the chunk-1–9
behavior; subprocess protects this chunk's protocol.

## Risks

- **Fixture-script drift.** The fixture `plan.py`/`implement.py`/
  `eval.py` must produce the same end state as the scripted hosts
  for `compose-smoke-subprocess` to pass with the same assertions.
  Mitigation: factor the deterministic shape (blob payload,
  metric values) into a small shared helper imported by both the
  fixture scripts and `eden_service_common.scripted` so divergence
  is mechanically prevented.
- **`git worktree` semantics on the bare repo.** Worktrees over a
  bare repo work but require care — the worktree's `.git` is a
  text file pointing at the bare repo's `worktrees/<name>/`
  directory. Permissions on the bare-repo volume need to allow the
  host user to write there. The 10b/10c volume already grants this
  to the `eden` user; verify with the new test before relying on
  it.
- **Worktree leak on host crash.** Mitigation in §D.5 (`git
  worktree prune` at host startup), but if a single bare repo is
  shared across multiple host containers (planner + implementer +
  web-ui), a prune in one might race a worktree-add in another.
  Mitigation: each host writes its worktrees under
  `<worktrees_dir>/<container_hostname>/<task_id>/` and only prunes
  its own subtree. Document this in the helper module.
- **Long-running planner subprocess as a SPOF.** A crashed Claude
  CLI or a stuck process kills planner availability for the whole
  experiment. Mitigation: respawn-on-crash (subject to a max
  respawn count) is in scope for this chunk; cross-restart session
  resumption (Claude `-c` after host restart) is a user-script
  concern.
- **Stdout-as-protocol fragility.** Anything the user's script
  prints to stdout that *looks* like a JSON event line could spoof
  the protocol. Mitigation: every event line carries the host-
  supplied `task_id`; the host rejects lines whose `task_id`
  doesn't match the currently dispatched task. Sentinel `ready`
  has no `task_id` and is only valid before the first dispatch.
  Document the spoofing risk in the protocol README.
- **`history` field unboundedness.** Plan tasks late in a long
  experiment carry a `history` array proportional to trial count.
  Mitigation for this chunk: cap at the last 50 completed trials,
  newest first; document the cap in the protocol README. Future
  refinement (windowing strategies, summarization) is a user
  script concern.
