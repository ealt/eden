**Findings**
- Critical: the implementer subprocess flow never creates the required `starting` trial before it creates the branch and submits the result. The plan introduces `trial_id` in `.eden/task.json`, then goes straight from output validation to `repo.create_ref(...)` and `store.submit(...)`, but there is no `store.create_trial(...)` step anywhere in the flow. That violates the existing role contract and the current reference pattern, both of which require the trial to exist before any repo write is made observable. See [eden-phase-10d-llm-worker-hosts.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-llm-worker-hosts.md:196), [eden-phase-10d-llm-worker-hosts.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-llm-worker-hosts.md:252), [03-roles.md](/Users/ericalt/Documents/eden/spec/v0/03-roles.md:85), and the current scripted host at [workers.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/workers.py:203).
- High: the command-source and path-resolution model is internally inconsistent. The plan says the host should run commands from the experiment config and that the existing fixture config already works unchanged, but the design actually introduces CLI flags / Compose env plumbing for the commands, and then runs implement/evaluate with `cwd` set to the per-task worktree. With the current fixture values `python3 plan.py`, `python3 implement.py`, and `python3 eval.py`, planner mode might work if `--plan-cwd` points at the experiment dir, but implementer and evaluator will not find those scripts from the worktree unless the host copies them there or rewrites them to absolute paths. That needs one explicit source of truth and one explicit relative-path rule. See [eden-phase-10d-llm-worker-hosts.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-llm-worker-hosts.md:17), [eden-phase-10d-llm-worker-hosts.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-llm-worker-hosts.md:90), [eden-phase-10d-llm-worker-hosts.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-llm-worker-hosts.md:218), [eden-phase-10d-llm-worker-hosts.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-llm-worker-hosts.md:297), [eden-phase-10d-llm-worker-hosts.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-llm-worker-hosts.md:371), [eden-phase-10d-llm-worker-hosts.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-llm-worker-hosts.md:398), and [config.yaml](/Users/ericalt/Documents/eden/tests/fixtures/experiment/.eden/config.yaml:4).

**Level Assessment**
- `1. Missing context`: Significant concerns. The plan does not clearly define where `plan_command` / `implement_command` / `evaluate_command` are sourced from at runtime, how relative paths are resolved, or how non-Compose deployments are expected to provide the experiment directory and command scripts. A reader cannot tell what the authoritative configuration surface is.
- `2. Feasibility`: Significant concerns. As written, implementer subprocess mode cannot satisfy the current store/spec contract because it omits trial creation entirely, and the command/cwd model does not actually support the “fixture works unchanged” claim. I would stop here and resolve these before evaluating alternatives, completeness, or edge-case coverage.

**Overall Assessment**
The direction is plausible, but the plan is not implementation-ready yet. Pin down the trial lifecycle for implementer mode and the command/config/path contract first; after that, the lower-level design tradeoffs will be much easier to evaluate.
