# EDEN Phase 10 chunk 10d: Subprocess-driven worker hosts

## Context

Phase 10 chunks 10b + 10c shipped the dockerized scripted services and
the `setup-experiment` bootstrap. The fixture experiment runs to
quiescence in Compose using **scripted** plan/implement/evaluate
behavior baked into the host process — no user code participates in
the loop.

The roadmap's Phase 10d wording is:

> 10d — LLM worker hosts: planner host (context-accumulating Claude
> session), implementer host (spawns per-task sandbox containers
> running `implement_command`), evaluator host.

Read literally, "LLM" is the headline. But the load-bearing
contribution is **letting user code drive the loop** — the planner,
implementer, and evaluator each run a user-supplied command from the
experiment config (`plan_command`, `implement_command`,
`evaluate_command`, all already accepted by the schema and present in
[`tests/fixtures/experiment/.eden/config.yaml`](../../tests/fixtures/experiment/.eden/config.yaml)). Whether
that command shells out to Claude, Codex, a local LLM, a
deterministic script, or `cat` is the user's business; the host's
contract is the **invocation protocol** between host and command.

This chunk defines and ships that invocation protocol. The
`context-accumulating Claude session` and `per-task sandbox
container` pieces are concrete consequences of the protocol — a
long-running planner subprocess that holds Claude session state
across plan tasks, and a per-task implementer subprocess whose cwd
is a freshly-created worktree the user's command commits into. **No
LLM client code lands in this repo**; users wire their own LLM into
their `*_command` scripts.

## Scope (one chunk)

Three new modes alongside the existing scripted profiles, selected
per-host via a CLI flag — `--mode {scripted,subprocess}`:

1. **Planner subprocess mode.** One long-running planner subprocess
   per host. Tasks dispatched as JSON lines on stdin; proposals
   returned as JSON lines on stdout, terminated by a sentinel.
2. **Implementer subprocess mode.** One short-lived subprocess per
   implement task. cwd is a freshly-created worktree pinned to the
   first parent commit. Subprocess produces a commit (or fails) and
   writes a JSON outcome file; host reads HEAD, validates parent
   reachability, pushes the work branch into the bare repo, submits
   the trial.
3. **Evaluator subprocess mode.** One short-lived subprocess per
   evaluate task. cwd is a freshly-created worktree pinned to the
   trial commit. Subprocess writes a metrics JSON file; host
   validates against `metrics_schema` and submits.

The fixture experiment grows three deterministic command scripts
(`plan.py`, `implement.py`, `eval.py`) that exercise the protocol
without any LLM dependency, so the existing `compose-smoke` job can
run a subprocess-mode end-to-end smoke without external services or
secrets.

## Design

### D.1 Mode selection and CLI shape

Each of the three host services gains a `--mode` flag:

```text
eden-planner-host    --mode {scripted,subprocess}  ...
eden-implementer-host --mode {scripted,subprocess} ...
eden-evaluator-host  --mode {scripted,subprocess}  ...
```

Default stays `scripted` so existing CI jobs and the merged
`compose-smoke` keep working unchanged. New flags appear only in
subprocess mode.

**Planner-host subprocess flags (added):**

- `--plan-command STR` — the command to launch (split via `shlex`).
- `--plan-cwd PATH` — working directory for the subprocess (default:
  `--repo-path` if set, else process cwd).
- `--plan-startup-deadline FLOAT` — seconds to wait for the
  subprocess to print a `ready` line (default 30).
- `--plan-task-deadline FLOAT` — seconds to wait for proposals from a
  single dispatched plan task (default 120).
- `--plan-shutdown-deadline FLOAT` — seconds to allow for graceful
  shutdown when the host stops (default 10).
- `--plan-env-file PATH` — additional env vars to inject (KEY=VALUE
  per line); intended for `ANTHROPIC_API_KEY` and similar.

**Implementer-host subprocess flags (added):**

- `--implement-command STR`
- `--worktrees-dir PATH` — base directory under which per-task
  worktrees are created (default: `/var/lib/eden/worktrees` in
  Compose; same volume as repo by convention).
- `--implement-task-deadline FLOAT` (default 600).
- `--implement-env-file PATH`

**Evaluator-host subprocess flags (added):**

- `--evaluate-command STR`
- `--worktrees-dir PATH`
- `--evaluate-task-deadline FLOAT` (default 300).
- `--evaluate-env-file PATH`

The implementer's existing `--repo-path` is required in subprocess
mode (the host needs to manipulate the bare repo); the evaluator
gains a new `--repo-path` requirement in subprocess mode.

### D.2 Planner subprocess protocol

Wire format: one JSON object per line, on both stdin and stdout. The
host writes "dispatch" lines, the worker writes "proposal" /
"plan-done" / "plan-error" / "ready" lines.

**Startup handshake.** When the host launches the subprocess, the
subprocess must print exactly one line `{"event": "ready"}` to
stdout within `--plan-startup-deadline`. Until ready, the host
ignores other lines (logged at debug). After ready, the host begins
dispatching tasks.

**Dispatching a plan task.** The host writes:

```json
{"event": "plan", "task_id": "plan-…", "experiment_id": "exp-1",
 "objective": {"expr": "score", "direction": "maximize"},
 "metrics_schema": {"score": "real"},
 "history": [{"trial_id": "trial-…", "status": "success",
              "commit_sha": "abc…", "metrics": {"score": 0.7}}]}
```

The `history` field is a flat list of completed trials available so
far (an LLM planner uses it to inform the next proposal; a
deterministic planner ignores it).

**Worker response.** The worker writes any number of proposal lines
followed by exactly one terminator:

```json
{"event": "proposal", "task_id": "plan-…", "slug": "p0",
 "priority": 1.0, "parent_commits": ["abc…"],
 "artifacts_uri": "file:///var/lib/eden/artifacts/plan-…/p0",
 "rationale": "free-form text"}
{"event": "plan-done", "task_id": "plan-…"}
```

`task_id` on every line scopes the response so a misbehaving worker
can't bleed state across tasks. `rationale` is optional; if present,
the host writes it to `<artifacts_dir>/proposals/<proposal_id>/rationale.md`
and stamps the proposal's `artifacts_uri` accordingly (overrides any
worker-supplied `artifacts_uri`). This anchors the chunk-9c "rationale
markdown rendered inline" path.

**Errors.** A `{"event": "plan-error", "task_id": "plan-…",
"reason": "..."}` line causes the host to submit
`PlanSubmission(status="error", description=reason)` and continue.
A protocol violation (missing required field, wrong `task_id`,
deadline exceeded) causes the host to: (a) treat the current task
as `plan-error`; (b) tear down and respawn the subprocess if the
violation suggests state corruption (deadline + truncated stdout);
(c) continue.

**Stdout discipline.** The worker's stdout is consumed line-by-line
on a background thread per task; non-JSON lines (or JSON missing
`event`) are logged at debug and dropped. Stderr is line-buffered
and forwarded to the host's logger at info, which gives users
visibility into LLM agent output.

### D.3 Implementer subprocess protocol

Per-task short-lived subprocess. The host:

1. Resolves `proposal.parent_commits[0]` against the bare repo and
   creates a fresh worktree at
   `<worktrees_dir>/<task_id>/` checked out to that commit, using
   `git worktree add --detach <path> <commit>`. (If
   `parent_commits` has multiple entries, the worktree is
   single-parent; the user's `implement_command` is responsible for
   producing a merge commit by writing one with `git commit-tree -p
   <p1> -p <p2>` or equivalent. The host's job is to start the
   worktree at the *first* parent.)
2. Writes the task brief to a JSON file inside the worktree at
   `.eden/task.json`:

   ```json
   {
     "task_id": "implement-…",
     "trial_id": "trial-…",
     "proposal_id": "proposal-…",
     "proposal_slug": "p0",
     "parent_commits": ["abc…", …],
     "rationale_path": "/var/lib/eden/artifacts/proposals/proposal-…/rationale.md",
     "output_path": ".eden/outcome.json"
   }
   ```

   `trial_id` is host-generated up front (matching the chunk-9c
   pattern of host-owned IDs that don't round-trip through the
   wire). `rationale_path` is absolute on the host filesystem and
   only present if the proposal was created with a rationale (will
   be present for proposals submitted via the planner subprocess
   protocol; not present for proposals minted by the scripted
   workers).
3. Spawns `implement_command` with cwd = worktree, environment
   extended with `EDEN_TASK_JSON=.eden/task.json` and
   `EDEN_OUTPUT=.eden/outcome.json`, plus the contents of
   `--implement-env-file` if supplied.
4. Awaits exit within `--implement-task-deadline`. On timeout,
   sends SIGTERM, waits 10 s, sends SIGKILL.
5. On exit-zero, reads `.eden/outcome.json` from inside the
   worktree:

   ```json
   {
     "status": "success",
     "commit_sha": "def…",
     "branch": "work/p0-trial-…",
     "description": "free-form summary"
   }
   ```

   Or:

   ```json
   {"status": "error", "description": "…"}
   ```

6. **Validation (success path):**
   - `commit_sha` exists in the bare repo's object database
     (`git -C <bare> cat-file -e <sha>`). The user's `git commit`
     inside the worktree updates the bare repo's object DB
     automatically (worktrees share the parent's `.git/objects`).
   - `is_ancestor(parent, commit_sha)` for **every** parent in
     `proposal.parent_commits` per spec §3.3 reachability.
   - `branch` is either absent (host generates
     `work/<slug>-<trial_id>`) or matches `^work/[A-Za-z0-9._/-]+$`
     and not yet present in the bare repo as a ref.
7. Creates the `work/<…>` ref in the bare repo via
   `repo.create_ref(...)`.
8. Submits via the same Phase 8a retry-before-orphan path the
   scripted implementer uses; on transport indeterminacy, reads
   back the committed task to classify success vs orphan.
9. Removes the worktree (`git worktree remove --force <path>`).
10. On exit-nonzero or timeout: submits
    `ImplementSubmission(status="error", description=…)`; cleans up
    the worktree.

**Failure modes covered:**

- Subprocess writes a `commit_sha` that's not a valid object →
  classified as `status=error` with description; trial-creation
  ordering ensures no orphan trial.
- Subprocess produces a commit whose parents don't satisfy spec
  §3.3 → same.
- Subprocess succeeds but exits non-zero → host trusts exit code
  (treats as error). Document this in the protocol README so users
  don't get bitten.
- Worktree cleanup failure → logged warning; the next task uses a
  new task-scoped path so leakage doesn't cascade.
- Crash mid-flow (host killed): the worktree leaks. The next host
  startup runs `git worktree prune --expire=1.day` against the bare
  repo as a maintenance step.

### D.4 Evaluator subprocess protocol

Symmetric to implementer:

1. Worktree at `trial.commit_sha` (detached).
2. Write `.eden/eval-task.json`:

   ```json
   {
     "task_id": "evaluate-…",
     "trial_id": "trial-…",
     "trial_branch": "trial/…-p0",
     "trial_commit_sha": "ghi…",
     "metrics_schema": {"score": "real"},
     "objective": {"expr": "score", "direction": "maximize"},
     "output_path": ".eden/eval-outcome.json"
   }
   ```

3. Run `evaluate_command` with cwd=worktree, env extended with
   `EDEN_TASK_JSON` and `EDEN_OUTPUT`.
4. Read outcome JSON:

   ```json
   {"status": "success", "metrics": {"score": 0.83},
    "artifacts_uri": "file:///var/lib/eden/artifacts/evaluate-…"}
   ```

   Or `status: error|eval_error`.
5. Validate metrics against `metrics_schema` using the same
   `Store.validate_metrics` path the chunk-9d evaluator route uses.
6. Submit; clean up worktree.

### D.5 Worktree management

Both the implementer and evaluator subprocess modes need ephemeral
worktrees over the bare repo. New helper module:

[`reference/services/_common/src/eden_service_common/worktrees.py`](../../reference/services/_common/src/eden_service_common/worktrees.py)

```python
class TaskWorktree:
    def __init__(self, repo_path: Path, base_dir: Path, task_id: str): ...
    def create(self, *, commit: str) -> Path: ...
    def head_sha(self) -> str: ...
    def remove(self) -> None: ...

def prune_worktrees(repo_path: Path) -> None: ...
```

Internally it shells out to `git worktree add --detach`, `git -C
<wt> rev-parse HEAD`, `git worktree remove --force`. The bare-repo
git invocation needs a `--git-dir <bare>` form because there's no
working directory; the worktree paths live outside the bare repo
proper.

The host process owns the worktree. The user's command sees only
its cwd; it should not be aware that this is a worktree (it's just
a checkout it can `git commit` in). The host's
`commit_exists`/`is_ancestor` queries go to the **bare** repo
(where the commit object lives after the worker's `git commit`
inside the worktree).

### D.6 Subprocess execution and signal handling

Implemented in a new `eden_service_common.subprocess` module so the
three host services share one execution path:

- `subprocess.Popen` with `start_new_session=True` (each subprocess
  is its own process group, so SIGTERM during host shutdown can
  cleanly kill the worker tree without nuking the host).
- Stdout consumed on a background thread per process; lines pushed
  into a thread-safe queue the main thread reads with timeouts.
- Stderr consumed on a separate background thread, forwarded to
  `logger.info(line, extra={"subprocess": "<role>", "task_id": …})`.
- Termination ladder: `SIGTERM`, wait `--*-shutdown-deadline`,
  `SIGKILL`. Same ladder used for per-task timeouts and for host
  graceful shutdown.
- On host receiving SIGTERM/SIGINT (existing `StopFlag` pattern),
  the subprocess receives the same signal; in-flight tasks are
  abandoned (no submission); claim sweeper at the next host startup
  reclaims them.

### D.7 Fixture experiment scripts

Three new files under
[`tests/fixtures/experiment/`](../../tests/fixtures/experiment/):

- `plan.py` — long-running JSON-line worker that prints `ready`,
  reads dispatched plan tasks from stdin, and writes one proposal
  per task. Deterministic: priority equals N − i for the i-th
  proposal of an N-proposal plan; `parent_commits` from
  `EDEN_BASE_COMMIT_SHA`. Written so the existing fixture config's
  `plan_command: "python3 plan.py"` invokes it.
- `implement.py` — short-lived; reads `EDEN_TASK_JSON`, makes a
  trivial blob change, runs `git add`/`git commit`, writes
  `EDEN_OUTPUT` with the resulting commit SHA. Mirrors the scripted
  implementer's blob shape so the existing `compose-smoke`
  assertions pass without parameterization.
- `eval.py` — short-lived; reads `EDEN_TASK_JSON`, emits a
  deterministic `score` matching the scripted evaluator.

These exist to make the protocol testable in CI without LLM
secrets. Real users supply their own `*.py` (or other interpreter)
that does LLM work.

The fixture's existing `tests/fixtures/experiment/.eden/config.yaml`
already declares the three commands; no schema or fixture-config
change needed.

### D.8 Compose updates

A new variant of the stack runs in subprocess mode by setting two
new optional `.env` keys:

```text
EDEN_WORKER_MODE=subprocess        # default: scripted
EDEN_EXPERIMENT_DIR=/etc/eden/experiment
```

When `EDEN_WORKER_MODE=subprocess`, each host service appends
`--mode subprocess --plan-command/--implement-command/--evaluate-command "${…}"`
to its `command:`. The fixture experiment directory is mounted into
each subprocess-mode host via a new `eden-experiment-dir` config-
mount or named volume; setup-experiment grows a flag
`--experiment-dir <path>` to copy the user's experiment dir into
the volume.

Compose-side flag plumbing uses the same `${VAR:?}` pattern already
in use; `${EDEN_PLAN_COMMAND:-}` etc. default to empty so the
scripted-mode default is unchanged.

The compose-smoke job stays in scripted mode (the merged shape from
10c) **plus** a new `compose-smoke-subprocess` matrix entry that
runs the same end-to-end assertions in subprocess mode against the
fixture scripts. Two CI signals: scripted protects the chunk-1–9
behavior; subprocess protects this chunk's protocol.

## Out of scope for this chunk

- **Container isolation** of `*_command` subprocesses. Deferred to
  a 10d follow-up sub-chunk. When that lands, the subprocess.Popen
  call is replaced with a `docker run` invocation; the worktree
  becomes a bind-mount; otherwise the protocol is unchanged.
- **Experiment-specific image** build pipeline. Same follow-up.
- **Auth-secret mounting** (Claude/Codex CLI auth dirs into the
  sandbox). Same follow-up.
- **Conformance test for the protocol.** The protocol is reference-
  specific (`spec/v0/09-…` is informative); Phase 11 conformance
  tests cover the spec-normative state machine, not the subprocess
  invocation shape.
- **Gitea integration.** Workers continue to use the Compose-shared
  bare repo. (Same posture 10b/c took.)
- **Push-driven event delivery** to subprocess workers. Plan
  dispatch is host-pull (host claims, then writes to subprocess
  stdin); reactive notifications about completed trials reach the
  planner only via the `history` field on the next plan task. A
  push-style trial-completed notification line is a Phase-12 item.

## Risks

- **Fixture-script drift.** The fixture `plan.py`/`implement.py`/
  `eval.py` must produce the same end state as the scripted hosts
  for `compose-smoke-subprocess` to pass with the same assertions.
  Mitigation: factor the deterministic shape (blob payload,
  metric values) into a small shared helper imported by both the
  fixture scripts and `eden_service_common.scripted` so divergence
  is mechanically prevented.
- **`git worktree` semantics on the bare repo.** Worktrees over a
  bare repo work but require care — the worktree's `.git` is a
  text file pointing at the bare repo's `worktrees/<name>/`
  directory. Permissions on the bare-repo volume need to allow the
  host user to write there. The 10b/10c volume already grants this
  to the `eden` user; verify with the new test before relying on
  it.
- **Worktree leak on host crash.** Mitigation in §D.5 (`git
  worktree prune` at host startup), but if a single bare repo is
  shared across multiple host containers (planner + implementer +
  web-ui), a prune in one might race a worktree-add in another.
  Mitigation: each host writes its worktrees under
  `<worktrees_dir>/<container_hostname>/<task_id>/` and only prunes
  its own subtree. Document this in the helper module.
- **Long-running planner subprocess as a SPOF.** A crashed Claude
  CLI or a stuck process kills planner availability for the whole
  experiment. Mitigation: respawn-on-crash (subject to a max
  respawn count) is in scope for this chunk; cross-restart session
  resumption (Claude `-c` after host restart) is a user-script
  concern.
- **Stdout-as-protocol fragility.** Anything the user's script
  prints to stdout that *looks* like a JSON event line could spoof
  the protocol. Mitigation: every event line carries the host-
  supplied `task_id`; the host rejects lines whose `task_id`
  doesn't match the currently dispatched task. Sentinel `ready`
  has no `task_id` and is only valid before the first dispatch.
  Document the spoofing risk in the protocol README.
- **`history` field unboundedness.** Plan tasks late in a long
  experiment carry a `history` array proportional to trial count.
  Mitigation for this chunk: cap at the last 50 completed trials,
  newest first; document the cap in the protocol README. Future
  refinement (windowing strategies, summarization) is a user
  script concern.
