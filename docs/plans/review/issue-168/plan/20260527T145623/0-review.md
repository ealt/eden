**Missing Context**

Assessment: the problem is well-defined, and the main grounding claims check out. The ideator subprocess host already writes `artifacts/ideas/<idea_id>/content.md` ([subprocess_mode.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/reference/services/ideator/src/eden_ideator_host/subprocess_mode.py:300)), the serve route is path-jail based rather than layout-aware ([artifacts.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/reference/services/web-ui/src/eden_web_ui/routes/artifacts.py:149)), the admin listing walks recursively ([admin_artifacts.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/reference/services/web-ui/src/eden_web_ui/routes/admin_artifacts.py:57)), and the web UI executor does not write artifact bytes today ([executor.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/reference/services/web-ui/src/eden_web_ui/routes/executor.py:676)).

- The plan still leaves one load-bearing naming choice unspecified: what exact on-disk name an idea with uploads should use inside `ideas/<idea_id>/`. [issue-168-hierarchical-artifacts-substrate.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/docs/plans/issue-168-hierarchical-artifacts-substrate.md:169) resolves evaluator naming, but not the single-upload / bundled-idea cases that already exist.

**Feasibility**

Assessment: the hierarchical layout is feasible, but the helper refactor in the plan is not implementable as written.

- D.1 says to pass an entity-scoped base dir into `predict_artifact_uri` / `write_artifact_bundle` ([plan](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/docs/plans/issue-168-hierarchical-artifacts-substrate.md:140)), but those helpers currently choose the basename themselves: text-only is always `<artifact_id>.md`, one-upload is `<artifact_id><ext>`, and bundles are `<artifact_id>.tar.gz` ([artifacts.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/reference/services/web-ui/src/eden_web_ui/artifacts.py:116), [artifacts.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/reference/services/web-ui/src/eden_web_ui/artifacts.py:151)). Passing `ideas/<idea_id>/` would produce `ideas/<idea_id>/<idea_id>.md`, not `ideas/<idea_id>/content.md`. The plan needs a full filename/layout policy, not just a directory helper.
- The plan should explicitly keep prediction and write in lockstep. Today `predict_artifact_uri()` hardcodes `content.md` for bundle collision checks ([artifacts.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/reference/services/web-ui/src/eden_web_ui/artifacts.py:143)), while the ideator route writes `idea.md` ([ideator.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/reference/services/web-ui/src/eden_web_ui/routes/ideator.py:393)). A layout refactor that does not unify that interface will preserve a real mismatch.

**Alternatives**

Assessment: the layout choice looks right; the better alternative is about helper placement.

- If the goal is one shared Python path-builder, I would decide up front to put it in `eden_service_common`, not `eden_web_ui.artifacts`. The plan already recognizes the import-boundary problem around the ideator host ([plan](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/docs/plans/issue-168-hierarchical-artifacts-substrate.md:330)). Making a worker host depend on the web UI package just to compute a filesystem path is the wrong dependency direction.

Assessment: the plan covers the big writers, but its explicit touch list is still incomplete.

- The test inventory understates how many places assume the old shape. The plan says `test_ideator_subprocess.py` likely already asserts `ideas/<idea_id>/content.md` ([plan](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/docs/plans/issue-168-hierarchical-artifacts-substrate.md:354)), but it currently does not ([test_ideator_subprocess.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/reference/services/ideator/tests/test_ideator_subprocess.py:88)). **Edge Cases And Risks**

Assessment: the main remaining risk is treating “UUID-based uniqueness” as equivalent to the spec’s no-overwrite requirement.

- The plan says the unique-stem hybrid neutralizes §5.4 “by construction” ([plan](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/docs/plans/issue-168-hierarchical-artifacts-substrate.md:417)), but both writers still overwrite an existing target if the same `artifact_id` is reused: the web UI uses `os.replace()` ([artifacts.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/reference/services/web-ui/src/eden_web_ui/artifacts.py:78), [artifacts.py](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/reference/services/web-ui/src/eden_web_ui/artifacts.py:237)), and `eden-manual` writes the tarball directly to the final path ([eden-manual](/Users/ericalt/Documents/eden-worktrees/plan-issue-168-artifacts-layout/reference/scripts/manual-ui/eden-manual:400)). If §5.4 is the design driver, the plan should require an explicit refuse-to-overwrite check or exclusive-create path.

Overall assessment: the plan is well-researched and most of its factual claims are correct, but it is not implementation-ready yet. One revision should pin the full filename policy, redesign the helper interface around full target paths rather than just directories, name the actual reader/test surfaces that must change, and fix the roadmap/no-overwrite points. After that, the approach looks sound.
