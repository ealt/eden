**Findings**

1. **`compose.logging-infra.yaml` is in scope but has no explicit validation path.** The plan now treats the socket-gated infra overlay as a first-party deliverable, but the execution gates only validate `compose.logging.yaml` and `smoke-logging.sh`; nothing exercises or even statically merges `compose.logging-infra.yaml`. That leaves the privileged overlay vulnerable to drift in `group_add`, socket mount wiring, or the alternate Alloy config with no planned catch point. At minimum, add a `docker compose -f compose.yaml -f compose.logging.yaml -f compose.logging-infra.yaml --env-file .env config >/dev/null` gate; ideally add a dedicated smoke when Docker-socket access is available. [docs/plans/issue-110-loki-grafana-overlay.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-110-loki/docs/plans/issue-110-loki-grafana-overlay.md:108) [docs/plans/issue-110-loki-grafana-overlay.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-110-loki/docs/plans/issue-110-loki-grafana-overlay.md:134) [docs/plans/issue-110-loki-grafana-overlay.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-110-loki/docs/plans/issue-110-loki-grafana-overlay.md:196)

2. **The infra-overlay `EDEN_DOCKER_GID` story still fails too softly.** The plan says non-docker-mode operators can set `EDEN_DOCKER_GID` manually, but today `setup-experiment.sh` writes `EDEN_DOCKER_GID=0` by default, so the overlay won’t fail fast on a missing value; it will come up with a plausible-looking but often wrong gid and then fail at runtime with Docker-socket permission errors. Since this overlay is intentionally privilege-sensitive, it should force an explicit correct gid rather than inherit a misleading default. A separate `${EDEN_LOGGING_DOCKER_GID:?…}` requirement in `compose.logging-infra.yaml`, or unconditional gid probing in setup, would make the failure mode much clearer. [docs/plans/issue-110-loki-grafana-overlay.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-110-loki/docs/plans/issue-110-loki-grafana-overlay.md:116) [reference/scripts/setup-experiment/setup-experiment.sh](/Users/ericalt/Documents/eden-worktrees/plan-issue-110-loki/reference/scripts/setup-experiment/setup-experiment.sh:402)

3. **The planned Alloy run command should make an explicit choice about usage reporting.** The proposed `alloy run` command omits `--disable-reporting`, and Alloy enables anonymous usage reporting by default. For a local reference stack, that is surprising outbound behavior unless the repo consciously wants it. I’d either add `--disable-reporting` to the planned command or explain why the default reporting posture is acceptable here. **Alternatives**
The main architectural choice now looks right: Alloy over Promtail, JSONL tailing over default socket discovery, and a separate privileged overlay for infra logs. The only minor simplification worth considering is using `loki.source.file`’s built-in file matching instead of a standalone `local.file_match` in the base config unless target reuse is actually needed. Source: https://grafana.com/docs/grafana-cloud/send-data/alloy/reference/components/local/local.file_match/

**Completeness**
The remaining completeness gap is the privileged overlay lifecycle: it needs its own validation story and a sharper env-var contract. Everything else reads substantially more complete than the previous draft.

**Edge Cases / Risks**
The biggest residual operational risks are silent drift in the infra overlay and confusing socket-permission failures when `EDEN_DOCKER_GID` is wrong. The unaddressed Alloy reporting default is smaller, but it is exactly the kind of surprise that should be decided explicitly in a reference deployment.

**Overall assessment**
This is close. The Promtail blocker is resolved, and the split between default logging and socket-requiring infra capture is a material improvement. I’d want the infra overlay validation path and `EDEN_DOCKER_GID` fail-fast story tightened before calling the plan converged.
