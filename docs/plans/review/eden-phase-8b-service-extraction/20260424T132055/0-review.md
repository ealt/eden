**Missing Context**

Assessment: significant concerns. I would stop here before evaluating feasibility, alternatives, completeness, or later risks, because the plan still leaves the core cross-process execution model underdefined.

- `CRITICAL` The plan never defines how subprocess workers receive the scripted behavior it depends on. [Plan lines 247-264](/Users/ericalt/Documents/eden/docs/plans/eden-phase-8b-service-extraction.md:247) say the hosts construct `ScriptedPlanner` / `ScriptedImplementer` / `ScriptedEvaluator` from shared scripted functions, and [lines 437-450](/Users/ericalt/Documents/eden/docs/plans/eden-phase-8b-service-extraction.md:437) say the E2E boots them via `python -m ...`; but the CLI only carries URLs, IDs, intervals, and prefixes. Today those workers require injected Python callables/factories at construction time ([workers.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/workers.py:79), [workers.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/workers.py:145), [workers.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/workers.py:216)). The plan needs a concrete cross-process config surface, or it needs to say 8b uses one fixed built-in script profile.

- `CRITICAL` The implementer host’s git/repo model is undefined, but the plan also expects real integration. The worker CLI has no repo/worktree path ([plan lines 162-182](/Users/ericalt/Documents/eden/docs/plans/eden-phase-8b-service-extraction.md:162), [lines 230-245](/Users/ericalt/Documents/eden/docs/plans/eden-phase-8b-service-extraction.md:230)), and the algorithm explicitly says the scripted implementer returns a deterministic fake `commit_sha` ([lines 511-513](/Users/ericalt/Documents/eden/docs/plans/eden-phase-8b-service-extraction.md:511)). That conflicts with the real `Integrator`, which requires a real branch and reachable commit in the repo ([integrator.py](/Users/ericalt/Documents/eden/reference/packages/eden-git/src/eden_git/integrator.py:210)). The current real-integrator E2E only works because it prebuilds actual work branches and returns those real SHAs ([test_end_to_end.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/tests/test_end_to_end.py:238)). The plan needs to choose and document one model.

- `SUGGESTION` ID allocation across restarts is underspecified. The plan exposes prefixes for proposals/trials/tasks ([plan lines 169-174](/Users/ericalt/Documents/eden/docs/plans/eden-phase-8b-service-extraction.md:169), [lines 242-243](/Users/ericalt/Documents/eden/docs/plans/eden-phase-8b-service-extraction.md:242)), but not the actual uniqueness scheme. In-process tests currently rely on injected iterators. A restarted standalone worker that starts counting from `proposal-1` or `trial-1` again will collide with persisted state. Define the allocator now, even if it is only a reference-only scheme.

- `SUGGESTION` The informative auth addendum is phrased ambiguously. [Plan lines 340-349](/Users/ericalt/Documents/eden/docs/plans/eden-phase-8b-service-extraction.md:340) say conforming servers “MAY, but MUST NOT, adopt this shape,” which reads as contradictory rather than clarifying. Rewrite it so a reader can tell whether the intent is “optional” or “discouraged but permitted.”

**Overall Assessment**

The direction is sensible, but the plan is not implementation-ready yet. The main gap is that it does not fully specify how standalone worker processes are configured and how the implementer produces repo-valid trial commits; until that is nailed down, the subprocess E2E path is not reviewable as a workable Phase 8b design.
