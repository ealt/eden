**Findings**
- High: the `termination_policy.duration` design is not implementable as written. The plan says strict Pydantic models can take ISO-8601 duration strings directly ([issue-157-cli-flags-to-config.md](/Users/ericalt/Documents/eden-worktrees/issue-157-plan/docs/plans/issue-157-cli-flags-to-config.md:119), [issue-157-cli-flags-to-config.md](/Users/ericalt/Documents/eden-worktrees/issue-157-plan/docs/plans/issue-157-cli-flags-to-config.md:141)), but `ExperimentConfig` models in this repo use `ConfigDict(strict=True)` ([config.py](/Users/ericalt/Documents/eden-worktrees/issue-157-plan/reference/packages/eden-contracts/src/eden_contracts/config.py:74)), and a local `uv run python` check shows `timedelta` rejects `"PT2H"` in strict mode. The plan needs a concrete replacement: either a custom validated string type in `eden_contracts._common` plus schema `format: duration`, or keep the field as a validated string and convert in `build_termination_policy()`.

- High: the multi-experiment story is internally inconsistent. The plan says no multi-experiment rework is in scope and that multi-experiment mode still has a single `--experiment-config` source ([issue-157-cli-flags-to-config.md](/Users/ericalt/Documents/eden-worktrees/issue-157-plan/docs/plans/issue-157-cli-flags-to-config.md:49)), but the current orchestrator has no `--experiment-config` flag at all in multi-experiment mode and still resolves `termination_policy` only from CLI ([cli.py](/Users/ericalt/Documents/eden-worktrees/issue-157-plan/reference/services/orchestrator/src/eden_orchestrator/cli.py:366), [cli.py](/Users/ericalt/Documents/eden-worktrees/issue-157-plan/reference/services/orchestrator/src/eden_orchestrator/cli.py:486), [multi_loop.py](/Users/ericalt/Documents/eden-worktrees/issue-157-plan/reference/services/orchestrator/src/eden_orchestrator/multi_loop.py:63)). If `--termination-policy` is dropped as planned ([issue-157-cli-flags-to-config.md](/Users/ericalt/Documents/eden-worktrees/issue-157-plan/docs/plans/issue-157-cli-flags-to-config.md:214)), control-plane deployments lose any configuration path. The plan needs to either narrow scope to single-experiment orchestrator only, or explicitly add the control-plane/config-registry plumbing.

- High: `ideas_per_ideation` is not actually promoted across the real ideator paths. The plan accepts scripted mode staying fixed at `1` and unconfigurable ([issue-157-cli-flags-to-config.md](/Users/ericalt/Documents/eden-worktrees/issue-157-plan/docs/plans/issue-157-cli-flags-to-config.md:226)), while also deleting the setup-experiment flag that currently feeds subprocess deployments ([issue-157-cli-flags-to-config.md](/Users/ericalt/Documents/eden-worktrees/issue-157-plan/docs/plans/issue-157-cli-flags-to-config.md:245)). But the base compose deployment runs the ideator in scripted mode with no experiment-config at all ([compose.yaml](/Users/ericalt/Documents/eden-worktrees/issue-157-plan/reference/compose/compose.yaml:316)), and subprocess ideation still gets its count through `EDEN_IDEAS_PER_IDEATION` ([compose.subprocess.yaml](/Users/ericalt/Documents/eden-worktrees/issue-157-plan/reference/compose/compose.subprocess.yaml:217), [cli.py](/Users/ericalt/Documents/eden-worktrees/issue-157-plan/reference/services/ideator/src/eden_ideator_host/cli.py:65)). As written, the new YAML field would be ignored in the default deployment and stop reaching subprocess children. This needs a concrete end-to-end wiring plan, or `ideas_per_ideation` should stay out of scope for this chunk.

- Medium: the validation plan contradicts itself on where `termination_policy required when dispatch_mode.termination == "auto"` is enforced. Section 3.2 says Pydantic-only because JSON Schema is awkward ([issue-157-cli-flags-to-config.md](/Users/ericalt/Documents/eden-worktrees/issue-157-plan/docs/plans/issue-157-cli-flags-to-config.md:177)), but the risks section later assumes schema-side `if/then` validation exists and parity will compare both sides ([issue-157-cli-flags-to-config.md](/Users/ericalt/Documents/eden-worktrees/issue-157-plan/docs/plans/issue-157-cli-flags-to-config.md:453)). Given the repo’s schema/model parity discipline, this should be specified once and done on both sides.

**Level Assessments**
- Missing context: not yet solid. The plan does not accurately describe which runtime modes already consume experiment config, especially orchestrator multi-experiment mode and the default scripted ideator path.
- Feasibility: significant concerns. One core type assumption is false (`strict=True` + `timedelta`), and two promoted knobs lose their actual runtime wiring when the current flags/env plumbing is removed.
- I did not evaluate alternatives, completeness, or edge cases in depth because these feasibility issues need to be resolved first.

**Overall Assessment**
The plan is not ready for implementation yet. It has the right general intent, but it needs another revision that reconciles the current runtime topology first: define a real duration type, decide whether multi-experiment orchestrator support is in or out, and make `ideas_per_ideation` genuinely reachable in both scripted and subprocess deployments.
