# Phase 10d follow-up A â€” container isolation of `*_command` subprocesses

## Context

Chunk 10d shipped `--mode subprocess` for the planner / implementer / evaluator
hosts: each host invokes a user-supplied command from
`experiment-config.yaml` (`plan_command` / `implement_command` /
`evaluate_command`) via `subprocess.Popen(shell=True)`. The
host supplies `EDEN_EXPERIMENT_DIR`, `EDEN_TASK_JSON`, `EDEN_OUTPUT`,
and (for implement / evaluate) `EDEN_WORKTREE`. Stdout/stderr are
piped (long-running JSON-line protocol for planner; per-task short-
lived for implementer / evaluator).

The shipped surface runs the user's command **in the worker host
container itself** â€” so a malicious or buggy `implement_command`
inherits the worker host's filesystem (writable bare repo + worktree
volume + artifacts), env (shared-token bearer, all `EDEN_*` vars),
and process namespace. The 10d plan deferred container isolation to
this follow-up; the subprocess.Popen call is the wrappable layer and
the JSON-line protocol stays unchanged when wrapped.

This plan covers **only** isolation. Sub-chunk B (Gitea-as-actual-
git-remote so workers don't share a writable bare-repo volume) is a
separate plan.

## Design

### D.1 Where to inject containerization

The 10d plan hinted at "wrap `subprocess.Popen` with `docker run`" as
the wrappable layer. There are three reasonable injection points:

1. **Inside `subprocess_runner.spawn()`** â€” make the helper itself
   container-aware.
2. **At each call site** â€” each role's `subprocess_mode.py`
   transforms the command before passing to `spawn`.
3. **At config-construction time in each host's `cli.py`** â€” build
   the wrapped command once, pass through the existing
   `*SubprocessConfig.command` field unchanged.

We pick **2/3 mixed**: the planner's command is built once at CLI
time (long-running subprocess, cwd = experiment dir, no per-task
state in the wrap). The implementer / evaluator wraps must be built
per task (cwd = per-task worktree path), so they live in
`subprocess_mode.py`'s `_run_subprocess`. The shared
command-builder + cidfile-lifecycle helpers go into a new
`reference/services/_common/src/eden_service_common/container_exec.py`
module.

### D.2 Wrap shape and DooD mount strategy

**The mount paths the wrap emits are seen by the host docker
daemon, NOT by the worker host container.** Under DooD (Docker
outside of Docker), the worker calls `/var/run/docker.sock`, which
talks to the *host* daemon, which resolves all `--mount source=â€¦`
references against its own filesystem. So a naĂŻve `-v <cwd>:<cwd>`
where cwd lives on a docker-managed named volume **does not work**:
the host daemon sees the named volume name in its catalog, not the
mount path inside the worker.

Two correct ways to give the spawned child container access to the
same files the worker host sees:

1. **Forward each named volume by name** (`--mount type=volume,
   source=<vol-name>,target=<path>`). The host daemon resolves the
   volume directly from its catalog. The child sees the same files
   as the worker because both mount the same underlying volume.
2. **Forward bind-mounts by host-side absolute path**
   (`--mount type=bind,source=<host-abs-path>,target=<path>`).
   Used for the experiment dir, where the host-side path is already
   captured by setup-experiment as `EDEN_EXPERIMENT_DIR_HOST`.

We use both, driven by deployment-supplied env vars rather than
hard-coded constants in code. The host CLI accepts:

- `--exec-volume <vol-name>:<target-path>[:ro]` (repeatable)
- `--exec-bind <host-abs-path>:<target-path>[:ro]` (repeatable)

The compose overlay sets these from the existing volume names
(`eden-bare-repo`, `eden-worktrees`, `eden-artifacts-data`) and from
`EDEN_EXPERIMENT_DIR_HOST`. Mount targets inside the child match
the worker host's own targets exactly â€” `/var/lib/eden/repo`,
`/var/lib/eden/worktrees`, `/var/lib/eden/artifacts`,
`/etc/eden/experiment-dir` â€” so the worker-internal paths the
worker passed via env (`EDEN_TASK_JSON`, `EDEN_OUTPUT`,
`EDEN_WORKTREE`, `EDEN_EXPERIMENT_DIR`) resolve to the same files
inside the child.

This is critical for git: the implementer's worktree under
`/var/lib/eden/worktrees/<host>/<task_id>/` contains a `.git` file
pointing into `/var/lib/eden/repo`. The child needs *both* the
worktrees volume AND the bare-repo volume mounted at those exact
paths or `git` inside the child fails. The plan's wrap mounts
both. (For the evaluator, ditto. The planner doesn't need either â€”
it only needs the experiment-dir bind.)

The wrap shape per task:

```bash
docker run --rm -i --init \
  --cidfile <unique-cidfile> \
  --label eden.host=<container-hostname> \
  --label eden.task_id=<task-id-or-host-id-for-planner> \
  --label eden.role=<role> \
  --mount type=volume,source=eden-bare-repo,target=/var/lib/eden/repo \
  --mount type=volume,source=eden-worktrees,target=/var/lib/eden/worktrees \
  --mount type=volume,source=eden-artifacts-data,target=/var/lib/eden/artifacts,readonly \
  --mount type=bind,source=<EDEN_EXPERIMENT_DIR_HOST>,target=/etc/eden/experiment-dir,readonly \
  -w <cwd-target-path-inside-container> \
  -e EDEN_TASK_JSON -e EDEN_OUTPUT -e EDEN_EXPERIMENT_DIR \
  [-e EDEN_WORKTREE]              # implementer/evaluator only
  [-e <USER-ENV-KEY>]â€¦             # one per parsed --*-env-file key
  <image> \
  bash -lc '<original-command>'
```

The exact set of `--mount` entries per role:

| role | mounts | cwd inside child |
|---|---|---|
| planner | `experiment-dir` bind ro | `/etc/eden/experiment-dir` |
| implementer | `eden-bare-repo` rw, `eden-worktrees` rw, `eden-artifacts-data` ro, `experiment-dir` bind ro | `/var/lib/eden/worktrees/<host>/<task_id>/` |
| evaluator | `eden-bare-repo` rw, `eden-worktrees` rw, `experiment-dir` bind ro | `/var/lib/eden/worktrees/<host>/<task_id>/` |

(The implementer's `eden-artifacts-data` mount is read-only â€” same
posture as today's worker host gets per `compose.subprocess.yaml`,
because the implementer reads the planner-written rationale only.
The evaluator does not get the artifacts mount because the chunk-9d
evaluator subprocess does not consult planner artifacts.)

### D.3 Container lifecycle and cidfile discipline

Three concerns: (a) reliable cleanup on SIGKILL, (b) reliable cleanup
on the happy path, (c) recovery from worker-host crash mid-task.

**(a) SIGKILL escalation.** `subprocess_runner.terminate()` already
sends SIGTERM â†’ wait â†’ SIGKILL. The local `docker run` process
forwards SIGTERM to the container (PID 1 = `tini` via `--init`),
which gives the workload `shutdown_deadline` to exit. On SIGKILL
escalation, the local docker-run client is killed but the container
keeps running until the workload exits or is killed externally.

Fix: `Subprocess.terminate()` gains an optional
`post_kill_callback: Callable[[], None] | None = None`. When set,
it runs **after** the SIGKILL escalation branch (AFTER `os.killpg`
of the local docker-run process group). The container_exec module
sets this callback to:

```python
def _kill_via_cidfile(cidfile: Path) -> None:
    cid = cidfile.read_text().strip() if cidfile.is_file() else ""
    if cid:
        subprocess.run(["docker", "kill", cid], check=False, timeout=10)
        subprocess.run(["docker", "rm", "-f", cid], check=False, timeout=10)
```

`check=False` because the container may already be gone via `--rm`.

**(b) Happy-path cleanup.** `--rm` removes the container on its own
exit, but the cidfile on the host filesystem stays. Reuse of a
cidfile path is fatal: `docker run --cidfile <path>` errors if the
file exists. So:

- Cidfile paths are unique per spawn: `<cidfile_dir>/<role>-<spawn-
  uuid>.cid`. (Not derived from task id â€” a reclaimed task may
  re-spawn after orphan-reaping, and we want a fresh path.)
- `Subprocess` gains a `cleanup_callbacks: list[Callable[[], None]]`
  field. The container_exec spawn helper appends a small
  `lambda: cidfile.unlink(missing_ok=True)` to it. Both the SIGTERM-
  ladder happy path inside `terminate()` and the call sites that
  drain the subprocess to natural EOF (the implementer/evaluator
  `_run_subprocess` while-loop) call a new
  `Subprocess.run_cleanups()` after `popen.wait()` completes.
  Idempotent â€” safe to call from multiple paths.

**(c) Worker-host crash mid-task.** A startup-time helper
`reap_orphaned_containers(*, role, host)` runs:

```bash
docker ps -aq \
  --filter label=eden.role=<role> \
  --filter label=eden.host=<host>
```

and `docker rm -f` everything that comes back. The
`eden.host=<container-hostname>` filter scopes to the calling
worker host's hostname only â€” mirrors the cross-host worktree
isolation already in place at chapter 6 of
`spec/v0/reference-bindings/worker-host-subprocess.md`. It also
removes any leftover cidfiles in the cidfile dir that don't
correspond to a live container (best-effort; non-fatal).

### D.4 Image strategy

Two layers, in priority order:

1. **Experiment-specific image** (preferred, optional) â€”
   `setup-experiment.sh` checks for a `Dockerfile` in
   `EDEN_EXPERIMENT_DIR`. If present, it builds:

   ```bash
   docker build -t "eden-experiment-${EXPERIMENT_ID}:dev" \
     "$EDEN_EXPERIMENT_DIR"
   ```

   and writes `EDEN_EXEC_IMAGE=eden-experiment-${EXPERIMENT_ID}:dev`
   into `.env`.

2. **Default runtime image** â€” if no experiment Dockerfile, fall back
   to a small `eden-runtime:dev` image we build alongside
   `eden-reference:dev`. This image has python3 + git + bash +
   ca-certificates and is meant for fixture experiments that don't
   need exotic deps.

The default is built from a new
`reference/compose/Dockerfile.runtime`. It is *separate* from
`reference/compose/Dockerfile` (the `eden-reference:dev` image, which
must stay tightly scoped to the EDEN service code).

The CLI flag is `--exec-image <name>`, required when
`--exec-mode docker` is selected and falling back to
`$EDEN_EXEC_IMAGE` if unset (to make the compose overlay symmetric).

### D.5 Auth-secret flow and the boundary it does NOT provide

User secrets (LLM API keys etc.) flow via the `--*-env-file` flags.
With containerization, the file is read on the worker host as
before, parsed by `parse_env_file`, and merged into the spawn env.
The wrap lists each parsed key as a `-e KEY` flag (no `=value`),
so docker copies it from the docker-run process's env. Net effect:
the secret never appears on the docker run command line (so it
doesn't show in `ps` output on the docker daemon's host or in shell
history of the worker host).

What this **does NOT do**:

- Secrets ARE visible in `docker inspect <container>` output to
  anyone with daemon access.
- Under DooD (the reference deployment), any other `*_command`
  running concurrently on the same host has full daemon access via
  the shared socket and can therefore read every other container's
  inspect output. So `*_command` instances are NOT isolated from
  each other's secrets.
- A malicious `*_command` can `docker run` arbitrary containers,
  read host files via volume mounts, etc. â€” DooD with a shared
  socket is a soft security boundary against bugs, not a hard one
  against malice.

This is documented plainly in Â§7 of the reference-binding chapter
update. The reference deployment trades that boundary for
operational simplicity; hardened deployments substitute sysbox /
Kata / a dedicated docker daemon per worker (invisible to EDEN
because the host just calls `docker run` either way).

### D.6 Docker socket access (DooD) â€” permissions

The current `eden-reference:dev` image runs as `USER eden` (uid
1000). The host docker socket on Linux is conventionally
`root:docker` mode `0660` â€” so a bare `eden` user inside the
container cannot use it.

Two options:

1. **GID-matching `group_add`** â€” at compose-load time, set
   `EDEN_DOCKER_GID` to the host's docker socket gid (read with
   `stat -c '%g' /var/run/docker.sock`) and use
   `group_add: ["${EDEN_DOCKER_GID}"]` on each subprocess-mode
   service. The container's `eden` user inherits that gid as a
   supplementary group â†’ the socket is rw without changing the
   user.
2. **`user: root`** â€” simpler, but expands the worker container's
   privilege. Acceptable for the reference because (per Â§D.5) DooD
   is already a soft security boundary, but option 1 is preferred.

We pick **option 1**. setup-experiment.sh gains:

```bash
if [[ "$EXEC_MODE" = "docker" ]]; then
    if [[ -S /var/run/docker.sock ]]; then
        EDEN_DOCKER_GID="$(stat -c '%g' /var/run/docker.sock 2>/dev/null \
            || stat -f '%g' /var/run/docker.sock)"
    else
        echo "setup-experiment: --exec-mode docker requires /var/run/docker.sock" >&2
        exit 2
    fi
fi
```

(`stat -c` is GNU; `stat -f` is BSD/macOS â€” try GNU first because
that's what GitHub-hosted runners and Linux developers have.)

`EDEN_DOCKER_GID` is written into `.env`. The compose overlay's
`group_add: ["${EDEN_DOCKER_GID:?}"]` resolves it at compose-up
time. On macOS / Docker Desktop, the socket forwarder typically
exposes the socket with the user's gid via vmnetd; the same `stat`
call returns it.

### D.7 What does NOT change

- The worker-host-subprocess **JSON-line protocol** is unchanged.
  Implementations of `plan.py` / `implement.py` / `eval.py` written
  against the chunk-10d reference binding work identically when
  wrapped.
- Spec normative chapters 03 and 04 are unchanged.
- `subprocess_runner.spawn()` signature stays back-compat (new
  optional kwarg only).
- The `--mode scripted` path is untouched (no docker dependency for
  scripted profiles; the existing `compose-smoke` job stays green).

## G. Out of scope for this sub-chunk

- **Network policy on subprocess containers** â€” `--network` is
  default. Operators who want network egress restricted set it via
  the experiment image or compose override.
- **Resource limits** (`--memory` / `--cpus` / `--pids-limit`).
  Same: operator-configured via the image or compose override.
- **Non-root execution inside the subprocess container.** The
  default `eden-runtime:dev` image runs as root; experiments that
  want a non-root uid set `USER` in their experiment Dockerfile.
- **DinD instead of DooD.** The wrap shape is the same; only the
  daemon socket origin differs. Out of scope.
- **Hardening the DooD security boundary** (sysbox, Kata, per-
  worker daemons). Out of scope; documented as the reference
  posture.
- **Sub-chunk B (Gitea-as-actual-git-remote).** Independent plan;
  this work doesn't change how implementers reach the bare repo.

## H. Risks / things to watch

- **Mount-target collision with worker host's own cwd.** If a
  worker host's cwd happens to fall outside any forwarded mount,
  the spawned child sees an empty `cwd`. This isn't possible in
  the planned compose deployment (every cwd is on a forwarded
  mount), but the wrap should reject such configs at build time
  for safety: `wrap_command` validates that `cwd_target` falls
  under at least one mount target, and raises `ValueError`
  otherwise.
- **Stale containers on host crash.** Mitigated by the
  `eden.host=<hostname>` label + startup-time
  `reap_orphaned_containers`. Cidfile dir is host-side mounted so
  cidfiles survive worker-host restart.
- **`docker run --cidfile` errors on existing path.** Cidfile paths
  are unique-per-spawn (uuid suffix); never reused. The reaper
  best-effort-removes orphaned cidfiles whose container is gone.
- **`bash -lc` and login-shell side effects.** A user's `~/.bashrc`
  may fail in a fresh container (no home dir). Mitigation:
  `eden-runtime:dev` ships an empty `/root/.bashrc`. Experiment
  images that override `USER` should likewise.
- **DooD security boundary is soft.** Documented in Â§D.5 and Â§7 of
  the reference-binding chapter and called out in AGENTS.md as a
  review checkpoint.
- **macOS Docker Desktop GID quirk.** The docker socket inside the
  Linux VM that Desktop runs has a Linux gid (often 0 for root, or
  a desktop-specific gid). The `stat -f` fallback handles macOS-
  side reads from the user's perspective; the in-container side
  uses whatever gid is on the *bind-mounted* socket inside the
  worker. setup-experiment computes that gid from inside a brief
  `docker run --rm -v /var/run/docker.sock:/var/run/docker.sock
  alpine stat -c '%g' /var/run/docker.sock`-style probe â€” yields
  the gid the worker container will see, which is exactly what
  `group_add` needs.
