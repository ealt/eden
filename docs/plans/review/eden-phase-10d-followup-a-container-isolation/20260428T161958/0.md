# Phase 10d follow-up A — container isolation of `*_command` subprocesses

## Context

Chunk 10d shipped `--mode subprocess` for the planner / implementer / evaluator
hosts: each host invokes a user-supplied command from
`experiment-config.yaml` (`plan_command` / `implement_command` /
`evaluate_command`) via `subprocess.Popen(shell=True)`. The
host supplies `EDEN_EXPERIMENT_DIR`, `EDEN_TASK_JSON`, `EDEN_OUTPUT`,
and (for implement / evaluate) `EDEN_WORKTREE`. Stdout/stderr are
piped (long-running JSON-line protocol for planner; per-task short-
lived for implementer / evaluator).

The shipped surface runs the user's command **in the worker host
container itself** — so a malicious or buggy `implement_command`
inherits the worker host's filesystem (writable bare repo + worktree
volume + artifacts), env (shared-token bearer, all `EDEN_*` vars),
and process namespace. The 10d plan deferred container isolation to
this follow-up; the subprocess.Popen call is the wrappable layer and
the JSON-line protocol stays unchanged when wrapped.

This plan covers **only** isolation. Sub-chunk B (Gitea-as-actual-
git-remote so workers don't share a writable bare-repo volume) is a
separate plan.

## Design

### D.1 Where to inject containerization

The 10d plan hinted at "wrap `subprocess.Popen` with `docker run`" as
the wrappable layer. There are three reasonable injection points:

1. **Inside `subprocess_runner.spawn()`** — make the helper itself
   container-aware.
2. **At each call site** — each role's `subprocess_mode.py`
   transforms the command before passing to `spawn`.
3. **At config-construction time in each host's `cli.py`** — build
   the wrapped command once, pass through the existing
   `*SubprocessConfig.command` field unchanged.

We pick **3**. Justification: subprocess_runner is role-agnostic
(it just runs a shell command and pipes stdio); per-role
`subprocess_mode.py` already encapsulates the per-task lifecycle
(worktree create + cwd + env). The CLI is where mode selection and
deployment knobs already live (`--mode`, `--*-env-file`,
`--worktrees-dir`). Adding `--exec-mode` here keeps the wrapping
visible at a single layer per role.

A small new helper module
`reference/services/_common/src/eden_service_common/container_exec.py`
houses the `wrap_command(...)` builder + the cidfile-based lifecycle
helpers; the three CLIs call it identically.

### D.2 Wrapped-command shape (per task)

The wrapped shell command, when `--exec-mode docker` is selected, is:

```bash
docker run --rm -i --init \
  --cidfile <cidfile> \
  --label eden.host=<container-hostname> \
  --label eden.task_id=<task-id-or-host-id-for-planner> \
  --label eden.role=<role> \
  -v <cwd>:<cwd> \
  -w <cwd> \
  -v <experiment-dir>:<experiment-dir>:ro \
  -e EDEN_TASK_JSON -e EDEN_OUTPUT -e EDEN_EXPERIMENT_DIR \
  [-e EDEN_WORKTREE]              # implementer/evaluator only
  [-e <USER-ENV-KEY>]…             # one per parsed --*-env-file key
  <image> \
  bash -lc '<original-command>'
```

Notes:

- **`--rm`** removes the container on exit. Combined with
  `--init`, this gives clean SIGTERM → graceful shutdown.
- **`-i` only (no `-t`)** so we get a real stdin pipe (planner
  protocol needs this) and signal forwarding from `docker run`'s
  parent shell to the container's PID 1 works.
- **`--init`** runs `tini` as PID 1 so SIGCHLD reaping + signal
  forwarding behave correctly even if the user's command is a shell
  one-liner.
- **Bind-mounting cwd at the same path** keeps `EDEN_TASK_JSON` /
  `EDEN_OUTPUT` (which are relative paths the host writes/reads on
  the host filesystem) coherent inside the container without
  rewriting them.
- **Experiment-dir mounted RO** so `python3 ${EDEN_EXPERIMENT_DIR}/plan.py`
  resolves both inside and outside the container at the same absolute
  path. Read-only is fine: the user's *_command shouldn't write to
  the experiment directory; outputs go to `EDEN_OUTPUT` under cwd.
- **`-e KEY` (no `=value`)** copies the variable's current value
  from the docker-run-caller's env, which is exactly the env the
  host already builds. Result: the env passed to spawn() flows
  through to the container without us having to JSON-encode it.
- **`bash -lc`** — `-l` makes login behavior available (so a user's
  conda/venv hook works); `-c` is what `shell=True` would do.

### D.3 Container lifecycle around SIGKILL

`docker run`'s signal forwarding is graceful only on SIGTERM. If the
host's `Subprocess.terminate()` escalates to SIGKILL after the
shutdown deadline, the local `docker run` process dies but the
container keeps running until its workload exits or is killed
externally.

Fix: every wrapped invocation writes its container id to a per-task
cidfile, and `subprocess_runner.terminate()` gains an optional
`post_kill_callback` (set by `container_exec.spawn_wrapped`) that:

1. Reads the cidfile (if it exists).
2. Runs `docker kill <cid>` synchronously (best-effort; ignores
   ProcessLookupError-shaped failures so it's safe to call when the
   container already exited).
3. Removes the cidfile.

This is invoked **only** on the SIGKILL escalation branch; the happy
path (SIGTERM → graceful exit) leaves the cleanup to `--rm`.

In addition, the `container_exec` module exposes
`reap_orphaned_containers(*, role: str, host: str) -> None`,
called once at host startup. It runs:

```bash
docker ps -aq \
  --filter label=eden.role=<role> \
  --filter label=eden.host=<host>
```

and `docker rm -f` everything that comes back. This catches the case
where the worker host crashed mid-task last run and left a container
behind. Cross-host safety: the `eden.host=<container-hostname>`
filter scopes to this host's hostname only, mirroring the cross-host
worktree isolation (`<worktrees_root>/<hostname>/<task_id>/`)
already in place at chapter 6 of the worker-host-subprocess
reference binding.

### D.4 Image strategy

Two layers, in priority order:

1. **Experiment-specific image** (preferred, optional) —
   `setup-experiment.sh` checks for a `Dockerfile` in
   `EDEN_EXPERIMENT_DIR`. If present, it builds:

   ```bash
   docker build -t "eden-experiment-${EXPERIMENT_ID}:dev" \
     "$EDEN_EXPERIMENT_DIR"
   ```

   and writes `EDEN_EXEC_IMAGE=eden-experiment-${EXPERIMENT_ID}:dev`
   into `.env`.

2. **Default runtime image** — if no experiment Dockerfile, fall back
   to a small `eden-runtime:dev` image we build alongside
   `eden-reference:dev`. This image has python3 + git + bash and is
   meant for fixture experiments that don't need exotic deps.

The default is built from a new
`reference/compose/Dockerfile.runtime`. It is *separate* from
`reference/compose/Dockerfile` (the `eden-reference:dev` image, which
must stay tightly scoped to the EDEN service code).

The CLI flag is `--exec-image <name>`. It is required when
`--exec-mode docker` is selected and falls back to
`$EDEN_EXEC_IMAGE` if unset (to make the compose overlay symmetric).

### D.5 Auth-secret flow

User secrets (LLM API keys etc.) already flow via the
`--*-env-file` flags. With containerization, the file is read on
the host as before, parsed by `parse_env_file`, and merged into the
spawn env. The wrapped command lists each parsed key as a `-e KEY`
flag (no value), so docker copies it from the docker-run process's
env. Net effect: the secret never appears on the docker CLI command
line (which would leak into `ps` / `docker inspect`).

### D.6 Docker socket access (DooD)

The worker host containers must be able to reach the host docker
daemon. We mount `/var/run/docker.sock` read-write from the host
into each subprocess-mode worker host container. This is "Docker
outside of Docker" — the worker uses the host's daemon to spawn
sibling containers, not nested ones. It's the standard pattern;
it's also security-sensitive (the worker effectively has root on
the host docker daemon).

This is **acceptable for the reference** and is documented as a
caveat in the reference-binding chapter §7. A hardened deployment
would substitute sysbox / Kata / a dedicated docker daemon per
worker; that substitution is invisible to EDEN because the host
just calls `docker run` either way.

The `docker` CLI itself is added to `eden-reference:dev` (a new
`apt-get install -y docker.io` step in the Dockerfile, or the
official static binary copied in — pick whichever installs faster
in the existing builder stage).

### D.7 What does NOT change

- The worker-host-subprocess **JSON-line protocol** is unchanged.
  Implementations of `plan.py` / `implement.py` / `eval.py` written
  against the chunk-10d reference binding work identically when
  wrapped.
- Spec normative chapters 03 and 04 are unchanged.
- `subprocess_runner.spawn()` signature and behavior are unchanged
  except for the optional `post_kill_callback` parameter.
- The `--mode scripted` path is untouched (no docker dependency for
  scripted profiles; the existing `compose-smoke` job stays green).

## Out of scope for this sub-chunk

- **Network policy on subprocess containers** — `--network` is
  default (bridge / compose-default). Operators who want network
  egress restricted can set it via the experiment image or compose
  override.
- **Resource limits** (`--memory` / `--cpus` / `--pids-limit`).
  Same: operator-configured via the image or compose override.
- **Non-root execution inside the subprocess container.** The
  `eden-runtime:dev` image runs as root by default; experiments
  that want a non-root uid set `USER` in their experiment
  Dockerfile.
- **DinD instead of DooD.** Out of scope; the wrap shape is the
  same for either, only the daemon socket origin differs.
- **Sub-chunk B (Gitea-as-actual-git-remote).** Independent plan;
  this work doesn't change how implementers reach the bare repo.
- **Updating branch protection to require
  `compose-smoke-subprocess-docker`.** Same posture chunks 10a /
  10b / 10c / 10d / 10e took for newly-added compose jobs: ship
  the job, watch a few clean runs, then promote in a follow-up.

## Risks / things to watch

- **Stale containers on host crash.** Mitigated by the
  `eden.host=<hostname>` label + startup-time `reap_orphaned_containers`.
  But the cidfile dir on the worker container needs to be a tmpfs
  or a host-mounted volume — if it's only on the worker container's
  ephemeral filesystem, a `docker compose stop` of the worker host
  before SIGKILL escalation loses the cidfile. The plan mounts a
  host-side cidfiles dir; that closes the gap.
- **Bind-mounting cwd at the same path inside vs outside the
  container.** Works trivially when cwd is under
  `<worktrees_root>` (host-side absolute path =
  `EDEN_EXPERIMENT_DIR_HOST/...` or similar). For the planner whose
  cwd is the experiment dir itself, we bind-mount it at its own
  host-side path. This relies on the existing
  `EDEN_EXPERIMENT_DIR_HOST` env var setup-experiment writes (which
  is host-side, not worker-side) — which is already correct for the
  current bind mount, so we use it directly.
- **`docker run` waits on its own — the host's read_line loop must
  keep draining stdout.** The implementer/evaluator
  `_run_subprocess` already drains in a `while sub.is_alive()`
  loop with a `read_line` deadline; this stays unchanged. The
  planner protocol's stdin/stdout flow goes via `docker run -i`
  which forwards both pipes; no separate concern.
- **`bash -lc` and login-shell side effects.** A user's `~/.bashrc`
  may fail in a fresh container (no home dir). Mitigation:
  `eden-runtime:dev` ships an empty `/root/.bashrc`. Experiment
  images that override `USER` should likewise.
- **DooD security boundary.** Documented in §7 of the reference-
  binding chapter and called out in AGENTS.md as a review checkpoint
  (a `*_command` running as root on the host docker daemon).
