**Findings**
- Two live replicas sharing the same `worker_id` still break the lease-fencing model. The plan treats uniqueness as an operator requirement at [eden-phase-12c-control-plane.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12c-control-plane.md:382), but lease ownership is keyed only by `holder: worker_id` at [eden-phase-12c-control-plane.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12c-control-plane.md:315), and renew/release are authorized only by matching that holder at [eden-phase-12c-control-plane.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12c-control-plane.md:244). The restart-recovery logic then explicitly says a replica recovers leases “held by its own `worker_id`” at [eden-phase-12c-control-plane.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12c-control-plane.md:1181). With two replicas on the same ID, both are the same principal from the control plane’s perspective, so both can renew/release the same lease and both can “recover” it after restart. That is worse than a normal misconfiguration: it defeats the plan’s core exclusivity guarantee. 12a-1 only required uniqueness within a registry at [eden-phase-12a-1-worker-identity.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-1-worker-identity.md:122), and 12a-2 left uniqueness as operator discipline at [eden-phase-12a-2-orchestrator-as-role.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-2-orchestrator-as-role.md:575). In 12c that is no longer enough; the plan needs an explicit runtime defense or an explicit “unsupported and fail-fast” startup check.

- Terminated experiments still have no coherent lease-release path after integration drains. 12a-3’s contract is that once an experiment is `terminated`, only integration runs until it drains, then the loop has no further work and may exit at [eden-phase-12a-3-lifecycle-policy.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:449) and [eden-phase-12a-3-lifecycle-policy.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12a-3-lifecycle-policy.md:740). In 12c, the held replica does `run_integration_only(experiment_id)` when state is terminated at [eden-phase-12c-control-plane.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12c-control-plane.md:419), but there is no corresponding “drained, so release lease and stop owning this experiment” rule. That matters because `unregister_experiment` requires `last_known_state == "terminated"` and no active lease at [eden-phase-12c-control-plane.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12c-control-plane.md:306). As written, a healthy replica can renew a no-op lease forever on a fully drained terminated experiment, which blocks unregister indefinitely. Releasing the lease is also not sufficient by itself, because the acquisition thread re-scans all registered experiments and will reacquire any unheld one at [eden-phase-12c-control-plane.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-12c-control-plane.md:431). The plan needs an explicit post-drain rule for terminated experiments: either a skip/reacquire filter, an additional drained bit, or a different unregister/ownership contract.

**Overall Assessment**
Completeness is now in decent shape, and the control-plane partition and checkpoint-import partial-success cases read coherently. The remaining blockers are both true edge-condition design gaps: duplicate `worker_id` currently defeats lease exclusivity, and terminated experiments have no stable end-state between “integration drained” and “operator unregisters.”
