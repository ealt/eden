**Findings**
- `Bug` [reference/services/task-store-server/src/eden_task_store_server/cli.py](/Users/ericalt/Documents/eden-worktrees/impl-issue-122-baseline-variant/reference/services/task-store-server/src/eden_task_store_server/cli.py:94), [app.py](/Users/ericalt/Documents/eden-worktrees/impl-issue-122-baseline-variant/reference/services/task-store-server/src/eden_task_store_server/app.py:80), [test_e2e.py](/Users/ericalt/Documents/eden-worktrees/impl-issue-122-baseline-variant/reference/services/orchestrator/tests/test_e2e.py:293): the feature is not actually “default-on” for new experiments. `base_commit_sha` is only persisted when task-store-server is started with the new optional `--base-commit-sha`/`EDEN_BASE_COMMIT_SHA`; when that flag is omitted, the experiment is created without a seed and the orchestrator skips baseline creation. The test comment at `test_e2e.py:293` explicitly relies on that behavior. That contradicts the plan/spec posture that new experiments auto-create/evaluate a baseline unless `baseline.enabled: false`. The fix is to make seed persistence part of the authoritative experiment-creation path, not an optional server flag on some startup paths, and then update the standard startup/test entrypoints to always populate it for newly created experiments.

- `Bug` [reference/packages/eden-storage/src/eden_storage/_checkpoint.py](/Users/ericalt/Documents/eden-worktrees/impl-issue-122-baseline-variant/reference/packages/eden-storage/src/eden_storage/_checkpoint.py:624): checkpoint import does not preserve “absent `base_commit_sha`” correctly. If the source checkpoint lacks `base_commit_sha`, import leaves the receiver’s existing seed untouched instead of clearing it. I verified this directly: importing a pre-field checkpoint into a receiver constructed with `base_commit_sha='a'*40` leaves that SHA in place. That violates the new round-trip contract and can manufacture a baseline-eligible experiment out of a source that explicitly had no seed recorded. The fix is to treat absence in the archive as a real write of `None` on import (`tx.base_commit_sha_update = (None,)`), not as “no update”.

- `Risk` [reference/services/orchestrator/src/eden_orchestrator/baseline.py](/Users/ericalt/Documents/eden-worktrees/impl-issue-122-baseline-variant/reference/services/orchestrator/src/eden_orchestrator/baseline.py:97), [baseline.py](/Users/ericalt/Documents/eden-worktrees/impl-issue-122-baseline-variant/reference/services/orchestrator/src/eden_orchestrator/baseline.py:138): `ensure_baseline_variant` does not fully implement the plan’s “verified read-back, not blind AlreadyExists” rule. The steady-state check only validates `kind` and `commit_sha`; it does not validate compatibility with the configured path/status (for example, override-metrics config vs an existing default-path baseline). Then the `AlreadyExists` race path returns immediately without any read-back at all. In single-mode startup there may be no “next iteration” verification, so a conflicting row at the deterministic id can be silently accepted. The fix is to factor a verifier that checks `kind`, `commit_sha`, and status/evaluation compatibility against current config, and run it both for the pre-existing-row path and immediately after `AlreadyExists`.

**Overall Assessment**
The load-bearing termination carve is implemented in the important runtime paths: spec text, `driver._integrate_successful_variants`, and `multi_loop._experiment_is_drained_terminated` are aligned, and the contracts/schema/event changes are mostly coherent.

I would not approve this as-is because the first two issues change observable behavior: the feature is only default-on in selected deployments, and checkpoint import can synthesize `base_commit_sha` that was absent in the source. I also ran targeted tests for the new storage/dispatch/orchestrator coverage; they passed, so these are review findings from uncovered integration paths rather than already-failing local tests.
