## Change Summary

- New `Variant.kind` discriminator; `"baseline"` marks the seed elevated to a variant. `idea_id` becomes conditionally-required (required unless `kind == "baseline"`) on schema + model + event payload, in lockstep.
- `create_variant` relaxed so a baseline MAY be created directly in `success` with config metrics (validated against `evaluation_schema` at create time), emitting `variant.started`+`variant.succeeded` atomically; per-kind create authority (`orchestrators` group for baselines) enforced at the wire.
- The load-bearing anti-deadlock carve: baselines are never integrated and are excluded from the integration-decision predicate, the termination-drain rule, `_integrate_successful_variants`, and `_experiment_is_drained_terminated`. State-view `running`/`attempted` counts also exclude baselines.
- `experiment.base_commit_sha` recorded on the runtime object (v7 column migration across backends + in-memory), threaded from a new task-store-server `--base-commit-sha` flag, surfaced on `read_experiment` (omit-when-absent), and round-tripped through checkpoints. `ensure_baseline_variant` (idempotent by verified read-back) runs in both orchestrator modes.
- Web-ui `variant.idea_id` consumers made null-safe for baselines; integrator carries a defensive reject behind the dispatch skip.

## Round 1 — addressed Codex round-0 feedback

- **Bug (checkpoint import absent-seed)** — FIXED. `_checkpoint.py` now stages `tx.base_commit_sha_update = (parsed.experiment_row.get("base_commit_sha"),)` **unconditionally**, so a seedless source clears any seed the receiver was constructed with (round-trip fidelity, 10-checkpoints.md §5). Added `test_base_commit_sha_absent_source_clears_receiver_seed` proving a receiver built with `base_commit_sha="e"*40` ends up `None` after importing a seedless checkpoint.
- **Risk (ensure_baseline_variant blind AlreadyExists)** — FIXED. Extracted `_verify_existing_baseline(existing, base_sha, experiment_id)` (kind + commit_sha identity check, fail-loud on drift) and now call it on BOTH the pre-existing-row path AND the post-`AlreadyExists` race path (the winner is read back and verified, not blindly accepted — single-mode has no next-iteration re-check). Status/evaluation are intentionally not checked (first-create-wins; baseline is immutable; a default-path baseline legitimately advances starting→success).
- **Bug (feature "not default-on")** — DEFERRED / explained, not a code change. The seed is recorded via `EDEN_BASE_COMMIT_SHA` (set by `setup-experiment.sh`, wired into the task-store-server command in `compose.yaml`), so new experiments in the canonical deployment ARE default-on — `smoke.sh` empirically creates AND evaluates the baseline (the orchestrator reads `experiment.base_commit_sha` and `ensure_baseline_variant` creates it; the baseline rides `evaluation_dispatch` to `success` and is correctly never integrated). The optional-flag posture is the §D.2 reconciled rule: a deployment that records no seed simply has no baseline and logs `baseline_enabled_but_no_base_commit_sha` (never crashes). The `test_e2e.py` / `test_subprocess_e2e.py` *unit* tests deliberately omit the flag (the documented legacy/skip path) to keep those heavyweight exact-count tests stable; the default-on path is covered by the Compose smokes + the deterministic `test_baseline.py`. This is a judgment difference about test posture, not a deployment-behavior gap.

## Round 2 — addressed Codex round-1 feedback

- **Risk (AlreadyExists race path untested)** — FIXED. Added two focused regression tests to `test_baseline.py` using a stateful `_RacingStore` fake (read_variant → NotFound on the first/identity-check call, then the concurrent `winner` on the post-create read-back; create_variant always raises `AlreadyExists`): `test_already_exists_race_accepts_valid_baseline_winner` (valid baseline winner → accepted, no raise) and `test_already_exists_race_rejects_squatting_winner` (non-baseline / wrong-seed winner → `RuntimeError` drift). This exercises the repaired race branch directly, not just the pre-existing-row path.
- The "not default-on" point was withdrawn by the reviewer in round 1 (accepted as an intentional deployment/test-posture choice).
