**Findings**
1. The auth/provisioning design is not implementable as written. The plan generates `GITEA_REMOTE_PASSWORD` and writes it into a credential helper, but it never binds that secret to an actual Gitea login, and it also assumes HTTPS even though the current stack exposes Gitea as plain HTTP. Without an explicit “set user password / mint git-usable credential” step and a TLS/trust-bootstrap story, the worker `git clone/fetch/push` path cannot work. See [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:94>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:123>), and current compose config [compose.yaml](</Users/ericalt/Documents/eden/reference/compose/compose.yaml:42>).

2. The integrator rollback story is incomplete once Gitea becomes the source of truth. Today `Integrator.integrate()` is `create_ref` then `store.integrate_trial`, with rollback implemented as a local `delete_ref` on store-side failure. The plan inserts `push_ref`, but it never specifies the required remote compensating delete for the case “push succeeded, `integrate_trial` failed”. Under chapter 6 §3.4, leaving `trial/*` on Gitea without `trial_commit_sha` + `trial.integrated` is a protocol violation. See [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:175>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:363>), and current code [integrator.py](</Users/ericalt/Documents/eden/reference/packages/eden-git/src/eden_git/integrator.py:155>).

3. The crash-recovery story only handles half the local-then-publish failure window. The risks section trims local-only `work/*` refs after restart, but says nothing about local-only `trial/*` refs in the orchestrator clone. That matters because integrator branch names are deterministic: a crash after local `create_ref` but before remote push can wedge the next retry on the stale local `trial/<trial_id>-<slug>` ref. The same under-specification affects the web UI: a one-time startup fetch is not enough for admin/implementer routes that read and delete refs long after startup. See [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:154>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:349>), [admin.py](</Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/admin.py:172>), and [implementer.py](</Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/implementer.py:250>).

4. The test plan does not hit the new behavior through the public entry points the repo explicitly requires. The credential-helper coverage is hand-waved down to argv inspection, which does not prove real git auth works. More importantly, there is no stated test that drives `Integrator.integrate()` through “remote push succeeded, store write failed, remote ref deleted”, nor a real test of the evaluator/implementer fetch-before-worktree path. That misses the repo’s “test the actual code path” pitfall. See [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:289>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:301>), and [AGENTS.md](</Users/ericalt/Documents/eden/AGENTS.md:149>).

5. The setup sequencing is still an open question in the plan, but it needs to be settled before implementation. `setup-experiment.sh` currently seeds the repo before any `compose up`, while this plan requires a live Gitea for repo creation and initial push. The risks section notices that, but leaves two mutually exclusive options and says “cleanest: option (a)” instead of making it part of the design. That is too unresolved for an implementation plan. See [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:123>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:374>), and current script [setup-experiment.sh](</Users/ericalt/Documents/eden/reference/scripts/setup-experiment/setup-experiment.sh:332>).

**Assessment**
Not ready to implement.

Concrete revisions required:
1. Specify a real HTTPS/auth bootstrap: how TLS is terminated/trusted, and exactly how the git credential used by workers is created in Gitea.
2. Extend the git design beyond `push_ref`/`fetch_ref`: the integrator needs an explicit remote-delete path, and the plan must wire `Integrator.integrate()` as `local create` → `remote publish` → `store.integrate_trial` → `remote compensating delete on store failure`.
3. Define restart/freshness reconciliation for both `work/*` and `trial/*`, plus the long-lived web-ui/admin clone.
4. Replace helper-only test ideas with public-path tests that exercise the real service/integrator flows.
5. Resolve setup-experiment sequencing and give a concrete test-surface inventory instead of “most tests can stay local-only.”
