**Findings**
1. The evaluator flow is still pointed at the wrong ref namespace. The revised plan says the evaluator should `git fetch origin trial/<id>` before worktree creation, but evaluation happens before integration; the evaluator must read the worker branch at `trial.branch` / `trial.commit_sha`, not `trial/*`. As written, this breaks chapter 3’s evaluator contract. See [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:345>) and [spec](</Users/ericalt/Documents/eden/spec/v0/03-roles.md:121>).

2. The proposed non-bare clone topology for implementer/evaluator is incompatible with the current worktree model, and the refresh command in the plan fails on a normal clone with `main` checked out. The plan says “use `--bare` for orchestrator/web-ui; non-bare otherwise so worktrees can be added”, but the existing subprocess hosts explicitly use worktrees of a bare repo, and `TaskWorktree` is documented that way. Separately, the proposed startup refresh `git fetch origin '+refs/heads/*:refs/heads/*'` is not safe for a non-bare checked-out clone. This needs to stay bare for the worker hosts or be redesigned more explicitly. See [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:227>), [worktrees helper](</Users/ericalt/Documents/eden/reference/services/_common/src/eden_service_common/worktrees.py:23>), [implementer CLI](</Users/ericalt/Documents/eden/reference/services/implementer/src/eden_implementer_host/cli.py:47>), and [evaluator CLI](</Users/ericalt/Documents/eden/reference/services/evaluator/src/eden_evaluator_host/cli.py:62>).

3. The “remote compensating delete failed” recovery path is still not actually specified. The new four-step ladder is the right shape, but when step `4a` fails the plan says the next startup reconciliation cleans it up. The reconciliation it defines only removes local refs that do not exist on the remote; it does not describe how to detect and remove a remotely-published orphan `trial/*` ref whose store-side `trial_commit_sha` / `trial.integrated` never committed. That is exactly the failure mode step `4a` was added to handle, and it still needs a real recovery rule. See [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:235>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:303>), and [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:456>).

4. The service-level transport-failure mapping is too broad for the evaluator. The plan says mid-operation Gitea unreachability should be caught and submitted as `status="error"` for the in-flight task. That is acceptable for implementer failure handling, but for evaluator infrastructure faults chapter 3 requires `eval_error`, not `error`, so the trial stays `starting`. This needs role-specific handling, not a blanket rule. See [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:258>) and [spec](</Users/ericalt/Documents/eden/spec/v0/03-roles.md:143>).

5. The document still has several stale contradictions that should be cleaned before implementation so the plan is self-consistent. It still says “HTTPS” in the context and flag naming even after switching to plain HTTP, the verification section still lists `uv run pytest -m gitea` even though the test strategy explicitly says no `pytest.mark.gitea`, and the risks section still carries pre-revision text (`0644`, unresolved option `(a)`, `work/*`-only orphan cleanup). See [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:15>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:225>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:484>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:557>), and [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-b-gitea-remote.md:588>).

**Assessment**
The revision materially improved the core integrator atomicity story, the auth bootstrap is now much more concrete, and the blast-radius claim is more defensible than in round 0.

It is still not ready to implement. The concrete revisions needed are:
1. Fix evaluator fetch semantics to use the worker branch / worker commit, not `trial/*`.
2. Resolve the bare vs non-bare topology explicitly and update the fetch strategy accordingly.
3. Add a real recovery rule for “remote publish succeeded, store write failed, remote compensating delete failed”.
4. Make git-transport failure handling role-specific, especially for evaluator `eval_error`.
5. Remove the stale contradictory text so the plan has one coherent transport, test, and risk story.
