**Feasibility**

There are still two plan-shaping feasibility problems in the new preferred path.

- The preferred exec-mode mechanism assumes a mode model that the current substrate does not have. The plan now says the `-2` trio wrapper reads `EDEN_EXEC_MODE_2`, and the wrapper examples pass `--exec-mode subprocess` / `--exec-mode docker` through `setup-experiment` ([issue-183-cross-mode-ci-smoke-matrix.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-183-cross-mode-ci-smoke-matrix/docs/plans/issue-183-cross-mode-ci-smoke-matrix.md:117), [issue-183-cross-mode-ci-smoke-matrix.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-183-cross-mode-ci-smoke-matrix/docs/plans/issue-183-cross-mode-ci-smoke-matrix.md:129), [issue-183-cross-mode-ci-smoke-matrix.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-183-cross-mode-ci-smoke-matrix/docs/plans/issue-183-cross-mode-ci-smoke-matrix.md:154), [issue-183-cross-mode-ci-smoke-matrix.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-183-cross-mode-ci-smoke-matrix/docs/plans/issue-183-cross-mode-ci-smoke-matrix.md:242)). But the real `setup-experiment` interface only accepts `--exec-mode {host,docker}` today, not `subprocess`, and existing subprocess mode is selected by layering `compose.subprocess.yaml`, not by env ([setup-experiment.sh](/Users/ericalt/Documents/eden-worktrees/plan-issue-183-cross-mode-ci-smoke-matrix/reference/scripts/setup-experiment/setup-experiment.sh:35), [setup-experiment.sh](/Users/ericalt/Documents/eden-worktrees/plan-issue-183-cross-mode-ci-smoke-matrix/reference/scripts/setup-experiment/setup-experiment.sh:419), [smoke-subprocess.sh](/Users/ericalt/Documents/eden-worktrees/plan-issue-183-cross-mode-ci-smoke-matrix/reference/compose/healthcheck/smoke-subprocess.sh:58)). Relatedly, “defaulting to match exp-1’s mode” is underspecified because exp-1 mode currently comes from overlay composition plus `EDEN_EXEC_MODE`, not one authoritative variable. Suggestion: either make the fallback overlay approach the primary design, or explicitly redefine the mode model as a real three-state contract (`scripted` / `subprocess` / `docker`) and call out the resulting changes to `setup-experiment`, `.env`, and the existing smoke docs.

- The preferred docker path still looks infeasible without a dedicated privilege overlay. The plan says the `-2` trio’s DooD socket mount and `group_add` would live on the `-2` services in `compose.multi-experiment.yaml`, gated on `EDEN_EXEC_MODE_2` ([issue-183-cross-mode-ci-smoke-matrix.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-183-cross-mode-ci-smoke-matrix/docs/plans/issue-183-cross-mode-ci-smoke-matrix.md:117), [issue-183-cross-mode-ci-smoke-matrix.md](/Users/ericalt/Documents/eden-worktrees/plan-issue-183-cross-mode-ci-smoke-matrix/docs/plans/issue-183-cross-mode-ci-smoke-matrix.md:151)). But the current substrate’s security posture is that the docker socket is mounted only in `compose.docker-exec.yaml`; that isolation is spelled out in the overlay header itself ([compose.docker-exec.yaml](/Users/ericalt/Documents/eden-worktrees/plan-issue-183-cross-mode-ci-smoke-matrix/reference/compose/compose.docker-exec.yaml:3)). A wrapper can choose CLI flags, but it cannot safely make privileged mounts appear and disappear. Suggestion: keep subprocess/scripted selection wrapper-driven if you want, but make docker-for-exp-2 explicitly use a separate `compose.multi-experiment.docker.yaml` overlay. For DooD privilege, the fallback shape is the sound shape.

Because those are still feasibility blockers, I’d stop here and not treat alternatives/completeness/risk as settled yet.

**Overall Assessment**

The revised plan is materially stronger than round 0. The remaining issue is concentrated in one area: the new preferred exec-mode design is still fighting the actual Compose/setup-experiment substrate. If you switch the plan to make dedicated `-2` overlays the default for mode-specific and especially privilege-specific behavior, or fully specify a new three-state mode contract, it should be ready for a deeper next-round review.
