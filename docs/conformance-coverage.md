# Conformance coverage matrix

Maps every RFC 2119 keyword line in `spec/v0/*.md` to the conformance
scenarios that cite it. Generated by `scripts/conformance-coverage.py`;
re-run to refresh.

Per [`spec/v0/09-conformance.md`](../spec/v0/09-conformance.md) §3,
the v1 / v1+roles / v1+roles+integrator suites assert MUSTs only;
SHOULDs and MAYs are interop guidance, not interop contracts. This
matrix surfaces both for visibility but classifies coverage gaps
by keyword level.

**Coverage rule.** A scenario whose docstring cites
`spec/v0/<chapter>.md §<sec>` covers that section AND every ancestor
section (e.g. a citation of §3.2 also counts as coverage of §3).
This mirrors
[`conformance/src/conformance/tools/check_citations.py`](../conformance/src/conformance/tools/check_citations.py)'s
section-walk logic.

**What this matrix is NOT.** A scenario citing a section is not
proof that every MUST in that section is *exercised* by the test —
only that the test claims authority from somewhere in that section.
A weak-coverage column would require reading each test body; that
step is recommended as a follow-up audit per finding #24.

## How to read the gap list

Three kinds of "uncovered" line show up in the priority list. Triage by kind before deciding whether a new scenario is warranted.

### 1. Structurally-coverage-immune chapters

[`00-overview.md`](../spec/v0/00-overview.md), [`01-concepts.md`](../spec/v0/01-concepts.md), and [`09-conformance.md`](../spec/v0/09-conformance.md) host MUSTs that are mostly restatements or self-references (the conformance chapter telling the IUT what conformance means; the overview saying schema and prose must agree). These rarely need their own scenarios — their content is enforced by scenarios that target the chapters those MUSTs point at. Flag these for a one-pass review to confirm each is genuinely a restatement, not a unique claim, then expect them to stay uncited.

### 2. Storage chapter ([`08-storage.md`](../spec/v0/08-storage.md))

Every MUST in chapter 08 currently appears uncovered. That's because [`spec/v0/09-conformance.md`](../spec/v0/09-conformance.md) §4 v1 covers chapter-08 §§1.1 and 1.7 *through* the wire chapters that use them (chapter 04 / 07), and the existing scenarios cite the wire chapter, not 08. This isn't a coverage gap so much as a **citation gap**: the assertion exists, but its authority chain doesn't surface. Two clean responses: (a) add cross-chapter ancestry to the matrix (a 04 §3.3 citation also counts as coverage of 08 §1.3 because the wire MUST inherits from the storage MUST), or (b) re-cite scenarios that exercise storage invariants to mention 08 explicitly. Option (b) is more honest but each test gets a longer docstring.

### 3. Data-model and event-payload shape (`02-data-model.md`, parts of `05-event-protocol.md`)

Many of these MUSTs are enforced **at schema-validation time** — the JSON Schema files under [`spec/v0/schemas/`](../spec/v0/schemas/) carry the constraint, and `eden-contracts` Pydantic models double-check via the `schema-parity` CI job. Because chapter 09 §6 makes the chapter-7 wire binding the IUT contract, a scenario asserts these MUSTs only when the wire surface lets a malformed payload reach the server (e.g. a malformed event ID, a non-finite metric, an out-of-vocabulary status). Schema-only constraints (timestamp pattern, SHA pattern, reserved metric names) ARE asserted indirectly because any IUT that fails them will reject conforming requests. Flag these for the inverse pass: confirm each schema-enforced MUST has *some* wire scenario that would catch a violator.

A future revision of this matrix could split the per-claim "covered by" column into:

| Tag | Meaning |
|---|---|
| `(scenario)` | Direct scenario citation. |
| `(schema)` | Enforced by the JSON Schema; IUT can't violate it without producing schema-invalid output. |
| `(restatement)` | This line restates a MUST whose authoritative home is elsewhere; coverage tracked at the home. |
| `(impl-only)` | Reference impl honors it but no scenario asserts it; latent coverage gap. |

That classification is one rung past what this matrix delivers today; it's the natural next step per finding #24's "high-value scenarios that would close the most-load-bearing gaps" follow-up.

## Per-claim assertion-coverage pilot — chapter 04

The line-coverage in the auto-generated tables below counts a row as "covered" if any cited scenario's docstring resolves to the row's section (or an ancestor). It does NOT verify that the cited scenario actually exercises the claim. This section reads each `04-task-protocol.md` MUST/MUST-NOT row's cited scenarios and tags the row by what coverage really exists. Method per finding #24's resolution direction; chapter 04 picked as the pilot because it has the highest concentration of MUSTs and the most-cited scenarios.

### Tag taxonomy used

A fifth tag was added to the original four named in the auto-generated section above; without it, two MUSTs that chapter 09 §3 explicitly opts out of certifying (atomicity-of-state-and-event, claim-token unforgeability) would be tagged misleadingly.

| Tag | Meaning |
|---|---|
| `(scenario)` | A cited scenario actually exercises the claim's behavior under realistic conditions. |
| `(consequence)` | The claim is one of the chapter 09 §3 black-box-impossible MUSTs (atomicity, unforgeability); the cited scenario asserts a testable proxy (uniqueness, post-event state read-back) and self-labels as a regression test. NOT a certification. |
| `(restatement)` | A scenario cites the section but its assertion does NOT exercise this particular MUST. The section may be partially covered by sibling claims; this specific line is not. |
| `(schema)` | Enforced by the JSON Schema; an IUT can't violate it without producing schema-invalid output. |
| `(impl-only)` | Reference impl honors it but no scenario asserts it via the wire. |
| `(uncovered)` | No scenario cites this section (or any ancestor). |

`(scenario)` and `(consequence)` together represent "as much coverage as the spec asks for". The other four tags are real gaps of varying severity.

### Methodology refinements surfaced during the pilot

1. **Five tags, not four.** The original four (`scenario`, `schema`, `restatement`, `impl-only`) didn't accommodate chapter 09 §3's two explicitly-uncertified MUSTs (atomicity, unforgeability). Added `(consequence)` for testable-proxy coverage with no certification claim.

2. **List-header lines are NOT independent MUSTs.** A line like `` `token` — an opaque value that MUST be: `` is a parser hit for "MUST" but the actual claims are in the bullets that follow. The auto-generator's keyword regex doesn't distinguish. A future revision should: (a) detect "MUST be:" patterns and merge the line with its sub-bullets, OR (b) tag these as `(list-header)` and exclude from coverage counting.

3. **Cross-chapter coverage is real and structurally hidden.** Two of chapter 04's five `(uncovered)` rows are tested under chapter 05 §2.2's composite-commit umbrella. The auto-generator's ancestor-walk is intra-chapter only; chapter 04's storage-side MUSTs that bind to chapter 05's event MUSTs (or chapter 07's wire MUSTs) don't get credited. Either teach the generator about composite-commit cross-chapter ancestry, or require multi-citation in scenario docstrings.

4. **Multi-MUST lines need finer claim-counting.** §4.3 line 121 packs two distinct MUSTs ("exactly one event" + "MUST clear claim") into one row. Both have coverage but via different cited tests at different sections. Tagging the row as `(scenario)` is correct net but loses the per-claim detail. A future revision should split lines into individual claims before tagging.

5. **MUST NOT in informative restatements double-counts.** Some lines whose primary MUST is in another chapter restate the rule for the local context (e.g. chapter 04 §1.3 restates chapter 05's atomicity invariant). The matrix counts these toward the local chapter's MUST total but they're not independent claims. A future revision should detect "as in [link] §X" / "per §X" prefixes and tag them `(restatement, see-other)`.

### Methodology refinements (additional, beyond chapter-04 pilot's list)

1. **`(impl-only)` is a real category, not a hypothetical.** (Continuing the chapter-04 pilot's numbered list as items 6+; renumbered as 1+ within this subsection per markdownlint convention.) The pilot tagged this as `(impl-only)`; chapter 09 §3 doesn't exclude it, but a wire-only conformance test cannot kill-9 the IUT and read back its state. Same pattern likely applies to other "deployment MUST persist X durably" rules in chapter 08. Worth a focused pass in a follow-up: name every `(impl-only)` MUST and decide whether the conformance contract needs a kill-9-and-restart fixture (chapter 09 §6 currently says no — the chapter-7 binding is the only IUT contract).

2. **The "X MUST be one of {a, b, c}" pattern is testable but rarely tested.** Chapter 03 has multiple MUSTs of this shape (`status` MUST be one of {success, error}; `parent_commits` MUST contain at least one SHA). Schema enforces them; conformance scenarios mostly test happy-path values. Adding wire scenarios that submit out-of-vocabulary values for each enum-shaped MUST would strengthen `(consequence)` rows; this is a separate generator-driven pass.
