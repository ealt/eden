# Conformance coverage matrix

Maps every RFC 2119 keyword line in `spec/v0/*.md` to the conformance
scenarios that cite it. Generated by `scripts/conformance-coverage.py`;
re-run to refresh.

Per [`spec/v0/09-conformance.md`](../spec/v0/09-conformance.md) ┬¦3,
the v1 / v1+roles / v1+roles+integrator suites assert MUSTs only;
SHOULDs and MAYs are interop guidance, not interop contracts. This
matrix surfaces both for visibility but classifies coverage gaps
by keyword level.

**Coverage rule.** A scenario whose docstring cites
`spec/v0/<chapter>.md ┬¦<sec>` covers that section AND every ancestor
section (e.g. a citation of ┬¦3.2 also counts as coverage of ┬¦3).
This mirrors
[`conformance/src/conformance/tools/check_citations.py`](../conformance/src/conformance/tools/check_citations.py)'s
section-walk logic.

**What this matrix is NOT.** A scenario citing a section is not
proof that every MUST in that section is *exercised* by the test ŌĆö
only that the test claims authority from somewhere in that section.
A weak-coverage column would require reading each test body; that
step is recommended as a follow-up audit per finding #24.

## How to read the gap list

Three kinds of "uncovered" line show up in the priority list. Triage by kind before deciding whether a new scenario is warranted.

### 1. Structurally-coverage-immune chapters

[`00-overview.md`](../spec/v0/00-overview.md), [`01-concepts.md`](../spec/v0/01-concepts.md), and [`09-conformance.md`](../spec/v0/09-conformance.md) host MUSTs that are mostly restatements or self-references (the conformance chapter telling the IUT what conformance means; the overview saying schema and prose must agree). These rarely need their own scenarios ŌĆö their content is enforced by scenarios that target the chapters those MUSTs point at. Flag these for a one-pass review to confirm each is genuinely a restatement, not a unique claim, then expect them to stay uncited.

### 2. Storage chapter ([`08-storage.md`](../spec/v0/08-storage.md))

Every MUST in chapter 08 currently appears uncovered by the auto-generator. That is because [`spec/v0/09-conformance.md`](../spec/v0/09-conformance.md) ┬¦4 v1 covers chapter-08 ┬¦┬¦1.1, 1.7, 1.8, 6 *through* the wire chapters that use them (chapter 04 / 07), and the existing scenarios cite the wire chapter, not 08. The per-claim audit below classifies chapter 08's MUSTs as `(restatement)` with explicit cross-references; the section's effective coverage is high once that lens is applied.

### 3. Data-model and event-payload shape (`02-data-model.md`, parts of `05-event-protocol.md`)

Many of these MUSTs are enforced **at schema-validation time** ŌĆö the JSON Schema files under [`spec/v0/schemas/`](../spec/v0/schemas/) carry the constraint, and `eden-contracts` Pydantic models double-check via the `schema-parity` CI job. Because chapter 09 ┬¦6 makes the chapter-7 wire binding the IUT contract, a scenario asserts these MUSTs only when the wire surface lets a malformed payload reach the server (e.g. a malformed event ID, a non-finite metric, an out-of-vocabulary status). Schema-only constraints (timestamp pattern, SHA pattern, reserved metric names) ARE asserted indirectly because any IUT that fails them will reject conforming requests or produce schema-invalid output. The per-claim audit tags these `(schema)`; effective coverage groups `(scenario)` + `(consequence)` + `(schema)` together.

## Per-claim assertion-coverage ŌĆö chapters 02 / 03 / 04 / 05 / 06 / 07 / 08 / 09

The auto-generator's line-coverage (above) counts a row as "covered" if any cited scenario's docstring resolves to the row's section (or an ancestor). It does NOT verify that the cited scenario actually exercises the claim, and it does NOT credit schema enforcement or cross-chapter ancestry. This section walks each normative chapter's MUST / MUST-NOT lines and tags each row by what coverage really exists.

### Tag taxonomy

| Tag | Meaning |
|---|---|
| `(scenario)` | A cited scenario actually exercises the claim's behavior. |
| `(consequence)` | The claim is one of [`09-conformance.md`](../spec/v0/09-conformance.md) ┬¦3's black-box-immune MUSTs (atomicity, unforgeability) or a wire-side projection of an off-wire MUST that chapter 09 ┬¦6 explicitly defers; the cited scenario asserts the testable proxy and self-labels as a regression test, NOT a certification. |
| `(schema)` | Enforced by the JSON Schema under [`spec/v0/schemas/`](../spec/v0/schemas/); an IUT cannot violate the constraint without emitting schema-invalid output, which the `schema-parity` CI job catches against the reference impl and which conforming clients reject. Counted as effective coverage because chapter 09 ┬¦6's IUT contract IS the wire binding. |
| `(restatement)` | The MUST's authoritative home is in another chapter; this row is a pointer / cross-reference. Coverage is tracked at the home. |
| `(impl-only)` | The reference impl honors the rule but no wire scenario asserts it; a real coverage gap. |
| `(list-header)` | A "X MUST be:" lead-in to sub-bullets; not an independent claim. Excluded from per-tag counts. |
| `(meta)` | A documentational / structural statement about the chapter itself; no testable behavior. Excluded from per-tag counts. |
| `(uncovered)` | No scenario, no schema cover, no restatement target ŌĆö a real gap. |

`(scenario)`, `(consequence)`, and `(schema)` together represent "as much coverage as chapter 09 ┬¦6 allows". `(restatement)`, `(impl-only)`, and `(uncovered)` are gaps of varying severity; `(restatement)` resolves with multi-citation or cross-chapter ancestor walk and is the mildest, `(uncovered)` is the most load-bearing.
