## Change Summary

- A three-service Compose stack: `postgres` (16.6-alpine, healthchecked), `gitea` (1.22.6-rootless, headless via INSTALL_LOCK + DISABLE_REGISTRATION, healthchecked), `blob-init` (one-shot busybox that mounts `eden-blob-data` so Compose actually creates the volume — empirically verified that an unmounted top-level `volumes:` declaration alone does NOT trigger creation).
- The two long-running services declare `depends_on: { blob-init: { condition: service_completed_successfully } }` so `compose up --wait` interprets the one-shot's exit-0 as success — also verified empirically that without this, `--wait` returns 1.
- `${VAR:?msg}` syntax for required secrets (POSTGRES_PASSWORD, GITEA_SECRET_KEY, GITEA_INTERNAL_TOKEN) so a missing `.env` fails loudly. `.env.example` ships dev-only weak credentials.
- Gitea SSH port wiring: `SSH_PORT` (advertised) tracks the host-side override; `SSH_LISTEN_PORT` is pinned at 2222 inside the container, so SSH clones survive a `GITEA_SSH_HOST_PORT` override.
- A `compose-smoke` CI job runs `healthcheck/smoke.sh` end-to-end on every PR. The script preflights tooling (docker, jq, curl, docker compose plugin), brings up the stack with `--wait`, asserts the project name resolved from `docker compose config` is `eden-reference` before applying the standard `<project>_<volume>` prefix, inspects each named volume, checks blob-init's exit code via `docker inspect --format '{{.State.ExitCode}}' eden-blob-init` (avoids `docker compose ps --format json` shape variations across Compose versions), confirms postgres + gitea are reachable.
- Branch protection is *not* updated to require `compose-smoke` in this chunk; deferred to a follow-up after the job has a few clean runs.
- Smoke test runs cleanly locally on the developer's Docker Desktop; markdown-lints clean.
