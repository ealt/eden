# EDEN Phase 10 chunk 10a: Compose Infrastructure

## Context

Phase 9 closed with a complete Web UI: a human can play any role
end-to-end through a browser, and `/admin/*` exposes observability and
operator-reclaim. Every service is currently run as a Python module
(`python -m eden_task_store_server …`, etc.) against ephemeral
SQLite + a local bare git repo.

Phase 10 packages the reference stack so it runs locally via
`docker compose up`. Per the roadmap, the phase splits into five
chunks; this plan covers **chunk 10a**:

> **10a** — Infrastructure containers (Postgres for durable backend,
> Gitea for git host, blob volume) + Compose skeleton that stands them
> up.

Subsequent chunks will dockerize the EDEN services themselves (10b),
add the experiment-registration script (10c), wire LLM workers (10d),
and ship the end-to-end Compose integration test (10e).

**Critical scope clarification.** 10a is *only* the infrastructure
layer: third-party stateful services (Postgres, Gitea), the named
volume that will eventually back blob storage, and the Compose file
that stands them up. **No EDEN services are dockerized in 10a.** No
new Python code in `reference/packages/` or `reference/services/`.
No `PostgresStore` implementation — `eden-storage` already ships
`SqliteStore` and `InMemoryStore`; the Postgres binding lands when a
consumer exists (10b alongside the dockerized `task-store-server`).
The blob package (`reference/packages/eden-blob/`) stays a `.gitkeep`
until 10d when the implementer host needs to publish artifacts.

**Postgres is for the EDEN task store, not for Gitea.** The roadmap
phrases 10a's deliverables as "Postgres for durable backend, Gitea
for git host," which scoped together can read as "Postgres backs
Gitea." It does not. Gitea ships with its own embedded SQLite and
that's what the 10a Compose file uses (`GITEA__database__DB_TYPE:
sqlite3`). Pointing Gitea at our Postgres would couple the git
host's recovery story to EDEN's task store schema — something a
reference deployment of EDEN should explicitly not do, since
production deployments should be free to swap either component
independently. The Postgres container in 10a is reserved for
`PostgresStore` consumption in 10b.

Why ship infra without consumers? Three reasons:

1. **Image-pin & config-shape stability lands first.** Pinning
   image tags, naming volumes, choosing the postgres user/db scheme,
   and picking the Gitea admin bootstrap pattern are all decisions
   that downstream chunks consume. Surfacing them now means 10b/10c/10d
   only have to write app code, not also negotiate infra shape.
2. **Smoke-testable in isolation.** A standalone Compose file with
   healthchecks can be brought up in CI with no EDEN dependencies,
   so Compose drift is caught even when the python suite is green.
3. **Chunk discipline.** The roadmap explicitly groups 10b + 10c into
   one chunk and lists 10a separately. Co-mingling 10a with backend
   bindings violates that grouping.

## Core Design Decisions

- **`reference/compose/` is the home for everything.** Compose file,
  `.env.example`, init scripts, README. No top-level `docker-compose.yml`
  at the repo root — that would imply the whole repo is "a docker
  thing," which it isn't.
- **One Compose file, no overrides yet.** A single
  `compose.yaml` in 10a. Profile-based service splits
  (`compose --profile workers`) are tempting but premature; 10b will
  add EDEN services to the same file, and 10e may add an override
  for the e2e test. Adding a second file now would force us to move
  things around in 10b.
- **Image tags pinned to specific minor versions, never `:latest`.**
  Selected versions:
  - `postgres:16.6-alpine` — current Postgres stable, alpine variant
    keeps the image small. Postgres 16 is the line direvo's monolith
    targeted; staying on it minimizes surprises when Phase 10b adds
    `PostgresStore`.
  - `gitea/gitea:1.22.6-rootless` — current Gitea stable, rootless
    variant runs as uid 1000 instead of root (Gitea publishes both
    flavors and explicitly recommends rootless for new deployments).
- **Default ports are off the well-known channels** (Postgres on 5433,
  Gitea on 3001) so a developer's existing local Postgres / Gitea
  doesn't collide. Documented in `.env.example`. Real production
  deployments override.
- **Secrets via `.env` only.** No secrets baked into the Compose file.
  `.env.example` ships canned dev defaults; the real `.env` is
  gitignored. CI uses `.env.example` as-is — these are throwaway
  values, not real credentials.
- **Healthchecks on every infra service.** Compose `depends_on`
  semantics with `condition: service_healthy` are the only safe way
  to chain bring-up; without healthchecks, "up" means "container
  process started," not "Postgres is accepting connections." Each
  service in 10a defines a `healthcheck:` block and downstream
  services in 10b will gate on `service_healthy` instead of bare
  `service_started`.
- **Volume strategy — named volumes, not bind mounts.** Named volumes
  (`eden-postgres-data`, `eden-gitea-data`, `eden-blob-data`) are
  Compose-managed, survive `compose down` (use `compose down -v` to
  wipe), and don't pollute the repo with `_data/` directories. Bind
  mounts work but tie data location to the developer's filesystem
  layout, which gets fragile when CI lays things out differently.
- **Postgres bootstrap via `/docker-entrypoint-initdb.d/`.** The
  official Postgres image runs every `*.sql` and `*.sh` in that
  directory the *first* time the data volume is empty. We ship one
  init SQL file: `01-create-eden-db.sql`, which creates the `eden`
  database and the `eden` role with an env-var-driven password.
  (`POSTGRES_DB=eden` + `POSTGRES_USER=eden` + `POSTGRES_PASSWORD=…`
  in `.env` is the simpler path and we use that — the init SQL is
  needed only if we want extensions or extra roles. **Decision:**
  use env-var bootstrap, **no init SQL file** in 10a. If a later
  chunk needs `pg_stat_statements` or a second role we add init SQL
  then. Keeps 10a smaller.)
- **Gitea bootstrap via env vars.** Gitea respects `GITEA__<section>__<key>`
  env vars at startup. We set the minimum needed for headless
  operation: install lock skipped, no telemetry, JWT secret pinned
  via `.env`, admin user *not* auto-created in 10a — 10c's
  `setup-experiment` script will create the admin via the API or
  `gitea admin user create` once there's a real consumer. (Creating
  an admin in 10a means we'd have to choose a token-issuing scheme
  before we have anything to issue tokens for.)
- **Blob volume = named volume + one-shot `blob-init` mounter.**
  Empirical Compose behavior verified during planning: a
  top-level `volumes:` declaration **does not** cause Compose to
  create the volume unless at least one service mounts it. A
  bare declaration (no mount, no init service) leaves
  `docker volume ls` empty after `compose up`. To make 10a's
  deliverable real, we ship a one-shot `blob-init` busybox
  container that mounts `eden-blob-data` at `/var/lib/eden/blobs`,
  runs `mkdir -p`, and exits 0. `restart: "no"` prevents the
  exited container from looping; `docker compose up --wait`
  semantics for one-shot services were also verified during
  planning — `--wait` returns exit 0 when a one-shot container
  exits 0, so the smoke test gets a clean signal. Downstream
  services in 10b/10d mount the same volume.
- **10a's blob-volume guarantee is "volume exists," not
  "writable by the eventual 10d producer."** `blob-init` runs as
  BusyBox root, and any in-container writability check 10a runs
  is also from a root container — that proves "root can write,"
  which is uninformative for the eventual non-root LLM-worker
  containers in 10d. We deliberately do **not** make a
  uid/gid/perm contract in 10a. 10d picks the user model when
  the first real producer lands and is responsible for any
  ownership/`chmod`/`chown` step it needs. 10a's smoke test
  asserts existence (`docker volume inspect`) and that the
  `blob-init` container exited 0 — nothing more about the volume.
- **Compose file format = `compose.yaml` (no `version:` field).**
  Modern Docker Compose deprecated the top-level `version:` key and
  Docker recommends using just `compose.yaml`. We name the file
  `compose.yaml` (not `docker-compose.yml`) and omit `version:`.
  `docker compose` (v2 plugin, the only supported flavor) accepts
  both filenames; users on extremely old `docker-compose` (v1, EOL)
  may be surprised, and that's an acceptable cost — v1 has been
  unmaintained for 18+ months.

## File-by-file design

### `reference/compose/compose.yaml`

Three services and three named volumes. `postgres` and `gitea`
mount their own data volumes; `blob-init` is a one-shot service
whose only purpose is to ensure `eden-blob-data` is created.

```yaml
name: eden-reference

services:
  postgres:
    image: postgres:16.6-alpine
    container_name: eden-postgres
    restart: unless-stopped
    environment:
      POSTGRES_DB: ${POSTGRES_DB:-eden}
      POSTGRES_USER: ${POSTGRES_USER:-eden}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:?POSTGRES_PASSWORD must be set}
    ports:
      - "${POSTGRES_HOST_PORT:-5433}:5432"
    volumes:
      - eden-postgres-data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER:-eden} -d ${POSTGRES_DB:-eden}"]
      interval: 5s
      timeout: 3s
      retries: 10
      start_period: 10s

  gitea:
    image: gitea/gitea:1.22.6-rootless
    container_name: eden-gitea
    restart: unless-stopped
    environment:
      GITEA__server__DOMAIN: localhost
      GITEA__server__ROOT_URL: http://localhost:${GITEA_HOST_PORT:-3001}/
      GITEA__server__HTTP_PORT: "3000"
      # SSH_PORT is the *advertised* port (shown in clone URLs); it must
      # match the host-side mapping so SSH clones work. SSH_LISTEN_PORT
      # is what Gitea binds inside the container, kept fixed at 2222
      # so the published mapping below stays valid even when a user
      # overrides GITEA_SSH_HOST_PORT to dodge a host collision.
      GITEA__server__SSH_PORT: "${GITEA_SSH_HOST_PORT:-2222}"
      GITEA__server__SSH_LISTEN_PORT: "2222"
      GITEA__database__DB_TYPE: sqlite3
      GITEA__database__PATH: /var/lib/gitea/data/gitea.db
      GITEA__security__INSTALL_LOCK: "true"
      GITEA__security__SECRET_KEY: ${GITEA_SECRET_KEY:?GITEA_SECRET_KEY must be set}
      GITEA__security__INTERNAL_TOKEN: ${GITEA_INTERNAL_TOKEN:?GITEA_INTERNAL_TOKEN must be set}
      GITEA__service__DISABLE_REGISTRATION: "true"
      GITEA__metrics__ENABLED: "false"
      GITEA__log__LEVEL: Warn
    ports:
      - "${GITEA_HOST_PORT:-3001}:3000"
      - "${GITEA_SSH_HOST_PORT:-2222}:2222"
    volumes:
      - eden-gitea-data:/var/lib/gitea
    healthcheck:
      test: ["CMD", "wget", "-q", "--spider", "http://localhost:3000/api/v1/version"]
      interval: 5s
      timeout: 3s
      retries: 20
      start_period: 30s

  blob-init:
    # One-shot service whose only job is to ensure eden-blob-data
    # exists at compose-up time. Compose does NOT create an unmounted
    # top-level volume — that was verified during planning by a quick
    # spike — so without this service the volume would not exist
    # until 10b/10d's first real mounter ran. 10a's contract is
    # "volume exists" only; ownership/perms are 10d's responsibility.
    image: busybox:1.36.1
    container_name: eden-blob-init
    restart: "no"
    # Deliberately no chmod/chown: 10a's contract is "volume exists,"
    # not "writable by the eventual 10d producer." Ownership belongs
    # to whichever uid/gid 10d picks for its blob-writing service.
    command: ["sh", "-c", "mkdir -p /var/lib/eden/blobs && echo blob-volume ready"]
    volumes:
      - eden-blob-data:/var/lib/eden/blobs

volumes:
  eden-postgres-data:
  eden-gitea-data:
  eden-blob-data:
```

Notes on shape choices:

- `name: eden-reference` is the project name. `compose up`,
  `compose down`, and `compose ps` use this; without it the project
  defaults to the directory name (`compose`) which is uselessly
  generic.
- `${VAR:?msg}` syntax for required secrets means `compose up`
  fails fast with a clear message if the user hasn't copied
  `.env.example` to `.env`. `${VAR:-default}` for optional values.
- Port collisions on the host are likely (a developer probably has
  Postgres 5432 in use; some have Gitea 3000). Defaulting host-side
  to 5433 / 3001 / 2222 sidesteps the most common collisions.
- `GITEA__security__INSTALL_LOCK: "true"` — skip the first-run
  installer wizard. With `INSTALL_LOCK=true` plus
  `DISABLE_REGISTRATION=true`, Gitea boots straight into headless
  mode. Admin user creation happens in 10c.
- The blob volume is declared at the top level and mounted by the
  one-shot `blob-init` service so Compose actually creates it at
  `compose up` (an unmounted declaration alone does not, per the
  empirical spike noted in Core Design Decisions). Downstream
  services in 10b/10d mount the same volume at `/var/lib/eden/blobs`.
  `--wait` exits 0 when `blob-init` exits 0; the smoke test re-checks
  via `docker volume inspect`.
- No `networks:` block: services land on Compose's default
  bridge network. Service discovery via container name
  (`postgres:5432`, `gitea:3000`) just works; explicit network
  declarations matter for multi-stack overlap, which 10a doesn't
  have.

### `reference/compose/healthcheck/smoke.sh`

Bash script, exits 0 on success. Used by the new CI job and runnable
locally for verification. Behavior:

```bash
#!/usr/bin/env bash
set -euo pipefail

# Bring the stack up using .env.example as the source of truth.
# (Copy to a temp .env if invoked from CI to avoid mutating the
# checked-in file.)
TMPENV="$(mktemp)"
trap 'rm -f "$TMPENV"; docker compose -f compose.yaml --env-file "$TMPENV" down -v' EXIT
cp .env.example "$TMPENV"

docker compose -f compose.yaml --env-file "$TMPENV" up -d --wait --wait-timeout 120

# Assert both long-running services are healthy.
# `--wait` already enforces this, but we re-check for clarity in CI logs.
docker compose -f compose.yaml --env-file "$TMPENV" ps

# Assert all three named volumes were created at compose-up time.
# blob-init mounts eden-blob-data so Compose creates it; postgres and
# gitea mount their own data volumes the usual way. Project-prefixed
# volume names follow the "<project>_<volume>" convention; the project
# name is fixed by `name: eden-reference` in compose.yaml and is also
# echoed below for traceability.
PROJECT="$(docker compose -f compose.yaml --env-file "$TMPENV" \
            config --format json | jq -r '.name')"
test "$PROJECT" = "eden-reference" \
    || { echo "unexpected project name: $PROJECT"; exit 1; }
for vol in eden-postgres-data eden-gitea-data eden-blob-data; do
    docker volume inspect "${PROJECT}_${vol}" >/dev/null
done

# Confirm blob-init actually exited 0 (--wait would have failed
# otherwise, but re-checking gives a useful CI log line). Using
# `docker inspect` against the explicit container_name avoids
# `docker compose ps --format json` shape variations across Compose
# versions (some emit NDJSON, some emit a JSON array).
test "$(docker inspect --format '{{.State.ExitCode}}' eden-blob-init)" = "0"

# Assert Postgres is accepting connections (independent of healthcheck).
docker compose -f compose.yaml --env-file "$TMPENV" exec -T postgres \
    pg_isready -U eden -d eden

# Assert Gitea API responds.
GITEA_PORT="$(grep ^GITEA_HOST_PORT "$TMPENV" | cut -d= -f2)"
curl -fsS "http://localhost:${GITEA_PORT}/api/v1/version" | grep -q '"version":'

echo "PASS"
```

Notes:

- The `trap` ensures teardown runs even on assertion failure, so
  CI runs don't leave dangling volumes.
- `--wait --wait-timeout 120` (Compose v2 feature) makes `up`
  block until every service is healthy or 2min elapses, exiting
  non-zero if any service fails. This is the cleanest way to
  encode "everything came up healthy" in a script.
- The volume name is the standard `<project>_<volume>` Compose
  convention. The script reads the project name from
  `docker compose config` and asserts it equals
  `eden-reference` (matching the `name:` field in `compose.yaml`)
  before applying the prefix, so a future rename of `name:`
  fails the smoke test loudly rather than silently checking
  the wrong volumes.

### `.github/workflows/ci.yml` — new `compose-smoke` job

```yaml
  compose-smoke:
    name: compose-smoke
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Verify docker compose is available
        run: docker compose version
      - name: Run compose smoke
        working-directory: reference/compose
        run: bash healthcheck/smoke.sh
```

GitHub Actions' `ubuntu-latest` runners ship with Docker and the
Compose v2 plugin pre-installed; no extra setup needed.

We do **not** add `compose-smoke` to branch protection in this PR.
Reasoning: the existing required checks (`docs-lint`,
`schema-validity`, `python-lint`, `python-typecheck`, `python-test`,
`schema-parity`) are sufficient to gate a 10a merge; adding
`compose-smoke` to protection on the *same* PR that introduces it
risks a chicken-and-egg if the job is flaky on first run. Add to
protection in 10b once the job has a few clean runs under its belt.

Order matters: each step assumes the previous one passed.

1. **Local compose up.** From `reference/compose/`:

   ```bash
   cp .env.example .env
   docker compose up -d --wait --wait-timeout 120
   docker compose ps
   ```

   Expected: `postgres` and `gitea` both `running (healthy)`;
   `blob-init` `exited (0)`. Then:

## Known Risks / Things to Watch

- **GitHub Actions Docker version drift.** `ubuntu-latest` updates
  Docker periodically; Compose v2 syntax we use (`--wait`,
  `--wait-timeout`, `name:` top-level field, no `version:` key) all
  require Docker Compose v2.20+. As of April 2026 `ubuntu-latest`
  ships v2.27+, so we have margin, but if the runner ever
  downgrades the smoke job will break loudly. We don't pin Docker
  in CI because GitHub controls that.
- **Gitea startup is slow.** First boot of `gitea/gitea:1.22.6-rootless`
  takes 15-25 seconds on a cold runner. The healthcheck
  `start_period: 30s` covers this; if it ever proves flaky,
  the next move is bumping to 60s rather than reducing the test.
- **Image-pin staleness.** `postgres:16.6-alpine` and
  `gitea/gitea:1.22.6-rootless` will be superseded. We pin
  deliberately; Phase 10e or a dedicated bump PR refreshes both
  together when the next Compose round lands.
- **Port-collision regret.** Defaults of 5433 / 3001 / 2222 sidestep
  the most common collisions but a developer running their own
  Gitea on 3001 will hit one. The README's Troubleshooting section
  documents the `.env` overrides; no other mitigation is worth it.
- **`.env.example` rotation.** The dev-only secrets in `.env.example`
  are intentionally weak. If a future contributor copies them to a
  real deployment they're in trouble. The README's Quickstart
  section labels the values as throwaway; we additionally make
  `POSTGRES_PASSWORD`, `GITEA_SECRET_KEY`, `GITEA_INTERNAL_TOKEN`
  required-with-error-on-missing in `compose.yaml`, so production
  users who delete `.env` get a clear failure rather than silently
  inheriting weak defaults.
- **Compose project-name leakage.** The `name: eden-reference`
  field means volumes get the prefix `eden-reference_`. If a
  developer also runs the predecessor's direvo monolith via
  Compose, project names won't collide. If they ever rename our
  `name:` field, every `docker volume ls` reference and every
  `compose down -v` invocation that targets historical data will
  miss the old volumes. We document the project name in the README.
- **Compose v1 (`docker-compose`) users.** v1 is EOL since June
  2023. Some legacy systems still ship it; those users will see
  syntax errors on `name:` and `--wait`. The README's prerequisites
  section names `docker compose` v2.20+ explicitly. No backcompat
  shim.
