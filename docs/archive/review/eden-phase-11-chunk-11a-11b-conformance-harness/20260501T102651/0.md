# EDEN Phase 11 chunk 11a + 11b — Conformance harness + state-machine scenarios

## Context

[`docs/roadmap.md`](../roadmap.md) Phase 11 chunks the conformance suite as
`11a + 11b` (one chunk), `11c` (one), `11d` (one). The harness and the
first scenarios travel together because the harness has nothing to
validate itself against until at least one scenario is in place — the
"first scenarios validate the harness itself" bootstrap problem.

[`spec/v0/`](../../spec/v0/) currently ships chapters 00 – 08 plus the wire
binding (07) and is missing chapter 09. [`conformance/`](../../conformance/)
is a `README.md` stub reserving the location.

This plan delivers exactly what chunk 11a + 11b says it should:

1. The harness — pytest-driven, IUT-adapter-shaped, two scenario marker
   groups (`conformance_core` and `conformance_wire_http`).
2. The first state-machine scenarios — task lifecycle, claim-token
   semantics, transactional event invariant.
3. The new normative chapter — [`spec/v0/09-conformance.md`](../../spec/v0/09-conformance.md)
   — defining the IUT contract, the assertion vocabulary, and the
   conformance-level taxonomy (`core` vs `core + wire-http`).
4. The reference IUT adapter that drives the existing
   `task-store-server` against the harness.
5. A self-validation scenario that asserts the harness rejects a
   deliberately-broken adapter.
6. A new `conformance` CI job that runs the suite against the reference
   impl on every PR.

## Reframe

The conformance suite is the **second** thing the spec promises (after
the spec itself) and the **first** thing that makes spec compliance an
externally-verifiable claim rather than an internal assertion. Until
this chunk lands, "the reference impl conforms to the spec" is a
sentence the maintainers say. After this chunk lands, "the reference
impl passes the v1 conformance suite, scenario-by-scenario, every PR"
is a sentence CI says.

The chunk's success criterion is therefore **not** "the reference impl
passes the suite" — that's a tautology if I author the suite by
inspecting the reference impl. The criterion is the chapter-09 +
scenarios pair: an independent reader of [`spec/v0/`](../../spec/v0/)
who builds their own task-store-server should be able to run the
harness against their service unmodified and have the pass/fail signal
be meaningful — which means **every assertion the harness makes MUST
be backed by a normative MUST in chapters 02 / 04 / 05 / 06 / 07.** No
"the reference impl happens to do this" assertions; every pass/fail
must be cite-able. That discipline is the load-bearing part of the
chunk; the pytest plumbing is incidental.

## Scope

### In scope (this chunk)

1. **Harness infrastructure** under [`conformance/harness/`](../../conformance/harness/):
   - `IutAdapter` Protocol — start/stop a subject, return a handle that
     exposes a wire-protocol base URL + bearer token.
   - pytest plugin that wires fixtures (`iut`, `wire_client`, `experiment_id`,
     `event_cursor_helper`) and the two scenario markers.
   - Fixture corpus — minimal experiment-config + empty bare repo
     factory used to spin a clean IUT per session (or per scenario,
     based on `--iut-mode` configurability).

2. **Reference IUT adapter** under
   [`conformance/adapters/reference/`](../../conformance/adapters/reference/):
   - Spawns `python3 -m eden_task_store_server` in a subprocess on an
     ephemeral port, parses the `EDEN_TASK_STORE_LISTENING` line for the
     port, and tears the subprocess down on `stop()`.
   - The reference adapter is a **task-store-only** subject. The chunk
     does NOT spawn the reference orchestrator or worker hosts — chunk
     11a + 11b's scope is task-store + wire-binding behavior. Role
     contracts (which would require driving the orchestrator alongside
     the store) are chunk 11c.

3. **State-machine scenarios** (`conformance_core` markers):
   - Task lifecycle: every legal transition; every illegal transition
     returns the spec-pinned rejection.
   - Claim tokens: §3.2 freshness, §3.3 authorization, §3.4 no-reclaim-
     while-claimed.
   - Submit idempotency: §4.2 content-equivalent acceptance vs divergent
     rejection.
   - Reclamation: §5.1 case matrix; §5.2 token invalidation; §5.4
     starting-trial reconciliation on implement reclaim.
   - Transactional event invariant: §1.3 — every state change has a
     corresponding event; subscribers cannot observe a state without
     the event.
   - Composite-commit invariants: chapter 5 §2.2 — `create_task(implement)`
     is atomic with the proposal `ready → dispatched` transition.

4. **Wire-binding-HTTP scenarios** (`conformance_wire_http` markers):
   - Path versioning: `/v0/` rooting; `X-Eden-Experiment-Id` header
     enforcement.
   - Status-code mapping: every operation returns the spec-pinned status.
   - RFC 7807 problem+json envelope shape on every non-2xx response.
   - Closed `eden://error/<name>` vocabulary: every observed error type
     is in the §7 table; no others appear.
   - Same-value idempotency on `integrate_trial` per §5.

5. **Self-validation scenario** under
   [`conformance/scenarios/_meta/`](../../conformance/scenarios/_meta/):
   A `pytest.mark.conformance_meta` scenario that runs the suite against a
   `BrokenAdapter` (a reference adapter wrapper that mutates one wire
   response — e.g. returning the wrong claim token) and asserts at least
   one core scenario fails. This proves the harness has teeth.

6. **Spec chapter [`spec/v0/09-conformance.md`](../../spec/v0/09-conformance.md)**:
   - §1 What conformance means.
   - §2 The IUT contract (wire endpoints + adapter shim shape).
   - §3 Assertion vocabulary (RFC 2119 in the suite, mapped to spec MUSTs).
   - §4 Conformance levels: `core`, `core + wire-http`.
   - §5 Scenario index (v1 = state-machine + wire-http; v2 hooks for
     11c/11d).
   - §6 The IUT-adapter shape.

7. **CI wiring**: new `conformance` job in
   [`.github/workflows/ci.yml`](../../.github/workflows/ci.yml) that
   `uv sync`s the workspace, then runs `uv run pytest -q conformance/`.
   Branch protection is **not** updated to require it in this chunk
   (same posture compose-smoke / compose-smoke-subprocess /
   compose-smoke-subprocess-docker / compose-e2e took for newly-added
   jobs — let it run cleanly for a few iterations on `main` first).

8. **AGENTS.md commands table** entry for
   `uv run pytest -q conformance/`.

### Out of scope (deferred to 11c / 11d / 12 / 13)

- **Role-contract scenarios** (planner / implementer / evaluator
  submission semantics): chunk 11c.
- **Integrator scenarios** (squash shape, eval-manifest shape, `work/*`
  access discipline): chunk 11d.
- **Multi-experiment / lease scenarios**: Phase 12.
- **Non-HTTP wire bindings**: not currently spec'd. The
  `conformance_core` / `conformance_wire_http` marker split is the
  affordance for them; no actual second binding exists.
- **Conformance against deployed Compose stack**: the reference adapter
  spawns one Python subprocess against an in-process `SqliteStore` at
  `:memory:`. Container-level conformance is `compose-smoke` /
  `compose-e2e`'s job.
- **Concurrency stress / chaos scenarios**: future Phase 11 follow-up if
  they prove necessary; the v1 suite asserts behavior at the spec
  level, not under fault injection.
- **Subscribe / push transports**: chapter 8 §2.1 / chapter 7 §6.2 long-
  poll subscribe IS in scope (the v1 task lifecycle scenarios verify
  subscribe yields each state-change event), but SSE/WebSocket push is
  OPTIONAL per chapter 7 §6.2 and out of scope here.

### B. The IUT-adapter Protocol

In [`conformance/harness/adapter.py`](../../conformance/harness/adapter.py):

```python
from __future__ import annotations

from dataclasses import dataclass
from pathlib import Path
from typing import Protocol


@dataclass(frozen=True)
class IutHandle:
    """Handle returned by IutAdapter.start; everything the harness needs
    to drive the IUT.
    """

    base_url: str           # e.g. "http://127.0.0.1:54321"
    shared_token: str       # bearer for Authorization: Bearer <token>
    experiment_id: str      # the experiment the IUT was started against


class IutAdapter(Protocol):
    """Conformance-suite adapter for an EDEN task-store implementation.

    A conforming third-party implementation provides ONE concrete
    subclass. The reference adapter under conformance/adapters/reference
    is one example; non-Python implementations write their own.

    The adapter's only obligation is lifecycle: start the subject in a
    state that matches the harness's preconditions; tear it down
    completely on stop. Adapters MUST NOT reach into the subject's
    internal state during a scenario; the harness exercises the wire
    binding only.
    """

    def start(
        self,
        *,
        experiment_config_path: Path,
        repo_path: Path,
        experiment_id: str,
    ) -> IutHandle: ...

    def stop(self) -> None: ...
```

The handle is **flat** — no callable for "fetch a fresh token" or
similar — because every secret a scenario needs is observable through
the wire binding. An adapter that needs to do extra setup (seed the
bare repo, run migrations) does so inside `start` before returning.

### C. The pytest plugin

In [`conformance/harness/plugin.py`](../../conformance/harness/plugin.py):

```python
import importlib
import os
import pytest
from pathlib import Path

from .adapter import IutAdapter, IutHandle
from .wire_client import WireClient
from .event_cursor import EventCursorHelper


def pytest_configure(config: pytest.Config) -> None:
    config.addinivalue_line(
        "markers",
        "conformance_core: scenario asserts a chapter-02/04/05/06/08 invariant; "
        "all conforming IUTs MUST pass.",
    )
    config.addinivalue_line(
        "markers",
        "conformance_wire_http: scenario asserts a chapter-07 wire-binding shape; "
        "an IUT claiming wire-http conformance MUST pass.",
    )
    config.addinivalue_line(
        "markers",
        "conformance_meta: harness self-validation; not part of any "
        "conformance level.",
    )


def pytest_addoption(parser: pytest.Parser) -> None:
    parser.addoption(
        "--iut-adapter",
        default="conformance.adapters.reference.adapter:ReferenceAdapter",
        help="Importable path to the IutAdapter implementation.",
    )
    parser.addoption(
        "--iut-mode",
        default="session",
        choices=("session", "scenario"),
        help="Restart the IUT per scenario (slow but isolated) or share "
        "across the whole session (fast).",
    )


@pytest.fixture(scope="session")
def iut_adapter_factory(pytestconfig: pytest.Config) -> "Callable[[], IutAdapter]":
    spec = pytestconfig.getoption("--iut-adapter")
    module_name, cls_name = spec.split(":")
    cls = getattr(importlib.import_module(module_name), cls_name)
    return cls
```

The fixture machinery for `iut`, `wire_client`, etc. follows the standard
pytest pattern: a session-scoped fixture starts the adapter once, a
function-scoped fixture issues a fresh `experiment_id` per test (via
`uuid4()`) so cross-scenario state doesn't leak. Scenario tests parametrize
on a fresh `experiment_id` even in `--iut-mode=session`, since the wire
binding is experiment-scoped per chapter 07 §1.3.

The `WireClient` in [`conformance/harness/wire_client.py`](../../conformance/harness/wire_client.py) is a **thin** httpx wrapper:
it talks raw HTTP/JSON, NOT through the existing `eden_wire.StoreClient`.
This is deliberate. The existing `StoreClient` does retry, decode-into-
typed-objects, problem+json-aware error parsing — i.e. it papers over
exactly the wire-binding details the conformance suite needs to assert.
The conformance harness needs to see status codes, problem+json envelope
fields, Content-Type headers, and raw response bodies. Thin wrapper, no
auto-decoding beyond `response.json()` for 2xx, no auto-retry, no
StoreClient.

### D. The reference IUT adapter

In [`conformance/adapters/reference/adapter.py`](../../conformance/adapters/reference/adapter.py):

```python
import secrets
import subprocess
import sys
import threading
import time
from pathlib import Path

from conformance.harness.adapter import IutAdapter, IutHandle


class ReferenceAdapter:
    """Spawns a fresh task-store-server subprocess against in-memory storage."""

    def __init__(self) -> None:
        self._proc: subprocess.Popen | None = None
        self._stderr_lines: list[str] = []
        self._stderr_thread: threading.Thread | None = None

    def start(
        self,
        *,
        experiment_config_path: Path,
        repo_path: Path,
        experiment_id: str,
    ) -> IutHandle:
        token = secrets.token_urlsafe(24)
        self._proc = subprocess.Popen(
            [
                sys.executable,
                "-m", "eden_task_store_server",
                "--store-url", ":memory:",
                "--experiment-id", experiment_id,
                "--experiment-config", str(experiment_config_path),
                "--shared-token", token,
                "--port", "0",
                "--host", "127.0.0.1",
            ],
            stdout=subprocess.PIPE,
            stderr=subprocess.PIPE,
            text=True,
        )
        port = self._read_port_announcement(timeout=15.0)
        self._start_stderr_drain()
        return IutHandle(
            base_url=f"http://127.0.0.1:{port}",
            shared_token=token,
            experiment_id=experiment_id,
        )

    def stop(self) -> None:
        if self._proc and self._proc.poll() is None:
            self._proc.terminate()
            try:
                self._proc.wait(timeout=5.0)
            except subprocess.TimeoutExpired:
                self._proc.kill()
                self._proc.wait(timeout=2.0)
        # Capture stderr if the test failed; pytest formatter will surface it.
        ...
```

The same `_read_port_announcement` helper used in
[`reference/services/orchestrator/tests/test_e2e.py`](../../reference/services/orchestrator/tests/test_e2e.py)
is duplicated here intentionally — copy is cheap, dependency from
`conformance/` into `reference/services/orchestrator/tests/` is wrong
direction. (`conformance/` MUST not import from `reference/` outside
of `conformance/adapters/reference/`. CI gates this; see §H.)

The reference adapter takes a deliberate shortcut: it uses
`SqliteStore(":memory:")` rather than `PostgresStore`. The conformance
suite doesn't care about durability semantics — that's
[`reference/packages/eden-storage/tests/`](../../reference/packages/eden-storage/tests/)'s
job. The suite cares about wire-observable behavior. SQLite-in-memory
gives the fastest possible test cycle.

### E. Scenario inventory

#### E.1 Core scenarios — task lifecycle (`test_task_lifecycle.py`)

Each test creates a fresh `plan` task and exercises one transition.
Citations indicate the spec MUST being tested.

| Test | Asserts |
|---|---|
| `test_create_task_pending` | After `POST /tasks`, `state == "pending"`, no `claim`. (chapter 04 §2; chapter 02 §3.1) |
| `test_pending_to_claimed` | `POST /tasks/{T}/claim` returns 200 + claim object; subsequent `GET /tasks/{T}` shows `state == "claimed"`. (chapter 04 §1.2; §3.2) |
| `test_claimed_to_submitted` | `POST /tasks/{T}/submit` with current token + plan payload returns 200; `state == "submitted"`. (chapter 04 §4.1) |
| `test_submitted_to_completed` | `POST /tasks/{T}/accept` returns 200; `state == "completed"`. (chapter 04 §4.3) |
| `test_submitted_to_failed_via_reject` | `POST /tasks/{T}/reject` with valid `reason` returns 200; `state == "failed"`. (chapter 04 §4.3) |
| `test_terminal_completed_rejects_writes` | After `accept`, every write op (claim, submit, accept, reject, reclaim) returns 409 `eden://error/illegal-transition`. (chapter 04 §1.1; §4.4) |
| `test_terminal_failed_rejects_writes` | Same as above for `failed`. (chapter 04 §1.1) |
| `test_pending_rejects_submit` | `POST /tasks/{T}/submit` against pending task returns 409 illegal-transition. (chapter 04 §1.2) |
| `test_pending_rejects_accept` | Likewise for `accept`. |
| `test_pending_rejects_reject` | Likewise for `reject`. |
| `test_claimed_rejects_accept` | Accept-without-submit returns 409 illegal-transition. |
| `test_claim_rejected_when_not_pending` | Re-claim a `claimed` task returns 409 illegal-transition. (chapter 04 §3.4) |

#### E.2 Core scenarios — claim tokens (`test_claim_tokens.py`)

| Test | Asserts |
|---|---|
| `test_token_is_present` | Successful claim response carries `claim.token`. (chapter 02 §3.4) |
| `test_token_is_unique_across_tasks` | Two parallel claims on different tasks produce different tokens. (chapter 04 §3.2 unique) |
| `test_token_is_unique_across_reclaim_cycles` | Reclaim → re-claim same task → token differs from prior claim. (chapter 04 §3.2 unique; §5.2) |
| `test_wrong_token_rejected` | `submit` with a fabricated token returns 403 `eden://error/wrong-token`. (chapter 04 §3.3) |
| `test_token_invalidated_by_reclaim` | After `POST /tasks/{T}/reclaim cause=operator`, the prior token returns 403 wrong-token on submit. (chapter 04 §5.2) |
| `test_no_reclaim_while_claimed_no_op_reject` | Claiming a `claimed` task by a second worker returns 409 illegal-transition. (chapter 04 §3.4) |

The harness does NOT attempt to assert "tokens are unforgeable" — that's
a cryptographic property and the spec uses RFC-2119 SHOULD/MAY language
for the mechanism (chapter 04 §3.2). The MUST is uniqueness, which is
testable.

#### E.3 Core scenarios — submit idempotency (`test_submit_idempotency.py`)

| Test | Asserts |
|---|---|
| `test_resubmit_content_equivalent_returns_200` | Second submit with identical payload returns 200, no state-change event added. (chapter 04 §4.2) |
| `test_resubmit_divergent_returns_409` | Second submit with different proposal_ids set returns 409 `eden://error/conflicting-resubmission`. (chapter 04 §4.2) |
| `test_plan_proposal_ids_compared_as_set` | Two submits with same proposal_ids in different order are content-equivalent. (chapter 04 §4.2 plan rule) |
| `test_evaluate_metrics_compared_as_json` | Evaluate resubmit with same metrics in different key order is content-equivalent. (chapter 04 §4.2 evaluate rule) |
| `test_resubmit_after_terminal_rejected` | After `accept`, resubmit with the original token returns 409 illegal-transition (NOT `wrong-token`, since the task is past terminal — the request is illegal regardless of token validity). (chapter 04 §4.4) |

Two submit-idempotency tests carry seeded data dependencies: a `plan`
test needs no external state; an `implement` test needs a `proposal`
in `state == "ready"` before the implement task can be created. The
harness ships a `_seeding` helper module that creates these via the
wire binding — no shortcuts through the adapter.

#### E.4 Core scenarios — reclamation (`test_reclamation.py`)

| Test | Asserts |
|---|---|
| `test_reclaim_operator_from_claimed` | `POST /tasks/{T}/reclaim cause=operator` against a claimed task: 200; state back to `pending`; claim cleared; `task.reclaimed` event with `cause=operator` in the log. (chapter 04 §5.1, §5.2; chapter 05 §3.1) |
| `test_reclaim_operator_from_submitted` | Same for submitted. (chapter 04 §5.1) |
| `test_reclaim_expired_from_claimed` | After issuing a claim with `expires_at` in the past, `POST /tasks/{T}/reclaim cause=expired` succeeds. (chapter 04 §5.1) |
| `test_reclaim_expired_against_submitted_rejected` | Cannot apply automatic reclaim to `submitted`. (chapter 04 §5.1) |
| `test_reclaim_terminal_rejected` | Cannot reclaim `completed` or `failed`. (chapter 04 §5.1) |
| `test_implement_reclaim_sets_starting_trial_to_error` | If implement reclaim happens after `create_trial(status="starting")`, the trial flips to `error` atomically. (chapter 04 §5.4) |
| `test_reclaim_cause_vocabulary_closed` | `POST /tasks/{T}/reclaim cause="garbage"` returns 400. (chapter 07 §2.6) |

#### E.5 Core scenarios — event invariants (`test_event_invariants.py`)

| Test | Asserts |
|---|---|
| `test_every_state_change_emits_event` | After each transition in §E.1, `GET /events` includes the matching event. (chapter 04 §1.3; chapter 05 §1) |
| `test_event_emitted_atomically_with_state_change` | Concurrent reads cannot observe the new state without the event also being present. The harness asserts this by polling `GET /tasks/{T}` and `GET /events` in alternation around an in-flight transition; if a state change is observed, the next read MUST yield its event. (chapter 04 §1.3 "atomically with") |
| `test_total_order_per_task` | `GET /events` filtered to a task ID returns events in transition order. (chapter 04 §6.1, chapter 05 §4) |
| `test_at_least_once_via_replay` | `GET /events?cursor=0` includes every event subsequent reads also see. (chapter 05 §4.4) |
| `test_subscribe_long_poll_yields_event` | Open `GET /events/subscribe?cursor=N` then issue a transition; the long-poll returns the new event within the chapter 7 §6.2 timeout. (chapter 07 §6.2) |
| `test_create_implement_task_atomic_with_proposal_dispatch` | `POST /tasks` for `implement` flips referenced proposal `ready → dispatched`; both are visible after one read; the matching `task.created` and `proposal.state-changed` (or equivalent, per chapter 5 §3) events both appear. (chapter 04 §2; chapter 05 §2.2) |

The atomicity test in `test_event_emitted_atomically_with_state_change`
is the **only** scenario in v1 that uses concurrent threads in the
harness. It is intentionally lightweight: the harness drives the
transition in one thread while another thread polls; it asserts the
ordering invariant holds across the polling window. A more rigorous
linearizability check (à la Jepsen) is out of scope.

#### E.8 Wire-http scenarios — problem+json envelope (`test_problem_json.py`)

For each error-class status code (400, 403, 404, 409), the test triggers
the error and asserts:

- `Content-Type: application/problem+json; charset=utf-8`
- Body parses as JSON and matches
  [`spec/v0/schemas/wire/error.schema.json`](../../spec/v0/schemas/wire/error.schema.json)
  (`type`, `title`, `status`, `detail`, `instance` all present and well-formed).
- `type` is a URI matching `eden://error/<name>`.
- `status` field equals the HTTP status.

#### E.9 Wire-http scenarios — error vocabulary (`test_error_vocabulary.py`)

| Test | Asserts |
|---|---|
| `test_observed_types_are_in_vocabulary` | Every error response observed across the wire-http suite has `type` in the chapter 07 §7 closed table. (chapter 07 §7 closure rule) |
| `test_no_internal_eden_error_type_for_auth` | Unauthenticated request returns 401 with `type` outside the `eden://error/` namespace (`eden://reference-error/unauthorized` for the reference IUT, or any non-`eden://error/` URI for a third party). (chapter 07 §12) |

The vocabulary-closure assertion is keyed off a module-level set
populated by every wire-http test that triggers an error: each test
registers the `type` it expects, and a session-scoped post-hook asserts
the union of observed types is a subset of the §7 vocabulary AND every
§7 entry was observed by at least one test (so the suite is exercised
exhaustively).

#### E.10 Wire-http scenarios — integrate idempotency (`test_integrate_idempotency.py`)

| Test | Asserts |
|---|---|
| `test_first_integrate_returns_200` | First call with `trial_commit_sha` returns 200 + empty body. (chapter 07 §5) |
| `test_same_value_idempotency` | Second call with same SHA returns 200; `read_range` shows exactly one `trial.integrated` event. (chapter 07 §5 same-value) |
| `test_different_sha_returns_409` | Second call with different SHA returns 409 invalid-precondition; the stored value is unchanged. (chapter 07 §5 different-value) |
| `test_integrate_against_non_success_trial_returns_409` | Trial in `status != "success"` rejects integrate with invalid-precondition. (chapter 07 §5 other preconditions) |

Integrate scenarios need a real `Trial` object with `status="success"`,
a `commit_sha`, and a reachable git artifact. The harness uses a
**fixture trial-shape**: it manufactures the trial's prerequisites
through the wire binding (create plan task → submit → accept → create
proposal → mark ready → create implement task → submit → accept → which
yields a trial with `commit_sha`) and then drives it to `success` by
creating an evaluate task and submitting `status=success` with valid
metrics. **No** shortcuts through the adapter; everything is observable
wire-protocol traffic. This setup is wrapped in a session-scoped
`integrated_trial_factory` fixture so it doesn't run per test.

The reference adapter's repo-path is set at adapter-`start` time. The
harness creates a temp dir, runs `python3 -m eden_service_common.repo_init
--repo-path <dir>` to seed it (since
[`reference/services/_common/src/eden_service_common/__init__.py`](../../reference/services/_common/src/eden_service_common/__init__.py)
already exposes `seed_bare_repo`, the adapter MAY call it directly via
import as a shortcut — that's reference-adapter-specific and a third-
party adapter would do its own seed). The integrator helper computes
the trial's commit SHA from the worker's submitted SHA following
chapter 6 §3.2.

### F. Self-validation scenario

In [`conformance/scenarios/_meta/test_self_validation.py`](../../conformance/scenarios/_meta/test_self_validation.py):

```python
import pytest

from conformance.adapters.reference.adapter import ReferenceAdapter

pytestmark = pytest.mark.conformance_meta


class _BrokenAdapter(ReferenceAdapter):
    """Wraps the reference adapter but injects a deliberately wrong
    behavior: the wire-http layer returns 200 OK for every wrong-token
    submit, instead of 403 wrong-token.

    The harness's E.2 test_wrong_token_rejected MUST fail when run
    against this adapter. If the harness passes against this adapter,
    the harness has no teeth.
    """
```

The test mechanism: the meta scenario uses `pytest.main(...)` to invoke
a sub-pytest run targeted at `conformance/scenarios/core/test_claim_tokens.py::test_wrong_token_rejected`
with `--iut-adapter=conformance.scenarios._meta.test_self_validation:_BrokenAdapter`,
captures the result code, and asserts it's non-zero. This is the
"detect-tampering" anchor: a future contributor who breaks a real
scenario will fail this meta test along with the underlying scenario,
which surfaces the integrity claim explicitly.

The broken adapter cannot be implemented purely as a Python subclass of
`ReferenceAdapter` because the broken behavior lives in the
task-store-server subprocess, not in the adapter layer. Three options:

(a) **Patch the reference subprocess at adapter-start time** by passing
    a `--reference-debug-misbehave-wrong-token-as-success` flag to
    `eden_task_store_server`. This requires adding a debug flag to the
    server. **Rejected**: pollutes production code with conformance-
    suite scaffolding.

(b) **Use an HTTP proxy in front of the IUT** that mutates the wrong-
    token 403 response into a 200. The broken adapter starts the real
    IUT on port X, then starts a proxy on port Y that forwards to X
    and rewrites one specific response. The handle's `base_url` points
    to port Y. **Selected**.

(c) **Replace the reference adapter wholesale with a hand-rolled FastAPI
    app** that misbehaves on one operation. **Rejected**: re-implements
    most of the wire binding just to break one rule.

The proxy in (b) lives at
[`conformance/scenarios/_meta/_misbehave_proxy.py`](../../conformance/scenarios/_meta/_misbehave_proxy.py)
and uses `aiohttp` (already a transitive dep of the reference services
via httpx/anyio chains, but pin it explicitly in
`conformance/pyproject.toml`). It's ~50 lines of code: forward every
request, special-case `/v0/experiments/{E}/tasks/{T}/submit` with a
wrong token by returning 200 instead of 403.

### G. Spec chapter 09

In [`spec/v0/09-conformance.md`](../../spec/v0/09-conformance.md), structure:

```markdown
# Conformance

## 1. Purpose

Conformance is a testable claim that an EDEN component implements the
behavior chapters 02 / 04 / 05 / 06 / 07 / 08 prescribe.

## 2. The IUT contract

A subject of conformance — an "implementation under test", IUT — MUST:

- Expose the wire binding from chapter 07 at a deployment-chosen base URL.
- Accept an experiment configuration whose shape matches `experiment-config.schema.json`.
- Be startable and stoppable via a documented adapter shim.

The chapter does NOT mandate an implementation language for the IUT or
for the suite that drives it. The reference suite is Python (see §6).

## 3. Assertion vocabulary

Every assertion in the suite is keyed off a normative MUST in chapters
02 / 04 / 05 / 06 / 07 / 08 (or, for §6.2 long-poll subscribe,
chapter 07 §6.2). The suite does NOT assert SHOULDs; SHOULDs are
implementation guidance, not interop contracts.

When the spec evolves, the suite evolves with it. A change that
removes a MUST or downgrades it to SHOULD MUST be accompanied by a
matching scenario removal or marker change.

## 4. Conformance levels

Conformance is claimed at one of two levels:

- **`core`** — the IUT implements the behavior in chapters 02 / 04 / 05 /
  06 / 08 over its chosen transport. The suite verifies this via the
  `conformance_core` marker group, exercising the IUT through the
  wire binding only.

- **`core + wire-http`** — the IUT additionally implements chapter 07,
  the HTTP wire binding. The suite verifies this via the
  `conformance_wire_http` marker group on top of the `conformance_core`
  group.

A v0 IUT that does NOT implement chapter 07 cannot be exercised by the
v1 suite as written, because the suite drives the IUT through HTTP. The
levels exist to anchor a future binding (gRPC, in-process) without
re-cutting the chapter; until that binding exists, every conforming v0
IUT is `core + wire-http`.

## 5. Scenario index (v1)

| Group | Scope | Marker | Spec citations |
|---|---|---|---|
| Task lifecycle | every legal/illegal transition | `conformance_core` | 04 §1, §3.4, §4 |
| Claim tokens | freshness, authorization, no-reclaim-while-claimed | `conformance_core` | 04 §3 |
| Submit idempotency | content-equivalent / divergent / terminal | `conformance_core` | 04 §4.2, §4.4 |
| Reclamation | case matrix; token invalidation; trial reconciliation | `conformance_core` | 04 §5 |
| Event invariants | atomicity, total order, replay, subscribe | `conformance_core` | 04 §1.3, §6; 05 §1, §4 |
| Path versioning | `/v0/` rooting; experiment-id header | `conformance_wire_http` | 07 §1 |
| Status codes | every operation's spec-pinned status | `conformance_wire_http` | 07 §§2–6 |
| Problem+json envelope | shape + content-type | `conformance_wire_http` | 07 §7 |
| Error vocabulary | closed `eden://error/<name>` set | `conformance_wire_http` | 07 §7 |
| Integrate idempotency | same-value / different-value / preconditions | `conformance_wire_http` | 07 §5 |

v2 (chunk 11c) extends with role-contract scenarios; v3 (chunk 11d)
adds integrator scenarios.

## 6. The IUT-adapter shape

The reference suite's adapter shape is documented in
`conformance/harness/adapter.py`. A non-Python IUT writes its own
adapter; the only contract is the lifecycle pair (start → handle, stop)
and the handle fields (`base_url`, `shared_token`, `experiment_id`).
This is informative; nothing in the chapter requires the suite be
written in Python.

## 7. Conformance posture vs the reference impl

The reference implementation in [`reference/`](../../reference/)
exists to demonstrate one valid implementation. It MUST pass the v1
suite; CI gates this on every PR via the `conformance` job. A reference-
impl test failure that CANNOT be cited to a normative MUST is either a
suite bug or a spec gap; the resolution path is one of: amend the
suite, amend the spec, fix the impl. The reference impl does NOT have
authority to reinterpret the spec.
```

The chapter is intentionally short. Conformance is a meta-document; the
real content is in chapters 02 / 04 / 05 / 06 / 07 / 08 and in the
suite itself. Chapter 09's job is to declare the relationship.

### I. Spec ↔ harness traceability

### K. Self-host guarantee

The harness's design lets it be invoked **without** the conformance/
package as a development dependency of the reference services — i.e.
running `uv run pytest -q reference/` doesn't pull in conformance
machinery. This is enforced by:

- `conformance/pyproject.toml` declares `conformance` as a separate uv
  workspace member with its own dependency closure (pytest, httpx,
  aiohttp).
- The reference services' tests do NOT import from `conformance/`.
- `pyproject.toml` (root) lists `conformance` under the
  `[tool.uv.workspace] members` array but the reference packages don't
  depend on it.

This is what lets a non-Python implementor consume the suite: they
clone the repo, `uv sync --package conformance` (or whatever the
non-Python equivalent is), and run pytest pointed at their adapter.
They never run `uv sync --all-packages`.

## Known risks / things to watch

- **Chapter-09 chickens-and-eggs.** The chapter cites the suite; the
  suite cites the chapter. To break the cycle the chapter is written
  first as the normative source, then the suite is written to match.
  When this chunk's plan goes to codex-review, expect the reviewer to
  push hard on chapter-09's section §3 ("does the suite assert SHOULDs?
  what about MAYs?"). The answer is the SHOULD/MAY guidance from
  RFC 2119: SHOULDs are interop guidance, not interop contracts; the
  suite asserts MUSTs only. Codify this explicitly in §3 of the chapter.

- **Atomicity test in §E.5 is the most fragile assertion.** The test
  polls `GET /tasks/{T}` and `GET /events` from a separate thread while
  the main thread drives a transition. Under fast-enough completion it
  may never observe the in-flight window, in which case the test is
  vacuously passing. Mitigation: the harness uses
  `event_cursor.subscribe_async()` (a long-poll wrapper) that BLOCKS on
  the next event, then issues a `GET /tasks/{T}` — if the state has
  advanced, the just-returned event MUST describe that state's
  transition. This is a tighter assertion than free-running poll-and-
  hope. (Inspired by the reference impl's
  [`reference/packages/eden-wire/`](../../reference/packages/eden-wire/)
  long-poll handling.) The plan's first round of review should pressure-
  test whether this is a sufficient atomicity assertion or a placeholder.

- **Self-validation broken-adapter is a moving target.** If a future
  refactor changes how the reference IUT formats wrong-token responses
  (say, the proxy needs to mutate a different envelope field), the
  meta scenario silently stops failing the broken adapter and the
  harness loses its self-test. Mitigation: the meta scenario also
  asserts that the inner pytest run named the specific failing test ID
  it expected — `test_claim_tokens.py::test_wrong_token_rejected` —
  not just "non-zero exit". A drift in error-shape that no longer
  causes that specific test to fail surfaces as a meta-test failure.

- **`integrated_trial_factory` setup is non-trivial.** The chain
  plan→propose→implement→evaluate→integrate is ~80 lines of fixture
  scaffolding; bugs in the fixture become bugs in unrelated scenarios.
  Mitigation: the fixture is itself unit-tested by a
  `conformance_meta`-marked test (
  `test_integrated_trial_factory_produces_status_success_trial`) — i.e.
  the fixture has a self-test that verifies its postcondition.

- **The wire-http vocabulary-closure assertion (§E.9) is order-
  dependent.** It needs every other wire-http test to have run first.
  Pytest's default test ordering is filesystem-walk-order which is
  stable but not strictly defined. Mitigation: the closure test is in
  `test_error_vocabulary.py` and is gated by a session-scoped
  autouse fixture that orders it last via
  `@pytest.mark.run(order="last")` (`pytest-ordering` plugin). Pin the
  plugin in `conformance/pyproject.toml`.

- **`aiohttp` proxy in §F may be flaky on CI.** The proxy is a small
  asyncio app; its lifecycle (start before adapter, stop after) needs
  care to avoid leaking sockets between meta-test invocations.
  Mitigation: the proxy is a context-managed object; the broken adapter's
  `start()` opens it, `stop()` closes it. Local test loop hits this
  ~50 times to verify socket cleanup is sound before the chunk PRs.

- **Drift between the chapter's scenario index (§5 of chapter 09) and
  the actual `conformance/scenarios/` tree.** The index is normative
  and the tree is the implementation; if they drift, the chapter
  becomes wrong. Mitigation: `tools/check_citations.py` ALSO validates
  that every scenario file's first-line marker comment matches an entry
  in the chapter's §5 table (or fails CI).
