**Findings**

- Bug — [test_event_delivery.py](/Users/ericalt/Documents/eden/conformance/scenarios/test_event_delivery.py:95): `test_subscribe_empty_batch_after_timeout` hard-codes `elapsed < 8.0` and uses a 10s client timeout. `07-wire-protocol.md §6.2` only says the server uses a server-chosen timeout; 30s is merely RECOMMENDED. A conforming IUT with a 15–30s long-poll would fail this conformance test. Remove the wall-clock bound from the suite and assert only the empty-batch/cursor behavior; keep the shorter timeout as a reference-adapter optimization, not a suite requirement.

- Bug — [test_event_delivery.py](/Users/ericalt/Documents/eden/conformance/scenarios/test_event_delivery.py:52): `test_subscribe_at_least_once_via_reconnect` never exercises `GET /events/subscribe` or a reconnect path. It calls `replay_from(0)` twice on `/events`, so an implementation with a broken long-poll subscriber would still pass. Rewrite this around `event_log.subscribe(...)`, reconnect from the prior cursor, and assert redelivery behavior on the subscribe endpoint itself.

- Bug — [check_citations.py](/Users/ericalt/Documents/eden/conformance/src/conformance/tools/check_citations.py:23): the plan’s load-bearing rule was “every scenario assertion must cite a normative MUST,” and the tooling was supposed to reject SHOULD/MAY citations and verify chapter-9 §5 coverage. The implementation only checks that a cited section exists and that certain filenames exist later in the file. It never inspects whether the citation points at normative MUST language, and it does not derive coverage from chapter 9 at all. That leaves the core conformance-discipline rule unenforced. Fix by parsing chapter 9’s scenario index, resolving each cited section’s text, and rejecting non-MUST citations.

- Bug — [adapter.py](/Users/ericalt/Documents/eden/conformance/src/conformance/adapters/reference/adapter.py:42) and [misbehaving_adapter.py](/Users/ericalt/Documents/eden/conformance/src/conformance/_meta/misbehaving_adapter.py:31): partial startup failures leak subprocesses. If `ReferenceAdapter._read_port_announcement()` raises after `Popen`, or `MisbehavingAdapter.proxy.start()` raises after `super().start()`, the fixture never reaches `yield`, so `stop()` is never called. Wrap startup in `try/except`, clean up partial state, then re-raise.

- Risk — [adapter.py](/Users/ericalt/Documents/eden/conformance/src/conformance/adapters/reference/adapter.py:95): `_read_port_announcement` claims a 15s timeout, but `proc.stdout.readline()` blocks indefinitely, so the deadline is not real. A hung server that never prints a newline can wedge CI forever. Use non-blocking I/O or a reader thread plus queue so the timeout is enforceable.

- Bug — [test_atomicity.py](/Users/ericalt/Documents/eden/conformance/scenarios/test_atomicity.py:18): the plan’s §E.5 scenario was a concurrent long-poll plus immediate/follow-up reads to probe state/event visibility windows. The implementation just does `claim`, then `replay_all()`, then `read_task()`. That is a post-hoc consistency check, not the designed best-effort atomicity regression. Implement the concurrent subscribe/read pattern from the plan or narrow the claim this test makes.

- Risk — [test_composite_commits.py](/Users/ericalt/Documents/eden/conformance/scenarios/test_composite_commits.py:60) and [test_composite_commits.py](/Users/ericalt/Documents/eden/conformance/scenarios/test_composite_commits.py:114): the plan said each composite-commit scenario should assert both component events are present. `test_implement_terminal_completes_proposal` checks only `proposal.completed`, and `test_evaluate_terminal_success_emits_trial_succeeded` checks only `trial.succeeded`. A server could drop `task.completed` on either path and still pass. Assert both events in those scenarios.

- Risk — [test_problem_json.py](/Users/ericalt/Documents/eden/conformance/scenarios/test_problem_json.py:16): `_assert_problem_envelope` never checks the `instance` value even though §7 requires the full five-field envelope, and the helper already accepts `instance_url`. Right now any non-empty string passes. Compare `body["instance"]` to the actual request URL or expected path.

- Risk — [AGENTS.md](/Users/ericalt/Documents/eden/AGENTS.md:139): plan §J required a new “Conformance assertions are spec citations, not impl observations” pitfall entry. The commands were added, but that guardrail was not. That misses one of the plan’s explicit codification steps.

- Risk — [test_task_lifecycle.py](/Users/ericalt/Documents/eden/conformance/scenarios/test_task_lifecycle.py:106): `test_terminal_failed_rejects_writes` does not match the plan’s “same for failed” coverage. It only checks `reclaim`, leaving claim/submit/accept/reject-on-`failed` untested. Expand it to mirror the `completed` sink-state test.

- Risk — [test_self_validation.py](/Users/ericalt/Documents/eden/conformance/scenarios/_meta/test_self_validation.py:117): the plan called for the broken run to fail with the specific substring `"Expected 403 wrong-token, got 200"`. The implementation accepts anything containing `"expected 403"` or `"wrong-token"`, which is looser and can let unrelated failures satisfy the meta-test. Tighten this to the planned message or an equivalently specific structured check.

**Assessment**

Broadly, the implementation hits the planned shape: chapter 9 exists, the harness/adapters/scenarios/meta test/CI job are all present, and the layering is sensible. The main problems are in the load-bearing parts of the design: the citation checker does not actually enforce MUST-only conformance assertions, the event-delivery and atomicity scenarios are materially weaker than the plan, and adapter startup/cleanup is fragile enough to leak or hang subprocesses.

I would not treat this as plan-complete yet. The scaffolding is there, but several of the tests that are supposed to make the suite authoritative are currently too weak or too opinionated to support the conformance claim the plan set out to make.
