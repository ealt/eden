**Findings**

1. Blocker: [ci.yml](/Users/ericalt/Documents/eden/.github/workflows/ci.yml:174) installs `httpx` with bare `pip` on `ubuntu-latest`, but the job never pins or activates a non-system Python. That is brittle at best and currently risky on GitHub-hosted Ubuntu: `ubuntu-latest` now maps to Ubuntu 24.04, GitHub’s runner image documents a system `Python 3.12.3` / `Pip 24.0`, and GitHub’s own runner-images issue tracker documents `pip install` failing there with PEP 668 unless `actions/setup-python` is used. Even if `pip` succeeds, [e2e.sh](/Users/ericalt/Documents/eden/reference/compose/healthcheck/e2e.sh:27) later imports via `python3`, so `pip`/`python3` skew is another avoidable failure mode. This should be `actions/setup-python` plus `python3 -m pip install ...`.

2. Medium: [AGENTS.md](/Users/ericalt/Documents/eden/AGENTS.md:74) still lists “Comprehensive Compose end-to-end test (with web-ui walkthroughs)” under “Commands that will exist in later phases,” even though 10e is already marked complete at [AGENTS.md](/Users/ericalt/Documents/eden/AGENTS.md:19) and the real command row exists at [AGENTS.md](/Users/ericalt/Documents/eden/AGENTS.md:69). That’s stale repo guidance.

3. Medium: [e2e_drive.py](/Users/ericalt/Documents/eden/reference/compose/healthcheck/e2e_drive.py:202) only uses `_fail()` for bad HTTP responses; raw `httpx` transport errors are uncaught. A connect timeout / refused connection / read timeout will dump a traceback instead of the intended concise diagnostic. That does not violate the happy path, but it weakens the failure-mode handling the script is otherwise aiming for.

**Assessment**

Against the intended plan, the implementation is otherwise solid. The staged bring-up in [e2e.sh](/Users/ericalt/Documents/eden/reference/compose/healthcheck/e2e.sh:82), the planner/admin route assumptions in [planner.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/planner.py:88), [admin.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/admin.py:307), [planner_list.html](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/templates/planner_list.html:32), [planner_claim.html](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/templates/planner_claim.html:27), [admin_task_detail.html](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/templates/admin_task_detail.html:37), and [planner_submitted.html](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/templates/planner_submitted.html:4) all line up. I did not find scope creep or missing plan-promised implementation.

On the specific questions:
- Race correctness: the planner-host startup race is addressed correctly. The 30s `/planner/` poll is sensible. The “orchestrator must still be running” check is in the right place, though not mathematically airtight: if the driver stalls badly, the orchestrator could still quiesce after that check and before submit. I’d treat that as residual risk, not a blocker.
- Termination drill: the blob-init expectation is correct because [compose.yaml](/Users/ericalt/Documents/eden/reference/compose/compose.yaml:72) defines a persistent one-shot container and stage 1 starts it at [e2e.sh](/Users/ericalt/Documents/eden/reference/compose/healthcheck/e2e.sh:46). A restarting container would fail the drill, which is appropriate. One bash caveat: the process-substitution producer at [e2e.sh](/Users/ericalt/Documents/eden/reference/compose/healthcheck/e2e.sh:200) is not protected by `pipefail`; if `docker compose ps -a` failed after partial output, the loop would not notice that directly.
- Driver robustness: CSRF scraping matches the current templates, session-cookie handling via `httpx.Client` is fine, and the admin reclaim `Location` equality matches the current redirect in [admin.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/admin.py:331).
- Docs: [docs/roadmap.md](/Users/ericalt/Documents/eden/docs/roadmap.md:201) is accurate. The only doc problem I found is the stale AGENTS “later phases” row. One minor note: the plan text itself has stale details in a couple places, but the shipped code/docs chose the correct side of those inconsistencies.

Overall: not ready to ship as-is because of the CI job’s Python install step. Fix that, and I would consider this ready.
