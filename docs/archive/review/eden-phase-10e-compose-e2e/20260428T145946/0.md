# EDEN Phase 10e — End-to-end Compose integration test

## Context

Phase 10b/10c shipped `compose-smoke` (scripted-mode 3-trial loop,
headless-host only). Phase 10d shipped `compose-smoke-subprocess`
(subprocess-mode same shape). Both verify that the orchestrator + the
three headless worker hosts can drive an experiment to quiescence; both
ignore the web UI surface, the admin-reclaim path, and the stack's
termination behavior.

Phase 10e closes that gap. The exit criterion in [`docs/roadmap.md`](../roadmap.md)
is "End-to-end Compose integration test exercising the full loop
including Web UI, admin-reclaim, and termination." This chunk delivers
exactly that, in one new `compose-e2e` CI job.

## Reframe

The value of `compose-e2e` is the **stack shape**, not duplicate role
coverage. The chunk-9b / 9c / 9d real-subprocess pytest tests in
`reference/services/web-ui/tests/test_*_e2e.py` already exercise each
role's UI flow against a forked task-store-server + web-ui pair on an
ephemeral port. What those tests can NOT prove is that the same flows
work when the web-ui is talking to a `task-store-server` running in
*another container*, sharing volumes with the *real* orchestrator and
*real* worker hosts, behind compose's healthcheck and restart-policy
machinery, on the published `${WEB_UI_HOST_PORT}` host port.

So this chunk picks **one** representative role flow (planner — the
simplest, no repo writes, no metrics-schema validation) and drives it
end-to-end against the live stack. Implementer and evaluator UI flows
are out of scope: their pytest e2e tests give equivalent role-contract
signal, and adding two more compose-bound walkthroughs multiplies
fragility (extra container starts/stops, extra orchestration races)
without adding stack-shape signal beyond the planner walkthrough.

## Roadmap delta

- Container isolation of the `*_command` subprocess (docker-in-docker,
  experiment-specific image build, auth-secret mounting): still
  deferred to a 10d follow-up sub-chunk per the chunk-10d plan.
- Workers integrate with Gitea as their actual git remote: still
  deferred to a 10d follow-up sub-chunk.
- Branch protection is **not** updated to require `compose-e2e` in
  this chunk — same posture 10c / 10d took for newly-added compose
  jobs (let it run cleanly for a few iterations on `main` first).

## Scope

### In scope

1. **Web UI planner walkthrough.** Sign in to the web UI on the
   published host port, GET `/planner/`, claim one plan task, submit a
   one-proposal draft, verify the task moved to `submitted` and the
   resulting trial integrates downstream.
2. **Admin-reclaim drill.** Claim a *separate* plan task via the web
   UI without submitting (so it sits in `claimed` state with a live
   token), POST `/admin/tasks/<id>/reclaim`, verify the redirect to
   `?reclaimed=ok`, and verify the event log shows a `task.reclaimed`
   event with `cause=operator`.
3. **Termination drill.** `docker compose stop --timeout 10` against
   the still-up stack, then verify every container exited cleanly:
   status=`exited`, exit code != 137 (the SIGKILL marker — anything
   else is acceptable, including 0 from a graceful handler and 143
   from a default-handler SIGTERM).
4. **End-state assertions.** Same final-state shape as compose-smoke,
   adjusted for `EDEN_PLAN_TASKS=4`: `>= 4 trial.integrated`,
   `>= 12 task.completed`, `>= 4` plan-task `task.completed`.
5. **CI wiring.** New `compose-e2e` job mirroring the existing
   `compose-smoke` and `compose-smoke-subprocess` jobs.

### Out of scope

- Subprocess-mode coverage. `compose-smoke-subprocess` already covers
  that path; layering UI walkthroughs on top is duplicative.
- Implementer / evaluator UI walkthroughs. Their chunk-9c / 9d
  pytest e2e tests give equivalent role-contract signal.
- Container isolation of the subprocess (10d follow-up).
- Gitea as the workers' actual git remote (10d follow-up).
- Branch-protection requirement for the new job (deferred follow-up).
- Multi-experiment / control-plane (Phase 12).
- Kubernetes deployment (Phase 13).

## Design

### A. Test layout

Two new files under `reference/compose/healthcheck/`:

- **`e2e.sh`** — bash wrapper. Owns: tooling preflight, setup-experiment
  invocation, `compose up -d --wait`, dispatch to the python driver,
  post-run event-log assertions via `jq`, termination drill, teardown.
- **`e2e_drive.py`** — python driver. Owns: web UI sign-in, CSRF
  extraction, planner-walkthrough claim/submit, admin-reclaim POST,
  authenticated event-log read-back, exit non-zero on assertion
  failure with a clear message.

The split is deliberate: bash is the natural shape for `compose` /
`docker inspect` / `jq` plumbing; python is the natural shape for
cookie-aware HTTP, form-CSRF round-tripping, and structured JSON
assertions. Mirroring the existing smoke shells keeps the CI job
shape uniform.

### B. Bring-up sequence

`e2e.sh` runs the same shape as `smoke.sh` for the bring-up phase, with
two changes:

1. After setup-experiment writes `.env`, override `EDEN_PLAN_TASKS=4`
   (default is 3). The extra plan task is what the web UI walkthrough
   consumes; the other 3 still go through the headless planner-host
   for end-state arithmetic that matches the existing smoke shape.

   Implementation: setup-experiment.sh writes `EDEN_PLAN_TASKS=3` into
   the .env file by default. After it returns, `e2e.sh` runs:

   ```bash
   sed -i.bak 's/^EDEN_PLAN_TASKS=.*/EDEN_PLAN_TASKS=4/' "$ENV_FILE"
   rm -f "${ENV_FILE}.bak"
   ```

   The `sed -i.bak` form is GNU/BSD-portable (BSD `sed -i ''` rejects
   the bare `-i`; GNU `sed -i.bak` accepts it identically).

2. Bring up the stack with `compose up -d --wait --wait-timeout 240`
   exactly as smoke does. The orchestrator will start dispatching as
   soon as task-store-server is healthy.

### C. Web UI planner walkthrough

The race against the headless planner-host is the central correctness
question for this drill. The planner-host polls every ~1.0s and claims
any pending plan task. If the test loses the race, its claim either
returns no eligible task or hits `WrongToken` on submit. Two viable
mitigations:

- (a) **Stop the planner-host** before the UI walkthrough, restart it
  after. Deterministic; cost is one `compose stop / start` round-trip.
- (b) **Use a separately-seeded plan task** with a prefix the
  planner-host doesn't claim. Planner-host doesn't filter by prefix;
  this would require code changes in the host to honor a prefix.

**Pick (a).** It costs one `compose stop planner-host` + one `compose
start planner-host` and works against the existing host code unchanged.
The drill is:

```bash
# Drill: stop planner-host so web UI wins the claim race
docker compose stop planner-host

# Run the python driver, which performs:
#  - GET  /signin (no body needed; the auth stub is a no-questions form)
#  - POST /signin -> 303 -> session cookie set
#  - GET  /planner/ -> 200, scrape the CSRF token from the rendered form
#  - POST /planner/<plan_task_id>/claim -> 303 to /planner/<id>/draft
#  - GET  /planner/<plan_task_id>/draft -> 200, scrape CSRF (claim form
#        and draft form are independent renders so re-scraping is the
#        chunk-9b pattern)
#  - POST /planner/<plan_task_id>/submit with one proposal row
#  - assert 303 to /planner/?submitted=ok

# Drill: claim a *second* plan task and abandon it for admin-reclaim
#  - POST /planner/<other_plan_task_id>/claim -> 303
#  - GET  /admin/tasks/?state=claimed -> 200, assert <other> visible
#  - POST /admin/tasks/<other>/reclaim -> 303 to ?reclaimed=ok

# Verify task.reclaimed cause=operator via authenticated event read
#  (the python driver uses StoreClient against the host-published port
#   on task-store-server is NOT host-published — task-store-server is
#   internal-only. Use `compose exec task-store-server curl …` in
#   e2e.sh for the post-run event check, NOT the python driver.)

# Restart planner-host so the reclaimed plan task gets re-claimed and
# the experiment proceeds to quiescence
docker compose start planner-host
```

**Important detail:** `task-store-server` does NOT publish a host port
(see compose.yaml line ~145; only `web-ui` publishes one). So the
python driver can talk to web-ui directly via `http://localhost:${WEB_UI_HOST_PORT}`,
but the post-walkthrough event-log assertion ("task.reclaimed event
with cause=operator was logged") is done from `e2e.sh` via
`compose exec task-store-server curl …` exactly the way `smoke.sh`
already reads the event log.

### D. Discovering plan-task IDs

The orchestrator seeds plan tasks with deterministic IDs derived from
the experiment ID and a sequence: typically `plan-1`, `plan-2`, etc.
(See `reference/services/orchestrator/` for the exact format.) The
test does NOT hard-code IDs; it queries `GET /admin/tasks/?kind=plan`
once after bring-up and picks two pending IDs:

- One for the success-path walkthrough (claim → submit)
- One for the admin-reclaim drill (claim → admin-reclaim, no submit)

This is robust against future changes to the orchestrator's task-id
naming convention.

### E. Submit-form payload

The chunk-9b planner draft form is one CSRF token plus a list of
proposal rows. Each row needs:

- `slug` — non-empty, ASCII-safe (the chunk-9b regex permits
  `[a-zA-Z0-9_-]+`)
- `priority` — float; the form parses `float()` directly
- `parent_commits` — comma-separated list; for this drill, single
  entry `EDEN_BASE_COMMIT_SHA` (read from the .env file passed via env
  to the python driver)
- `rationale` — markdown text; one line is fine

The driver submits ONE proposal row, mirroring the
`tests/fixtures/experiment/plan.py` long-running planner's default
`EDEN_PROPOSALS_PER_PLAN=1` shape so downstream arithmetic matches.

### F. Quiescence wait

After the python driver returns success, `e2e.sh` waits up to 240s for
the orchestrator container to enter `exited` status with code 0,
exactly the way `smoke.sh` does. The wait timeout is bumped from
smoke's 180s because:

- 4 plan tasks (vs 3 in smoke) → ~33% more work
- The brief planner-host stop+start window adds startup-latency budget

### G. End-state assertions

Same shape as smoke.sh, adjusted for `EDEN_PLAN_TASKS=4`:

```bash
# >= 4 trial.integrated  (every plan task -> 1 trial via fixture's
#                         EDEN_PROPOSALS_PER_PLAN=1)
# >= 12 task.completed   (4 plan + 4 implement + 4 evaluate)
# >= 4 task.completed where task_id starts with "plan-"

# Additional 10e-specific assertions:
# >= 1 task.reclaimed with cause=operator
#                        (the admin-reclaim drill produced exactly one)
```

The drill's success-path proposal becomes one of the 4 trials; the
admin-reclaimed plan task is re-dispatched after planner-host
restarts and produces its own trial. So the arithmetic still works.

### H. Termination drill

After the end-state assertions pass, `e2e.sh` runs:

```bash
docker compose -f compose.yaml --env-file "$ENV_FILE" stop --timeout 10
```

Then asserts, for every `eden-*` container in `compose ps -a`:

- `State.Status == "exited"`
- `State.ExitCode != 137` (137 = 128 + 9 = SIGKILLed)

The non-137 form (rather than == 0) is deliberate: services that don't
install a SIGTERM handler exit with 143 (128 + 15 = signaled by SIGTERM
with the default handler), which is *also* a clean termination — the
container received SIGTERM and exited promptly, just without
graceful-shutdown bookkeeping. Treating 143 as a failure would make
the drill fragile against benign upstream changes (e.g. a future
busybox version or postgres-image internal change). What we actually
care about is "the container did NOT need to be killed."

A subtle case: `eden-blob-init` and (potentially) `eden-repo-init`
have already exited before `compose stop` runs. Their exit codes
should be 0 (they're one-shots that completed successfully during
bring-up). The drill checks them all; success there is `ExitCode == 0`,
not the != 137 rule (which would let a misbehaving one-shot pass).
To keep the assertion uniform, the drill checks `ExitCode != 137`
universally and additionally checks `ExitCode == 0` for the
one-shots by container name.

### I. Teardown

`e2e.sh`'s `cleanup()` trap runs `docker compose down -v` on EXIT,
exactly as smoke.sh does. The termination drill's `compose stop` does
NOT call `down -v` itself — that would lose the volumes before the
teardown drill verifies anything. `down -v` is the trap's
responsibility; `stop` is the drill's.

### J. CI wiring

Add `compose-e2e` to `.github/workflows/ci.yml` mirroring
`compose-smoke-subprocess`:

```yaml
compose-e2e:
  name: compose-e2e
  runs-on: ubuntu-latest
  timeout-minutes: 20  # 5 more than smoke jobs to absorb the drill
  steps:
    - uses: actions/checkout@v4
    - name: Verify docker compose + jq + python3 are available
      run: |
        docker compose version
        jq --version
        python3 --version
    - name: Install python driver deps
      run: pip install 'httpx>=0.27,<0.30'
    - name: Run compose e2e
      working-directory: reference/compose
      run: bash healthcheck/e2e.sh
```

The `httpx` install is the only new dependency — version-pinned to a
narrow range matching the rest of the workspace. `jq` and python are
already present on `ubuntu-latest`.

### K. Failure-mode handling

| Failure | Surface | Recovery |
|---|---|---|
| planner-host doesn't stop in time | python driver claim returns no eligible task → driver exits non-zero with diagnostic | shell trap dumps `compose logs planner-host`, then teardown |
| Web UI sign-in 5xx | driver exits non-zero | shell trap dumps `compose logs web-ui` |
| Submit fails reachability check | `/planner/<id>/submit` returns 200 with form re-rendered showing error | driver scrapes the form for an error banner and exits non-zero with the banner text |
| Admin-reclaim 403 (CSRF) | redirect to `?error=csrf` | driver exits non-zero |
| Admin-reclaim 404 | task disappeared (reclaimed by sweeper before drill) | driver exits non-zero with diagnostic; raise the claim TTL via `EDEN_WEB_UI_CLAIM_TTL_SECONDS` (NOT a current flag — would need to be added) |
| Orchestrator doesn't quiesce in 240s | bash assertion fails | shell trap dumps `compose logs --tail 60 orchestrator planner-host implementer-host evaluator-host` |
| Container SIGKILLed during stop | bash assertion fails on != 137 | shell trap dumps `compose logs --tail 30` for the offending container |

Re the claim-TTL row above: the chunk-9b web-ui defaults to a 60s TTL.
For the admin-reclaim drill, the claim is held for ~1s before the
admin-reclaim POST runs, so TTL expiry is not a concern. We do NOT
add a new TTL flag; if the drill grows in scope later such that the
claim might age past 60s before reclaim, revisit then.

### L. Deterministic plan-task seeding

The orchestrator's `--plan-tasks N` flag seeds tasks at startup. With
`EDEN_PLAN_TASKS=4` they're all created promptly, before the python
driver runs. The driver's `GET /admin/tasks/?kind=plan` will see all 4
in `pending` state during the brief planner-host-stopped window.

There's a remaining race: between `compose stop planner-host`
returning and the python driver reading the admin task list, some
implementer-host or other component shouldn't be touching plan tasks.
Implementer-host filters by `kind=implement` so it can't claim plan
tasks. Web-ui doesn't auto-claim. The only relevant actor is the
sweeper — but the sweeper runs in `run_orchestrator_loop` and only
reclaims expired claims; with the planner-host gone, no claims exist
to expire. So the window is safe.

## Tests / verification

This chunk's deliverable IS a test (compose-e2e). There are no new
unit tests. The verification path is:

1. `bash reference/compose/healthcheck/e2e.sh` passes locally
2. `compose-e2e` job goes green on CI for the PR
3. The existing `compose-smoke` and `compose-smoke-subprocess` jobs
   still pass (no regressions to the base smoke shape)
4. All other CI jobs (markdown, schema, ruff, pyright, pytest,
   schema-parity, python-test-postgres) still green

## Lessons-learned hooks (preview for /codify after impl review)

- Race-against-headless-host pattern (compose stop / drill / start)
  may recur in future compose-bound tests; consider a shared helper
  if a third instance shows up.
- 137-vs-not-137 termination check is more robust than == 0; codify
  if it survives codex-review.

## Out of scope (explicit)

- Branch-protection update for `compose-e2e` (deferred follow-up).
- Three-role UI walkthrough (planner only; implementer + evaluator
  rely on chunk-9c / 9d real-subprocess pytest coverage).
- Subprocess-mode UI walkthrough (compose-smoke-subprocess covers
  the subprocess loop; UI flows are mode-agnostic).
- New CLI flags on web-ui or hosts.
- New normative spec content. The compose layout is a deployment
  detail, not a protocol concern.
