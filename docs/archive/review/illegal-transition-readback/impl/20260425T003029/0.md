## Change Summary

- Both `_retry_submit_with_readback` (implementer) and
  `_retry_submit` (planner) now treat `IllegalTransition` as a
  read-back-required failure rather than a definitive orphan.
  Definitive short-circuits remain `WrongToken` (claim was
  reclaimed under us; we couldn't have won),
  `ConflictingResubmission` (spec-defined "different won"), and
  `InvalidPrecondition` (definitive store rejection).
- The implementer's existing read-back logic was extracted into
  a `_readback(...)` helper so the IllegalTransition arm can
  call it directly without duplicating the body.
- The planner gained read-back for the first time. The existing
  return signature of `_retry_submit` (`(bool, banner)`) was
  preserved; on `IllegalTransition` or retry-exhaustion the
  function reads back the task + submission, returns
  `(True, None)` if `submissions_equivalent(prior, ours)` and
  `(False, "eden://error/conflicting-resubmission")` otherwise.
  Pending / claimed states fall through to the transport-flavored
  banner the existing code already used.
- 5 new test sub-cases (3 implementer + 2 planner). One existing
  planner test was updated to reflect the new contract.
