# EDEN Phase 11 chunk 11c — Role-contract conformance scenarios

## Context

[`docs/roadmap.md`](../roadmap.md) Phase 11 chunks the conformance suite as
`11a + 11b` (one chunk, complete), `11c` (one, this chunk), `11d` (one,
next).

[`spec/v0/09-conformance.md`](../../spec/v0/09-conformance.md) §4 names
three conformance levels:

- **v1** (shipped in 11a + 11b) — task-store + wire-binding subset
  (chapters 02 / 04 / 05 / 07 + 08 §1.1, §1.7).
- **v1+roles** (this chunk) — adds chapter 03 role-contract MUSTs.
- **v1+roles+integrator** (chunk 11d) — adds chapter 06 §3.4 integrator
  MUSTs.

11c's single deliverable is the **v1+roles addendum**: a set of black-box
scenarios under [`conformance/scenarios/`](../../conformance/scenarios/)
that assert the testable subset of chapter 03's role-contract MUSTs
through the chapter-7 HTTP binding, plus the chapter-9 prose updates that
make the new level claimable.

The reference impl is expected to pass on the first attempt. The scenarios
were authored against the spec, not against the impl, so a failure on the
reference impl is either an impl bug or a spec gap worth investigating —
not a green light to amend the test to match impl behavior.

## Reframe

Chapter 03 contains MUSTs that span four roles (planner, implementer,
evaluator, integrator) and many wire endpoints. Most of those MUSTs are
NOT directly black-box testable through the chapter-7 binding alone,
because they constrain *what a worker host does internally* (e.g. §3.2
step 1: "persist a trial with `status == 'starting'` before making any
repository write observable to other roles"). The IUT for the conformance
suite is the **task store + event log + wire binding** — *not* the worker
hosts. Worker-internal invariants are out of black-box scope by
construction.

What IS testable is the wire-binding's enforcement of the chapter-03
**submission semantics**: §2.4 (planner submission shape and
preconditions), §3.4 (implementer submission shape, idempotency), §4.2 +
§4.4 (evaluator submission shape, metrics-schema validation, eval_error
non-grafting). These are what the IUT observes and acts on — and what a
non-conforming IUT could silently get wrong, with consequences invisible
to the worker.

Chapter 03 §5 (integrator) is deliberately scoped out of this chunk and
into 11d, matching the §4 conformance-level structure.

## Spec gap surfaced by this chunk: §4.4 evaluator idempotency

While drafting the evaluator-resubmission test, codex-review (round 0)
flagged a real internal conflict in the spec:

- **Chapter 04 §4.2** (canonical idempotency rule for the task store)
  defines evaluate equivalence as `trial_id` + `status` + `metrics`
  (as JSON values). `artifacts_uri` is not listed.
- **Chapter 03 §4.4** (role-side restatement) says "identical
  `trial_id` + `status` + `metrics` + `artifacts_uri` MUST be
  accepted." `artifacts_uri` IS listed.

The reference impl follows §4.2: see `submissions_equivalent` in
[`reference/packages/eden-storage/src/eden_storage/submissions.py`](../../reference/packages/eden-storage/src/eden_storage/submissions.py)
which deliberately excludes `artifacts_uri` per its docstring.

This is the kind of "spec wins" disagreement chunk 11c is supposed
to surface. The resolution this chunk takes:

**Amend `spec/v0/03-roles.md` §4.4 to drop `artifacts_uri` from
the equivalence formula and defer to chapter 04 §4.2 as the
canonical statement.** Rationale: §4.2 is the wire-binding /
task-store side; §4.4 is the role-side restatement. The role-side
should defer to the task-store side, not contradict it. The impl
already follows §4.2.

The new §4.4 sentence will read approximately:

> Resubmission is idempotent under the same rules as §3.4 and
> [`04-task-protocol.md`](04-task-protocol.md) §4.2: identical
> normative fields (`trial_id`, `status`, `metrics`) MUST be
> accepted; inconsistent resubmission MUST be rejected.
> `artifacts_uri` is NOT part of equivalence — the first
> submission's `artifacts_uri` is the committed one.

The role-rule idempotent-resubmit test will assert this:
identical normative fields succeed; the test will NOT vary
`artifacts_uri` between calls (which would have hit the spec
ambiguity).

## Conformance posture for v1+roles

Three new chapter-9 §5 index groups, one per worker role:

| Group | Scope |
|---|---|
| Planner submission | Drafting-proposal precondition; status vocabulary; proposal-set semantics. |
| Implementer submission | Submission-shape preconditions; trial-binding; status vocabulary; idempotency. |
| Evaluator submission | Status vocabulary; metrics-schema validation; per-status trial-side writes; eval_error non-grafting. |

**Group structure decision: split by role, not combined.** A single
"Role contracts" group is conceivable, but the per-role split mirrors
chapter 03's §2 / §3 / §4 structure, makes the chapter-9 §5 coverage
check granular (a missing group is easier to spot than a missing test
*within* a group), and lets a future v1+roles+x extension add narrower
groups without renaming existing ones.

**Chapter 9 prose updates:**

- §3 currently lists `02 / 04 / 05 / 07 / 08` as the "v1 suite asserts
  MUSTs from these chapters" set. Extend to acknowledge that v1+roles
  adds chapter 03 and v1+roles+integrator adds chapter 06.
- §5 gains a v1+roles index table with the three groups above.
- §7 reference-posture wording already accommodates "v1+roles after
  chunk 11c"; no change needed.

**No reference-binding chapter changes.** Chapter 03's submission shapes
are already specified prose-side; this chunk doesn't introduce new wire
endpoints.

## The "assert end-state, not endpoint" pattern (load-bearing design choice)

Several chapter-03 MUSTs allow the IUT latitude on **where** in the wire
flow the rejection surfaces. Two examples:

1. §3.4: "commit_sha — required when status == success." A conforming
   IUT can reject this at `/submit` (shape validation) OR at `/accept`
   (acceptance validation, the orchestrator's `validate_terminal`
   path). Both are conforming.
2. §4.2: "metrics keys MUST be a subset of metrics_schema; values
   satisfy per-metric type rules." Same latitude — submit-time or
   accept-time.

If a scenario pins the failing status code at one specific endpoint,
it codifies a reference-impl quirk as a conformance contract and locks
third-party IUTs into the reference's choice. That is a silent spec
expansion via test-suite drift — exactly what chapter 9 §1 says the
suite is **not** authorized to do ("the spec is the source of truth;
the suite is one faithful implementation of 'what would test that?'").

The pattern this chunk adopts:

```text
1. Drive the flow through every endpoint that could legally reject.
2. Treat any 4xx response as "conforming-rejected" — short-circuit and
   assert the trial-side end-state still satisfies the MUST.
3. Treat any 2xx response at /submit as "must reject downstream" —
   continue to /accept and apply the same logic.
4. Assert the terminal observable state the MUST cares about (e.g.
   `trial.status != "success"`, `trial.metrics is None`).
```

Where the spec MUST is unambiguous about the endpoint (e.g. §3.4
"duplicate submit disagreeing on commit_sha MUST be rejected" — the
rejection clearly belongs at `/submit` because that's the only
endpoint a duplicate hits), the test pins the endpoint as usual.

## Scope

### In scope (this chunk)

1. **`spec/v0/09-conformance.md` §3 + §5 updates** for v1+roles + the
   index table.

2. **Three new scenario files under `conformance/scenarios/`:**

   - `test_planner_submission.py` — group `Planner submission`, citing
     §2.4. Tests:
     - submit-with-drafting-proposal → 409 illegal-transition
     - submit-with-unknown-proposal → 409 illegal-transition
     - zero-proposal status=success accepted (the "MAY produce zero
       proposals" branch)
     - status=error keeps any partially-drafted proposals in `drafting`
       and emits no `proposal.dispatched`. Wire shape: the test sends
       `{"kind": "plan", "status": "error", "proposal_ids": []}`. §2.4
       reads "the submission payload MUST include: `proposal_ids` ...
       MAY be empty" with no condition on `status` — so `proposal_ids`
       MUST be present on every plan submission, empty on error. The
       reference impl's `PlanSubmission.proposal_ids` happens to have a
       `()` default that lets a missing key parse, but a non-Python IUT
       might enforce the literal-spec MUST. This chunk fixes
       `_seed.submit_plan` to always include `proposal_ids` (it
       currently omits when `status="error"`), so the helper produces
       a wire-shape that's conforming under both readings.

   - `test_implementer_submission.py` — group `Implementer submission`,
     citing §3.4. Tests:
     - unknown trial_id → 409 illegal-transition
     - cross-proposal trial_id (trial under proposal_b, task on
       proposal_a) → 409 illegal-transition
     - status=success without commit_sha — drive submit + accept;
       observable end-state: `trial.status != "success"` (uses the
       end-state-not-endpoint pattern)
     - accepted success writes `commit_sha` onto the trial. Asserts
       `trial.commit_sha == submission.commit_sha` after `/accept` only;
       does NOT assert atomicity with `task.completed` /
       `proposal.completed` events — that composite-commit is a chapter
       04 §4.3 / chapter 05 §2.2 invariant already covered by
       `Composite commits` (chunk 11b). Citing only §3.4 keeps this
       test inside its declared `Implementer submission` group per the
       chapter-9 §5 traceability rule.
     - duplicate submit with same commit_sha is idempotent (one
       `task.submitted` event)
     - duplicate submit with different commit_sha → 409
       conflicting-resubmission
     - **status=error round-trip** (new in revision 1): submit with
       `status="error"` succeeds, the orchestrator's reject path runs,
       the trial's `status` becomes `"error"` per §3.4 ("The trial
       MUST be persisted with `status == 'error'`"), and a
       `task.failed` event names the impl task. Cited at §3.4 from
       the role-contract angle (a Composite-commits test in chunk 11b
       already exercises the chapter-05 §2.2 atomicity angle).

   - `test_evaluator_submission.py` — group `Evaluator submission`,
     citing §4.2 / §4.4. Tests:
     - mismatched trial_id (submission != task.payload) → 409
     - undeclared metric key — drive submit + accept; observable
       end-state: `trial.status != "success"`
     - type-violating metric (`retries: 1.5` against the fixture's
       `retries: integer`) — same pattern
     - accepted success writes metrics + completed_at atomically with
       `trial.succeeded`
     - eval_error keeps trial in `starting` AND discards
       submission-carried metrics + artifacts_uri
     - retry-exhausted `declare-eval-error` after a metrics-bearing
       eval_error submit does NOT graft prior metrics onto the trial
     - role-rule idempotent resubmit (one `task.submitted` event)
     - **artifacts_uri NOT part of equivalence** (new in revision 1):
       per the §4.4 amendment in this chunk, a resubmit with identical
       `trial_id` + `status` + `metrics` but a *different*
       `artifacts_uri` MUST be accepted (200) AND emit only one
       `task.submitted` event. The first submission's `artifacts_uri`
       is the committed one. Pins the §4.4 amendment so a future
       impl regression that incorrectly tightens equivalence to
       include `artifacts_uri` is caught.
     - **status=error trial-side write** (new in revision 1): submit
       with `status="error"` and a metrics object, run the
       orchestrator reject path, then read the trial. §4.4 says
       "metrics — set to the submission's `metrics` when status ∈
       {'success', 'error'}", so the trial's `metrics` MUST equal
       the submission's metrics and `trial.status == 'error'`.
       Distinguishes `error` from `eval_error` on the trial-side
       write rule, which the existing `eval_error keeps starting`
       test does not exercise.

3. **Spec amendment:** `spec/v0/03-roles.md` §4.4 evaluator-idempotency
   formula update per §"Spec gap surfaced by this chunk" above.

4. **Helper extensions:** `conformance/src/conformance/harness/_seed.py`

     None = None` parameter so the new artifacts_uri-not-equivalence

   - `submit_plan` is changed to always include `proposal_ids` in the
     wire payload (it currently omits when `status="error"`), per the
     §2.4 reading that `proposal_ids` is unconditionally part of the
     submission shape. This is conforming under both literal-spec and

### Out of scope (this chunk)

1. **Worker-internal invariants** — §1.2 (workers MUST NOT write
   non-owned objects), §3.2 step 1 (starting-trial-before-repo-write),
   §3.3 (work/* branch uniqueness), §4.3 (evaluator non-interference).
   These constrain the worker's behavior, not the IUT's, and a
   black-box test through the chapter-7 binding cannot prove or
   disprove them.

2. **Integrator §5 / chapter 06 MUSTs** — deferred to 11d per the
   chapter-9 §4 level taxonomy.

3. **Adding helpers for malformed-by-design wire shapes.** Where a test
   needs a payload that violates the role contract — e.g. implementer
   `status=success` without `commit_sha` — the test issues a direct
   `wire_client.post(...)` call inline rather than getting a helper.
   Adding `submit_implement_no_sha` would normalize the malformed shape
   into the same surface as well-formed submissions and obscure that
   it's deliberately broken. (Note: the `_seed.submit_evaluate`
   extension in §"In scope" item 4 is different — that's normalizing
   how a *valid* wire shape with explicit metrics on `eval_error` is
   sent, not papering over a malformed shape.)

4. **New `eden_*` package code.** This chunk is conformance-suite-only;
   no reference impl changes are expected. If a test fails on the
   reference impl during execution, that's an impl bug or a spec gap
   worth investigating, NOT a signal to relax the test.

5. **Chapter 03 §3.4 "commit_sha MUST equal the tip of the worker
   branch"** — black-box impossible. The IUT receives `commit_sha`
   from the implementer's submission and writes it to the trial; the
   IUT never sees the worker branch tip. Asserting equality requires
   walking out to a git remote, which is the worker's domain not the
   IUT's. Out of scope.

6. **Citation overlap with existing groups.** `Submit idempotency`
   (chunk 11b) already covers some idempotency cases citing chapter 04
   §4.2. The new role-contract groups cite §03 §3.4 / §4.4 for the
   same observable behavior — this is deliberate. Chapter 03's role
   contracts express the worker's view of idempotency; chapter 04 §4.2
   expresses the task-store's view. Both citations are normatively
   load-bearing, and the per-test docstring spells out which of the
   two MUSTs the test asserts.

7. **Branch-protection update.** No new CI job in this chunk — the
   existing `conformance` job already runs the full suite under
   `pytest -q conformance/`. Branch protection already requires it.

## Reference impl behavior the tests rely on

These are observed behaviors of the reference task store
([`reference/packages/eden-storage/src/eden_storage/_base.py`](../../reference/packages/eden-storage/src/eden_storage/_base.py))
that the tests' assertions pin to spec MUSTs. If the reference impl
diverges from spec, the test should fail and the impl should be
fixed, not the test.

- `_validate_submission_ref_binding` raises `IllegalTransition` (→ 409
  `eden://error/illegal-transition`) at submit-time for: planner
  submitting drafting/unknown proposals; implementer submitting
  unknown trial or trial under wrong proposal; evaluator submitting
  trial_id != task.payload.trial_id.
- `_validate_implement_acceptance` returns a non-None reason (→
  `IllegalTransition` from `_accept_implement`) when commit_sha is
  None on a status=success submission.
- `_validate_evaluate_acceptance` calls `_validate_metrics`; an
  undeclared key or type-violation raises `InvalidPrecondition`,
  which `_validate_evaluate_acceptance` wraps as a string reason,
  which `_accept_evaluate` raises as `IllegalTransition`.
- `_reject_evaluate` only writes trial-side metrics on
  `submission.status == "error"` AND `reason != "validation_error"`;
  on eval_error and on success-with-validation_error, the trial stays
  unchanged.
- `declare_trial_eval_error` writes only `status="eval_error"` and
  `completed_at`; never copies metrics from a prior eval_error
  submission.

These are spec-conforming behaviors; the tests verify them.

## Fixture dependencies

The conformance harness uses
[`conformance/src/conformance/fixtures/minimal-experiment.yaml`](../../conformance/src/conformance/fixtures/minimal-experiment.yaml)
as the experiment config every adapter starts against. It declares:

```yaml
metrics_schema:
  score: real
  retries: integer
```

The evaluator-submission tests rely on this shape:

- `test_success_metric_wrong_type_must_not_complete_trial` — sends
  `retries: 1.5`, depends on `retries: integer`.
- `test_success_metrics_outside_schema_must_not_complete_trial` —
  sends `undeclared_key: 99`, depends on the schema NOT declaring
  that key.

If the fixture's metrics_schema is ever changed, these tests need to
be reviewed.

## Risks

- **Reference impl might fail one of the new assertions on first
  run.** Mitigation: read the spec MUST + the reference impl side by
  side before authoring each test, so the test is provably faithful
  to the spec; if it fails the impl, the impl is buggy. (Outcome: no
  failures observed on first run — the impl was developed against the
  spec.)

- **Fixture metrics_schema couples test to fixture.** Mitigation:
  documented in §"Fixture dependencies" so a future fixture change
  surfaces this as a conscious decision.

- **End-state-not-endpoint pattern might mask a regression where the
  IUT silently lets `commit_sha`-less success through to a successful
  trial.** Mitigation: the tests assert `trial.status != "success"`
  AND `trial.metrics is None` (where applicable) — both fields would
  have to lie for the assertion to pass with a buggy IUT. The 4xx
  short-circuit branch is also exercised by the impl (which rejects
  at /accept), so the "happy" path through the test does verify the
  reference's specific behavior.

- **Chapter 9 §3 prose extension might over-claim what v1+roles
  asserts.** Mitigation: the wording added is "The v1+roles suite
  extends this to include [03-roles.md]"; it does not claim chapter
  03 is *exhaustively* asserted — only that some MUSTs are. This
  matches the §6 informative IUT-adapter wording's discipline.

- **The end-state-not-endpoint pattern is one I'm introducing in this
  chunk; it doesn't have prior precedent in the suite.** Codex-review
  on the plan should weigh whether this is the right discipline going
  forward, or whether tighter assertions (specific endpoint, specific
  status code) would catch more impl bugs at the cost of locking in
  reference-impl choices. The choice has cross-chunk implications:
  chunk 11d's integrator scenarios will face the same question
  (where does §3.4 ladder rejection surface? `/integrate` vs
  `/trials/{id}` read-back?), and 11c's answer is the precedent.
