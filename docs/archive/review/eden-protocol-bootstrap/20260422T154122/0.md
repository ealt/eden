# EDEN: Protocol-First Repo Bootstrap

## Context

`/Users/ericalt/Documents/eden` is the successor to `/Users/ericalt/Documents/direvo`. The predecessor has a working monolith at `direvo/src/eden/` and a design doc at `eden/docs/plans/eden-microservices-refactor.md`.

**The reframe (critical):** EDEN is not a product-with-microservices. EDEN is a **protocol** for orchestrating directed code evolution — a specification that defines components, their responsibilities, the messages between them, and the invariants they must honor. This repo provides:

1. **The normative protocol specification** — versioned, formal, with machine-readable schemas for every wire format and state machine. This is the authoritative artifact; semantics live here, not in code.
2. **A reference implementation** — one complete, working stack that conforms to the spec. Explicitly labeled as *one* valid implementation, not *the* implementation.
3. **A conformance suite** (roadmapped) — black-box tests that any independently-built component can run against itself to prove it conforms. This is what makes swappability real rather than aspirational.

Analogues: Language Server Protocol (spec + reference servers + clients), Model Context Protocol (spec repo + reference SDKs), OpenTelemetry (spec - reference SDKs per language), Kubernetes CRI/CNI/CSI (wire APIs + many interoperable implementations).

Anyone should be able to implement their own orchestrator, planner host, evaluator, storage backend, git host, or UI in any language, and have it interoperate with other conforming components.

User decisions already captured:

- GitHub: **personal account, private**, name `eden`.
- Existing microservices plan: **move to archive**; new spec + roadmap become authoritative.
- GitHub setup: repo create + push + CI (ruff/pyright/pytest + markdownlint + JSON Schema validation) + branch protection on `main`.

## Core Design Decisions

- **Spec-first, impl-second.** Every cross-component boundary is defined first in `spec/` (human-readable normative text + JSON Schema + state machine) before any code for it is written in `reference/`. When they disagree, the spec wins and the impl gets a bug.
- **Wire protocols as the contract**, not Python types. Components communicate over documented message formats (JSON over HTTP for control calls, SQL or JSON over some transport for tasks/events — to be pinned in Phase 1). Python bindings in `reference/packages/eden-contracts/` are generated from / aligned with the JSON Schemas and exist as a convenience for Python implementors, **not** as the source of truth.
- **Single repo, clear boundaries.** `spec/`, `reference/`, `conformance/`, `docs/` at top level. Can split into separate repos later if needed (MCP/LSP pattern), but day 0 discipline is structural, not geographical.
- **Versioned spec.** `spec/v0/…` from day 0. Breaking changes go to `v1/`, not mutated in place. Reference impl declares which spec version it targets.
- **uv workspace for the reference impl**, members under `reference/services/*` and `reference/packages/*`. Root `pyproject.toml` is the workspace root; the spec itself has no Python.
- **Python 3.12+, ruff + pyright**, embedded config ported from direvo.
- **`CLAUDE.md` → `AGENTS.md` symlink** (user convention).

### B. The spec itself (`spec/v0/`)

Written in RFC-style prose: **normative language** (MUST / SHOULD / MAY per RFC 2119), numbered sections, explicit cross-references, and *no* references to specific technology choices. Example boundaries:

- `04-task-protocol.md` defines the task state machine, the claim-token rule, the submit-idempotency rule, and the wire format for task objects. It does **not** say "Postgres `tasks` table" — that's a reference-impl detail. It may say "a conforming task store MUST provide atomic claim with linearizable semantics" and leave the mechanism to the implementor.
- `05-event-protocol.md` defines the event object shape, the transactional invariant (event insert MUST be atomic with the state change it describes), and the delivery guarantees subscribers can rely on. Whether that's LISTEN/NOTIFY, Kafka, or a WebSocket is out of scope.
- `06-integrator.md` pins git topology invariants: namespaces (`work/*`, `trial/*`, `main`), sole-integrator rule, squash rule, eval-manifest shape. Applies regardless of git host.

Every wire-format object has a JSON Schema under `spec/v0/schemas/`. The Markdown docs link to them. CI validates they're valid schemas and that the reference impl's Pydantic models match.

### C. `reference/` at bootstrap

- `reference/README.md` — "What's implemented, what's stubbed, what spec version is targeted." At bootstrap: `targets: eden-protocol/v0`, all services empty.
- Each `reference/services/*/` and `reference/packages/*/` — a `.gitkeep` only. Per-member `pyproject.toml` files land as each phase adds code.
- **No code yet.** Bootstrap is structure; Phase 1 is the spec content.

### D. `conformance/` at bootstrap

- `conformance/README.md` only. Describes the intended shape: scenarios drive an implementation-under-test via its advertised protocol endpoints; assertions verify invariants. Actual suite is a later roadmap phase once enough protocol surface is stable to test.

### E. `docs/roadmap.md`

Rewritten as spec-first phases. Each phase: Goal / Exit Criteria / Non-Goals / Rough Size.

- **Phase 0 — Bootstrap.** *(This task.)* Repo, scaffold, docs shell, GitHub. No spec content yet.
- **Phase 1 — Spec v0 drafts for core concepts.** Write `01-concepts`, `02-data-model`, JSON Schemas for config/task/event/proposal/trial. Exit: schemas validate `tests/fixtures/experiment/.eden/config.yaml` (migrated from direvo in this phase).
- **Phase 2 — Spec v0: role contracts + task protocol.** `03-roles`, `04-task-protocol`. Full state machine enumerated with legal/illegal transitions. Exit: state machine is testable (next phase will).
- **Phase 3 — Reference contracts package.** `reference/packages/eden-contracts/` — Pydantic bindings aligned with the JSON Schemas. CI enforces schema ↔ model parity.
- **Phase 4 — Spec v0: events + integrator + storage.** `05-event-protocol`, `06-integrator`, `08-storage`. Transactional invariant formalized; git topology pinned.
- **Phase 5 — In-memory reference dispatch loop.** One process, scripted workers for all three roles, everything in-memory, no git. Proves the protocol's state machines are implementable. Exit: state-machine conformance tests (first conformance scenarios) pass.
- **Phase 6 — Reference storage backend.** `eden-storage` with one concrete impl (SQLite for dev). Loop persists.
- **Phase 7 — Reference git integrator.** `eden-git`, `work/*` → squash → `trial/*` with eval manifest against a local bare repo.
- **Phase 8 — Cross-process reference: orchestrator + workers over a documented transport.** Extract what was in-proc to a wire protocol. First point at which a third-party worker in any language could participate.
- **Phase 9 — Reference Web UI.** Claim/submit flows for all roles.
- **Phase 10 — Reference Compose stack.** All services in Docker Compose (Gitea, Postgres, blob, services). Equivalent to the old plan's Milestone 1 but now explicitly "one valid deployment topology."
- **Phase 11 — Conformance suite v1.** Formalize scenarios that drive an implementation-under-test. Reference impl is the first subject, but the harness is deployment-agnostic.
- **Phase 12 — Spec v0 → v0.1 maintenance; multi-experiment (leases).** Old Milestone 2 scope.
- **Phase 13 — Kubernetes reference deployment.** Old Milestone 3.

Each phase explicitly lists what it does *not* do, to prevent creep.

## Known Risks / Things to Watch

- **uv workspace with empty globbed members.** Defer the workspace block until Phase 1 if `uv sync` rejects it; confirmed during execution.
- **Branch protection before any CI run has landed on main.** Required checks can't reference a check that's never run. Order: push → first CI run completes → enable protection.
- **`gh auth status` failing.** Surface clearly and stop; don't guess.
- **Protocol versioning discipline from day 0** — tempting to write `spec/` without a `v0/` subdirectory "for now." Resist: adding the version layer later means rewriting every cross-reference.
- **"Reference" naming overload.** `reference/` (top-level) is the reference *impl*; `docs/archive/` (not `docs/reference/`) holds the old plan. Avoiding the collision is why the old plan goes to `archive/`, not `reference/`.
