## Round 1 changes (in response to Round 0 review)

**All four Round 0 findings addressed:**

1. **Bug — experiment-config was too impl-specific.** Fixed. Removed `planner_root`, `workspace`, `plan_command`, `implement_command`, `evaluate_command`, and `file_permissions` from the normative schema and from `02-data-model.md` §2. Required normative fields are now only the host-independent core: `parallel_trials`, `max_trials`, `max_wall_time`, `metrics_schema`, `objective`. `convergence_window` and `target_condition` remain optional. A new §2.4 explicitly permits additional top-level fields for implementation-specific role binding / tuning and forbids a conforming orchestrator from rejecting configs solely because they carry such fields. Role binding is now deferred to Phase 2's `03-roles.md`, matching the framing. The fixture still validates under the relaxed schema (its direvo-specific fields pass through as additional properties).

2. **Bug — task `claim` mutual exclusion not enforced.** Fixed. `schemas/task.schema.json` now uses `{"if": state==claimed, "then": required claim, "else": not required claim}`. `02-data-model.md` §3.4 now explicitly states `claim` MUST be present when `state == "claimed"` and MUST NOT be present otherwise.

3. **Bug — prose said UTC but schemas accepted offsets.** Fixed. Every timestamp field in `task.schema.json`, `event.schema.json`, `proposal.schema.json`, and `trial.schema.json` now has a `pattern` that restricts the value to a UTC `Z`-suffixed form (e.g. `2026-04-23T18:04:01Z` or `…Z` with optional fractional seconds). Prose in `02-data-model.md` §1.2 is strengthened to state explicitly that offset forms MUST NOT be used.

4. **Risk — integer vs real metric typing underspecified.** Fixed. `02-data-model.md` §1.3 now defines exactly what each declared type admits: `integer` is a JSON number with no fractional component (`1.0` accepted; `1.5` rejected), `real` is any JSON number, `text` is a JSON string, `null` is allowed for any declared type. The trial schema's `metrics` remains per-experiment — §7.2 now explicitly calls out that the per-metric type check is a runtime responsibility of the orchestrator because the schema cannot express per-experiment types generically.
