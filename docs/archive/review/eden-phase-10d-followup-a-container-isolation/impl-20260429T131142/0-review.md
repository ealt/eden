**Findings**

1. Medium: `compose.subprocess.yaml` makes the Docker socket and supplementary group part of the subprocess overlay even when `EDEN_EXEC_MODE=host`, so host mode is no longer the chunk-10d “no Docker access” path the plan/spec describe. The planner, implementer, and evaluator services all unconditionally add `group_add` and `/var/run/docker.sock` mounts in [compose.subprocess.yaml](/Users/ericalt/Documents/eden/reference/compose/compose.subprocess.yaml:73), [compose.subprocess.yaml](/Users/ericalt/Documents/eden/reference/compose/compose.subprocess.yaml:132), and [compose.subprocess.yaml](/Users/ericalt/Documents/eden/reference/compose/compose.subprocess.yaml:197). That is a security and deployment drift: on some hosts it grants daemon access in host mode, and on remote/nonstandard Docker setups it can make host mode depend on a local socket path that should have been irrelevant.

2. Medium: the new docker-backed tests do not actually prove the two highest-risk behaviors they claim to lock down. The “git mount shape” test in [test_container_exec_integration.py](/Users/ericalt/Documents/eden/reference/services/_common/tests/test_container_exec_integration.py:108) only bind-mounts a normal repo; it does not model the production case that mattered here, namely a worktree whose `.git` points into a separately mounted bare repo volume. The “post-kill callback” test in [test_container_exec_integration.py](/Users/ericalt/Documents/eden/reference/services/_common/tests/test_container_exec_integration.py:207) only calls `kill_via_cidfile()` directly; it never drives the SIGKILL branch in [subprocess_runner.py](/Users/ericalt/Documents/eden/reference/services/_common/src/eden_service_common/subprocess_runner.py:162). The smoke script also overclaims this point in [smoke-subprocess-docker.sh](/Users/ericalt/Documents/eden/reference/compose/healthcheck/smoke-subprocess-docker.sh:157): the planner sibling disappearing after `compose stop` can happen via clean SIGTERM, so it does not prove `post_kill_callback` fired.

**Open Question**

The plan text said startup reaping would also best-effort clean stale cidfiles, but `reap_orphaned_containers()` in [container_exec.py](/Users/ericalt/Documents/eden/reference/services/_common/src/eden_service_common/container_exec.py:221) only talks to Docker and never inspects the cidfile dir. Because cidfiles are uuid-scoped, I don’t think this blocks ship, but it is plan/impl drift worth either implementing or deleting from the docs.

**Assessment**

Not ready to ship as-is. The core execution path looks solid: the JSON-line / `outcome.json` protocol is preserved, `Subprocess.terminate()` has the right callback ordering by inspection, volume `name:` discipline and `eden:1000` identity match the plan, and the `logging.py` `extra=` change is justified rather than scope creep because the new lifecycle code logs through the standard logging API. The concrete revisions I’d require are: gate the socket/group privilege to true docker-exec mode, and strengthen the docker-marker coverage so it exercises the real worktree+bare-repo mount shape and the actual SIGKILL escalation path.
