**Findings**

- Medium: The plan now has two different, conflicting algorithms for `EDEN_DOCKER_GID`. Section D.6 says `setup-experiment.sh` should read the host socket gid directly with `stat ... /var/run/docker.sock` and write that to `.env` ([plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:275>)), but the risks section says the correct value on Docker Desktop should come from an in-container probe against the bind-mounted socket ([plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:553>)). Those are not equivalent on macOS. The plan should pick one method and use it consistently in the design, implementation surface, and verification steps.

- Medium: The `post_kill_callback` wiring is still underspecified. The design says `Subprocess.terminate()` takes `post_kill_callback` ([plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:143>)), but the implementation surface only explicitly adds `cleanup_callbacks` to `spawn()` ([plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:335>)) and later says the planner’s `post_kill_callback` is “registered on the spawn” ([plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:356>)). That leaves it unclear whether the callback is stored on the `Subprocess` instance or must be threaded explicitly into every `terminate()` call. The plan should settle on one API model so the SIGKILL cleanup path is unambiguous.

- Low: The new test coverage is much better, but two important choices are still left as “TBD at impl time.” The Docker integration tests are “pre-step or inline … whichever is simpler” ([plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:448>)), and the orphan-recovery smoke is optional with a unit-test fallback if CI cost is high ([plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:482>)). Since orphan reaping and crash cleanup were core review risks, I’d prefer the plan commit now to the exact coverage shape rather than defer that judgment to implementation.

**Assessment**

This is substantially improved. The earlier correctness issues around named-volume forwarding, bare-repo visibility, cidfile cleanup on normal exit, and the soft secret boundary are now addressed.

I would not call it fully ready yet, but it’s close. The remaining revisions are narrow: unify the `EDEN_DOCKER_GID` detection story, make the `post_kill_callback` lifecycle explicit in the API design, and pin down the exact Docker-backed coverage you intend to ship. Once those are tightened, I’d consider the plan ready to implement.
