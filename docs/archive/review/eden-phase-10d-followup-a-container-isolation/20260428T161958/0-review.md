**Findings**

- High: The plan’s mount model does not actually make implementer/evaluator child containers usable under DooD. The proposed wrapper mounts only `<cwd>` and the experiment dir at the same absolute paths ([plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:56>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:84>)), but today those worktrees live in the `eden-worktrees` named volume and the worktree’s `.git` points back into `/var/lib/eden/repo` ([compose overlay](</Users/ericalt/Documents/eden/reference/compose/compose.subprocess.yaml:90>), [compose overlay](</Users/ericalt/Documents/eden/reference/compose/compose.subprocess.yaml:137>), [implementer flow](</Users/ericalt/Documents/eden/reference/services/implementer/src/eden_implementer_host/subprocess_mode.py:180>), [evaluator flow](</Users/ericalt/Documents/eden/reference/services/evaluator/src/eden_evaluator_host/subprocess_mode.py:240>)). Under a host daemon, `-v /var/lib/eden/worktrees/...:/var/lib/eden/worktrees/...` resolves on the host, not inside the worker container, and even if that path existed the child still needs the bare repo mounted at `/var/lib/eden/repo` for Git to work. As written, the plan breaks the current `implement_command` contract rather than isolating it. The revision needed here is explicit daemon-visible mounts for repo/worktrees/artifacts, not a raw `cwd` bind assumption.

- High: The Docker socket plan is missing a permissions story, so the worker hosts would likely fail every `docker` call. The runtime image currently runs as `USER eden` ([Dockerfile](</Users/ericalt/Documents/eden/reference/compose/Dockerfile:35>)), while the plan only adds the CLI and mounts `/var/run/docker.sock` ([plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:178>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:268>)). On Linux/CI that socket is normally `root:docker` with `0660` permissions; without `user: root`, `group_add`, or a dynamic GID-matching entrypoint, `docker ps/run/rm` from the worker will be permission-denied. This needs to be specified in the compose/runtime design, not left implicit.

- Medium: The cidfile lifecycle is incomplete and will leave stale files on normal exit. The plan only removes the cidfile in the SIGKILL escalation callback ([plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:107>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:117>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:264>)), but not after successful or non-escalated exits. That is especially risky for the long-lived planner subprocess and for reclaimed task IDs if the cidfile path is deterministic, because `docker run --cidfile ...` will trip over an existing file on reuse. The revision needed is “unlink cidfile on every terminal path, not only after forced kill,” plus a defined naming/reuse policy.

- Medium: The test plan is too end-state-only for the new failure modes it introduces. The proposed coverage checks string assembly and final event counts ([plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:317>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:326>)), but it does not exercise host-crash orphan reaping, stale-cidfile reuse, or the actual mount/env shape seen by the spawned child container. Those are the places this design is most likely to fail. I would require at least one Docker-backed integration test that kills/restarts a worker host mid-task and one targeted check that the child can run `git` in the mounted worktree.

- Medium: The secret-handling section overstates the boundary. `-e KEY` without `=value` avoids exposing secrets in the host `ps` command line, but it does not keep them out of `docker inspect`; the env still lands in container metadata, and with the mounted Docker socket the workload effectively has daemon-level visibility anyway ([plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:168>), [plan](</Users/ericalt/Documents/eden/docs/plans/eden-phase-10d-followup-a-container-isolation.md:178>)). For a reference implementation that may be acceptable, but the plan should say that plainly. The required revision is to narrow the claim and document that DooD does not isolate env-file secrets from a malicious `*_command`.
