# Implementation Review: eden-phase-10bc-dockerize-and-setup — Round 1

## Round 0 issues addressed

- **Bug — `_postgres_schema.current_version` hardcoded to `public.schema_version`.**
  Fixed in [`_postgres_schema.py`](../../../../../reference/packages/eden-storage/src/eden_storage/_postgres_schema.py): probe `to_regclass('schema_version')` (no schema qualifier) so the lookup uses the active `search_path`. Reopen on a non-default schema now finds the migrations table the first invocation created.

- **Bug — password DSN interpolation.**
  Fixed in [`setup-experiment.sh`](../../../../../reference/scripts/setup-experiment/setup-experiment.sh): the password is percent-encoded via a one-line `python3 -c 'import urllib.parse; …'` call before being interpolated into the DSN. Hex-only auto-generated passwords are encoding-safe by construction; the encoding matters when `--postgres-password <raw>` carries reserved URI characters.

- **Risk — option parser crashes on missing flag value.**
  Fixed: a new `require_value()` helper validates `[[ $# -ge 2 ]]` before each `shift 2` and prints a clear "<flag> requires a value" message instead of letting `set -u` surface "unbound variable".

- **Risk — smoke.sh asserted `trial.integrated` instead of the plan's `task.terminated`.**
  Note: `task.terminated` is not an event type the system emits; the terminal task event is `task.completed`. The smoke script now asserts both `trial.integrated >= 3` (full pipeline) **and** `task.completed >= 9` (3 plan + 3 implement + 3 evaluate) **and** `>= 3 plan-task.completed` events specifically. The plan's intent — "each plan task reached a terminal state" — is satisfied by the new plan-task `task.completed` filter.

## Round 0 plan-drift cleanups addressed

- **`setup-experiment.sh` now defaults to the fixture config** when no positional path is given (matches plan §E "defaults to repo's fixture at `tests/fixtures/experiment/.eden/config.yaml`"). Verified: `bash setup-experiment.sh` (no args) succeeds.

- **New dedicated [`tests/test_postgres_store.py`](../../../../../reference/packages/eden-storage/tests/test_postgres_store.py)** (5 tests) covers postgres-only behavior: reopen on non-default schema, reopen with different `experiment_id` rejected, reopen with changed `metrics_schema` rejected, reopen-without-arg inherits persisted schema, event-id counter resumption across reopen. All pass against a live Postgres.

- **Web UI README documents `/healthz`**: a new "Healthcheck" section under "Auth model" describes that `GET /healthz` is unauthenticated by design (compose healthchecks must work before sign-in; reveals only "process is up").

- **Plan updated to say `data text`, not `data jsonb`** with reasoning: psycopg returns `jsonb` columns as Python dicts, which would diverge from SQLite's "always a string" return and require per-backend branching in every `_get_*` method. Migrating to `jsonb` is a Phase-12+ concern.
