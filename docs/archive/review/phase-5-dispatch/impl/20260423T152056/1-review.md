**Findings**

- **Bug** — malformed submissions still do not reliably become `task.failed(reason="validation_error")`; several paths still throw and abort the loop. The driver only pre-validates worker-declared `success` submissions ([driver.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/driver.py:79)), and even there `validate_acceptance()` does not cover field-level validation like invalid `commit_sha` or invalid `artifacts_uri` ([store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:485), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:972), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:992)). The actual write then raises out of `_validated_update()` during accept/reject ([store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:782), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:852), [store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:900)). I reproduced all three cases: implement `status="success"` with `commit_sha="not-a-sha"`, evaluate `status="success"` with `artifacts_uri="not a uri"`, and evaluate `status="error"` with schema-invalid metrics; all raise instead of emitting a terminal `task.failed`. That is still a 04 §4.3 violation. The new tests also mask this by asserting raw exceptions on `accept()` / `reject()` instead of driver-level terminalization ([test_store_hardening.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/tests/test_store_hardening.py:410), [test_store_hardening.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/tests/test_store_hardening.py:566)).

- **Bug** — the mutable-state leak is closed for tasks/proposals/trials/events, but it still exists for submissions. `read_submission()` returns the stored object directly ([store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:251)), and `submit()` stores the caller-owned submission object directly ([store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:481)). `EvaluateSubmission` is a frozen dataclass, but its nested `metrics` dict is still mutable; mutating either `store.read_submission(...).metrics` or the original submission object after `submit()` changes the committed result in place. That reopens the “observable state changed without going through the store” problem for result payloads and can corrupt idempotency / validation decisions.

- **Risk** — the event-log API is clearer now, but the storage-chapter gap remains: the store only exposes an `events()` snapshot ([store.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/store.py:295)). If Phase 5 sign-off is only about the in-process dispatch/state-machine behavior, that deferral is understandable. If it is meant to claim conformance to 08 §2.1 / 05 §4 as written, `read_range` / replay / subscription semantics are still missing.

**Assessment**

Needs one more round. The main Round 0 issues around task/trial/event aliasing and task-binding were fixed, but the new validation path is incomplete in a way that still breaks spec-required terminalization, and submissions still leak mutability through the public API.
