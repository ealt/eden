## Change Summary

- `InMemoryStore` collapses the task store, event log, and proposal/trial persistence into a single in-process object (collapsing is allowed by `08-storage.md` §7). Every mutating public method stages writes in a `_Tx` and applies them via a single `_commit`; precondition failures raise before commit, so readers never observe a partial state change.
- All six composite commits enumerated in `05-event-protocol.md` §2.2 land atomically: implement dispatch, implement terminal, evaluate terminal (success/error), implement reclaim with in-flight trial, retry-exhausted `trial.eval_errored`, and trial promotion.
- The implementer's `commit_sha` lands on the trial atomically with the implement-accept composite commit (task.completed + proposal.completed), not at submit time. Rationale: the trial's `commit_sha` is not itself a state-machine-owned field (trial status stays `starting` until the evaluator writes it), and writing it inside the composite keeps every protocol-owned mutation behind the transactional invariant.
- Workers go through the store's public API only; they never touch the store's internal dicts. The store enforces role negative rules from `03-roles.md` §1.2 structurally because there is no back door.
- The dispatch driver (`run_experiment`) is deliberately thin: it shuttles work between roles by polling `list_tasks` / `list_proposals` / `list_trials` and finalizes submitted tasks by reading the recorded submission. It does not implement retry policy, priority ordering, or any scheduling beyond "do every piece of work currently possible, to quiescence."

## Notable design decisions to audit

1. **`commit_sha` write timing.** See above. Alternative: have `submit` for implement also write `commit_sha` on the trial atomically. That would match `03-roles.md` §3.2 ("set the trial's commit_sha to the tip of the work/* branch" as an implementer step) more literally. Current design interprets §3.2 as "the implementer provides commit_sha as an output; the orchestrator persists it atomically with terminal acceptance."
2. **Content equivalence for resubmit.** `_submissions_equivalent` compares status + role-specific canonical fields per `04-task-protocol.md` §4.2: plan compares `proposal_ids` as sets; implement compares `trial_id` + `commit_sha`; evaluate compares `trial_id` + `metrics` (dict equality) + `artifacts_uri`.
3. **Submission storage separate from the task object.** Submissions live in `InMemoryStore._submissions` keyed by task_id, not on the task's extra fields. This keeps the `Task` Pydantic model aligned with the wire schema (no result field) and makes the submission cleared on reclaim via a single set operation.
4. **Automatic reclaim of submitted tasks.** Per §5.1, only operator reclaim may move a submitted task. `reclaim(task_id, cause)` enforces this by rejecting `expired` / `health_policy` against `submitted`.
5. **Trial `commit_sha` precondition on evaluate dispatch.** `create_evaluate_task` requires `trial.commit_sha` is set. This follows from `03-roles.md` §4.1 (the orchestrator ensures `T.commit_sha` is set before dispatch).

## What's explicitly NOT in this chunk (by roadmap non-goals)

- No git integration — commit SHAs are fabricated by scripted workers; no `work/*` branches are actually written. Lands in Phase 7.
- No SQLite or any persistence. The store is in-memory only. Lands in Phase 6.
- No cross-process protocol. Lands in Phase 8.
- No retry policy, priority scheduling, or operator-facing observability beyond the event snapshot. Lands later.

## Review Status

### Round 0 — addressed

All three Round-0 bugs and the tests-overstate-conformance risk were addressed. The two risks were accepted with narrower scope.

**Bug: mutable leak.** Every read (`read_task`, `read_proposal`, `read_trial`, `list_tasks`, `list_proposals`, `list_trials`, `events`) and every creation (`create_proposal`, `create_trial`, and the `create_*_task` return value) now goes through `_deep`, which deep-copies the Pydantic model. Stored objects and returned objects no longer share identity. New `TestReadIsolation` scenarios assert that mutating a read or the instance passed to `create_proposal` does not corrupt store state.

**Bug: submission not bound to task payload.** Added `_validate_submission_ref_binding` at `submit` time and per-kind `_validate_*_acceptance` helpers. The store now rejects:

- a plan submission referencing unknown proposals or proposals still in `drafting` (03-roles §2.4),
- an implement submission whose `trial_id` belongs to a different proposal, lacks a `commit_sha` when `status=success`, or points to a trial that is not `starting`,
- an evaluate submission whose `trial_id` does not match the task's payload, or whose trial is not `starting`/has no `commit_sha`, or whose `metrics` is absent on `success`.

The driver now calls `store.validate_acceptance` before deciding between `accept` and `reject`; malformed success submissions become `task.failed(reason=validation_error)` per 04-task-protocol §4.3. Negative tests added in `TestSubmissionBinding` and `TestValidationErrorRouting`.

**Bug: trial writes bypassed Pydantic validation; no metrics-schema enforcement.** Replaced every `model.model_copy(update=...)` with `_validated_update`, which dumps + re-validates. Invalid `commit_sha` / `artifacts_uri` now fail via Pydantic's pattern and RFC-3986 validators. Added a `metrics_schema: MetricsSchema | None` constructor parameter and `_validate_metrics`, which enforces 08-storage §4.3: every key must be in the schema; every value must match the declared type (integer / real / text); `bool` is rejected for numeric types (spec §1.3 treats the three storage types as distinct). Negative tests in `TestMetricsSchemaEnforcement` and `TestFieldValidationOnUpdate`.

**Open question: evaluate resubmit includes `artifacts_uri`.** Codex flagged this as defensible (03-roles §4.4 mentions it; 04 §4.2 omits it). I went with the 04 §4.2 reading: equivalence is `status + trial_id + metrics`; `artifacts_uri` differences are ignored and the first submission is canonical. Scenarios in `TestEvaluateResubmitEquivalence` cover both sides of this.

**Risk: `subscribe()` was a misleading name for a snapshot.** I removed `subscribe` entirely; `events()` is the only read interface and its docstring is explicit that it returns an ordered snapshot. A conforming subscription / replay API is a Phase-6 concern (where the storage backend becomes real) and out of Phase-5 scope.

**Risk: role negative rules not structurally enforced.** Accepted with a docstring note in `store.py` making the scope boundary explicit: any caller can call `create_trial`, `integrate_trial`, etc. The scripted workers comply by construction; detecting non-compliance in an arbitrary deployment is Phase 11's conformance suite. An in-process reference could only enforce via role-scoped handles, which would be substantial surface for no extra protocol guarantee.

**Risk: tests overstated conformance.** 21 new scenarios cover the gaps (mutation isolation, submission binding, malformed-success routing, metrics-schema enforcement, field validation on update, evaluate resubmit equivalence per 04 §4.2). Total: 57 dispatch scenarios + 80 contracts scenarios = 137 pytest; still all green.

### Round 1 — addressed

**Bug: submission mutability leak.** `submit()` now deep-copies the submission before storing; `read_submission()` deep-copies on return. `EvaluateSubmission.metrics` is a dict (not frozen) so the ``frozen`` dataclass alone wasn't enough — the deep copy is the fix. New test `TestReadIsolation.test_submission_metrics_mutation_does_not_leak` verifies both the caller-side mutation and the reader-side mutation cannot change the committed submission.

**Bug: malformed submissions still crashed instead of routing to `validation_error`.** The store now exposes `validate_terminal(task_id)` returning ``(decision, reason)`` where ``decision`` is one of ``accept`` / ``reject_worker`` / ``reject_validation``. The driver consults this before every terminal decision. Additions:

- `_validate_implement_acceptance` now dry-runs `_validated_update(trial, commit_sha=…)`, so an invalid SHA surfaces as ``reject_validation`` instead of raising.
- `_validate_evaluate_acceptance` now dry-runs the full trial write (status=success, metrics, artifacts_uri, completed_at), catching invalid URIs before the real update.
- New `_validate_evaluate_error` covers the error path: a `status="error"` evaluate submission with schema-invalid metrics is routed to ``reject_validation`` rather than blowing up inside `_reject_evaluate`.
- `_reject_evaluate` now skips trial writes whenever `reason=="validation_error"` (treated like `eval_error`: trial stays in ``starting``, task is terminal with no invalid field landing on the trial).

Tests updated: `TestFieldValidationOnUpdate` now asserts end-to-end terminalization — invalid `commit_sha` / `artifacts_uri` produce `task.failed(validation_error)` and the invalid values never land on the trial. `TestMetricsSchemaValidationIndependentOfSuccess` now asserts that an error submission with bad metrics also becomes `validation_error` with no trial mutation.

**Risk: `events()` still snapshot-only (no `read_range` / replay / subscription).** Accepted; Phase 5's scope is the state machine and transactional invariant, and the snapshot surface is sufficient for the in-process scripted workers and tests. A conforming subscription / replay API is a Phase 6 concern, where the storage backend becomes real. Documented explicitly in the module docstring.

### Round 2 — addressed

**Bug: malformed evaluate `status="error"` left the trial in starting.** Correct. Per 03-roles.md §4.4, `status="error"` always drives the trial to `error` — the worker declared trial failure, independent of whether the payload fields are valid. Refactored `_reject_evaluate`:

- Trial-side transition is now a function of `submission.status`, not of the orchestrator's reject `reason`.
- `status="error"` → trial transitions to `error`, `completed_at` is written, `trial.errored` emitted. If `reason=="validation_error"`, the invalid `metrics`/`artifacts_uri` are dropped; the trial-side transition itself still happens.
- `status="eval_error"` → trial stays in `starting` (§4.4 evaluator- side failure).
- `status="success"` (only reachable via validation_error on a malformed success) → trial stays in `starting` (orchestrator never observed a valid success verdict).

The test that previously encoded the wrong behavior (`test_error_submission_with_bad_metrics_routed_to_validation_error`) now asserts the correct post-condition: `trial.status=="error"`, `metrics=None` (dropped), `completed_at` set, `trial.errored` in the log.

**Nit: stale driver docstring.** Updated to reference `validate_terminal` rather than `validate_acceptance` and to call out the validation-error path for `status="error"` submissions too.
