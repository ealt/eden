# EDEN Phase 9 chunk 1 — Reference Web UI shell + planner module

## Goal

Stand up the reference Web UI service (`reference/services/web-ui/`) with
enough surface to play one role — planner — through a browser end-to-end
against the existing Phase 8b service stack. This phase delivers roadmap
units **9a (UI shell)** and **9b (planner module)** as a single chunk;
implementer (9c), evaluator (9d), and observability (9e) are deferred
to subsequent chunks.

The exit bar is: a human, given the URL of a running task-store-server
and the shared bearer token, can sign in, see the configured experiment,
claim a pending `plan` task, draft proposals through a web form, submit,
and see the resulting state reflected back through the same UI on the
next page load. Submission must round-trip through `eden-wire`'s
`StoreClient` against the real task-store-server (no in-process
shortcut).

## Non-goals

- **9c–9e** — implementer module, evaluator module, observability views,
  and admin-reclaim are out of scope for this chunk. The shell is
  designed to accommodate them, but no placeholder stubs are added; an
  empty navigation entry that 404s is worse than no entry.
- **Real auth.** The wire protocol uses a shared bearer token (chapter
  07 §12); this chunk inherits that. No per-user accounts, sessions
  beyond a signed cookie, or RBAC. Real auth is Milestone 3.
- **Multi-experiment switcher.** Phase 12 introduces the control plane;
  for now the UI service is configured at start-up with a single
  `--experiment-id` and `--task-store-url`, mirroring the orchestrator
  service.
- **Compose / Dockerization.** Phase 10. The UI service runs locally via
  `python3 -m eden_web_ui …` for now.
- **In-UI code editing**, artifact previews, or trial diff views.
- **WebSocket / SSE push.** The UI is request-response; the next page
  load reflects state. Phase 9e revisits live updates.
- **Static asset bundling / minification.** A single CSS file is served
  as-is; no build step.
- **Blob storage for proposal artifacts.** Phase 9 chunk 1 writes
  rationale markdown to `--artifacts-dir` and references it via
  `file://` URIs. This matches the placeholder scheme the scripted
  planner uses today; Phase 10 introduces real blob storage.

## Backwards-compatibility policy

Pre-1.0, version 0.0.1, no external consumers. Wire-protocol surface
does not change in this chunk. If a contract change is needed, it
lands in `eden-wire` / spec chapter 07 in lockstep, not as a side
effect of UI work.

## Tech stack — decision and alternatives considered

**Chosen stack:** FastAPI + Jinja2 + HTMX, plus a single hand-written
CSS file. The UI service is a separate process under
`reference/services/web-ui/` that acts as a **backend-for-frontend
(BFF)**: it holds the `--shared-token` (the same reference-bearer the
orchestrator and worker hosts already use), runs `eden_wire.StoreClient`
in-process to talk to the task-store-server, and exposes only
HTML/HTMX-friendly routes to the browser. The browser never sees the
shared token; it gets a session cookie signed with a server-side
secret.

This BFF split is a property of the architecture, not the rendering
choice — the same split could host an SPA, a Phoenix-LiveView-style
push UI, or plain server-rendered HTML. The rendering choice (HTMX
over SPA) is argued separately below.

### Why this stack

1. **No new toolchain.** FastAPI, Jinja2, and httpx are already direct
   or transitive dependencies of `eden-wire` and the existing services.
   HTMX is a single CDN-hosted JS file (or vendored under `static/`).
   No Node, no Vite, no npm lockfile in a Python-first repo.
2. **One language end-to-end.** Server-side rendering keeps planner /
   implementer / evaluator forms in Python, where the typed
   `PlanSubmission` / `ImplementSubmission` / `EvaluateSubmission`
   payloads they construct already live (`eden_storage.submissions`).
   No duplicate validation logic in a JS layer.
3. **`StoreClient` already satisfies the `Store` Protocol.** The UI
   service consumes the same surface the orchestrator and worker
   hosts use. We don't add a second client implementation; we don't
   expose the wire protocol directly to the browser.
4. **Reference-impl ethos.** The repo's job is to demonstrate one
   conforming implementation, readable to someone evaluating whether
   to build a different one. An SPA buries the role-contract logic
   behind JSON-over-fetch plumbing; server-rendered forms keep the
   spec-to-code mapping legible.
5. **Auth surface stays small.** The browser holds a session cookie
   only; it never gets the shared bearer. The threat model is
   deliberately spelled out in §B (a leaked cookie can act as the
   named worker_id through the UI but cannot bypass the UI service
   to talk to the task-store directly); the goal here is to keep
   the wire-protocol auth surface unchanged, not to claim the
   cookie is unforgeable.

### Alternatives considered

#### A. BFF-backed SPA (React / Vue / Svelte talking to the UI service's JSON API)

- **Pros:** Richer interactions; separates view logic from server
  state cleanly; idiomatic for production webapps; the BFF split
  preserves the shared-bearer-stays-server-side property.
- **Cons:**
  - Adds a Node toolchain, a build step, and a JS lockfile to a repo
    whose every other artifact is Python or Markdown. Doubles the CI
    matrix (frontend lint / build / test alongside `uv run …`).
  - Duplicates validation: form fields validated client-side, then
    re-validated server-side, then re-validated by the wire schemas.
    Three sources of truth for "is this submission well-formed."
  - The role-contract logic — "for plan submission, create N proposals,
    mark them ready, submit with their ids" — moves into JS, away
    from where `PlanSubmission` is defined.
- **Verdict:** Defensible if a richer client experience is the
  primary goal. For a reference impl whose job is to make the
  spec-to-code mapping legible, the costs outweigh the benefits.

#### A'. Browser-direct SPA (no BFF; browser holds the shared bearer)

Listed separately because it's the failure mode of a careless SPA
choice, not a defensible alternative on its own.

- **Cons:** Exposes the chapter 07 §12 shared bearer to the browser;
  forces a CORS story; deletes the wire/UI auth boundary the BFF
  preserves.
- **Verdict:** Explicitly out — included only to flag that picking
  "SPA" without the BFF is strictly worse than either A or the
  chosen stack.

#### B. Plain server-rendered Jinja with progressive enhancement (no HTMX)

This is the closest neighbor to the chosen stack.

- **Pros:** Exactly the same dependency graph minus 14 KB of JS;
  forms degrade to full-page POST + 303 redirect on every action;
  no client-side state to reason about; trivially testable with a
  plain HTTP client.
- **Cons:**
  - Claim-then-show-form needs either a server-side redirect or a
    two-page flow; not a deal-breaker but it's clunkier.
  - Any partial update — e.g. "you claimed task X, now this row
    moves out of pending" — either reflows the whole page or
    requires hand-rolled `fetch` + `outerHTML` swaps. HTMX
    automates exactly that with `hx-target` / `hx-swap`.
- **Verdict:** Live alternative, not a strawman. We pick HTMX for
  the partial-update ergonomics it gives the planner-list →
  claim-form transition; if HTMX adds friction during impl, the
  fallback is this option, not a heavier framework.

#### C. Plain HTML + vanilla `fetch` + manual DOM replacement

- **Pros:** No JS dependency at all.
- **Cons:** Reimplements HTMX badly. Every form ends up reinventing
  the same `fetch → response.text() → element.outerHTML = …`
  plumbing. The bug surface and code volume both grow per route.
- **Verdict:** Strictly dominated by option B (drop the JS entirely
  and reload) or the chosen stack (use HTMX for the swap).

#### D. Phoenix-LiveView-style server-pushed UI (e.g. via SSE/WebSocket)

- **Pros:** "Live" feel without a JS framework; great fit for the
  observability views in 9e.
- **Cons:** Phase 9 chunk 1 is request-response by scope. Pulling
  the live-update transport in now adds complexity (subscribe
  endpoint, reconnect logic) without a place to use it; a stranded
  human claim doesn't push, it times out. Revisit specifically for
  9e.
- **Verdict:** Right tool for 9e, wrong scope for 9a + 9b.

#### E. Server-side React via Next.js (or Remix)

- **Pros:** Server-rendered HTML at first paint, hydrates to an SPA;
  best-of-both for a production product.
- **Cons:** Brings the entire Node toolchain (and a build step) for
  marginal benefit over plain Jinja in a reference UI. The
  hydration step needs a JSON API anyway, recreating most of the
  SPA cost.
- **Verdict:** Wrong fit for a reference impl whose readers are
  evaluating spec conformance, not frontend architecture.

#### F. Server-side framework with built-in templates (Starlette, Quart, Flask)

- **Pros:** Same shape as FastAPI for our purposes.
- **Cons:** FastAPI is already in the dependency graph for
  `eden-wire`. Adding a second Python web framework just for the UI
  doubles the surface for "how does this differ from `eden-wire`?"
  questions in review.
- **Verdict:** No reason to introduce a second framework.

#### G. Embed the UI in `task-store-server`

- **Pros:** One fewer process to run.
- **Cons:**
  - Conflates the normative wire-protocol surface (chapter 07) with
    a non-normative UI; reviewing one no longer means reviewing
    only one. A third-party task-store-server reimplementation
    could not be expected to ship a UI.
  - The UI service holds the shared token on the browser's behalf
    and adds session-cookie state; baking that into the task-store
    pollutes a clean wire-protocol service.
- **Verdict:** Separate process is the right boundary.

#### H. CLI-only for Phase 9 chunk 1

- **Pros:** No new service at all.
- **Cons:** The roadmap exit criterion for Phase 9 is "a human can
  fully play any one role for at least one trial via the Web UI."
  CLI-only doesn't satisfy that, and Phase 10 (Compose)
  presupposes the UI service exists to dockerize.
- **Verdict:** Doesn't meet the phase exit bar.

### What this decision does **not** preclude

- A third-party SPA-style implementation against the same wire
  protocol — this is the whole point of the protocol-first repo.
- Switching off HTMX later if it turns out to be friction; the
  templates would still render fine on full-page submits.
- Phase 9e revisiting the request-response model with SSE for the
  observability view, which has the strongest case for live updates.

## What gets built

  `uvicorn`, `jinja2`, `itsdangerous` (for signed session

- `README.md` — what the service is, how to run it, the auth model,

- `src/eden_web_ui/__init__.py`
- `src/eden_web_ui/__main__.py` — launch via `python3 -m eden_web_ui`.
- `src/eden_web_ui/cli.py` — argparse with `add_common_arguments`
  (gives us `--task-store-url`, `--experiment-id`, `--shared-token`,

### B. Auth model (UI scope only — does not change wire protocol)

- The UI service is started with `--shared-token <T>` and passes
  `T` as the bearer to its `StoreClient`.
- The browser receives an `itsdangerous`-signed session cookie
  holding `{"worker_id": "<--worker-id>", "csrf": "<random hex>"}`
  once the user "signs in". The sign-in for Phase 9 is a
  no-questions form ("Continue as <worker-id>"); real auth is
  Milestone 3. The cookie carries the worker_id so subsequent
  claim/submit requests know who the human is, and a per-session
  CSRF token used by mutating routes.
- **Cookie attributes** are pinned: `HttpOnly=True`, `SameSite=Lax`,
  `Secure=False` (toggleable via `--secure-cookies` for deployments
  behind TLS — defaults off so localhost dev doesn't break).
  `Path=/`, no `Domain`. Cookie max-age tracks the session-secret
  rotation: any restart with a fresh `--session-secret`
  invalidates outstanding sessions.
- **CSRF protection** for every mutating route (POST /planner/...).
  A hidden `_csrf` form input carries the per-session token; the
  server compares against the cookie's `csrf` field via constant-
  time compare and rejects mismatches with 400. `SameSite=Lax`
  alone is **not** treated as sufficient — explicit token check
  is added because reference UIs in the wild often get cross-site
  POST'd through HTML form action attributes.
- **Bearer-token hygiene.** The `--shared-token` value never reaches
  the browser, the rendered HTML, the server logs, or the test
  fixture log files. Specifically:
  - The session cookie does not contain it.
  - JSON-line logs (via `eden_service_common.logging`) elide
    `Authorization` headers.
  - The bearer-leak test in `tests/test_security_invariants.py`
    (file inventory in §A) asserts no log line, no rendered
    response body, and no `Set-Cookie` value contains the token's
    string.
- **Session cookie does not bypass the wire boundary.** A leaked
  cookie can act on behalf of the worker_id it names (claim, submit
  through the UI) but cannot bypass the UI service to talk to the
  task-store directly — the shared bearer is held only in the UI
  process. Until Milestone-3 auth, "compromise of the cookie ==
  compromise of that worker_id's UI actions" is the documented
  threat model.

### C. Planner request flow (the spec-to-code map)

For each planner action, the corresponding `Store` (i.e.
`StoreClient`) calls are pinned here so the codex review can check
the implementation against this spec rather than rederiving it from
the templates.

The planner module is informed by the `experiment_config` loaded at
start-up. The planner page surfaces:

- The experiment's `objective` (read-only banner).
- The experiment's `metrics_schema` (read-only side panel showing
  metric names and types). The planner contract per
  `spec/v0/03-roles.md` §2 explicitly grants the planner read
  access to `metrics_schema` and `objective`; surfacing both is
  required to satisfy that contract.
- A "context" panel listing the most recent proposals
  (`store.list_proposals(state=None)` capped at the 20 most
  recent) and trials (`store.list_trials(status=None)` capped at
  the 20 most recent). This satisfies the implicit "what has
  already been tried" read access the planner needs to draft
  meaningfully.

The proposal form itself collects per-proposal slug, priority,
parent-commits, and rationale-markdown.

1. **List pending plan tasks** (`GET /planner`):
   - `store.list_tasks(kind="plan", state="pending")` — render rows.
2. **Claim** (`POST /planner/{task_id}/claim`):
   - CSRF check.
   - `claim = store.claim(task_id, worker_id=session.worker_id,
     expires_at=now() + timedelta(seconds=claim_ttl_seconds))`.
   - Save `claim.token` in the session keyed by `task_id`.
   - Render `planner_claim.html` with an empty proposal-row plus
     "add another" / "submit (success)" / "submit (error)" buttons.
3. **Submit success** (`POST /planner/{task_id}/submit`):
   - CSRF check.
   - Parse and **fully validate** N proposal rows from the form
     before any store mutation. Validation includes: slug
     pattern, priority float-parse, parent_commits non-empty,
     rationale-markdown non-empty.
   - **Phase 1 of the write — proposals as `drafting`.** For each
     proposal: write its rationale markdown to
     `<artifacts-dir>/<proposal_id>.md`, build a `file://` URI,
     then `store.create_proposal(Proposal(state="drafting", ...))`.
     A `drafting` proposal is invisible to the orchestrator's
     `list_proposals(state="ready")` dispatch path, so a
     mid-loop failure here orphans only `drafting` proposals,
     which are safe to ignore (no implement task is dispatched).
   - **Phase 2 of the write — flip to `ready`.** Once every
     `create_proposal` has returned, loop again and call
     `store.mark_proposal_ready(proposal_id)` for each.
   - **Phase 3 of the write — submit.**
     `store.submit(task_id, token, PlanSubmission(status="success",
     proposal_ids=tuple(proposal_ids)))`.
   - Drop the token from the session.
   - Render `planner_submitted.html`.

   The narrowest unsafe window — orphaned `ready` proposals not
   yet referenced by a submitted plan task — exists between
   Phase 2 and Phase 3. If Phase 3 fails after Phase 2 succeeded
   for at least one proposal, the orchestrator could in principle
   dispatch implement tasks for ready-but-unreferenced proposals.
   We accept this narrow window for the reference impl,
   documented as a known limitation; the spec-level fix
   (atomic ready-and-submit) is out of scope for Phase 9 chunk 1
   and applies equally to the existing scripted planner host.
   The error page on Phase-3 failure surfaces the orphaned
   proposal IDs explicitly so an operator can manually reclaim
   the plan task and let the orchestrator re-dispatch a fresh
   plan attempt.
4. **Submit error** (`POST /planner/{task_id}/submit?status=error`):
   - CSRF check.
   - `store.submit(task_id, token, PlanSubmission(status="error"))`.
   - No proposals are written.
   - Drop the token. Render `planner_submitted.html`.
5. **Phase-3 retry-before-orphan.** Per
   `spec/v0/07-wire-protocol.md` §2.4 (and §8.1 on idempotency) the
   resubmit semantics are defined: a content-equivalent resubmit
   returns 200, a divergent resubmit returns 409
   `eden://error/conflicting-resubmission`. Before declaring
   proposals orphaned, the UI retries the same `submit` call up
   to 3 times with exponential backoff on transport failures,
   and on a clean response treats the submission as committed.
   The orphaned-proposals error page is reached only after the
   retries are exhausted with an indeterminate result, or the
   server returns a definitive error.
6. **Reclaim path on expired/invalid claim token.** If any
   `store.claim` / `store.submit` returns one of the closed
   wire-protocol errors —
   - `eden://error/wrong-token` (presented token != stored
     token; chapter 07 §7),
   - `eden://error/illegal-transition` (state-machine
     violation, e.g. another worker reclaimed after expiry),
   - `eden://error/conflicting-resubmission` (resubmit
     disagreed with committed payload),
   - `eden://error/invalid-precondition` (e.g. trial state
     differs from required) —
   the form re-renders the planner list with a banner naming
   the canonical error type and offering "claim again." Field
   input is preserved across the re-render so the user does not
   lose work-in-progress.
7. **Errors** at any other point: re-render the form with
   field-level messages; do not lose user input.

`proposal_id` is generated server-side using `uuid.uuid4().hex`, the
same pattern the orchestrator service uses for task IDs.

### D. Expired-claim sweeper (cross-package, makes TTL operational)

Setting `expires_at` on every claim is just metadata; recovery requires
something that actually calls `reclaim(task_id, "expired")` on
expired claims. The reference stack has no such sweeper today; this
chunk adds one. Without it, the `--claim-ttl-seconds` flag would be
load-bearing on paper only.

- New module `eden_dispatch.sweep` exposing
  `sweep_expired_claims(store, *, now) -> int` (returns count
  reclaimed): iterates `store.list_tasks(state="claimed")`, parses
  each `expires_at` (skipping rows where it is `None`), and for
  each one strictly less than `now` calls `store.reclaim(task_id,
  "expired")`. Per-task errors are logged but do not abort the
  sweep — sweeping is best-effort; the next iteration retries.
- `reference/services/orchestrator/src/eden_orchestrator/loop.py`
  calls `sweep_expired_claims(store, now=datetime.now(UTC))` once
  per iteration before `run_orchestrator_iteration`. This couples
  expired-claim recovery to the same poll cadence as the rest of
  the dispatch loop, with no new background thread or wire-level
  notification. Adding the call here (vs. inside
  `run_orchestrator_iteration`) keeps the latter semantically
  pure: it stays the dispatch-and-finalize body, sweeping is a
  separate cross-cutting maintenance pass.
- New test
  `reference/packages/eden-dispatch/tests/test_sweep.py` covering:
  expired claim is reclaimed and emits `task.reclaimed`; non-expired
  claim is left alone; absent `expires_at` is left alone; per-task
  reclaim failure does not abort the sweep.
- The Phase 8b real-subprocess e2e test remains green: it
  doesn't exercise expired claims, but the sweeper is a no-op on
  fresh claims.

This is the smallest cross-package change that makes the UI's
TTL claim real today, and it's reusable by every future role
that issues claims (implementer / evaluator UI modules in 9c /

## Risks

- **Stranded human claims.** Mitigated end-to-end: the UI sets
  `expires_at = now + --claim-ttl-seconds` on every claim, and the
  new §D sweeper (`eden_dispatch.sweep.sweep_expired_claims`,
  invoked from the orchestrator service's loop) actually issues
  the `expired` reclaim. Phase 9e adds an explicit
  operator-reclaim button as a faster recovery path. The UI must
  not retry claims server-side for a user who doesn't return —
  that would mask the issue.
- **Partial proposal writes.** The Phase 1 → Phase 2 → Phase 3
  ordering in §C narrows the unsafe window to "ready proposals
  not yet referenced by a submitted plan task." The Phase-3
  retry-before-orphan policy (per chapter 07 §8.1 idempotent resubmit)
  collapses transport-only failures into successes; only a
  definitive divergent-payload server response or 3 indeterminate
  retries in a row falls through to the operator-recovery error
  page. A spec-level fix (atomic `mark_ready_and_submit`) is out
  of scope for Phase 9 chunk 1; the residual gap applies equally
  to the existing scripted planner host.
- **Experiment-config drift.** The UI loads its own copy of
  `experiment-config.yaml`; if it diverges from the
  task-store-server's copy, planner-side validation could pass on
  inputs the evaluator's metrics_schema would later reject. Phase
  12's control plane is the right fix. Until then there is no
  operational cross-check (`ExperimentConfig` carries no
  `experiment_id` field, so a content-equivalence comparison is
  not implementable today — §A documents this explicitly).
- **CSRF / cookie attribute regressions.** The cookie/CSRF policy
  is enforced by `tests/test_security_invariants.py`. Any change
  that loosens an attribute or skips the CSRF check breaks the
  test, by design.
- **Bearer-token leak via templating.** Mitigated by the
  bearer-leak unit test, the JSON-line log redaction, and the
  manual `grep -r "$EDEN_SHARED_TOKEN"` verification step. A
  reviewer should still spot-check any new template that
  receives the configuration object.
- **`itsdangerous` session-secret rotation.** Out of scope for this
  chunk; restarting the service with a different `--session-secret`
  invalidates outstanding sessions, which is the documented
  behavior. A real auth phase replaces this anyway.
- **Vendored HTMX bit-rot.** Pin the version in `pyproject.toml`
  comments and check the SHA-256 in `web-ui/README.md`; bumping is a
  one-line follow-up but not silent.
- **HTMX surprises.** If a partial-update interaction turns out to
  need a feature HTMX doesn't cleanly support, fall back to
  full-page submits for that flow rather than reaching for a JS
  framework. The CSS is already a single file; the templates degrade
  gracefully.
- **Test scope creep.** The existing Phase 8b e2e test takes ~3
  seconds because it forks five processes. Adding a second e2e test
  that forks two more (task-store + UI) keeps it on the order of
  another ~3 seconds; this is acceptable. If e2e total exceeds
  ~30 seconds we revisit consolidating, but not preemptively.
- **Reviewers' default expectation of a SPA.** The "Tech stack —
  decision and alternatives considered" section is deliberately
  detailed precisely because reviewers may default to "modern means
  React." The codex-review should specifically scrutinize that
  decision.
