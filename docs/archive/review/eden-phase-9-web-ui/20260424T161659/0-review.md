**Findings**

- **Critical** — The plan cannot yet satisfy the planner input contract because the UI has no source for experiment config or objective. The proposed CLI only wires `--task-store-url`, `--experiment-id`, `--shared-token`, `--host`, `--port`, `--session-secret`, and `--worker-id` in [eden-phase-9-web-ui.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-9-web-ui.md:174), and the app is explicitly scoped to a `StoreClient` in [eden-phase-9-web-ui.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-9-web-ui.md:54). But the planner contract requires access to experiment config, including `metrics_schema` and `objective`, in [03-roles.md](/Users/ericalt/Documents/eden/spec/v0/03-roles.md:40), and the `Store` protocol only exposes store data plus `experiment_id` in [protocol.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/protocol.py:88). As written, a human planner can claim and submit, but not actually perform the role with the required inputs.

- **Critical** — The submit flow does not handle partial proposal writes in a spec-safe way. The plan says to `create_proposal`, `mark_proposal_ready`, and only then `submit` in [eden-phase-9-web-ui.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-9-web-ui.md:249), while the error path is just `PlanSubmission(status="error")` in [eden-phase-9-web-ui.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-9-web-ui.md:257). The spec requires that on planner error, partially written proposals remain `drafting` and not be dispatched in [03-roles.md](/Users/ericalt/Documents/eden/spec/v0/03-roles.md:68). Because proposal creation and ready-marking are separate store mutations in [protocol.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/protocol.py:240) and [protocol.py](/Users/ericalt/Documents/eden/reference/packages/eden-storage/src/eden_storage/protocol.py:244), any mid-loop wire/store failure can leave `ready` proposals behind an unsubmitted task.

- **High** — The claim-lifecycle story is not safe enough for a human UI. The happy path claims without `expires_at` in [eden-phase-9-web-ui.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-9-web-ui.md:245), but the risks section assumes a “claim TTL” safety net in [eden-phase-9-web-ui.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-9-web-ui.md:350). The spec makes `expires_at` optional in [04-task-protocol.md](/Users/ericalt/Documents/eden/spec/v0/04-task-protocol.md:76); without it, abandoned browser claims rely only on operator reclaim or a task-store-defined health policy in [04-task-protocol.md](/Users/ericalt/Documents/eden/spec/v0/04-task-protocol.md:131), neither of which this chunk provides. For 9a+9b, that means a closed tab can strand a plan task indefinitely.

- **High** — The auth section understates the security requirements. The plan explicitly says “No CSRF token is added” and relies on same-origin cookie behavior in [eden-phase-9-web-ui.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-9-web-ui.md:228), while also claiming that a leaked cookie cannot bypass authorization in [eden-phase-9-web-ui.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-9-web-ui.md:78). That is not accurate: in this chunk, the cookie is the authorization for UI actions. If you keep this model, the plan should at minimum pin cookie attributes (`HttpOnly`, `Secure` when applicable, `SameSite`), origin/referrer checks or CSRF tokens for mutating routes, and “never log bearer or session material.”

- **High** — The stack analysis is directionally reasonable, but the alternatives section is partly strawmanned and misses the closest alternatives. The strongest reasons for FastAPI + Jinja2 are “Python-only repo”, “reuse `StoreClient`”, and “keep the shared bearer server-side” in [eden-phase-9-web-ui.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-9-web-ui.md:60). But the SPA critique in [eden-phase-9-web-ui.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-9-web-ui.md:93) assumes browser-direct task-store calls; a BFF-backed SPA would not require exposing the shared bearer. The doc also skips the most relevant nearby alternative: plain server-rendered Jinja/FastAPI with progressive enhancement and no HTMX. I would still likely choose the current stack, but the rationale needs to be tightened.

- **High** — The planner output model is incomplete because the plan never says how the markdown form becomes the required proposal artifact. The form layer is described as “markdown form” / `ProposalTemplate`-like inputs in [eden-phase-9-web-ui.md](/Users/ericalt/Documents/eden/docs/plans/eden-phase-9-web-ui.md:188), but each proposal must carry `artifacts_uri` in [proposal.py](/Users/ericalt/Documents/eden/reference/packages/eden-contracts/src/eden_contracts/proposal.py:32), and the planner contract requires uploading plan text / rationale and populating that URI in [03-roles.md](/Users/ericalt/Documents/eden/spec/v0/03-roles.md:50). The reference stack currently uses placeholder file URIs, which is fine for v0, but the plan needs to say so explicitly.

**Assessment**

- **Tech stack decision:** FastAPI + Jinja2 is a defensible choice for the reference UI. HTMX is also plausible, but it is not yet justified over plain server-rendered Jinja with selective enhancement. The current “reference-impl ethos” argument is weaker than the operational arguments, and the SPA alternative is framed too narrowly. The auth split “browser cookie, server bearer” does keep the wire protocol unchanged, but only if the UI continues to act as a BFF and you tighten the cookie/CSRF story.

- **Scope:** `9a + 9b` is the right chunk boundary; it matches the roadmap in [docs/roadmap.md](/Users/ericalt/Documents/eden/docs/roadmap.md:184). For `9c/9d/9e` to land cleanly later, this chunk needs durable invariants around role-neutral claim/session handling, experiment-context sourcing, artifact persistence, and recovery from partial writes and abandoned claims.

- **Spec-to-code correctness:** The pinned `list_tasks` / `claim` / `submit` calls are broadly aligned with the task protocol, and using `proposal_ids` as a set-valued submission matches [04-task-protocol.md](/Users/ericalt/Documents/eden/spec/v0/04-task-protocol.md:105). The gaps are the missing planner inputs, missing artifact strategy, and missing handling for mid-flight failures that would leave proposals `ready` before task submission succeeds.

- **Test plan:** The three-layer shape is good if each layer owns a distinct concern. Right now `TestClient` and `httpx.AsyncClient` against the same in-memory app are close to redundant unless the middle layer explicitly owns cross-request flow and recovery cases. Missing tests: expired/invalid claim token, abandoned claim with expiry, partial proposal-write failure, cookie attributes/CSRF policy, and a “shared bearer never leaks into HTML/cookies/logs” assertion.

- **Verification gates:** Useful baseline, but not exhaustive. I would add merge-blocking checks for the unhappy paths above, plus a documented claim-expiry/reclaim story and an automated assertion that the UI can recover from an expired token without creating duplicate or prematurely-ready proposals.

- **Risks:** The list is missing the biggest implementation risks: no source for experiment config, no artifact persistence plan for planner markdown, no recovery story for ambiguous `create_proposal` / `mark_proposal_ready` failures, and the fact that claim TTL is not actually present unless the UI sets it.

- **Backwards-compatibility / non-goals:** Mostly precise. The main tension is that “no wire/API changes” is only credible if the UI gains planner context via a local `--experiment-config` input or equivalent. If the only viable fix is a new wire endpoint, that should be called out now, not discovered mid-implementation.

**Overall**

Specific changes are required first; this is not ready to implement as-is.

The minimum fixes I’d make before coding are: add an explicit source for experiment config/objective, specify how planner markdown becomes `artifacts_uri`, define a spec-safe recovery strategy for partial proposal writes, require claim expiry or an equivalent stranded-claim mechanism for human use, and tighten the auth/alternatives section so the stack decision is argued on real constraints rather than a weakened comparison set.
