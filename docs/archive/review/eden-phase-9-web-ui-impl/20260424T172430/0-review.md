**Findings**
- High: the browser UI cannot actually author multiple proposals, even though the plan and manual smoke require it. `GET /planner/{task_id}/draft` always renders exactly one row via `row_indices = [0]` in [planner.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/planner.py:109), and [planner_claim.html](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/templates/planner_claim.html:27) has no “add another” control or alternate flow to grow the form. The tests mask this by hand-posting duplicate field names directly in [test_partial_write_recovery.py](/Users/ericalt/Documents/eden/reference/services/web-ui/tests/test_partial_write_recovery.py:24), which a human cannot produce from the shipped UI. As implemented, chunk 1 misses the plan’s multi-proposal planner surface.

- High: retry-before-orphan is not implemented for the real deployment path. `_retry_submit` only handles store-domain exceptions in [planner.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/planner.py:240), but the real `StoreClient` surfaces transport failures as raw `httpx` exceptions from `_request()` in [client.py](/Users/ericalt/Documents/eden/reference/packages/eden-wire/src/eden_wire/client.py:113) and `submit()` in [client.py](/Users/ericalt/Documents/eden/reference/packages/eden-wire/src/eden_wire/client.py:287). A dropped connection/read timeout therefore bypasses the retry/orphan logic and becomes a 500. I verified this locally by forcing `httpx.ConnectError` under `TestClient`; the route returned 500 instead of retrying.

- High: claim tokens are not stored “in the session keyed by task_id” as the plan says; they live in a process-global `_CLAIMS` map keyed only by `(worker_id, task_id)` in [planner.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/planner.py:41). Because sign-in always uses the configured `worker_id` in [auth.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/auth.py:29), a second browser session with the same worker id can open `/planner/{task}/draft` and reuse the first session’s claim. I reproduced this locally: session 2 received 200 on the draft page after only session 1 claimed the task. That weakens the UI auth boundary and breaks the intended claim ownership model.

- Medium: the promised expired-claim e2e coverage was weakened. The orchestrator loop does call the sweeper in [loop.py](/Users/ericalt/Documents/eden/reference/services/orchestrator/src/eden_orchestrator/loop.py:68), but [test_stranded_claim_recovers](/Users/ericalt/Documents/eden/reference/services/web-ui/tests/test_e2e_real_subprocess.py:256) does not prove that integration; it spawns only task-store-server + web-ui, then invokes `sweep_expired_claims()` directly over `StoreClient` at [test_e2e_real_subprocess.py](/Users/ericalt/Documents/eden/reference/services/web-ui/tests/test_e2e_real_subprocess.py:348). That covers the helper over HTTP, not the actual “once per orchestrator iteration” operational path the plan promised.

- Medium: the submit-time reclaim/error recovery path from the plan is not implemented. The plan said `wrong-token`, `illegal-transition`, `conflicting-resubmission`, and `invalid-precondition` on submit should return the user to the planner list with the canonical error name and preserved input. Instead `_retry_submit()` treats those outcomes as generic failure and [submit_plan()](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/planner.py:226) always renders the orphan page, losing the draft. There is no test covering the promised preserved-input recovery, and even the partial-write test file still claims a Phase 1 case that was never added in [test_partial_write_recovery.py](/Users/ericalt/Documents/eden/reference/services/web-ui/tests/test_partial_write_recovery.py:1).

**Notes**
- The core plan promises that are present look sound: `--experiment-config` is required and loaded in [cli.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/cli.py:27), claim TTL is applied in [planner.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/planner.py:97), artifact writes are tmp-and-rename atomic in [artifacts.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/artifacts.py:17), CSRF checks happen before planner-side store mutations in [planner.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/planner.py:95) and [planner.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/routes/planner.py:145), constant-time compare is in [sessions.py](/Users/ericalt/Documents/eden/reference/services/web-ui/src/eden_web_ui/sessions.py:64), and the sweeper semantics themselves match chapter 04 §5 in [sweep.py](/Users/ericalt/Documents/eden/reference/packages/eden-dispatch/src/eden_dispatch/sweep.py:32).
- The HTMX → plain-Jinja swap is still defensible for claim/list navigation. The problem is not “lack of HTMX” in general; it is that the fallback never replaced HTMX’s add-row ergonomics with any server-rendered alternative. I would not reintroduce HTMX just to save this chunk, but I would require an actual multi-row authoring flow before calling the fallback complete.
