## Change Summary

- `eden-git` now ships an `Integrator` that composes `GitRepo` with a `Store` to promote `success` trials to canonical `trial/*` commits. Order of operations: write commit object → `create_ref` (CAS) → `store.integrate_trial`; compensating `delete_ref` on store-side failure; `AtomicityViolation` if the compensating delete also fails.
- §2 preconditions (`status == success`, `commit_sha` reachable from `branch` tip), §1.4 reachability against declared `parent_commits`, and §2 metrics validity are all enforced up front. Metrics validation routes through the new public `Store.validate_metrics` method.
- §5.3 idempotency is verified end-to-end: second call on an already- promoted trial re-derives the manifest bytes, re-constructs the expected squash tree from the worker tip, and compares to the committed artifacts; `CorruptIntegrationState` on any mismatch.
- Eval-manifest bytes are deterministic (sorted keys, `indent=2`, trailing newline) to make the §5.3 re-derivation stable.
- The `eden-dispatch` driver now accepts `integrate_trial: Callable[[str], object]` (typically `Integrator.integrate`) instead of the Phase 5 placeholder `integrator_commit_factory`.
- Spec §3.4 was tightened to make the post-promotion reading of the cross-artifact invariant explicit, and a design note at `spec/v0/design-notes/integrator-atomicity.md` captures the full rationale and alternatives. This unblocked convergence of the plan review after three rounds where codex kept flagging the plan-local interpretation.
- All 336 tests pass; ruff and pyright are clean; markdownlint-cli2 is clean; `scripts/spec-xref-check.py` resolves all 240 §-refs.

## Review Status

### Round 0 — findings addressed

- **Bug — §5.3 replay after work/* pruning.** Fixed in `integrator.py::_check_idempotent`. The manifest-bytes check no longer depends on the worker commit being alive — it now reads the committed trial commit's tree and manifest blob directly. The tree-shape re-verification (§3.2 invariant) is now conditional on `commit_exists(trial.commit_sha)` being true, since §1.3 explicitly permits work/* cleanup. New regression test `TestIdempotentReplayAfterWorkerPruned` runs `git gc --prune=now` between integrations and asserts the second call is a no-op.
- **Bug — non-finite floats in metrics / manifest JSON.** Fixed in two layers: `_base.py::_validate_metrics` now rejects non-finite floats for `real` metrics via `math.isfinite()`; `_manifest.py` sets `allow_nan=False` on `json.dumps` as defense in depth. New test `TestNonFiniteMetricsRejected::test_nan_metric_rejected_at_integration` seeds a trial with `float("nan")` and asserts `NotReadyForIntegration`.
- **Risk — dispatch → real integrator coverage gap.** Fixed by `test_end_to_end_with_real_integrator` in `eden-dispatch/tests/test_end_to_end.py`. The test builds a real `GitRepo` in `tmp_path`, pre-creates three `work/*` branches, seeds the scripted implementer to return the real SHAs/branches, wires `Integrator.integrate` into `run_experiment`, and asserts the `trial/*` refs exist, the subject lines match §3.3, and the `trial.integrated` events land.

All 339 tests pass. ruff / pyright / markdownlint / spec-xref clean.

### Round 1 — findings addressed

- **Bug — §5.3 replay silently accepted corrupt tree after worker prune.** You reproduced this by rewriting the stored trial/* commit to a tree with an extra file but the same eval.json, deleting work/*, and running `git gc --prune=now`; the second `integrate()` accepted it as no-op. I agree that skipping the tree check when the worker commit is pruned is too permissive — §5.3's no-op path requires the tree to satisfy §3.2, and without the worker tree we cannot prove that. Fixed in `integrator.py::_check_idempotent`: when the worker commit is no longer reachable, the integrator now raises `CorruptIntegrationState` rather than silently treating the replay as a no-op. Updated regression test `TestReplayAfterWorkerPruned::test_replay_after_worker_gc_raises_corrupt_state` confirms this. New test `TestCorruptTrialTreeRejected::test_tree_with_extra_path_rejected` exactly reproduces your scenario (extra file in trial/* tree with the same eval.json, work/* still present) and asserts `CorruptIntegrationState`.

Deployment consequence: a deployment that wants §5.3 replay idempotence after eager work/* cleanup must either retain the worker commit/tree via a separate anchor ref, or accept that replay after cleanup surfaces as `CorruptIntegrationState` requiring operator intervention. This is the literal §5.3 reading; a looser reading (trust §1.2 to protect against external writers, skip tree re-verification) is permissible in a conforming third-party impl but the reference impl takes the strict reading.
