# Phase 7b ‚Äî Integrator flow (`eden_git.integrator`)

## Goal

Ship the integrator that composes the Phase 7a `GitRepo` subprocess wrapper with the Phase 6 `Store` to produce canonical `trial/*` commits per [`spec/v0/06-integrator.md`](../../spec/v0/06-integrator.md). Given a `success` trial with a recorded `commit_sha`, the integrator produces a single-commit squash whose tree is the worker-tip tree plus exactly the eval-manifest file at `.eden/trials/<trial_id>/eval.json`, writes the `trial/*` ref, writes `trial_commit_sha` on the trial, and appends `trial.integrated` ‚Äî atomically with respect to in-process observers.

See [`docs/roadmap.md`](../roadmap.md) ¬ßPhase 7 for context. Phase 7a is shipped (PR #12, merged `0fd58bd`).

## API surface

```python
class Integrator:
    def __init__(
        self,
        *,
        store: Store,
        repo: GitRepo,
        author: Identity,
        committer: Identity | None = None,
        clock: Callable[[], datetime] | None = None,
    ) -> None: ...

    def integrate(self, trial_id: str) -> IntegrationResult: ...
```

- `store` supplies trial reads + proposal lookup (for `slug`) + the composite `integrate_trial` commit.
- `repo` is the experiment's git repository (bare or non-bare). The integrator writes `trial/*` refs here.
- `author`/`committer` stamp the squash commit (¬ß6 leaves identity to the implementation).
- `clock` is injected for deterministic timestamps in tests; defaults to `datetime.now(tz=UTC)`.

```python
@dataclass(frozen=True)
class IntegrationResult:
    trial_id: str
    trial_commit_sha: str
    branch: str            # e.g. "trial/<trial_id>-<slug>"
    already_integrated: bool  # True on idempotent no-op path (¬ß5.3)
```

### Error hierarchy

```python
class IntegratorError(RuntimeError): ...                 # base
class NotReadyForIntegration(IntegratorError): ...       # ¬ß2: status != success, commit_sha absent, etc.
class ReachabilityViolation(IntegratorError): ...        # ¬ß1.4
class EvalManifestPathCollision(IntegratorError): ...    # ¬ß3.2: worker carried a file at the manifest path
class CorruptIntegrationState(IntegratorError): ...      # ¬ß5.3: trial_commit_sha set but ref missing / tree disagrees
class AtomicityViolation(IntegratorError): ...           # ¬ß3.4: compensating rollback itself failed
```

`IntegratorError` wraps `GitError` / `StorageError` only when the underlying operation should not be retried as-is. Otherwise those raise through unchanged so callers see the root cause.

## Algorithm

Executed in order, with the rollback rules documented in ¬ß3.4.

1. **Load trial.** `trial = store.read_trial(trial_id)`. Raise `NotFound` through.
2. **Idempotency check (¬ß5.3).**
   - If `trial.trial_commit_sha is not None`: - Resolve the SHA on the `trial/*` ref. If the ref exists, points at the recorded SHA, and the commit's tree satisfies ¬ß3.2 (tree derives from `trial.commit_sha`'s tree plus exactly the manifest path, manifest blob matches ¬ß4.2), return `IntegrationResult(..., already_integrated=True)`. - Otherwise raise `CorruptIntegrationState`.
3. **Promotion preconditions (¬ß2).** Verify:
   - `trial.status == "success"` ‚Äî else `NotReadyForIntegration`.
   - `trial.commit_sha is not None` ‚Äî else `NotReadyForIntegration`.
   - `trial.branch is not None`, the branch exists, and `trial.commit_sha` is reachable from the branch tip. The integrator normalizes the `work/*` branch name to a fully qualified ref before resolving, since `GitRepo.resolve_ref` requires `refs/heads/‚Ä¶` form (verified against Phase 7a's `show-ref --verify` implementation): `branch_ref = f"refs/heads/{trial.branch}"`. Then `tip = repo.resolve_ref(branch_ref)` ‚Äî `None` means branch absent ‚Üí `NotReadyForIntegration`. Reachability check: `tip == trial.commit_sha` OR `repo.is_ancestor(trial.commit_sha, tip)`. This matches ¬ß2 bullet 2 literally ("`commit_sha` resolves to a commit on the trial's `branch`") ‚Äî the branch tip need not equal `commit_sha`; additional commits pushed to `work/*` after evaluation are tolerated, they simply are not part of the squash tree. Raise `NotReadyForIntegration` if `commit_sha` is not reachable from the tip.
   - **Metrics validate against `metrics_schema` (¬ß2 last paragraph).** Call `store.validate_metrics(trial.metrics)`. Raises `InvalidPrecondition` on violation; the integrator surfaces this as `NotReadyForIntegration` and produces no `trial/*` commit, no `trial_commit_sha` write, no `trial.integrated` event (¬ß5.1). This is defense-in-depth: the orchestrator already validates at the evaluate terminal, but ¬ß2 is normative ("MUST NOT promote a trial whose `metrics` do not validate"), so the integrator enforces it directly rather than relying on the upstream guard.
4. **Proposal lookup.** `proposal = store.read_proposal(trial.proposal_id)`. The branch name uses `proposal.slug` per ¬ß3.1.
5. **Reachability check (¬ß1.4).** For each parent in `trial.parent_commits`, verify `repo.is_ancestor(parent, trial.commit_sha)`. On any failure raise `ReachabilityViolation`. This defends against a work branch whose tip does not descend from the declared parents.
6. **Manifest-path collision check (¬ß3.2).** If `repo.tree_entry_exists(trial.commit_sha, f".eden/trials/{trial.trial_id}/eval.json")` returns true, raise `EvalManifestPathCollision`. The integrator MUST NOT overwrite.
7. **Build squash tree.**
   1. Build the manifest JSON bytes per ¬ß4.2 via `_manifest.py`.
   2. `blob_sha = repo.write_blob(manifest_bytes)`.
   3. `worker_tree = repo.commit_tree_sha(trial.commit_sha)`.
   4. `tree_sha = repo.write_tree_with_file(worker_tree, f".eden/trials/{trial.trial_id}/eval.json", blob_sha, mode="100644")`. (`write_tree_with_file` already rejects on nested-path collisions ‚Äî belt-and-braces behind step 6.)
8. **Commit object.**
   - `message = f"trial: {trial.trial_id} {proposal.slug}\n"` (¬ß3.3).
   - `commit_sha = repo.commit_tree(tree_sha, parents=list(trial.parent_commits), message=message, author=self.author, committer=self.committer or self.author, author_date=..., committer_date=...)`.
   - Timestamps derive from `self.clock()`; tests inject a fixed clock.
9. **Create `trial/*` ref.** `branch = f"trial/{trial.trial_id}-{proposal.slug}"` `repo.create_ref(f"refs/heads/{branch}", commit_sha)` (CAS with zero-oid ‚Äî fails if the ref already exists).
10. **Atomic store write.** `store.integrate_trial(trial.trial_id, commit_sha)`. If this raises: `repo.delete_ref(f"refs/heads/{branch}", expected_old_sha=commit_sha)` to compensate. Then re-raise the original error. If `delete_ref` itself raises, wrap both errors in `AtomicityViolation` ‚Äî this is the one case where operator intervention is required (¬ß3.4).
11. **Return** `IntegrationResult(..., already_integrated=False)`.

### Atomicity design (¬ß3.4)

Three observable artifacts: the `trial_commit_sha` field on the trial, the `trial.integrated` event, and the `trial/*` git ref. ¬ß3.4 pins:

1. "The integrator MUST roll back any already-performed step" on failure.
2. "A dangling `trial/*` ref with no `trial_commit_sha` field and no `trial.integrated` event is a protocol violation."
3. "Implementations MAY use store-level transactions, outbox patterns, compensating deletes, or any other mechanism."
4. "The observable invariant is that a reader of any one of the three artifacts MUST observe the other two."

Rules 1+3 explicitly permit compensating deletes. Rules 2+4 define the invariant readers observe. Read together, the spec requires that **post-reconciliation** any reader sees a consistent tuple: success leaves all three, rollback leaves none. The Phase 7b integrator achieves this by making the `trial_commit_sha` field the **commit marker** ‚Äî the single artifact whose presence indicates a successful integration ‚Äî and ordering writes so the field lands last.

**Chosen order: [commit object, create ref, append event, write field].** The ref and event land *inside* `store.integrate_trial` as separate sub-steps of the store's atomic commit for the reference backends:

1. **Write commit object.** Orphan, content-addressed; invisible to ref-walking readers.
2. **`repo.create_ref`** (CAS with zero-oid). Gives ¬ß1.2 single-writer discipline a mechanical guard.
3. **`store.integrate_trial(trial_id, commit_sha)`**. The store's existing atomic operation appends `trial.integrated` and writes `trial_commit_sha` in a single transaction.
4. On failure of step 3: compensating `repo.delete_ref(..., expected_old_sha=commit_sha)`. Both ref and the two store writes are now absent.

**Reader discipline.** Protocol-conforming readers consult artifacts in the order: `trial_commit_sha` ‚Üí event log ‚Üí `trial/*` ref. The field is the single authoritative marker. Because it is written atomically with `trial.integrated` and strictly after `create_ref`, the following is guaranteed:

- Field set ‚áí ref written + event written. Reader transitions to examining the other two and finds them.
- Field absent ‚áí treat trial as not-integrated. Reader does not look at the ref even if it transiently exists pre-step-3. This satisfies the ¬ß3.4 invariant "a reader of any one of the three artifacts MUST observe the other two" ‚Äî a reader consulting the field as the canonical marker never encounters a state where the marker is set but the other two are missing.

The dispatch driver and every in-repo component check the field first (see `_promote_successful_trials` in `eden_dispatch/driver.py`, which already filters by `trial.trial_commit_sha is not None`). Phase 7b's design preserves that convention: the field's presence is the integration signal.

**External git readers.** A shell running `git log trial/*` directly has no access to the field and walks refs only. ¬ß3.4's "compensating deletes" mechanism is by construction incompatible with zero-width windows for such external readers ‚Äî rollback writes a ref, then deletes it. The reference implementation accepts this trade:

- On success (the common case), all three land and every reader observes all three.
- On rollback (the rare case), an external ref-walking reader MAY observe the ref before the compensating delete; after stabilization the ref is gone. A subsequent ref walk sees the rolled-back state. ¬ß3.4 explicitly names compensating deletes as a permitted mechanism; a permitted mechanism cannot simultaneously be a violation.

Stronger guarantees ‚Äî an instantaneous invariant surviving concurrent fs-level readers ‚Äî would require either (a) a two-phase commit coupling the store's SQL transaction with git's ref write, or (b) an outbox where the ref write is derived from committed store state. Neither mechanism is required by ¬ß3.4; both trade against design simplicity and wire-protocol scope, and are more natural to introduce in Phase 8 when the cross-process wire is defined.

**On compensating-delete failure** ‚Äî if step 3 raises and `repo.delete_ref(...)` also raises, neither rollback succeeded. The integrator raises `AtomicityViolation` wrapping both causes. This is the irrecoverable state ¬ß3.4 names as requiring implementation-defined mitigation; the wrapped exception identifies exactly which `trial/*` ref is dangling so an operator can delete it manually.

**Alternate orderings rejected:**

- `[commit object, store.integrate_trial, create_ref]`. If ref-create fails after the store committed, both the field and event are already written. Events are append-only per chapter 5; there is no "undo integration" event. The rollback path is strictly worse than the chosen order's single compensating ref-delete.
- Outbox pattern with a dedicated outbox table in the store. Would give stronger external-reader guarantees but requires extending the `Store` protocol and all backends with outbox read/claim/ack operations ‚Äî a disproportionate scope increase for Phase 7b. ¬ß3.4 explicitly lists outbox as *one* permitted mechanism, not *the* required mechanism. Revisit in Phase 8 when the wire-protocol introduces cross-process readers that actually benefit from it.

### What the integrator does NOT do

- **Does not observe events or poll.** The integrator is a passive function: callers drive `integrate(trial_id)` when they decide. The dispatch loop (`eden_dispatch.driver`) already has the `_promote_successful_trials` scheduler; it will be updated to call `Integrator.integrate(trial_id)` in place of its current placeholder `integrator_commit_factory` hook.
- **Does not manage work-branch retention** (¬ß1.3). The integrator neither deletes nor renames `work/*` branches; that is a deployment concern.
- **Does not populate optional manifest fields** `evaluator` or `artifacts` (¬ß4.3, ¬ß4.4). The v0 role contracts in `03-roles.md` ¬ß4.4 do not surface evaluator identity or per-file artifact inventory in the evaluator's submission shape; when role-bindings add that metadata (Phase 10's LLM evaluator is the likely source) the integrator grows two optional `Callable` hooks. Keeping the shape minimal now avoids inventing a submission field ahead of spec demand.

## New `Store.validate_metrics` method

The integrator needs access to the experiment's `metrics_schema` to enforce ¬ß2's "MUST NOT promote a trial whose `metrics` do not validate." The schema is a `Store` construction parameter (private `self._metrics_schema`) and `_validate_metrics` already implements the check for submit-time validation. Phase 7b promotes it to a public `Store` method:

```python
# In Store Protocol (reference/packages/eden-storage/src/eden_storage/protocol.py)
def validate_metrics(self, metrics: dict[str, Any]) -> None:
    """Validate metrics against the experiment's metrics_schema.

    Raises ``InvalidPrecondition`` on violation; returns None on pass.
    No-op when the store was constructed without a ``metrics_schema``.
    Chapter 8 ¬ß4 (storage) and chapter 6 ¬ß2 (integrator) both
    reference this guard; this is the public entry point for both.
    """
    ...
```

`_base.py` exposes the public method as a thin wrapper:

```python
def validate_metrics(self, metrics: dict[str, Any]) -> None:
    self._validate_metrics(metrics)
```

**Why on Store, not on the integrator or a separate validator object.** The schema lives in the store; exposing a standalone validator would require duplicating the construction path that wires the schema in, for no gain. Both submit-time and integration-time validation pass through the same code, which is the invariant ¬ß2 actually wants (the two validators agree by construction).

**Parameter choice.** `metrics: dict[str, Any]` rather than `trial_id: str`. The integrator already has the `Trial` object loaded in step 1; passing its `metrics` dict through avoids a redundant store round-trip and keeps the validator unit-testable without a trial fixture.

## Eval-manifest builder (`_manifest.py`)

Separate module so manifest shape has its own test target. Signature:

```python
def build_manifest(trial: Trial) -> bytes: ...
```

Builds a JSON object with the required fields from ¬ß4.2 (sourced directly from the `Trial` model). Emits the optional `description` and `artifacts_uri` only when present on the trial. Produces UTF-8 bytes with `sort_keys=True` + `indent=2` + trailing newline so the blob is byte-stable across repeated promotions for idempotency check in step 2.

- Required fields copied verbatim from the trial: `trial_id`, `proposal_id`, `commit_sha`, `parent_commits`, `metrics`, `completed_at`.
- Optional `artifacts_uri` and `description` copied only if present.
- No synthesis. No transformation. ¬ß4.2 closing paragraph: "a conforming integrator MUST NOT synthesize or transform required-field values."
- Raises `ValueError` if any required trial field is `None`. Surfaces as `NotReadyForIntegration` at the integrator layer (the conditions that prevent those fields from being set are the same as ¬ß2's preconditions; this is a guard, not a rewrite).

## Updating `eden-dispatch` driver

The driver currently accepts an `integrator_commit_factory: Callable[[str], str] | None` that returns a fake commit SHA. Phase 7b replaces this with a real `Integrator` call:

```python
def run_experiment(
    ...,
    integrator: Integrator | None = None,
) -> None:
```

`_promote_successful_trials` becomes:

```python
def _promote_successful_trials(store: Store, integrator: Integrator) -> bool:
    progress = False
    for trial in store.list_trials(status="success"):
        if trial.trial_commit_sha is not None:
            continue
        integrator.integrate(trial.trial_id)
        progress = True
    return progress
```

The `integrator_commit_factory` parameter is removed entirely ‚Äî it was a Phase 5 placeholder, not a supported surface. Driver tests either supply a real `Integrator` backed by a test `GitRepo` or omit the integrator argument and assert trials remain at `status=success` with `trial_commit_sha=None`.

## Tests

### `test_manifest.py`

- Required-field shape: builds manifest from a fully populated trial, asserts every ¬ß4.2 required key + value equal to trial field.
- Optional fields: omitted when absent, present when set.
- Byte stability: two calls on the same trial produce identical bytes (load-bearing for idempotency in `test_integrator.py`).
- Rejects when any required field is `None` (trial with no `commit_sha` / `metrics` / `completed_at`).

### `test_integrator.py`

Set up with a non-bare `GitRepo`, an `InMemoryStore`, a real proposal and trial seeded into the store, and a real worker branch at `trial.commit_sha` that descends from `trial.parent_commits`.

**Success path:**

- `integrate(trial_id)` returns `already_integrated=False`.
- `refs/heads/trial/<trial_id>-<slug>` exists and resolves to a commit whose parents are `trial.parent_commits` in order.
- Commit subject line is `trial: <trial_id> <slug>`.
- Commit tree's entry at `.eden/trials/<trial_id>/eval.json` is a blob matching `build_manifest(trial)`.
- Every other tree entry equals the corresponding entry under `trial.commit_sha`'s tree (no drift, no drops).
- `store.read_trial(trial_id).trial_commit_sha == result.trial_commit_sha`.
- `store.events()` contains one `trial.integrated` event referencing the trial + sha.

**Idempotent replay (¬ß5.3 bullet 1):**

- Call `integrate(trial_id)` twice. Second call returns `already_integrated=True` with the same SHA. No new ref write, no new event. Assert len(events) equals expected + 1 total (not + 2).

**Corrupt state (¬ß5.3 bullet 2):**

- Trial has `trial_commit_sha` set but ref does not exist ‚Üí raises `CorruptIntegrationState`, no side effects.
- Trial has `trial_commit_sha` set and ref exists but tree does not satisfy ¬ß3.2 (manifest missing or extra file) ‚Üí `CorruptIntegrationState`.

**Reachability rejection (¬ß1.4):**

- Seed a trial whose `commit_sha` is unreachable from its `parent_commits` (e.g. by rewriting `commit_sha` to point to a separate commit chain). `integrate(trial_id)` raises `ReachabilityViolation`. Assert no ref created, no `trial_commit_sha` written, no `trial.integrated` event.

**Manifest-path collision (¬ß3.2):**

- Seed a worker branch whose tree already contains `.eden/trials/<trial_id>/eval.json`. Raise `EvalManifestPathCollision`. No side effects.

**Precondition failures (¬ß2):**

- `status=starting`, `status=error`, `status=eval_error` each raise `NotReadyForIntegration`. No ref, no field, no event.
- `commit_sha=None` ‚Üí `NotReadyForIntegration`.
- `branch` does not exist ‚Üí `NotReadyForIntegration`.
- `branch` resolves to a SHA unrelated to `trial.commit_sha` (not an ancestor either direction) ‚Üí `NotReadyForIntegration`.
- **`branch` tip is equal to `commit_sha`** ‚Üí promotion succeeds (edge of ¬ß2 bullet 2 where the branch hasn't advanced).
- **`branch` tip is a descendant of `commit_sha`** ‚Üí promotion succeeds (¬ß2 "commit_sha resolves to a commit on the branch" is satisfied when commit_sha is reachable from branch tip, per plan step 3). Asserts the squash tree derives from `commit_sha`, not from the branch tip ‚Äî extra commits on the branch are excluded from the squash.

**Metrics re-validation (¬ß2, ¬ß5.1):**

- A trial with metrics containing a key absent from the experiment's `metrics_schema` ‚Äî seeded directly via store internals (bypassing submit-time validation) ‚Äî raises `NotReadyForIntegration` when passed to `integrate`. Assert no ref, no field, no event.
- A trial with metrics of the wrong declared type raises similarly.
- A store constructed without a `metrics_schema` integrates trials with arbitrary metrics (no-op validator path).

**Atomic rollback (¬ß3.4):**

- Monkey-patch `store.integrate_trial` to raise. Run `integrate(trial_id)`. Assert:
  - The `trial/*` ref was deleted (compensating write).
  - The original `store.integrate_trial` exception propagates through `Integrator.integrate`.
  - Trial `trial_commit_sha` remains `None`.
  - No `trial.integrated` event appears.
  - The commit object is still in the object DB (orphan; GC will clean it). Verified via `repo.commit_exists(sha)` returning True.
- Monkey-patch both `store.integrate_trial` and `repo.delete_ref` to raise. Assert `AtomicityViolation` surfaces, wrapping both errors.

**Commit identity:**

- Custom `author` / `committer` / `clock` are honored. Commit has `commit.gpgsign=false` (inherited from `GitRepo`).
- Default `committer = author` when only `author` is supplied.

**Multi-parent proposal:**

- A trial with two `parent_commits` produces a commit with two parents in order. (¬ß3.2 supports multi-parent squash.)

## Known risks

- **Manifest byte-stability is load-bearing.** The idempotency check in step 2 re-derives `build_manifest(trial)` and compares to the committed blob. If serialization is not deterministic (insertion order, whitespace, trailing newline), a second invocation will falsely report corruption. Enforced via `sort_keys=True`, `indent=2`, trailing `\n`, and a dedicated byte-stability test.
- **Compensating ref-delete can fail.** In pathological cases (fs error during rollback), both writes fail and we surface `AtomicityViolation`. This is the operator-intervention state (¬ß3.4); tests exercise it and assert the wrapped cause is visible.
- **Zero-oid hash-algo.** Inherited from Phase 7a ‚Äî `GitRepo` queries `rev-parse --show-object-format` and caches. Integrator's `create_ref` / `delete_ref` calls route through that.
- **Ambient git config.** Inherited from Phase 7a's `_sanitized_git_env()` + `commit.gpgsign=false` per invocation; extends to every integrator invocation automatically.
