# Implementation Review: eden-phase-9c-implementer-module

## Plan
- Path: `docs/plans/eden-phase-9c-implementer-module.md`
- Summary: Add the implementer role to the reference Web UI service. The user does git work out-of-band, pushes their tip commit to the bare repo, then claims an `implement` task in the UI and submits the resulting `commit_sha`. The UI verifies §3.3 reachability locally, creates a `starting` trial, writes the `work/<slug>-<trial_id>` ref, then submits with retry-before-orphan + a committed-state read-back.

## Change Summary

- Added an opt-in implementer module to the reference Web UI service, gated on a new `--repo-path` CLI flag. When the flag is omitted the implementer routes return 404 and the navigation hides the link, preserving pre-9c invocation surface.
- Implemented the spec-to-code flow per §C: validate the form, run §3.3 reachability via `GitRepo.commit_exists` + `is_ancestor` against every parent in `proposal.parent_commits`, run a Pre-Phase-1 ref-collision guard, then `create_trial(starting)` → `repo.create_ref(work/...)` → `store.submit` with retry-before-orphan and a `read_task` + `read_submission` + `submissions_equivalent` read-back on retry exhaustion.
- `trial_id` is server-only (kept in an in-process `_CLAIMS` dict alongside the claim token); no hidden form field; a forged form value is ignored.
- Added the §A.1 artifact-rendering trust boundary: only `file://` URIs that resolve inside `--artifacts-dir` and are ≤ 1 MiB render inline; all other shapes render link-only.
- Five test files cover per-route validation, the cross-request flow + reachability rejections + status=error, every branch of §C-recovery via monkeypatched failures, security invariants (bearer/CSRF/cookie/trust-boundary/`trial_id` non-leak), and a real-subprocess two-process e2e test driving the full flow over real HTTP.
- All gates pass locally: `uv run pytest -q` (494 passed), `uv run pytest -m e2e -q` (4 passed including the new implementer e2e), `uv run ruff check .` (clean), `uv run pyright` (clean), markdownlint (clean).

## Review Status

### Round 0 (codex)
4 findings. All addressed:

1. **Bug — `artifacts_uri` rendered as `<a href>` without scheme check** in `implementer_claim.html`. Fixed by gating the `<a href>` on `scheme in {http, https, file}`; other schemes render as plain `<code>` with an "(unrenderable scheme)" note. Added `TestArtifactUriRendering::test_javascript_uri_is_not_hyperlinked` to pin the guarantee.
2. **Risk — read_task probe failure returned `auto`** in `_retry_submit_with_readback`. Changed to return `transport` so the orphan template renders the indeterminate prose, not the misleading "auto-recovers" prose. Added `TestReadbackProbeFailure::test_read_task_failure_renders_transport_orphan` which asserts the orphan page contains "read-back failed" and does **not** contain "auto-recovers".
3. **Risk — Phase 1 `create_trial` transport failure crashes the route**. Wrapped in `except Exception` (after the existing `DispatchError` catch) and rendered the orphan page with `recovery_kind="transport"`. The orphan template's transport branch was generalized to cover all three indeterminate cases (read-back probe failure, store invariant violation, Phase 1 transport failure). Added `TestPhase1TransportFailure::test_create_trial_transport_failure_renders_indeterminate_orphan`.
4. **Risk — sub-case A / A' tests didn't exercise the read-back path** because the first `submit` returned cleanly. Rewrote both monkeypatches: the first (and every subsequent) call commits via `original_submit(...)` AND raises a transport-shape exception, forcing the route's retry loop to drain to exhaustion and reach the read-back. Sub-case A' additionally calls `store.reject` between commit-and-raise to put the task in `failed` before the read-back observes it.
